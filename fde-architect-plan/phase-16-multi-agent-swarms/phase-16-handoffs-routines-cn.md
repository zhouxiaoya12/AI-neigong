# 路由与例程——无状态编排

> OpenAI 的 Swarm（2024 年 10 月）将多智能体编排提炼为两个原语：**例程**（指令 + 工具作为系统提示）和**路由**（一个返回另一个 Agent 的工具）。没有状态机、没有分支 DSL——大语言模型通过调用正确的路由工具来路由。OpenAI Agents SDK（2025 年 3 月）是生产级后继。Swarm 本身仍是最清晰的概念参考——它的全部源代码仅几百行。这种模式具有病毒式传播力，因为 API 表面大致是"智能体 = 提示 + 工具；路由 = 返回智能体的函数"。局限：无状态，因此记忆是调用者的问题。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** 第16阶段 · 04（原语模型）
**时间：** 约60分钟

## 问题

每个多智能体框架都要求你学习它的 DSL：LangGraph 的节点和边、CrewAI 的 crew 和任务、AutoGen 的 GroupChat 和管理器。DSL 是真实的抽象，但它们让事情感觉比需要的更重。

Swarm 朝相反方向推进：使用模型已有的工具调用能力。路由成为工具调用。编排器是当前持有对话的智能体。状态机隐含在智能体的系统提示中。

## 概念

### 两个原语

**例程。** 一个定义智能体角色和可用工具的系统提示。把它看作一组有范围的指令："你是一个分诊智能体；如果用户询问退款，路由给退款智能体。"

**路由。** 智能体可以调用的一个工具，返回一个新的 Agent 对象。Swarm 运行时检测 Agent 返回值并切换活跃智能体以进行下一轮。

这就是整个抽象。

```python
def transfer_to_refunds():
    return refund_agent  # Swarm 看到 Agent 返回 → 切换活跃智能体

triage_agent = Agent(
    name="triage",
    instructions="将用户路由到合适的专家。",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

分诊智能体的系统提示使其基于用户消息选择正确的路由。大语言模型的工具调用完成路由。

### 为什么具有病毒式传播力

- **小 API。** 仅需学习两个概念。
- **使用模型已有的能力。** 工具调用在各提供商已是生产级别的。
- **没有状态机负担。** 你不描述图；智能体的提示描述它们路由给谁。

### 无状态的代价

Swarm 在运行之间显式无状态。框架在运行期间维护消息历史，但不持久化任何内容。记忆、连续性、长运行任务——全部是调用者的问题。

在生产中（OpenAI Agents SDK，2025 年 3 月），这是主要变化之一：SDK 添加了内置的会话管理、护栏和跟踪，同时保留了路由原语。

### Swarm/路由适用的场景

- **分诊模式。** 前线智能体将用户路由到专家。
- **基于技能的路由。** "如果任务需要代码，调用编码员；如果需要研究，调用研究员。"
- **短的有界对话。** 客户支持、FAQ 到工单、简单工作流。

### Swarm 困难的场景

- **带共享记忆的长会话。** 路由将对话状态重置为新智能体的提示加历史。在调用者管理记忆的情况下，智能体之间没有持久状态。
- **并行执行。** 路由是一次一个——活跃智能体切换。并行需要调用者编排多个 Swarm 运行。
- **审计和重放。** 无状态运行难以精确重放；大语言模型的路由选择不是确定性的。

### OpenAI Agents SDK（2025 年 3 月）

生产级后继添加了：

- **会话状态。** 跨运行的持久线程。
- **护栏。** 输入/输出验证钩子。
- **跟踪。** 每次工具调用和路由被记录。
- **路由过滤器。** 控制路由时传输什么上下文。

路由原语保留下来；生产便利性围绕它添加。

### Swarm vs GroupChat

两者都使用大语言模型驱动的路由，但它们在**谁选择下一个**上不同：

- GroupChat：一个选择器（函数或大语言模型）从外部选择下一个发言者。
- Swarm：当前智能体通过调用路由工具选择其后继。

Swarm 是"智能体决定接下来是什么"；GroupChat 是"管理者决定接下来是什么"。Swarm 的决策存在于活跃智能体的工具调用中；GroupChat 的决策存在于 `GroupChatManager` 中。

## 构建它

`code/main.py` 从头实现 Swarm：一个 Agent 数据类、一个路由机制（工具返回 Agent）和一个检测智能体切换的运行循环。

演示：一个分诊智能体路由到退款、销售或客服专家。每个专家有自己独立的工具。运行循环打印每次路由。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-handoff-designer.md` 为给定任务设计路由拓扑：存在哪些智能体、它们可以调用哪些路由、传输什么上下文。

## Ship It

检查清单：

- **路由日志。** 每次路由写入带有来源智能体、目标智能体、上下文快照的跟踪事件。
- **上下文传输规则。** 决定路由时传输什么：完整历史（昂贵）、最近 N 条消息或摘要。
- **路由护栏。** 路由到具有不同工具权限的专家必须经过认证——否则提示注入可以强制不需要的路由。
- **循环检测。** 两个智能体来回路由是常见故障；用简单的最近 K 项环形检查来检测。
- **回退智能体。** 如果路由目标不存在，回退到安全的默认值。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份以极简主义哲学贯穿的技术教材——"两个原语（例程 + 路由）就是全部抽象"的核心主张，本身就是一种教学设计。文档将 OpenAI 的 Swarm（概念参考）与 Agents SDK（生产后继）并置，形成了"原则→局限→修复"的完整叙事：Swarm 演示什么是可能的最简单形式，Agents SDK 演示什么必须在生产中添加（会话、护栏、跟踪）。这种"先纯粹后实用"的结构比直接从生产框架开始更能建立深层理解。

### 二、知识结构梳理

**认知基础：** 多智能体编排可以简化为 LLM 已经具备的能力——工具调用。例程是"角色定义"，路由是"角色切换工具"。关键在于：状态机不是被显式描述的，而是从智能体的系统提示中隐式涌现。

**工程模式：** Swarm 的无状态设计（运行间不持久化）既是优雅的来源（最小 API 表面），也是生产化的主要障碍（记忆是调用者的问题）。Agents SDK 在保留路由原语的基础上添加了会话状态、护栏、跟踪和路由过滤器四个生产化维度。

**落地实践：** Swarm vs GroupChat 的对比——前者是"智能体选择后继"（自组织），后者是"管理者选择后继"（中心化）。路由循环检测、上下文传输策略（完整/最近N条/摘要）、回退智能体构成容错体系。

### 三、核心洞察

1. **"几百行源代码"是 Swarm 最强大的特性。** 不是功能多，而是简洁到足以让工程师在一个下午理解全部抽象。为什么重要？在框架膨化的时代（每个新框架都引入更多 DSL 概念），'少即是多'本身就是架构立场。几百行的参考实现比几十页的文档更有教学价值。

2. **无状态是架构选择，不是限制。** Swarm 坚持无状态不是因为做不到有状态，而是因为"记忆责任"应该在正确的架构层解决——调用者管理记忆，Swarm 管理路由。为什么重要？它区分了"编排逻辑"（Swarm 的领域）和"持久化逻辑"（调用者的领域），这种关注点分离在复杂系统中极其珍贵。

3. **"智能体选择后继" vs "管理者选择后继"是两种格格不入的路由哲学。** Swarm 的自组织（agent-decides）更灵活但更难审计，GroupChat 的中心化（manager-decides）更可控但引入单点瓶颈。为什么重要？选择 Swarm 还是 GroupChat 不是技术偏好问题，是对"自主权 vs 可审计性"的权衡。

4. **路由循环是不够显式的状态管理的最直接后果。** 两个智能体在没有中央管理者的情况下互相路由，形成无限循环。为什么重要？它是自组织系统最经典的故障模式——局部决策最优不等于全局行为最优。循环检测（最近 K 项环形检查）是最小可行的全局保护。

5. **路由过滤器的引入揭示了"上下文传输"作为一个独立工程问题的重要性。** 路由时传输全部历史会污染新智能体的上下文；传输太少会丢失关键信息。正确的答案是"刚好够"——而"刚好够"是一个需要显式设计的策略。为什么重要？它在两个原语模型的基础上添加了第三维度：不是"谁在说话"或"怎么切换"，而是"切换时带走什么"。

6. **Swarm 到 Agents SDK 的演化轨迹是 AI 工程的缩影。** 2024 年 10 月的概念原型→2025 年 3 月的生产级框架，6 个月内完成的添加（会话、护栏、跟踪、过滤器）恰好定义了"从 demo 到生产"的差距。为什么重要？它为学生提供了一个真实的案例研究：任何 AI 系统从原型到生产的过程，本质上是围绕极简核心添加-ility（reliability, auditability, governability）。

### 四、教学建议

1. **先让学生手写路由互的逻辑。** 不介绍 Swarm 的 `Agent` 数据类和自动检测，而是让学生手动实现："当一个工具返回一个 Agent 对象时，将活跃智能体改为该 Agent。"这个 10 分钟的练习会让他们深刻理解 Swarm 为什么简洁——核心机制只有几行代码。

2. **运行 `code/main.py` 三次，每次修改路由目标。** 第一次正常运行；第二次故意让路由到不存在的智能体（触发回退）；第三次让两个智能体互相路由（触发循环）。让学生看到"正确运行"和"故障运行"的完整图谱。

3. **用"电话转接"作为类比。** 分诊智能体=前台总机，路由=转接电话，活跃智能体=当前通话人。学生只需要理解总机如何决定转给谁——这就是 Swarm 的全部。

4. **对比表练习。** 让学生填写一个表格：P1 列是"Swarm 模式"，P2 列是"GroupChat 模式"，对比维度包括"谁决定路由"、"状态在哪"、"提示注入风险"、"审计难度"、"并行支持"。这种结构化的对比让学生不只是"知道两个模式"，而是能"判断该用哪个"。

5. **让学生阅读 Swarm 的源代码。** 既然只有几百行，就应该直接读源代码。标注出：(a) 路由检测的代码，(b) 运行循环的代码，(c) 无状态的证据（没有数据库、没有文件写入）。

6. **用"添加会话状态"作为扩展练习。** 在 Swarm 的极简运行循环之上，让学生添加一个简单的会话存储（内存中 dict 即可）。体验"从无状态到有状态"的过程——哪些代码需要修改，哪些不需要。

### 五、值得补充的内容

1. **路由的原子性保证。** 如果在路由过程中发生异常（网络故障、模型超时），系统应该留在哪个智能体？是重试、回退还是升级给人类？

2. **多步骤工作流中的路由模式。** Swarm 展示的是单步路由（分诊→专家）。如果工作流需要"分诊→专家A→专家B→综合"的多跳路由链，Swarm 模式如何优雅处理？

3. **路由决策的可解释性。** 大语言模型为什么选择路由给专家A而不是专家B？这个决策的解释应该用什么格式记录（日志、结构化字段、还是自然语言理由）？

4. **路由与权限的关联设计。** 不同的智能体有不同的工具权限——某些工具只有特定专家可以访问。路由时如何验证目标智能体的权限？如果分诊智能体错误地将用户路由给具有过高权限的智能体，风险是什么？

5. **Swarm 模式在非 OpenAI 生态系统中的等价实现。** Anthropic 的 Claude Code 子智能体（`Task`）和 LangGraph 的工具调用形式都可以看作路由模式的不同实现——它们的对应关系和实现差异值得系统比较。

### 六、一句话总结

Swarm 的哲学是所有多智能体编排中最极致的极简主义——两个原语定义全部（例程是角色，路由是切换），其他都是调用者的责任；这种"不做太多"的克制恰恰是它病毒式传播的原因，也是它需要生产级 SDK 扩展才能安全部署的原因。

# 🎓 Agent 架构课：路由与例程——几百行代码怎么干掉几万行的框架

同学们好。我是你们的FDE工程老师，今天讲的是 路由与例程。

**LangGraph 要你学节点+边。CrewAI 要你学 crew+task+process。AutoGen 要你学 GroupChat+Manager。Swarm 只要两个原语——例程和路由——全部代码几百行。**

例程 = 一个智能体的系统提示 + 可用工具。路由 = 一个返回另一个 Agent 对象的函数。工具调用本身变成了路由机制：

```python
def transfer_to_refunds():
    return refund_agent  # Swarm 检测到返回 Agent → 切换活跃智能体

triage_agent = Agent(
    instructions="你是分诊员。退款请求 → 路由退款智能体。",
    tools=[transfer_to_refunds]
)
```

没有状态机 DSL，没有图配置。编排隐含在智能体自己决定"调用哪个路由工具"的过程中。局限：无状态——记忆是调用者的问题。但大部分多智能体编排不需要持久状态机。

## 结课清单

1. **两个原语 = 例程 + 路由。** 足以覆盖大多数多智能体编排。
2. **路由 = 工具调用。** 模型自己决定何时换人——不需要图配置。
3. **总代码量 = 几百行。** 如果一个框架需要你学三天文档，先问是不是真的需要。

最后一句话：**最简单的多智能体架构不是最强大的框架——是最少的抽象。Swarm 用几百行证明了：有时候你只需要一个智能体能把球传给另一个智能体。**

# 💼 从业者故事：路由与例程——几百行代码怎么干掉几万行的框架

2024 年 10 月，我在做一个客服系统的多智能体改造。当时市面上有三个选择：LangGraph、CrewAI、AutoGen。我花了两个星期读文档，每个框架的抽象都不一样——LangGraph 是图（节点+边），CrewAI 是船员（crew+task+process），AutoGen 是群聊（GroupChat+Manager）。每个都要求你学一套新的 DSL，每个都有几百页文档。我当时的感受是：这帮人是不是在用框架复杂性来证明框架价值？

然后 OpenAI 发布了 Swarm。我看了 README，翻了下源代码——**只有几百行。** 两个概念：路由（routine）和路由（handoff）。没有图。没有 DSL。没有状态机。就两个。

我当时在工位上笑出了声。不是因为 Swarm 多强——它连持久化都没有。是因为它证明了一件事：**多智能体编排的核心不是复杂架构，是大语言模型已有的工具调用能力。** 你不需要学一个新框架。你只需要让你的智能体调用一个返回另一个智能体的函数。

## 两个原语——用大白话重新讲

**例程（Routine）** 就是你的智能体。一个 system prompt + 一堆工具。"你是分诊台的。用户说退款就转给退款专家，说下单就转给销售，其他转给客服。"

**路由（Handoff）** 就是你的智能体调用的一个函数，返回值是另一个智能体。Swarm 的运行循环检测到这个返回值，然后下一轮对话由新智能体接手。

就这。整个 Swarm 的"框架"就是一个 while 循环：

```python
while True:
    response = active_agent.call(messages)
    if response is an Agent:   # 这就是路由
        active_agent = response
    else:
        break  # 结束了
```

你看。while 循环 + if 判断。几百行代码的精华就是这 5 行。剩下的代码全是日志、错误处理和一些示例智能体。

这就是为什么它具有病毒式传播力。不是因为功能多，是因为**你学完它只需要 10 分钟**。相比 LangGraph 的 StateGraph + Node + ConditionalEdge 的六层抽象，Swarm 简单到像是在嘲讽整个行业。

## 但无状态是一把双刃剑——"记忆是调用者的问题"翻译过来就是"你自求多福"

Swarm 最大的设计选择是无状态——每次运行之间不保存任何东西。智能体的上下文在调用结束后就蒸发了。如果你想让智能体记住"用户上次说他的订单号是 ABC123"，你自己记。

这在 demo 里很优雅。在生产里是一场噩梦。

我团队第一次在生产用 Swarm 的时候，用户问"你上次不是说要给我退款 50 元吗"，智能体说"我没有上次的记录"。用户炸了。因为我们忘了——路由不是"传递记忆"，路由是"切换说话的人"。新的智能体只知道系统提示 + 当前消息历史。它不知道 3 轮之前的对话里退款智能体承诺了 50 元。

这就是为什么 OpenAI Agents SDK（2025 年 3 月）加了会话状态。不是"Swarm 做错了"，是"无状态不适合需要记忆的场景"。Swarm 是为短对话设计的——分诊、路由、简单问答。超过 5 轮的对话，你需要 SDK 的持久会话。

## Swarm vs GroupChat——不是谁好谁坏，是你信任谁

两种路由哲学：

- **Swarm（智能体决定）**：当前智能体通过调用路由工具选择"下一个是谁"。决策在智能体内部。
- **GroupChat（管理者决定）**：一个中央管理者观察所有智能体，选择"下一个是谁"。决策在管理者内部。

用人话翻译：

- Swarm 是"你想跟谁说话？你自己选"。
- GroupChat 是"你们别吵，我来决定你跟谁说话"。

Swarm 更灵活——智能体可以基于自己的判断选择后继。但也更危险——一个被提示注入欺骗的智能体可以把用户路由给具有更高权限的智能体。GroupChat 更安全——管理者统一把关。但也更慢——每轮都需要管理者的判断。

选哪个？问自己一个问题：**你更怕智能体自己做错决定，还是更怕中央管理者成为瓶颈？** 如果你做的是企业级权限系统（退款只能退款专员做），GroupChat 更安全。如果你做的是轻量级路由（FAQ 分诊），Swarm 更快。

## 四个生产陷阱——都是自己踩过的

**陷阱一：路由循环——两个智能体互相甩锅。** 分诊智能体说"这是技术问题，转给技术"。技术智能体说"这不归我管，转回分诊"。分诊智能体又说"这是技术问题..."。两个 LLM 互相推诿的优雅程度堪比真人客服。解决方案简单到爆：维护最近 3 次路由的智能体对，如果 A→B→A→B 出现，强制终止并升级给人类。

**陷阱二：上下文传输的三种策略，选错的代价是信息丢失或上下文污染。** 路由时可以传完整历史（贵、污染新智能体的窗口）、最近 N 条消息（可能丢失关键上下文）或摘要（有损压缩）。我现在的默认策略是：短对话（<10 轮）传完整历史，长对话传摘要（由上一个智能体在路由前生成）。永远不要传"全部什么都不删"——那跟单智能体的上下文饱和是一样的死法。

**陷阱三：路由目标不存在。** 你的退款智能体下线了（你改了名字或者删了），但分诊智能体的路由工具里还有 `transfer_to_refunds`。分诊智能体调用它，返回了一个不存在的智能体。你的 while 循环怎么办？必须有回退智能体——一个兜底的"抱歉，当前无法处理，请稍后再试"的角色。不是优雅降级，是保底不死。

**陷阱四：把 Swarm 的几百行当成生产方案。** Swarm 的 README 里写得很清楚——"Swarm 是实验性的，不适合生产。"但我知道一定会有人把它直接扔到生产环境（我见过）。它缺少会话持久化、缺少护栏、缺少跟踪、缺少路由过滤器。这些不是"nice to have"，是"没它们就不知道为什么炸了"。从 Swarm 学概念，用 Agents SDK 做生产。

## 最后一句

Swarm 告诉我们：多智能体编排不需要 DSL、不需要图、不需要状态机——你把"切换说话的人"做成一个函数返回值，大语言模型的工具调用就帮你路由了。但记住，极简是理解的最佳工具，不是生产的最佳状态。你的框架从 Swarm 开始，但不应该停在 Swarm。

> 发朋友圈的素材：几百行代码的多智能体框架，核心逻辑是一个 while 循环里检测函数返回值是不是 Agent。如果返回值是 Agent，切换。不是 Agent，结束。你的微服务框架有多少行？
