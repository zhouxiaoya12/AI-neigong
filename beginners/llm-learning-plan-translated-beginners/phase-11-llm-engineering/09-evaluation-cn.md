# LLM 评估：别说"看起来不错"，用数据说话

> **原文：** Evaluation & Testing LLM Apps  
> **预计时间：** ~25 分钟  

---

## 🚗 开场翻车：改了系统提示，答案完整性掉 34%，11 天没人发现

你发布了一个客服聊天机器人。演示完美。两周后，有人改了系统提示减少幻觉。"改对了"——幻觉率下降。但答案完整性也下降了 34%，因为模型拒绝回答不够 100% 确定的事。

**11 天没人注意到。** 自助服务收入下降，工单激增。

**"凭感觉评估" = 祈祷。祈祷不是工程实践。**

---

## 🧠 三种评估方法

| 方法 | 速度 | 成本 | 最适合 |
|------|------|------|--------|
| **自动指标** | 快 | 便宜 | BLEU/ROUGE——快速近似 |
| **LLM-as-Judge** | 快 | 中等 | 用强模型按评分标准打分 |
| **人工评估** | 慢 | 贵 | 金标准——校准自动化评估 |

---

## 🔧 怎么做

```python
# 评估数据集 = 输入 + 期望输出 + 边缘案例
eval_set = [
    {"input": "退款政策？", "expected": "30天内全额退款"},
    {"input": "怎么退订？", "expected": "设置→订阅→取消"},
    # ...至少 50-100 条
]

# 每次改提示/换模型/调参数后跑一遍
for case in eval_set:
    output = llm.generate(case["input"])
    score = judge.score(output, case["expected"])
    if score < threshold:
        alert("质量退化！")
```

---

## 🏆 焊死在脑子里的东西

1. **"看起来不错"不是评估——是祈祷。** 5 个例子测不出退化。
2. **LLM-as-Judge 是性价比最高的自动评估。** 与人类判断相关 82-88%。
3. **每次变更都跑评估。** 质量退化时阻止部署。

---

> **你不会在没测试的情况下部署 Web 应用。同理，别在没评估的情况下改 LLM 提示。评估是你和"用户悄悄流失"之间唯一的东西。**
