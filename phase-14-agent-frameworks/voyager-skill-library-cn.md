# 技能库与终身学习（Voyager）

> Voyager（Wang et al., TMLR 2024）将可执行代码视为技能。技能有名称、可检索、可组合，并通过环境反馈精炼。这是 Claude Agent SDK skills、skillkit 以及 2026 年技能库模式的参考架构。

**类型：** 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 07（MemGPT）、Phase 14 · 08（Letta Blocks）
**用时：** 约 75 分钟

## 学习目标

- 说出 Voyager 的三大组件——自动课程、技能库、迭代提示——以及各自的作用。
- 解释为什么 Voyager 将动作空间设为代码，而非原始命令。
- 实现一个基于标准库的技能库，包含注册、检索、组合和失败驱动精炼。
- 将 Voyager 模式映射到 2026 年 Claude Agent SDK skills 与 skillkit 生态。

## 问题

每个会话都从零重建所有能力的 Agent，犯了三个错误：

1. **浪费令牌。** 每个任务都重新激发相同的推理。
2. **丢失进度。** 在会话 A 中学到的纠正无法迁移到会话 B。
3. **长程组合失败。** 复杂任务需要能力层级；一次性提示表达不了它们。

Voyager 的答案：把每个可复用能力当作一块命名代码，存入库中，按相似度可检索，可与其他技能组合，并通过执行反馈精炼。

## 概念

### 三大组件

Voyager（arXiv:2305.16291）围绕以下三者组织 Agent：

1. **自动课程。** 好奇心驱动的提议器，基于 Agent 当前的技能集合和环境状态选择下一个任务。探索是自底向上的。
2. **技能库。** 每条技能是可执行代码。任务成功时添加新技能。技能按查询与描述的相似度检索。
3. **迭代提示机制。** 失败时，Agent 收到执行错误、环境反馈和自我验证输出，然后精炼技能。

Minecraft 评估结果（Wang et al., 2024）：比基线多 3.3 倍独特物品，制石工具快 8.5 倍，制铁工具快 6.4 倍，地图穿越距离长 2.3 倍。这些数字是 Minecraft 专属的，但模式是可迁移的。

### 动作空间 = 代码

大多数 Agent 发出的是原始命令。Voyager 发出的是 JavaScript 函数。一条技能是：

```javascript
async function craftIronPickaxe(bot) {
 // 挖 3 块铁矿
 await mineIron(bot, 3);
 // 砍 2 根木棍
 await mineStick(bot, 2);
 // 放置工作台
 await placeCraftingTable(bot);
 // 合成铁镐
 await craft(bot, 'iron_pickaxe');
}
```

由子技能组合而成。以描述和嵌入向量为键存储。以程序的形式被检索，而不是提示。

这就是 2026 年 Claude Agent SDK 的技能：一个命名的、可检索的代码块加上指令，Agent 按需加载。

### 技能检索

新任务"制作一把钻石镐"。Agent：

1. 将任务描述嵌入为向量。
2. 在技能库中查询 top-k 相似技能。
3. 检索到 `craftIronPickaxe`、`mineDiamond`、`placeCraftingTable` 等。
4. 从检索到的原语加上新逻辑，组合出新技能。

这就是 MCP resources（Phase 13）和 Agent SDK skills 实现的模式：在知识/代码表面进行检索，范围限定于当前任务。

### 迭代精炼

Voyager 的反馈回路：

1. Agent 编写一个技能。
2. 技能在环境中运行。
3. 返回三种信号之一：`success`（成功）、`error`（含栈追踪）、`self-verification failure`（自我验证失败）。
4. Agent 用该信号作为上下文重写技能。
5. 循环直到成功或达到最大轮次。

这是 Self-Refine（第 05 课）在代码生成上的应用，以环境为真值进行验证。CRITIC（第 05 课）是同一模式，只是以外部工具为验证器。

### 课程与探索

Voyager 的课程模块——基于 Agent 已有的和尚未完成的事情——提议任务，如"在湖边建造庇护所"。提议器使用环境状态 + 技能库存来选择刚好超出当前能力的任务——探索的最佳区间。

对于生产级 Agent，这转化为一个"缺失什么"操作符：给定当前技能库和领域，还有哪些技能我们尚未覆盖？团队通常以课程审查的方式手动实现这一步骤。

### 此模式在何处会出错

- **技能库腐化。** 同一条技能被添加了 10 次，描述略有不同。在写入时做去重；检索只返回一条。
- **组合技能漂移。** 父技能依赖于已被精炼的子技能。给技能做版本管理；父技能锁定 v1 就不会悄悄获得 v3。
- **检索质量。** 当库增长到几百条以上时，基于技能描述的向量检索会退化。用标签过滤器和硬约束作为补充（"仅 `category=tooling` 的技能"）。

## 构建

`code/main.py` 实现一个基于标准库的技能库：

- `Skill`——名称、描述、代码（字符串）、版本、标签、依赖。
- `SkillLibrary`——注册、搜索（令牌重叠度）、组合（依赖拓扑排序）、精炼（更新时版本号递增）。
- 一个脚本化 Agent，注册三条原语技能，组合第四条，遇到失败，然后精炼。

运行：

```bash
python3 code/main.py
```

输出展示库写入、检索、组合、一次失败执行、以及 v2 精炼——Voyager 的端到端循环。

## 使用

- **Claude Agent SDK skills**（Anthropic）——2026 年参考：每条技能有 description、code 和 instructions；Agent 会话期间按需加载。
- **skillkit**（npm: skillkit）——跨 Agent 技能管理，支持 32+ AI 编码 Agent。
- **自定义技能库**——领域特定（数据 Agent 的 SQL 技能，基础设施 Agent 的 Terraform 技能）。Voyager 模式可以向下缩放。
- **OpenAI Agents SDK `tools`**——在谱系低端；每个 tool 是一个轻量技能。

## 发布

`outputs/skill-skill-library.md` 为任意目标运行时生成一个 Voyager 风格的技能库，内置注册、检索、版本管理和精炼。

## 练习

1. 为 `compose()` 添加依赖循环检测器。当技能 A 依赖 B 而 B 又依赖 A 时会发生什么？报错还是警告？
2. 实现按技能版本锁定。当父技能组合了 `crafting@1` 时，`crafting@2` 的精炼不得悄悄升级父技能。
3. 用 sentence-transformers 嵌入向量（或 BM25 标准库实现）替换令牌重叠度检索。在 50 条技能的玩具库上测量 retrieval@5。
4. 添加一个"课程"Agent：给定当前库和领域描述，提议 5 条缺失的技能。每周调用一次。
5. 阅读 Anthropic 的 Claude Agent SDK skill 文档。将玩具库移植到 SDK 的 skill schema。可发现性有何变化？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 技能（Skill） | "可复用能力" | 命名代码块 + 描述，可按相似度检索 |
| 技能库（Skill Library） | "Agent 的怎么做记忆" | 技能的持久化存储，可搜索、可组合 |
| 课程（Curriculum） | "任务提议器" | 由当前能力缺口驱动的自底向上目标生成器 |
| 组合（Composition） | "技能 DAG" | 技能调用技能；执行时做拓扑排序 |
| 迭代精炼（Iterative Refinement） | "自修正循环" | 环境反馈 + 错误 + 自我验证折叠回下一版本 |
| 代码即动作空间（Action-Space-as-Code） | "可编程动作" | 发出函数而非原始命令，以获得时间扩展行为 |
| 写时去重（Dedup on Write） | "技能折叠" | 近似重复的描述折叠为一条规范技能 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Agent 工程课程中工程密度最高的一课。它不讲抽象理论，而是直面一个所有做 Agent 的人都撞过的墙——可复用性缺失。文档的最大优势在于"模式抽象"：把 Voyager 论文中 Minecraft 特定的实现提炼为三个通用组件（课程、技能库、迭代提示），然后直接映射到 2026 年的工程现实（Claude Agent SDK、skillkit）。适合有生产 Agent 构建经验的工程师，不适合纯算法背景、无工程直觉的新手。其最大亮点是它将一个学术论文模式直接翻译为可操作的工程实践。

### 二、知识结构梳理

**认知基础层：** Self-Refine（第 05 课）是迭代精炼的前置知识；MemGPT 和 Letta Blocks（Phase 14·07-08）为"Agent 的记忆可以跨越一次对话"这一前提打下基础。学生必须先理解：Agent 的状态不必在每次会话后归零。

**工程模式层：** 四个关键设计决策构成了整个技能库模式——① 动作空间选择代码而非命令、② 检索依赖语义相似度而非精确匹配、③ 组合基于 DAG 拓扑排序而非线性拼接、④ 精炼由失败驱动而非人工干预。每个决策都是"为什么不选另一个"的教学点。

**实践应用层：** Claude Agent SDK skills 是工业级产品化，skillkit 是跨 Agent 生态的连接器，自定义领域技能库是该模式在不同规模下的应用。三层从"看懂原理"到"能落地"再到"能推广"递进。

### 三、核心洞察（备课时的关键理解）

1. **技能是代码，不是自然语言。** Voyager 最被低估的设计决策。提示是模糊的——LLM 每次解读都有偏差。代码是确定性的——相同输入永远得到相同输出。但更重要的是：代码可以被另一个函数调用、可以被执行引擎验证、可以用版本号追踪变更。这三个性质是自然语言根本不具备的。

2. **技能库解决的是"冷启动"问题。** 没有持久化记忆的 Agent 每次对话都在付"认知启动税"。有技能库的 Agent 像是已经工作了三年的工程师——大部分时候在回忆已掌握的方案，少数时候在创新。这直接转化为令牌消耗的下降。

3. **迭代精炼是该模式能跑通的必要条件。** 没有任何 Agent 能一次性写对复杂技能。Voyager 的优雅之处在于把"失败"设计为系统的燃料，而非故障。三元反馈（成功/错误/自验失败）比二元反馈（过/不过）多提供了调试所需的信息量——自验失败尤其关键，因为能跑不意味着做对了。

4. **检索质量是技能库的阿喀琉斯之踵。** 20 条技能时 Jaccard 足够。200 条时向量嵌入勉强扛住。2000 条时连向量都开始退化——需要标签筛选、分层索引、多路召回。这是生产 Agent 最容易低估的工程问题：检索引擎不是"装上就行"的东西，它是需要持续维护的。

5. **组合技能的版本漂移是一颗定时炸弹。** 对 `smeltIron@v1` 的一次精炼可能悄悄改变所有依赖它的父技能的行为。这跟 npm 的依赖地狱是同一个问题——只不过出错对象从 CI 管道变成了你的 Agent。版本锁定是最小化解决方案，但长期看需要技能兼容性契约。

6. **课程模块是"元认知"的工程实例化。** "我下一步该学什么？"不是人类独有的问题。Voyager 把它变成一个可计算的差距分析——已有技能 vs. 领域需求。这对 Agent 的自主学习、多任务泛化、以及在开放世界中持续成长至关重要。

7. **这个模式的最大亮点是向下和向上同时缩放。** 3 条技能也能用，300 条也能用。初级 Agent 和复杂 Agent 共享同一套 API，仅有的区别是库的大小和检索的精确度。好的架构具备双向缩放能力——这是它的标志。

### 四、教学建议

1. **用对比表格开场，不要用定义。** 直接让学生看到"有技能库 vs. 无技能库"在令牌消耗、错误重复率、长程任务成功率上的差异。感官冲击比概念阐述有效十倍。

2. **现场跑 `main.py` 是这节课的核心，不是附件。** 让学生亲眼看到 v1 在冶炼步骤失败、v2 修复后成功。这个"见证失败→修复→成功"的体验，比任何 PPT 都深刻。预留完整的 30 分钟编码时间。

3. **"动作空间 = 代码"值得单独拿出 5 分钟争论。** 问学生："你认为 Agent 应该发出自然语言命令，还是函数调用？"先让他们争论 3 分钟，然后你给出三列对比表（精确性、可组合性、可验证性）。争论后的结论记得更牢。

4. **检索质量退化问题用类比引入。** 先让他们写 Jaccard 检索，然后问："如果库里有 1000 条技能，怎么保证 Top-5 的相关性？"这个问题让嵌入向量、标签过滤、多路召回的需求变得自然。

5. **用 npm 依赖地狱来类比技能版本漂移。** `smeltIron@v1` 变了接口，所有依赖它的父技能都"坏了"——npm 用户的日常噩梦。让学生意识到 Agent 技能库的管理问题和软件包管理是同构的。

6. **三个陷阱（腐化、漂移、检索退化）放在一个"技能库增长时间线"里讲。** 当库从 10 条长到 1000 条，这三个问题依次爆发。用一个叙事把它们串联起来，比分别讲解更有记忆点。

7. **练习 5（Claude Agent SDK 移植）是最有价值的延伸。** 它让抽象概念和 2026 年的真实产品对接。如果课堂时间允许，带学生看一眼真正的 SDK API 文档——这节课就从学术讨论变成了可操作的职业技能。

### 五、值得补充的内容

1. **技能路由（Skill Router）。** 库里有 500 条技能时，Top-K 检索结果里可能混入不相干的东西。需要一个路由层来筛选"与当前上下文相关"的技能子集，而不只是按相似度排序。

2. **技能冲突检测。** 两条检索到的技能给出了矛盾方案（一个说"直接合成"，一个说"先冶炼"）——Agent 怎么裁决？这需要比"执行后失败"更高层的精炼机制。

3. **跨 Agent 技能可移植性。** skillkit 解决了格式统一问题，但没解决语义一致性问题。Agent A 的"挖矿"技能在 Agent B 的环境里可能完全不能工作，因为环境假设不同。

4. **冷启动问题。** Voyager 依赖环境探索来生成技能，但刚上线的生产 Agent 什么技能都没有。需要种子技能集（seed skills），或从已有文档自动生成初始技能。

5. **安全与权限。** 如果技能是任意可执行代码，生产环境中谁审查其安全性？谁决定"这个 Agent 能跑哪些技能"？Agent SDK 已用白名单机制解决，但教学文档中未展开。

### 六、一句话总结

**技能库不是让 Agent 变得更聪明——而是让它不再为已经学会的事情重新支付认知成本。代码替代提示，检索替代推理，精炼替代遗忘：这就是 Agent 的终身学习。**

# 🎓 Agent 架构课：技能库——让 Agent 记住自己学会的事

同学们好。我是你们的 Agent 架构老师。

今天这节课，我们先不谈模型架构、不谈推理优化。那些是算法工程师头疼的事。我们今天的核心问题是——**你的 Agent 是怎么"忘记"事情的？** 以及，怎么让它不再忘记。

## 想象你是一个 Agent，我们来做个思想实验

假设你是一个 Minecraft Agent。上一轮对话，用户说："帮我做个铁镐。" 你花了一堆 token 去想：第一步挖铁矿，第二步砍木棍，第三步放进熔炉，第四步合成。中间还出了个小错——你忘了冶炼步骤，直接把矿石扔进了合成台。试了两次才做对。终于成功了。

这一轮对话，用户说："帮我做个钻石镐。"

你现在会怎么做？

如果你没有技能库，你会**从头开始推理**。从"钻石怎么获取"开始。每一步都是新的。更惨的是——你上次在冶炼环节踩过的坑，这一次会再踩一遍。

这就是我们行业里不说但心知肚明的"Agent 健忘症"。它有三个具体症状：

- 💸 **Token 浪费。** 同一段推理被重复产生，反复计费。
- 🔄 **错误重复。** 在会话 A 已经学会的事，会话 B 完全不记得——因为 Agent 没有"学到"这个概念。
- 🧩 **长程任务崩溃。** 真正有用的任务——比如"从零搭建一个 Minecraft 基地"——需要十几步。没有可复用的能力，一次性 prompt 写不下，也写不对。

## 两条路，两种哲学

面对这个问题，业界走出了两条截然不同的路。

**第一条路：微调。** 把 Agent 的底层模型用历史对话再训练一轮。每做对一次任务，就更新一次模型权重。这条路听起来很理想——让 Agent"内化"它的经验。但问题是什么？微调是改大脑。你每学会一个新技能就做一次微调？那张 GPU 账单能把你的预算烧穿。而且微调后的模型只保证概率倾向，不保证精确行为。

**第二条路：技能库。** 不要改 Agent 的大脑。给它一本外挂的"技能笔记本"。每次任务，Agent 翻开笔记本查："我会做这个吗？" 会就直接用，不会就现场创造，做对了再记进去。

这就是 Voyager 的基本直觉：**代码是你学会的事，检索是回忆的过程，精炼是你从错误中学习。**

一个微调让模型"总体上更厉害"——一个技能库让模型"精确地记住该怎么做某件事"。这两者在 Agent 架构中的位置完全不同。

## 三大组件：拆开看 Voyager 是怎么工作的

Voyager 把一个持续学习的 Agent 拆成了三个组件。我用自己的话讲——不背定义，我们看它为什么这样设计。

### 组件一：自动课程——"我下一步该学什么？"

这个问题非常难。如果你给 Agent 一个固定的任务列表，它只能做你预设过的事。如果你随机给任务，它会浪费大量时间在要么太简单、要么太难的任务上。

Voyager 的方案是：**选刚好高出当前能力的任务。** 这是 Vygotsky 的"最近发展区"思想——教育学的经典概念，被 Voyager 变成了一个工程机制。

具体怎么做？Agent 扫一眼自己已有的技能（"我已经会挖矿、会冶炼"），然后看一眼环境（"附近有钻石矿脉"），然后提议："挖三颗钻石，做个钻石镐。"

为什么这很聪明？因为课程模块不需要知道"钻镐怎么做完"——它只需要知道"钻石镐比铁镐多一个步骤，正好够难"。

在生产系统中，你不需要一个完整的自动课程系统。大多数团队每周跑一次差距分析：用 LLM 扫描已有技能清单和领域需求，自动列出 5-10 条缺失的技能。这就是你的周度"Agent 学习计划"。

### 组件二：技能库——"我已经学会做什么了？"

这是整个 Voyager 的心脏。

一条技能是什么？**一段可执行的代码，加上描述、标签、版本、依赖关系。** 不是自然语言。不是"如何做铁镐"的一段说明。是真正的、能跑的函数。

为什么选代码？我告诉你三个理由，它们加在一起就是判决书。

**第一，精确性。** "把一个铁矿石放进熔炉，等待它变成铁锭"——这个自然语言描述，LLM 每次读都可能理解得略有不同。但 `smelt_iron(ores) → iron_ingot` 这个函数调用——输入和输出是确定的，永远不变。

**第二，可组合性。** 代码天然支持函数嵌套。`craftIronPickaxe` 调用 `mineIron` 再调用 `smeltIron`。这个调用关系图就是技能依赖 DAG。你不需要额外设计"组合语言"——Python 的函数调用就是你的组合语言。

**第三，可验证性。** 代码运行后要么返回结果，要么抛出异常。自然语言写的东西你只能靠 LLM 自检"看着对不对"——而那是不靠谱的。

检索是怎么做的？Agent 收到"做一个钻石镐"：

1. 把这句话编码为查询向量
2. 在技能库里找描述最接近的 Top-K 条技能
3. 拿到 `craftIronPickaxe`、`mineDiamond`、`placeCraftingTable`
4. 用这些做积木，加上新逻辑，拼出 `craftDiamondPickaxe`

这里有件事我现在就要提醒你：**检索质量会随着库增长而退化。** 20 条技能时随便什么相似度算法都好用。200 条时你需要正经的嵌入向量。2000 条时向量也会开始给你不相关的结果——你需要标签过滤、分层索引、甚至多路召回来兜底。生产中的技能库运维成本不在代码里，在检索引擎上。

### 组件三：迭代精炼——"错了，该怎么改？"

Voyager 做的最漂亮的设计决策之一是：**把"执行失败"当成燃料，而不是错误。**

标准的工作流：写代码 → 跑 → 失败 → 人手动改。
Voyager 的工作流：写代码 → 跑 → 收到信号 → 自动重写 → 再跑 → 直到成功或超限。

注意它收到的不是"成功/失败"二元信号，而是三元：

- ✅ `success`——入库，结束。
- ❌ `error`（带完整栈追踪）——分析哪里出错，重写。
- ⚠️ `self-verification failure`——代码能跑但结果不对——重写。

第三种信号是关键。代码不抛异常不等于做对了事。如果你的函数少调了一步冶炼，它不会报错——它会静默返回错误的结果。Voyager 用一个 LLM 自检步骤来抓这类问题："这段代码的操作顺序对吗？最终真能产出铁镐吗？"

它的底层就是 Self-Refine——我们在第 5 课讲过的迭代自修正模式。区别在于验证源：普通 Self-Refine 靠 LLM 自己看——Voyager 靠执行环境给你真值反馈。

## 生产的真相：这个模式在真实世界会怎么坏

好，讲完原理，我现在跟你们说几个在生产里真正发生的事情。这些不是论文里写的——是我见过的。

### 陷阱一：技能库腐化

你的 Agent 用了三个月。技能库里出现了 12 条名字和内容几乎一模一样、只是描述差了几个词的技能。你检索"做铁镐"，Top-5 里有 4 条是重复的。

解决方案：**写时去重。** 每次注册新技能，先跟库里已有的技能比描述相似度。超过 80% 相似？合并，不新增。别怕误合并——比一堆重复好得多。

### 陷阱二：组合技能漂移

你的 `craftIronPickaxe@v1` 调用了 `smeltIron@v1`。然后有人精炼了 `smeltIron`，现在它是 `smeltIron@v2`，接口变成了 `smelt_iron(ores, fuel_type)`。你的 `craftIronPickaxe` 还什么都不知道——但它开始报参数错误了。

这就是版本管理的缺失。技能不是无版本的。每改一次，版本号 +1。父技能锁定依赖的版本号——`craftIronPickaxe@v1` 永远依赖 `smeltIron@v1`，除非你做显式的兼容性迁移。技能库的版本管理和 npm 的 `package.json` 是同构问题。

### 陷阱三：循环依赖

技能 A："我能做铁镐，但我需要铁锭 → 依赖 B"
技能 B："我能做铁锭，但我需要铁镐来挖矿 → 依赖 A"

A 依赖 B，B 依赖 A。你的 `compose()` 函数在这时会做什么？如果你用的是拓扑排序（Kahn 算法），它会自然地暴露这个问题——排序后的节点数会少于总节点数。这时候不是你该输出 warning 还是 error 的问题——这是必须报错的问题。循环依赖意味着这两条技能在设计上就是错的。

## 从 2023 到 2026：这跟你今天的工作有什么关系？

你们可能会想："这是一个 2024 年的 Minecraft 论文，我一个 2026 年做 SaaS Agent 的，为什么要学这个？"

因为这个论文没有死在 Minecraft 里。它的模式成了行业标准。

Voyager 技能：`{name, description, code, tags, version, dependencies}`
Claude Agent SDK skill（2026）：`{name, description, code, instructions, version}`
skillkit（npm）：统一格式，横跨 32+ AI 编码 Agent，可移植

模式完全一致：持久化、可检索、可组合、可精炼、按需加载。

区别在哪？产品化。Voyager 的嵌入向量检索变成了 SDK 的内置 API。Voyager 的环境反馈循环变成了 SDK 的会话内 skill 热更新。Voyager 的课程模块变成了 SDK 的 skill discoverability ——"当前任务，哪些技能可能用得上？"不再是手动查询，而是 Agent 自动评估。

论文和产品的差距只有两点：工程细节和生态规模。模式是一样的。

## 什么时候不该用技能库

教架构的人如果不讲"什么时候别用"，就是在画饼。

技能库不适合这四种情况：

1. **一次性 Agent。** 如果你只跑一次就丢掉这个 Agent，建技能库是纯开销。
2. **无重复模式的领域。** 假如你的每个任务完全不一样——没有任何可复用的子步骤——技能库就是存储器浪费。
3. **技能比任务还大。** 如果你的"技能"等同于"完成整个任务的全流程"，那不是技能库——那是宏录制器。
4. **没有环境反馈的系统。** 如果你无法验证技能执行结果（比如纯推理/创意任务），迭代精炼就做不了。你不能在没有真值信号的系统里搞 Self-Refine。

## 结课：如果你的 Agent 需要技能库，这是你的设计清单

1. 技能载体选代码，不选自然语言。确定性比灵活性重要。
2. 检索引擎从嵌入向量起步，再用标签过滤兜底，长期准备多路召回。
3. 写入时必须做去重检查。相似描述不产生新技能条目。
4. 每个技能带版本号。父技能锁死依赖的特定版本。
5. 组合技能做拓扑排序（Kahn 算法），并检测循环依赖。
6. 精炼用失败驱动循环，三元反馈优于二元。
7. 课程模块不需要完整系统——每周一次 LLM 差距分析通常就够用。
8. 在 50 条技能的测试库上，retrieval@5 要 ≥ 80% 才算及格。

最后一句话——我希望你们明天还记得这句：

**技能库要解决的，不是 Agent 不够聪明——而是它把已经学会的东西反复遗忘。记住自己做过的事、检索已有能力、在失败后改进——这是 Agent 的终身学习，也是好的架构设计的起点。**
