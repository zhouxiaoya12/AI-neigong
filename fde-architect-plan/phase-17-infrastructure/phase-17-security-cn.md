# 安全 — 密钥、API 密钥轮换、审计日志、防护栏

> 通过集中式密钥库（HashiCorp Vault、AWS Secrets Manager、Azure Key Vault）消除密钥蔓延。绝不在配置文件、版本控制中的环境变量文件或电子表格中存储凭证。使用 IAM 角色替代静态密钥；CI/CD 使用 OIDC。AI 网关模式是 2026 年的解决方案：应用 → 网关 → 模型提供商，网关在运行时从密钥库拉取凭证。在密钥库中轮换，所有应用在几分钟内自动获取——无需重新部署，无需在 Slack 上问"谁有新密钥"。轮换策略 ≤90 天；每次提交时用 TruffleHog / GitGuardian / Gitleaks 扫描。零信任：MFA、SSO、RBAC/ABAC、短期令牌、设备姿态。PII 脱敏使用实体识别在转发前掩码 PHI/PII；一致性令牌化（Mesh 方法）将敏感值映射到稳定的占位符，使 LLM 保留代码/关系语义。网络出口：LLM 服务在专用 VPC/VNet 子网中，仅白名单 `api.openai.com`、`api.anthropic.com` 等；阻止所有其他出站流量。2026 年事件驱动因素：Vercel 供应链攻击——通过被攻陷的 CI/CD 凭证，在数千个客户部署中泄露了环境变量。

## 学习目标

- 列举四种密钥管理反模式（VCS 中的配置文件、硬编码环境变量、电子表格、静态密钥）并指出其替代方案。
- 解释 AI 网关从密钥库拉取凭证的模式是 2026 年的生产标准。
- 实现一个带一致性令牌化的 PII 脱敏器（相同值 → 相同占位符），使语义得以保留。
- 说出 2026 年 Vercel 供应链事件及其对 CI/CD 凭证安全的教训。

## 问题

一个实习生提交了包含 API 密钥的 `.env` 文件。他们很快删除了它。密钥已经在 git 历史中——GitGuardian 扫描发现了它，你的轮换流程是"在 Slack 上通知团队，更新 40 个配置文件，重新部署所有服务"。8 小时后，一半服务已上线，另一半还在等待部署窗口。

另外，用户提示包含"我的 SSN 是 123-45-6789"。提示被发送到 OpenAI。你有 BAA 协议但内部策略是在转发前掩码 PII。你没有这样做。

另外，你的 EKS 集群中的 LLM Pod 可以访问任何互联网主机。有人通过 DNS 查询将数据泄露到攻击者控制的域名。没有任何阻拦。

LLM 服务的安全必须同时应对这三个向量：密钥库支持的凭证管理、PII 脱敏、网络出口过滤、审计日志。

## 概念

### 集中式密钥库 + IAM 角色拉取

**密钥库**：HashiCorp Vault、AWS Secrets Manager、Azure Key Vault、GCP Secret Manager。单一事实来源。

**IAM 角色**：应用/网关通过其 IAM 身份认证，而非静态密钥。密钥库在令牌有效期内返回密钥。

**AI 网关模式**：网关在请求时从密钥库拉取 `OPENAI_API_KEY`。在密钥库中轮换；下一个请求自动获取新密钥。无需重新部署。

### 轮换策略 ≤ 90 天

所有 API 密钥、密钥库根令牌、CI/CD 凭证。尽可能自动化轮换。手动轮换需要记录和跟踪。

### 密钥扫描

- **TruffleHog** —— 正则 + 熵检测，针对提交。
- **GitGuardian** —— 商业产品，高准确率。
- **Gitleaks** —— 开源，在 CI 中运行。

每次提交都运行。检测到新密钥时阻止 PR。

### 零信任姿态

- 所有账户要求 MFA。
- 通过 SAML/OIDC 的 SSO。
- RBAC（基于角色）或 ABAC（基于属性）的细粒度访问控制。
- 短期令牌（小时级，非天级）。
- 设备姿态 —— 仅允许带磁盘加密的公司设备。

### PII / PHI 脱敏

在提示离开你的基础设施之前：

1. 实体识别（spaCy NER、Presidio、商业方案）。
2. 掩码匹配的实体：`"我的 SSN 是 123-45-6789"` → `"我的 SSN 是 [SSN_TOKEN_A3F]"`。
3. 一致性令牌化（Mesh 方法）：相同值映射到相同的占位符，使 LLM 保留关系语义。
4. 可选的 LLM 响应反向映射。

静态正则过滤器捕获基本模式；NER 捕获更多。两者都使用。

### 输入 + 输出防护栏

输入：阻止已知越狱、禁止主题；按用户速率限制。

输出：正则脱敏泄露的密钥（API 密钥模式、拒绝上下文中泄露的邮箱模式），对策略违规的分类器。

### 网络出口白名单

LLM 服务在专用子网中：
- 白名单：`api.openai.com`、`api.anthropic.com`、向量数据库端点、密钥库端点。
- 其他所有：丢弃。
- DNS 通过仅白名单的解析器（避免 DNS 隧道泄露）。

### 审计日志

每次 LLM 调用的不可变日志，包含：
- 时间戳。
- 用户 / 租户。
- 提示哈希（非原始提示，保护隐私）。
- 模型 + 版本。
- Token 计数。
- 成本。
- 响应哈希。
- 任何触发的防护栏。

按监管要求保留（SOC 2 1 年，HIPAA 6 年）。

### 2026 年 Vercel 事件

供应链攻击：被攻陷的 CI/CD 凭证在数千个客户部署中泄露了环境变量。教训：CI/CD 凭证等同于生产凭证。存储在密钥库中。范围最小化。积极轮换。

### 你应该记住的数字

- 轮换策略：≤ 90 天。
- 每次提交扫描：TruffleHog / GitGuardian / Gitleaks。
- Vercel 2026：CI/CD 凭证被攻陷 → 数千客户环境变量泄露。
- 审计日志保留：SOC 2 = 1 年，HIPAA = 6 年。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份紧贴 2026 年生产实践的 LLM 安全指南，以"真实事件驱动"的方式组织——2026 年 Vercel 供应链攻击作为开篇的血泪教训贯穿全文。文档同时覆盖了三个安全维度（密钥管理、PII 保护、网络安全），避免了常见安全课程"只讲一个维度"的片面性。最精彩的部分是"AI 网关从密钥库拉取凭证"的生产模式——它将密钥轮换从一个需要重新部署的"运维噩梦"变成了一个运行时自动生效的"静默操作"，学生在理解这个模式后会重新审视自己的安全架构设计。

### 二、知识结构梳理

- **认知基础层**：安全的核心问题不是"有没有安全措施"，而是"安全措施是否与实际工作流程兼容"。四种反模式（VCS 中的密钥、硬编码、电子表格、静态密钥）之所以普遍存在，不是因为工程师不知道它们不好，而是因为替代方案太繁琐——直到密钥库 + 网关模式出现。
- **工程模式层**：集中式密钥库 + IAM 角色拉取 + 网关模式构成了 2026 年的标准安全架构。PII 脱敏（实体识别 + 一致性令牌化）和网络出口白名单是另外两个关键工程模式。零信任姿态（MFA、短期令牌、设备姿态）定义了人员层面的安全基线。
- **落地实践层**：TruffleHog/GitGuardian/Gitleaks 的 CI 集成，≤90 天的轮换策略，审计日志的分层保留（热/温/冷），这些都是可以直接落地到生产环境的实践。练习从代码层面（脱敏器实现）到策略层面（保留层级设计）逐步深化。

### 三、核心洞察

1. **"API 密钥在 git 历史中"是一个不可逆的事件**——删除提交不能撤销泄露。一旦密钥出现在 git 历史中，轮换是唯一正确的响应。"清理历史"只是掩盖问题，不能修复已经发生的泄露。为什么重要：它迫使团队建立一个"泄露→轮换"而非"泄露→删除提交"的响应流程。
2. **一致性令牌化（Mesh 方法）是语义保留的关键**——如果相同的 SSN 每次被替换为不同的随机占位符，LLM 就无法理解"这是同一个人"。为什么重要：PII 保护不是简单地删除敏感信息，而是在保护隐私和保留语义之间找平衡。
3. **CI/CD 凭证是生产级凭证**——Vercel 2026 事件的核心教训是：工程师习惯把 CI/CD 视为"内部工具"，其凭证风险被严重低估。CI/CD 凭证可以触及所有部署环境，攻陷它就是攻陷一切。为什么重要：它要求密钥库策略覆盖 CI/CD，而非仅覆盖运行时。
4. **DNS 隧道是一个容易被忽略的泄露向量**——攻击者不需要直接访问外部 API；一个 DNS 查询就足以泄露数据（通过将编码数据嵌入子域名）。为什么重要：网络出口策略必须包括 DNS 层面的控制，仅 IP 白名单不够。
5. **审计日志的"不可变性"比"详细程度"更重要**——一个可以被修改的详细日志比一个不可变的简洁日志更危险，因为前者创造了虚假的安全感。为什么重要：设计审计日志时，防篡改是第一优先级。
6. **"反向令牌化"的复杂性不值得**——将 LLM 响应中的占位符替换回真实 PII 值，意味着你需要信任 LLM 不会在响应中"创造"新的 PII 或错误地关联敏感信息。为什么重要：这是一个看似合理的功能，但其安全风险可能超过其便利性。

### 四、教学建议

1. **用"实习生提交 .env"的真实故事作为开场**——几乎每个有经验的工程师都见过或经历过这种情况。用这个共鸣点引出密钥管理的系统性解决方案。
2. **做一次"git 历史密钥扫描"实战**——让学生在自己的仓库上运行 TruffleHog 或 Gitleaks，看看能发现什么。实际的发现比理论讲解更有说服力。
3. **模拟一个"Vercel 式供应链攻击"的场景**——设计一个角色扮演练习：你发现 CI/CD 凭证泄露了，接下来 24 小时你应该做什么？这是很好的危机响应训练。
4. **PII 脱敏的"边界测试"**——给学生一些精心设计的测试用例，如 "my friend whose SSN is the same as mine"、"the patient in room 302 has the same condition as SSN 123-45-6789"，测试脱敏器的语义保留能力。
5. **"如果我是攻击者"环节**——让学生从攻击者视角思考：如果我是攻击者，我会尝试哪些出口通道（直接 API 调用、DNS 隧道、ICMP 隧道、HTTP 头注入）？这培养安全直觉。
6. **审计日志的"法官测试"**——如果一份审计日志作为法庭证据，它能否经得起交叉质询？这条思路帮助学生理解为什么时间戳精度、不可变性和完整性校验很重要。
7. **制作"安全清单"卡片**——将核心实践（密钥库、扫描、轮换、脱敏、出口白名单、审计日志）制作成可打印的检查清单，供团队在实际项目中使用。

### 五、值得补充的内容

1. **密钥库的高可用性和灾难恢复**——如果密钥库宕机，所有依赖它的服务都会失败。这些内容探讨密钥库自身的运维挑战（多区域复制、缓存降级策略）。
2. **微分段（Micro-segmentation）与 LLM 服务**——除了子网级别的白名单，更细粒度的服务间网络策略（如 Kubernetes NetworkPolicy）可以提供额外防护。
3. **PII 脱敏的性能开销**——在生产流量下，NER 模型的推理延迟可能成为瓶颈，需要讨论批处理、缓存和模型轻量化的权衡。
4. **联邦学习/多方安全计算与 LLM 安全**——当多个组织需要共享模型但不共享数据时，如何在保持 PII 保护的同时协作。
5. **AI 安全合规的跨辖区差异**——GDPR、HIPAA、PCI DSS 对 LLM 服务的要求有何不同，特别是数据驻留和被遗忘权。

### 六、一句话总结

LLM 安全不是"加一道防火墙"，而是三个维度（密钥管理、PII 保护、网络安全）的系统性工程——密钥库 + 网关模式将密钥轮换从"部署事件"变成"运行时事件"，一致性令牌化在隐私和语义之间寻找平衡，而 Vercel 2026 事件提醒我们：CI/CD 凭证就是生产凭证。

# 🎓 Agent 架构课：LLM 安全不是加锁，是设计金库——从凌晨三点轮换密钥说起

同学们好。我是你们的FDE工程老师，今天讲的是 LLM 基础设施安全。

**凌晨三点十七分。GitGuardian 扫到了一个 API Key 在公开仓库 4 小时了。权限是 `*`——所有模型、所有账单、所有 fine-tune。** 这不是"会不会发生"的问题——是"什么时候发生"。

LLM 基础设施安全的核心原则：① **最小权限**——每个服务一个 Key，不共享。读模型的 Key 不能调 fine-tune。② **密钥轮换**——定期自动更换。别等出事再换。③ **API 网关作唯一入口**——所有请求经过网关审计，不直连提供商。④ **速率限制 + 预算告警**——异常用量自动切断。⑤ **日志审计**——谁在什么时候调了什么模型花了多少 token，全部可追溯。

跟 Phase 13 的 MCP 安全互补：Phase 13 安全管的是"MCP 服务器能不能信"，Phase 17 安全管的是"你的 API Key 有没有泄露、你的网络有没有被监听、你的账单有没有被盗刷"。

## 结课清单

1. **最小权限。** 每个服务独立 Key，权限只给需要的。
2. **网关作唯一入口。** 不直连提供商，全部经过审计层。
3. **密钥轮换自动化。** 别等到出事。
4. **基础设施安全 ≠ 模型安全。** 这是传输层、访问层、账单层的安全。

最后一句话：**你的模型可能很安全，但你的 API Key 在公开仓库里躺了 4 小时。LLM 安全不是训练时的事——是运维时的事。**

我盯着 Slack 里 47 条未读消息，手心出汗。不是第一次了，但这次不一样——这 Key 绑的是生产账号，月账单 6 万美金。如果攻击者拿到了，他不需要黑进我们的服务器，他可以直接用我们的账号调 GPT-4 生成几万条诈骗邮件，账单全算我们头上。

这就是安全的第一课：**密钥泄露不是"会不会"的问题，是"什么时候"的问题。** 我到今天都记得那个实习生——他没错，错的是我们把密钥放在 `.env` 文件里还以为"开发环境没关系"。关系大了。开发环境就是攻击者的跳板。

## 密钥管理的核心不是技术，是心态

很多团队的安全讨论让我想起我奶奶把存折藏在枕头底下——"只要没人发现就行"。工程师也一样：密钥放在 CI/CD 变量里、环境变量里、甚至 Excel 里，然后自我安慰"我们内部安全还行"。

2026 年 Vercel 那场供应链攻击打的就是这个脸。攻击者不是攻破了 Vercel 的服务器，是拿到了一套 CI/CD 凭证。然后呢？数千个客户的 `.env` 全部曝光，包括 API Keys、数据库密码、第三方服务的 Token。你猜这些客户里有多少人第二天才发现？答案是：有人一周后收到 AWS 账单才发现。

**做安全架构的第一原则：假设凭证已经泄露了。** 不是假设它能挡住，是假设它已经挡不住了，然后设计被泄露之后怎么办。

## AI 网关模式为什么是最优雅的解法

传统做法：轮换密钥 → 改 40 个配置文件 → 重启所有服务 → 祈祷没有漏掉一个。这他妈哪是轮换，这是俄罗斯轮盘赌。

AI 网关模式把问题翻转了：密钥只存在于密钥库（Vault / AWS Secrets Manager）里。网关在每次请求时去拉，缓存几秒钟。你在密钥库轮换，下一个请求自动拿到新 Key。**十二秒。** 不是十二小时。不是十二个部署窗口。十二秒。

我去年帮一个团队做迁移，他们原来轮换一次 OpenRouter Key 要两天——两天里新旧 Key 并存，攻击面加倍。迁移到网关模式后，我当着他们面在 Vault 里点了一下"rotate"。然后说："刷新一下你的 App。"他刷新了。一切正常。他沉默了三秒钟，然后说了一句我这辈子忘不了的话："你他妈早干嘛去了。"

不是我不想早说。是大多数团队觉得"网关模式"听起来像过度设计——"就一个 API Key 而已，至于吗？" 至于。Vercel 事件之后，至于的团队都活下来了。

## PII 脱敏：你以为你删干净了，模型替你想起来了

一致性令牌化（Mesh 方法）是我最喜欢的工程创新之一。逻辑很简单："张三的身份证号是 310xxx" → 每次出现都替换成 `[PID_X7K]`。模型读到两个 `[PID_X7K]`，知道"这是同一个人"。语义保留了，隐私脱敏了。

但我见过太多团队的理解停留在"用正则替换掉数字"。不行的。你把"310xxx199001011234"换成 `[REDACTED]`，模型就丢了"同一个身份证号出现了两次"这个信息。如果一个医疗记录里同一个 SSN 出现了三次，模型需要知道这是同一个病人。否则诊断就是猜谜。

**一致性令牌化的精髓是：你在保护隐私的同时保护了语义。** 大多数工程师做反了——他们保护了隐私但破坏了语义，然后抱怨"模型怎么变蠢了"。不是模型变蠢了，是你把它的上下文给切碎了。

## DNS 隧道泄露：我踩过的最贵的坑

这事说起来丢人。我们在 EKS 上部署了 vLLM 集群，网络策略只允许出站访问 `api.openai.com` 和 `api.anthropic.com`。我以为万无一失。

然后有一天审计日志莫名其妙爆了——有提示的 MD5 哈希在外部攻击者控制的 DNS 日志里出现了。怎么出去的？

**DNS 查询。** 攻击者在我们的集群里拿到了一个 Pod 的 shell（别问怎么拿到的，问就是 RBAC 配错了），然后他不需要直接连接外部服务器。他只需要把想偷的数据编码成子域名：`encoded-data.attacker-domain.com`。我们的 DNS 服务器转发了这个查询——因为 DNS 不在白名单里。

这个洞花了我们 4 天排查。修起来只要一行：DNS 转发只允许白名单解析器。但发现它的代价是 4 天没睡好觉。

## 反模式速查

1. **"开发环境用静态 Key 没关系"** —— 有关系。开发环境 = 生产环境的跳板。
2. **"我们用的是私有仓库所以安全"** —— Vercel 也是私有仓库。
3. **"审计日志存 MySQL，我们加了索引"** —— 审计日志的核心不是查询速度，是不可变性。能被 `UPDATE` 的日志不叫审计日志，叫笔记。
4. **"反向令牌化——把脱敏内容替换回真实值——看起来是个好功能"** —— 它是一个会爆炸的好功能。你信任模型不会在输出里"创造"新的 PII 吗？

## 收尾

安全工程的悲剧在于：你做对了，什么都不会发生。没人感谢你。你做错了一次，所有人都知道了。

所以记住这句话：**LLM 安全不是你加的那道防火墙，是防火墙被绕过之后你还能睡着的底气。** 密钥泄露不可怕，不能轮换才可怕。PII 穿越不可怕，没有审计才可怕。凌晨三点的告警不可怕，告警来了你不知道怎么办才可怕。

*——当你下次想往 `.env` 里塞一个生产 Key 的时候，想想 Vercel 那几千个工程师凌晨三点盯着监控面板的样子。*
