# 原生稀疏注意力（DeepSeek NSA）

> 在 64k 令牌时，注意力消耗解码延迟的 70-80%。每个开放模型实验室都有修复它的方案。DeepSeek 的 NSA（ACL 2025 最佳论文）是立住了的那个：三个并行的注意力分支——压缩的粗粒度令牌、选择性保留的细粒度令牌、以及用于局部上下文的滑动窗口——通过学习门控组合在一起。它是硬件对齐的（内核友好），原生可训练的（在预训练中工作，不是在推理时外挂），在 64k 解码上它比 FlashAttention 更快，同时匹配或超过全注意力的质量。本课从头构建三个分支，并展示为什么这种稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python（标准库）
**先修：** 第 7 阶段 · 12（KV 缓存、FlashAttention），第 7 阶段 · 15（注意力变体），第 10 阶段 · 16（差分注意力）
**时间：** 约 60 分钟

## 学习目标

- 说出 NSA 的三个注意力分支以及每个分支捕获什么。
- 解释为什么 NSA 是"原生可训练"的，而之前的稀疏注意力方法只能在推理时使用。
- 以压缩块大小和选择 top-k 为函数，计算 NSA 在 64k 上下文时相对于全注意力的注意力计算节省量。
- 在标准库 Python 中实现三分支组合，并在短合成序列上验证门控权重的行为。

## 问题

序列长度 N 上的全注意力需要 `O(N^2)` 时间和每层 `O(N)` 的 KV 缓存。在 64k 令牌时，计算量和内存带宽数字是灾难性的。NSA 论文中的理论估计测量值：在 64k 时，注意力占解码总延迟的 70-80%。一切下游指标——TTFT（首令牌时间）、令牌/秒、每百万令牌成本——都被注意力成本主导。

稀疏注意力是显而易见的答案。之前的尝试分为两类。固定模式稀疏性（滑动窗口、步进式、块局部）丢弃信息，在长程召回任务上失败。推理时稀疏性（KV 缓存剪枝、H2O、StreamingLLM）应用于在密集注意力上预训练的模型，只恢复了一小部分潜在加速，因为模型从未学习过通过稀疏模式路由信息。

原生稀疏注意力（Yuan 等人，DeepSeek + 北大 + 华盛顿大学，ACL 2025 最佳论文，arXiv:2502.11089）两者都做到了：一种模型在预训练中学习的稀疏模式，实现为内核对齐的算法，在推理时真正实现计算节省。两年后，NSA 或其直接后继将成为每个前沿长上下文模型上的默认注意力。

## 概念

### 三个并行分支

对于每个查询，NSA 运行三次注意力，分别针对 KV 缓存的三种不同视图：

1. **压缩分支。** 令牌被分组到大小为 `l`（通常 32 或 64）的块中。每个块通过一个小型学习 MLP 压缩为一个摘要令牌。查询在这些压缩令牌上进行注意力，获得整个序列的粗粒度视图。

2. **选择分支。** 使用压缩分支的注意力分数，识别与当前查询最相关的 top-k 个块。从这些块中读取细粒度（未压缩）令牌，查询对这些令牌全部进行注意力。可以把压缩分支注意力视为选择的路由信号。

3. **滑动窗口分支。** 查询对最近的 `W` 个令牌（通常 512）进行注意力，获取局部上下文。这个分支捕获其他两个可能遗漏的结构密集型短程模式（语法、局部共指）。

三个分支的输出通过学习的逐位置门控组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是来自查询上一个小型 MLP 的门控权重。它们不需要加起来等于 1——可以独立加权分支。

### 为什么这是"原生可训练"的

选择步骤（top-k 块）是离散的。离散操作会中断梯度流。之前的稀疏注意力工作要么跳过通过选择的反向传播（限制训练），要么使用不产生真正推理稀疏性的连续松弛。

NSA 绕过了这个问题：压缩分支注意力本身就是对整个序列的可微粗粒度注意力。top-k 操作只是重用压缩分支的最高注意力分数来选择加载哪些细粒度块。梯度通过压缩分支分数流动（这些分数影响压缩输出和选择逻辑），选择的块对最终输出的贡献也是可微的。不可微的 `top_k` 操作在前向计算图上是一个无操作——它只控制哪些块从内存中加载。

这就是 NSA 可以被端到端用于预训练的原因。模型学习通过三个分支联合路由信息，产生一个在推理时真正实现承诺加速的稀疏模式。

### 硬件对齐的内核

NSA 的内核是为现代 GPU 内存层级设计的。内核按 GQA 组加载查询（外循环），为每组获取相应的稀疏 KV 块（内循环），并在 SRAM 上运行注意力。因为每个查询组看到相同的选择块（选择是按查询组而不是按查询头的），KV 加载在组内摊销。算术强度保持高位。

论文报告 Triton 内核在 64k 解码上比 FlashAttention 快 9 倍，加速比随序列长度增长。前向和反向内核都已提供。

### 计算预算

设 `N` 为序列长度，`l` 为压缩块大小，`k` 为 top-k 选择数，`w` 为滑动窗口，`b` 为选择块大小（通常等于 `l`）。

- 压缩分支：每个查询 `O(N/l)` 个键，总计 `O(N * N / l)`。
- 选择分支：每个查询 `O(k * b)` 个键，总计 `O(N * k * b)`。
- 滑动分支：每个查询 `O(w)` 个键，总计 `O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

当 `N = 64k, l = 64, k = 16, b = 64, w = 512` 时：每个查询的成本为 `1000 + 1024 + 512 = 2536 个键`。全注意力是 `64000 个键`。计算量减少 25 倍。

当 `N = 128k, l = 64, k = 16, b = 64, w = 512` 时：每个查询的成本为 `2000 + 1024 + 512 = 3536 个键`。全注意力是 `128000 个键`。减少 36 倍。收益随序列长度增长——这正是设计的全部意义。

### 对比

| 方法 | 可微 | 真实推理加速 | 长程召回 |
|--------|---------------|----------------------|-------------------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 步进/块稀疏 | 是 | 是 | 部分 |
| KV 剪枝（H2O, StreamingLLM） | 不适用（推理时） | 是 | 部分 |
| MoBA（Moonshot） | 部分 | 是 | 良好 |
| NSA | 是（原生） | 是（64k 时 9x） | 匹配全注意力 |

MoBA（Moonshot，arXiv:2502.13189）是并行发表的工作，采用类似的"三比一好"方法，将 MoE 原理应用于注意力块。NSA 和 MoBA 是 2026 年长上下文预训练需要了解的两个架构。

## 构建它

`code/main.py` 在短合成序列上实现三个分支并展示：

- 压缩 MLP（为了教学清晰，使用简单的均值池化基线；真实 NSA 使用学习 MLP）。
- 由压缩分支分数驱动的 top-k 块选择。
- 对最后 `w` 个令牌的滑动窗口注意力。
- 门控组合。
- 与全注意力对比的计算量打印。

### 步骤 1：将令牌压缩到块中

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 步骤 2：压缩分支注意力

对压缩键运行查询的 softmax 注意力。压缩分支分数同时作为 top-k 选择的信号。

### 步骤 3：Top-k 块选择

选择 `k` 个最高压缩分支分数块的索引。加载这些块中的原始未压缩令牌，并对它们运行注意力。

### 步骤 4：滑动窗口注意力

取最后 `w` 个令牌，对其运行标准注意力。

### 步骤 5：门控 + 组合

查询上的小型 MLP 产生三个门控权重。最终输出是三个分支输出的加权和。

### 步骤 6：计算量统计

打印每个分支每个查询关注的键数以及总数。与 `N`（全注意力）对比。在使用 `l = 32, k = 4, w = 128` 的 1024 令牌合成序列上，NSA 看到每个查询 `32 + 128 + 128 = 288` 个键，而全注意力是 1024——少了 3.5 倍。

## 使用它

NSA 正在 DeepSeek 自家的长上下文预训练流水线中使用。截至 2026 年 4 月，在公共推理栈中的集成状态：

- **DeepSeek 内部**：原生，已发布的权重使用 NSA 或其继任者 DSA（DeepSeek 稀疏注意力）。
- **vLLM**：针对 DeepSeek-V3.x 权重的实验性 NSA 支持开发中。
- **SGLang**：NSA 基准测试已发布；生产路径跟随 vLLM。
- **llama.cpp / CPU**：不支持；内核分解的开销在 CPU 吞吐量下不值得。

何时使用 NSA：

- 针对 64k+ 上下文的预训练或继续训练，有认真的计算预算。
- DeepSeek 自有长上下文检查点的推理。权重是 NSA 原生的。

何时不使用：

- 服务现有的密集注意力预训练模型。没有继续训练就无法改装 NSA。
- 上下文在 16k 以下。三分支开销主导了节省。
- 单批次交互式聊天。延迟敏感的解码受益，但只在长上下文下。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇从"问题的量级"切入的技术教学文档，开篇就用 64k 上下文中注意力消耗 70-80% 解码延迟的硬数据立住了问题的严重性。文档最精彩的是"为什么 NSA 是原生可训练"的解释——把离散选择操作如何保留梯度流讲得异常清晰。三分支结构+门控组合的设计既有直觉（压缩→选择→局部），又有严格的数学推导（计算预算、O 复杂度），是理论与工程的完美平衡。

### 二、知识结构梳理

**基础层**：全注意力的 O(N^2) 瓶颈；KV 缓存的内存增长；为什么固定模式稀疏（滑动窗口/步进）失败于长程召回；为什么推理时剪枝（H2O/StreamingLLM）无法达到理论加速。

**模式层**：NSA 的三分支架构——压缩分支（O(N/l) 粗粒度视图）、选择分支（top-k 细粒度）、滑动窗口（局部结构）；压缩分数重用于路由的巧妙设计；硬件对齐内核的内存层级利用（GQA 组摊销 KV 加载）。

**应用层**：计算量缩减（64k: 25x, 128k: 36x）；与 FlashAttention 的 9x 加速对比；生产栈集成状态（DeepSeek 原生、vLLM/SGLang 实验性）；何时使用/不使用的决策框架；与 MoBA 的并行架构对比。

### 三、核心洞察

1. **"原生可训练"是关键差异点**：之前十年的稀疏注意力研究解决的是"推理时稀疏"问题——在已训练的密集模型上砍路径。NSA 解决的是"让模型学会稀疏"——在预训练中形成信息路由策略。这两个问题是完全不同的难度级别。

2. **压缩分数的双重用途是天才工程**：压缩分支的注意力分数既用于产生压缩输出，又用于选择细粒度块。不需要额外的路由网络，不需要独立的梯道路径。一个计算服务两个功能。这种"免费获得"的设计在深度学习中罕见而珍贵。

3. **计算量节省随序列长度增长**：64k → 25x 节省，128k → 36x 节省。NSA 的设计使得问题越大，收益越大。这是"面向未来"的设计——随着上下文长度继续增长，NSA 的价值曲线是上升的。

4. **选择是按查询组而非按查询头**：这是硬件对齐的深层含义。如果每个注意力头选择不同的块，KV 加载无法在组内共享。按组选择意味着 KV 缓存加载一次，组内所有头共享——算术强度由此来。

5. **三分支是"容错设计"**：压缩分支提供全局但低分辨率的视图；选择分支提供定位的高分辨率视图；滑动窗口提供局部结构。没有任何一个分支独自承担所有职责。如果一个分支失败（滑动窗口在超长程任务上无用），另外两个分担。

6. **离散操作不破坏梯度流的洞察**：`top_k` 在前向图中是选择操作——它决定从内存中读取什么，不修改任何计算。梯度通过被选择的块流入，通过压缩分数流入。`top_k` 本身不从梯度的角度存在。这让 NSA 可以端到端预训练。

7. **与 MoBA 的并行发展不是偶然的**：2025 年初，"多重注意力/块选择"作为解决方案同时出现在两个独立实验室，说明这是一个"时机成熟"的想法——当 Transformer 上下文增长到 64k 时，问题模式迫使每个人朝同一个方向看。

### 四、教学建议

1. **从一张"注意力图的热力可视化"开始**：展示一个 64k 序列的注意力图——学生应该看到巨大的 N×N 矩阵，其中绝大多数条目是噪声。NSA 的动机应该是一张图就能讲清楚的东西。

2. **三分支的"隐喻化"教学**：压缩分支 = 扫读目录（粗粒度），选择分支 = 精读相关章节（细粒度），滑动窗口 = 记住上一句（局部连贯）。这个隐喻让学生在听到数学推导之前就建立了直觉。

3. **计算量推导要手把手做**：不要只展示公式 N*(N/l + k*b + w)——让学生在黑板上逐步推导。64k/64=1000（压缩），16*64=1024（选择），512（滑动）→ 2536 vs 64000。25x 的数字让他们记住。

4. **不可微操作的梯度流分析要画图**：前向计算图——标注哪些边有梯度流，哪些没有。`top_k` 是图中的黑洞（无梯度通过），但被选择的块仍然通过注意力计算的梯边与压缩分数相连。这一图值万字。

5. **门控 MLP 的行为可视化**：让学生在不同查询上观察 g_cmp、g_sel、g_win 的变化。查询接近序列开头时，选择分支权重是否上升？查询在最后几句内时，滑动窗口权重是否主导？门控行为是检查"模型是否正确理解三个分支各自用途"的试金石。

6. **对比练习**：让学生用相同的合成序列运行全注意力、仅滑动窗口、仅滑动/压缩分支（无选择）、完整 NSA。测量每个变体的召回率。缺失任何一个分支的代价应该清晰可见。

7. **最终决策题**："你的公司要训练一个 128k 上下文的客服模型。你有 500 GPU 的预算。你会选择 NSA、MoBA 还是标准的 FlashAttention？" 让学生基于成本、延迟和精度做出架构决策。

### 五、值得补充的内容

1. **NSA 的预训练收敛曲线**：NSA 论文第 5 节应该有三分支训练的损失曲线。预训练中，模型花了多少步学会有效利用选择分支？压缩分支和选择分支的损失贡献在训练过程中如何演变？

2. **块大小的选择策略**：`l = 32` vs `l = 64` vs `l = 128` 不仅仅是计算量差异——块大小决定了压缩分支的"分辨率"。太小（l=16）导致压缩分支键数过多，太大（l=128）导致摘要过于粗糙，top-k 选择精度下降。这个权衡值得量化。

3. **NSA 在 DeepSeek-V3 中的实际集成**：DeepSeek-V3 使用 NSA 还是 DSA？它们之间的架构差异是什么？V3 的门控 MLP、压缩块大小、top-k 具体数字？

4. **NSA vs DIFF Attention 的成本/收益**：两者都改善长上下文注意力，但路径完全不同。NSA 通过"选择"来减少信息，DIFF 通过"减法"来减少噪声。在什么场景下一者优于另一者？能结合吗？

5. **多轮对话场景下的 KV 缓存增长对 NSA 的影响**：在多轮对话中，KV 缓存持续增长。NSA 的三个分支如何适应动态增长的缓存？压缩和选择分支是否需要重新计算？

### 六、一句话总结

NSA 的核心洞察是"高质量注意力不需要看到一切——只需要看到正确的东西"：通过让模型在预训练中学习"挑什么看"，它实现了比 FlashAttention 快 9 倍的长上下文解码，同时不牺牲召回率。

# 🎓 Agent 架构课：原生稀疏注意力——你不需要看完整个图书馆才能找到那一页

## "你的模型在 64k 令牌上每秒只能生成 5 个令牌。注意力占了 78% 的时间。你打算砍掉从哪些令牌读信息？"

这是我在 2025 年春天评估一个 64k 上下文客服部署时面对的实际问题。全注意力是杀手——不是因为模型不够聪明，是因为它花了太多时间去看不该看的东西。在 64,000 个令牌中，相关的可能只有一千个左右。另外 63,000 个在消耗 HBM 带宽、SRAM 空间、计算周期，不提供任何有用信号。

### 问题的本质

让我说清楚全注意力在长上下文下发生了什么的物理层面。你的注意力操作在计算 `softmax(QK^T/sqrt(d))`。每个查询必须取自己和每一个键的点积。N 个查询，N 个键 = N² 次点积。在 64k 时，那是 41 亿次点积——每个头，每层，每个解码步骤。

传统的方法是"砍掉一些键"——滑动窗口只看最后 512 个，步进式注意力跳着看，KV 缓存剪枝扔掉"不重要"的条目。这些方法是有效的，但它们有一个共同缺陷：模型从未在训练中学会通过这种稀疏模式路由信息。你在推理时砍掉东西，模型没有准备好在这些约束下工作。结果？召回率下降，幻觉增加，"中间丢失"效应恶化。

### 两条路径：推理时稀疏 vs 原生训练稀疏

业界在这个问题上分裂了。一条路径说："先用密集注意力训练模型，然后在推理时聪明地剪枝。" 这更简单，兼容已经训练好的权重。但剪枝是一个信息破坏性操作——而梯度从未流过剪枝决策。

另一条路径说："在预训练中就把稀疏性编码进去。" 这更难——需要让稀疏选择可微或至少允许梯度绕过选择。但一旦成功，你在推理时拿到的东西是模型已经适应了的稀疏模式，不是强加的。

我为什么在乎？因为我见过太多团队花了几个月优化推理时剪枝，然后在 128k 上下文的海捞针基准上看到了 40% 的召回率下降。你可以在推理时剪出一个漂亮的延迟数字，但不要误以为你保持了质量。

### 深入原理：为什么三分支设计

NSA 的三分支设计不是因为"三个比一个好"（虽然碰巧是的）——而是因为每一个分支做一件其他分支不擅长的事。

**压缩分支**给你全局视图，但分辨率低。一个 64 令牌的块被池化成一个向量——你看到的是"这一块大致在说什么"，不是具体的哪个词在哪里。这在你需要知道"相关信息在文档的前 1/3 还是后 1/3"时已经够用，但不足以精确回忆一个数字或名字。

**选择分支**给你高分辨率但窄范围。你只在压缩分支说"这里重要"的地方放大细看。选择分支成本取决于 k 和 b——k 控制了狭窄程度，b 控制了每次放大的粒度。

**滑动窗口**给你局部结构。任何语言模型都需要知道前几个词——句法、局部共指、最近的主题。滑动窗口用少量令牌覆盖了这些，避免了压缩分支的粗粒度和选择分支的选错可能性。

门控 MLP 学习的是"当前查询需要多少全局、多少局部、多少精确定位"。这不是一个开/关开关——它是一个连续的混合权重。

为什么压缩分数重用于选择？这个问题被问了很多。答案是双重的。第一，重用节省了计算——你不需要额外的路由网络。第二，更重要的是，重用保证了梯度流。压缩分数的梯度影响两个东西：压缩输出和选择逻辑。模型学习在同一个表示空间中同时优化"全局上下文理解"和"相关性判断"。相关性判断不需要一个独立的目标函数。

### 从 HBM 到 SRAM：内核的现实

NSA 在 GPU 上实现时的数字比理论数字更好。让我解释为什么。

GPU 内存是一个层级结构：HBM 很大（80GB）但慢（~2TB/s），SRAM 很小（~40MB）但极快（~19TB/s）。全注意力必须从 HBM 加载整个 N×d 的 KV 缓存到 SRAM，计算，再写回去。在 64k 时，KV 缓存大到无法一次装进 SRAM——你必须分块加载（FlashAttention 做的就是这件事），但块的数量还是 O(N)。

NSA 的稀疏性在这里大放异彩。对于每个查询，必须从 HBM 加载的 KV 字节减少 25-36 倍。更少的 HBM 加载 = 更低延迟。在 SRAM 中运行的计算仍然很快——但瓶颈变成了计算，不是加载。这就是硬件对齐的含义：稀疏模式不仅减少了数学上的计算量，还匹配了实际硬件的层级结构。

DeepSeek 报告的 9x 加速在 64k 时是有具体内核可复现的——不是理论估计，是在 H800 上测的 Triton 内核。更高的加速比（>9x）在 128k 时被观察到。

### 反模式：什么时候 NSA 不是答案

**反模式 1："我有一个 Llama 3 的 70B 模型，我想在推理时加 NSA。"** 你不能。稀疏模式是学到的，不是配置的。在密集注意力上预训练的模型没有压缩键的表示、没有门控 MLP、没有选择行为。你从零开始训练或继续训练；你不能事后改装。

**反模式 2："我的模型永远只在 8k 上下文中。NSA 应该更省计算吧？"** 在 8k 时，三分支的开销——压缩 MLP、门控、两套额外的注意力前向路径——几乎等于全注意力的成本。收益在 32k 以下很小，在 64k 以上巨大。

**反模式 3："我直接把 k 设成 1，把三个分支减少到基本上都是一个分支。"** 你不要同时放弃全局和局部视图。k=1 可能漏掉真正关键的信息。k=16 是论文默认，经过了海捞针和长文档问答的验证。调参空间大，但非常规值没有经过良好校准。

### 结语

NSA 是 "训练时设计架构"优于"推理时优化"的证明。花预训练计算量学习一个稀疏模式比花推理计算量暴力剪枝更有效——前提是你有预训练的控制权。

你现在知道了：三分支结构（压缩、选择、滑动）是必需的而非可选的。压缩分数重用于选择是因为梯度流，不是因为懒。硬件对齐 = 稀疏模式形状匹配 GPU 内存层级。门控学习混合比。9x 在 64k 时的加速不是理论数字——是可以复现的 Triton 内核指标。DeepSeek 已经把它推进生产了。

**金句：** "全注意力是读整本书找一句话。NSA 是先看目录（压缩），再看相关章节（选择），同时记得上一段是什么（滑动窗口）。三个动作都在同一瞬间完成，而且互不冲突。" 📊
