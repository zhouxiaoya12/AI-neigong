# 监管者/编排器-工作者模式

> 一个主导智能体规划和委派；专业工作者在并行上下文中执行并汇报回来。这是 Anthropic 研究系统背后的模式（Claude Opus 4 为主导，Sonnet 4 为子智能体），在内部研究评估中比单智能体 Opus 4 高出 +90.2%。Anthropic 的工程博文报告称 BrowseComp 上 80% 的方差仅由令牌用量解释——多智能体获胜的主要原因是每个子智能体获得新鲜的上下文窗口。本课从原语构建监管者模式，并涵盖 2026 年生产部署的工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置课程：** 第16阶段 · 04（原语模型）
**时间：** 约75分钟

## 问题

研究是单智能体系统失败的典型任务。你问"多智能体系统在 2023 年到 2026 年间发生了什么变化？"一个单智能体按顺序阅读五篇论文，上下文中塞满了论文文本，然后必须对所有内容进行推理。读到第五篇时，它已经忘记了第一篇。它无法并行化。

监管者模式解决了这个问题：一个主导智能体规划搜索，将每个子问题委派给工作者，然后综合。每个工作者获得自己的 200k 令牌窗口用于一个狭窄的问题。主导从不看到原始论文——只看到工作者的摘要。

Anthropic 的生产研究系统报告在内部研究评估上比单一 Opus 4 高出 +90.2%。同一篇博文指出，BrowseComp 上 80% 的方差仅由*令牌用量*解释。每个子智能体的新鲜上下文是主要机制。

## 概念

### 模式

```
                 ┌──────────────┐
                 │   主导       │  规划、分解、
                 │  (Opus 4)    │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工作者1 │  │ 工作者2 │  │ 工作者3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         新鲜          新鲜          新鲜
         上下文        上下文        上下文
```

主导从不阅读原始材料。工作者在主导综合之前看不到彼此的工作。每个箭头都是一个带有狭窄产物的转交。

### 为什么有效

三种机制：

1. **每个子智能体有新鲜上下文。** 探索"FIPA-ACL 遗产"的工作者不携带主导花了 40k 令牌进行规划的历史。它为一个问题获得 200k 窗口。
2. **通过提示实现专业化。** 主导的提示是"分解和综合"，而不是"研究"。每个工作者的提示是狭窄的："找出X发生了什么变化"。聚焦的提示产生聚焦的输出。
3. **并行性。** 工作者并发运行。墙钟时间大致为 `max(worker_times) + plan + synthesis`，而不是 `sum(worker_times)`。

### 工程经验（Anthropic 2025）

Anthropic 的博文列出了几条在 2026 年仍然相关的生产经验：

- **根据查询复杂度调整投入规模。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。主导必须估算这一点，而不是调用者。
- **先广后深。** 先分解为广泛的子问题，然后如果答案值得深入，再为每个子问题生成更多工作者。
- **Rainbow 部署。** 智能体是长时间运行和有状态的。传统的蓝绿部署不适用。Anthropic 使用 rainbow：新版本逐步推出，旧版本逐渐退出。
- **令牌用量占主导地位。** 多智能体约是单智能体的 15 倍令牌消耗。仅在任务价值证明成本合理时才运行。

### LangGraph 转向

LangGraph 最初提供了一个 `langgraph-supervisor` 库，带有一个高级 `create_supervisor` 助手。在 2025 年，LangChain 将推荐转向通过工具调用直接实现监管者模式，因为工具调用能更好地控制*监管者看到什么*（上下文工程）。该库仍然可用；文档现在推荐工具调用形式。

### 故障模式

- **主导虚构计划。** 如果主导生成了不分解实际问题的子问题，工作者会在错误的目标上进行精确研究。
- **工作者过度探索。** 没有明确的边界限制，工作者会漂移到分配的子问题之外，污染综合步骤。
- **综合冲突。** 两个工作者返回矛盾的事实。主导必须要么重新询问（增加一轮），要么明确记录分歧。默默地选择一方是最坏的失败：用户永远不知道有过分歧。

### 监管者不适用的情况

- **顺序任务。** 如果步骤 2 确实需要步骤 1 的输出，并行没有任何好处。使用流水线（CrewAI Sequential、LangGraph 线性图）。
- **简单查询。** 单智能体更快、更便宜地处理它们。在生成工作者之前使用主导的"规模投入"检查。
- **严格确定性。** 监管者使用大语言模型选择的委派。当审计/重放比适应性更重要时，静态图更好。

## 构建它

`code/main.py` 使用 `threading` 实现了一个包含三个并行工作者的监管者。主导将查询分解为子问题，工作者在每个子问题上并发运行，主导进行综合。没有真正的大语言模型——工作者被脚本化为模拟获取和摘要。

关键结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回一个假摘要（在生产中可以是任何使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工作者，join，然后综合。

运行：

```
python3 code/main.py
```

输出显示计划、带有开始/结束时间戳的并行工作者跟踪以及最终综合。你可以看到墙钟时间的优势：三个 0.3 秒的工作者在大约 0.35 秒内运行，而不是 0.9 秒。

## 使用它

`outputs/skill-supervisor-designer.md` 接受用户查询并生成监管者模式设计：主导系统提示、工作者角色、子问题分解规则和综合模板。在构建新的研究类智能体系统之前使用。

## Ship It

部署监管者模式前的检查清单：

- **模型配对。** 主导使用推理层模型（Opus 类、`o3` 类）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 任何运行时间超过 2 倍中位时长的工作者被终止；主导要么以更窄范围重新生成，要么在没有它的情况下继续。
- **每个工作者的令牌上限。** 硬性限制（例如 10 倍预期综合输入）防止失控工作者消耗预算。
- **可观测性。** 跟踪主导的计划、每个工作者的工具调用和综合。这是任何事后调试的基础。
- **Rainbow 推出。** 有状态的长时间运行智能体需要渐进式版本过渡，而非热替换。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这份教材以 Anthropic 的实际生产系统为锚点，将"监管者模式"从一个抽象架构模式具体化为有数据、有经验教训、有工程检查清单的可操作指南。+90.2% 的性能提升和"80% 方差来自令牌用量"这两个数据点堪称"说服力炸弹"，让学习者在第一页就理解了监管者模式的价值。文档将理论研究（Anthropic 博文）、框架演进（LangGraph 转向）和工程实践（Rainbow 部署、超时、令牌上限）编织成一条清晰的叙事线，没有任何多余的抽象。

### 二、知识结构梳理

**认知基础：** 研究是单智能体失败的典型任务——顺序阅读多文档导致上下文溢出和遗忘。监管者模式的三个胜利机制（新鲜上下文、专业化提示、并行执行）分属三个独立维度：空间（上下文隔离）、质量（聚焦提示）和时间（并发）。

**工程模式：** 主导（高级模型，负责规划和综合）+ 工作者（低成本模型，负责执行）的非对称模型配对策略。模式包含五个完整的故障模式（虚构计划、过度探索、综合冲突、不适用于顺序/简单/确定性场景），形成"什么情况下不适用"的反面知识体系。

**落地实践：** Rainbow 部署（有状态长运行智能体的版本过渡）是分布式系统工程的精细洞察；15 倍令牌成本的警告将"技术可行性"锚定在"经济可行性"上；LangGraph 从 DSL 向工具调用的转向展示了"上下文控制"优先于"框架便利"的工程哲学。

### 三、核心洞察

1. **"80% 的方差来自令牌用量"是整份文档最基本也最反直觉的数据。** 它意味着在 BrowseComp 上，多智能体的真正优势不是"更多模型"或"更好的模型"，而是"对每个子问题使用完整的 200k 令牌窗口"。为什么重要？它重塑了我们对多智能体架构的投资方向——瓶颈不是模型能力，是上下文管理。

2. **非对称模型配对是成本效益的精确平衡。** 主导用 Opus 4（昂贵、推理强），工作者用 Sonnet 4（便宜、执行快）。同样的任务如果全部用 Opus 4 做，成本远高于单智能体。为什么重要？它展示了多智能体不是"用更多好模型"，而是"模型选择是经济优化问题"。

3. **"主导从不看到原始论文"是有损架构的典范。** 每个工作者的上下文窗口被完整利用，但传递到主导的是有损摘要。这既是效率的来源（上下文净化），也是风险的来源（压缩丢失了信息）。为什么重要？它意味着监管者模式的质量上限不是工作者的能力，而是摘要质量——压缩算法的质量决定了最终答案的质量。

4. **Rainbow 部署揭示了 AI 智能体与无状态微服务的根本差异。** 传统蓝绿部署假设服务可以随时切换；但一个有状态的、正在进行研究的智能体不能简单地"杀掉重启"。为什么重要？它将智能体基础设施的问题域从"模型推理"扩展到"分布式状态管理"——这是很多 AI 工程师不熟悉的领域。

5. **综合冲突的"沉默选择"是最危险的故障模式。** 当两个工作者返回矛盾事实时，主导选择一方并继续——用户永远不知道有过分歧。比"错误的答案"更糟的只有"错误且不可审计的答案"。为什么重要？它强调了"暴露不确定性"比"掩盖分歧"更有价值的工程原则。

6. **LangGraph 从 DSL 到工具调用的转向是上下文工程优先于框架便利的宣言。** 工具调用形式让开发者控制监管者看到什么；DSL 形式更"方便"但黑盒化了信息流。为什么重要？在 AI 系统中，信息的流动路径（谁看到什么）比架构是否优雅重要得多。

### 四、教学建议

1. **先让学生自己经历单智能体的研究失败。** 给一个需要 5 篇论文才能回答的问题。让学生用单智能体试一遍，记录他们在哪个阶段丢失上下文。挫折感是最好的教学前置。

2. **画一张大图：监管者模式的完整信息流。** 标注每个智能体的窗口大小、令牌流向和摘要压缩率。让学生看到信息如何在系统中"蒸发"和"凝聚"的路径。

3. **运行 `code/main.py` 两次——一次用 3 个工作者，一次用 1 个。** 比较墙钟时间和输出质量。讨论：如果工作者数量增加到 10 个，主导的综合步骤会不会成为新瓶颈？

4. **模拟"综合冲突"场景。** 给两个工作者分配相反的输入（例如"写 Python 2 的 print 语法"和"写 Python 3 的 print 语法"），让学生看到主导如何处理矛盾信息。讨论：沉默选择 vs 暴露分歧的两种策略，什么时候该选哪一个。

5. **以 Anthropic 博文为阅读作业。** 让学生标注出生产经验中与课堂 demo 的差距（Rainbow 部署、令牌上限、可观测性——这些在 demo 中都不存在）。训练他们用"生产视角"审视实验代码。

6. **将 LangGraph 的 DSL 和工具调用两种实现并排比较。** 让学生指出工具调用形式中"控制信息流"的具体代码点在哪里。这种对比学习让抽象概念落地到具体代码。

### 五、值得补充的内容

1. **工作者超时的替代策略。** 除了"终止并继续"，还可以选择"终止并以更窄的范围重新生成工作者"、"回退到默认答案"、或"升级给人类"。不同场景应选择不同策略。

2. **综合步骤的提示工程细节。** 当主导收到多个工作者的摘要时，综合提示应该如何结构化？"按主题聚类摘要→指出冲突→给出最终答案"的三段式模板可能很有价值。

3. **工作者生成的动态调整。** "先广后深"原则的具体实现：什么信号触发"需要更深"——是摘要中包含未解决的不确定性，还是发现了新的子问题？

4. **成本-质量边界上的定量模型。** 15 倍令牌成本是一个重要的警告，但缺少更细粒度的分析：什么时候额外的令牌值得花（按任务类型分类）？能否根据主导对查询复杂度的判断，动态选择工作者数量？

5. **与 LangChain/LangGraph 之外框架的对比。** CrewAI 的 Hierarchical Process 和 AutoGen 的 GroupChat 也实现了类似模式——它们与监管者模式的关键差异是什么？

### 六、一句话总结

监管者模式的三个胜利机制——新鲜上下文、专业化提示和并行执行——分别解决了单智能体的空间瓶颈、质量瓶颈和时间瓶颈，但代价是一个本质上有损的信息压缩链和显著的成本倍增。

# 🎓 Agent 架构课：监管者模式——别让老板干活，老板只分活和验收

同学们好。我是你们的FDE工程老师，今天讲的是 监管者模式。

**Anthropic 的生产研究系统：Opus 4 当监管者，Sonnet 4 当工作者。比单 Opus 4 高出 +90.2%。BrowseComp 上 80% 的方差仅由 token 用量解释——每个子智能体拿到新鲜上下文窗口是主因。**

模式很简单：一个主导智能体规划搜索、分解问题、委派给工作者、综合结果。主导从不看到原始数据——只看到工作者的摘要。每个工作者有独立的 200k token 窗口用于一个狭窄问题。

为什么有效？因为人类组织也是这么工作的。你不会让 CEO 读 50 篇论文然后写报告——你让 5 个分析师各读 10 篇然后汇报摘要。监管者模式把组织层级映射到了智能体拓扑上。

## 结课清单

1. **监管者 ≠ 单智能体放大版。** 监管者只规划和综合，不执行。
2. **+90.2% 是实测数字。** Anthropic 内部研究评估——不是营销。
3. **新鲜上下文窗口是主因。** 比"大模型更聪明"更重要的因素是"小模型拿到更干净的输入"。

最后一句话：**别让老板干活。监管者模式的力量不在"最强的模型做所有事"——在"每个模型只做自己最擅长的事"。**

# 💼 从业者故事：监管者模式——别让老板干活，老板只分活和验收

今年三月我们上线了一个"智能研究助手"，用来帮分析师从 50 篇 ArXiv 论文里提取趋势。第一版用的是单智能体——一个 Opus 4，读了 50 篇论文，然后写报告。测试阶段看起来还行，因为测试只用了 3 篇论文。上线那天，分析师给了 50 篇，结果报告里的第 3 段和第 47 段自相矛盾："2024 年 MoE 是主流趋势"和"MoE 在 2024 年被证实不可扩展"出现在同一份报告里。分析师截图发到了全员 Slack 频道。那个表情包我现在还留着。

翻日志才发现——智能体读到第 43 篇论文的时候，第 5 篇论文的内容已经不在窗口里了。它根据后半段论文重新发明了一个"事实"，而这个"事实"恰好和前面的结论矛盾。没有人告诉它，它也不知道。这就是单智能体研究的经典死法：**不是模型胡说，是它手里的证据被自己挤出去了。**

## 监管者模式——不是"有个老板"，是"老板不干活"

很多新人以为监管者模式就是"有一个智能体指挥其他智能体"。那太浅了。监管者模式的核心是一个反直觉的设计原则：**主导智能体永远不碰原始数据。**

你想想。一个传统的研究流程是什么样的？你（研究员）会亲自打开论文、读、做笔记、写报告。你的老板只会在周会上问"进展如何"。监管者模式就是把这个分工搬到了 AI：

- **主导（老板）**：只看摘要。不看论文原文。决定"我们要研究什么"和"这些发现加在一起意味着什么"。
- **工作者（研究员）**：每个人只读几篇论文。读完给老板一个 500 字的简报。不需要知道其他研究员在读什么。

为什么这很重要？因为**主导的上下文窗口是整条链最珍贵的资源。** 它要综合所有工作者的发现。如果主导的窗口被论文原文塞满了，它就没空间综合了。所以最蠢的设计就是让主导去读原始文档——你那叫浪费管理带宽。

Anthropic 的数据是最硬的证据：BrowseComp 上 80% 的性能方差来自令牌用量，不是模型选择。翻译成人话：**你用 Opus 4 做单智能体，不如用 Opus 4 当主导 + 三个 Sonnet 4 当工作者——不是因为模型更强，是因为每个工作者的 200k 窗口都是新鲜的。**

## 三个机制拆开揉碎讲

**机制一：新鲜上下文。** 这听起来简单，但它是整个模式 80% 的价值。你可以做一个思想实验：你有两个选择。选择 A：一个 200k 窗口的 Opus 4 读 50 篇论文。选择 B：5 个 200k 窗口的 Sonnet 4，每个读 10 篇论文，然后一个 Opus 4 读 5 份摘要。哪个输出质量高？答案是 B。不是 Sonnet 比 Opus 强——Sonnet 比 Opus 弱多了——是因为每个 Sonnet 的 200k 都只装了 10 篇论文。Opus 那个 200k 装了 50 篇，挤成一团。上下文的质量 > 模型的质量。

**机制二：专业化提示。** 主导的 system prompt 是"你是研究总监。你的工作是分解问题、分配子任务、综合发现。你从不直接做研究。"工作者的 system prompt 是"你是某个具体领域的专家。你只回答分配给你的子问题。"一个 prompt 里同时塞两件事就是毒药——跟你在简历上写"精通前后端运维 AI"一样，HR 看到直接扔。

**机制三：并行执行。** 三个工作者同时跑，墙钟时间 = max(工作者时间) + 规划 + 综合。不是 sum(工作者时间)。50 篇论文，5 个工作者，每个 10 篇，并行跑完大概相当于 10 篇论文的串行时间。单智能体读 50 篇就是 50 篇的串行。省的不是算力，是用户等着出结果时刷手机的时间。

## 生产现实——三个你没想过的问题

**问题一：主导"虚构计划"。** 主导的 Task 是"把问题拆成子问题"。如果它拆出来的子问题跟原始问题对不上——比如你问"MoE 架构的演进"，它拆成了"MoE 在 NLP 中的应用""MoE 在 CV 中的应用""MoE 的训练技巧"——然后第三个工作者发现市面上关于"MoE 训练技巧"的论文很少，就自己编了一段。主导不知道工作者在编，因为它不读原始论文。最终报告看起来很专业，但第三部分是幻觉。你 debug 的时候会发现：整个链条的真相从第一步就歪了。

**问题二：工作者"过度探索"。** 你让工作者研究"2024 年 Transformer 的替代架构"。工作者读了两篇关于 Mamba 的论文觉得很有意思，顺藤摸瓜找到了 2023 年的 S4 论文，又找到了 2022 年的 HiPPO 理论，最后给你的摘要里有 60% 是 2022 年的内容。你没有问 2022 年。但工作者觉得很相关。它没有错——它只是热情过头了。这个问题在单智能体里也存在，但在多智能体里更隐蔽：因为主导看不到原始论文，它无法判断工作者的摘要是否跑题。

**问题三：综合冲突是最大的质量杀手。** 工作者 A 说"X 是趋势"，工作者 B 说"X 在衰退"。两个可能都对——A 看的是开源社区，B 看的是企业采购数据。但主导看到的只是两句矛盾的话。它能做什么？三个选项：（1）选一个看起来更靠谱的——这是最坏的，因为用户不知道有分歧；（2）暴露分歧，说"A 认为 X，B 认为 Y，可能因为数据源不同"——这是好的；（3）发起第二轮，让 A 和 B 验证对方的证据——这是贵的但最好的。大多数系统默认做（1），因为（2）和（3）需要额外设计。

## 反模式——这些坑我替你们踩过了

**反模式一：所有工作者用同一模型。** 我看到太多人搞 3 个 Opus 4 当工作者。兄弟，Opus 4 的定价是 Sonnet 4 的 3 倍。你的三个工作者并行跑，一次查询的 API 成本够买一杯奶茶。主导用 Opus（需要强推理），工作者用 Sonnet（需要快和执行）。非对称配对不是为了省钱——是为了把钱花在唯一需要深度推理的环节。

**反模式二：不给工作者设超时。** 你的工作者可能陷入无限循环——工具调用→读文档→工具调用→读更多文档→...→窗口满了还在调用。设置硬超时：超过中位执行时间 2x 就 kill。主导要么重新生成一个更窄范围的工作者，要么在没有这个结果的情况下继续。宁可部分报告缺失，不能让整个查询卡住。

**反模式三：没有令牌预算。** 每个工作者消耗的 token 量是未知的。你以为它就读 10 篇论文，结果它调用了 50 次搜索。按次计费的系统里，一个失控的工作者能吃掉你一天的 API 配额。给每个工作者设令牌上限——比如预期综合输入的 10 倍——超了就截断。这是成本控制，不是技术优化。

**反模式四：Rainbow 部署听不懂，跳过。** 如果你做的是服务化部署（不是一次性脚本），你会碰到一个恶心的场景：你升级了主导的 system prompt，旧的工作者还在跑，新的工作者用新 prompt，主导收到两种格式的摘要，综合逻辑崩了。对，智能体部署不是无状态的——它们有记忆、有进行中的任务、有版本不一致的通信合约。处理的方式是渐进式过渡（Rainbow），不是重启。

## 最后一句

监管者模式告诉我们的不是"AI 需要管理"——是**管理者的工作（规划、委派、综合）和执行者的工作（读、写、算）必须跑在不同的上下文窗口里**。别让老板干活，也别让干活的人做综合。各司其职，各有一个干净的脑子。
