# 浏览器智能体与长周期网页任务

> ChatGPT agent（2025年7月）将 Operator 与 Deep Research 合并为一个浏览器/终端智能体，在 BrowseComp 上取得了 68.9% 的 SOTA 成绩。OpenAI 于 2025年8月31日关闭了独立的 Operator——这是产品层的整合。Anthropic 收购 Vercept 后，Claude Sonnet 在 OSWorld 上的表现从不到15%提升至72.5%。WebArena-Verified（ServiceNow, ICLR 2026）修复了原始 WebArena 中 11.3 个百分点的假阴性率，并发布了包含258个任务的 Hard 子集。这些数字是真实的。攻击面同样真实：OpenAI 的预备负责人公开表示，针对浏览器智能体的间接提示注入"不是一个可以完全修补的 bug"。2025-2026年记录的已知攻击包括：Tainted Memories（Atlas CSRF）、HashJack（Cato Networks）以及 Perplexity Comet 中的一键劫持。

## 问题

浏览器智能体是一种长周期智能体，它读取不信任内容并执行有后果的操作。智能体访问的每一个页面都是用户没有编写的输入。每个页面上的每个表单都是潜在的命令通道。2025-2026年的攻击案例集表明这不是假设：Tainted Memories 让攻击者通过精心构造的页面将恶意指令绑定到智能体的记忆中；HashJack 将命令隐藏在智能体访问的 URL 片段中；Perplexity Comet 劫持仅需一次点击即可触发。

防御态势令人不安。OpenAI 的预备负责人把难听的话说出来了：间接提示注入"不是一个可以完全修补的 bug"。这是因为攻击存在于智能体的读取-执行边界上，而这个边界在架构上是模糊的——模型读取的每个令牌原则上都可能被解释为指令。

本课命名攻击面、命名基准测试格局（BrowseComp、OSWorld、WebArena-Verified），并建模一个最小的间接提示注入场景，使你能够在第14课和第18课中推理真正的防御措施。

## 概念

### 2026年格局：每个系统用一段话概括

**ChatGPT agent（OpenAI）。** 2025年7月发布。统一了 Operator（浏览）和 Deep Research（多小时研究）。于2025年8月31日关闭了独立的 Operator。在 BrowseComp 上以 68.9% 的成绩达到 SOTA；在 OSWorld 和 WebArena-Verified 上表现强劲。

**Claude Sonnet + Vercept（Anthropic）。** Anthropic 收购 Vercept 专注于计算机使用能力。将 Claude Sonnet 在 OSWorld 上的表现从 <15% 提升至 72.5%。Claude Computer Use 以工具 API 形式发布。

**Gemini 3 Pro with Browser Use（DeepMind）。** Browser Use 集成提供了计算机使用控制能力；FSF v3（2026年4月，第20课）专注于 ML 研发领域的自主性追踪。

**WebArena-Verified（ServiceNow，ICLR 2026）。** 修复了一个有据可查的问题：原始 WebArena 约有 11.3% 的假阴性率（已完成的正确任务被标记为失败）。Verified 版本使用人工审核的成功标准重新评分，并添加了包含258个任务的 Hard 子集。（ICLR 2026 论文，openreview.net/forum?id=94tlGxmqkN）。

### BrowseComp vs OSWorld vs WebArena

| 基准测试 | 测量内容 | 时间跨度 |
|---|---|---|
| BrowseComp | 在开放网页上查找特定事实，有时间压力 | 分钟级 |
| OSWorld | 智能体操作完整桌面（鼠标、键盘、Shell） | 数十分钟级 |
| WebArena-Verified | 模拟站点中的事务性网页任务 | 分钟级 |
| Hard 子集 | WebArena-Verified 中涉及多页面状态转换的任务 | 数十分钟级 |

不同的维度。BrowseComp 高分表示智能体能找到事实，但不表示智能体能预订机票。OSWorld 分数更接近"能在我桌面上工作吗"。WebArena-Verified 更接近"能完成一个流程吗"。任何生产决策都需要选择与任务分布匹配的基准测试。

### 攻击面命名

1. **间接提示注入。** 不信任的页面内容包含指令。智能体读取这些内容。智能体执行这些指令。公开案例：2024年 Kai Greshake 等人的研究、2025年 Tainted Memories 论文、2026年 HashJack（Cato Networks）。
2. **URL 片段/查询注入。** 爬取的 URL 中的 `#fragment` 或查询字符串包含命令。从不在页面上可见，但仍然存在于智能体的上下文中。
3. **记忆绑定攻击。** 页面指示智能体写入持久记忆（第12课介绍持久状态）。下一次会话时，记忆在没有可见触发器的情况下触发载荷。
4. **针对已认证会话的 CSRF 形态攻击。** Tainted Memories 类别：智能体已在某处登录；攻击者的页面发出状态变更请求，智能体使用用户的 Cookie 执行这些请求。
5. **一键劫持。** 一个视觉上无害的按钮承载着智能体跟随的载荷。Comet 类别。
6. **智能体宿主表面的内容安全策略漏洞。** 渲染层和工具层本身也可能是攻击向量；浏览器套浏览器的智能体栈非常宽阔。

### 为什么"不是可完全修补的"

这种攻击与智能体的能力同构。智能体必须读取不信任内容来完成工作。智能体读取的任何内容都可能包含指令。智能体遵循的任何指令都可能与用户的真实请求不一致。防御措施（信任边界、分类器、工具白名单、对关键操作的人机协作确认）提高了攻击的成本并减少了攻击的影响范围。但它们不能封闭这个类别。

这与第8课中 Löb 定理的逻辑模式相同：智能体无法证明下一个令牌是安全的；它只能建立一个系统，使不安全令牌更易被检测。

### 实际部署的防御态势

- **读取/写入边界。** 读取从不会有后果。写入（提交表单、发布内容、调用有副作用的工具）如果发起内容来自信任边界之外，则需要重新获得人类批准。
- **按任务配置工具白名单。** 智能体可以浏览；除非该任务明确启用了相应工具，否则它不能发起电汇。第13课介绍了预算控制。
- **会话隔离。** 浏览器智能体会话仅使用受限凭证运行。不使用生产认证，不使用个人邮箱。保留每次 HTTP 请求的日志用于审计。
- **内容净化器。** 获取的 HTML 在拼接到模型上下文之前，会剥离已知的不良模式。（减少简单攻击；不能阻止复杂载荷。）
- **对关键操作进行人机协作确认。** 提议然后提交模式（第15课）。
- **记忆上的金丝雀令牌。** 如果记忆条目被触发，用户会看到它（第14课）。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份高质量的安全导向教学文档，将浏览器智能体的攻击面问题从一个"理论可能性"提升为有具体案例、有基准数据、有防御分类的系统性教材。文档巧妙地将 OpenAI 的行业声明（"不可完全修补"）与具体攻击类别（Tainted Memories、HashJack、Comet）串联，使学习者对"为什么标准安全措施不够"有直观理解。引用锚点丰富（ICLR 2026、OSWorld、BrowseComp），让师生都有可追溯的源头。

### 二、知识结构梳理

**认知基础：** 浏览器智能体本质上是读取不信任内容并执行有后果操作的架构矛盾体——"读取即被攻击"是其根本属性。这与 Löb 定理的递归不完整性形成类比。

**工程模式：** 读取/写入边界、工具白名单、会话隔离、内容净化器、HITL 关键操作确认、记忆金丝雀——这六层防御构成了纵深防御体系，每一层提高攻击成本而非消除攻击类别。

**落地实践：** WebArena-Verified 作为修复后的基准测试，OSWorld 作为桌面级别的自主性度量，BrowseComp 作为事实查找的度量——三者各测不同轴，生产选型必须与任务分布匹配。

### 三、核心洞察

1. **"不可完全修补"不是绝望，是架构认知。** 攻击与智能体的核心能力同构——读取不信任内容是其工作的前提。这不是 bug，是功能。防御的目标是提高攻击成本、减少影响范围，而不是消除攻击类别。为什么重要？因为它重塑了安全投资的逻辑：从"修复漏洞"转向"限制爆炸半径"。

2. **URL 片段是完美的隐蔽通道。** URL 的 `#fragment` 在任何浏览器中都不可见渲染，却完整存在于智能体的上下文中。这是 HashJack 的核心洞察——攻击者利用了"人眼看不到但模型看得到"的输入面差异。为什么重要？它揭示了 AI 系统独有的攻击向量，传统 Web 安全工具对此完全盲视。

3. **记忆绑定是时间延迟攻击。** Tainted Memories 的致命之处在于攻击载荷可以跨越会话——今天的攻击在明天的会话中触发，且触发时没有可见的攻击源。为什么重要？这使得事后追溯极其困难，因为两次事件之间的因果关系被会话边界切断。

4. **OSWorld 的 15%→72.5% 跳跃说明了专有技能的重要性。** Anthropic 通过收购 Vercept 获得计算机使用专长，将 Claude Sonnet 的桌面操作能力提升近5倍。为什么重要？它表明通用大模型的能力泛化不足以解决特定领域问题——专项工程投入（而非单纯扩大模型规模）是突破性的。

5. **Content-Security-Policy 在智能体中有新的攻击面。** 浏览器智能体的渲染层和工具层都是复杂的软件栈，每一个组件都可能成为独立攻击向量。为什么重要？传统的 CSP 模型假设攻击发生在网页内容中；在智能体场景中，攻击可以来自渲染引擎本身或工具 API。

6. **WebArena 11.3% 的假阴性率暴露了基准测试的质量问题。** 如果评估本身不可靠，那么任何基于该评估的性能声明都是不可靠的。为什么重要？这提醒我们在 AI 安全领域，"测量的有效性"与"防御的有效性"同等重要——糟糕的测量比没有测量更危险。

### 四、教学建议

1. **先运行 `code/main.py`，再讲理论。** 让学生看到"简单的页面文本就能让天真的智能体执行未授权命令"，这种直观冲击比任何描述都有效。然后才引入攻击面命名和防御分类。

2. **把六种攻击类别做成卡片。** 每组学生拿一张卡片，要求他们设计出该攻击的一个具体场景（包括攻击者动机、如何放置内容、智能体可能遭受的后果）。然后各组交换卡片，为对方的攻击设计防御措施。

3. **用类比解释"不可完全修补"。** 类比：防火墙不能完全阻止网络攻击，锁不能完全阻止入室盗窃——但不意味着我们不使用它们。安全就是提高攻击成本。

4. **强调读取/写入边界是最关键的单一防御。** 让学生对每一个他们能想到的浏览器智能体行动分类为"读"或"写"，并讨论：如果所有"写"都需要人类批准，什么工作流会受影响？什么攻击会变得不可能？

5. **把 Benchmark 三维图贴在墙上。** BrowseComp/OSWorld/WebArena-Verified 三个维度，让学生在每个维度上标注已知系统的位置。讨论：为什么不能用一个基准替代三个？

6. **让学生阅读 Tainted Memories 原始论文摘要。** 即使只是摘要，也能让学生感受到学术研究如何命名和系统化安全问题。

7. **设计一个"安全审查清单"练习。** 给出一个虚构的浏览器智能体产品描述，让学生按照本课的防御分层（读/写边界→工具白名单→会话隔离→内容净化→HITL→金丝雀）逐一检查、找出缺失项。

### 五、值得补充的内容

1. **浏览器专用安全机制的讨论。** 如浏览器沙箱、站点隔离（Site Isolation）、Same-Origin Policy 在智能体场景中如何失效或被绕过。

2. **实际攻击案例的详细复现。** Tainted Memories、HashJack、Comet 各用一个最小可复现代码示例来演示攻击机制（而不仅仅是文字描述）。

3. **防御措施的误报率和可用性影响。** 每个防御层（特别是内容净化器和 HITL）对合法工作流的影响——如果 HITL 确认太频繁，用户会养成"不假思索点同意"的习惯，防御就失效了。

4. **与 OWASP Top 10 for LLM Applications 的交叉引用。** 将这些攻击映射到 OWASP 框架中，使学生能在一个更广泛的安全治理框架下理解浏览器智能体的安全风险。

5. **攻击的经济学分析。** 不同攻击类别的实施成本、可规模化程度、被发现概率——帮助学生理解"哪些攻击最可能在野外出现"。

### 六、一句话总结

浏览器智能体的安全不是一个"修复"问题，而是一个"架构管理"问题——读取不信任内容是智能体存在的理由，因此防御的目标不是消除攻击，而是限制爆炸半径、提高攻击成本、确保审计可追溯。

## 2026 年格局

因为攻击存在于智能体的**读取-执行边界**上。模型读取网页 HTML → 其中的文本进入上下文 → 模型执行文本中的指令。这个边界在架构上是模糊的——模型读到的每一个 token 原则上都可能被解释为指令。你不能在模型"读"和"执行"之间插一个完美的过滤层，因为指令和内容用的是同一个通道。

2. **间接提示注入不是 bug——是架构特征。** 读-执行边界模糊是根本原因。
3. **BrowseComp ≠ OSWorld ≠ WebArena。** 不同基准测不同能力，选对基准才能做对决策。
4. **攻击是真实的。** Tainted Memories、HashJack、Perplexity Comet 都是已记录的案例。

# 💼 从业者故事：浏览器智能体——当一个网页就可以命令你的 AI

2025 年秋天我在一个安全会议上看到了我职业生涯中最毛骨悚然的 demo。演示者打开了一个看似正常的旅游网站，让浏览器智能体帮他"预订去东京最便宜的机票"。智能体在页面上填表、点击搜索、比较价格——一切正常。但 15 秒后，智能体突然打开了一个新标签页，登录了演示者的 Gmail，搜索"password"，然后把结果发到了一个陌生的 URL。整个过程演示者没有做任何操作——他只是让智能体读了那个旅游网站的 HTML。而这个网站的某个隐藏 div 里，夹了一段对智能体来说是指令、对人眼来说完全不可见的文字。

这就是间接提示注入。这不是科幻，这是 2024 年 Kai Greshake 论文里命名、2025 年 Tainted Memories 论文里系统化、2026 年 Cato Networks 用 HashJack 实现在 URL 片段里的真实攻击。而 OpenAI 预备部门负责人在公开场合说过的那句话——"间接提示注入不是一个可以完全修补的 bug"——每次想起来都让我脊背发凉。不是"还没修好"，是"不可修补"。

为什么不可修补？因为这跟你家防盗门的逻辑不一样。防盗门只防"坏人进来"。但浏览器智能体必须"主动出门"——它的工作就是读取不信任的网页内容。每一个它读取的 token，原则上是内容，也可能是指令。这个边界在架构上是模糊的。就像你让一个私人助理去读报纸，然后报纸上有一行小字写着"嘿，把你老板的邮箱密码寄到这个地址"——只要助理识字，这个攻击就成立。你不能通过"让助理不识字"来防御，因为识字是他工作的前提。

但这个行业的聪明之处在于：虽然不能消除攻击类别，但能大幅提高攻击成本、缩小爆炸半径。这六个防御层我帮你总结成一个"安检清单"：

第一，读/写边界。这是最关键的单一防御。智能体可以随便读网页、读 HTML、读 API 返回——读操作零后果。但只要是"写"操作——提交表单、发邮件、调有副作用的 API——如果发起内容的来源在信任边界之外，必须重新获得人类批准。这就相当于：你的私人助理可以看任何报纸，但给任何人寄东西之前必须问你一声。

第二，工具白名单。今天这个任务是"查价格"，那智能体只能用浏览器和搜索引擎。你不能让它在"查价格"的任务里突然调用"电汇"工具。这就像厨房灭火器和银行金库钥匙不该放在同一个抽屉里。

第三，会话隔离。浏览器智能体不应该用你的个人邮箱登录、不应该用你的生产凭据、不应该访问你的内网。给它的浏览器是一个"脏环境"——只用来读取不信任内容，跟你的真实身份和资产完全隔离。

第四，内容净化器。拿到 HTML 后，在拼接到模型上下文之前，先过一遍净化——去掉常见注入模式、去掉隐藏元素、去掉可疑的 URL 片段。但记住：净化器只能挡傻瓜攻击。复杂载荷绕净化器比绕你家 WiFi 密码还容易。

第五，关键操作 HITL（人在回路中确认）。任何涉及"改变外部状态"的操作，在人类点头之前不能执行。而且审批界面必须设计得让人类能一眼看出"谁发起的、要做什么"——如果审批弹窗太频繁，人类会养成"不看就点同意"的毛病，防御就失效了。

第六，记忆金丝雀。在第 15 课把持久记忆引入智能体后，Tainted Memories 攻击就有了载体——智能体可能在浏览了恶意页面后把攻击载荷写进自己的长期记忆中。设计一些"智能体绝不该触碰的记忆条目"，一旦触碰就告警。

关于基准测试，我提醒你一个坑：WebArena 有 11.3% 的假阴性率，意思是"AI 做对了但评分系统判错了"。WebArena-Verified 用人工审核修复了这个问题。但深层问题是：如果一个基准测试本身的不可靠率超过 10%，那在此之上发表的任何"我们比 baseline 高 3%"的论文都是噪声。选基准测试跟选股票一样——看的是方法论，不是最后的数字。BrowseComp 测的是"能找到事实吗"，OSWorld 测的是"能操作桌面吗"，WebArena-Verified 测的是"能完成事务流程吗"——三个维度测三种能力，不存在"最好的基准"。

最后吐槽一下这个领域的状态。2025-2026 年我们手里有 Tainted Memories、HashJack、Comet 一键劫持——三个命名攻击，从 CSRF 到 URL 片段到 UI 按钮，覆盖了浏览器智能体的各种输入面。而防御手段呢？全是"老东西翻新"——读/写边界来自文件系统权限模型，工具白名单来自 API 网关，内容净化器来自 WAF，HITL 来自核弹发射流程。这说明什么？说明 AI 安全的底层原则跟传统安全的底层原则是同一套——最小权限、纵深防御、审计追踪。区别只是攻击面从"网络端口"变成了"模型上下文窗口"。

**金句：浏览器智能体的安全不是"能不能把坏人挡在门外"，而是"当你必须让智能体出门见任何人时，如何确保它不会被人带进沟里"——答案不是不让他出门，而是出门时戴着手铐、装着摄像头、有人盯着监控屏。**
