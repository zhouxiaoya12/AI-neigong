# 动作预算、迭代上限与成本调控器

> 一个中型电商 Agent 的月度 LLM 成本从 $1,200 跳到了 $4,800——在团队启用了"订单追踪"技能之后。这不是定价 bug。这是一个 Agent 发现了一个新循环并在里面持续花钱。微软的 Agent Governance Toolkit（2026 年 4 月 2 日）将针对这类问题的防御固化为代码：每次请求的 `max_tokens`、每任务的令牌和美元预算、每日/每月上限、迭代上限、分层模型路由、提示缓存、上下文窗口化、昂贵操作的人类在环检查点、预算突破时的断路器。Anthropic 的 Claude Code Agent SDK 以不同的名称提供了同样的原语。金融速度限制——例如在 10 分钟内消费超过 $50 就切断访问——比月度上限更快地捕获循环。

**类型：** 学习
**语言：** Python（标准库，分层成本调控器模拟器）
**前置课程：** Phase 15 · 10（权限模式）、Phase 15 · 12（持久化执行）
**用时：** 约 60 分钟

## 问题

自主 Agent 在每一轮都花真钱。聊天机器人的坏输出是一个坏回复；Agent 的坏循环是一张账单。行业记录的失败模式术语是"钱包拒绝服务"（Denial of Wallet）——Agent 持续推理、持续调用工具、持续计费，没有任何东西阻止它，因为没有任何东西被设计来阻止它。

修复方案不是一个数字。它是一组不同时间尺度和粒度的限制堆栈：每次请求、每任务、每小时、每天、每月。一个设计良好的堆栈在几分钟内捕获失控循环，几小时内捕获缓慢泄漏，一天内捕获一个坏发布。同一个堆栈在 Agent 是长程和自主时保持预算可控。

这是一节工程课：数学是简单的，纪律是团队失败的地方。以下限制列表全部在微软 Agent Governance Toolkit 或 Anthropic Claude Code Agent SDK 文档中被命名。

## 概念

### 成本调控器堆栈

1. **每次请求 `max_tokens`。** 简单。防止任何单次调用发出无界补全。
2. **每任务令牌预算。** 整个运行中不超过 N 个令牌。达到上限时硬停止。
3. **每任务美元预算。** 和令牌一样但是以货币为单位。Claude Code 中的 `max_budget_usd`。
4. **每工具调用上限。** 不超过 N 次 `WebFetch` 调用、N 次 `shell_exec` 调用等。
5. **迭代上限（`max_turns`）。** Agent 循环总迭代次数；防止无限推理循环。
6. **每分钟 / 每小时 / 每天 / 每月上限。** 滚动窗口。在不同时间尺度上捕获泄漏。
7. **金融速度限制。** 例如，"如果消费在 10 分钟内超过 $50，切断访问。"在月度上限触发前就捕获基于循环的燃烧。
8. **分层模型路由。** 默认使用较小的模型；仅在分类器判断任务值得时才升级到更大的模型。
9. **提示缓存。** 系统提示和稳定上下文存储在供应商缓存中；重新发送的令牌成本接近零。
10. **上下文窗口化。** 压缩/摘要化使活跃上下文保持在阈值以下；直接降低令牌成本。
11. **昂贵操作的人类在环检查点。** 在执行已知昂贵的操作（长时间工具调用、大下载、昂贵的模型升级）之前，需要人类确认。
12. **预算突破时的断路器。** 任何上限触发时会话中止。上限被记录；需要单独的重新启用路径。

### 为什么是堆栈，而不是一个上限

一个月度上限仅在钱包烧光后捕获失控 Agent。一个每次请求上限在会话层面什么都捕获不到。不同的失败模式需要不同的时间尺度：

- **失控循环**（Agent 卡在 5 秒重试中）：由速度限制捕获。
- **缓慢泄漏**（Agent 每任务做约 2 倍预期工作）：由每日上限捕获。
- **坏发布**（新版本使用 5 倍令牌）：由每周/每月上限捕获。
- **正当激增**（真实需求，非 bug）：由小时/天上限加清晰日志捕获。

### Claude Code 的预算接口

Claude Code Agent SDK 暴露（公开文档）：

- `max_turns`——迭代上限。
- `max_budget_usd`——美元上限；违反时会话中止。
- `allowed_tools` / `disallowed_tools`——工具允许列表和拒绝列表。
- 在工具使用前的钩子点用于自定义成本核算。

与权限模式阶梯（第 10 课）结合。一个没有 `max_budget_usd` 的 `autoMode` 会话是无治理的自主。Anthropic 明确将 Auto Mode 框架为需要预算控制；分类器与成本是正交的。

### 欧盟 AI 法案、OWASP Agentic Top 10

微软的 Agent Governance Toolkit 涵盖了 OWASP Agentic Top 10 和欧盟 AI 法案第 14 条（人类监督）的要求。对于在欧盟的生产部署，日志记录和上限执行不是可选的。

### 观察到的 $1,200→$4,800 案例

微软文档中的真实案例：一个电商 Agent 在添加了新工具后月度成本翻了三倍。该工具允许 Agent 在每次会话中轮询订单状态。无循环检测。无每工具上限。无周环比增长告警。修复方案是每工具上限加每日增长告警。这是一个模板：每个新工具面都是一个新潜在循环；每个新工具都需要自己的上限和自己的告警。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 16 高级 Agent 工程中最"接地气"的一课——不讲架构模式，讲的是"你的 Agent 这月花了多少钱，会不会下月翻三倍"。文档以微软 Agent Governance Toolkit 和 Claude Code Agent SDK 为蓝本，将成本控制从"项目管理的担忧"变成"12 层堆栈的工程规范"。$1,200→$4,800 的真实案例是这节课的灵魂——不只因为数字吓人，因为故障原因是"启用了订单追踪技能后 Agent 开始轮询"——完全没有恶意，只是 Agent 发现了一个能做的事就开始做了。适合即将部署自主 Agent 的生产团队。

### 二、知识结构梳理

**成本调控器堆栈层：** 12 层限制——从细粒度（每请求 max_tokens）到粗粒度（月度上限），从令牌到美元，从频率（迭代上限）到速度（金融速度限制），从预防（分层路由、提示缓存、上下文窗口化）到止损（断路器、人类在环检查点）。每层覆盖不同的时间尺度和失败模式。

**失败模式-时间尺度映射层：** 失控循环→速度限制（分钟级）。缓慢泄漏→每日上限（小时级）。坏发布→每周/每月上限（天级）。正当激增→小时/天上限+日志（不是 bug，但仍需可见）。

**真实案例层：** $1,200→$4,800 的电商 Agent——新工具=新循环=新账单。修复=每工具上限+增长告警。模板化：每个新工具需要自己的上限和告警。

### 三、核心洞察（备课时的关键理解）

1. **"单一月度上限 = 钱包烧光后才发现的告警"是整个成本调控的起点认知。** 一个 Agent 从 $1,200 跳到 $4,800 花了可能两周。月度上限在 $4,800 才触发——你已经在第 28 天才知道问题。速度限制在 10 分钟内就能截断。

2. **不同时间尺度的上限覆盖不同失败模式——这是堆栈存在的理由。** 不是"多设几层更安全"——是"不同类型的成本泄漏有不同的扩散速度。"失控循环 = 分钟级爆发。缓慢泄漏 = 天级蔓延。坏发布 = 周级累加。如果只有一层——你只覆盖了一种。

3. **金融速度限制是成本调控器堆栈中最被低估的一层。** "$50/10 分钟"——这个数字不大——但它在以下场景中作用最大：Agent 陷入重试循环，每 5 秒调一次付费 API。月度上限：$1,000→需要 200 次 10 分钟窗口耗尽。速度限制：第 10 分钟就截断。速度限制是"循环检测"的财务等价物。

4. **"每个新工具 = 一个新的潜在循环"是 $1,200→$4,800 案例教的最重要的一课。** 团队没做错任何事——他们加了一个订单追踪功能。Agent 开始用这个功能在每个会话中轮询。没有任何硬上限阻止它。模板：加新工具→同时加该工具的每工具调用上限→同时加该工具的成本增长告警。

5. **分层模型路由不是省钱技巧——是"默认安全的架构决策"。** 大多数查询不需要 Opus 级别的推理。用 Sonnet 或 Haiku 做默认——只有分类器判断任务需要时才升级。这个决策不是省钱——是把"默认成本"保持在低位。

6. **Claude Code 的 `autoMode` 无 `max_budget_usd` = 无治理自主——这不是 bug，这是设计中被明确标注的风险。** Anthropic 在文档中明确说 Auto Mode 需要预算控制。不是"建议"——是"没有预算控制的自主模式是未完成品"。

7. **欧盟 AI 法案第 14 条让日志记录和上限执行从"好实践"变成"法律要求"。** 在欧盟部署 Agent——你需要人类监督机制。成本调控器堆栈中的断路器日志、人类在环检查点、上限触发记录——这些不是工程优化，是合规证据。

### 四、教学建议

1. **用 $1,200→$4,800 的故事开场。** 这是真实的。一个电商 Agent，添加了一个功能，月度成本翻三倍。故障原因不是 bug——是 Agent 发现能做一件事就开始做了。让学生感受"没有成本上限的 Agent = 给它一张无限额信用卡。"

2. **12 层堆栈不需要逐条背——需要按时间尺度分组。** 请求级（max_tokens、迭代上限）→ 任务级（令牌/美元预算、每工具上限）→ 窗口级（分钟/小时上限、速度限制）→ 累积级（日/周/月上限、断路器）。四组。四层防线。

3. **速度限制用一个模拟 demo 来讲。** 让 Agent 陷入轮询循环，每 5 秒调一次 API。速度限制 10 分钟内截断 vs 月度上限两天后才触发。看到时间差异 = 理解堆栈价值。

4. **分层模型路由的"分类器判断"值得一个案例。** "用户问'今天天气怎么样'"——分类器判断：用 Haiku。"用户问'帮我分析这个 100 页的合同中的风险条款'"——分类器判断：用 Opus。同样的系统，不同的模型，成本差 50 倍。

5. **每工具上限用"订单追踪工具"的案例来讲。** 工具上线前→无上限。Agent 每个会话轮询 3 次。工具上线后→上限 5 次/会话。"为什么是 5？"——因为正常使用最多 3 次，留 2 次余量。这是上限设计的思维过程——不是随便设一个数。

6. **练习 3（映射上限类型到失败模式）是理解"为什么是堆栈"的关键练习。** 让学生逐条配对——速度限制↔失控循环、每日上限↔缓慢泄漏、月度上限↔坏发布——配对完了，堆栈的理由自动清晰。

7. **练习 4（为 50 个 issue 分类定价并设预算）是最诚实的。** 让学生为一个真实任务估计成本，然后设定 `max_budget_usd = 2× 估计`。问"为什么是 2 倍？"讨论欠估计的风险。"为什么不是 10 倍？"讨论上限太宽松失去意义。

### 五、值得补充的内容

1. **告警策略——上限触发时通知谁？** 速度限制触发→运维 Slack 频道。月度上限触发→项目经理和工程经理。每工具上限触发→该工具的维护团队。分级告警匹配分级上限。

2. **上限的调优周期。** 上限不是设一次就不变的。新 Agent 上线第一周——上限 2× 估计值。第二周——用实际数据调整到 1.5×。稳定后——锁死。调优周期是上限策略的一部分。

3. **不同供应商的令牌成本差异对上限设置的影响。** Opus token 是 Sonnet 的 5 倍。如果你用分层路由——美元上限必须基于加权模型成本而非统一令牌计数。

4. **提示缓存的省钱效果量化。** 系统提示 10K tokens，每分钟调一次——不用缓存 = 每次 $0.15，用缓存 = 每次 $0.015。日省 90%。这个数字值得算给学生看。

5. **与 Phase 16·01（长程 Agent）和 Phase 16·03（持久化执行）的串联。** 长程 Agent 的跨度翻倍趋势 → 成本翻倍趋势。持久化执行的重放 → 重放不需要重新付费（活动结果从日志取）→ 检查点策略本身也是成本控制策略。

### 六、一句话总结

**Agent 的成本控制不是设一个数字——是建一个堆栈。失控循环用速度限制在 10 分钟内截断、缓慢泄漏用每日上限在 24 小时内截断、坏发布用每周上限在一周内截断、新工具用每工具上限在上线前就截断。每层上限对应一种失败模式的扩散速度，缺失任何一层就是缺失一种检测。你的 Agent 不是不需要花钱——是你需要知道它在花多少、花在哪里、什么时候该停。**

# 🎓 Agent 架构课：成本调控——你的 Agent 这个月花了多少钱？

同学们好。我是你们的 Agent 架构老师。

前面三节课我讲了长程 Agent 的趋势、监管者模式、持久化执行。这三节课有一个共同的前提——**Agent 在花钱。** 每一个 LLM 调用、每一次工具执行、每一个额外的推理回合——都是真金白银。

今天这节课不讲架构。讲一个更根本的问题：**你怎么确保你的 Agent 不会下个月花掉你三倍的预算——因为你加了一个无害的新功能？**

这是真实发生过的事。微软的文档记录了一个案例：一个中型电商 Agent，月度 LLM 成本从 $1,200 跳到了 $4,800。不是因为模型涨价。不是因为用户暴增。是因为团队加了一个"订单追踪"技能。

Agent 学会了这个新技能后，开始在每次会话中轮询订单状态。没有循环检测。没有每工具调用上限。没有周环比增长告警。Agent 只是发现了一个能做的事，然后开始做了。**这就是"钱包拒绝服务"——你的 Agent 没恶意，只是在花你的钱。**

## 一个上限不够

如果你只有一个月度预算上限——$5,000/月。Agent 从 $1,200 跳到 $4,800 花了可能两周。你在第 14 天收到告警——"Agent 本月已花费 $4,800。"你去看——是那个订单追踪技能在狂轮询。你加了个上限。但 Agent 已经多花了 $3,600。

如果你有一个速度限制——"10 分钟内消费超过 $50，切断访问。"Agent 开始轮询，第 10 分钟消费到了 $50。上限触发。Agent 被切断。运维收到告警。第 10 分钟——不是第 14 天。Agent 只多花了 $38。这就是堆栈思维：不同时间尺度的上限捕获不同扩散速度的失败。

## 12 层——按时间尺度分组

成本调控器堆栈不是 12 个独立配置项。是四组，对应四种检测窗口。

**第一组：请求级。** `max_tokens`（单次调用上限）和迭代上限（`max_turns`）。这是你的第一道防线——防止一个单次 LLM 调用无限生成，或一个 Agent 无限推理。毫秒到秒级检测。

**第二组：任务级。** 每任务的令牌预算和美元预算（`max_budget_usd`）。每工具调用上限——`WebFetch` 最多 5 次、`shell_exec` 最多 3 次。一个完整运行在达到硬上限时中止。分钟级检测。

**第三组：窗口级。** 速度限制——"10 分钟 $50"——和每分钟/每小时上限。这是你的循环探测器。Agent 陷入重试循环每 5 秒调一次付费 API——速度限制在第 10 分钟截断。每小时上限在更慢的泄漏上给第 60 分钟的安全网。

**第四组：累积级。** 每日、每周、每月上限。这些不是循环检测器——是异常检测器。新版本上线，Agent 的每次运行消耗了 5 倍令牌。不是 bug——新版本就是更贵。但每日上限在第一天就让你看到"昨天我们花了平时的 5 倍"——你回滚了版本，而不是等到月末账单。

## 不止上限——省钱的设计

堆栈里有四层不是"触发就停"——是"从一开始就不花不必要的钱。"

**分层模型路由。** 不是每个问题都需要 Opus 4。用户问"今天天气怎么样"——用 Haiku。用户说"帮我分析这 100 页合同的风险条款"——分类器说"这个需要 Opus。"默认用小模型，升级只在必要的时候。成本节省不是 10%——是 90%。Opus 是一次 Haiku 调用的 50 倍成本。

**提示缓存。** 你的系统提示 10K tokens，每分钟被调用一次。不用缓存 = 每次重新发送 10K tokens = 每次 $0.15。用缓存 = 供应商端存着这份提示 = 每次 $0.015。一天省 $194。

**上下文窗口化。** Agent 的对话历史在增长——8K、16K、32K。每次 LLM 调用都在处理更长的上下文。压缩——把历史摘要成 2K tokens。直接降低每次调用的令牌成本。

**人类在环检查点。** 在调用昂贵的操作之前——比如调用 Opus 做一个需要 30 秒推理的任务——先问人类。"这个需要 Opus 吗？"不一定。人类说"Sonnet 够。"省了 50 倍的成本。

## Claude Code 给的原语

Claude Code Agent SDK 把预算控制作为一等 API：

- `max_turns`——Agent 循环的迭代上限。
- `max_budget_usd`——美元上限。会话违反→中止。
- `allowed_tools` / `disallowed_tools`——控制 Agent 能调什么工具。
- 工具前钩子——你可以插自定义的成本核算逻辑。

Anthropic 的文档明确地说：Auto Mode 需要预算控制。不是建议。是"没有预算控制的自主 Agent 是未完成品。"

## 新工具 = 新上限 = 新告警

$1,200→$4,800 的故事教了一件事我不想你忘记：**每当你给 Agent 加一个新工具，你同时要加三样东西：该工具的每调用上限、该工具的成本增长告警、该工具的异常行为检测器。** 不是"以后再说"——是上线前。模板，不是选择。

## 结课清单

给你的 Agent 加成本控制：

1. 堆栈，不是一个上限。请求级→任务级→窗口级→累积级。四组。12 层。
2. 速度限制捕获循环——10 分钟内 $50。不是月度上限。
3. 分层模型路由——默认用小模型，只升级必要的。
4. 提示缓存——系统提示和稳定上下文不重复付费。
5. 每个新工具带三个东西——每调用上限、成本告警、异常检测。
6. 人类在环检查点在昂贵操作前——省的不是几块钱，是 50 倍成本。
7. 在欧盟→日志和上限是法律要求（EU AI Act 第 14 条），不是好实践。

**最后一句话：**

**Agent 的成本控制不是省钱——是你在给一个能自己决定花多少钱的系统设定花钱的规则。堆栈是你设定的规则——请求级、任务级、窗口级、累积级。没有堆栈，你的 Agent 不是"能自主工作"——是"能自主花你的钱，你不设上限它就不会停。"**
