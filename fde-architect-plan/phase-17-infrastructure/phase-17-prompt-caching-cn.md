## 学习目标

- 区分 L2 提示/前缀缓存（提供商端的 KV 复用）和 L1 语义缓存（在相似提示时绕过 LLM）。
- 解释 Anthropic 的 `cache_control` 显式标记和两种 TTL 选项（5 分钟 vs 1 小时）及其价格倍数。
- 给定命中率、提示/响应组合和 Token 价格，计算预期月度节省。
- 指出使账单膨胀 5-10 倍的并行化反模式和使命中率归零的动态内容反模式。

## 问题

你给 RAG 服务加了提示缓存。账单不变。你测量命中率 — 是 7%。你的提示看起来是静态的但其实不是 — 系统提示包含精确到分钟的当前日期、一个请求 ID 和一个为了多样化随机重排的示例。每个请求都写入一个新缓存条目，读取零次。

另外，你的 Agent 对每个用户问题并行运行 10 个工具调用。所有 10 个请求在第一次缓存写入完成之前到达提供商。10 次写入，零次读取。你的账单是"启用缓存"预期成本的 5-10 倍。

缓存是一个协议，不是一个标志。两层缓存，两种不同的失败模式。

## 概念

### L2 — 提供商提示/前缀缓存

提供商存储可缓存前缀的注意力 KV，并在下一个匹配该前缀的请求中复用它。你为一次写入付费，读取几乎免费。

**Anthropic（Claude 3.5 / 3.7 / 4 系列）**：在请求中使用显式的 `cache_control` 标记。你标记哪些块是可缓存的。TTL：5 分钟（写入成本为基价的 1.25 倍）或 1 小时（写入成本为基价的 2 倍）。缓存读取：Claude 3.5 Sonnet 上 $0.30/M vs $3.00/M 全新 — 便宜 10 倍（docs.anthropic.com，截至 2026-04）。不同模型（Opus/Haiku 单独发布）价格不同；始终交叉检查实时定价页面。

**OpenAI**：提示 ≥1024 Token 时自动缓存（platform.openai.com，2026-04）。无需显式标志。在当前 gpt-4o/gpt-5 价格表上，缓存输入比全新便宜约 10 倍。文档和发布说明中均未公布官方命中率基线；社区报告在精心设计提示的情况下集中在 30-60%。通过监控 `usage.cached_tokens` 来测量你自己的。

**Google（Gemini）**：通过显式 API 进行上下文缓存；100 万 Token 上下文意味着缓存的回报更高。

**自托管（vLLM、SGLang）**：Phase 17 · 06 涵盖 RadixAttention — 使用自己的计算资源实现相同模式。

### L1 — 应用级语义缓存

在调用 LLM 之前，对提示进行哈希、嵌入，查找相似的已缓存请求（余弦相似度高于阈值，通常 0.95+）。命中时返回缓存响应。未命中时调用 LLM 并缓存结果。

开源：Redis Vector Similarity、GPTCache、Qdrant。商业：Portkey Cache、Helicone Cache。

供应商准确率声明指的是返回的缓存响应在语义上合适的频率 — 而不是你命中的频率。生产命中率：

- 开放式聊天：10-15%。
- 结构化 FAQ / 支持：40-70%。
- 代码问题：20-30%（小的变体杀死命中）。
- 语音 Agent 重复提示：50-80%（语音规范化固定集合）。

### 并行化反模式

你的 Agent 并行发出 10 个工具调用。所有 10 个都有相同的 4K Token 系统提示。Anthropic 缓存写入是逐请求的；第一次缓存写入在提供商看到提示后约 300 毫秒完成。请求 2-10 在同一毫秒窗口内到达，每个都看到缓存未命中。你支付 10 次写入溢价，0 次读取折扣。

修复：使用顺序优先批处理 — 先单独发请求 1，然后在请求 1 的缓存填充后发出请求 2-10。为第一个工具调用增加 300 毫秒延迟；节省 5-10 倍的账单。

### 动态内容反模式

你的系统提示看起来像：

```
你是一个有用的助手。当前时间是 14:32:17。
用户 ID：abc123。今天是星期二...
```

每个请求都是唯一的。每个请求都写入。零命中。

修复：将所有真正静态的内容移到可缓存前缀中；在缓存边界之后追加动态内容：

```
[可缓存]
你是一个有用的助手。[规则、示例、说明]
[/可缓存]
[动态，不可缓存]
当前时间：14:32:17。用户：abc123。
```

ProjectDiscovery 通过这种方式将缓存命中率从 7% 提升到 74%，并公开发布了拆解过程。

### 为隔夜工作负载叠加批处理 + 缓存

批量 API（Phase 17 · 15）在 24 小时周转时间下提供 50% 折扣。在此基础上加上缓存输入，你可以得到约 10 倍的叠加效果。隔夜分类、标注和报告生成工作负载可以通过叠加降至同步无缓存成本的约 10%。

价格点捕获于 2026-04，来自链接的供应商文档，每隔几个月就会变动 — 在依赖之前重新检查。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇非常实用的"缓存经济学"课程，接地气、案例驱动。文档不只是讲缓存的 API 怎么用，而是直接切入两个生产环境中最常见的反模式（并行化陷阱和动态内容陷阱）— 这正是大多数"缓存教程"不会告诉你的。ProjectDiscovery 从 7% 到 74% 命中率的案例拆解非常有说服力，让抽象的"优化提示可缓存性"变成了可操作的步骤（把动态内容移出前缀）。价格数据标注了 2026-04 的时间戳，非常诚实。不足在于 L1 语义缓存的向量检索性能没有涉及（索引选择、近似搜索 vs 精确搜索的取舍），以及没有讨论缓存失效策略（缓存漂移问题）。

### 二、知识结构梳理

- **分层概念层**：L2 提供商前缀缓存（KV 复用 — Anthropic `cache_control`、OpenAI 自动缓存、Google 上下文缓存、自托管 RadixAttention）和 L1 应用级语义缓存（嵌入相似度匹配 — GPTCache、Redis Vector）。两层分别解决不同问题。
- **经济学层**：写入溢价（1.25x vs 2x）、读取折扣（10x）、TTL 对命中率的影响、盈亏平衡公式。隔夜工作负载叠加批处理 + 缓存的 10% 成本目标。
- **反模式层**：并行化反模式（N 次写入零次读取，300ms 写入窗口内的竞态）、动态内容反模式（日期/ID/随机化示例在前缀中杀死缓存）。ProjectDiscovery 的案例展示了什么是正确做法。

### 三、核心洞察

1. **缓存是一个协议，不是一个标志。** 打开"启用缓存"开关不等于缓存生效。提示工程决定了缓存命中率 — 7% 和 74% 的差异来自提示结构的重构，不是 API 参数。
2. **并行化反模式是隐蔽的账单杀手。** 300 毫秒内发起 10 个相同前缀的请求，全部错过首次缓存写入，账单膨胀 5-10 倍。这个时间窗口是如此之小，以至于很多团队甚至没意识到问题存在。
3. **动态内容在前缀中的破坏性是绝对的。** 一个时间戳就能让整个缓存策略失效。缓存命中是一个二进制命题：前缀完全匹配才有命中，99% 匹配也是未命中。
4. **L1 和 L2 缓存解决不同的问题。** L2 省钱和延迟（KV 复用），L1 完全跳过 LLM 调用（响应复用）。L1 的命中率低得多（10-70%），但每次命中的收益也大得多（零 API 调用）。
5. **语义缓存的准确率 vs 命中率权衡是经典的精确率/召回率问题。** 0.95 阈值保证准确性但牺牲命中率，0.85 提高命中率但引入错误响应。产品决策取决于错误响应的容忍度。
6. **隔夜批处理 + 缓存的叠加效应可达 90% 成本削减。** 批量 API 50% 折扣 × 缓存读取 10% 成本 ≈ 5% 总成本。这是分析类工作负载的最佳成本策略。
7. **Anthropic 的显式 `cache_control` vs OpenAI 的自动缓存各有优劣。** 显式让你完全控制但需要工程投入；自动缓存零配置但可能缓存了你不想缓存的东西，也可能没缓存你想要的。

### 四、教学建议

1. **从"7% 命中率"的尴尬场景开场。** 展示加了缓存但账单没变的实际截图，让学员感受"我加了缓存为什么没效果"的困惑——这是绝大多数团队的真实体验。
2. **并行化反模式用时间线图展示。** 横轴是时间，纵轴是 10 个请求在 300ms 窗口内并发到达，标注第一个缓存写入完成的时间点。视觉化解释为什么 2-10 号请求都错过了。
3. **ProjectDiscovery 的案例要做成"提示重构前后"的并排对比。** 左边是带时间戳的前缀（7% 命中率），右边是分离的动态内容（74% 命中率）。让学员自己找差异。
4. **盈亏平衡计算需要动手做。** 给定请求频率（每分钟 N 个），分别计算 5 分钟 TTL 和 1 小时 TTL 的写入/读取次数和总成本。让学员在 Excel 或代码中完成。
5. **语义缓存的阈值选择可以做成互动实验。** 用一个包含 20 个相似但不同的提示的数据集，让学员手动标注哪些应该在 0.95、0.90、0.85 阈值下命中，感受准确率/召回率的权衡。
6. **强调"先跑 `code/main.py` 再讨论"。** 模拟器的并行化开关是一个直接的实验，让学员切换标志看账单差异，理解比讲 10 分钟都有效。
7. **扩展讨论话题：缓存失效策略。** 当基座模型升级（v1 → v2）时，缓存的是旧模型的响应。语义相同但"质量应该不同"。这是缓存工程中的难点问题。

### 五、值得补充的内容

1. **向量数据库的选型对 L1 缓存性能的影响。** Redis vs Qdrant vs FAISS 在 10 万条目下的查询延迟和召回率差异 — 这对高流量系统很关键。
2. **缓存预热策略。** 如何在部署时预填充热门前缀以减少冷启动成本。企业级系统的常见需求。
3. **缓存命中率的监控和告警。** 什么样的命中率下降值得触发告警？如何区分"流量模式变了"和"缓存配置坏了"？
4. **多模型提供商混合场景下的缓存策略。** 当你同时使用 Anthropic 和 OpenAI，如何统一管理 L2 缓存策略并避免重复计价。
5. **关于 Anthropic 和 OpenAI 缓存定价模型的差异，做一个对比表格。** 包括 TTL、写入溢价、读取折扣、Token 粒度下限 — 便于学员选型。

### 六、一句话总结

**提示缓存不是"开个开关就能省钱"的免费午餐 — 7% 和 74% 命中率的差别在于你是否愿意重构提示结构；并行化和动态内容是两头悄悄偷走你预算的沉默账单杀手。**

# 🎓 Agent 架构课：提示缓存——你以为开了缓存就省钱？先看看你的命中率再哭

同学们好。我是你们的FDE工程老师，今天讲的是 提示缓存。

**你开了 Anthropic 的 prompt caching，心想"90% 的成本省了！"。然后账单来了——命中率 12%。** 因为你把每次都不一样的 `user_message` 放在了缓存前缀里。

缓存规则：**缓存的是前缀。** 两个请求共享相同前缀的部分才被缓存。正确做法：系统提示 + 工具声明放前缀（不变），用户消息放后缀（每次不同）。错误做法：把整个对话历史放前缀——每次都变，缓存永远不命中。

关键数字：Anthropic 缓存最短前缀 = 1024 token（Claude Sonnet/Haiku）或 2048 token（Opus）。前缀短于这个 → 不缓存。TTL = 5 分钟未使用后过期。命中后 TTL 重置。

## 结课清单

1. **缓存的是前缀。** 不变的放前面，会变的放后面。
2. **最短前缀 1024-2048 token。** 太短不缓存。
3. **TTL 5 分钟。** 活跃使用自动续期。
4. **先测命中率再开香槟。** 开了缓存不等于省钱——命中率说了算。

最后一句话：**提示缓存不是开了就省钱——是设计对了才省钱。把不变的放前面，把会变的放后面。顺序反了，缓存就是个装饰品。**

# 💼 从业者故事：提示缓存——你以为开了缓存就省钱？先看看你的命中率再哭

我犯过一个特别蠢的错误。上线一个新 Agent 之后，信心满满地打开了 Anthropic 的 prompt caching。按文档说，缓存命中可以省 90% 的成本。我心想"这不就是白捡钱吗"。

月底账单出来的时候我差点把咖啡喷在屏幕上——不仅没降，还比上个月高了 8%。我翻了一个下午的日志才搞明白：缓存命中率是 7%。

百分之七。

不是缓存坏了，是我的 Prompt 里有一个"当前时间 2026-03-15 14:32:17"。每秒钟都不一样，所以每秒钟都是一个全新的前缀。每一次请求都是缓存未命中，每一次未命中都要支付 1.25 倍的写入溢价。我不是在省钱，我是在主动多花钱。

这就是提示缓存最残忍的真相：**它不是开关，是协议。** 你打开了缓存不等于缓存生效。缓存生效的前提是你的前缀必须是静止的——像山一样一动不动。任何动态内容放在前缀里，就像往平静的湖面扔了一块石头，命中率直接归零。

## 两个反模式，教会了我所有关于缓存的事

**反模式一：并行化陷阱。** 你的 Agent 并行调用了 10 个工具。10 个请求都有相同的系统提示——理论上应该全部命中缓存。但 10 个请求几乎同时到达提供商，而第一次缓存写入需要约 300ms。结果：10 个请求全部"看到"缓存是空的，各自独立写了一份缓存。你支付了 10 次写入溢价，读取了 0 次。

我第一次发现这件事的时候，整个人都不好了。我们在高并发场景下花钱比不缓存还快 5-10 倍。修复方法也简单：把请求串行化——第一个请求单独发，等缓存写完之后（加个 300ms 延迟）再发剩下的 9 个。多等 300ms，省下 90% 的成本。这个数学每个人都应该算一遍：300ms 的延迟 vs 9 倍成本的差别。

**反模式二：动态内容污染前缀。** 你的 System Prompt 里有时间戳、用户 ID、随机排列的 few-shot 示例。每一个看起来"很小"的动态字段，都是在告诉提供商"这个前缀是新的一样"。缓存前缀匹配是二进制全匹配——99% 一样也是未命中。ProjectDiscovery 把动态内容从前缀里剥离出去之后，命中率从 7% 跳到 74%。74%！这不是微调，是重构。

修复手法：把你的 Prompt 像切蛋糕一样分成两块。上面的"静态层"——角色定义、规则、固定示例——标记为 `cache_control`。下面的"动态层"——时间戳、用户信息、查询本身——追加在缓存边界之后。静态层越厚，省钱越多。

## L1 和 L2 缓存：别搞混了

L2（提供商级别）缓存的是注意力 KV——就是模型"读懂了你的前缀"的那个中间状态。命中率高的话，你能省掉预填充的算力成本，延迟也少一大截。

L1（应用级别）语义缓存是在调用 LLM 之前，先把用户的问题做嵌入，跟以前回答过的问题比相似度。命中就直接返回缓存答案，完全跳过 LLM 调用。省的不是算力折扣，是整笔 LLM 调用费。

但 L1 的命中率天生低。开放式聊天就 10-15%，只有结构化的 FAQ 场景才能到 40-70%。而且语义缓存有准确率问题——相似度阈值设 0.95，保证答案靠谱但命中少；设 0.85，命中了但可能返回错误答案。这是经典的精确率 vs 召回率权衡。

我现在的策略是两层都用：L2 省钱 + 省延迟，L1 作为"完全相同或几乎相同问题"的快速通道。不要指望 L1 救你的账单，它的天花板是 L2 的十分之一。

## 隔夜批处理 + 缓存 = 终极省钱套餐

如果你有一些不需要实时返回的任务——比如每天晚上对当天的用户对话做分类、给长文档打标签、生成周报——你可以叠加两个利器：批处理 API（50% 折扣）加上提示缓存（90% 输入成本减免）。叠完后总成本降到同步无缓存的约 5%。花 5 块钱做以前 100 块的事。

这个组合对分析类工作负载是杀手级的。我们有一个客户的每日舆情分析 pipeline，之前在同步模式下一个月花了四万块，切换到"隔夜批处理 + 缓存"之后降到两千块。老板还以为账单打错了。

## Anthropic 的显式标记 vs OpenAI 的自动缓存

Anthropic 让你用 `cache_control` 手动标记哪些块是可缓存的。好处是完全控制，坏处是你需要工程投入去设计哪个块该缓存。OpenAI 自动缓存 ≥1024 Token 的提示，零配置但你也零控制——它可能帮你缓存了你不想缓存的（浪费 TTL），也可能漏了你想要的。

从工程实践看：如果你有固定的系统提示模板，用 Anthropic 的显式标记能榨出最高的命中率。如果你的提示动态性强、结构不固定，用 OpenAI 的自动缓存至少零维护成本。选哪个不是技术问题，是"你愿意为省更多钱付出多少工程时间"的问题。

还有一个大家经常忽略的细节：Anthropic 的 TTL 有两种——5 分钟（1.25x 写入溢价）和 1 小时（2x 写入溢价）。如果你的请求间隔密度够大，5 分钟 TTL 的性价比最高。但如果用户的对话间隔经常超过 5 分钟（比如客服系统用户可能在思考后再追问），1 小时 TTL 虽然写入贵但命中率高。这个抉择需要看你的真实流量分布——不能拍脑袋，要算。

**提示缓存是所有省钱手段里 ROI 最高的——但前提是你理解它不是一个参数，而是一个需要你重构 Prompt 结构的工程合同。7% 和 74% 的差距不是缓存的好坏，是你把动态内容放在哪里。先重构，再开缓存，最后看账单微笑。**
