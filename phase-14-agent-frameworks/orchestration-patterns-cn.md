# 编排模式：监管者、群体、层级

> 四种编排模式在 2026 年的框架中反复出现：监管者-工作者、群体/点对点、层级、辩论。Anthropic 的指导原则："关键在于为你的需求构建正确的系统。"从简单起步；仅当单个 Agent 加上五种工作流模式仍不够时，才增加拓扑结构。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 12（工作流模式）、Phase 14 · 25（多 Agent 辩论）
**用时：** 约 60 分钟

## 学习目标

- 说出四种反复出现的编排模式及各自适合的场景。
- 描述 2026 年 LangChain 的建议：基于工具调用的监管 vs 监管器库。
- 解释 Anthropic 的"构建正确系统"规则及其如何控制拓扑选择。
- 基于标准库和通用脚本化 LLM 实现全部四种模式。

## 问题

团队在需要之前就奔向"多 Agent"。四种模式在框架中反复出现；一旦能说出它们的名字，你就能选对——或者完全跳过拓扑。

## 概念

### 监管者-工作者（Supervisor-worker）

- 一个中央路由 LLM 将任务分派给专家 Agent。
- 决策：返回自身、转交专家、终止。
- 专家之间不互相通信；所有路由经过监管者。

框架：LangGraph `create_supervisor`、Anthropic 编排器-工作者、CrewAI 层级流程。

**2026 年 LangChain 建议：** 通过直接工具调用实现监管，而非使用 `create_supervisor`。这提供了更精细的上下文工程控制——你精确决定每个专家看到什么。

### 群体 / 点对点（Swarm / peer-to-peer）

- Agent 通过共享工具面直接转交。
- 无中央路由器。
- 比监管者延迟更低（更少跳数）。
- 更难推理（无单一控制点）。

框架：LangGraph swarm 拓扑、OpenAI Agents SDK 转交（当所有 Agent 均可互相转交时）。

### 层级（Hierarchical）

- 监管者管理子监管者，子监管者管理工作者。
- 在 LangGraph 中实现为嵌套子图；在 CrewAI 中为嵌套工作组。
- 可扩展到大规模 Agent 群体，代价是操作复杂性。

何时需要：当单个监管者的上下文预算无法容纳所有专家的描述时。

### 辩论（Debate）

- 并行提案者 + 迭代交叉评审（第 25 课）。
- 不完全是编排——更偏向验证——但在框架中以拓扑选择的形式出现。

### CrewAI 的 Crew vs Flow

CrewAI 正式划分了两种部署模式：

- **Flow** 用于确定性事件驱动自动化（推荐的生产环境起点）。
- **Crew** 用于自主角色协作。

这与上述四种模式正交，但映射到拓扑：Flow 通常是监管者或层级；Crew 通常是带有 LLM 路由器的监管者。

### Anthropic 的指导原则

"在 LLM 空间中的成功不是构建最复杂的系统。而是为你的需求构建正确的系统。"

决策顺序：

1. 单个 Agent + 工作流模式（第 12 课）——从此起步。
2. 监管者-工作者——当你有 2-4 个专家时。
3. 群体——当延迟比推理清晰度更重要时。
4. 层级——仅当监管者上下文预算不够时。
5. 辩论——当准确性比成本更重要时。

### 此模式在何处会出错

- **拓扑优先思维。** "我们需要多 Agent"——在搞清楚多 Agent 解决什么问题之前。
- **群体中的弹跳转交。** A -> B -> A -> B。使用跳数计数器。
- **虚假层级。** 三层是因为"企业级"；实际只有两个团队。合并。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 14 课程中"拓扑决策"的核心一课。它不是实现指南——它是**架构选型手册**。文档将四种在 2026 年各框架中反复出现的编排模式归纳为一个清晰的决策树，从上节课（第 12 课，工作流模式）的"先别用 Agent"延伸到"不得不用多 Agent 时怎么选"。最大的价值在于"反模式"部分——拓扑优先思维、弹跳转交、虚假层级——这三条是每个做过生产多 Agent 系统的人都踩过的坑。适合已有工作流模式基础、正在规划多 Agent 拓扑的架构师。

### 二、知识结构梳理

**认知基础层：** 第 12 课（工作流模式）是本节课的直接前置——学生必须理解"单个 Agent + 五种工作流"是默认方案，才会理解"什么时候必须加拓扑"。第 25 课（多 Agent 辩论）是辩论模式的前置。CrewAI 的 Crew/Flow 二分法提供了部署视角。

**架构模式层：** 四种模式按复杂度递进——监管者（2-4 专家，中心路由）、群体（低延迟，无中心）、层级（大规模，嵌套管理）、辩论（准确性优先，并行评审）。每种模式有其成本、延迟和可调试性 trade-off。

**工程实践层：** 2026 年 LangChain 的"工具调用监管"建议是该层的关键——不用框架的 `create_supervisor`，而是手写工具调用来实现监管。这呼应了第 12 课"简单胜过复杂"的原则，并提供了更深层的上下文工程控制。

### 三、核心洞察（备课时的关键理解）

1. **四种模式不是菜单——是决策树。** 文档给出的 Anthropic 决策顺序（单 Agent → 监管者 → 群体 → 层级 → 辩论）是线性约束：每往下一层，复杂性代价增加，必须由需求的复杂性来支付。

2. **监管者-工作者是四种模式中使用频率最高的。** 2-4 个专家、一个中央路由——覆盖了 90% 的多 Agent 需求。群体、层级、辩论是剩余的 10%，分别优化延迟、规模和准确性。

3. **"基于工具调用的监管"是 2026 年的关键工程洞察。** LangChain 建议放弃 `create_supervisor` 库，改用直接工具调用实现监管。原因：库自动管理上下文，但你失去了决定"每个专家看到什么"的能力。手写工具调用给你精确的上下文工程控制权。

4. **群体的延迟优势是以可推理性的丧失为代价的。** 没有中央路由器意味着少一跳延迟——但也意味着出了问题你没有一个地方查"谁决定把这个转给谁"。跳数计数器是群体的最小化安全网。

5. **层级唯一合理的采用理由是"监管者的上下文预算不够"。** 不是因为"我们是大公司所以应该有层级"——是因为 20 个专家的描述塞不进一个监管者的上下文窗口。如果没有上下文窗口的硬约束，层级就是过度设计。

6. **辩论不是编排——是验证。** 它出现在拓扑选择里是因为框架把它当作一种编排方式。但它的本质是让多个 LLM 互相验证对方的工作，而不是分配任务。用辩论不是因为需要"多个 Agent 协作"——是因为准确性的边际收益大于成本的边际增长。

7. **CrewAI 的 Crew/Flow 区分是部署模式，不是拓扑模式。** Flow 是确定性的——事件驱动，适合审计和合规场景。Crew 是自主的——适合探索和开放式场景。同一个拓扑（比如监管者）可以部署为 Flow 模式或 Crew 模式，取决于你需要确定性还是灵活性。

### 四、教学建议

1. **用"什么时候你觉得单个 Agent 不够了"来开场。** 让学生举出自己遇到过的场景——任务太多、类型太杂、单个 LLM 搞不定。这些就是拓扑需求的种子。先有痛点，后有方案。

2. **四种模式用"调用图"可视化对比。** 画四个图：监管者（星形）、群体（网状）、层级（树状）、辩论（并行）。让学生一眼看到信息流的差异，比文字描述快得多。

3. **弹跳转交问题用现场演示。** 写一个没有跳数计数器的 swarm，让它 A->B->A 弹跳三次，看它浪费的令牌和时间。然后加上跳数计数器，再跑。视觉化效果比解释有效。

4. **"基于工具调用的监管"值得和 `create_supervisor` 做对比。** 用同一个任务，一次用 LangGraph 的 `create_supervisor`（5 行配置），一次用直接工具调用（30 行手写），对比上下文的控制粒度。

5. **决策树的每个节点都要追问"为什么会走到下一步"。** 单 Agent → 监管者：因为专家类型超过了单个 LLM 能同时处理的。监管者 → 群体：因为延迟要求不允许经过中心路由器。监管者 → 层级：因为上下文窗口装不下所有专家描述。每个升级都必须有一个可量化的理由。

6. **"虚假层级"作为反模式，适合让学生互相审查。** 给出一段有虚假层级的架构描述，让学生指出哪两层可以合并，并给出合并后的新架构。这是诊断能力的训练。

7. **练习 4（性能分析）是最全面的。** 在同一工作负载上跑四种模式，记录延迟、成本、准确性、可调试性。这张对比表是学生离开课堂后做架构决策时的参考表。

### 五、值得补充的内容

1. **混合模式的成本模型。** 实际系统通常是监管者 + 辩论（质量门控）或 监管者 + 群体（对延迟敏感的子任务）。需要逐模式的令牌成本公式。

2. **监管者的负载均衡。** 当一次任务同时需要多个专家时，监管者是串行调用还是并行？这是第 12 课"并行化"模式在多 Agent 场景下的延伸。

3. **跨拓扑的可观测性。** 四种模式的轨迹形状不同（星形、网状、树形、并行），对日志格式和可视化有不同的要求。LangGraph 的 tracing 在这些拓扑上的表现各不相同。

4. **从监管者到群体的渐进迁移。** 文档提到群体更难推理，但没说怎么安全地迁移。需要一个 checklist：① 标注延迟敏感的子任务 → ② 为这些子任务建立点对点转交通道 → ③ 用跳数计数器限界 → ④ 保持监管者作为全局回退。

5. **自主 Agent 框架（AutoGen）对四种模式的映射。** 第 12 课提到 AutoGen v0.4 的 Actor 模型——它如何映射到监管者/群体/层级？这对从 LangChain 迁移到 AutoGen 的团队很重要。

### 六、一句话总结

**多 Agent 系统不是 Agent 多了就更好——四种拓扑不过是四种控制权分布方式。选择的标准不是复杂度，而是"上下文预算、延迟预算、准确性预算——哪个是瓶颈？"**

# 🎓 Agent 架构课：多 Agent 编排——四种拓扑，四种控制权分布

同学们好。我是你们的 Agent 架构老师。

上节课我们讲了五种工作流模式。我告诉你们——大多数时候你不需要 Agent，单个 LLM 加工作流就够了。今天这节课，我假设你已经遇到一个真正的情况：**单个 Agent 搞不定了。**

不是因为你想要多 Agent——是因为任务本身要求多个不同领域的专业知识，或者延迟不允许你串行处理，或者准确性要求你必须让多个视角交叉验证。

今天的核心问题是：**当你真的需要多个 Agent 一起工作时，谁决定下一个 Agent 做什么？** 答案只有四种。

## 一个你一定会遇到的场景

假设你在做一个客服系统。一个用户发来一条消息——可能是在抱怨产品故障（需要退款专家），可能是在报告网站白屏（需要 bug 专家），可能是在问你们的定价方案（需要销售专家）。

只有一个 LLM 处理所有这三种？它会在退款的语气里塞 bug 修复的技术术语，在销售话术里写错了产品名。

你需要三个专家 LLM：退款、bug、销售。但是——**谁来决定"这条消息给哪个专家？"** 这就是编排问题的核心。

业界有四种答案。我们今天挨个拆开看。

## 模式一：监管者-工作者——你第一个该学的多 Agent 模式

监管者-工作者的结构是最直观的：一个中央"监管者"LLM，下面挂着一组"工作者"专家。

工作流程：用户消息进来 → 监管者读一遍 → 判断"这是退款问题"→ 转给退款工作者 → 退款工作者给出回复 → 监管者检查 → 结束。

关键特征：
- **所有路由经过监管者。** 退款工作者和 bug 工作者永远不会直接对话。它们甚至不知道彼此存在。
- **监管者决定三种动作：** 转给专家、回到自己（需要再想想）、终止（回答用户）。
- **专家是被调用的工具。** 每个专家是一个函数，接受输入、返回输出。监管者像调度器一样调用它们。

**什么时候用：** 这是 90% 多 Agent 场景的答案。2-4 个专家，分工明确，一个中央路由——够了。

**一个关键的工程细节：** LangChain 有一个 `create_supervisor` 函数，一行代码创建监管者。但 2026 年的建议是——**别用这个函数。** 用直接工具调用实现监管。

为什么？因为 `create_supervisor` 自动管理上下文——你不知道每个专家看到了什么。手写工具调用让你精确决定：退款专家只看到退款相关的上下文，bug 专家只看到 bug 相关的上下文。上下文控制权不能交给框架——这是你作为架构师的核心职责。

## 模式二：群体——去中心化，低延迟，高混乱

群体模式的反直觉之处：**没有监管者。** 每个 Agent 都可以直接把任务转交给任何其他 Agent。

工作流程：用户消息进来 → Agent A 读一遍 → "这不太像我该处理的，但 Agent C 可能更懂" → 直接转给 Agent C → Agent C 处理 → 回复用户。

为什么不用监管者？**延迟。** 监管者模式中每条消息要经过一个额外的跳（监管者读→转发）。如果延迟是你的瓶颈——比如实时聊天系统——群体模式少一跳。

代价是什么？**可推理性的丧失。** 出了问题，你没有一个地方去查"谁决定把这个转给谁的"。每个 Agent 自己做转交决策，轨迹是分布式的。

**群体有一个最经典的失败模式：弹跳转交。** Agent A 说"我不懂这个，转给 B。"Agent B 说"我也不懂，转回给 A。"A 再次转给 B。无限弹跳。解决方案很简单——**跳数计数器。** 每条消息带一个 `hop_count`，转交时 +1。到了 3 次就锁死，强制当前 Agent 给出回复。

## 模式三：层级——上下文预算不够的时候

层级听起来很"企业级"，但它的出现理由其实非常具体：**你的监管者上下文窗口装不下所有专家的描述了。**

假设你有 20 个专家 Agent。一个监管者要了解所有 20 个专家的能力描述，才能做路由决策。20 条描述塞进一个上下文窗口——满了。监管者开始丢信息，路由质量下降。

解决方案：引入子监管者。顶级监管者管理 3 个子监管者（售前、售后、技术），每个子监管者管理 3-5 个专家。顶级监管者只需要知道 3 个子监管者的能力——而不是 20 个专家的。上下文预算解决了。

**什么时候不该用层级：** 如果你的专家数量不超过 8 个——不需要层级。不要因为"我们是大公司"就加层级。层级是上下文预算的解决方案，不是组织架构的映射。

## 模式四：辩论——让多个视角互相挑战

辩论的运作方式和其他三种完全不同。它不是分配任务——是验证答案。

工作流程：用户提问 → 三个 LLM 并行给出答案 → 三个答案被彼此阅读 → 每个 LLM 被要求评审其他两个答案 → 综合结论。

这不是编排——这是验证。但它被框架作为"多 Agent 拓扑"提供，因为它在架构决策上和前三种并列。

**什么时候用辩论：** 当错误成本极高时。医疗诊断、法律建议、金融风险评估——一个 LLM 的回答不够可靠，你需要多个视角交叉验证。代价是成本——辩论是最贵的模式，因为你要为每个问题调用至少 6 次 LLM（3 个提案 + 3 个评审）。

辩论模式的源头是一篇叫"Society of Minds"的论文（Du et al., 2023）——这不在今天的阅读材料里，但你可以把它理解为"用多个模型的共识近似真值"。

## 四种模式的关系：一张决策树，不是菜单

```
单个 Agent + 五种工作流（第 12 课）
   ↓ 2-4 个专业领域，需要分派？
   监管者-工作者
   ↓ 延迟是最紧迫的约束？
   群体（点对点）
   ↓ 专家太多，监管者上下文装不下？
   层级（嵌套监管者）
   ↓ 错误成本极高，需要多重验证？
   辩论（并行评审）
```

每一步向下走，成本都在增加。每一层也都在缩小适用范围。**建筑法则：只在必须时才往上爬。**

## CrewAI 的一个有用区分：Flow vs Crew

CrewAI 是 2026 年多 Agent 框架中的一个，它做了一个有用的区分——但必须理解清楚，因为它容易被误读。

**Flow** 是确定性的。事件触发 → 执行 → 结束。你知道每一步会发生什么。适合生产环境、需要审计的场景。

**Crew** 是自主的。你把角色分配给 Agent，它们自己决定怎么协作。适合探索性任务。

这个区分和四种拓扑是正交的——同一个监管者拓扑，可以用 Flow 模式（步骤预定义）实现，也可以用 Crew 模式（Agent 自主运行）实现。你的选择取决于一个问题：**你的工作流是可预测的，还是需要 Agent 自己决策？**

## 生产中的三个反模式

### 反模式一：拓扑优先思维

"我们的新项目要用多 Agent 系统！"——在你们搞清楚有哪些不同类型的子任务之前。

拓扑是解决方案，不是需求。如果你不知道自己需要几个专家、每个专家负责什么、它们之间的信息流是什么——你不是在选拓扑，你是在装饰架构图。

### 反模式二：弹跳转交

A 转给 B，B 转回给 A。没有跳数计数器的 swarm 就是一台燃烧令牌的永动机。

解决：永远在转过手里加 `hop_count`。3 次后锁死。这不是优化——是保险。

### 反模式三：虚假层级

一个系统有三个监管者层、两个子监管者层——但实际只有 5 个专家。这是"企业架构图"误导了"软件架构"。层级有唯一的合理入场条件：上下文预算。没有这个硬约束，层级就是过度设计。

## 结课清单

如果你要设计一个多 Agent 系统的拓扑：

1. 先检查第 12 课的工作流模式——单个 Agent 加五种工作流是否足够了？
2. 如果不够：需要几个专家？2-4 个 → 监管者-工作者。
3. 延迟是关键吗？→ 群体，但必须加跳数计数器。
4. 专家超过 8 个吗？→ 层级，子监管者分组。
5. 错误成本极高吗？→ 辩论，但接受成本翻倍。
6. 任何时候：能用直接工具调用实现监管，就不要用框架的 `create_supervisor`。
7. 任何时候：不要因为"看起来很酷"加拓扑。每加一层都加了延迟、成本和调试难度。

**最后一句话：**

**多 Agent 编排不是"让更多 Agent 一起工作"——是"在谁决定下一步"这个问题上，选择控制权的分布方式。监管者是你替他们决定，群体是他们自己决定，层级是分层决定，辩论是互相质疑。选哪种只取决于一个问题——信息在这个系统里应该怎么流动？**
