# 评审 Agent：将构建者与评分者分离

> 写代码的 Agent 不能给自己的代码评分。评审者是第二个循环，有不同的系统提示、不同的目标，以及对构建者产出的只读访问权。构建者与评审者之间的差距是大多数可靠性所在的地方。

**类型：** 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 38（验证门控）
**用时：** 约 55 分钟

## 学习目标

- 陈述为什么同一个 Agent 不能可靠地评审自己的工作。
- 构建一个评审 Agent 循环，消费构建者制品并产出结构化评审报告。
- 撰写一个评审评分量表，对具体维度进行评分，而非靠感觉。
- 将评审者接入工作台，使人类审查步骤从一份真实的制品开始。

## 问题

你让 Agent 修一个 bug。它编辑了四个文件，运行了测试，报告完成。验证门控确认验收运行了且范围保持。门控说 `passed: true`。你合并了。两天后发现修复解决了 bug 的错误一半。

验收是必要的，但不是充分的。评审者问的是验收不能问的问题：这解决了正确的问题吗？它是否在没有标记的情况下扩展了范围？它是否文档化了本应被质疑的假设？它是否将工作台留在了下一次会话可以接上的状态？

## 概念

### 评审评分量表

五个维度，每个评分 0 到 2。

| 维度 | 问题 |
|------|------|
| 问题匹配度 | 变更是否按所述解决了任务，而非附近的某个任务？ |
| 范围纪律 | 编辑是否限制在契约内，还是契约被有计划地扩展了？ |
| 假设 | 所有隐藏假设是否被写在了可审查的地方？ |
| 验证质量 | 验收命令是否实际证明了目标，还是证明了一个更弱的版本？ |
| 交接就绪度 | 下一次会话能否干净地从当前状态接上？ |

满分 10 分。低于 7 分是软失败；低于 5 分是硬失败。

### 评审者是独立角色，而非独立模型

你可以用与构建者相同的模型来运行评审者。纪律在于角色分离：不同的系统提示、不同的输入、对 diff 无写访问权。姿态的改变就是信号的改变。

### 评审者不能编辑 diff

评审者读取 diff、状态、反馈、裁决。它写一份报告。它不修补 diff。如果报告说"修复这个"，下一个构建者轮次做修复；评审者回到评审。混用角色击败了差距。

### 评审评分量表 vs 验证门控

验证门控检查确定性事实：验收是否运行、规则是否通过、范围是否保持。评审者做定性判断：这是正确的工作吗、有文档吗、交接可用吗。两者都是必需的。

## 构建

`code/main.py` 实现：

- 一个 `ReviewerInputs` 数据类，捆绑评审者读取的制品。
- 一个评分量表评分器，每个维度一个函数。每个函数对本课是确定性和桩级的；真实实现会调用 LLM。
- 一个 `review_report.json` 写入器，带有五个分数、总分和裁决（`pass`、`soft_fail`、`hard_fail`）。
- 两个演示案例：干净的变更和"正确的测试，错误的问题"变更。

运行：

```bash
python3 code/main.py
```

输出：写入磁盘的两份评审报告和维度分数的控制台表格。

## 生产中的模式

真实数据：Cloudflare 2026 年 4 月的 AI Code Review 系统在 30 天内对 5,169 个仓库的 48,095 个合并请求运行了 131,246 次评审。中位评审在 3 分 39 秒内完成。多达七个专业评审者（安全、性能、代码质量、文档、发布管理、合规、工程 Codex）在评审协调器下并行运行，进行查重发现和严重性判断。顶级模型专门保留给协调器；专业评审者运行在更便宜的层上。

四种模式使这一规模化运行成为可能。

**专业评审者池，而非一个大评审者。** 一个带 5 维度评分量表的评审者适用于单人仓库。一旦代码库有安全关键、性能关键和文档界面，拆分为带更小提示词的专业评审者。协调器做查重；专业评审者从不跑完整评分量表。模型层级分离随之而来：便宜的专业评审者，昂贵的协调器。

**偏差缓解作为设计需求，而非优化。** LLM 评判者表现出四种可靠偏差（Adnan Masood，2026 年 4 月）：位置偏差（GPT-4 在 (A,B) vs (B,A) 排序上约 40% 不一致）、冗长偏差（对更长输出约 15% 分数膨胀）、自我偏好（评判者偏好来自同一模型家族的输出）、权威性（评判者过度评价对已知作者的引用）。缓解：评估两种排序并只计算一致获胜；使用显式奖励简洁性的 1-4 量表；跨模型家族轮换评判者；评分前去除作者名称。

**校准集，而非靠感觉。** 一个 10-20 个任务的历史集合，带有已知正确裁决。每次提示变更时在其上运行评审者。如果与历史记录的一致性低于 80%，评分量表需要修订才能发布评审者。

**与门控的混合规范。** 验证门控处理确定性检查。评审者处理语义检查。Anthropic 2026 年的指导对此分割是明确的：不要让评审者重做门控已经证明的事。

## 使用

生产模式：

- **Claude Code 子 Agent。** 评审子 Agent 在构建者关闭任务后运行。它在 PR 上发布带评分量表分数的评论。
- **OpenAI Agents SDK 转交。** 构建者在任务完成时转交给评审者。评审者可以带回发现列表转交回，或向上升级给人类。
- **双模型配对。** 构建者运行在更快更便宜的模型上。评审者运行在更强的模型上，带更小的上下文，专注判断。

## 发布

`outputs/skill-reviewer-agent.md` 生成项目特定的评审评分量表、一个挂接到构建者制品的评审 Agent 桩、以及与验证门控的集成，使人类审查从一份书面报告开始而非空白页。

## 练习

1. 添加一个对你的产品领域特定的第六维度。论证为什么它不被现有五维度吸收。
2. 用两种不同的系统提示运行评审者（简洁、冗长）。哪个产生人类更可能阅读的报告？
3. 为每个维度添加 `confidence` 字段。当最低维度的置信度低于 0.6 时拒绝发出报告。
4. 构建一个校准集：10 个历史任务关闭的已知正确裁决。在其上运行评审者。它在何处与历史记录不一致？
5. 添加"请求更多证据"的能力：评审者可以在评分前要求构建者运行特定测试。什么是正确的回退以使其不形成循环？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 评审评分量表（Reviewer rubric） | "检查清单" | 五维度 0-2 评分，每个维度有书面问题 |
| 软失败（Soft fail） | "需要修订" | 总分低于 7；构建者收到待解决发现 |
| 硬失败（Hard fail） | "拒绝" | 总分低于 5 或任意维度 0 分；停止并呈现给人类 |
| 角色分离（Role separation） | "不同提示" | 相同模型可以扮演两个角色；纪律在于输入和姿态 |
| 置信度下限（Confidence floor） | "不发出低信号报告" | 当评分量表不确定时拒绝发出裁决 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 16 高级 Agent 工程中可靠性工程的顶点——将"质量保证"从确定性检查（验证门控）提升到语义判断（评审 Agent）。核心洞察简洁而深刻：**写代码的 Agent 不能评分。** 不是因为它不聪明——是因为它的姿态是"生成"，评审者的姿态是"质疑"。两个姿态需要一个显式的角色分离。Cloudflare 的 131K 评审运行数据将这节课从理论锚定到工业实践。适合已理解验证门控、正在建立 Agent 质量体系的架构师。

### 二、知识结构梳理

**评分量表层：** 五维度评审——问题匹配度、范围纪律、假设、验证质量、交接就绪度——覆盖了"对不对""超没超""说没说明""测没测准""能不能交接"五个质量面。每个维度 0-2 分，总分 0-10。

**工程模式层：** 三个关键设计——评审者不能编辑 diff（角色分离的强制）、评分量表 vs 验证门控的分工（确定性 vs 语义）、校准集（防止评分量表漂移）。

**生产规模化层：** Cloudflare 的四模式——专业评审者池（7 个专家 + 协调器）、偏差缓解（位置/冗长/自我偏好/权威性）、校准集、混合规范——将单评审者模式升级为企业级。

### 三、核心洞察（备课时的关键理解）

1. **"同一个模型，不同的系统提示"——角色分离的本质不是模型差异，是姿态差异。** 构建者的提示是"解决这个问题。"评审者的提示是"告诉我这个问题为什么没有被真正解决。"同一个 Claude Sonnet 可以扮演两个角色——只要系统提示不同、输入不同、评审者无写权限。这不是模型能力的问题——是角色设计的问题。

2. **"评审者不能编辑 diff"是防止角色坍缩的强制规则。** 如果评审者能改 diff——它就不再是评审者，而是第二轮构建者。两个角色合二为一——你回到了 Self-Refine（第 02 课）的单 Agent 自评审。强制只读——评审者只写报告。

3. **四种 LLM 评判者偏差（位置、冗长、自我偏好、权威）是评审 Agent 的隐藏杀手。** 位置偏差：答案排第一被评分更高。冗长偏差：更长的输出被评更高——即使冗余。自我偏好：GPT 评判者偏好 GPT 生成的输出。权威性：引用知名作者被加分。如果在评分量表设计时不缓解——你的评审分数反映了风格而非质量。

4. **Cloudflare 的 7 专业评审者 + 协调器架构证明了"评审不是一个 Agent——是一个系统。"** 安全、性能、代码质量、文档、发布管理、合规、工程 Codex——七个不同的视角。协调器做查重（两个评审者标记了同一段代码的问题合并为一个发现）和严重性判断。便宜模型做专业评审，顶级模型做协调——这是成本优化的规模化评审。

5. **校准集是所有产品化评审系统的必要条件。** 10-20 个历史任务，每次改评分量表后跑一致性。低于 80%→评分量表有问题。没有校准集——你不知道"评分量表变了"和"Agent 变差了"之间的区别。校准集是评审系统自身的评估。

6. **"混合规范"是 Anthropic 2026 年的明确建议——不要让评审者重做门控的工作。** 门控检查验收运行了没、规则通过了没。评审者问"这解决了正确的问题吗？"两个层次、不同对象、不重叠。重叠 = 浪费令牌，或更糟——两个层级给出不同答案时混乱。

7. **"请求更多证据"是防止评审者做出不完全知情判断的能力。** 评审者看 diff——不确定一个假设是否合理。它可以要求构建者跑一个特定测试。但要带退避——两次请求后评审者必须做出有或没有额外证据的裁决。不设退避——评审变成无限问询。

### 四、教学建议

1. **用"自己写的代码自己看"的类比开场。** "你刚写完一个复杂函数——你觉得它没问题。你同事看了，指出一个边界条件错误。你的代码有问题但你看不出来。Agent 也一样。"这是评审者独立性的直观论证。

2. **五维度评分量表用"五个问题"来讲——不要背维度名。** "解决了正确的问题吗？没超范围？假设写下来了吗？测试真的证明了吗？下一个人能接上吗？"五个问题 = 五维度。

3. **"评审者不能编辑 diff"——让学生设计一个违反这个规则的场景。** "Agent 评审发现了一个 bug，顺手改了——报告说'顺便修好了。'"然后问"为什么这个有问题？"因为评审者的改从来没被独立评审过。

4. **四种偏差用一个实验来展示。** 给同一个 LLM 评判者两份答案——同样质量，一个更长。评分是否有差异？再给两份——同样质量，一份来自 GPT 一份来自 Claude。GPT 评判者是否偏好 GPT？让学生亲眼看到偏差。

5. **Cloudflare 的 131K 运行数据是一个锚点。** "你刚写的评审 Agent 代码——规模化到 5 千个仓库、5 万个 MR、13 万次运行——需要什么改动？"答案就在文档的四模式中。

6. **校准集的建立是练习 4，也是最应该做的练习。** "找 10 个历史任务的关闭记录。你的评审 Agent 对它们评几分？和历史裁决一致吗？不一致——是历史错了还是评审评分量表错了？"这是评审系统自身的质量控制。

7. **"请求更多证据"加退避——练习 5 是防止评审者和构建者之间无限循环的练习。** "评审者说'我不确定'→构建者跑测试→评审者再说'还是不确定'→构建者再跑→"两轮退避规则——实用且有教学价值。

### 五、值得补充的内容

1. **评审者的"过度批评"和"过度宽容"需要在评分量表偏差中补充。** 四种已知偏差是风格偏差。但评审者还有方向偏差——总是太严（每个变更都 soft_fail）或总是太宽（每个变更都 pass）。需要方向偏差的检测和校准。

2. **评审者和构建者的反馈循环——评审发现如何被纳入构建者的下一次运行。** 软失败→构建者读报告→修复→重新运行验证门控→重新评审。这是一个额外的循环，成本在令牌和延迟上。

3. **多评审者之间的冲突——当安全评审者说"通过"而性能评审者说"失败"时。** Cloudflare 的协调器做严重性判断和去重——但没说怎么解决两个专业评审者之间的分歧。需要一个冲突解决协议。

4. **评审评分量表的"领域适应性"——通用五维度在特定领域的有效性。** 在安全领域，五维度够吗？在金融领域，"监管合规"维度是否需要被显式添加？通用评分量表 vs 领域评分量表。

5. **评审 Agent 与 Phase 15·11（评估驱动开发）的关系。** 评审 Agent 是另一种形式的评估——评审运行在每次任务完成后，评估套件运行在每次 PR 上。两者互补——评审是定性的、评估是定量的。

### 六、一句话总结

**评审 Agent 给你的不是"让 Agent 再检查一遍"——是"让一个和构建者有不同的系统提示、不同的输入、对 diff 只有只读权的 Agent 来问构建者不会问自己的问题。"构建者的姿态是完成，评审者的姿态是质疑。把两个姿态放在不同的角色中——不是同一个 Agent 的两个提示——是 Agent 可靠性的结构基础。**

# 🎓 Agent 架构课：评审者——为什么同一张嘴不能既说"我写完了"又说"它是对的"

同学们好。我是你们的 Agent 架构老师。

前面六节课我讲了长程趋势、监管者模式、持久化执行、成本调控、人类在环、多 Agent 辩论。这些课都在不同层面回答"怎么让 Agent 更可靠。"今天这节课是可靠性工程的最后一环——也是最容易被跳过的。

**你让 Agent 修了一个 bug。Agent 修了，跑了测试，说"做完了。"你信了。两天后发现它修的是 bug 的错误一半。测试是通过了——测试在测一个更弱的条件。Agent 完成了你让它做的事——但不是你真正需要它做的事。**

验证门控（第 15 课）告诉你"验收运行了、范围保持了、规则通过了。"评审者问的是门控不能问的问题：**"这真的解决了问题吗？"**

## 为什么同一张嘴不能既说"做完了"又说"是对的"

给你一个类比。你写完了一个复杂函数。你觉得它没问题——你就是写出它的人。你的同事走过来，读了你的代码，指出一个你漏掉的边界条件。

你漏掉它的原因不是你不够聪明——是因为**你创建它时的思维模式和检查它时需要的思维模式不同。** 创建是"我要让它工作。"检查是"我要让它在所有不工作的情况下也能被捕获。"

Agent 也一样。构建者的系统提示是"完成任务。"评审者的系统提示是"找出任务没有被真正完成的证据。"同一个模型可以扮演两个角色——但需要不同的系统提示、不同的输入、且评审者不能碰 diff。

## 五个问题——不是五个维度

评审者的评分量表看起来是五个维度——但我叫它们五个问题。因为"维度"听起来像你必须背的东西。"问题"是你在读评审报告时问的东西。

**问题一：它解决了正确的问题吗？** Agent 修了登录按钮的样式——但 bug 报告说的是登录按钮不工作。样式不等于功能。0 分。

**问题二：它在范围内吗？** Agent 修登录按钮时顺便重构了整个认证模块。"顺便"不是范围。1 分——确实需要一些重构，但应该标记出来而不是默默做。

**问题三：假设写下来了吗？** Agent 假设用户数据库使用 bcrypt。它没写下来。下一个 Agent 接手时不知道这个假设，可能用 SHA-256 替代 bcrypt。0 分。

**问题四：测试真的证明了吗？** 测试通过了登录按钮的 UI 渲染——但没有测试实际登录逻辑。验收命令运行了，但它证明的是一个更弱的目标。1 分。

**问题五：下一个人能接上吗？** diff 清晰、假设文档化了、状态干净。但交接文档缺少"为什么选择 bcrypt 而不是 Argon2id"的理由。1 分。

总分 3/10。硬失败。不是 Agent 没做事——是它做的是一个和实际需求有系统性差距的事。这种差距只有评审者能发现。

## 四种偏差——你的评审者可能在骗你

LLM 评判者有四种已知的、可靠的偏差。你不认识它们——你的评审分数反映的是风格不是质量。

**位置偏差。** 答案 A 和答案 B 一样好。但 A 排在 B 前面。GPT-4 在 40% 的情况下对 (A,B) 和 (B,A) 给出不同分数。交换顺序后得分不一致→评判不可靠。

**冗长偏差。** 长回答被评更高分——即使长出来的部分是冗余。评审者说"这篇写得好"——因为它是 10 段而不是 3 段。实际上 3 段说了同样的事。

**自我偏好。** GPT 评判者偏好 GPT 生成的输出。Claude 评判者偏好 Claude 的输出。不是故意的——但检测得到。

**权威性偏差。** "正如 Knuth 在《计算机程序设计艺术》中所说"——引用了知名作者，评审者给更高分，即使引用和论点无关。

缓解不是"不用 LLM 做评判者"——是**设计评分量表来抵消偏差。** 评估两种排序并只保留一致赢的。用 1-4 量表并显式说"简洁是加分项，冗长是减分项。"跨模型家族轮换评判者。评分前去掉作者名。

## Cloudflare 做了 131,246 次评审，30 天

我想给你一个数字来锚定这节课。Cloudflare 在 2026 年 4 月公布了他们的 AI Code Review 系统。30 天内——5,169 个仓库、48,095 个合并请求、131,246 次评审。中位评审 3 分 39 秒。

不是一个评审者。七个专业评审者：安全、性能、代码质量、文档、发布管理、合规、工程 Codex。一个协调器做去重和严重性判断。

顶层模型只用在协调器上——因为它做最难的判断（"这个发现是 blocker 还是 warning？"）。专业评审者用更便宜的模型——因为它们有更窄的任务和更小的提示词。

你的评审 Agent 不需要从七个专家开始。从五个问题的一个评审者开始——然后在特定领域上失控时拆分出专家。

## 结课清单

添加评审 Agent：

1. 评审者是独立角色——不同的系统提示、不同的输入、对 diff 只读。
2. 五个问题 = 五个维度。总分 10。低于 7 = 软失败。低于 5 = 硬失败。
3. 评审者不碰 diff——只写报告。混用角色 = 击败了差距。
4. 评审评分量表 ≠ 验证门控。门控是确定性事实，评审是语义判断。两个都需要。
5. 四种偏差必须缓解——位置、冗长、自我偏好、权威性。
6. 校准集——10-20 个历史任务，每次改评分量表后跑一致性。
7. 从五个问题的一个评审者起步——需要时拆分出专家。

**最后一句话：**

**Agent 可靠性最被低估的基础设施不是一个检查、不是一个门控——是"构建"和"评审"之间的那道墙。墙的这边是完成任务的姿态，墙的那边是质疑完成姿态的姿态。用不同的系统提示、不同的输入、对 dif 只有只读权来建这道墙——不是"让 Agent 再自检一次"，是"让另一个角色来做你的 Agent 不愿意对自己做的事。"**
