# 生产环境扩展 — 队列、检查点、持久化

> 将多 Agent 系统扩展到数千个并发运行需要**持久执行**。LangGraph 的运行时在每个超级步（super-step）后写入一个检查点，用 `thread_id` 作为键（默认使用 Postgres）；Worker 崩溃会释放租约，另一个 Worker 继续执行。Agent 可以无限期等待人类输入。**MegaAgent**（arXiv:2408.09955）运行了每个 Agent 一个的生产者-消费者队列，具有三种状态（Idle / Processing / Response）和两层协调（组内聊天 + 组间管理聊天）。**纤程/异步** 在 LLM 流式处理中优于每任务一线程：线程 99% 时间空闲等待 Token，而纤程在 I/O 上协作式让出。反面观点：Ashpreet Bedi 的 "Scaling Agentic Software" 主张**FastAPI + Postgres + 其他什么都不加**直到负载证明不够 — 简单架构比预期走得更远。本课构建持久检查点日志、带状态转换的每 Agent 工作队列、异步 vs 线程对比演示，并总结务实的"从简单开始"规则。

**类型：** 学习 + 构建
**语言：** Python（标准库、`asyncio`、`sqlite3`）
**前置课程：** Phase 16 · 09（并行 Swarm 网络）、Phase 16 · 13（共享内存）
**时间：** 约 75 分钟

## 问题

一个多 Agent 原型在一台笔记本上用三个 Agent 在内存事件循环中工作正常。你迁移到生产环境：

- Agent 有时运行数小时（长时间研究、等待人类介入）。
- Worker 进程崩溃。重启后状态丢失。
- 峰值负载是平均的 10 倍；需要水平扩展。
- 用户按 Agent 运行次数付费；需要精确一次计费语义。

内存事件循环做不到这些。你需要在下面放一个持久执行层。2026 年的标准选项有：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每 Agent 生产者-消费者）。
4. 手写 FastAPI + Postgres（Bedi 的观点）。

本课构建每种方案的一个微型版本。

## 概念

### 持久执行模式

持久执行引擎在每个"步"（LangGraph 术语中的超级步）后持久化完整的程序状态。崩溃时：

```
worker 在步骤中崩溃
  -> 租约超时
  -> 另一个 worker 接管 thread_id
  -> 从最后一个检查点恢复
  -> 无重复副作用
```

这需要：

- **可序列化状态。** 所有 Agent 状态必须是可持久化的。含有活跃数据库连接的闭包无法存活。
- **确定性恢复。** 给定相同状态和相同输入，Agent 产生相同的动作（或依赖于外部确定性的 LLM 调用预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的，或使用去重键。

LangGraph 在每个超级步后写入检查点；Temporal 在每个 Activity 后写入；Restate 使用事件溯源日志。三者实现同一模式。

### LangGraph 的运行时

每个 Agent 有一个 `thread_id`；状态是类型化字典；每个超级步向检查点表写入一行。恢复时，运行时从最后一个检查点重放，而不是从头开始。Agent 可以 `interrupt()` 等待人类输入；运行时持久化并释放 Worker。当输入到达时，任何 Worker 都可以恢复。

这是 2026 年 4 月参考级的生产设计。

### MegaAgent 的每 Agent 队列

arXiv:2408.09955 描述了一个大规模实验：数千个并发 Agent 在一个集群中。架构：

```
agent i:
  state ∈ {Idle, Processing, Response}
  in_queue   <- 发送给 agent i 的消息
  out_queue  -> 回复 + 副作用

协调器：
  组内聊天（同一组的 agents）
  组间管理聊天（高层路由）
```

两层协调使组内对话可以密集进行，而组间保持稀疏 — 这是保持数千个 Agent 成本线性的模式。

### 异步 vs 每任务一线程

LLM 调用是 I/O 密集型的。等待下一个 Token 的线程 99% 时间空闲。线程每个占用约 1MB RAM；10,000 个并发调用仅栈就占 10GB。

纤程（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O 上协作式让出。同样的 10,000 个调用在进程中轻松容纳。在 LLM-Agent 规模上，异步不是优化 — 它是架构。

例外：CPU 密集型后处理（嵌入、分词技巧）仍需要线程或进程。将 I/O 层与 CPU 层分离。

### Bedi 的反面观点

"Scaling Agentic Software"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就过度工程化了。务实的默认方案：

- FastAPI + Postgres。
- 每个 Agent 运行是一行；状态用乐观并发原地更新。
- 后台任务通过 `pg_notify` 或简单的 Celery Worker。
- 重试策略在应用代码中定义。

对于约 100 个并发 Agent 运行以下的负载，这通常就足够了。当测量到它失败时再升级。

规则：当你遇到简单架构无法解决的具体问题时，再采用持久执行框架。过早采用会在仪式感上消耗时间而没有回报。

### 精确一次语义

对于付费的 Agent 运行，你需要"有效精确一次"（至少一次投递 + 幂等消费者）。工程手段：

- **每次运行的唯一去重键。** 包含在每个副作用调用中。
- **发件箱模式。** 副作用先写入表，然后一个独立进程执行它们。两步都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，调度补偿。

这些都是数据库工程模式，不是 LLM 特有的。LLM 的额外成本只是 LLM 调用较慢；其他都是标准分布式系统。

### 彩虹部署

Anthropic 的 Multi-agent 研究系统使用"彩虹部署"：多个版本的 Agent 运行时并发运行，所以长时间运行的 Agent 不必在每次代码部署时被杀死。对一部分流量金丝雀部署新版本；当旧版本上的 Agent 完成后再退役旧版本。

这是长时间运行的有状态系统的标准做法；2026 年的适配是 Agent 可能存活数小时，因此部署周期必须适应。

### 标准生产检查清单

- 持久状态（检查点、快照，或发件箱 + 可重放日志）。
- 幂等副作用。
- LLM 调用的异步 I/O 层。
- 至少一次投递加去重。
- 有状态工作负载的彩虹/金丝雀部署。
- 可观测性：每 Agent 链路追踪、超级步审计、重试计数器。

## 上线加固

标准的生产加固：

- **从简单开始（Bedi 规则）。** FastAPI + Postgres，直到测量到它失败。
- **在优化之前先对一切进行仪表化。** 每运行延迟直方图、每步耗时、重试次数、失败分类。
- **副作用的发件箱模式。** 特别是支付和外部 API 调用。
- **彩虹部署。** 永远不要在部署期间杀死正在运行的 Agent。
- **当遇到以下具体问题时再采用持久执行引擎（Temporal / LangGraph / Restate）：** 长达数小时的人类等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层用异步。** 线程仅用于 CPU 密集型后处理。

## 练习

1. 运行 `code/main.py`。确认检查点恢复正常工作；测量异步 vs 线程并发差异。
2. 实现一个**发件箱**表：每个工具调用先写入发件箱，然后一个独立的 goroutine/task 执行。通过运行工具调用两次来验证幂等性。
3. 模拟**彩虹部署**：两个并发运行时版本；将新 thread_id 的一半路由到每个版本；确认旧版本上的进行中线程没有被中断。
4. 阅读 LangGraph 的运行时文档（链接见下）。找出运行时哪些功能在手写 FastAPI + Postgres 版本中最难复制。这是采用的理由，还是可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理聊天）是显式的。画草图说明如何用两个队列族将其映射到消息队列。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 持久执行 | "持久化程序状态" | 引擎在每个超级步后写入状态；崩溃恢复是确定性的。 |
| 超级步 | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "Agent 运行标识符" | 绑定检查点和恢复逻辑的键。 |
| 幂等性 | "重试安全" | 重复一个副作用产生与一次尝试相同的结果。 |
| 发件箱模式 | "解耦副作用" | 将意图写入表；独立执行器执行并标记完成。 |

| 检查点 | "状态快照" | 超级步边界上的序列化状态；恢复的关键。 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇结构清晰、实用性极强的生产环境扩展指南。文档的特点是"多方案对比 + 动手验证 + 务实的工程判断"，不是教科书式的单向输出，而是提供了 Temporal、LangGraph Runtime、MegaAgent 和 FastAPI+Postgres 四种方案的微型实现，让学员自己对比。Bedi 的"从简单开始"反观点是亮点，避免了学员陷入"必须上 Temporal"的教条主义。不足在于 SQLite 检查点日志与真实 Postgres 生产方案差距较大，彩虹部署的代码模拟也偏简单。

### 二、知识结构梳理

- **问题层**：内存事件循环的局限性 — 无崩溃恢复、无水平扩展、无精确一次语义。引出四种 2026 年标准方案。
- **模式层**：持久执行的三个必要条件（可序列化状态、确定性恢复、幂等副作用），LangGraph 超级步检查点机制，MegaAgent 的两层协调队列，异步纤程 vs 线程的取舍。
- **实践层**：Bedie 的"先简单再升级"规则，发件箱模式，彩虹部署，标准生产检查清单。从"过度工程化"到"按需升级"的工程哲学贯穿始终。

### 三、核心洞察

1. **持久执行不是新技术，是分布式系统基本功。** LLM 调用只是"慢"而已，其余的模式（检查点、发件箱、补偿事务）都是经典分布式系统手段，不需要重新发明。
2. **异步是 LLM Agent 的架构基础，不是优化。** 线程在 LLM 场景下的空闲率高达 99%，纤程/协程从内存和调度开销上碾压线程。这不是"可选优化"，是架构设计的基本约束。
3. **Bedi 的反观点是务实的清醒剂。** FastAPI + Postgres 能覆盖 100 并发以下的场景，过早引入 Temporal 只增加运维负担。这个"先测量再升级"的原则应该是所有 Agent 系统的默认立场。
4. **MegaAgent 的两层协调（组内密集 + 组间稀疏）是成本线性扩展的关键。** 如果所有 Agent 在同一个通信平面上全连接，消息量是 O(n²)；分层后 O(n)。
5. **彩虹部署是 Agent 特化的部署策略。** 常规微服务可以瞬间替换，但 Agent 运行可能持续数小时，必须有版本共存机制。
6. **发件箱模式是精确一次语义的兜底方案。** 写入意图和实际执行分离，配合去重键，在任一层失败时都能保证最终一致性。
7. **"检查点是否确定性的"这个问题在 LLM 场景下要打问号。** 同一个状态 + 同一个输入，LLM 的输出可能不同（temperature > 0），真正的确定性恢复可能需要外部确定性预言机。

### 四、教学建议

1. **先让学员体验"崩溃"的痛。** 跑一个没有检查点的 Agent，手动 kill 进程，然后恢复不了。再引入检查点，效果对比鲜明。
2. **异步 vs 线程的对比演示是关键动手环节。** 500 个并发调用的规模刚好能展示差异而不至于太复杂。建议让学员调整并发数看拐点。
3. **发件箱模式的动手实现要强调去重键的重要性。** 让学员先实现没有去重键的版本，再触发重复执行，观察双扣费问题。
4. **用"餐馆后厨"类比解释超级步和检查点。** 厨师每完成一道菜（超级步）就在单子（检查点）上打勾；换了厨师（Worker 切换）新厨师能从最后一个勾继续。
5. **Bedi 规则需要一个"什么时候升级"的判断表。** 建议做一个决策矩阵：并发数 < 100 → FastAPI+PG；100-1000 → LangGraph Runtime；> 1000 → Temporal + 队列。虽然简化但实用。
6. **强调"先仪表化再优化"。** 很多团队跳过可观测性直接上 Temporal，钱花了但不知道改善了哪里。延迟直方图和重试计数器应该在第一版就有了。
7. **彩虹部署的练习值得升级为小组讨论。** 实际场景中涉及版本兼容协议和状态迁移，代码模拟过于简化，需要补充理论。

### 五、值得补充的内容

1. **Temporal 与 LangGraph Runtime 的深度对比。** 文档只提了"都实现同一模式"，但 Temporal 的 Saga 模式、Activity 重试策略和 Workflow 超时机制与 LangGraph 的超级步模型有显著差异，值得深入。
2. **水平扩展中的状态分片策略。** 当 Agent 数量超过单实例容量时，如何按 thread_id 分片到不同 Postgres 实例或 Worker 池。
3. **Kubernetes 上的 Agent 自动扩缩容实战。** HPA 基于哪些指标（队列深度、Agent 计数、延迟百分位）最有效。
4. **成本模型。** 检查点写入的存储成本、Temporal 服务器的运维成本、异步 vs 线程在云环境中的实例成本 — 这些数字对架构决策很关键。

### 六、一句话总结

**生产环境多 Agent 系统的本质是分布式系统基本功加上 LLM 特有的慢速 I/O：先简单后复杂，先测量后优化，持久执行是兜底不是默认。**

# 🎓 Agent 架构课：生产环境扩展——你那台笔记本上的 Agent 不是产品

同学们好。我是你们的FDE工程老师，今天讲的是 Agent 生产化。

**3 个 Agent 在笔记本内存事件循环里丝滑流畅。500 个 Agent 需要持久检查点、工作队列、精确一次语义。** 你的 demo 不是产品。

生产环境需要什么？① **持久检查点**——Worker 崩溃后另一个 Worker 从检查点继续。LangGraph 用 Postgres 每超级步写一次。② **工作队列**——Agent 在队列里取任务，处理完放回。MegaAgent 用生产者-消费者队列 + 三种状态（Idle/Processing/Response）。③ **纤程/异步优于每任务一线程**——线程 99% 时间空闲等 token，纤程在 I/O 上协作让出。

反面观点：Ashpreet Bedi 主张 **FastAPI + Postgres + 别的什么都不加**——简单架构走得比预期远。直到负载证明不够才加复杂度。

## 结课清单

1. **内存事件循环 ≠ 生产环境。** 崩溃 = 状态丢失。
2. **持久检查点是底线。** Postgres 一个表就够了。
3. **纤程 > 线程用于 LLM 工作负载。** 等 token 的时候让出 CPU。
4. **从简单开始。** FastAPI + Postgres 先跑起来，负载证明不够再加。

最后一句话：**在你笔记本上跑通的 3 个 Agent 是 demo。在生产环境跑 500 个 Agent 是另一门学科。先装检查点，再谈 scale。**

# 💼 从业者故事：生产环境扩展——你那台笔记本上的 Agent 不是产品

我有一个非常尴尬的经历。去年在做技术分享的时候，我在台上 live demo 我们的多 Agent 系统，跑了 3 个 Agent 协同工作，丝滑流畅，掌声雷动。然后一个观众问："你们生产环境能同时跑多少个？"我说"理论上几千个都没问题"。他说"那你现在跑 500 个试试"。

我跑了。电脑风扇起飞了。第 312 个 Agent 开始 OOM，第 478 个超时，第 500 个直接没起来。整个 demo 以我开始删除进程、关闭 Chrome 标签页来释放内存的尴尬场面收场。从那以后我再也不在笔记本上吹牛了。

那天的教训是：**能跑通和能跑量之间隔着一个生产环境。** 笔记本上的内存事件循环跑 3 个 Agent 美滋滋，跑 500 个就是灾难。你需要的不只是"更多的机器"，而是整个持久执行层的重新设计。

## 崩溃不可怕，崩溃以后状态没了才可怕

2026 年 3 月我们一个 Worker 进程在凌晨 3 点挂掉了。重启之后，所有 Agent 的状态都丢了——用户在等的回复、进行到一半的推理链、17 分钟的对话上下文——全没了。客服收到了一批"为什么我的 Agent 突然不理我了"的投诉。

这就是为什么检查点不是"nice to have"，是"have it or die"。LangGraph 在每个超级步之后写入检查点的设计，本质上跟打游戏存档是一样的——不是每次走一步都存盘，但关键节点之后必须存。Worker 崩溃只是换了个机器继续跑，用户根本不知道底下死了一个进程。

我现在的硬性要求：任何 Agent 运行超过 30 秒的，必须有检查点。超过 5 分钟的，必须有可恢复的检查点加幂等副作用。别跟我扯"我们不会崩溃"——你们会的，只是还没遇到负载。

## 异步不是优化，是底线

讲一个特别简单的数学题：一个线程在等待 LLM 返回 Token 的时候，99% 的时间在干什么？答案是：在睡觉。因为 LLM 调用平均 1-2 秒的延迟，每个 Token 之间还要等几毫秒，你的线程就像一个在麦当劳等汉堡的人——大部分时间在划手机。

10,000 个并发 LLM 调用用线程：每个线程 1MB 栈，光栈就 10GB，还不算上下文切换的开销。同样 10,000 个调用用 asyncio 纤程：一个进程轻松吃下，因为纤程在 I/O 等待的时候主动让出 CPU，不占栈空间。

这不是优化，这是架构。如果你在生产环境用线程做 LLM 调用，你就是在烧钱换简单。Goroutine（Go）和 tokio（Rust）同理——异步是 LLM Agent 系统的默认姿势，不是加分项。唯一的例外是 CPU 密集型后处理（比如本地嵌入计算），那个才需要线程或独立进程池。

## Bedi 的反面观点：先简单，后复杂

我第一次看到 Ashpreet Bedi 那篇文章的时候心里不太舒服，因为我刚刚花了三周搭 Temporal 工作流引擎。然后他说："大多数团队在测量负载之前就过度工程化了。FastAPI + Postgres 就够了。"

我冷静下来想了想，他说得对。如果你的并发 Agent 数不到 100，FastAPI + Postgres 加一个 Celery Worker 就够用了。每个 Agent 运行是数据库里的一行，状态用乐观并发更新，重试在应用代码里写。简单，粗暴，能跑。

但问题是——你怎么知道你不会超过 100？答案是：先测量。上线第一周的压力测试和真实流量数据就是你决策的依据，不是直觉。我们有个客户坚持"先上 Temporal 准没错"，结果运维成本比业务增长更快，最后还是退回了 FastAPI + Postgres。

我的判断规则：并发 < 100 → FastAPI + PG；100-1000 → LangGraph Runtime；> 1000 → Temporal + 消息队列。这个划分粗糙但有指导意义。关键是"先测量再升级"，不是"先升级再后悔"。

## 发件箱模式：最无聊也最重要的模式

Agent 做一个工具调用，比如支付。你发请求给支付 API，然后想把结果写回数据库。但支付 API 返回成功，你的数据库写入却失败了。用户扣了钱，你的系统不知道。这就是经典的"副作用不一致"问题。

发件箱模式的解法特别朴实：别直接干活，先把"我打算干什么"写到一个专门的表里，然后另一个独立进程去执行。每次执行前检查去重键，确保不会重复扣款。两步都是幂等的——失败了重来，重复了去重。

这个模式跟 LLM 一点关系都没有，纯粹是数据库工程。但 2026 年做 Agent 系统，你 80% 的时间不是在调 Prompt，而是在做这些"无聊的"分布式系统基础设施。如果有人告诉你 Agent 系统只是"调 API 就行"，他肯定没上过生产。

## 彩虹部署：别在部署的时候杀人

Anthropic 的多 Agent 研究系统有一个细节特别打动我：他们用彩虹部署，就是新旧版本的运行时同时跑，长时间运行的 Agent 不会因为代码部署被强制终止。

你想想，一个用户花了 30 分钟跟 Agent 协作调试一个 Bug，你一个 `kubectl apply` 把它杀了。用户再打开的时候状态全没了。这不是技术问题，是产品体验问题。Agent 系统的部署周期必须适应 Agent 的生命周期，而不是反过来。

我们现在的方案：滚动更新时保留旧版本的 Pod 直到它上面的 Agent 运行全部完成或超时（48 小时硬上限）。新请求走新版本，旧请求在旧版本上自然结束。简单，但比不做强多了。

**生产环境的 Agent 系统是 80% 的分布式系统基本功加 20% 的 LLM 特有慢速 I/O。不要被 AI 的炫酷迷了眼，先把检查点、发件箱、幂等性、异步这四个地基打牢。打牢了，你怎么浪都稳。**
