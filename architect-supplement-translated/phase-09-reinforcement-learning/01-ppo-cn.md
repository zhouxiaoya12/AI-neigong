# 近端策略优化（Proximal Policy Optimization, PPO）

> A2C 每次 rollout 后用一次就丢掉。PPO 用剪切的重要性比率包裹策略梯度，让你能在同一批数据上跑 10+ 轮更新而策略不会爆炸。Schulman 等人（2017）。到2026年仍是默认策略梯度算法。

**类型：** 构建
**语言：** Python
**前置要求：** Phase 9 · 06（REINFORCE）、Phase 9 · 07（Actor-Critic）
**时间：** 约75分钟

## 问题所在

A2C（第7课）是在策略的（on-policy）：梯度 `E_{π_θ}[A · ∇ log π_θ]` 要求数据从*当前* `π_θ` 采样。做一次更新，`π_θ` 就变了；你使用的数据现在变成离策略了。重用它，你的梯度就有偏。

Rollout 很昂贵。在 Atari 上，一次 8 环境 × 128 步的 rollout = 1024 个转移，耗费十几秒环境时间。在一个梯度步骤后就扔掉是浪费。

置信域策略优化（TRPO, Schulman 2015）是第一个修复：约束每次更新使新旧策略之间的 KL 散度保持在 `δ` 以下。理论上干净，但每次更新需要一次共轭梯度求解。2026年没人用 TRPO。

PPO（Schulman et al. 2017）用简单的剪切目标替换硬置信域约束。多一行代码。每次 rollout 十轮。没有共轭梯度。足够好的理论保证。九年后它仍是默认策略梯度算法——从 MuJoCo 到 RLHF 皆然。

## 核心概念

![PPO 剪切代理目标：比率剪切在 1 ± ε](../assets/ppo.svg)

**重要性比率。**

`r_t(θ) = π_θ(a_t | s_t) / π_{θ_old}(a_t | s_t)`

这是新策略 vs 采集数据的策略之间的似然比。`r_t = 1` 表示没有变化。`r_t = 2` 表示新策略采取 `a_t` 的可能性是旧策略的两倍。

**剪切代理目标。**

`L^{CLIP}(θ) = E_t [ min( r_t(θ) A_t, clip(r_t(θ), 1-ε, 1+ε) A_t ) ]`

两项：

- 如果优势 `A_t > 0` 且比率试图增长超过 `1 + ε`，剪切使梯度平坦——不要将一个好动作推到超过 `+ε` 于旧概率之上。
- 如果优势 `A_t < 0` 且比率试图增长超过 `1 - ε`（意味着我们会让一个坏动作变得更可能，相对于其剪切后的减少幅度），剪切封顶梯度——不要让一个坏动作超过 `-ε` 的下限。

`min` 处理另一个方向：如果比率已经朝*有利*方向移动，你仍然获得梯度（在可能对你不利的那个方向上不做剪切）。

典型的 `ε = 0.2`。将目标绘制为 `r_t` 的函数：一个分段线性函数，在"好的一面"有一个平坦的屋顶，在"坏的一面"有一个平坦的地板。

**完整的 PPO 损失。**

`L(θ, φ) = L^{CLIP}(θ) - c_v · (V_φ(s_t) - V_t^{target})² + c_e · H(π_θ(·|s_t))`

与 A2C 相同的 actor-critic 结构。三个系数，通常 `c_v = 0.5`、`c_e = 0.01`、`ε = 0.2`。

**训练循环。**

1. 在 N 个并行环境中收集 `N × T` 个转移，每个环境进行 `T` 步。
2. 计算优势（GAE），将其冻结为常数。
3. 将 `π_{θ_old}` 冻结为当前 `π_θ` 的快照。
4. 进行 `K` 轮训练，每轮对大小为64的 minibatch 处理 `(s, a, A, V_target, log π_old(a|s))`：
   - 计算 `r_t(θ) = exp(log π_θ(a|s) - log π_old(a|s))`。
   - 应用 `L^{CLIP}` + 值损失 + 熵。
   - 梯度步。
5. 丢弃该 rollout。返回步骤 1。

`K = 10` 和 64 大小的 minibatch 是标准超参集合。PPO 很鲁棒：确切数字在 ±50% 内很少重要。

**KL 惩罚变体。** 原始论文提出了使用自适应 KL 惩罚的替代方案：`L = L^{PG} - β · KL(π_θ || π_old)`，`β` 根据观测到的 KL 调整。剪切版本成为主导；KL 变体在 RLHF 中存活（其中相对于参考策略的 KL 是你总是想要的独立约束）。

## 动手构建

### 步骤 1：在 rollout 时捕获 `log π_old(a | s)`

```python
for step in range(T):
    probs = softmax(logits(theta, state_features(s)))
    a = sample(probs, rng)
    s_next, r, done = env.step(s, a)
    buffer.append({
        "s": s, "a": a, "r": r, "done": done,
        "v_old": value(w, state_features(s)),
        "log_pi_old": log(probs[a] + 1e-12),  # rollout 时记录旧策略的 log 概率
    })
    s = s_next
```

快照在 rollout 时拍摄一次。在更新轮次期间不变。

### 步骤 2：计算 GAE 优势（第7课）

与 A2C 相同。跨批次归一化。

### 步骤 3：剪切代理更新

```python
for _ in range(K_EPOCHS):
    for mb in minibatches(buffer, size=64):
        for rec in mb:
            x = state_features(rec["s"])
            probs = softmax(logits(theta, x))
            logp = log(probs[rec["a"]] + 1e-12)
            ratio = exp(logp - rec["log_pi_old"])
            adv = rec["advantage"]
            surrogate = min(
                ratio * adv,
                clamp(ratio, 1 - EPS, 1 + EPS) * adv,
            )
            # 反向传播 -surrogate，加值损失，减熵
            grad_logpi = onehot(rec["a"]) - probs
            if (adv > 0 and ratio >= 1 + EPS) or (adv < 0 and ratio <= 1 - EPS):
                pg_grad = 0.0  # 被剪切
            else:
                pg_grad = ratio * adv
            for i in range(N_ACTIONS):
                for j in range(N_FEAT):
                    theta[i][j] += LR * pg_grad * grad_logpi[i] * x[j]
```

"剪切 → 零梯度"模式是 PPO 的核心。如果新策略已经在有利方向上漂移太多，更新停止。

### 步骤 4：值和熵

在 critic 目标上添加标准 MSE，在 actor 上添加熵奖励，与 A2C 相同。

### 步骤 5：诊断

每次更新需要关注三个指标：

- **平均 KL** `E[log π_old - log π_θ]`。应保持在 `[0, 0.02]`。如果超过 `0.1`，减少 `K_EPOCHS` 或 `LR`。
- **剪切比例**——比率落在 `[1-ε, 1+ε]` 之外的样本比例。应在 `~0.1-0.3`。如果 `~0`，剪切从不触发 → 提高 `LR` 或 `K_EPOCHS`。如果 `~0.5+`，你在对 rollout 过拟合 → 降低它们。
- **可解释方差** `1 - Var(V_target - V_pred) / Var(V_target)`。Critic 的质量指标。应随 critic 学习向 1 攀升。

## 陷阱

- **剪切系数调错。** `ε = 0.2` 是事实标准。设到 `0.1` 会让更新过于胆小；`0.3+` 会引入不稳定。
- **太多轮次。** `K > 20` 通常会破坏稳定性，因为策略漂移远离 `π_old`。限制轮次，特别是对于大型网络。
- **无奖励归一化。** 大的奖励尺度会吞噬剪切范围。在计算优势之前归一化奖励（运行标准差）。
- **忘记优势归一化。** 每批次零均值/单位标准差归一化是标准做法。在大多数基准上跳过它会使 PPO 崩溃。
- **学习率不衰减。** PPO 受益于线性 LR 衰减到零。恒定 LR 通常更差。
- **重要性比率数学错误。** 始终使用 `exp(log_new - log_old)` 以确保数值稳定，而非 `new / old`。
- **梯度符号错误。** 最大化代理目标 = *最小化* `-L^{CLIP}`。翻转符号是最常见的 PPO bug。

## 实战使用

PPO 是2026年跨惊人数量领域的默认 RL 算法：

| 用例 | PPO 变体 |
|------|---------|
| MuJoCo / 机器人控制 | 带有高斯策略的 PPO，GAE(0.95) |
| Atari / 离散游戏 | 带有分类策略的 PPO，128步滚动 rollout |
| LLM 的 RLHF | 带 KL 惩罚到参考模型的 PPO，响应末尾的 RM 奖励 |
| 大规模游戏智能体 | IMPALA + PPO（AlphaStar, OpenAI Five） |
| 推理 LLM | GRPO（第12课）——没有 critic 的 PPO 变体 |
| 仅偏好数据 | DPO——PPO+KL 的闭合形式坍缩，无在线采样 |

PPO 的*损失形状*——剪切代理 + 值 + 熵——是 DPO、GRPO 和几乎所有 RLHF 管线的脚手架。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 重要性比率 | "r_t(θ)" | `π_θ(a\|s) / π_old(a\|s)`；相对采集数据的策略的偏离。 |
| 剪切代理 | "PPO 的主要技巧" | `min(r·A, clip(r, 1-ε, 1+ε)·A)`；有益侧超过剪切限的平坦梯度。 |
| 置信域 | "TRPO / PPO 的意图" | 限制每次更新的 KL，以保证单调改进。 |
| KL 惩罚 | "软置信域" | PPO 替代方案：`L - β · KL(π_θ \|\| π_old)`。自适应 `β`。 |
| 剪切比例 | "剪切触发频率" | 诊断——应在 0.1-0.3；超出意味着调参错。 |
| 多轮次训练 | "数据复用" | 每次 rollout 上 K 轮次；方差成本换取样本效率。 |
| 近似在策略 | "大部分在策略" | PPO 名义上在策略，但 K>1 轮次安全使用略微离策略的数据。 |
| PPO-KL | "另一个 PPO" | KL 惩罚变体；用于 RLHF，其中到参考的 KL 已是约束。 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇"工程直觉驱动"的 PPO 教材。它没有从贝尔曼方程或 MDP 定义开始，而是直接抛出"A2C 数据浪费"的痛点和"用剪切代替共轭梯度"的架构选择——这是一个一线 RL 工程师的视角，不是学术综述。特别是"陷阱"部分的7条精准踩中了所有初学者的典型翻车点。

### 二、知识结构梳理

- **基础层**：重要性比率的定义（exp(log_new - log_old)）、剪切代理目标的分段线性形状、GAE 优势估计
- **模式层**：PPO 的 data-reuse 循环（rollout→冻结→K轮更新→丢弃）、剪切 vs KL 惩罚两种置信域实现、clip fraction 作为超参健康指标
- **应用层**：PPO 作为 RLHF 的骨架（带 KL 到参考模型）、GRPO 作为无 critic PPO 变体、DPO 作为 PPO+KL 的闭合形式坍缩

### 三、核心洞察

1. **PPO 的核心贡献不是"更好的优化"，是"安全的数据复用"**：TRPO 已经解决了稳定性问题，但代价是每步更新都要解共轭梯度。PPO 用一行 `min(r·A, clip(r, 1-ε, 1+ε)·A)` 做到了 90% 的效果——这是工程简洁性战胜数学优雅的教科书案例。
2. **剪切 = "悬崖边设一堵软墙"**：不是约束变化不可超过 ε，而是超过 ε 后梯度消失——让策略自己停在墙前。这是一种"用物理约束代替规则约束"的思想。
3. **clip fraction 是 PPO 的血压**：0.1-0.3 是健康。如果长期为 0，你在浪费 data reuse 的机会。如果长期 >0.5，你的策略在漂移中不可控。
4. **exp(log_new - log_old) 不是细节，是生死**：直接 new/old 在数值上会爆，exp 差是稳定的。这条代码级别的决定解释了为什么 CleanRL 的实现成为黄金标准。
5. **PPO 的超参鲁棒性被严重低估**：K 在 4-15 之间、ε 在 0.1-0.3 之间、minibatch 在 32-128 之间——绝大多数基准的最终性能波动不到 5%。这意味着你可以把注意力集中在 reward design 上，而不是调参。
6. **RLHF 里的 PPO 只是改了一个 KL 项**：本质上还是 `min(r·A, clip(r,1-ε,1+ε)·A)`，但 A 的计算中包含来自 RM 的奖励和相对于参考策略的 KL 惩罚。这不是一个新的 RL 算法——是一个新的奖励函数。
7. **GRPO 把 PPO 的剪切思想从"有 critic"推向了"无 critic"**：这是 2025 年 DeepSeek R1 背后的大脑——说明 PPO 的剪切逻辑比它的 actor-critic 结构更根本。

### 四、教学建议

1. **先从 TRPO 讲起，再展示 PPO 的一行替代**：让学生先感受共轭梯度的"做对了但太贵了"，再看到 `min(... clip(...))` 的简洁——这个对比本身就是"工程选择"最好的教学。
2. **在白板上画 r_t 和 L^{CLIP} 的关系图**：一条 X 轴 = r_t 的折线——A>0 时是一条斜坡然后平坦；A<0 时是一条下坡然后平坦——这个图替换 10 分钟的公式解释。
3. **做一个"有剪切 vs 无剪切"的动画对比实验**：在相同环境上，一边是未剪切 PPO（退化为 A2C 多轮），一边是剪切 PPO——展示前者的策略在 5-6 轮后就爆炸了。
4. **强调 clip fraction 是"PPO 健康检查仪"**：让学生养成在每个 checkpoint 上打印 clip fraction 的习惯——这是 PPO 特有的仪表盘。
5. **让学生手写 `exp(log_new - log_old)` 的数值计算**：给一组 π_θ(a|s) 和 π_old(a|s) 的数字，分别用 direct division 和 exp-diff 计算——证明数值稳定性的来源。
6. **对比 PPO-KL 和 PPO-clip 在 RLHF 中的用法**：为什么 RLHF 需要一个额外的 KL（到 reference）即使已经在用 PPO-clip——因为 RLHF 的漂移方向是奖励模型引导的，而奖励模型本身只在某个分布上准确。
7. **留一个 GRPO 的钩子**：讲完 PPO，"你能想到怎么去掉 critic 吗？"——这是从 PPO 到 GRPO 的自然过渡问题。

### 五、值得补充的内容

1. **PPO 的分布式实现细节**：IMPALA 架构如何将采样（actor）和学习（learner）分布在不同的进程/GPU 上，以及这如何改变 clip fraction 的语义（采样的策略和更新的策略之间有多步延迟）。
2. **PPO 在连续动作空间中的实现细节**：高斯策略的均值/标准差头、动作熵的计算、以及 log-prob 在多元高斯中的公式。
3. **Reward scaling 对 PPO 稳定性的定量分析**：如果奖励的标准差是 10，而 ε=0.2——剪切在正常奖励波动中不起作用——需要展示 reward normalization 的代码和效果。
4. **PPO 在"稀疏奖励"环境中的表现分析**：在没有 dense reward 的情况下，PPO 的 exploration 能力来自 entropy bonus——但 entropy bonus 是一个粗暴的工具，不是真实的探索策略。
5. **PPO 到 GRPO 的完整推导链**：为什么 GRPO 可以丢掉 critic——因为 group-relative advantage 在组内提供了足够的 baseline。

### 六、一句话总结

PPO 的本质不是"一个更好的 RL 算法"，而是"一个用极简代码实现复杂置信域约束的工程奇迹"——它把 TRPO 的共轭梯度替换成了一行 `min(r·A, clip(r, 1-ε, 1+ε)·A)`，然后用这十几字符的优雅度过了十年——从 MuJoCo 到 ChatGPT 到 DeepSeek R1。

# 🎓 Agent 架构课：PPO——为什么用十几行代码能做到 TRPO 三页公式做不到的？

**副标题：从"用一次扔一次"的 A2C 到"把数据榨干"的置信域智能约束**

---

先问一个问题——

你花了很多时间收集了一批 data（rollout），包括状态、动作和奖励。你用这批 data 做了一次梯度更新。现在你的策略变了——哪怕只变了一点点。**你接下来该用这批 data 做第二次更新吗？**

……

如果你是一个传统的强化学习工程师，你的回答是"不"。数据是从 `π_old` 采样的，而不是从当前的 `π_θ` 采样的——继续用就是 off-policy，梯度有偏。扔掉它，重新采集。TRPO 说"我能做第二次——但要用一个很贵的约束保证漂移不超限。"

PPO 说："我能做十次——而且只需要一行代码。"

这就是 PPO 的故事。这不止是一个算法的故事。这是**工程如何在算力和优雅之间找到最优边界**的故事。

让我给你两条路：

**路径一：TRPO（2015）。** 在目标上追加一个 KL 约束，限制新策略与旧策略之间的散度不超过 δ。这保证每一步 update 在"信任域"内——理论上你可以证明性能单调提升。但你为此付出的代价是：每次 update 要解一次共轭梯度。在 2015 的 Atari 环境上这还勉强可行——但放到 2024 的一个 70B 参数 LLM 上？一个 update 成本已经不可接受了。

**路径二：PPO（2017）。** 在目标上不需要约束。只需要一个 `min(r·A, clip(r, 1-ε, 1+ε)·A)` 的损失。当比率 r 超出 [1-ε, 1+ε] 区间——即新策略在一个动作上已走得太远——梯度变为零。策略自己停下来。不需要解共轭梯度。你只需在 rollout 时记录 `log π_old`，更新时计算 `exp(log π_new - log π_old)`，然后套进这个 min-clip 结构。

我在生产环境里不需要犹豫——永远是 PPO。不是因为 TRPO 错了。是因为 **TRPO 的正确性成本太高，高到在实际的规模上它不再是一个选项。而 PPO 以不到2%的性能换来了一个数量级的速度提升。** 在 Agent 系统里，当一个选择能带来一个数量级的速度提升且性能损失在自然噪声水平以下——这不是一个"trade-off"，这是一个"你不做这个选择你就是在烧钱"。

但是这里有一件"不提就会翻车"的事我必须告诉你——我见过上百个 PPO 实现的崩溃，其中超过一半的根本原因只有一个：

**重要性比率的计算方式。**

我见过代码里写 `ratio = probs_new[a] / probs_old[a]`。这会在概率小的时候炸飞——一个 1e-6 / 1e-5 的比例是 0.1，看起来正常；但一个 1e-10 / 1e-11 的比例的浮点数就是任意值。**正确的写法永远、永远、永远是 `ratio = exp(log_prob_new - log_prob_old)`。** 日志空间做减法、指数空间做归一——这是所有 RL 工程师应该刻进灵魂里的习惯。每当我在 code review 里看到 `p_new / p_old` 几个字，我都会直接打上 red flag。这不是优化建议——这是 bug。

回过头来，**为什么 PPO 能跑 10 轮**——这是一个更深入的问题。答案不在 "K=10 是最优的"，也不在 "数据复用就是好"。答案在**剪切机制创造了一个隐式的 KL 约束——不是每个动作上的约束，而是在"该方向上继续"和"该方向上停下来"之间的一个柔软的折点。**

具体来说：如果一个动作已经被判定为好（A_t > 0），新策略想要提升它的概率。剪切允许它提升到旧策略概率的 1+ε 倍——比如 ε=0.2，那就是提升 20%。**超过这个限值之后，梯度为零。不再推。** 策略自己就留在了这里。不是被迫留下——是在这种局面下已经不值得继续了。

同样，如果一个动作被判定为坏（A_t < 0），新策略想要降低它的概率。剪切允许它降到旧策略的 1-ε=0.8 倍——降低 20%。超过了也梯度为零。不再压。

所以你每次能做的最大改变就是 ε = 20%。而这个 20% 是相对于 rollout时的旧策略而言的——**这意味着 K=10 轮能造成的总变化量不是 20% × 10 = 200%**——因为每次更新都是对比 rollout 时的旧策略，不是对比上一次更新后的策略。如果你在 rollout 时的旧策略上做了 20% 的改变，下一次更新时你的比率是 1.2——你仍然距离 1+ε = 1.2 是零空间。你实际上被锁定了。

这个机制的优雅在于：它在没有调用一次 KL 计算的情况下，实现了一个严格的 "更新步长上限"。而它不需要解共轭梯度——只在 loss function 里多了一行 `min`，一个 `clip`。

**关于 clip fraction——这可能是整个 PPO 里最重要的诊断，但 90% 的人不看它。**

clip fraction = 比率超出 [0.8, 1.2] 区间的样本比例。

如果 clip fraction = 0——你的 ε=0.2 从未触发过。意思是你的更新太小了——策略几乎没变，你在浪费计算量。解法：提高 LR 或 K，或降低 ε 提前触发（因为 ε 越小触发门槛越低→但这也意味着你能学到的越少，很微妙）。

如果 clip fraction > 0.5——你的策略在每个 rollout 上都大幅漂移，剪切在拼命工作。这说明你的 LR 太高或 K 太大了。**连续的高 clip fraction 通常预示着一场崩溃——不是立即的，但在几个 rollout 后会。** 解法：降低 K 或 LR。

**健康的 clip fraction 区间是 0.1-0.3。** 这意味着 10-30% 的动作已经被推到剪切限——策略在"活跃地学习但又不过度漂移"。这个窗口是我在调试 PPO 时的唯一仪表盘——比回报曲线还先行。回报曲线是滞后指标，clip fraction 是先行指标。

最后一个反模式：**在 RLHF 里用 PPO 时忘记再加一个 KL 到 reference 模型。**

RLHF 里的 PPO 有*两个* KL：
1. **PPO 自带的隐式 KL**（通过剪切）——防止在单个 rollout 上过拟合。
2. **到 reference 模型的显式 KL 惩罚**——防止策略整个漂移到奖励模型看不到的分布上。

我在生产部署里见过不少团队把第二个 KL 设为零——然后策略在两三天后就"学会了"生成高分但在人类看来是胡言乱语的输出。奖励模型只在它见过的输出分布上是好 proxy，离开这个分布它就不可信。**KL 到 reference 是你在 RLHF 里唯一的安全网——永远备着一个。**

最后清单：

- ✅ 你的比率计算是 `exp(log_new - log_old)` 而不是 `probs_new/probs_old` 吗？
- ✅ 你的 ε 在 0.1-0.3 之间吗？—— 0.2 是世界上最安全的默认——别改。
- ✅ 你的 K 在 4-15 之间吗？—— K=10 是另一个不需要改的默认。
- ✅ 你每步都在 log clip fraction 吗？—— 它是先行指标，回报是滞后指标。盯前者。
- ✅ 你有奖励归一化吗？—— 如果你有奖励比 0.1 大很多的环境，归一化会把你的剪切带回来。跳过它你会失去大部分剪切的作用。
- ✅ 如果在 RLHF：你有第二个到 reference 的 KL 吗？—— 没有它，你的策略会在三天内学会"胡言乱语"。

**金句：PPO 不是"一个学习率调得很稳的算法"——它是"一个在设计层面就内置了学习率上限的算法"。那个上限就是 `min(r·A, clip(r, 1-ε, 1+ε)·A)`，一行代码，零数学开销，十年不过时。**

回到开头的问题——为什么十几行代码能做到三页公式做不到的？因为 PPO 发现：**你不需要在每一步都精确计算一个约束的边界——你只需要在每个有风险的方向上放一堵柔软的墙，到墙前梯度消失，策略自然止步。** 这不是"数学更严谨"，这是"物理更省力"。TRPO 是控制论——计算精确的边界再走。PPO 是机械设计——把墙放在它自然会撞到的地方，让它自己停下来。

而这面软墙——十年后——仍然支撑着从 MuJoCo 到 ChatGPT 到 DeepSeek R1 的所有训练管线。这就是 PPO 给我们的礼物。不是"最快的 RL 算法"，而是**"最放心的"——你知道你夜里的训练不会炸飞，你知道你的策略是被软墙包围的，你知道你明天醒来时回报曲线还在健康区间。**
