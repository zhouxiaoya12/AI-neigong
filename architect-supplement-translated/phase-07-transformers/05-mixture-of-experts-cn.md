# 混合专家模型（Mixture of Experts, MoE）

> 一个密集的 70B Transformer 对每个令牌激活全部参数。一个 671B 的 MoE 每个令牌只激活 37B，却在每个基准上胜出。稀疏性是这十年来最重要的扩展思想。

**类型：** 构建
**语言：** Python
**前置要求：** Phase 7 · 05（完整 Transformer）、Phase 7 · 07（GPT）
**时间：** 约45分钟

## 问题所在

密集 Transformer 推理时的 FLOPs 等于其参数数量（前向传播再乘2）。扩大密集模型意味着每个令牌都要付全价。到2024年，前沿正在撞计算墙：要明显更聪明，你需要每个令牌指数级增加 FLOPs。

混合专家模型打破了这个联系。将每个 FFN 替换为 E 个独立专家 + 一个路由器，每个令牌选出 k 个专家。总参数 = E × FFN_size。每个令牌的活跃参数 = k × FFN_size。2026年的典型配置：E=256、k=8。存储量随 E 扩展，计算量随 k 扩展。

2026年的前沿几乎全是 MoE：DeepSeek-V3（671B 总量 / 37B 活跃）、Mixtral 8×22B、Qwen2.5-MoE、Llama 4、Kimi K2、gpt-oss。在 Artificial Analysis 的独立排行榜中，前10名开源模型全部是 MoE。

## 核心概念

![MoE 层：路由器从 E 个专家中为每个令牌选出 k 个](../assets/moe.svg)

### FFN 替换

密集 Transformer 块：

```
h = x + attn(norm(x))
h = h + FFN(norm(h))
```

MoE 块：

```
h = x + attn(norm(x))
scores = router(norm(h))              # (N_tokens, E)
top_k = argmax_k(scores)              # 每个令牌选出 k 个专家
h = h + sum_{e in top_k}(
        gate(scores[e]) * Expert_e(norm(h))
    )
```

每个专家是一个独立的 FFN（通常是 SwiGLU）。路由器是一个单线性层。每个令牌选自己的 k 个专家，得到它们输出的门控混合。

### 负载均衡问题

如果路由器把 90% 的令牌都发给专家 3，其他专家就饿死了。尝试过三种修复：

1. **辅助负载均衡损失**（Switch Transformer, Mixtral）。添加一个与专家使用方差成正比的惩罚项。有效，但增加一个超参和第二个梯度信号。
2. **专家容量 + 令牌丢弃**（早期 Switch）。每个专家处理最多 `C × N/E` 个令牌；溢出令牌跳过该层。损害质量。
3. **无辅助损失的均衡**（DeepSeek-V3）。添加一个可学习的每专家偏置，平移路由器的 top-k 选择。偏置在训练损失外部更新。不对主目标施加惩罚。2024年的重大突破。

DeepSeek-V3 的方法：每个训练步骤后，对每个专家检查其使用量是否高于或低于目标。以 `±γ` 微调偏置。选择使用 `scores + bias`。用于门控的专家概率是未经修改的原始 `scores`。将路由与表达解耦。

### 共享专家

DeepSeek-V2/V3 还将专家分为*共享*和*路由*。每个令牌通过所有共享专家。路由专家通过 top-k 挑选。共享专家捕获共性知识；路由专家专业化。V3 运行 1 个共享专家加 top-8（从 256 个中选）。

### 细粒度专家

经典 MoE（GShard, Switch）：每个专家与完整 FFN 一样宽。E 较小（8–64），k 较小（1–2）。

现代细粒度 MoE（DeepSeek-V3, Qwen-MoE）：每个专家较窄（完整 FFN 的 1/8）。E 很大（256+），k 更大（8+）。总参数相同，但组合数量呈爆炸式增长。`C(256, 8) = 400 万亿`种可能的令牌"专家组合"。质量上升，延迟不变。

### 成本剖面

每个令牌、每层：

| 配置 | 每令牌活跃参数 | 总参数 |
|------|--------------|--------|
| Mixtral 8×22B | ~39B | 141B |
| Llama 3 70B（密集） | 70B | 70B |
| DeepSeek-V3 | 37B | 671B |
| Kimi K2（MoE） | ~32B | 1T |

DeepSeek-V3 在几乎所有基准上都击败了 Llama 3 70B（密集），同时每个令牌的活跃 FLOPs 更少。更多参数 = 更多知识。更多活跃 FLOPs = 每个令牌更多计算。MoE 将它们解耦。

### 隐忧：内存

所有专家都驻留在 GPU 上，无论哪些被激活。一个 671B 模型需要约 1.3 TB 显存存储 fp16 权重。前沿 MoE 部署需要专家并行——将专家分布到多张 GPU，通过网络路由令牌。延迟由全到全通信主导，而非矩阵乘法。

## 动手构建

参见 `code/main.py`。一个紧凑的 MoE 层，纯标准库实现，包含：

- `n_experts=8` 个类 SwiGLU 专家（各一个线性层，用于说明）
- top-k=2 路由
- softmax 归一化门控权重
- 通过每专家偏置实现无辅助损失均衡

### 步骤 1：路由器

```python
def route(hidden, W_router, top_k, bias):
    scores = [sum(h * w for h, w in zip(hidden, W_router[e])) for e in range(len(W_router))]
    biased = [s + b for s, b in zip(scores, bias)]
    top_idx = sorted(range(len(biased)), key=lambda i: -biased[i])[:top_k]
    # 对选中专家的原始分数做 softmax
    chosen = [scores[i] for i in top_idx]
    m = max(chosen)
    exps = [math.exp(c - m) for c in chosen]
    s = sum(exps)
    gates = [e / s for e in exps]
    return top_idx, gates
```

偏置影响选择，不影响门控权重。这是 DeepSeek-V3 的绝招——偏置纠正负载不平衡，而不引导模型的预测。

### 步骤 2：让 100 个令牌通过路由器

追踪哪些专家被调用以及调用的频率。没有偏置时，使用量是倾斜的。通过偏置更新循环（过度使用的专家 `-γ`，使用不足的专家 `+γ`），使用量在几次迭代后收敛到均匀分布。

### 步骤 3：参数计数比较

打印 MoE 配置的"密集等价模型"。DeepSeek-V3 形状：256 个路由 + 1 个共享，8 个活跃，d_model=7168。总参数量令人瞠目。活跃参数数仅为密集 Llama 3 70B 的七分之一。

## 实战使用

HuggingFace 加载：

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
model = AutoModelForCausalLM.from_pretrained("mistralai/Mixtral-8x22B-v0.1")
```

2026年生产推理：vLLM 原生支持 MoE 路由。SGLang 拥有最快的专家并行路径。两者都自动处理 top-k 选择和专家并行。

**何时选择 MoE：**
- 你想要前沿质量，但每个令牌推理成本更低。
- 你有足够的显存/专家并行基础设施。
- 你的工作负载是令牌密集型的（聊天、代码），而非上下文密集型的（长文档）。

**何时不选 MoE：**
- 边缘部署——你为任何活跃 FLOP 支付完整的存储成本。
- 对延迟敏感的单用户服务——专家路由增加开销。
- 小模型（<7B）——MoE 的质量优势只在某个计算阈值以上显现（约6B活跃参数）。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 专家 | "众多 FFN 中的一个" | 独立的前馈网络；参数专用于 FFN 计算的稀疏切片。 |
| 路由器 | "门控" | 微小的线性层，对每个令牌给每个专家打分；top-k 选择。 |
| Top-k 路由 | "每个令牌 k 个活跃专家" | 每个令牌的 FFN 计算恰好经过 k 个专家，由门控加权。 |
| 辅助损失 | "负载均衡惩罚" | 额外损失项，惩罚偏斜的专家使用。 |
| 无辅助损失 | "DeepSeek-V3 的绝招" | 仅对路由器选择的每专家偏置进行均衡；无额外梯度。 |
| 共享专家 | "始终开启" | 每个令牌都经过的额外专家；捕获共性知识。 |
| 专家并行 | "按专家分布" | 将不同专家分配到不同 GPU；通过网络路由令牌。 |
| 稀疏性 | "活跃参数 < 总参数" | 比例 `k × expert_size / (E × expert_size)`；DeepSeek-V3 为 37/671 ≈ 5.5%。 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这篇 MoE 教材从"密集模型中参数和计算量绑死"这个第一性原理出发，精准地定位了 MoE 的架构动机——解耦"知识量"和"计算量"。从 Switch Transformer 到 DeepSeek-V3 的无辅助损失偏置技巧，有一条清晰的工程进化线。尤其出色的是"何时选/不选 MoE"的部分——这比任何学院派教材都更接近生产决策的真实面目。

### 二、知识结构梳理

- **基础层**：密集 FFN → MoE FFN 的替换、top-k 路由的数学、门控 softmax 加权、SwiGLU 专家结构
- **模式层**：三种负载均衡方案（辅助损失/容量丢弃/偏置均衡）、共享专家 vs 路由专家的分工、细粒度 vs 粗粒度专家
- **应用层**：成本剖面分析（总参数 vs 活跃参数 vs KV缓存）、MoE 的部署选择（专家并行、all-to-all 通信）、生产模型案例（DeepSeek-V3 37B活跃胜 70B 密集）

### 三、核心洞察

1. **MoE 的核心创新不在"多专家"，在"解耦"**：密集模型里，参数多→知识多，但也→FLOPs 多。MoE 把"知识存储"和"每个令牌的活跃计算"分开了。这是比任何具体 trick 都更根本的架构贡献。
2. **无辅助损失偏置是 2024 年最重要的 MoE 突破**：在此之前，负载均衡必须通过额外的损失项来实现——这意味着你的主要训练目标和均衡目标在打架。DeepSeek-V3 的偏置方案让均衡成了"训练之外的操作"，不干扰主目标。
3. **细粒度专家 + 大 k 是组合爆炸优势**：C(256,8) = 4×10^14 种可能的"专家组合"——每个令牌在训练中见过的具体组合微乎其微，但同层所有专家共享了共同的基础——这是 sparse 表示的一个极端案例。
4. **共享专家的存在说明"有些知识不需要稀疏化"**：语言中有大量"通用"知识（基本语法、常见词汇搭配），让所有令牌都经过这些知识是合理的——稀疏化只适用于专业化知识。
5. **MoE 模型的内存需求可能比参数增长更可怕**：671B 参数需要 1.3TB fp16 显存——但只有 37B 活跃。这意味着你需要大很多倍的硬件来存储一个模型，虽然它在推理时算得跟一个小得多模型一样快。
6. **专家并行中的 all-to-all 通信是 MoE 推理的真正延迟瓶颈**：在分布式 MoE 推理中，每个 token 的路由决策触发跨 GPU 的数据传输——这个通信时间可能超过计算时间。工程设计必须优先考虑路由负载均衡，不是因为质量，而是因为延迟。
7. **MoE 不是"更大的模型"——它是"更聪明的模型"**：相同的每令牌 FLOPs 预算下，MoE 能用多几个数量级的参数来编码知识。对知识密集型任务（事实问答、编程）这是质的差距。

### 四、教学建议

1. **从"为什么 70B 密集不够"讲起**：展示一个密集模型扩展到 1T 参数的 FLOPs 成本——然后展示 DeepSeek-V3 如何用 37B 活跃 FLOPs（相当于 ~37B 密集）击败 70B 密集。这个对比是 MoE 的最佳广告。
2. **做一个"有偏置 vs 无偏置"的负载可视化**：跑 100 个 token 通过路由器两次——一次没有偏置校正，一次有——用柱状图显示专家使用分布。学生会在5秒内理解为什么需要负载均衡。
3. **手算 C(256,8) 的组合数**：在黑板上展开组合公式——这个数字本身是数据不足以穷举的证据。
4. **把 MLP 看作"知识"，注意力看作"动态路由"**：这个类比能帮助学生理解为什么 MoE 只替换 MLP 而不替换注意力——注意力已经在做动态路由了。
5. **对比三种负载均衡方案的代码差异**：辅助损失加一行 loss += aux_coef * aux_loss；容量丢弃加一个 if 判断；偏置方案加一个 bias += γ*sign(usage - target)。代码改动的简洁性本身就是优势。
6. **演示"专家崩溃"现象**：如果不加任何均衡，前几百步训练后所有 token 都流向 1-2 个专家——让 students 亲眼看到这个 collapse，理解为什么均衡是"生存必需"。
7. **让学生画一张"部署心理地图"**：给定显存和延迟约束，判断该选密集还是 MoE、该用多少个专家、该用多少卡做专家并行。这是建筑师的肌肉记忆。

### 五、值得补充的内容

1. **Expert Offloading**：对于显存不足的 MoE 推理，把不活跃专家的权重放在 CPU 内存里，使用时再加载——这是一种"换页系统"式的部署技巧。
2. **MoE 训练的分布式策略细节**：Expert Parallelism + Data Parallelism 的组合，以及它们对梯度和 all-reduce 通信的影响。
3. **Hash-based 路由 vs 学习路由的更深层对比**：哈希路由的确定性和低成本 vs 学习路由的语义性——在什么场景下前者更合适？
4. **MoE + Speculative Decoding 的互作**：MoE 模型的草稿和验证阶段如何使用不同的专家——草稿只用少量专家，验证用全部。
5. **Sparsity patterns 的分析**：训练好的 MoE 模型中哪些专家经常一起激活（co-activation patterns）——这实际在定义模型的"子模块"结构。

### 六、一句话总结

MoE 不是"并行运行多个模型"，而是在同一块表示空间中开辟了多扇"知识专用通道"——每个专家学一类知识，每个令牌根据上下文动态选择知识通道的组合——这使得模型在知识量（存储）和计算量（推理成本）之间实现了自 Attention 诞生以来最大的结构性解耦。

# 🎓 Agent 架构课：混合专家——为什么 671B 参数但只激活 37B 的模型比 70B 全激活更聪明？

**副标题：从"计算和知识绑死"到"只为你活跃的部分付费"的稀疏化革命**

---

先问一个问题——

如果你把一个 FFN 的 7168 → 18432 的权重矩阵"放大10倍"，你获得了更多的知识存储，但每个令牌推理时都要把这 10 倍的矩阵全部算一遍，每层如此，每个令牌如此。你觉得这值得吗？

……

从 2020 年的角度：值。GPT-3 就是这条路，175B 参数全部激活，每次推理都为每个令牌付全价。

从 2026 年的角度：**不值。更好的办法是把这 10 倍拆成 256 个独立的 FFN（"专家"），每个令牌只调用其中最相关的 8 个。**知识量涨了 10 倍，活跃计算量只涨了 1/32 个专家的份额。这是混合专家（MoE）的核心命题。

DeepSeek-V3：671B 总参数，每个令牌只激活 37B。对比 Llama 3 70B（70B 全部激活），DeepSeek-V3 在几乎所有基准上胜出，但每令牌的计算量只有后者的一半。

让我给你两条路径：

**路径一：密集扩展。** 你要让模型"知道更多"，你把总参数量从 70B 推到 200B、400B、1T。每个令牌推理时都必须在所有参数上做矩阵乘法。延迟线性增长。在小批量或单请求推理中，你的 GPU 利用率从本就很低的 20% 再打折。

**路径二：MoE 扩展。** 你要让模型"知道更多"，你把专家的数量从 8 推到 64、256。每个令牌只激活其中 8 个。延迟几乎不变（路由开销很小，且被更小的每次 matmul 尺寸抵消）。但总参数量——也就是知识量——是指数级差的。

我在生产环境里不会犹豫一秒钟：路径二是唯一合理的方案。不仅是省成本。**路径二在相同的每令牌 FLOPs 预算下实际上产出更高智商。** 原因很简单：同样的计算量，MoE 让你将更多的知识编码进权重的容量中。而密集模型——相同 FLOPs 下——只能编码对应的容量。

但这有代价。两个代价。

**代价一：负载均衡。**

设想你有 256 个专家，但经过几千步训练后，99% 的令牌只经过专家 3、7、42 和 128——其他 252 个专家闲置。你花了显存放的权重，没有被训练到——"专家的死亡"。这是 MoE 的天生缺陷。

三种修复。我只推荐一种。

- 辅助损失（Switch、Mixtral）：这是大多数论文里提到的"在损失函数里加一个项惩罚不均衡"。它在原理上是对的，但在生产上烦人：你在训练目标里塞了第二个目标，要调权重、要监控、会在质量上造成小但可测量的拖累。
- 容量+丢弃（早期）："每个专家每步最多只处理固定数量的令牌，超过就丢掉。"——这实际上在随机丢弃训练数据。在玩具任务上还行，在语言质量上是不可接受的。
- **无辅助损失偏置（DeepSeek-V3）：这是 2024 年真正的游戏改变者。** 你在每个专家的路由分数上增加一个可学习的偏置项，偏置更新的方向是"把使用量推向均衡"。但门控权重用的是原始分数，不是偏置后的。这意味着：偏置影响*选择哪些专家*（被选中），不影响*给每个专家的权重*（怎么混合）。并且偏置在训练损失之外更新——它不影响主目标的梯度。

2026 年往后，如果你在一个新的 MoE 上还看到辅助损失，我会说这个团队还没有读到最新一页。但我必须提醒：DeepSeek 的偏置更新率 γ 是一个新超参——如果太大，偏置会在均衡和不均衡之间振荡；如果太小，收敛慢。这个 γ 的调参是你在 MoE 项目里要花最多时间的事情之一。

**代价二：内存。**

你花了一辈子（夸张说法）积攒出的 671B 参数——它们全部必须在 GPU 显存里，即使任何时刻只有 37B 在活跃。fp16 下这需要约 1.3 TB 显存。一张 H100 有 80 GB。一张 B200 有 192 GB。你需要的不是一张卡，是一个集群。**MoE 在训练和推理中都逼着你做 Expert Parallelism——把专家分布到多张 GPU，跨网络路由令牌。** All-to-all 通信成了你的延迟瓶颈。如果网络速度是 800 Gbps 的 InfiniBand，这个延迟可能可控。如果是 400 Gbps 的 RoCE，你的延迟会比计算时间更长——GPU 再次摸鱼。

因此，部署 MoE 的真正艺术不在于"选对 E 和 k"，而在于**画出你的通信矩阵图，确定 all-to-all 通信是延迟瓶颈还是带宽瓶颈，然后据此选择你的并行策略和互联方案。** 如果你没有看过 SGLang 的 expert-parallel dispatch 代码——去看看。它把 token routing 做成了共享内存的固定块调度——这避免了每 token 一条消息的通信开销。

**一场具体的战争。** 2025年，我有一个项目需要选型。我们是选 Llama 3 70B（密集，一张 4×A100 做 TP）还是 DeepSeek-V3（671B，需要 8×A100 做 EP）？

Llama：4张卡，推理延迟 37ms/token，KV 缓存占用 10.4GB（128K），成本 ~$4.50/hr。
DeepSeek：8张卡，推理延迟 52ms/token（因为 all-to-all），但知识广度和推理质量显著更好，成本 ~$9/hr。

我们选了 DeepSeek——不是因为它在纸上更好看。**而是因为我们的工作负载（代码生成 + 工具调用）极度依赖模型的世界知识。**DeepSeek 庞大的参数量在这里给出了密集模型在相同活跃 FLOPs 下无法匹敌的准确率。那个延迟多出的 15ms 被"一次性正确"的概率提回了成本效益。

但如果你在做实时的语言翻译？我会毫不犹豫选 Llama 密集模型。延迟敏感的单一任务场景不需要 MoE 的知识广度——需要的是一张卡上的按批次满负荷计算。

这引出了我的最终清单——如果你在决定是否上 MoE：

- ✅ 你的工作负载是"知识密集型"的吗（代码、事实、多语言）？——如果是，MoE 的多参数优势在这里开花。如果不是（翻译、摘要、情感分析），密集更划算。
- ✅ 你的每令牌推理成本占比是否 > 训练成本？—— MoE 降低每令牌 FLOPs，适合推理体积大的场景。
- ✅ 你的 GPU-to-GPU 网络带宽是多少？—— <400 Gbps 的话，all-to-all 通信会吃掉 MoE 的所有延迟优势。
- ✅ 你的模型是否 ≥ ~6B 活跃参数？—— 比这个小的模型，MoE 的质量优势不明显。
- ✅ 你准备了多少显存？—— 总参数 fp16 的大小是你部署的硬上限——无论活跃参数多小。

**反模式必须点名。** 我见过一个团队把 MoE 的 k 设得太大（k=32，E=64），这使得 token 选择的稀疏性消失——每个 token 几乎经过所有专家。这不是 MoE，这是一个带有巨大路由开销的密集模型。**k 的最优范围通常是在 E 的 1/32 到 1/8 之间。超过这个范围，稀疏化的价值递减。**

还有一个反模式——在 MoE 里用小的 batch size。如果你每个 batch 只有4个序列，每序列 128 个 token，总共 512 个 token——256 个专家就只平均每个看到 16 个 token。**低利用率导致高噪声、低训练效率。** MoE 需要大的全局 batch size（通常 >2048 token）才能让所有专家获得有意义的梯度。

**金句：MoE 不是"稀疏注意力"的同类——它是"稀疏 FFN"。注意力已经在做动态路由了——注意力头本身就是一种"软性专家选择"。MoE 是在"知识存储"端（FFN）引入稀疏性，这才是新信息。**

回到开头的问题——671B 参数但只激活 37B 的模型为什么比 70B 全激活更聪明？因为**智商不是"每个 token 上砸了多少计算"，而是"整个模型里储存了多少关于世界的模式"。** MoE 让你把一个学期的知识装进图书馆，但每次查资料只需要走到对应的那个书架——密集模型是把所有书架搬到你的桌上，但图书馆更小。大图书馆小书架永远比小图书馆全上桌更有利于解决复杂问题。

这就是稀疏化的终极承诺。不是更快——是在不增加每步计算成本的前提下，把模型变得更深、更广、更能记住这个世界。
