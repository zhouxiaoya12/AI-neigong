# 动作预算、迭代上限和成本调控器

> 一个中型电商Agent的月度LLM成本在其团队启用"订单追踪"技能后从$1,200跃升至$4,800。这不是定价Bug。这是一个Agent发现了一个新循环并在其中持续消费。微软的Agent治理工具包（2026年4月2日）将防御此类问题的方案编码为：每次请求的`max_tokens`、每个任务的令牌和美元预算、每天/每月上限、迭代上限、分层模型路由、提示缓存、上下文窗口化、昂贵操作上的人机协同确认检查点、预算超支的终止开关。Anthropic的Claude Code Agent SDK以不同的名称提供了相同的原语。金融速度限制——例如在10分钟内消费超过$50则切断访问——比月度上限更快捕获循环。

**类型：** 学习
**语言：** Python（标准库，分层成本调控器模拟器）
**前置条件：** 阶段15 · 10（权限模式）、阶段15 · 12（持久执行）
**时间：** ~60分钟

## 问题

自主Agent在每一回合花费真实的金钱。聊天机器人的糟糕输出是一个糟糕的回复；Agent的糟糕循环是一张账单。行业记录的这个失败模式的术语是"Denial of Wallet"——Agent持续推理、持续调用工具、持续计费，没有什么能阻止它，因为没有设计任何东西来阻止它。

修复方案不是一个数字。它是一个在不同时间尺度和粒度上的限制堆栈：每次请求、每个任务、每小时、每天、每月。一个设计良好的堆栈在几分钟内捕获失控循环，在几小时内捕获缓慢泄漏，在一天内捕获糟糕的发布。同一个堆栈确保Agent在长周期和自主运行时保持预算。

这是一个工程课程：数学很简单，纪律是团队失败的地方。下面的限制列表全部在微软Agent治理工具包或Anthropic Claude Code Agent SDK文档中被命名。

## 核心概念

### 成本调控器堆栈

1. **每次请求的`max_tokens`。** 简单。防止单次调用发出无界的补全。
2. **每个任务的令牌预算。** 跨整个运行，不超过N个令牌。达到上限时硬停止。
3. **每个任务的美元预算。** 与令牌相同，但是以货币计。Claude Code中的`max_budget_usd`。
4. **每个工具调用上限。** 不多于N次`WebFetch`调用、N次`shell_exec`调用等。
5. **迭代上限（`max_turns`）。** Agent循环总迭代次数；防止无限推理循环。
6. **每分钟/每小时/每天/每月上限。** 滚动窗口。在不同时间尺度上捕获泄漏。
7. **金融速度限制。** 例如，"如果10分钟内消费超过$50，切断访问。"在月度上限触发之前捕获循环燃烧。
8. **分层模型路由。** 默认使用较小模型；仅当分类器判断任务值得时升级到较大模型。
9. **提示缓存。** 系统提示和稳定上下文存储在供应商缓存中；重新发送的令牌成本接近零。
10. **上下文窗口化。** 压缩/摘要以将活跃上下文保持在阈值以下；直接降低令牌成本。
11. **昂贵操作上的人机协同确认检查点。** 在已知为昂贵的操作（长工具调用、大下载、昂贵的模型升级）之前，需要人的批准。
12. **预算超支终止开关。** 当任何上限触发时，会话中止。上限被记录；需要单独的重新启用路径。

### 为什么是堆栈而非单一上限

单一月度上限仅在钱包已空后才捕获失控Agent。单一每次请求上限在会话级别什么都不捕获。不同的失败模式需要不同的时间尺度：

- **失控循环**（Agent卡在5秒重试中）：由速度限制捕获。
- **缓慢泄漏**（Agent每个任务做约2倍预期工作量）：由每日上限捕获。
- **糟糕发布**（新版本使用5倍令牌）：由每周/每月上限捕获。
- **合法激增**（真实需求，非Bug）：由小时/天上限+清晰日志捕获。

### Claude Code的预算表面

Claude Code Agent SDK公开（公开文档）：

- `max_turns` — 迭代上限。
- `max_budget_usd` — 美元上限；超支时会话中止。
- `allowed_tools` / `disallowed_tools` — 工具允许列表和拒绝列表。
- 工具使用前的Hook点，用于自定义成本核算。

与权限模式阶梯（第10课）结合使用。没有`max_budget_usd`的`autoMode`会话是无治理的自主性。Anthropic明确将Auto Mode框架为需要预算控制；分类器与成本是正交的。

微软的Agent治理工具包覆盖OWASP Agentic Top 10和EU AI法案第14条（人类监督）要求。对于在EU的生产部署，记录和上限执行不是可选的。

### 观察到的$1,200 → $4,800案例

微软文档中的真实案例：一个电商Agent的月度成本在添加新工具后增长了两倍。该工具允许Agent在每次会话期间轮询订单状态。没有循环检测。没有每个工具的上限。没有周对周增长的告警。修复方案是每个工具的上限加上每日增长告警。这是一个模板：每个新工具表面都是一个新的潜在循环；每个新工具需要自己的上限和告警。

4. 为一个真实任务（例如，"分类处理仓库中的50个issue"）的价格进行夜间无人运行估算。将`max_budget_usd`设置为你的点估计值的2倍。论证2倍的理由。

## 📝 教师备课总结与读后感

**文档评价：** 这是一个看似简单但极具生产价值的工程课程。成本调控器在概念层面不复杂——就是设置各种上限——但文档通过真实的$1,200→$4,800案例揭示了"为什么工程纪律比数学模型更重要"。文档将成本控制从"运维的事后补救"重新定位为"Agent设计的核心组件"，这不只是教学生怎么省钱，更是教学生怎么设计"能安全自主运行"的系统。最精彩的部分是"为什么是堆栈而非单一上限"的分析——不同时间尺度的上限对应不同的故障模式，这体现了系统工程的层次化思维。

**知识结构：** 成本调控器堆栈从最微观（每次请求max_tokens）到最宏观（月度上限）构成12层防御体系。按时间尺度可分为三类：（1）会话级（max_tokens、task budget、per-tool cap、max_turns、max_budget_usd）—防止单次运行失控；（2）实时级（velocity limit、tiered routing）—捕获快速燃烧；（3）周期级（hourly/daily/monthly caps）—捕获泄漏和趋势。提示缓存和上下文窗口化是成本优化手段，HITL检查点和终止开关是安全机制。

**核心洞察（5-7条）：**
1. "Denial of Wallet"——Agent的糟糕循环不是用户体验问题，是账单问题——这个命名揭示了Agent自主性的隐性成本，对企业和开发者都是最直接的痛点。
2. 单一上限不够——不同时间尺度的上限解决不同问题——这不是"多加几层更安全"，而是"不同故障模式有不同的检测时间窗口"。
3. 真实案例$1,200→$4,800的根因是"新增工具=新增潜在循环"——每个新工具都是一个未被评估的成本向量，这个洞察适用于所有Agent扩展场景。
4. 速度限制（$50/10分钟）vs月度上限的区别本质上是"实时断路器"vs"月终对账单"——前者在损失可忽略时截断，后者在损失已经发生后才告知。
5. 分层模型路由是"成本-能力"的动态权衡——不是"什么时候都用最好的模型"，而是"只在需要时才用"，这是一个经济学优化而不仅是技术选择。
6. 提示缓存是被低估的成本优化手段——在一个长周期Agent中，系统提示可能被重复发送成百上千次，缓存将这部分成本降至接近零。
7. 微软治理工具包覆盖EU AI法案和OWASP Top 10意味着成本调控器不只是一个"省钱工具"，而是一个合规要求——这给了工程团队推行的"政治资本"。

**教学建议（5-7条）：**
1. 开场用"钱包漏洞"演示——让模拟Agent在没有成本上限的情况下运行5分钟，然后展示账单数字——视觉冲击力比任何理论讲解都有效。
2. 成本调控器堆栈的讲解应该采用"逐层添加"的方式——从最基础的单次max_tokens开始，逐渐添加速度限制、每日上限、每月上限，每次添加后让学生思考"这层能捕获什么问题？还有哪些问题它会漏掉？"
3. 真实案例$1,200→$4,800应该被制作成一个"迷你案例研究"——给出时间线（week 1→week 4的成本图表），让学生分析问题出现在哪周、为什么团队没发现、应该在哪周设置什么上限。
4. 分层模型路由的概念应该结合"实际价格数据"讲解——展示不同模型的每次请求成本差异（如GPT-4o mini vs GPT-4o vs Claude Opus），然后演示"一个Agent从mini升级到Opus做了100次调用"的成本变化。
5. 练习2（为浏览器Agent设计每工具上限）是培养"成本风险分析"能力的好练习——让学生思考不同工具的成本特征（搜索$0、下载大文件$、调用付费API$$）、失败概率和循环风险。
6. 练习4（为50个issue分类任务定价）是实战演练——让学生体验"如何在不确定的任务长度上设置预算上限"，培养工程判断而非套用公式。
7. 最后可以做一个"成本调控器审计清单"——给出10个Agent部署场景（"电商客服Agent""代码审查Agent""研究Agent""社交媒体Agent"），让学生判断每个场景最重要的3个成本调控器是什么。

**值得补充（3-5条）：**
1. 补充一个"成本调控器的用户体验影响"分析——过于激进的上限可能导致Agent在合法任务中途被切断，如何在安全性和可用性之间平衡？
2. 增加"成本异常检测"的告警策略——不仅设置上限，还要设置"异常模式检测"（比如"本周成本分布与历史分布的KL散度超过阈值"），捕获那些没有超过上限但行为异常的Agent。
3. 补充"多供应商成本对比"——同一个Agent使用不同模型供应商的相同任务成本可能差5-10倍，如何做动态路由以优化成本？
4. 增加"成本调控器的社会维度"——当Agent被上限切断时，谁该负责？开发者（设得太低）？产品经理（定义的任务太复杂）？运维（没有监控到趋势）？
5. 补充"开源工具链"的介绍——除了微软和Anthropic的方案，社区中有哪些开源的成本调控器实现？如何集成到自己的Agent框架中？

**一句话总结：** 成本调控器不是让你省钱——是让你的Agent在自主运行时不会在你看不见的地方烧掉你一个月的预算。

# 🎓 Agent 架构课：成本调控器——你不给Agent花钱的自由，它就会给你花出你一年的自由！

一个真实的数字开场：**一个电商团队的Agent月度成本从$1,200跳到了$4,800。不是因为模型涨价了，不是因为用户变多了。是因为团队给Agent添加了一个新工具——"订单追踪"。Agent发现它可以在每次会话中轮询订单状态，于是它就这么做了。没有循环，没有bug——它就是发现了更多可以做的事情，然后去做了。这就是我见过的最昂贵的"功能发布"。**

现在我问你们一个问题：你的Agent有没有设置预算上限？

## "Denial of Wallet"：为什么Agent的无限自由是你的财务噩梦

在传统软件中，一个糟糕的功能通常只浪费用户的时间。一个死循环通常只浪费CPU周期。但在Agent系统中，Agent的每一个"想法"都在花钱——每次LLM调用、每次工具执行、每次重试。一个Agent的无限自主性 = 一张无限额度的信用卡。

这个行业造了一个词来描述这个问题："Denial of Wallet"——你的钱包不是被偷了，而是被Agent合法地、自动地、持续地消耗着，而你甚至不知道它在发生。你的会计部门月底收到账单时才发现的。

## 为什么一个上限不够：时间尺度的工程学

很多人觉得"我设一个月度$500的上限不就行了？"我来告诉你为什么不行。

假设你的Agent有一个月度$500的上限。平均每天$16。在某一个下午，你的Agent陷入了一个轮询循环——每5秒调用一次LLM，成本$0.01/次。你猜这个循环在一个下午（4小时）会烧多少钱？$28。没到月度上限。Agent继续烧。到第二天早上，你的运营团队才能从对账单上发现异常。而此时Agent已经烧了$100+。

这就是为什么你需要多层上限——不同时间尺度的上限捕获不同速度的问题：

**速度限制（velocity limit）**捕获最快的——"10分钟内超过$50就切断"。这是你的断路器。它不在乎"月度预算还剩多少"，它只问"现在花钱的速度是否正常"。

**每次请求上限（max_tokens）**是防火墙——防止一次调用就消耗天文数字的令牌。

**迭代上限（max_turns）**是结构性的——不管你花了多少钱，一个Agent不应该无休止地"思考"。

**每日/每月上限**是战略性的——它们捕获的不是"突发燃烧"，而是"悄悄泄漏"。一个Agent每个任务比预期多花20%——没人会注意到，直到月底账单多出$800。

我在生产系统中采用的原则是：**用最细粒度的上限尽量在最早期捕获问题，用最粗粒度的上限做最后的兜底。** 速度限制捕获循环，迭代上限捕获逻辑错误，每日上限捕获泄漏，月度上限确保整体不超预算。

## 分层模型路由：不是省钱，是给Agent合理的计算资源

这里有一个容易被误读的概念。分层模型路由——默认用小模型，关键任务才升级到大模型——很多人觉得这是"抠门"。但在我看来，这不是成本优化，是资源分配优化。

你的Agent一天做100个任务。其中90个是简单的："把这个issue分配给谁""这个邮件该怎么回复"。剩下10个是复杂的："分析这个代码库的安全漏洞"。如果你用同一个大模型处理所有100个任务，你在浪费计算资源——简单任务不需要Claude Opus级别的推理能力。

分层路由的真正价值不是"少花钱"，而是"把钱花在刀刃上"。一个分类器判断任务复杂度，然后路由到合适的模型。这就像你不会让一个外科医生去量体温——不是说他量不准，而是说这不值得。

## 真实案例的教训：每个新工具都是一个新的成本向量

$1,200到$4,800的案例有一个深刻的教训：**每次你给Agent添加一个新工具，你不仅在扩展它的能力，你也在创建一个新的潜在成本循环。** 订单追踪工具看起来无害——"查询一下订单状态嘛"。但Agent的逻辑是"越多越好"——它发现了可以查询订单状态，于是它在每次用户会话结束时都查一次，然后发现"这个订单两天没更新了，我得每小时查一次"。

这不是Agent的错。Agent被设计成"尽最大努力完成任务"。问题是没有人告诉它"查询订单状态应该在什么频率"、"什么样的查询是过度的"。所以它在自己的逻辑框架内做最优选择——而这个"最优"对成本来说是最坏的。

这就是为什么每工具上限很重要。不是所有的工具调用等值。一次文件读取可能花$0.001。一次LLM推理可能花$0.01。一次外网API调用可能花$0.05。如果你的Agent可以无限制地调用昂贵工具，你的预算上限形同虚设——因为Agent可能在达到迭代上限之前就烧掉了预算。

## 结语清单

在部署你的自主Agent之前，回答：
1. 你的Agent有每次请求的max_tokens限制吗？这是最简单但最容易被忽略的上限。
2. 你的Agent有一个速度限制吗？比如10分钟$50。如果没有，一个循环能在你的月度上限触发之前烧多少钱？
3. 你的每个工具都有调用次数上限吗？哪些工具最贵？它们的上限应该最小。
4. 你有没有做分层模型路由？你的Agent在做简单任务时是否在使用和复杂任务一样的大模型？
5. 你的成本调控器堆栈是否在CI中测试过？没有经过验证的上限，在关键时刻可能不会触发——而你不会希望用生产环境的$4,800账单来发现这一点。

**金句：给Agent自主权但不给预算上限，就像给一个青少年一张无限额度的信用卡然后说"我相信你的判断"。问题不是他会不会乱花钱——问题是你不知道"乱花钱"在他的逻辑框架里可能对应着什么。**
