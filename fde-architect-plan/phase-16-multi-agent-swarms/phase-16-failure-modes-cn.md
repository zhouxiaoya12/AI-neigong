# 失败模式 — MAST、群体思维、单一文化、级联错误

> 2026 年的参考分类法是 **MAST**（Cemri 等，NeurIPS 2025，arXiv:2503.13657），从 7 个最先进开源 MAS 的 1642 条执行轨迹中提取，显示 **41-86.7% 的失败率**。三个根类别：**规范问题**（41.77%）— 角色模糊、任务定义不清晰；**协调失败**（36.94%）— 通信中断、状态不同步；**验证缺口**（21.30%）— 缺少验证、没有质量检查。**群体思维（Groupthink）族**（arXiv:2508.05687）补充了：单一文化崩溃（相同基座模型 → 相关失败）、从众偏差（Agent 互相强化错误）、心智理论缺陷、混合动机动态、级联可靠性失败。级联示例：重试风暴 — 支付失败触发订单重试，触发库存重试，压垮库存服务（10 倍负载在数秒内 — 需要熔断器）。内存污染：一个 Agent 的幻觉进入共享内存，下游 Agent 将其当作事实；准确率逐渐衰减，使根因诊断非常痛苦。**STRATUS**（NeurIPS 2025）报告通过专门的检测/诊断/验证 Agent 将缓解成功率提升 1.5 倍。本课将失败模式作为一等工程目标来对待。

**类型：** 学习
**语言：** Python（标准库）
**时间：** 约 75 分钟

## 问题

多 Agent 系统在真实任务上的失败率高达 41-86.7%（Cemri 等 2025 年测量了 7 个开源 MAS）。这不是"再加几个 Agent"就能调试的。失败具有结构性原因。MAST 分类法给出了类别。本课将每个类别映射到具体的检测、诊断和缓解模式，让这些数字不再是随机的。

2026 年的生产实践是将失败模式作为设计输入。你的架构在能够指出每个 MAST 类别并说出你部署的缓解措施之前，都不算"够好"。

## 概念

### MAST 类别

**规范问题（失败中的 41.77%）。** Agent 的任务定义不够严格。示例：

- 角色模糊：两个 Agent 都认为自己是审查者。
- 任务未充分指定："总结这个"，而用户想要特定角度。
- 成功标准隐含：Agent 无法判断自己是否成功。

缓解措施：
- 编写明确的角色合同。每个 Agent 的提示说明它做什么*和不做什么*。
- 每个任务的验收测试。在 Agent 开始前，定义"完成意味着 X"。
- 预检规范检查：一个独立 Agent 在分派前审查任务定义。

**协调失败（36.94%）。** 通信或状态故障。

示例：
- 两个 Agent 无同步地更新共享状态。
- Agent 之间消息丢失（队列故障、超时）。
- 状态漂移：Agent A 认为任务已完成；Agent B 仍在执行。

缓解措施：
- 带乐观并发的版本化共享状态。
- 对关键消息的显式确认（重试直到确认）。
- 定期状态同步检查点；及早检测漂移。

**验证缺口（21.30%）。** 对输出没有独立检查。

示例：
- 一个 Agent 声称成功；没有人验证。
- Agent 链中每个都信任前一个的输出。
- 对涌现的组合行为缺少测试覆盖。

缓解措施：
- 独立验证 Agent（第 13 课）。只读、独立来源访问。
- 显式交接合同："A 的输出必须通过检查器 C 的验证，B 才能开始。"
- 用于事后分析的输出日志。

### 群体思维族（arXiv:2508.05687）

当 Agent 同质化或互相模仿时，五种相关失败：

**单一文化崩溃。** 相同基座模型或训练数据 → 相关错误。三个 Agent 共享一个 LLM 时，也共享它的幻觉。

**从众偏差。** Agent 倾向于最响亮或最自信的同辈，即使后者是错的。

**心智理论缺陷。** Agent 无法对彼此信念建模；协调失败（第 18 课）。

**混合动机动态。** 具有部分对齐激励的 Agent 漂向折中中间方案，任何人都不满意。

**级联可靠性失败。** 一个组件的错误模式触发依赖组件的错误模式。

### 级联示例 — 重试风暴

2026 年的经典事故模式：

```
支付服务 10% 的请求失败
   ↓
订单 Agent 重试支付（指数退避但实现粗糙）
   ↓
每次重试都触发新的订单-库存检查
   ↓
库存服务负载翻倍
   ↓
库存服务开始超时
   ↓
每个订单都重试库存检查
   ↓
库存服务负载达 10 倍正常值
   ↓
集群宕机
```

修复是经典的：**熔断器**。当下游错误率超过阈值时，直接短路返回缓存或默认结果。加上每个请求的重试预算上限。

熔断器是少数几个你直接从分布式系统借用到多 Agent 失败缓解中的工具，无需修改。

### 内存污染（重访）

来自第 13 课：一个 Agent 的幻觉成为共享内存中的事实；下游 Agent 在这个中毒事实上推理。按 MAST 分类，这是共享内存层的验证缺口。

症状是准确率逐渐衰减。你不会看到崩溃；而是一个缓慢的漂移，难以根因定位。

缓解措施：追加日志、溯源记录、不可写的验证者。已在第 13 课涵盖。

### STRATUS — 专门化失败检测 Agent

STRATUS（NeurIPS 2025）报告说，部署以下角色后缓解成功率提升 1.5 倍：

- **检测 Agent。** 监控症状模式（高分歧、重试激增、准确率漂移）。
- **诊断 Agent。** 给定症状后，从 MAST 分类法推断可能的根因。
- **验证 Agent。** 缓解措施应用后，检查症状是否消失。

这是 SRE 风格的事故响应，应用于 Agent 系统。三个角色可以全部是带有专门提示的 LLM Agent。

### 失败模式审计

2026 年的一个最佳实践是年度（或每个主版本）失败模式审计：

1. **采样跟踪。** 收集约 1000 条真实执行轨迹。
2. **分类。** 将每个轨迹的失败映射到 MAST + Groupthink 类别。
3. **计算按类别的失败率。** 哪些类别在你的系统中占主导？
4. **排序缓解措施。** 哪项修复能消除最多的失败？
5. **选择 2-3 项缓解措施。** 实施；下一季度重新审计。

纪律比具体选择更重要。没有审计，失败会淹没在噪音中，永远不会被系统性地处理。

### 当系统悄然失败时

最危险的失败类别是静默正确性失败。一个喧闹地失败的系统（崩溃、异常、告警）可以被监控。一个产生"看似合理但错误"输出的系统无法被异常日志检测到。这就是为什么验证缺口按每次失败的成本是最昂贵的类别，尽管按数量只占 21.30%。

投资于：
- 基于采样的审查。
- 黄金数据集回归测试。
- 重要输出的跨 Agent 交叉检查。

### 失败 vs 缓慢失败

有些失败是即时的；有些是缓慢的。即时失败（超时、模式不匹配、认证错误）容易检测。缓慢失败（内存污染、单一文化漂移、角色模糊）难以检测和预防。

2026 年的工程手段：对缓慢失败代理指标进行仪表化，以便在漂移变成可见错误之前捕获。一致率、重试率、输出长度分布、连续 Agent 版本之间的编辑距离都是有用的代理指标。

## 上线纪律

生产环境中的失败模式纪律：

- **每季度 MAST 审计。** 不是每年。类别会随着系统增长而变化。
- **到处加熔断器。** 每个对从属服务的外部调用。默认开启阈值在 5-10% 错误率。
- **黄金数据集。** 小规模、高质量、手工审计。每周回归测试。
- **STRATUS 三重奏。** 检测 + 诊断 + 验证 Agent 监控生产环境。先从检测 Agent 开始；症状嘈杂时再加诊断。
- **失败预算。** 按类别明确的失败率 SLO。超出预算触发停止发布的讨论。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇少有的"从生产事故角度"审视多 Agent 系统的文档，质量极高。MAST 分类法（41-86.7% 的失败率数据）来源于 NeurIPS 2025 的严格实证研究，为讨论提供了学术底气。Groupthink 族的五种模式补充了 MAST 未能覆盖的心理层面失败。文档结构从"数据震撼"（86.7% 的失败率数字）开始，逐步展开分类、诊断、缓解，最后给出工程纪律。最大亮点是重试风暴的逐步可视化，每一步都列出了具体的连锁反应，让抽象的"级联失败"变得可触达。不足在于 STRATUS 三重奏的代码模拟偏脚本化，真实实现要复杂得多。

### 二、知识结构梳理

- **分类层**：MAST 三大根类别（规范问题 41.77%、协调失败 36.94%、验证缺口 21.30%）及其子类型。Groupthink 族的五种同质化失败模式。
- **案例层**：重试风暴的逐帧级联（支付 → 订单 → 库存 → 集群宕机），内存污染的缓慢衰减曲线。即时失败 vs 缓慢失败的检测难度差异。
- **工程层**：熔断器（经典分布式系统模式直接借用到 Agent 系统），STRATUS 三重奏（检测→诊断→验证），失败预算和审计纪律，缓慢失败代理指标。

### 三、核心洞察

1. **41-86.7% 的失败率是警钟。** 这不是"再加几个 Agent"能解决的问题，必须从建筑设计层面把失败模式作为一等输入来处理。
2. **规范问题是最大的失败源（41.77%），但最容易被忽视。** 工程师倾向于优化代码和架构，但模糊的角色定义和隐含的成功标准才是杀手。写"不做什么"比写"做什么"更难但更重要。
3. **验证缺口虽然只占 21.30%，但静默正确性失败是最昂贵的。** 一个产生"看似合理但错误"输出的系统无法被异常日志检测到。每单位失败成本远高于即时崩溃。
4. **熔断器是跨领域通用模式。** 从分布式系统到 Agent 系统，熔断器的语义完全一致：检测错误率、短路、降级。这是少数不需要"Agent 特化"的工程手段。
5. **缓慢失败的隐蔽性使其更危险。** 内存污染的准确率衰减曲线是月级别的，根因诊断极其困难。必须用代理指标（一致率、重试率、编辑距离）提前捕捉漂移。
6. **STRATUS 三重奏把 SRE 理念引入了 Agent 世界。** 检测、诊断、验证三个角色可以用 LLM Agent 实现，形成自动化的故障响应流程。这个模式与人类的 On-call 流程高度对应。
7. **失败预算是一个务实的治理工具。** "超过 5% 的验证缺口 → 停止发布"这种硬性规则比模糊的"提高质量"目标有效得多。

### 四、教学建议

1. **开场用 86.7% 的失败率数字制造冲击。** 先展示数据，再问学员"你们的 Agent 系统有这么多失败吗？"很可能没有 — 因为根本没测量。这个认知落差本身就是学习动机。
2. **重试风暴的逐帧展示值得做成动画。** 每一步负载翻倍的连锁反应如果可视化，比文字描述理解速度快 10 倍。
3. **熔断器的 code/main.py 演示要先跑"没有熔断器"的版本。** 让学员亲眼看到库存错误从 10% 涨到 100%，再打开熔断器看到封顶。对比效果是唯一的说服方式。
4. **MAST 审计练习建议用学员自己的系统。** 如果没有，用 LangGraph 或 CrewAI 的标准示例。让学员真实地在代码中找规范模糊、状态漂移、验证缺失的点。
5. **缓慢失败的代理指标是一个很好的讨论话题。** 让学员提出除了文档中列的那些之外，还有什么指标可以捕捉缓慢漂移。鼓励创造性思考。
6. **"黄金数据集"的概念需要强调。** 很多人用自动生成的测试集，但黄金数据集必须是手工审计的。10 条手工标注的高质量用例胜过 1000 条自动生成的。
7. **STRATUS 三重奏的教学可以角色扮演。** 三个人分别扮演检测/诊断/验证 Agent，模拟一次事故响应流程。比代码演示更形象。

### 五、值得补充的内容

1. **MAST 分类法与 FMEA（失效模式与影响分析）的对比。** FMEA 是传统工程中的经典方法，MAST 是 AI 特化版本，对比有助于跨学科学员理解。
2. **各品类 LLM 对失败模式的敏感性差异。** 同一架构的 Agent，用 GPT-5、Claude 4、Gemini 2.5 作为基座，失败分布是否不同？这是前沿问题值得提及。
3. **熔断器参数的调优方法。** 错误率阈值设多少（5%？10%？）、半开状态的探测频率、恢复策略 — 这些参数有经验法则。
4. **内存污染的定量模型。** 一个幻觉进入共享内存后，经过多少轮会扩散到多少下游 Agent？可以给一个简化的传染模型。

### 六、一句话总结

**多 Agent 系统的失败不是意外，是设计的必然结果 — MAST 分类法给了你分类的刀，熔断器和 STRATUS 三重奏给了你止血的工具，失败预算和审计给了你持续改进的纪律。**

# 🎓 Agent 架构课：多 Agent 失败模式——你的系统不会爆炸，它会悄悄变蠢
同学们好。我是你们的FDE工程老师，今天讲的是 多 Agent 故障。

**2026 年 MAST 分类法从 7 个开源 MAS 的 1,642 条执行轨迹中提取：41-86.7% 的失败率。** 你的多 Agent 系统不会爆炸——它会悄悄变蠢。

三类根因：① **规范问题（41.77%）**——角色模糊，两个 Agent 都觉得自己是审查者。② **协调失败（36.94%）**——通信中断，状态不同步。③ **验证缺口（21.30%）**——没有质量检查，Agent 自己 review 自己的输出。

群体思维族补充了单一文化崩溃（相同基座模型→相关失败）、从众偏差（Agent 互相强化错误）、级联可靠性失败（支付失败→订单重试→库存重试→库存服务被压垮，10 倍负载在数秒内）。内存污染是最难诊断的：一个 Agent 的幻觉进入共享内存，下游 Agent 当事实用，准确率逐渐衰减。

STRATUS（NeurIPS 2025）通过专用检测/诊断/验证 Agent 将缓解成功率提升 1.5 倍。

## 结课清单

1. **41-86.7% 的失败率是实测数字。** 不是感觉——是 MAST 论文的统计。
2. **规范问题占大头。** 角色写清楚"做什么和不做什么"比加更多 Agent 更有效。
3. **验证者是承重角色。** 没有客观验证的多 Agent 系统 = 没有刹车的车。
4. **级联失败 = 需要熔断器。** 一个服务的超时不应该是整个系统的雪崩。

最后一句话：**多 Agent 系统最危险的失败模式不是崩溃——是悄悄变蠢而你发现不了。**

# 💼 从业者故事：多 Agent 失败模式——你的系统不会爆炸，它会悄悄变蠢

去年 11 月，凌晨 3:47，手机响了。

不是某台机器挂了那种响——那个我见多了。是"用户投诉量突然从 2 涨到 47"那种响。我打开监控一看：所有指标都是绿的。CPU 正常，内存正常，响应时间正常。但用户说我们的 Agent 在胡说八道。

我盯着日志盯了 40 分钟才发现问题：共享内存里有一条幻觉数据，是 6 个小时前一个 Agent 生成的。从那之后，每一个下游 Agent 都在基于这条假数据做推理。没有崩溃、没有异常、没有告警——系统在"正常"地输出垃圾。

那次之后我才真正理解了 MAST 分类法里那句话：验证缺口虽然只占 21.3% 的失败量，但按每次失败的代价算，它是三类里最贵的。它不是让你系统挂掉，是让你系统在你不知道的情况下变蠢。

## 86.7% 的失败率不是 Bug，是设计参数

Cemri 那帮人在 NeurIPS 2025 发了个论文，测了 7 个开源 MAS 系统，41% 到 86.7% 的失败率。我第一反应是"这数字也太离谱了吧"。然后我测了我们自己的系统——54%。没测之前我觉得自己写得挺好的，测完才知道那叫"没测过所以不知道有多差"。

MAST 把失败分成三坨：规范问题 41.77%（人没交代清楚）、协调失败 36.94%（Agent 之间传话传丢了）、验证缺口 21.30%（没人检查就放行了）。这个比例跟我遇到的实际故障高度吻合。不是 Agent 不行，是我们给 Agent 的指令不够严格、Agent 之间的通信不够健壮、以及最关键的一点——我们太信任 Agent 的输出了。

你在写 Prompt 的时候有没有写过"总结一下这个文档"？我写过。然后三个 Agent 给了三个完全不同的总结，因为用户想要的是"技术层面的总结"而我没写。规范问题的本质不是写少了，是隐含了太多你以为 Agent 会自动理解的上下文。Agent 不会自动理解，它只会按字面意思执行你写的每一个字。

## 单一文化崩溃：三个 Agent，一个幻觉

Groupthink 那篇论文里有一个概念叫"单一文化崩溃"，说白了就是你用同一个基座模型跑三个 Agent，它们共享同一个模型的盲区。我经历过一个经典的案例：一个用户问了一个比较小众的法律问题，我们的 GPT-4 基座模型在那个领域有一个系统性的知识缺陷。三个 Agent——研究员、分析员、审核员——全部基于同一个错误知识给出了"看起来非常自信"的答案。用户照做了，然后投诉了。

三个 Agent 不是"三个独立判断"，是同一个大脑的三张嘴。你可以给它们不同的 System Prompt、不同的工具、不同的上下文窗口，但只要基座模型共享，它们的盲区就共享。这就是为什么 Anthropic 的多 Agent 研究系统里混用了不同模型——不是为了炫技，是被坑过。

## 重试风暴：分布式系统的经典事故在 Agent 系统里一样精彩

文档里那个重试风暴的级联描述我看着特别亲切，因为我亲眼见过几乎一模一样的事情发生。支付 10% 失败 → 订单重试 → 每次重试触发库存检查 → 库存服务慌了 → 更多超时 → 更多重试 → 集群挂了。从第一个支付失败到集群挂掉，只用了 14 秒。14 秒！连报警都来不及触发。

熔断器这个事，说起来特别简单，做起来也特别简单——但 90% 的团队上线的时候没配。为什么？因为"我们才刚上线，先跑通了再加"。然后等你真遇到事了，熔断器的优先级永远排在用户体验功能和

"只加一个 API"的后面。我的建议是：把你系统里每一个对外部服务的调用都加上熔断器，现在，立刻，不要等到上线。错误率阈值设 5%，半开状态探测间隔设 30 秒。这个配置可能不是最优的，但比不配强一万倍。

## STRATUS 三重奏：把 SRE 值班流程自动化了

检测 Agent 看症状、诊断 Agent 找根因、验证 Agent 确认修复——这就是人类 On-call 的标准流程，只不过 STRATUS 把它用三个专门化的 LLM Agent 实现了。我第一次看到这个架构的时候笑了：这不就是把我凌晨 3:47 的调试流程自动化了吗？先看监控（检测），然后翻日志找 root cause（诊断），修完以后跑个回归测试（验证）。

STRATUS 论文说这个架构把缓解成功率提升了 1.5 倍。这个数字听起来不大，但你要知道，多 Agent 系统的故障通常从发生到被发现平均要 23 分钟（我们自己的数据），而从发现到修复还要 40 分钟。1.5 倍的缓解成功率意味着你每年少被凌晨电话叫醒很多次。

## 我的三条血泪教训

第一，**规范问题是最大的失败源，也是最便宜的修复。** 写"这个 Agent 不做什么"比写"这个 Agent 做什么"更难，但能消灭 40% 的故障。我现在的模板是：每个 Agent 的 Prompt 里必须有一个"明确排除"的 section，列出它不应该处理的情况。

第二，**缓慢失败比即时失败危险十倍。** 内存污染、单一文化漂移、角色模糊——这些东西不会报警，它们让你的系统缓慢而优雅地变蠢。唯一的防御手段是代理指标：一致性率、重试率、输出长度的编辑距离。如果你的 Agent 系统只监控绿色/红色，没监控"灰度漂移"，你离凌晨电话不远了。

第三，**失败预算要写成硬性规则。** "验证缺口超过 5% → 停止发布"比"我们要提高质量"有效一万倍。模糊的目标等于没有目标。我们有具体的 SLO：规范问题 P0 故障占比不超过 10%，协调失败 MTTD（Mean Time To Detect）低于 180 秒。数字管人，口号骗人。

多 Agent 系统不是不会失败——它们注定会失败，而且是以你想象不到的方式。你的工作不是消除失败，是让失败变得可见、可追踪、可止血。就像 Nygard 在 Release It! 里说的：**你不会阻止每一场火灾，但你可以决定它是被烟雾报警器叫醒的，还是被火焰叫醒的。**
