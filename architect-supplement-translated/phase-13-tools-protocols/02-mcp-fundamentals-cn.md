# MCP 基础——基元、生命周期、JSON-RPC 基础

> MCP之前的每一个集成都是一次性方案。Model Context Protocol（模型上下文协议），由Anthropic于2024年11月首次发布，现由Linux基金会下的Agentic AI Foundation管理，标准化了发现和调用，让任何客户端都能与任何服务器通信。2025-11-25规范定义了六个基元（三个服务器端，三个客户端）、一个三阶段生命周期和JSON-RPC 2.0线格式。学会这些，本阶段MCP章节的其余内容就变成了阅读。

**类型：** 学习
**语言：** Python（标准库，JSON-RPC解析器）
**前置条件：** 第13阶段 · 01到05（工具接口和函数调用）
**时间：** ~45分钟

## 学习目标

- 说出所有六个MCP基元（服务器端：工具、资源、提示；客户端：根、采样、启发），并各给一个使用场景。
- 走过三阶段生命周期（初始化、操作、关闭），说出每个阶段谁发送什么消息。
- 解析和发出JSON-RPC 2.0请求、响应和通知信封。
- 解释 `initialize` 中的能力协商是什么，以及没有它会出什么问题。

## 问题

在MCP之前，每个使用工具的Agent都有自己的协议。Cursor有类似MCP但不兼容的工具系统。Claude Desktop用的是另一套。VS Code的Copilot扩展用的是第三套。一个构建了"Postgres查询"工具的团队要写三次相同的工具，每次都面向不同主机的API。复用它需要复制代码。

结果是一场一次性集成的寒武纪大爆发，和生态系统速度的天花板。

MCP通过标准化线格式修复了这个问题。一个MCP服务器能在每一个MCP客户端中工作：Claude Desktop、ChatGPT、Cursor、VS Code、Gemini、Goose、Zed、Windsurf，到2026年4月超过300个客户端。1.1亿月SDK下载量。10,000+公开服务器。Linux基金会在2025年12月的Agentic AI Foundation下接管了管理。

本阶段使用的规范版本是 **2025-11-25**。它添加了异步任务（SEP-1686）、URL模式启发（SEP-1036）、带工具的采样（SEP-1577）、增量范围同意（SEP-835）和OAuth 2.1资源指示符语义。第13阶段 · 09到16覆盖这些扩展。本课止于基础。

## 概念

### 三个服务器基元

1. **工具。** 可调用动作。与第13阶段 · 01相同的四步循环。
2. **资源。** 暴露数据。通过URI寻址的只读内容：`file:///path`、`db://query/...`、自定义方案。
3. **提示。** 可复用模板。主机UI中的斜杠命令；服务器提供模板，客户端填入参数。

### 三个客户端基元

4. **根。** 服务器被允许触及的URI集合。客户端声明它们；服务器尊重它们。
5. **采样。** 服务器请求客户端的模型执行一次补全。启用无需服务器端API密钥的服务器托管Agent循环。
6. **启发。** 服务器在运行中向客户端用户请求结构化输入。表单或URL（SEP-1036）。

MCP中的每个能力恰好属于这六个之一。第13阶段 · 10到14深入覆盖每一个。

### 线格式：JSON-RPC 2.0

每条消息是一个JSON对象，带有这些字段：

- 请求：`{jsonrpc: "2.0", id, method, params}`。
- 响应：`{jsonrpc: "2.0", id, result | error}`。
- 通知：`{jsonrpc: "2.0", method, params}`——没有 `id`，不期望响应。

基础规范有约15个方法，按基元分组。重要的有：

- `initialize` / `initialized`（握手）
- `tools/list`、`tools/call`
- `resources/list`、`resources/read`、`resources/subscribe`
- `prompts/list`、`prompts/get`
- `sampling/createMessage`（服务器到客户端）
- `notifications/tools/list_changed`、`notifications/resources/updated`、`notifications/progress`

### 三阶段生命周期

**阶段1：初始化。**

客户端发送 `initialize`，附带其 `capabilities` 和 `clientInfo`。服务器以其 `capabilities`、`serverInfo` 和其使用的规范版本响应。客户端消化响应后发送 `notifications/initialized`。从此以后，任何一方都可以根据协商的能力发送请求。

**阶段2：操作。**

双向。客户端调用 `tools/list` 发现，然后调用 `tools/call` 执行。如果服务器声明了能力，可以发送 `sampling/createMessage`。当工具集变化时，服务器可以发送 `notifications/tools/list_changed`。当用户更改根范围时，客户端可以发送 `notifications/roots/list_changed`。

**阶段3：关闭。**

任一方关闭传输。MCP中没有结构化的关闭方法；传输层（stdio或Streamable HTTP，第13阶段 · 09）携带连接结束信号。

### 能力协商

`initialize` 握手中的 `capabilities` 是合约。服务器示例：

```json
{
  "tools": {"listChanged": true},
  "resources": {"subscribe": true, "listChanged": true},
  "prompts": {"listChanged": true}
}
```

服务器声明它可以发出 `tools/list_changed` 通知并支持 `resources/subscribe`。客户端通过声明自己的来同意：

```json
{
  "roots": {"listChanged": true},
  "sampling": {},
  "elicitation": {}
}
```

如果客户端没有声明 `sampling`，服务器不得调用 `sampling/createMessage`。对称：如果服务器没有声明 `resources.subscribe`，客户端不得尝试订阅。

这就是阻止生态系统漂移的机制。一个不支持采样的客户端仍然是有效的MCP客户端；一个不调用 `sampling` 的服务器仍然是有效的MCP服务器。它们只是不使用该功能。

### 结构化内容和错误形状

`tools/call` 返回一个 `content` 数组，包含类型块：`text`、`image`、`resource`。第13阶段 · 14向该列表添加了MCP应用（`ui://` 交互UI）。

错误使用JSON-RPC错误码。规范定义的附加码：`-32002`"找不到资源"，`-32603`"内部错误"，加上MCP特定的错误数据作为 `error.data`。

### 客户端能力 vs 工具调用细节

一个常见混淆：`capabilities.tools` 是客户端是否支持工具列表变更通知。客户端是否会调用特定工具是一个运行时选择，由其模型驱动，不是一个能力标志。能力标志是规范级别的合约。模型的选择是正交的。

### 为什么用JSON-RPC而不是REST？

JSON-RPC 2.0（2010年）是一个轻量级双向协议。REST是客户端发起的。MCP需要服务器发起的消息（采样、通知），所以JSON-RPC以其对称的请求/响应形状是自然的适配。JSON-RPC还可以干净地在stdio和WebSocket/Streamable HTTP上组装，无需重新发明HTTP的请求形状。

## 使用

`code/main.py` 提供一个最小的JSON-RPC 2.0解析器和发射器，然后手工走过 `initialize` → `tools/list` → `tools/call` → `shutdown` 序列，打印每条消息。没有真实传输；只有消息形状。对照延伸阅读中链接的规范验证每个信封。

要看的内容：

- `initialize` 双向声明能力；响应有 `serverInfo` 和 `protocolVersion: "2025-11-25"`。
- `tools/list` 返回一个 `tools` 数组；每项有 `name`、`description`、`inputSchema`。
- `tools/call` 使用 `params.name` 和 `params.arguments`。
- 响应 `content` 是 `{type, text}` 块数组。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------|---------|
| MCP | "模型上下文协议" | 模型到工具发现和调用的开放协议 |
| 服务器基元 | "服务器暴露什么" | 工具（动作）、资源（数据）、提示（模板） |
| 客户端基元 | "客户端让服务器用什么" | 根（范围）、采样（LLM回调）、启发（用户输入） |
| JSON-RPC 2.0 | "线格式" | 对称的请求/响应/通知信封 |
| `initialize` 握手 | "能力协商" | 第一条消息对；服务器和客户端声明它们支持的功能 |
| `tools/list` | "发现" | 客户端询问服务器其当前工具集 |
| `tools/call` | "调用" | 客户端要求服务器以参数执行一个工具 |
| `notifications/*_changed` | "突变事件" | 服务器告诉客户端其基元列表已变化 |
| 内容块 | "类型化结果" | 工具结果中的 `{type: "text" | "image" | "resource" | "ui_resource"}` |
| SEP | "规范演化提案" | 命名草案提案（例如异步任务的SEP-1686） |

## 📝 教师备课总结与读后感

### 一、文档整体评价
这是一份"协议即产品"思维的优秀教材。文档不试图让学生读80页规范，而是提取六个基元+三阶段生命周期+JSON-RPC 2.0三个核心概念，用精确的JSON示例展示每条消息的形态。能力协商作为"互操作性的基石"被置于核心位置，呼应了整篇文档的价值观——标准化不是规范写得好，是生态参与者知道彼此能做什么。

### 二、知识结构梳理
- **基础层**：JSON-RPC 2.0的三个信封类型（请求/响应/通知）、六基元的分类逻辑（服务器暴露 vs 客户端提供）、线格式的精确定义
- **模式层**：三阶段生命周期（初始化→操作→关闭）的设计模式、对称能力协商作为生态互操作性的契约模式、服务器主动推送(`notifications/*_changed`)的异步通信模式
- **应用层**：`tools/list`→`tools/call`的发现-调用循环、能力标志的声明与验证、MCP在300+客户端/10000+服务器生态中的实际运行

### 三、核心洞察
1. **"六个基元"不是分类学的产物，是设计约束的穷举**——工具（做）、资源（看）、提示（模板）、根（安全边界）、采样（回调LLM）、启发（问用户）。每一类对应Agent交互中的一个不可回避的需求。任何人设计Agent协议都会落到这六个桶里。
2. **能力协商是MCP的核心创新，不是规范附录**——对称的能力声明（服务器说"我能做X"，客户端说"我能接收X"）让"部分兼容"成为合法的兼容方式。一个不支持采样的客户端仍是有效MCP客户端。这种设计避免了协议膨胀成单一"全功能"怪物。
3. **JSON-RPC 2.0选择是在"双向性"和"简单性"之间的折中**——REST单向、WebSocket但需要额外握手机制。JSON-RPC 2010年已经在LSP等协议中验证过，Anthropic选它不是因为技术先进，是因为它足够简单且方向对了。
4. **SEP机制是MCP从"公司协议"到"社区标准"转变的标志**——Linux基金会管理、SEP演化模型、多个SEP在2025-11-25版本中被纳入，这是MCP从一次性的公司开源转向可持续社区治理的关键信号。
5. **`notifications/*_changed` 是状态同步的基石**——工具的列表可以动态变化（服务器热加载新工具），而客户端需要实时感知。这个看似微小的设计确保了MCP是动态的、活的协议，而非一次性工具注册表。
6. **内容块的类型化设计是对未来的预留空间**——2025-11-25规范定义了 `text`、`image`、`resource`，第13阶段 · 14加入 `ui_resource`。这个可扩展的设计确保今天写MCP服务器的人不会被未来的交互UI扩展甩下。

### 四、教学建议
1. **用"万能遥控器"做类比开场**——每个电视品牌有自己的遥控器，你永远找不到对的按钮。然后出现了一个标准协议（HDMI-CEC）：任何遥控器能操作任何电视。MCP和HDMI-CEC的故事一模一样，但学生们都买过电视——类比瞬间穿透。
2. **让学生手写JSON-RPC消息**——不依赖代码，先在纸上写出一条完整的 `initialize` 请求和响应。一旦学生会手写，协议就不再是"配置"，而是"可以调试的东西"。
3. **角色扮演三阶段生命周期**——三个学生分别扮演"客户端"、"服务器"和"观察者"。客户端发 `initialize`，服务器回 `capabilities` + `serverInfo`，客户端回 `notifications/initialized`。然后进入操作阶段随意发请求。游戏化+体感记忆 > 阅读记忆。
4. **能力协商做"缺失实验"**——用 `code/main.py`，故意删除服务器能力声明中的 `tools.listChanged`，让客户端尝试发送这个通知，观察会发生什么。失败是最好的老师。
5. **六个基元的"缺失练习"**——给出5个场景（如"需要用户批准操作"、"需要知道当前文件系统路径"、"需要生成一个常用提示模板"），让学生映射到六个基元。如果映射不上，讨论这个场景是否超出MCP范围。
6. **对比MCP与其他Agent协议**——LangChain Tool、Semantic Kernel Plugin、OpenAI Function Calling。画一张表对比六个基元在每个框架中的等价物和缺失项。对比建立协议的"为什么更好"。
7. **让学生读一节真正的规范文本**——80页规范不要全读，但选 "Architecture concepts" 章节让学生精读10分钟，然后用自己的话在课堂上讲给同学听。教授是最好的学习方式。

### 五、值得补充的内容
1. **MCP与LSP（语言服务器协议）的家族相似性**——LSP标准化了编辑器和语言工具之间的通信，MCP标准化了Agent和工具之间的通信。两者都是JSON-RPC 2.0、都有初始化握手和能力协商。讲这层关系让学生意识到"这不是第一次有人做协议标准化，模式是成熟的"。
2. **Streamable HTTP vs stdio传输的选择指南**——文档提到了这两种传输方式（第13阶段 · 09覆盖），但没有讲何时选哪种。补充：stdio适合本地单用户工具（如文件系统操作），Streamable HTTP适合远程多租户服务（如云端数据库查询）。选错传输 = 架构回炉。
3. **六基元中"启发(elicitation)"最被低估的原因**——工具可以让Agent做事，但有些决策必须由人类做出（如超大额支付确认）。启发是"人类在循环中"的程序化接口。这应该是Agent架构课中"何时把控制权交还给人"的讨论起点。
4. **MCP的竞争格局**——A2A（Google的Agent-to-Agent协议）和MCP的关系和定位差异。MCP偏"模型到工具"，A2A偏"Agent到Agent"。当学生问"为什么不用A2A"时，你需要的不是防御性回答，是对问题的清晰分类。

### 六、一句话总结
MCP的本质不是规范文档，不是JSON-RPC 2.0，不是六个基元——它是一个承诺：任何MCP客户端的用户，都能用任何MCP服务器提供的工具。而实现这个承诺的，是能力协商——不是"你必须支持一切"，而是"告诉我你能做什么，我们试试看"。

# 🎓 Agent 架构课：协议设计——为什么MCP会成为你下一个十年里最重要的API

"协议"这个词会让你想到什么？

TCP/IP、HTTP、USB、蓝牙。这些都是你每天都在用但从来看不到的东西。它们之所以不可见，是因为它们太稳定了，稳定到你忘了它们被设计出来就是来解决一个根本问题：**互操作性**——A产品能跟B产品说话，而不需要A和B认识同一个开发团队。

**MCP就是Agent世界的USB。**

我来解释这个论断。然后我会告诉你为什么"六个基元"不是任意的——它们恰好覆盖了你的Agent系统在真实运行中必须处理的所有交互。不多不少。六个。

---

## 问题本质：当每个Agent都有自己的工具API

在MCP之前，我见过的最荒谬的事是这个：

一个团队同时用了三个Agent产品——Cursor写代码、Claude Desktop做日常问答、Gemini做长文档分析。他们有一个"查数据库"的工具，需要在三个Agent中都能用。

他们写了三次。三次。

不是复制粘贴——是三个完全不同的API。Cursor的插件系统、Claude的tool-use格式、Gemini的function-calling。同样的SQL查询，三种代码。

**我为什么在乎？** 作为架构师，我看到的是三个问题：重复劳动（浪费工程师时间）、不一致行为（三个实现会有三种bug）和安全盲区（每个都需要独立审计）。而MCP用"一个协议"解决了这三个问题。一口井。

---

## 六基元的设计哲学：为什么是六个而不是四个或八个

让我给你展开这六个基元。不看定义——看"为什么会需要"：

**服务器端：**
1. **工具。** 因为Agent需要做事情。查数据库、发邮件、调API。没有工具你就只有一个聊天机器人。
2. **资源。** 因为Agent需要看东西。文件内容、数据库结果、"你现在在哪个文件夹"。工具是"做"，资源是"看"。
3. **提示。** 因为Agent的提示模板需要在服务器端管理。你有一个"写发布说明"的提示模板，更新一次，所有客户端受益。

**客户端端：**
4. **根。** 因为Agent不能访问你整个文件系统。给数据库服务器 `./data/` 根目录，它就不能看到 `~/.ssh/`。安全边界。
5. **采样。** 因为有些时候，服务器需要反过来让客户端的模型干活。一个数据库服务器想要LLM帮你总结查询结果——它自己没有API密钥，但你的客户端有。
6. **启发。** 因为Agent不能什么事都自己做决定。大额支付、删除用户数据、发送广播邮件——这些事需要人点头。

六个。每一个都不可删除。你试过去掉任何一个吗？

- 没有资源 → 所有数据必须在工具调用参数中传输 → 炸上下文。
- 没有采样 → 服务器必须自带API密钥 → 安全噩梦。
- 没有启发 → 要么Agent什么事都不敢做 → 无用；要么什么都敢做 → 危险。

六个是最小完备集。不是最小集——有四个就不能工作。而是刚好那六个你没法少。

---

## 两条路径：能力协商 vs 全有全无

**路径A：全功能必要性。**

协议说"要兼容，你必须实现全部六个基元和全部子功能"。MCP Server如果少实现一个能力就不合格。这听起来"严谨"，但看看现实：一个"查天气"的工具需要采样吗？不需要。一个"生成发布说明"的提示服务器需要跟用户启发吗？不需要。

全有全无的结果是没人遵守——或者大家实现一堆stub函数，然后在runtime报错。

**路径B：能力协商。**

MCP的方式：服务器说"我有这些能力：tools + tools.listChanged"。客户端说"我有这些能力：roots + sampling + elicitation"。两边知道彼此能做什么。不能做的功能不被调用。

这允许"部分MCP"存在。一个只做工具发现的服务器、一个只支持采样的客户端——仍然是有效的MCP节点。生态不是由"全功能旗舰"组成的——是由"做好一件事的节点"组成的。

**生产教训：** 标准化协议的生命力不取决于"它包含多少功能"，而取决于"缺少某些功能时它是否能优雅降级"。MCP的能力协商是优雅降级的教科书级实现。

## 深入原理：JSON-RPC 2.0为什么是正确的选择

JSON-RPC有一个反常识的优势：它不是REST。

REST是客户端发起、服务器响应。单向。Web就是这样的。

但MCP需要双向。服务器需要主动告诉客户端"我的工具列表变了"（`notifications/tools/list_changed`）。服务器需要反过来请求客户端"帮我用你的LLM跑一次推理"（`sampling/createMessage`）。

这些是服务器→客户端的消息。REST做不到——除非加WebSocket或Server-Sent Events，但那等于重新发明了JSON-RPC已经做好的事情。

JSON-RPC 2.0于2010年标准化。到2024年MCP采用它时，它已经在LSP（语言服务器协议）中服务了数百万开发者。这不是"新技术"——这是"验证过的技术"。Anthropic做了一个聪明的选择：不用自己发明线格式，而是复用已经debug了14年的东西。

## 反模式：把"兼容MCP"当成营销标签

2025年下半年，每家公司都说自己"兼容MCP"。但"兼容MCP"可能意味着：

- "我们的服务器实现了 `tools/list`"（最低限度）
- "我们的服务器实现了全部三个服务器基元"
- "我们的客户端能解析 `tools/call` 的响应"
- "我们的客户端通过了MCP合规套件"

这四个档次的能力差距巨大。而用户看到"兼容MCP"以为它们都一样。

**我的建议：** 要求每个自称"兼容MCP"的产品发布其能力声明。不是"我们兼容"——是"我们支持 `tools`、`resources`、`prompts`、`roots`、`sampling`、`elicitation`。"六个单词。如果他们不能写出具体支持了哪几个，就不是真正的兼容。

## 结语：架构师的决策清单

在你的Agent系统中接入MCP时：

1. **先决定你用哪些基元。** 不是所有6个你都需要。先列出你的真实需求，再映射到基元。最小化能力声明降低攻击面和复杂度。
2. **测试能力协商。** 故意让客户端声明比实际少一个能力，确认服务器的优雅降级不会崩溃。这是比"常规路径测试"更重要的测试。
3. **选传输层时先画架构图。** stdio用于本地、Streamable HTTP用于远程。不要用Streamable HTTP跑本地工具（多一次网络开销），也不要用stdio跑远程工具（安全隔离不够）。
4. **工具列表变更通知必须有。** 如果你的服务器热加载新工具（比如用户上传了自定义脚本），确保发出 `notifications/tools/list_changed`。没有这个通知的MCP系统是一个"需要手动刷新"的系统——这是负UX。
5. **读一次规范中的错误处理章节。** MCP的错误码（`-32002`、`-32603`）和标准HTTP错误码不同。不要在日志里看到 `-32603` 就去搜HTTP 500的含义。

**金句：MCP给你的不是"一个更好的工具调用方式"——它给你的是"所有工具调用方式中的那一个"。"标准化"本身的价值，超过任何单一标准的技术质量。**

想想USB：1996年第一版USB 1.0又慢又笨，但它的存在让串口、并口、PS/2口全部消失。不是因为它更快——是因为它只有一个。MCP在Agent时代的角色，和USB在I/O外设时代的角色一模一样。
