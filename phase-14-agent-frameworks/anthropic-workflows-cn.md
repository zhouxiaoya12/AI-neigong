# Anthropic 工作流模式：简单胜过复杂

> Schluntz 和 Zhang（Anthropic, 2024 年 12 月）区分了工作流（预定义路径）和 Agent（动态工具使用）。五种工作流模式覆盖大多数场景。从直接 API 调用起步。仅在步骤无法预测时才引入 Agent。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 01（Agent 循环）
**用时：** 约 60 分钟

## 学习目标

- 说出 Anthropic 的五种工作流模式：提示链、路由、并行化、编排器-工作者、评估器-优化器。
- 解释 Agent 与工作流的区别以及各自的工程成本。
- 判断何时选工作流而非 Agent（反之亦然）。
- 基于标准库和脚本化 LLM 实现全部五种模式。

## 问题

团队为一个单次函数调用就能解决的问题引入多 Agent 框架。成本是真实存在的：框架增加了层数，模糊了提示词，隐藏了控制流，并催生了过早的复杂性。Schluntz 和 Zhang 在 2024 年 12 月的文章是引用最多的行业反驳：从简单起步，仅在复杂性物有所值时再增加。

## 概念

### 工作流 vs Agent

- **工作流。** LLM 和工具通过预定义代码路径编排。工程师拥有控制图。
- **Agent。** LLM 动态指挥自身的工具并自行决定步骤。模型拥有控制图。

两者各有其用。工作流更便宜、更快、更容易调试。Agent 能解决开放式问题，但故障模式更难推理。

### 增强型 LLM

全部五种模式的基础：一个 LLM 接入三种能力——搜索（检索）、工具（行动）、记忆（持久化）。任何 API 调用都可以使用这些能力。

### 五种模式

1. **提示链（Prompt chaining）。** 第 1 次调用的输出是第 2 次调用的输入。当任务有清晰的线性分解时使用。步骤之间可选择性加入程序化门控。

2. **路由（Routing）。** 一个分类器 LLM 选择调用哪个下游 LLM 或工具。当类别不同的输入需要不同处理时使用（客服 L1 vs 退款 vs bug 报告 vs 销售）。

3. **并行化（Parallelization）。** 同时运行 N 个 LLM 调用，聚合结果。两种形态：分段（不同数据块）和投票（相同提示词，N 次运行，多数投票/综合）。

4. **编排器-工作者（Orchestrator-workers）。** 一个编排器 LLM 动态决定运行哪些工作者（也是 LLM），并综合它们的输出。类似 Agent 循环，但编排器不会无限循环。

5. **评估器-优化器（Evaluator-optimizer）。** 一个 LLM 提案，另一个 LLM 评估。迭代直到评估器通过。这是 Self-Refine（第 05 课）的泛化。

### 工作流胜过 Agent 的场景

- **可预测的任务。** 如果你能枚举所有步骤，就应该枚举。
- **成本受限的任务。** 工作流有有界的步骤数；Agent 可能螺旋。
- **合规受限的任务。** 审计者想要阅读控制图，而非从轨迹中推断。

### Agent 胜过工作流的场景

- **开放式研究。** 下一步取决于上一步返回了什么。
- **变长任务。** 分钟到小时级别的工作，步骤数未知。
- **新领域。** 当你还不知道正确的工作流时——先探索，后固化。

### 上下文工程伴侣

"面向 AI Agent 的有效上下文工程"（Anthropic 2025）是该学科的规范化表述：200k 窗口是预算，不是容器。该包含什么、何时压缩、何时让上下文增长。详见 Phase 14 上下文压缩课程（本课程重新编号前的 Phase 14 第 06 课）。

## 构建

`code/main.py` 基于 `ScriptedLLM` 实现全部五种工作流模式：

- `prompt_chain(input, steps)`——顺序执行。
- `route(input, classifier, handlers)`——分类 + 分发。
- `parallel_vote(prompt, n, aggregator)`——N 次运行，聚合。
- `orchestrator_workers(task, workers)`——编排器选择工作者。
- `evaluator_optimizer(task, proposer, evaluator, max_iter)`——循环直到通过。

运行：

```bash
python3 code/main.py
```

每种模式打印其运行轨迹。每种模式的代码量约为 10-15 行；框架的成本以千行计。

- 当你想获得 Claude Code harness 的形态但不想自己重建时，选择 Claude Agent SDK。

## 练习

1. 实现带置信度阈值的路由。低于阈值 → 升级到人工处理。客服 L1 场景下阈值应该设为多少？
2. 为 `parallel_vote` 添加超时。某次调用挂起时会发生什么？存在缺失投票时如何聚合？
3. 将 `evaluator_optimizer` 改为 bandit：在迭代中保留 top-2 输出，以免晚期出现的好结果被晚期的坏结果覆盖。
4. 组合提示链与路由：一个路由器选择三条链之一。测量令牌成本，与单个大提示词的替代方案对比。
5. 选一个你生产环境中的功能。画出工作流图。数步骤数。这里 Agent 真的会更合适吗？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 工作流（Workflow） | "预定义流程" | 工程师拥有的 LLM 与工具调用图 |
| Agent | "自主 AI" | 模型拥有的控制图；动态工具指挥 |
| 增强型 LLM（Augmented LLM） | "带工具的 LLM" | LLM + 搜索 + 工具 + 记忆；原子单元 |
| 提示链（Prompt Chaining） | "顺序调用" | 第 N 次调用的输出是第 N+1 次调用的输入 |
| 路由（Routing） | "分类器分发" | 选择哪个链/模型处理输入 |
| 并行化（Parallelization） | "扇出" | N 个并发调用；按分段或投票聚合 |
| 编排器-工作者（Orchestrator-workers） | "调度器 Agent" | 编排器 LLM 动态选择专家 LLM |
| 评估器-优化器（Evaluator-optimizer） | "提案者 + 评判者" | 迭代直到评估器通过；Self-Refine 的泛化 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是整个 Phase 14 课程中最"反框架"的一课。它不是技术教程——它是一篇**架构决策指南**，直接回答"你什么时候需要 LangChain？大概率不需要"。文档的独特价值在于：它不教你用工具，它教你判断**什么时候不该用工具**。Anthropic 2024 年 12 月的这篇博客是行业内关于 Agent 架构"从简原则"的最高引文献。适合任何被告知"我们这个项目需要多 Agent 系统"的工程师——这节课给他们反驳的弹药。

### 二、知识结构梳理

**认知基础层：** Agent 循环（Phase 14·01）是本节课的认知前提。学生必须先理解"Agent 是什么"——一个带工具和记忆的主循环——然后才能理解"为什么大多数时候你不需要它"。增强型 LLM（LLM + 检索 + 工具 + 记忆）是所有五种模式的原子单元。

**工程模式层：** 五种工作流是一个复杂度递进序列。提示链 = 线性，路由 = 条件分支，并行化 = 扇出，编排器-工作者 = 动态调度，评估器-优化器 = 迭代收敛。五种模式覆盖了 80% 的 LLM 应用场景，每种模式 10-15 行代码即可实现。

**决策框架层：** 文档的核心不是五种模式的实现，而是"Agent vs 工作流"的决策标准——可预测性、成本约束、合规要求。这是架构技能，不是编码技能。

### 三、核心洞察（备课时的关键理解）

1. **"Agent vs 工作流"的区分本质上是控制权归属问题。** 工作流的控制图由工程师绘制——每一步的调用、分支、聚合都是显式的。Agent 的控制图由模型动态生成——下一步做什么是 LLM 的推理结果。控制权归属直接决定了可调试性、可审计性和令牌成本的可预测性。

2. **增强型 LLM 是该课被最频繁跳过的概念。** 大多数学生直奔五种模式，忽略了基础——单个 LLM 接入搜索、工具、记忆这三种能力后，已经可以解决 60% 的问题。五种工作流模式是在这个原子单元之上增加的编排层。

3. **"从简单起步"不是哲学口号——是工程经济学。** 一个 10 行代码的提示链和一个 2000 行的 LangGraph 实现，在调试时间、入职成本、故障排查难度上的差距是指数级的。架构的复杂性必须由需求的复杂性来支付。

4. **提示链和评估器-优化器是五种模式中使用频率最高的两种。** 提示链覆盖所有"线性分解"任务（起草→编辑→润色），评估器-优化器覆盖所有"有质量门控"的任务（代码→测试→修复）。其余三种（路由、并行化、编排器）是特定场景的降本/提速手段。

5. **编排器-工作者是最容易误用为 Agent 的模式。** 它看起来像 Agent（动态选择工作者），但它不是——编排器不会无限循环，步骤数有界。如果你发现自己在给编排器加"最多重试 N 次"的逻辑，你就已经在造 Agent 了。

6. **"上下文工程"是该课的隐藏主题。** 200k 上下文窗口不是让你把所有东西都塞进去的——它是预算，需要用压缩、检索和记忆管理来分配。五种工作流模式隐含了一个上下文管理假设：每次 LLM 调用只拿到它需要的上下文，而不是全量。

7. **文档的胆量在于它来自 Anthropic——一个卖 Agent SDK 的公司。** 一篇题为"大多数时候你不需要 Agent"的文章，出自一家正在卖 Agent SDK 的公司。这种坦诚赋予了它极高的可信度，也是它在行业内被反复引用的原因。

### 四、教学建议

1. **开场直接展示对比。** 一个任务用直接 API 调用 3 行解决 vs. 用 LangGraph 200 行实现。让学生看到差距——在讲任何模式之前先建立"不是所有问题都需要复杂方案"的认知锚点。

2. **五种模式用"什么时候你觉得需要它"来引入，而不是用定义。** 提示链——当你发现上一个输出是下一个输入的时候。路由——当你发现不同的问题需要不同的回答风格。让学生先感觉到需求，再给模式命名。

3. **增强型 LLM 值得花 5 分钟单讲。** 让学生写一个带工具调用的 LLM 主循环（就一个 while 循环 + 解析 + 调用工具），然后说"这就是你的原子单元，五种模式都是在这上面加的编排。"

4. **"Agent vs 工作流"的决策标准适合用表格争论。** 在黑板上画一个表：左边列"什么时候用工作流"，右边列"什么时候用 Agent"。每行一个场景（可预测/不可预测、成本敏感/成本无关、合规/探索），让学生填。争论本身就是学习。

5. **现场编码演示比 PPT 重要。** 五种模式每种 10-15 行——完全可以在课堂上现场写。看到提示链从一个 `chain([call1, call2, call3])` 变成 5 行代码，比看架构图有说服力得多。

6. **练习 5（对比你自己的功能）是最有价值的作业。** 让学生拿一个真实项目中的功能，画工作流图，数步骤，然后问自己"这里用 Agent 真的更好吗？"这是他们离开课堂后唯一会做的事。

7. **讨论 LangGraph/LangChain 的反模式。** 让学生分享自己见过/写过的"为了一个 if-else 引入了 300 行框架配置"的经历。这不是技术批评——是架构卫生教育。

### 五、值得补充的内容

1. **工作流到 Agent 的渐进迁移路径。** 文档说"先工作流，后 Agent"，但没说具体怎么迁移。需要一个具体的迁移 checklist：① 找到工作流中"不知道下一步该做什么"的分支点 → ② 把该点的 LLM 从单步决策升级为循环决策 → ③ 在循环中加入退出条件和成本上限。

2. **五种模式的令牌成本对比。** 提示链 = N 次调用，并行化 = N 次调用（并发但总量相同），路由 = 2 次调用（分类 + 处理），编排器 = (1 + K) 次调用，评估器-优化器 = 2×M 次调用。给出每种模式的令牌预算公式，学生才能做成本决策。

3. **评估器-优化器与 Voyager 的对应关系。** 第 09 课的 Voyager 中，迭代精炼就是评估器-优化器的一个实例。这是跨课串联的绝佳材料。

4. **混合模式：一个生产系统很少只用一种模式。** 实际系统通常是路由 + 提示链（路由选一个链）或 编排器 + 评估器（编排器分配任务，评估器检查质量）。

5. **LangGraph 的权衡分析。** 什么时候 LangGraph 真的物有所值？答案是：① 需要跨节点持久化状态 AND ② 需要可中断/可恢复的执行 AND ③ 需要人类在环审批。大多数项目只满足 0-1 个条件。

### 六、一句话总结

**Agent 架构的第一原则不是"让它更自主"——而是"不要在 10 行代码能解决的问题上引入 1000 行框架。"**

# 🎓 Agent 架构课：为什么你不需要 Agent——五种工作流模式的真相

同学们好。我是你们的 Agent 架构老师。

今天这节课的标题可能会冒犯一些人。我准备跟你们说一句反直觉的话：**在你职业生涯的绝大多数时间里，你不需要 Agent。**

我知道你可能刚看完一个 LangChain 教程、刚装完 CrewAI、刚搭建了一个"多 Agent 协作系统"的 demo。我今天不是要告诉你这些工具是错的。我是要告诉你——**在选它们之前，先检查你是不是在用一个 10 行函数能解决的事，买了一个 1000 行的框架。**

## 一个我们都要承认的事实

有一个真实现象，我希望你们记在心里。任何加过框架的人——无论是 React、Django 还是 LangChain——都知道这个规律：

**一个框架最少把代码行数乘以 5，把调试时间乘以 3，把入职成本乘以 10。**

如果你需要框架的能力，这些成本是值得的。如果你不需要，这就是你在给自己的项目上税。

Anthropic 在 2024 年 12 月发了一篇博客，由 Schluntz 和 Zhang 署名。我跟你讲——这是过去两年里被引用最多次的 Agent 工程文章。不是因为它发明了什么新东西。是因为它说了别人不敢说的实话：**从直接 API 调用起步。只有五种工作流模式值得你记住。Agent 是你的最后手段，不是你的第一步。**

## 一个核心区分：谁拥有控制图？

这节课我会反复提一个概念叫"控制图"（graph）。什么叫控制图？就是你的系统里，谁决定下一步做什么。

在工作流里，**你拥有控制图。** 你写代码说：第一步做 A，A 的结果给 B，B 完成了就结束。每一步都是你预先定义的。LLM 是被调用的工具——它不知道自己在一个更大的流程里。

在 Agent 里，**模型拥有控制图。** 你给 Agent 一个目标和一堆工具，模型自己决定先用哪个工具、用几次、什么时候说"我做完了"。你不写步骤——你写的是"这个 Agent 能做什么，以及它什么时候该停"。

这个区别不是学术上的。它决定了：
- **调试。** 工作流出 bug，你打日志，跟着代码路径追。Agent 出 bug，你得读一整段轨迹，从中间推断"它在第 3 步为什么会选那个工具"。
- **成本。** 工作流的步骤数是静态的。Agent 的步骤数你不知道——它可能转 15 圈才停下来。
- **合规。** 审计者能读你的工作流图——10 个节点，清清楚楚。审计者读 Agent 轨迹——200 步工具调用日志，需要推断"这算不算合规操作"。

所以原则很简单：**当一个任务的所有步骤你都能预见到时，你写工作流。当步骤不可预见时，你才考虑 Agent。**

## 先搭原子：增强型 LLM

在讲五种模式之前，有一个基础单元要先建好。我称为**增强型 LLM**。

就是一个 LLM，加上三样东西：

- 🔍 **搜索/检索。** LLM 可以查资料——RAG、向量数据库、API 调用。
- 🔧 **工具。** LLM 可以执行操作——运行代码、读文件、调用外部服务。
- 🧠 **记忆。** LLM 可以跨调用保持状态——记住上一步做了什么。

这个增强型 LLM 是你所有工作流的原子单元。五模式架构不是什么神秘的框架——就是一个增强型 LLM，外面套了一层编排逻辑。

一个增强型 LLM 的核心代码长什么样？就一个 while 循环：

```python
while not done:
    response = llm.call(prompt + context + tool_results)
    if response.wants_tool:
        result = execute(response.tool_name, response.tool_args)
        tool_results.append(result)
    else:
        done = True
return response.text
```

这就是你的原子单元。今天这五种模式，全部是在这个 while 循环外面增加的编排逻辑。而一个 LangGraph 项目，往往是把这 10 行代码膨胀成了 500 行配置。你在付的是"编排层的可读性"——但如果你的编排逻辑只有一层，你不需要为它买一个框架。

## 模式一：提示链——你第一个该学的模式

提示链是做 LLM 最自然的模式。你用 LLM 做了一件事，输出给了下一个 LLM，下一个的输出给了第三个。

例子：你想写一篇技术博客。
- 链 1：生成大纲。
- 链 2：根据大纲写正文。
- 链 3：校阅正文，修正语法和风格。

三个 LLM 调用，顺序执行，每个的输出是下一个的输入。没有分支、没有循环、没有分类——一条直线。

什么时候用？**当你发现"我把这个任务拆成几步以后，每步的输出质量都提高了"。** 这是信号——你应该用提示链。

什么时候不用？**当你发现链中某一环"我不确定这个输入应该给哪个模块"——下一步是路由。**

### 模式二：路由——什么时候一个 LLM 不够，需要两个

提示链是直线。路由是岔路。

你是一个客服系统。用户输入来了。"我的衣服破了，要退款"——这是 L1 客服问题。"你们的系统在付款页面白屏了"——这是 bug 报告。"你们的数据合规政策是什么"——这是合规查询。

你用一个 LLM 处理三种完全不同的问题？它会在客服的语气里塞 bug 修复步骤，在合规查询里塞退款链接。

路由的方案：**加一个分类器 LLM。** 它的唯一任务是读用户输入，决定这是 L1 客服、bug 报告、还是合规查询。然后调用对应的处理 LLM。分类器的提示词是高度特化的——"你是一个分类器，只输出类别标签，不要做任何其他事情。"

路由的令牌成本是 2 次调用——一次分类，一次处理。比一个"处理一切"的巨型提示词更便宜、质量更高，因为每个处理 LLM 的提示词是专精的。

### 模式三：并行化——同一个问题，多个角度看

并行化有两种变体。我们分开讲，因为它们解决的问题不一样。

**第一种：分段（sectioning）。** 你有 10 页的合同要摘要。你不能一次全丢给 LLM——超窗口了。分成 5 段，每段 2 页，同时送给 5 个 LLM 并行摘要。5 个结果聚合——得到整份合同的摘要。

**第二种：投票（voting）。** 你有一个代码审查任务。一条提示词，同时发给 3 个 LLM 实例。三个结果是三份审查意见。聚合方式——多数投票（如果两个说"这一段有 bug"就算有），或者综合（让第四个 LLM 把三份意见合并为一份）。

分段是吞吐量优化（更快），投票是可靠性优化（更准）。你选哪种取决于你的瓶颈是时间还是准确率。

这里有一个你以后会踩的坑：**并行调用中的超时问题。** 如果你的 `parallel_vote` 发出 3 个请求，其中一个挂了或超时了——你怎么聚合？2 票算多数还是 3 票？这个边界条件需要在你写代码时就决定，不是出问题以后再说。

### 模式四：编排器-工作者——最容易被误用的模式

这个模式看起来最像个 Agent。一个"编排器"LLM 读任务描述，决定"这个问题需要哪些专家"，然后把子任务派给不同的工作者 LLM。

例子：用户说"我的网站加载很慢，而且搜索功能也不对。"

编排器分析：这是两个子问题——性能问题（需要前端 + 后端专家）和搜索问题（需要搜索工程师）。编排器调用前端工作者、后端工作者、搜索工作者，拿到三份报告，综合成一份"你的问题分析"。

**它为什么不是 Agent？** 因为编排器不会无限循环。它读完任务，在第一步就决定"我要调用这三个工作者"，然后一次性调用，综合结果，结束。它不迭代、不探索、不"试一个工具看看再决定下一步"。

如果你发现自己在给编排器加"如果结果不满意，重新分配"的逻辑——你的编排器已经变成了 Agent。不是坏事——意味着你的任务够复杂，值得升级。但你要知道你在升级。

### 模式五：评估器-优化器——"写了再改"

这是 Self-Refine（第 05 课）的泛化，也是我们第 09 课 Voyager 里迭代精炼的基础。

一个 LLM 生成答案。另一个 LLM 评估答案。评估器说"通过"——结束。评估器说"不通过，问题在这里：……"——生成器拿到反馈，重写，再评估。循环到通过或达到上限。

评估器和生成器**不能是同一个提示词模板。** 如果生成器和评估器共享同一个提示结构，LLM 会同意自己的输出。这叫橡皮图章，是 Self-Refine 的经典失败模式。

评估器的输出必须是结构化 JSON：

```json
{
  "pass": false,
  "violations": [
    {"severity": "high", "description": "缺少错误处理"},
    {"severity": "medium", "description": "变量命名不规范"}
  ],
  "suggested_fixes": [
    "为 API 调用添加 try-except",
    "将 xyz 重命名为 customer_id"
  ]
}
```

不是"这看起来还行"——是逐条列出问题、严重程度、修复建议。没有结构化反馈，精炼器无法高效工作。

## 五种模式的关系：一张图收尾

这五种模式不是孤立的——它们是一个复杂度递增的序列：

```
直接 API 调用
   ↓ 任务可以线性拆开？
   提示链
   ↓ 不同输入需要不同处理？
   路由
   ↓ 一个任务可以同时做多件事？
   并行化
   ↓ 不确定事先需要哪些专家？
   编排器-工作者
   ↓ 需要质量门控和迭代？
   评估器-优化器
   ↓ 步骤完全不可预见？
   Agent
```

每一步都在增加复杂性——也在增加令牌成本和调试难度。原则是：**从你确定需要的那一层开始，不要跳级。**

## 什么时候不该用这些模式

我教架构的时候有一条铁律：不讲"什么时候不该用"就是画饼。

**不该用工作流的场景：**
- 你无法枚举步骤。如果"查阅资料 → 分析 → 得出结论"这个循环里"查阅资料"可能产生 20 个新问题需要再查——这不是提示链能做的。你需要 Agent。
- 你想用一个模式解决所有问题。不要把路由强套在一个提示链够用的场景上。不要把编排器-工作者强套在一个提示链够用的场景上。

**不该用 Agent 的场景：**
- 你能量化步骤数。如果你能说出"这个任务就是 4 步"——写 4 步的工作流，不要写一个"自己决定"的 Agent。Agent 会在 4 步能搞定的事上走 8 步。
- 你需要精确的成本预测。每次 Agent 调用你不知道它会花多少钱。工作流的成本是确定的。
- 你需要审计。审计者读不了 200 步的 Agent 轨迹。他们能读你 10 个节点的流程图。

## 结课清单

如果你今天只带走一件事：**当你听说"我们的新项目要用多 Agent 框架"时，你先问三个问题：**

1. 这个任务有可枚举的步骤吗？→ 写工作流，不要写 Agent。
2. 这个任务需要幻数多的迭代吗？→ 评估器-优化器，上线 4 轮。
3. 这个任务的预算能承受不可预测的令牌消耗吗？→ 不能就不要用 Agent。

五模式实现清单：
1. 增强型 LLM = LLM + 检索 + 工具 + 记忆——你的原子单元。
2. 提示链 = 顺序调用——覆盖所有线性分解任务。
3. 路由 = 分类器 + 分发——不同输入需要不同处理。
4. 并行化 = 分段（吞吐量）或投票（可靠性）——N 个并发调用。
5. 编排器-工作者 = 一次动态任务分配——不是 Agent，不循环。
6. 评估器-优化器 = 生成→评估→重写→循环——质量门控。
7. Agent 是你的最后一步——不是你的第一步。

**最后一句话——我希望你明天还记得：**

**Agent 架构的精髓不是让系统更自主——而是在 10 行代码能解决的问题上，别引入 1000 行框架。简单不是偷懒——简单是工程判断。**
