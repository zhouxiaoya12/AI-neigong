## 🚗 开场翻车：跑了 3 小时的 Agent，崩在第 2 小时 59 分

一个团队让 Agent 做长时间研究——读 50 篇论文、写综合报告。在笔记本上跑通了：3 个 Agent，内存事件循环。

搬上生产。Agent 跑了一个研究任务 2 小时 59 分。还剩 1 分钟就要写最终报告了。**Worker 进程崩了。**

没有检查点。没有状态保存。3 小时的推理和工具调用全部丢失。从头开始。

**内存事件循环在笔记本上很好——在生产环境是定时炸弹。**

## 🧠 核心概念：持久执行

生产环境 Agent 需要一种能力：**每一步之后保存状态。崩了从最后一步恢复。**

```
Agent 跑第 37 步
      ↓
写完第 37 步的状态到数据库（检查点）
      ↓
Agent 跑第 38 步... Worker 崩溃！
      ↓
另一个 Worker 接管
      ↓
从第 37 步的检查点恢复 ← 不是从头开始
      ↓
继续跑第 38 步
```

这需要三个条件：
1. **状态可序列化** — Agent 的所有状态必须能存进数据库
2. **恢复是确定性的** — 同样的状态 + 同样输入 = 同样的动作
3. **副作用幂等** — 如果第 38 步"发了一封邮件"在崩之前已经执行了，恢复后不能再发

---

## 🔧 三种实现方式

### 方式 1：LangGraph 检查点（最成熟）

LangGraph 每个 `thread_id` 对应一个 Agent 运行。每个"超级步"之后自动写检查点到 Postgres。

```python
# LangGraph 风格的检查点
config = {"configurable": {"thread_id": "run-abc123"}}

# 跑 Agent——每步自动保存
for event in agent.stream(input, config):
    # 如果崩了，同一个 thread_id 从最后一个检查点恢复
    pass
```

**Worker 崩了 → 租约超时 → 另一个 Worker 拿同一个 thread_id 恢复。**

### 方式 2：消息队列 + 状态存储

```
Agent A 的出队消息 → Agent B 处理 → 写状态到 Postgres → 发结果到队列
```

每个 Agent 是独立的消费者。状态持久化在 Postgres。队列（SQS/RabbitMQ/Redis）负责路由。

### 方式 3：FastAPI + Postgres（KISS 路线）

Ashpreet Bedi（2026）的建议：**大多数团队在需要之前就过度工程化了。**

```
FastAPI + Postgres + 简单的后台 Worker
每个 Agent 运行 = 数据库里一行
状态用乐观并发原地更新
```

**在并发 <100 Agent 运行之前，这通常够了。** 当真的遇到瓶颈了，再升级到 LangGraph/Temporal。

## ⚡ 异步 vs 多线程：Agent 该用哪个

LLM 调用的特点：**99% 时间在等网络回包。** 线程在等，但占着 1MB 内存。

| | 1 个线程 | asyncio 协程 |
|---|---|---|
| 内存 | ~1MB | ~几 KB |
| 1000 并发 | 1GB 内存 | 几 MB 内存 |
| 适合 | CPU 密集 | I/O 密集（LLM 调用） |

**Agent 是典型 I/O 密集场景。用 asyncio，别用每任务一线程。**

## 🏆 焊死在脑子里的东西

1. **笔记本上的内存事件循环 ≠ 生产环境。** 生产必须持久化状态。

2. **检查点 = 每一步之后保存。** 崩了从最后一步恢复，不是从头来。

3. **KISS 原则：FastAPI + Postgres 在 100 并发以下通常够了。** 别过早用 LangGraph/Temporal。

4. **Agent 用 asyncio，别用多线程。** I/O 等网络，线程浪费内存。

---

## 📝 练习答案

**方案：KISS — FastAPI + Postgres + 简单检查点。**

理由：
- 每天 50 个任务，并发 <10 → 不需要消息队列
- 偶尔崩溃 → 简单检查点（每步写 Postgres）就够了
- 2 人团队 → 不要引入 LangGraph/Temporal 的学习成本
- 每个任务的状态存一行，崩了从最后一行恢复
- 如果未来涨到每天 5000 个任务，再评估是否加队列

---

> **生产环境的 Agent 不是魔法——是数据库里的一行状态 + 一个 while 循环 + 每一步写一次检查点。KISS 走到走不动了再加复杂度，别反过来。**
