# 自托管推理引擎选型 — llama.cpp、Ollama、TGI、vLLM、SGLang

> 2026 年四大推理引擎主导自托管推理。根据硬件、规模、生态选择。**llama.cpp** 在 CPU 上最快——最广的模型支持，对量化和线程的完全控制。**Ollama** 是开发笔记本的一键安装方案，比 llama.cpp 慢约 15-30%（Go + CGo + HTTP 序列化），在生产级负载下吞吐量差距达 3 倍。**TGI 于 2025 年 12 月 11 日进入维护模式**——仅修复 bug，原始吞吐量比 vLLM 慢约 10%，但历史上拥有顶级的可观测性和 HuggingFace 生态集成。该维护状态使其成为长期高风险选择——新项目更安全的默认选择是 SGLang 或 vLLM。**vLLM** 是通用生产默认引擎——v0.15.1（2026 年 2 月）新增 PyTorch 2.10、RTX Blackwell SM120、H200 优化。**SGLang** 是智能体多轮 / 前缀密集型场景的专家——400,000+ GPU 投入生产（xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS）。硬件约束：仅 CPU → 只能用 llama.cpp。AMD / 非 NVIDIA → 只能用 vLLM（TRT-LLM 锁定 NVIDIA）。2026 年流水线模式：开发 = Ollama，预发布 = llama.cpp，生产 = vLLM 或 SGLang。全程使用相同的 GGUF/HF 权重。

## 问题

你的团队开始一个新的自托管 LLM 项目。一个工程师说用 Ollama，另一个说 vLLM，第三个说"TGI 不是开箱即用吗？"三个人在各自的情境下都是对的。没有人对所有情境都是对的。

在 2026 年，选择决策树很重要：硬件第一，规模第二，工作负载第三。而一个具体的 2025 年事件——TGI 于 12 月 11 日进入维护模式——改变了新项目的默认选择。

## 概念

### 五大引擎

| 引擎 | 最适合 | 备注 |
|------|--------|------|
| **llama.cpp** | CPU / 边缘 / 最少依赖 / 最广模型支持 | CPU 上最快，完全控制 |
| **Ollama** | 开发笔记本，单用户，一键安装 | 比 llama.cpp 慢 15-30%；生产吞吐量差距 3 倍 |
| **TGI** | HF 生态，受监管行业 | **2025 年 12 月 11 日进入维护模式** |
| **vLLM** | 通用生产，100+ 用户 | 广泛的生产基线；2026 年 2 月 v0.15.1 |
| **SGLang** | 智能体多轮，前缀密集型负载 | 400,000+ GPU 投入生产 |

### 硬件优先决策

**仅 CPU** → llama.cpp。Ollama 也可以但较慢。没有其他引擎在 CPU 上有竞争力。

**AMD GPU** → vLLM（AMD ROCm 支持）。SGLang 也可用。TRT-LLM 锁定 NVIDIA，因此出局。

**NVIDIA Hopper（H100 / H200）** → vLLM 或 SGLang 或 TRT-LLM。三者都是顶级。

**NVIDIA Blackwell（B200 / GB200）** → TRT-LLM 是吞吐量领先者（第 17 阶段 · 07）。vLLM 和 SGLang 紧随其后。

**Apple Silicon（M 系列）** → llama.cpp（Metal）。Ollama 封装了这个。

### 规模第二决策

**1 用户 / 本地开发** → Ollama。一条命令，首 token 秒级。

**10-100 用户 / 小团队** → vLLM 单 GPU。

**100-10k 用户 / 生产** → vLLM 生产栈（第 17 阶段 · 18）或 SGLang。

**10k+ 用户 / 企业** → vLLM 生产栈 + 分离式架构（第 17 阶段 · 17）+ LMCache（第 17 阶段 · 18）。

### 工作负载第三决策

**通用聊天 / 问答** → vLLM 在广泛默认上胜出。

**智能体多轮（工具、规划、记忆）** → SGLang 的 RadixAttention（第 17 阶段 · 06）占主导。

**前缀重度复用的 RAG** → SGLang。

**代码生成** → vLLM 很好；SGLang 在缓存上略有优势。

**长上下文（128K+）** → vLLM + 分块预填充；SGLang + 分层 KV。

### TGI 维护陷阱

Hugging Face TGI 于 2025 年 12 月 11 日进入维护模式——今后仅有 bug 修复。历史上：顶级的可观测性，一流的 HF 生态集成（模型卡片、安全工具），原始吞吐量略逊于 vLLM。

对于 2026 年的新项目：默认不选 TGI。现有 TGI 部署可以继续但最终应迁移。SGLang 和 vLLM 是更安全的默认选择。

### 流水线模式

开发（Ollama）→ 预发布（llama.cpp）→ 生产（vLLM）。全程使用相同的 GGUF 或 HF 权重。工程师在笔记本上快速迭代；预发布环境镜像生产量化；生产环境是最终的推理目标。

### Ollama 注意事项

Ollama 对开发来说很棒。对共享生产来说不太好：Go HTTP 序列化增加开销，并发管理比 vLLM 简单，OpenTelemetry 支持滞后。在 Ollama 擅长的地方使用它——单用户、一条命令——并在共享场景切换到 vLLM。

### 自托管 vs 托管是一个独立决策

第 17 阶段 · 01（托管超大规模平台）、· 02（推理平台）涵盖了托管方案。本课程假设你已经决定自托管。自托管的理由：数据驻留、自定义微调、大规模总拥有成本、领域模型在托管平台不可用。

### 你应该记住的数字

- TGI 维护模式：2025 年 12 月 11 日。
- vLLM v0.15.1：2026 年 2 月；PyTorch 2.10；Blackwell SM120 支持。
- SGLang 生产规模：400,000+ GPU。
- Ollama 与 llama.cpp 的吞吐量差距：慢 15-30%；生产负载下 3 倍差距。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份高度实用的 2026 年推理引擎选型指南，以"决策树"思维贯穿全文——硬件 → 规模 → 工作负载。文档最突出的特点是它用非常具体的"信号"（TGI 维护日期、Kong 基准测试百分比、Ollama 的 3 倍吞吐量差距）代替模糊的推荐，使学习者可以独立做出技术决策而不依赖权威背书。同时，文档坦诚地指出 Ollama"在开发上很棒，在生产上不怎么样"，这种"场景化评价"比简单的"好/坏"二元判断更有工程教育价值。

### 二、知识结构梳理

- **认知基础层**："没有万能引擎"是核心认知。五大引擎各有所长，选择不是找"最好"的，而是找"最匹配约束"的。硬件约束是硬边界（CPU 只能 llama.cpp，AMD 不能用 TRT-LLM），这个认知防止了不切实际的选型。
- **工程模式层**：三层决策树（硬件 → 规模 → 工作负载）将混沌的选型空间压缩为可操作的决策路径。流水线模式（开发 Ollama → 预发布 llama.cpp → 生产 vLLM，相同权重）是另一个关键工程模式——它解决了"开发体验"和"生产性能"之间的矛盾。
- **落地实践层**：TGI 维护模式的教训是"生态信号"比技术指标更重要——一个引擎进入维护模式后，即使当前性能可控，长期风险也不可接受。vLLM 生产栈（第 18 课）和 SGLang 的 RadixAttention（第 06 课）的具体实现细节在相关课程中展开，本课负责"在正确的时间把你送到正确的课程"。

### 三、核心洞察

1. **TGI 维护模式是 2026 年最重要的生态信号**——它改变了整个推理引擎市场的默认选择。一个新团队在 2021-2024 年可能默认 TGI，但 2026 年必须默认 vLLM 或 SGLang。为什么重要：技术选型的默认值决定了成千上万个项目的技术方向，一个维护公告的影响远超其表面内容。
2. **"流水线模式"解决了开发-生产不一致的问题**——用相同的 GGUF/HF 权重贯穿开发到生产，消除了"开发时用 4-bit 量化，生产时用 8-bit，效果对不上"的经典陷阱。为什么重要：这是工程教育中最被低估的实践之一——让开发者的体验与生产环境一致。
3. **Ollama 的 3 倍吞吐量差距是"隐性技术债"**——一个团队可能从 Ollama 开发版直接上线，然后发现生产吞吐量远低于预期。这种差距不是因为 Ollama"差"，而是因为它不是为共享生产设计的。为什么重要：它提醒我们"开发工具"和"生产工具"之间有一道必须被正视的鸿沟。
4. **硬件约束是第一道过滤器，不是最后一个考虑因素**——很多技术讨论跳过硬件直接比较引擎特性。但如果你只有 CPU，llama.cpp 是唯一答案。为什么重要：它教会学生"约束驱动设计"的思维方式——先识别硬约束，再在其中优化。
5. **SGLang 的 400,000+ GPU 规模是"被采纳的信号"**——xAI、LinkedIn、Cursor、Oracle 等重量级用户的选择本身就是信息。为什么重要：大规模生产采纳是比基准测试更可靠的"可行性证明"。
6. **TRT-LLM 的"NVIDIA 锁定"是一种战略取舍**——在 Blackwell 上它是最快的，但代价是无法迁移到 AMD 或其他硬件。为什么重要：它迫使团队在"最佳性能"和"硬件灵活性"之间做明确的权衡。

### 四、教学建议

1. **用"三个工程师争论 Ollama/vLLM/TGI"的故事作为开场**——这个真实场景立即让学生理解为什么需要系统性的选型方法，而不是凭个人经验投票。
2. **制作一个可交互的"决策树计算器"**——一个简单的 Web 页面，输入硬件（CPU/AMD/NVIDIA/Apple Silicon）、规模（1/100/10k）、工作负载（聊天/智能体/RAG/代码/长上下文），输出推荐引擎和理由。让学生在不同约束下反复尝试。
3. **强调日期的重要性**——告诉学生技术文档有"保质期"。TGI 在 2025 年 12 月 11 日之前和之后是完全不同的评估对象。培养"检查文档时效性"的习惯。
4. **组织一场"TGI 迁移辩论"**——让一组学生为"继续用 TGI"辩护，另一组论证"迁移到 vLLM/SGLang"。这不仅锻炼技术论证能力，也模拟了真实团队中的技术决策讨论。
5. **用"相同权重，不同引擎"的实验演示**——在实际硬件上用相同模型权重分别跑 Ollama 和 llama.cpp，展示吞吐量和延迟的实际差异。数字比理论更有说服力。
6. **制作"引擎选型陷阱"清单**——常见错误：用 Ollama 上生产、在仅 CPU 环境试图用 vLLM、忽略 TGI 维护状态、在 AMD 上尝试 TRT-LLM。作为快速检查清单。
7. **引入"迁移成本"作为第四个决策维度**——除了硬件、规模、工作负载，从旧引擎迁移到新引擎的成本（重写配置、重新测试、团队学习）也应该被纳入决策。

### 五、值得补充的内容

1. **推理引擎的内存管理差异**——vLLM 的 PagedAttention、SGLang 的 RadixAttention 和 llama.cpp 的量化策略在内存使用上的具体差异，以及这对硬件选型的影响。
2. **多模型并发推理**——当一个 GPU 节点需要同时服务多个不同模型时（如通用模型 + 代码模型 + 嵌入模型），各引擎的 LoRA 适配器和模型切换能力。
3. **推理引擎的社区治理模式**——vLLM（UC Berkeley 主导，社区活跃）、SGLang（多公司贡献）、llama.cpp（ggml 生态）、Ollama（单一公司）的治理差异对长期可靠性的影响。
4. **边缘推理场景的引擎选择**——在移动设备、嵌入式设备、浏览器端（WebLLM/WebGPU）等极端资源受限环境下的推理方案。
5. **推理引擎的 Python 以外语言绑定**——对于需要 Go、Rust、C++ 客户端的生产系统，各引擎的 API 兼容性和非 Python SDK 生态。

### 六、一句话总结

2026 年推理引擎选型的核心不是"哪个最好"，而是"硬件决定边界→规模决定复杂度→工作负载决定优化方向"——TGI 的维护模式改变了默认选择，llama.cpp 独占 CPU，vLLM 是生产通用基线，而 SGLang 在前缀密集和智能体场景中无可替代。

# 🎓 Agent 架构课：推理引擎选型不是选最好的，是选最不会在凌晨爆炸的

同学们好。我是你们的FDE工程老师，今天讲的是 自托管推理。

**"咱们用 Ollama 上生产吧，开发用得很顺。"——这句话我在三年里听过至少 20 次。每次结局都一样：凌晨爆炸，PagerDuty 响了。**

Ollama 是开发工具——一键启动，CPU/GPU 自动适配，API 兼容 OpenAI。生产推理引擎是另一个物种。选型矩阵：vLLM（吞吐量最优，PagedAttention，连续批处理）、TGI（HuggingFace 原生，最全模型支持）、TensorRT-LLM（NVIDIA GPU 上最快但配置最复杂）、llama.cpp（CPU 推理，边缘设备）。

关键指标不是"跑起来了"——是在你的并发量下 TTFT 和 TPOT 不退化。vLLM 的 PagedAttention 在 KV cache 管理上做到接近零浪费，高并发下吞吐量远超其他。但部署复杂度也最高。

## 结课清单

1. **Ollama = 开发。vLLM = 生产。** 不要搞混。
2. **不是选"最好的"——是选"在你的 load 下不会炸的"。**
3. **PagedAttention 是高并发下的关键优势。** KV cache 管理效率决定吞吐量上限。

最后一句话：**开发环境和生产环境的推理引擎不是同一个东西。Ollama 让你开心地开发，vLLM 让你安心地睡觉。**

## 引擎选型决策树——或者说"别他妈选错工具"速查表

我做选型有一条铁律：**硬件第一，规模第二，工作负载第三。** 不是反过来。反过来的团队都在事后加班修性能问题。

硬件是硬边界。你只有 CPU？闭着眼选 llama.cpp。别纠结了。没有第二个选项。vLLM 在 CPU 上不是慢的问题，是根本启动不起来。我见过一个团队在 CPU-only 服务器上试了三天 vLLM，最后 CTO 问我有没有办法在 CPU 上优化 vLLM。我说有——买 GPU。

AMD GPU？更微妙。TRT-LLM 直接出局——它是 NVIDIA 的亲儿子，NVIDIA 独占。有人问我"AMD 和 NVIDIA 不都是 GPU 吗？"技术上是的。生态上不是。就像同一条街上的两家面馆，一家的配料只能在那家用。vLLM 的 ROCm 支持是 2025 年底才成熟的，之前你基本选 SGLang 或者认命。

## TGI 的维护状态——2026 年最重要的生态信号

2025 年 12 月 11 日。HuggingFace 宣布 TGI 进入维护模式——只修 bug，不加功能。

我给团队通知这个时，一个同事的表情像听说自家楼下的超市关门了。"但 TGI 不是挺好的吗？"是挺好的。但一个进入维护模式的推理引擎就像一辆停产的汽车——你还能开，但配件会越来越难找，修车师傅会越来越少，而其他车厂的发动机一年更新两轮。

**技术选型最容易被忽视的维度是"活着的信号"。** 一个引擎还在被积极开发，意味着：新的 GPU 架构会第一时间适配、新的量化方法会第一时间支持、新的 PyTorch 版本会第一时间兼容。TGI 进入维护模式的那一刻，这些"第一时间"全部消失了。

这也是为什么我现在对 2026 年的新项目默认推 vLLM 或 SGLang。不是因为 TGI 不好，是因为**不再进步的引擎就是在倒退**。

## vLLM vs SGLang：一句话说清楚

如果你做的是通用聊天/问答——vLLM 是安全牌。生态最广，踩坑的人最多，StackOverFlow 上能搜到答案的概率最高。

如果你做的是 Agent 多轮交互——SGLang 的 RadixAttention 是作弊器。前缀缓存命中率在 80% 以上的场景下，SGLang 能把延迟砍到 vLLM 的一半。xAI 的 Grok、Cursor 的代码补全、GCP 的推理服务，都在用 SGLang。400,000+ GPU。这不是实验室玩具，这是工业级基础设施。

我个人的经验法则：先跑 vLLM。当你的前缀复用率超过 60%（做个简单的日志分析就能算出来），再考虑迁移到 SGLang。不要一开始就 SGLang——它的调试体验比 vLLM 粗糙。你需要在"工程师时间"和"GPU 时间"之间做权衡。

## 流水线模式——我最常讲但最少人做的实践

开发用 Ollama → 预发布用 llama.cpp → 生产用 vLLM。**同一套权重文件贯穿三个阶段。**

为什么重要？因为模型量化带来的行为差异是"幽灵 Bug"——开发环境和生产环境的结果不一样，但偏差太小不至于直接报错，恰好大到让产品经理在评审会上说"为什么生产环境的回答比本地差？"

用相同权重贯穿开发到生产，等于告诉你："开发时看到的答案，就是用户将要看到的答案。"这不是技术问题，这是信任问题。产品经理不信任 AI 团队，通常不是因为模型不好，是因为开发和生产表现不一致。这个 pipeline 模式解决的就是这个信任问题。

## 反模式速查

1. **"Ollama 开发用得好，直接上生产"** —— 3 倍吞吐量差距不是"优化一下"能解决的。Ollama 的架构决定了它的并发上限。
2. **"我们一直用 TGI，为什么要换"** —— 因为 TGI 不会再更新了。你的 GPU 会更新，你的用户量会增长，你的模型会变大。引擎不更新，就是等死。
3. **"llama.cpp 只适合边缘计算"** —— 错。llama.cpp 在 CPU 上独步天下，在 Apple Silicon M 系列上也是。它不是"低端选择"，它是"正确的硬件选择"。
4. **"所有引擎都差不太多"** —— 在你只有 1 个用户的时候确实差不多。在你有 10000 个用户的时候，选错引擎的代价可能是三倍的 GPU 账单。

## 收尾

推理引擎选型最怕的不是选错，是"不知道自己在选什么"——把开发工具当生产工具，把停产的引擎当长期依赖，把通用引擎用在专用场景里然后抱怨性能不行。

记住：**引擎选型的选择不是关于性能，是关于凌晨三点你最不想处理什么问题。** 用对了引擎，凌晨三点你在睡觉。用错了引擎，凌晨三点你在盯着 Grafana 面板上的延迟曲线，手边是第三杯咖啡，心里想着"明天就说服老板批那批 H100 的预算"。
