# 一个 Agent 不够？那就来一群——多 Agent 编排

> 预计阅读：30分钟 | 难度：进阶 | 前置条件：理解了单 Agent 循环
>
> 你将学到：为什么单个 Agent 会撞墙？什么时候该拆分成多个 Agent？监管者模式、Handoff、以及多 Agent 系统的五大翻车现场。

---

## 🚗 开场翻车：一个 Agent 三份工

去年有人让一个"全能 Agent"同时做三件事：
- 读 5 篇论文，写文献综述
- 基于论文写一个 Python 实现
- 写完代码后做代码审查

Agent 开始了。先读第一篇论文——读完开始写代码，写到一半想起"我还没读另外四篇"，切回去读第二篇，读了两段又想起"刚才写的代码有个 bug"，切回去改 bug……**来回横跳了 47 轮，三个任务全都没干完，API 费用烧了 $47。**

这不是 Agent "不聪明"。这是任务超出了单 Agent 的物理极限。

---

## 🤔 单 Agent 的天花板——三个要命的限制

想象一个 Agent 是一个**只有一张办公桌的人**：

```
┌─────────────────────────────────────┐
│          Agent 的办公桌             │
│  (上下文窗口 = 200k tokens)        │
│                                     │
│  📄 论文1(30k) 📄 论文2(30k)       │
│  📄 论文3(25k) 📄 论文4(28k)       │
│  📄 论文5(32k)                      │
│  💻 代码文件(15k)                   │
│  📋 测试结果(20k)                   │
│  📝 审查笔记(18k)                   │
│  ...                                │
│  ████████████ 办公桌满了 ██████████ │
│  第5轮读到的关键信息已经掉到桌下了   │
└─────────────────────────────────────┘
```

### 天花板一：上下文饱和

每调用一次工具（读文件、调 API），结果就塞进上下文窗口。到第 30 轮，Agent 已经消费了 150k tokens——第 5 轮的关键细节被挤到边缘，模型"看不见"它了。

**就像你让一个人记住 50 页资料然后答题——他记不住第 3 页的内容，因为脑子里已经在塞第 48 页。**

### 天花板二：角色混淆

一个系统提示写："你是研究员、编码员、审查员和测试员"。结果？一半研究一半编码、从来没完成审查。因为模型在四个角色之间摇摆——提示越长越模糊。

**就像一个人同时当厨师、服务员、收银和洗碗工——上菜时忘了放盐，收钱时忘了算账。**

### 天花板三：顺序瓶颈

Agent 读文件 A → 读文件 B → 读文件 C。三次串行调用。文件 A 和文件 B 之间没有任何依赖关系，但 Agent 必须等 A 读完才能读 B。

**就像你让一个人先去看第 1 本书，看完再看第 2 本书——而不是派三个人各看一本。**

---

## 🔧 解决方案：拆！

多 Agent 的核心思想简单到一句话：**不要用一个大 Agent 干所有事。拆成几个小 Agent，每个只管一件事。**

```
                    监管者 Agent
                   "把这个任务拆成3块"
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ 研究员    │  │ 编码员    │  │ 审查员    │
    │          │  │          │  │          │
    │ 只读文档  │  │ 只写代码  │  │ 只找bug   │
    │ 只找模式  │  │ 只看规格  │  │ 只检查质量 │
    │          │  │          │  │          │
    │ 上下文：  │  │ 上下文：  │  │ 上下文：  │
    │ 只有论文  │  │ 只有代码  │  │ 只有审查  │
    └──────────┘  └──────────┘  └──────────┘
```

每个 Agent 有：
- **一个聚焦的系统提示**（"你只负责找 bug，其他事别管"）
- **独立的上下文窗口**（不跟其他 Agent 的垃圾挤在一起）
- **清晰的输入/输出合约**（收到研究笔记，产出代码）

## 📊 多 Agent 的四种模式

### 1. 流水线（Pipeline）

```
研究 → 编码 → 审查 → 测试
 A 完成 → B 开始 → C 开始 → D 开始
```

上一个的输出是下一个的输入。适合**顺序依赖**的任务。

**比如：** "先研究 API 文档 → 再写代码 → 再审查 → 再跑测试"——后一步必须等前一步，没法并行。

### 2. 并行扇出（Parallel Fan-out）

```
        "研究三个方案"
         /    |    \
    方案A  方案B  方案C （同时跑）
         \    |    /
        "综合三个结果"
```

同一个问题，三个 Agent 各研究一个方向。**适合独立的子任务。**

**比如：** "比较 React/Vue/Svelte 三个框架的 Server Components 方案"——三个 Agent 各研究一个框架，没有依赖。

### 3. 监管者（Supervisor）⭐ 最常用

```
         监管者 Agent
        "规划和委派"
        /    |    \
    工作者1 工作者2 工作者3
        \    |    /
         监管者综合
```

一个"老板 Agent"把任务拆成子问题，派给"员工 Agent"，然后汇总结果。

**这是 Anthropic 内部研究系统用的模式。** 他们用 Claude Opus 4 当监管者，Sonnet 4 当工作者——内部评估比单 Agent 高出 90.2%。

**核心优势：** 每个工作者拿到的是**新鲜的上下文窗口**（空的 200k tokens），不为监管者的规划历史付 token 账单。

### 4. 分层（Hierarchical）

```
        CEO Agent
       /    |    \
    VP1    VP2    VP3
    / \    / \    / \
  员工 员工 员工 员工 员工 员工
```

多层嵌套。**适合大型复杂项目。** Claude Code 的 Task 子 Agent 就是这种——父 Agent 生成子 Agent，子 Agent 可以做聚焦的工作。

## 💀 多 Agent 的五大翻车现场

### 翻车 1：监管者编了一个不存在的问题

监管者说"你们三个分别研究 A、B、C"。但 C 跟原问题毫无关系——监管者幻觉了一个子问题。三个 Agent 在错误的方向上精确地研究——做得越好越离谱。

### 翻车 2：工作者漂移

你让一个 Agent "只研究 React Server Components 的性能"。它看着看着发现 React 的文档太少了，"顺手"研究起了 Next.js——带回来一堆跟任务无关的信息，污染了综合环节。

### 翻车 3：综合冲突

Agent A 说"方案一最快"，Agent B 说"方案一最慢"。监管者拿到两个矛盾的结果。最坏的失败是什么？**监管者默默选了其中一个，用户永远不知道发生过分歧。**

### 翻车 4：token 账单爆炸

多 Agent 系统的 token 消耗约是单 Agent 的 15 倍。三个 Agent 各用 200k tokens = 600k tokens。加上监管者的规划 token。**多 Agent 只在任务价值证明成本合理时才用。**

### 翻车 5：简单的活杀鸡用牛刀

"今天天气怎么样？"——开五个 Agent 来回答？一个 Agent 3 步搞定的事，用监管者模式花了 15 步。**多 Agent 不是默认选项——先试单 Agent，撞墙了再加人。**

## 🏆 焊死在脑子里的东西

1. **单 Agent 的天花板是真实的。** 上下文饱和、角色混淆、顺序瓶颈——不是模型的错，是架构的物理极限。

2. **多 Agent = 各管一摊 + 独立上下文 + 清晰合约。** 每个 Agent 只有一个工作、一个窗口、一个输入输出格式。

3. **监管者模式是最通用的。** 一个老板拆任务、派活、综合结果。Anthropic 自己也是这么用的。

4. **多 Agent 比单 Agent 贵 15 倍。** 只在任务价值证明成本合理的时候开。

5. **先试单 Agent。撞墙了再加 Agent。** 多 Agent 不是默认选项。

## 📝 练习答案

**任务拆分：** "分析五个开源项目的架构，写对比报告"

**模式：** 并行扇出 → 流水线结合。

**拆分：**
1. **监管者 Agent：** 把任务拆成 5 个子任务 + 1 个综合任务
2. **工作者 1-5（并行）：** 各分析一个项目——"分析 React 架构"、"分析 Vue 架构"……每个独立上下文窗口
3. **综合 Agent（等工作者完成）：** 拿到 5 份分析报告，写对比总结

**总共 7 个 Agent。** 5 个并行跑（等最慢的那个），1 个规划，1 个综合。如果单 Agent 需要 60 轮，这个方案大概 15-20 轮 + 并行时间。

---

> **一个 Agent 是打工人，一群 Agent 是公司。没有老板的公司是菜市场——吵得很热闹，什么正事都没干。有了老板但员工乱跑的公司是迷宫——进得去出不来。拆得好、管得住的多 Agent 才是生产线。**
