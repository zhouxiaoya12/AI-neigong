# OpenAI Agents SDK：转交、护栏、追踪

> OpenAI Agents SDK 是构建在 Responses API 之上的轻量级多 Agent 框架。五个原语：Agent、Handoff、Guardrail、Session、Tracing。转交是名为 `transfer_to_<agent>` 的工具。护栏在输入或输出时触发。追踪默认开启。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 01（Agent 循环）、Phase 14 · 06（工具使用）
**用时：** 约 75 分钟

## 学习目标

- 说出 OpenAI Agents SDK 的五个原语。
- 解释转交：为什么被建模为工具、模型看到的名称形态、以及上下文如何传递。
- 区分输入护栏、输出护栏和工具护栏；解释 `run_in_parallel` 与阻塞模式。
- 实现一个基于标准库的运行时，包含转交 + 护栏 + span 风格追踪。

## 问题

无法干净委派的 Agent 最终会把所有东西塞进一个提示词。没有护栏的 Agent 会泄露 PII、输出违规策略的内容、或无限循环。OpenAI 的 SDK 将三个使多 Agent 工作可管理的原语固化为代码。

## 概念

### 五个原语

1. **Agent。** LLM + 指令 + 工具 + 转交。
2. **Handoff（转交）。** 委托给另一个 Agent。在模型中表示为名为 `transfer_to_<agent_name>` 的工具。
3. **Guardrail（护栏）。** 对输入（仅第一个 Agent）、输出（仅最后一个 Agent）或工具调用（每个函数工具）的验证。
4. **Session（会话）。** 跨轮次的自动对话历史。
5. **Tracing（追踪）。** 内置 span，覆盖 LLM 生成、工具调用、转交、护栏。

### 转交即工具

模型在其工具列表中看到 `transfer_to_billing_agent`。调用它会告诉运行时：

1. 复制对话上下文（或通过 `nest_handoff_history` beta 折叠它）。
2. 用目标 Agent 的指令初始化它。
3. 由目标 Agent 继续运行。

这是监管者模式（第 13 课 / 第 28 课）的产品化。

### 护栏

三种类型：

- **输入护栏。** 在第一个 Agent 的输入上运行。在任何 LLM 调用之前拒绝不安全或超出范围的请求。
- **输出护栏。** 在最后一个 Agent 的输出上运行。捕获 PII 泄露、策略违规、格式错误响应。
- **工具护栏。** 在每个函数工具上运行。验证参数、检查权限、审计执行。

模式：

- **并行（默认）。** 护栏 LLM 与主 LLM 同时运行。降低尾部延迟。如果被触发，主 LLM 的工作被丢弃（令牌浪费）。
- **阻塞（`run_in_parallel=False`）。** 护栏 LLM 先运行。如果被触发，不在主调用上浪费令牌。

触发时抛出 `InputGuardrailTripwireTriggered` / `OutputGuardrailTripwireTriggered`。

### 追踪

默认开启。每次 LLM 生成、工具调用、转交和护栏都发出一个 span。`OPENAI_AGENTS_DISABLE_TRACING=1` 可选择退出。`add_trace_processor(processor)` 将 span 扇出到你的自有后端，与 OpenAI 的后端并列。

### 会话

`Session` 在后端（SQLite、Redis、自定义）中存储对话历史。`Runner.run(agent, input, session=session)` 自动加载和追加。

### 此模式在何处会出错

- **转交漂移。** Agent A 转交给 Agent B，B 又转回给 A。添加跳数计数器。
- **护栏绕过。** 工具护栏仅在函数工具上触发；内置工具（文件读取器、网页获取）需要单独的策略。
- **过度追踪。** Span 中包含敏感内容。配合 OTel GenAI 内容捕获规则（第 23 课）——外部存储，按 ID 引用。

## 构建

`code/main.py` 基于标准库实现 SDK 形态：

- `Agent`、`FunctionTool`、`Handoff`（作为带转移语义的函数工具）。
- `Runner`，具备输入/输出/工具护栏、转交分发、跳数计数器。
- 一个简单的 span 发送器，展示追踪形态。
- 一个分诊 Agent，根据用户查询转交给账单或支持；一个输入被护栏拦截。

运行：

```bash
python3 code/main.py
```

输出显示两次成功的转交、一次输入护栏拦截、以及一个反映真实 SDK 所发出内容的 span 树。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| Agent | "LLM + 指令" | SDK 中的 Agent 类型；拥有工具和转交 |
| Handoff（转交） | "转移" | 模型调用以委托给另一个 Agent 的工具 |
| Guardrail（护栏） | "策略检查" | 对输入 / 输出 / 工具调用的验证 |
| Tripwire（触发线） | "护栏拦截" | 护栏拒绝时抛出的异常 |
| Session（会话） | "历史存储" | 运行间持久化的对话记忆 |
| Tracing（追踪） | "Span" | 内置可观测性，覆盖 LLM + 工具 + 转交 + 护栏 |
| 阻塞护栏 | "顺序检查" | 护栏先运行；拦截时不浪费令牌 |
| 并行护栏 | "并发检查" | 护栏同时运行；延迟更低，拦截时浪费令牌 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 14 课程中"产品化原语"的关键一课。LangGraph 课讲了执行引擎——这是基础设施。这节课讲的是"产品框架怎么把基础设施包装成可用的原语"。五个原语（Agent、Handoff、Guardrail、Session、Tracing）精确覆盖了多 Agent 系统的五类需求：能力封装、任务委托、安全校验、状态持久化、可观测性。文档在"护栏的并行 vs 阻塞"和"转交即工具"两个设计决策上有最深的工程洞察。适合已理解编排模式和状态机引擎、正在选型产品框架的架构师。

### 二、知识结构梳理

**认知基础层：** Agent 循环（第 01 课）——SDK 的 `Runner` 就是对 Agent 循环的封装。工具使用（第 06 课）——转交被建模为一种特殊工具，这是整节课最核心的认知前提。编排模式（第 13/28 课）——SDK 的转交本质上是监管者模式的产品化。

**框架原语层：** 五个原语不是孤立的——它们是一个完整的 Agent 运行时。Agent 封装能力，Handoff 分配任务，Guardrail 执行策略，Session 保持记忆，Tracing 提供可见性。每个原语解决多 Agent 系统中的一个具体痛点。

**工程决策层：** 两个关键技术选择——"转交 = 工具"（而非独立 API）和"护栏模式可选并行/阻塞"——是该 SDK 区别于 LangGraph 和 Claude Agent SDK 的核心差异。

### 三、核心洞察（备课时的关键理解）

1. **"转交即工具"是整个 SDK 最聪明的架构决策。** 转交不是独立于 tools 的另一个原语——它就是一个工具，名字叫 `transfer_to_<agent_name>`。这意味着一件事：**模型不需要学习一个新概念**。它已经知道怎么调用工具。转交只是工具列表里的一个特殊条目。这比创建一个独立的"handoff API"更简洁、更少训练成本、更少出错。

2. **护栏的并行/阻塞 trade-off 是工程经济学的精妙实例。** 并行护栏：护栏 LLM 和主 LLM 同时跑——延迟 = max(护栏延迟, 主调用延迟)。但护栏触发时主调用的令牌全浪费了。阻塞护栏：先跑护栏——延迟 = 护栏延迟 + 主调用延迟。拦截时不浪费令牌。选择取决于"护栏触发率 × 浪费令牌成本 vs. 增加的延迟"。这是一个可以用数字计算的工程决策。

3. **三类护栏覆盖了整个链路的三个脆弱点。** 输入护栏防"进来的东西不对"，输出护栏防"出去的东西不对"，工具护栏防"中间操作不对"。缺任何一类，你都有一个未被保护的面。

4. **追踪默认开启是一把双刃剑。** 好的一面：零配置可观测性。坏的一面：Span 中可能包含敏感内容（用户消息、LLM 回复）。文档明确警告"过度追踪"——必须配合 OTel 的内容捕获规则，敏感内容外部存储、Span 中只保留 ID 引用。

5. **`nest_handoff_history` 是上下文管理的杀手功能。** 当 Agent A 转交给 Agent B 时，B 是否需要看到 A 的完整对话历史？如果历史很长——B 的上下文窗口被吃掉。`nest_handoff_history` 将 A 的对话折叠为一条摘要，B 只拿到一个紧凑的上下文。这是上下文预算管理在转交场景的精确应用。

6. **Session 是"状态持久化"在产品框架中的标准形态。** LangGraph 把状态持久化作为核心卖点（检查点存储），OpenAI Agents SDK 把它包装为一个轻量原语（Session）。两者解决同一问题，但 LangGraph 的状态模型更显式、更强大，SDK 的 Session 更简单、更省配置。

7. **和 Claude Agent SDK 的对比是天然的跨框架教学材料。** Claude SDK 用"子 Agent + 会话存储"实现转交——本质是同一个模式（监管者），但实现哲学不同。Claude 偏 "managed harness"，OpenAI 偏 "composable primitives"。这不是谁更好——是对"控制粒度 vs. 开箱即用"的不同取舍。

### 四、教学建议

1. **用"转交 = 工具"这个点开场。** 问学生："如果你要设计一个多 Agent 系统，你怎么让 Agent A 把任务交给 Agent B？" 通常会得到"独立 API"或"消息队列"的答案。然后展示 SDK 的方案——"就把它当一个工具，名字叫 transfer_to_b"。简洁的力量令人印象深刻。

2. **护栏的并行 vs 阻塞值得画一个延迟图。** 横轴是护栏触发率，纵轴是平均延迟 + 平均浪费令牌成本。两条线交叉的点就是决策边界。让学生看到"工程决策可以量化"。

3. **三类护栏用三个真实失败案例引入。** 输入护栏：用户让客服 Agent 生成恶意代码。输出护栏：Agent 在回复中输出了用户的信用卡号。工具护栏：Agent 调用了"退款"工具但参数是负数金额。每个案例对应一类护栏。

4. **追踪的 span 树值得在课堂上现场看。** 拉一个真实 SDK 的 trace，展示 LLM span → tool call span → handoff span → guardrail span 的嵌套关系。学生需要看到"可观测性不是日志——是有结构的调用树"。

5. **和 LangGraph 的对比是这节课的天然延伸。** 同一个任务（分诊 + 转交给专家），用 LangGraph 实现（显式状态图）vs 用 Agents SDK 实现（转交即工具）。让学生看到"同一个模式，两种框架哲学"。

6. **跳数计数器是每个多 Agent 系统的基本功。** 练习 1（转交跳数计数器）不是可选的——是每个做多 Agent 的人都必须实现的。A→B→A 弹跳在生产中太常见了。

7. **练习 5（移植到真实 SDK）是最诚实的教学。** 让学生用自己的标准库实现和真实的 `openai-agents-python` 对比——发现自己建模错了什么。这是从"我懂了"到"我真的懂了"的桥梁。

### 五、值得补充的内容

1. **护栏的误报率和漏报率的量化。** 输入护栏拒绝了 10% 的正常请求——用户体验受损。输出护栏漏了 1% 的 PII——合规风险。需要一个方法论来决定护栏的阈值。

2. **多供应商 Agent 的转交实现。** 当分诊 Agent 是 GPT-5、账单 Agent 是 Claude Opus 时——转交怎么工作？上下文格式怎么转换？这是"自定义"场景的核心挑战。

3. **追踪的成本。** 默认开启追踪意味着每个请求额外产生 span 序列化和网络传输的延迟和成本。对于每秒 10K 请求的系统，这值得量化。

4. **与 LangGraph 检查点的互操作。** 如果用 Agents SDK 做转交、用 LangGraph 做状态管理——两者怎么协同？这是混合架构的常见问题。

5. **护栏的"拒绝消息"设计。** 当输入护栏触发时——用户看到什么？"你的请求被拒绝"不够好。需要给出具体的拒绝理由和申诉路径。

### 六、一句话总结

**OpenAI Agents SDK 做了一个很少框架做到的决策：把转交当作工具、把护栏当作可选的延迟-成本 trade-off、把追踪当作默认——三个简约选择合在一起，构成了多 Agent 系统最干净的工程接口。**

# 🎓 Agent 架构课：OpenAI Agents SDK——三个让你少犯错的工程决策

同学们好。我是你们的 Agent 架构老师。

上节课我们讲了 LangGraph——状态机引擎、检查点、持久化恢复。那是基础设施。今天这节课讲的是**在基础设施之上，一个产品框架怎么把多 Agent 操作打包成五个干净的原语**。

我先给你们一个数字：**大多数多 Agent 系统的 bug 集中在三个地方——转交出错、输出违规、不知道 Agent 内部发生了什么。** OpenAI Agents SDK 用五个原语来打这三个点。我们一个一个看。

## 原语一：Agent——不是"智能体"，是"能力封装单元"

SDK 里 Agent 的定义非常窄：**一个 LLM + 一份指令 + 一组工具 + 一组可以转交的目标。**

注意它不是什么。它不是一个"自主思考的智能体"。它是你封装一个专精能力的单元。账单 Agent 只知道账单的事。支持 Agent 只知道支持的事。它们的智能不来自 Agent 这个概念——来自你的指令 + 它们的工具。

这个"窄定义"是故意的。越窄的 Agent 越可靠、越容易调试、越少产生幻觉。

## 原语二：Handoff——为什么转交不该是一个独立 API

这是 OpenAI Agents SDK 做的最聪明的架构决策，我单独拿出来讲。

我问你们：如果你要设计多 Agent 系统，Agent A 怎么把任务交给 Agent B？

大多数人的第一答案是：**加一个独立 API。** `handoff(target="billing_agent", context=...)`。Agent A 调用这个 API，运行时切换执行上下文到 Agent B。

OpenAI 的方案不是这个。他们把转交做成一个**工具**——名字叫 `transfer_to_billing_agent`。

为什么这是天才设计？

**第一，模型不需要学习新概念。** LLM 已经知道怎么调用工具。它在工具列表里看到 `transfer_to_billing_agent`，就像看到 `search_knowledge_base` 一样——它是一个函数，接受参数，被调用时发生一些事情。转交不引入新的认知负担。

**第二，转交的触发和工具调用共享同一套机制。** 日志、追踪、护栏——所有这些对工具调用有效的，对转交自动有效。不需要为转交单独建一套可观测性。

**第三，`nest_handoff_history` 解决了上下文膨胀。** 当 Agent A 转交给 B 时——B 需要看到 A 的完整对话吗？如果 A 已经聊了 20 轮——B 的上下文窗口直接被占满。SDK 的 `nest_handoff_history` 选项让 A 的对话被压缩为一条摘要。B 接手的上下文是小而干净的。

**第四，转交是监管者模式的最简实现。** 一个分诊 Agent 有三个转交目标（账单、支持、销售）——这就是一个监管者-工作者系统。不是用 `create_supervisor()`，是用 `tools = [transfer_to_billing, transfer_to_support, transfer_to_sales]`。

## 原语三：Guardrail——三层防线，一个决策点

护栏在三个位置检查你的 Agent：

**输入护栏：** 用户消息进来 → 护栏 LLM 检查 → 拒绝？通过？拒绝时抛出 `InputGuardrailTripwireTriggered`，终止整个过程，**一个字节的 LLM 生成都没触发。** 你在做的是"在花钱之前先检查值不值得花"。

**输出护栏：** Agent 处理完 → 生成回复 → 护栏 LLM 检查 → PII 泄露？违规内容？格式错误？拒绝时抛出 `OutputGuardrailTripwireTriggered`。你在做的是"在发给用户之前先检查该不该发"。

**工具护栏：** Agent 调用了一个函数工具（比如"退款"）→ 护栏检查参数 → 金额是负数？权限不足？拒绝执行。

这里有一个关键工程决策：**护栏怎么跑？并行还是阻塞？**

- **并行（默认）：** 护栏 LLM 和主 LLM 同时启动。延迟 = max(护栏延迟, 主调用延迟)。如果主 LLM 先返回但护栏还没跑完——你等。如果护栏先返回并通过——主 LLM 的结果直接放行。如果护栏拒绝——主 LLM 的结果被丢弃。**令牌浪费了，但延迟最优。**

- **阻塞（`run_in_parallel=False`）：** 护栏先跑。如果被触发——主 LLM 根本没启动，零令牌浪费。如果通过——主 LLM 启动。**令牌最优，但延迟增加了。**

选哪个？取决于一个简单的数学：**护栏触发率 × 每次触发浪费的令牌成本 vs. 增加的延迟成本。**

如果一个护栏的触发率是 0.1%（比如绝大多数正常用户的输入都会通过），并行模式省下来的延迟值得那 0.1% 的令牌浪费。如果触发率是 30%（比如你的应用经常被恶意用户攻击），阻塞模式省下来的令牌成本值得那一点点延迟。

## 原语四：Session——状态持久化的轻量版本

`Session` 是 SDK 对"跨轮次记忆"的回答。

`Runner.run(agent, input, session=session)`——每次调用自动加载这个 session 的历史对话，追加当前轮次，持久化回后端。

这和 LangGraph 的检查点模型有本质区别：
- LangGraph：每一步都做检查点，状态是完整的类型化字典，恢复是精确到步的。
- Agents SDK Session：按轮次持久化对话历史——检查点粒度更大，但配置更简单。

选哪个？如果你需要"在第 38 步崩溃后精确恢复"——LangGraph。如果你需要"用户下一次打开 App 时记得之前的对话"——SDK Session 更合适。

## 原语五：Tracing——默认开启，双刃剑

追踪默认开启。每次 LLM 生成、工具调用、转交、护栏触发——一条 span。

这是什么？**你的 Agent 系统自带可观测性。** 不需要装 OpenTelemetry SDK、不需要配置收集器、不需要手动埋点。调一下 `add_trace_processor` 把 span 路由到你的监控后端——结束了。

但这把刀有两面。Span 里包含你的用户消息和 Agent 回复——也就是你的用户数据。如果你的监控后端没有数据脱敏，这些 span 就是合规定时炸弹。

SDK 的做法是：配合 OTel 的内容捕获规则。敏感内容不在 Span 里存储——Span 里只放一个内容 ID。实际内容存在外部安全存储中。

## 生产反模式

### 反模式一：转交弹跳

分诊 Agent → 账单 Agent。账单 Agent 说"这其实是支持问题" → 转回分诊。分诊说"这看起来像账单" → 转给账单。循环。

解决方案：**跳数计数器。** 超过 N 次转交（默认 3），强制当前 Agent 给出最终回复。

### 反模式二：只设输出护栏，不设输入护栏

"输出时检查一下就好了"——但你的 Agent 已经为那个恶意输入花掉了 5000 个令牌做推理。输入护栏的价值不仅在安全——在**成本保护。** 在花钱之前拒绝，胜过花了钱再拦截。

### 反模式三：忽视追踪中的敏感内容

开启追踪 → 一个月后合规审计 → "你们的监控日志里存了用户的完整对话？"

这是真实发生过的事。追踪中的内容管理不是"以后再说"——是上线第一天就要配好的。

## 什么时候用 OpenAI Agents SDK，什么时候用别的

- **用 Agents SDK：** 你的产品以 OpenAI 模型为主，你需要快速的转交 + 护栏开箱即用，你想要零配置追踪。
- **用 Claude Agent SDK：** 你的产品以 Claude 为主，你需要托管 harness + 内建会话存储。
- **用 LangGraph：** 你需要显式状态管理、持久化恢复、自定义检查点存储后端。
- **自己写：** 你需要多供应商、联邦部署、或对框架的每一层都有精确控制。

大多数产品从 Agents SDK 起步就够了。只在 SDK 的抽象妨碍你时才升级。

## 结课清单

如果你在用 OpenAI Agents SDK 搭多 Agent 系统：

1. 每个 Agent 的职责是窄的——窄 = 可靠。
2. 转交是工具，不是独立 API——利用现有的工具调用机制而非创造新概念。
3. 输入护栏 → 输出护栏 → 工具护栏：三层全要，缺一不可。
4. 护栏模式选并行还是阻塞——这是一个可以用触发率和令牌成本计算的决策。
5. 跳数计数器写死在转交逻辑里——不是可选功能。
6. 追踪中的敏感内容规则上线第一天就配好。
7. `nest_handoff_history` 在转交给频繁调用的 Agent 时开启——保护上下文预算。

**最后一句话：**

**多 Agent 系统的复杂度不在 Agent 数量——在三个脆弱点：谁把任务交给谁、输出是否能出去、以及你是否知道内部发生了什么。OpenAI Agents SDK 用五个原语——Agent、Handoff、Guardrail、Session、Tracing——精确覆盖这三个点。转交是工具而非 API，护栏是延迟-成本的量化选择，追踪默认开启但必须有内容保护——这三个工程决策定义了它的设计品味。**
