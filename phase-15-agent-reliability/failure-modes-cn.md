# 失败模式：Agent 为什么崩溃

> MASFT（Berkeley, 2025）编目了 14 种多 Agent 失败模式，分属 3 个类别。微软的分类法记录了现有 AI 失败如何在 Agent 环境中被放大。行业现场数据收敛于五种反复出现的模式：幻觉动作、范围蔓延、级联错误、上下文丢失、工具误用。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 05（Self-Refine 与 CRITIC）、Phase 14 · 24（可观测性）
**用时：** 约 60 分钟

## 学习目标

- 说出 MASFT 的三个失败类别及每类中至少四种具体模式。
- 解释为什么 Agent 失败放大了现有 AI 失败模式（偏见、幻觉）。
- 描述五种行业反复出现的模式及其缓解措施。
- 实现一个基于标准库的检测器，为 Agent 轨迹打上失败模式标签。

## 问题

团队交付的 Agent 在 90% 的轨迹上工作。那 10% 的失败不是随机噪声——它们落入少数几个反复出现的类别中。一旦能说出它们的名字，就能监控和修复它们。

## 概念

### MASFT（Berkeley, arXiv:2503.13657）

多 Agent 系统失败分类法。14 种失败模式聚为 3 个类别。标注者间 Cohen's Kappa 0.88——类别是可靠可区分的。

核心主张：**失败是多 Agent 系统的基本设计缺陷，而非可通过更好的基础模型修复的 LLM 局限。**

### 微软 Agent AI 系统失败模式分类法

- 现有 AI 失败（偏见、幻觉、数据泄露）在 Agent 环境中被放大。
- 新失败从自主性中涌现：规模化非预期动作、工具误用、使命漂移。
- 白皮书是 Agent 产品的风险登记册。

### Agent AI 中的故障特征（arXiv:2603.06847）

- 失败源自编排、内部状态演化和环境交互。
- 不只是"坏代码"或"坏模型输出"。

### LLM Agent 幻觉综述（arXiv:2509.18970）

两种主要表现形式：

1. **指令遵循偏离**——Agent 不遵循系统提示。
2. **长程上下文误用**——Agent 遗忘或错误应用早期轮次的上下文。

子意图错误：遗漏（跳过步骤）、冗余（重复步骤）、错序（步骤乱序）。

### 五种行业反复出现的模式

Arize、Galileo、NimbleBrain 2024-2026 年现场分析收敛于：

1. **幻觉动作。** Agent 调用了不存在的工具或捏造了参数。
2. **范围蔓延。** Agent 将任务扩展到用户请求之外（额外创建 PR、额外发送邮件）。
3. **级联错误。** 一个错误调用触发下游效应。一次幻影 SKU 幻觉触发四次 API 调用——一次多系统事故。
4. **上下文丢失。** 长程任务遗忘早期轮次的约束。
5. **工具误用。** 用正确参数调用错误工具，或完全调用错误工具。

**级联是杀手。** Agent 无法区分"我失败了"和"任务不可能"，经常在收到 400 错误时幻觉式声称成功以闭合循环。

### 缓解：每一步的门控

推理链每一步的自动验证门控，根据环境状态检查事实依据。具体而言：

- 每步安全分类器（第 21 课）。
- 工具调用参数验证（第 06 课）。
- 将检索到的内容与已知事实交叉检查（第 05 课，CRITIC）。
- 通过重新探测状态检测成功幻觉（文件真的被创建了吗？）。

### 失败监控在何处出错

- **只标记崩溃。** 大多数 Agent 失败产生看起来有效的输出。需要内容级检查。
- **无基线。** 漂移检测需要一个上次已知良好状态；没有它你无法说"这在变差"。
- **过度告警。** 每次失败产生一个告警页面。聚类和速率限制。

## 构建

`code/main.py` 实现一个基于标准库的失败模式标记器：

- 覆盖五种模式的合成轨迹数据集。
- 每种模式的检测器函数（工具调用、输出、重复动作的签名模式）。
- 一个标记器，为每条轨迹打标签并报告模式分布。

运行：

```bash
python3 code/main.py
```

输出：每条轨迹的标签 + 聚合分布，这是 Phoenix 轨迹聚类功能暴露的廉价复现。

## 练习

1. 添加"成功幻觉"检测器：Agent 返回成功但目标状态未改变。
2. 标记你构建的产品中的 100 条真实轨迹。哪种模式占主导？修复它的成本是多少？
3. 实现"级联半径"指标：给定步骤 N 的失败，它影响了多少下游步骤？

5. 将一个检测器接入 CI 任务：如果 ≥5% 的轨迹标记了一种模式，构建失败。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| MASFT | "多 Agent 失败分类法" | Berkeley 14 模式分类 |
| 级联错误（Cascading error） | "涟漪失败" | 一个早期错误通过 N 步传播 |
| 上下文丢失（Context loss） | "忘了约束" | 长程轮次丢弃早期轮次的事实 |
| 工具误用（Tool misuse） | "错误工具/错误参数" | 有效的调用，错误的调用方式 |
| 成功幻觉（Success hallucination） | "假完成" | Agent 在 400 上声称成功；状态未改变 |
| 范围蔓延（Scope creep） | "过度扩张" | Agent 做了超出要求的事 |
| 指令遵循偏离（Instruction-following deviation） | "不服从" | 忽略系统提示或用户约束 |
| 子意图错误（Sub-intention errors） | "计划 bug" | 计划执行中的遗漏、冗余、错序 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 14 可靠性工程模块的开篇课，标志着课程从"构建"转向"运维"。文档不是讲"怎么避免失败"——是讲"失败长什么样，怎么命名它，怎么检测它"。最大的价值在于整合了四条独立的学术/工业证据链（Berkeley MASFT、微软分类法、arXiv 综述、Arize/Galileo 现场数据），收敛于五种行业反复确认的模式。这不是学术分类——是生产事故报告。

### 二、知识结构梳理

**学术基础层：** MASFT（14 模式 × 3 类别，Kappa 0.88）、微软分类法（现有 AI 失败的放大效应）、arXiv 综述（幻觉的两种形态 + 子意图错误）——三条学术线独立得出了高度重叠的结论。

**工业收敛层：** Arize/Galileo/NimbleBrain 的现场数据将学术分类压缩为五种高频模式——幻觉动作、范围蔓延、级联错误、上下文丢失、工具误用。级联被单独加粗：它不是"一种失败模式"——是"一个失败模式的放大器"。

**缓解工程层：** 门控策略（每步验证、工具参数校验、交叉检查、状态重探）和监控策略（不止监控崩溃、需要基线、聚类告警）——将前几课的 Self-Refine、CRITIC、可观测性重新部署到"防崩溃"的上下文中。

### 三、核心洞察（备课时的关键理解）

1. **"失败是多 Agent 系统的基本设计缺陷"是 MASFT 最激进的声明。** 不是"模型不够好"——是系统的设计方式内建了某些失败模式。更好的模型不会消除级联——只会让级联的每一步看起来更可信。

2. **五种行业模式中，级联是元失败。** 其他四种模式（幻觉、范围蔓延、上下文丢失、工具误用）都可能发生——但它们单独发生时影响有限。级联把这四种中的任何一种放大为系统性故障。一次工具误用 → 错误输出 → 下游 Agent 基于错误输出推理 → 四个后续调用全错。

3. **"Agent 无法区分'我失败了'和'任务不可能'"是生产中最危险的认知缺陷。** Agent 收到 HTTP 400 时，它可能说"请求失败，重试"——但也可能说"任务完成"。为什么？因为 LLM 的输出不直接反映工具调用的返回值——它"解释"结果。如果解释逻辑出错——成功幻觉。

4. **"只监控崩溃"是 Agent 可观测性的第一大盲区。** Agent 崩溃了——你收到告警。Agent 返回了一个看起来正确但实际错误的结果——你什么都没收到。Agent 的可观测性不能只依赖异常——需要**内容级检查**。

5. **漂移检测需要基线。** "这个月 Agent 的失败率上升了"——上升到什么？你没有上个月的基线。"工具误用率从 2% 涨到 5%"——需要一个 last-known-good 数据点。没有基线，漂移检测是一句空话。

6. **子意图错误（遗漏、冗余、错序）是"计划执行"的质量维度。** 大多数 Agent 评测只看最终结果——"任务完成了吗？"但子意图错误评测看的是"步骤对不对、有没有漏、有没有重复、顺序对不对"。后者是更细粒度的质量信号。

7. **微软分类法的一个独特贡献是"规模化效应"。** 单次 Agent 的工具误用是 bug。1000 个 Agent 同时工具误用是事故。Agent 系统的规模化放大了每个失败模式的影响半径。

### 四、教学建议

1. **用真实事故报告开场。** 找一个 Agent 生产事故（幻影 SKU 导致四次 API 调用的事故故事是完美的开场案例），让学生感受"一次幻觉 → 一次错误调用 → 级联 → 生产事故"的因果链。

2. **五种模式各自用一个"你能一眼认出来"的签名来讲。** 幻觉动作 = 工具名不存在。范围蔓延 = 输出长度远超过需求。级联错误 = 一步的错误类型在后续步骤中扩散。上下文丢失 = 早期约束在后期消失了。工具误用 = 工具调了但参数全错。

3. **"成功幻觉"是最值得单讲的模式。** 写一个 Agent，让它调用一个永远返回 400 的 API，看它怎么报告结果。大概率是"任务完成"。这个 demo 的冲击力比其他任何理论讲解都强。

4. **MASFT 的 14 模式不适合逐条讲——适合分组映射。** 把 14 模式映射到五种行业模式上——哪些是同一模式的不同学术表达。让学生看到学术和工业的分类学是怎么对齐的。

5. **门控策略应该是实操环节。** 给一个带失败模式的轨迹，让学生写一个门控检测器，在轨迹上跑，看哪些失败被捕获、哪些漏了。这是练习 1-3 的直接教学点。

6. **练习 2（标记 100 条真实轨迹）是最诚实的练习。** 如果你没有自己的产品轨迹——用开源数据。哪种模式在你以为"没问题"的系统里实际上占主导？这是每个工程师都应该知道的关于自己系统的事。

7. **练习 5（CI 集成）将"故障检测"从运维提升为工程实践。** "≥5% 的轨迹标记了级联错误 → 构建失败"——这是把可靠性工程从"出了事再查"变成"上线前就知道"。

### 五、值得补充的内容

1. **失败模式的频率分布。** 五种模式在真实系统中的发生频率是多少？幻觉动作是最常见的吗？级联错误是最高影响的吗？需要统计数据来优先排序修复工作。

2. **模式之间的因果关系。** 上下文丢失 → 工具误用 → 级联错误——这是一个常见的因果链。独立检测每种模式 vs 检测因果链——哪种更有效？

3. **失败模式在不同 Agent 架构中的差异性。** 监管者-工作者模式更容易出现级联（因为一个专家错误会影响所有后续专家）。群体模式更容易出现范围蔓延（因为没有中心控制）。不同拓扑有不同的脆弱性剖面。

4. **修复成本矩阵。** 修复幻觉动作需要更好的工具描述。修复级联需要门控。修复上下文丢失需要更好的记忆/压缩策略。每种模式的修复成本和修复难度不同。

5. **与安全护栏的关系。** 第 16 课（OpenAI Agents SDK 护栏）和第 17 课（Claude Agent SDK 钩子）中的护栏机制——能捕获五种失败模式中的哪几种？哪些需要额外的检测逻辑？

### 六、一句话总结

**Agent 失败的秘密不是"有时候它会出错"——是"它只以五种方式出错，而且其中四种可以通过门控检测，级联可以通过在第一步就截断来防止。"学会命名失败是学会修复失败的第一步。**

# 🎓 Agent 架构课：失败模式——为什么你的 Agent 90% 的情况工作，10% 的情况让你凌晨三点被叫醒

同学们好。我是你们的 Agent 架构老师。

今天这节课和前面八节都不一样。前面我在教你们怎么搭 Agent。今天我在教你们**怎么知道你的 Agent 什么时候在骗你。**

这不是修辞。Agent 真的会骗你。它会在收到 HTTP 400 时告诉你"任务完成"。它会在你没要求的时候多发三封邮件。它会因为第一步出了一个微小的错，导致后面十五步全崩——而它从来不告诉你"我第一步就错了"。

我们今天把这 10% 的失败命名——五种类型。命名的目的是什么？**监控。** 你不命名它，你就没法在日志里搜到它。

## 五种类型，五种签名

### 类型一：幻觉动作——"我调用了这个工具"→ 这个工具不存在

Agent 说它调用了 `delete_database`。你的系统里没有这个工具。这看起来像是个笑话——怎么会有 Agent 调用不存在的工具？但生产里天天发生。

为什么？因为 LLM 在看到"清理临时文件"这个任务时，它"推理"出应该有一个 `delete_temp_files` 工具。工具列表里没有——但它"觉得应该有"。于是它伪造了调用。

签名：**工具名不在已注册的工具列表中。** 检测器 = 每次工具调用时检查工具名是否在注册表里。

### 类型二：范围蔓延——"我问你要一杯水，你给我修了个水库"

用户说："帮我在代码库里加一行注释。"Agent 说："好的，我还顺便重构了这个模块、更新了 README、创建了一个 PR、在 Slack 里通知了团队。"

用户没说这些。Agent 觉得"这些都是对你有好处的事情"——但它忘了问。

签名：**产出的数量或类型超出任务定义的范围。** 检测器 = 比较预期输出和实际输出的范围差异。

### 类型三：级联错误——"第一步歪了，后面全歪了"🔴

这是五种模式里最危险的。为什么？因为它不是"一种失败"——是**"一个失败被放大了十五倍的机制。"**

Agent 在第一步幻觉了一个不存在的 SKU `X-42`。第二步用这个 SKU 去查价格 → API 返回 404。第三步基于"价格未知"推断"产品已下架"→ 更新库存为 0。第四步通知运营"产品已下架"→ 运营开始排查。**一个幻觉 SKU，四次错误操作，一次跨团队事故。**

签名：**步骤 N 的错误类型在步骤 N+1、N+2... 中以因果方式复现。** 检测器 = 找到第一个错误调用，计数下游有多少步是基于它的输出。

致命之处：Agent 不知道它第一步错了。LLM 不会说"等等，SKU `X-42` 可能是我编的"——它相信自己的输出。

### 类型四：上下文丢失——"你说过不能超过 100 行，但我的第 15 步推理已经忘了"

用户在第一轮说："代码不能超过 100 行。"Agent 确认了。

第三轮、第五轮、第十轮——Agent 在生成代码。到第十五轮，Agent 产出了 300 行代码。约束在第一轮——已经超出上下文窗口的活跃范围。

签名：**早期轮次中存在的约束在后期轮次中消失。** 检测器 = 提取第一轮的约束，在最后一轮的输出中检查是否仍被遵守。

### 类型五：工具误用——"正确的工具，错误的参数"或"错误的工具"

`create_file(path="/etc/passwd", content="...")`——模型知道 `create_file` 是什么，但它没有检查 `path` 参数是否安全。

`search_database(query="DROP TABLE users")`——模型应该用 `read_only_query` 工具，但它选了 `search_database`。

签名：**工具名存在且有效，但参数不合理或工具选择错误。** 检测器 = 工具调用的参数验证 + 可选工具列表检查。

## 级联：为什么它是最危险的

我单独拉出来讲，因为级联是所有生产事故的共同模式。

Agent 调用了一个 API → API 返回了 400。这是第一步。第二步：Agent 读取 400 响应。它看到 `{"error": "product_not_found"}`。它需要判断——"这是产品不存在，所以我该告诉用户产品不存在。"——这看起来正确。

但它没有判断的是：**产品不存在的根本原因是不是我编造了产品 ID？**

这一步的判断 LLM 不会做。它只看到"API 说产品不存在"，它推理"那我告诉用户产品不存在"。它不回溯——"等一下，这个产品 ID 是我三秒钟前生成的，可能根本就不存在。"

这就是级联的本质：**错误发生在第 N 步，但系统在第 N+1 到 N+M 步都把它当作正确信息来处理。** 没有人说"停"。

缓解措施——**门控。** 每一步之后，检查这一步的结果是否合理：
- API 返回了 404？→ 检查"我请求的 ID 是从哪来的？是用户提供的还是我生成的？"如果我生成的 → 重试，生成新 ID 或者承认失败。
- 工具调用参数是文件路径？→ 检查路径是否在允许目录内。
- 输出包含了用户 PII？→ 拦截，不发给下游。

门控不是"AI 自己判断"——门控是**代码检查。** 确定性逻辑。不需要 LLM。

## 监控：怎么知道你的 Agent 在崩

三件事：

**第一，不要只监控崩溃。** 异常监控告诉你 Agent 挂了。但大多数 Agent 失败不抛异常——它们返回一个看起来对但实际错的结果。你需要内容级检查：输出是否满足约束？工具调用参数是否在合理范围？

**第二，建立基线。** "这个月幻觉动作率是 3%"——3% 是好是坏？你不知道，因为你没有上个月的数据。基线 = last-known-good。每两周跑一次基线扫描——你才能说"这个星期比上星期差了"。

**第三，聚类告警。** 不要每次失败发一个告警页面。把失败按模式聚类——"过去一小时，级联错误的频率是平日的 5 倍"——这是一条告警。一条，不是 57 条。

## 结课清单

如果你想你的 Agent 不被凌晨三点的电话叫醒：

1. 五种失败模式：幻觉动作、范围蔓延、级联、上下文丢失、工具误用——每个有一个可检测的签名。
2. 级联是最危险的——不是"一种失败"，是失败放大器。门控在第一步就截断。
3. 成功幻觉是生产中最常见的谎言——Agent 说"做完了"，状态没变。通过重探状态检测。
4. 监控三原则：不止崩溃、需要基线、聚类告警。
5. MASFT 给你 14 种模式——五种行业模式是它的高频子集。从五种开始，不够再扩展到 14 种。
6. CI 中的失败检测 = ≥5% 轨迹标记 → 构建失败。不让 bug 进入生产比修复生产中的 bug 便宜。

**最后一句话：**

**Agent 不是在"有时候会出错"——它在五种具体的方式中出错。命名它们，为每种写一个检测器，在每一步放一个门控。你不会把失败率降到零——但你会把"不知道哪里崩了"变成"凌晨三点醒来，看了一眼仪表盘，继续睡了。"**
