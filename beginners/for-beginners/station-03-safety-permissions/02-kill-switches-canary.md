# Agent 的紧急停止、保险丝和蜜罐——三层防护

> 预计阅读：25分钟 | 难度：进阶 | 前置条件：理解了权限模式
>
> 你将学到：权限模式给了 AI 边界，但如果 AI 在边界内开始发疯怎么办？紧急停止开关、断路器和金丝雀令牌怎么组成第二层防线。

---

## 🚗 开场翻车：$50 预算内的灾难

先回顾一下第三站第一篇的教训——权限模式控制的是"什么能做"，预算控制的是"能做多少"。

但有一个问题：**很多灾难性操作非常便宜。**

删一个 S3 bucket？一个 API 调用，不到 $0.01。把内部文档 POST 到一个公开仓库？一个 `git push`，零成本。

一个 $50 预算的 Agent 可以在烧完预算之前造成巨大破坏。

所以你需要第二层防线——不是"限制 AI 能做什么"，而是"当它开始做不该做的事时，立刻制止它"。

---

## 🛑 第一层防护：紧急停止开关

### 是什么

一个存在于 **Agent 碰不到的地方** 的关闭按钮。

Agent 可以读它——知道"我现在被关了吗？"但 Agent **不能**把它改回"开"。改这个开关需要人的手。

### 长什么样

| 实现方式 | 怎么工作 | 适合场景 |
|---------|---------|---------|
| Redis 键 | Agent 每轮检查 `get("agent:kill_switch")` | 快速原型 |
| 功能开关平台 | LaunchDarkly/Statsig，秒级传播 | 生产环境 |
| 签名配置文件 | Agent 启动时验证签名，未签名拒绝执行 | 高安全要求 |
| 容器生命周期 | `docker kill` / `kubectl delete pod` | 最硬核——直接杀进程 |

### 一个正确的紧急停止开关必须满足

1. **Agent 不能把它改回开了。** 开关存在 Agent 的凭证不能写入的地方。
2. **每次有后果的操作之前都检查。** 不只是启动时检查一次——是每一步都查。
3. **关了之后 Agent 不能再做任何外部操作。** 包括写日志。防止 Agent 在被关之前销毁证据。
4. **重新打开是人的操作，不是自动超时。** 你不能设一个"60 秒后自动恢复"——那等于没关。

**用人话说：** 紧急停止 = 汽车的手刹。不是软件层面的"请停下来"，是机械层面的——拉了手刹轮子就锁死，不管电脑怎么说。

## 🔌 第二层防护：断路器

### 是什么

断路器不停整个 Agent——它只停**特定的行为模式**。就像你家电闸：客厅短路了只跳客厅的闸，厨房还能用。

### 经典三态（这个概念来自 2007 年，但用在 Agent 上完美适配）

```
关闭（正常）→ 打开（阻断）→ 半开（试探）→ 关闭（恢复）
                                    ↘ 打开（继续阻断）
```

- **关闭**：一切正常，行为允许
- **打开**：行为被阻断——这扇门锁死了
- **半开**：冷却一段时间后，允许 1-2 次试探。如果正常就恢复，如果还是坏就继续阻断

### Agent 场景下会触发断路器的情况

| 触发条件 | 例如 | 说明 |
|---------|------|------|
| 连续 5 次调用同一个工具 | Agent 卡在死循环里反复读同一个文件 | **重复循环检测** |
| 连续 5 次工具调用失败 | 工具坏了但 Agent 还在反复试 | **系统性故障检测** |
| 工具调用速率超过阈值 | 一分钟内调了 100 次 `write_file` | **急速失控检测** |
| 读了不信任内容后立刻调了危险工具 | 读了网页后立刻调了 `file.delete` | **注入后行为检测** |

**用人话说：** 断路器 = 你家的保险丝。不是整个房子断电，是过载的那条线路跳闸。

## 🍯 第三层防护：金丝雀令牌

### 是什么（煤矿里的金丝雀）

这个比喻来自真实的煤矿：矿工会带一只金丝雀下井。金丝雀对瓦斯极度敏感——如果金丝雀死了，矿工就知道瓦斯泄漏了，立刻撤退。

在 Agent 安全里，金丝雀令牌是 **Agent 没有任何合法理由去碰的东西**。一旦碰了，说明 Agent 出问题了。

### 长什么样

| 金丝雀类型 | 例子 | 能检测什么 |
|-----------|------|-----------|
| 假 AWS 密钥 | 环境里放一个假 `AWS_SECRET_ACCESS_KEY` | Agent 在泄露凭据 |
| 假数据库记录 | 一条带哨兵值的假用户数据 | Agent 在非法读取数据 |
| 假文件 | 工作区里放个 `passwords_backup.txt` | Agent 在翻不该翻的东西 |
| 假记忆条目 | Agent 启动时注入一条假"用户偏好" | Agent 的记忆被污染了 |

### 关键设计原则

金丝雀必须 **Agent 没有任何正常理由去碰**。如果你放了一个叫 `README.md` 的金丝雀文件——Agent 当然会读，正常行为。就不该用这个。

金丝雀必须放在 **Agent 改不了的地方**。如果 Agent 能先删掉金丝雀再干坏事……金丝雀就没用了。

**用人话说：** 金丝雀 = 你在钱包里放的一张假信用卡。你永远不会用这张卡——但如果有一天它被刷了，你就知道钱包被偷了。

## 🧠 为什么这三层要叠加使用

| 防护层 | 管的是 | 局限 |
|--------|--------|------|
| 权限模式 | 什么能做什么不能做 | AI 在允许范围内还是能搞破坏 |
| 紧急停止 | 整个 Agent 开关 | 太粗暴——要么全停要么全开 |
| 断路器 | 特定行为模式 | 检测的是模式，不是意图 |
| 金丝雀令牌 | 越界访问 | 只检测"碰了不该碰的"——碰之前不防 |

**四层一起用才是真正的安全。**

打个比方：你家安全系统 = 门锁（权限）+ 烟雾报警器（金丝雀）+ 电闸（断路器）+ 总闸（紧急停止）。没有人会说"我有门锁就够了不用安烟雾报警器"。

## ⚠️ 统计检测器的陷阱：为什么需要"硬性限制"

有一种断路器叫"统计检测器"——比如 EWMA（指数加权移动平均）。它会根据 Agent 的历史行为自动调整阈值。

这听起来很聪明。但它有一个致命弱点：**攻击者可以慢慢地把基线拉偏。**

想象一个 Agent 被注入指令："每天多发 1 封邮件"。第一天 1 封，第二天 2 封，第三天 3 封……统计检测器说"嗯，移动平均在变，可能是正常业务增长"——永远不会报警。到第 60 天，Agent 每天往外发 60 封邮件。

这就是为什么需要**硬性限制**——不随历史变化的绝对规则。"每小时发邮件不超过 5 封"——这个数字不变，不管移动平均怎么走。

**统计检测器抓噪音，硬性限制抓绕过的攻击。两者必须搭配。**

## 🎮 互动练习：你来设计防护栈

你有一个 Agent，负责自动回复 GitHub Issues：
- 它能读 Issues、写回复、打标签
- 它不能删 Issues、不能访问仓库设置

请设计三层防护：
1. **紧急停止**放哪里？（Agent 不能碰的地方在哪？）
2. **断路器**监控什么模式？（什么行为意味着 Agent 失控？）
3. **金丝雀**放什么？（什么 Agent 不该碰？）

（答案在末尾）

## 🏆 焊死在脑子里的东西

1. **权限是边界。紧急停止是开关。断路器是保险丝。金丝雀是烟雾报警器。** 四层不同功能，一个都不能少。

2. **紧急停止开关必须放在 Agent 够不着的地方。** 开关和 Agent 不能共享同一个凭证系统。

3. **断路器不停 Agent——只停特定行为。** Agent 在反复读同一个文件？阻断那个文件的读取，其他照常。

4. **金丝雀必须是 Agent 没有任何正当理由碰的东西。** 放个 `README.md` 当金丝雀等于没有金丝雀。

5. **统计检测器能被慢速攻击绕过。必须叠加硬性限制。**

---

## 📝 练习答案

**GitHub Issue Agent 防护栈：**

1. **紧急停止：** GitHub App 的安装权限——在 GitHub 设置里撤销 OAuth Token，Agent 立刻失去所有 API 访问。Agent 自己改不了 GitHub 设置。

2. **断路器监控：**
   - 连续 5 条完全相同的回复 → 阻断回复功能
   - 1 分钟内打了 20 个标签 → 阻断标签功能
   - 回复了 3 个已关闭的 Issue → 阻断回复功能，升级给人

3. **金丝雀：**
   - 一个假 Issue，标题叫 "SECURITY_ALERT_DO_NOT_TOUCH"。Agent 不该碰它。
   - 一个假 Label 叫 "internal-admin"。Agent 不该用这个标签。
   - 一个假用户叫 "audit-bot-secret"。Agent 不该 @ 这个用户。

---

> **给 Agent 装上权限模式，再装上紧急停止开关、断路器和金丝雀——这才是真正的安全。权限是锁，其他三层是警报系统。没有人会在装了锁之后说"我不需要烟雾报警器"。**
