# Agent 可观测性：Langfuse、Phoenix、Opik

> 三个开源 Agent 可观测性平台主导 2026 年。Langfuse（MIT）——每月 6M+ 安装，追踪 + 提示管理 + 评估 + 会话回放。Arize Phoenix（Elastic 2.0）——深度 Agent 特定评估、RAG 相关性、OpenInference 自动仪表化。Comet Opik（Apache 2.0）——自动提示优化、护栏、LLM 评判者幻觉检测。

**类型：** 学习
**语言：** Python（标准库）
**前置课程：** Phase 14 · 23（OTel GenAI）
**用时：** 约 45 分钟

## 学习目标

- 说出三个顶级开源 Agent 可观测性平台及其许可证。
- 区分每个平台的最强项：Langfuse（提示管理 + 会话）、Phoenix（RAG + 自动仪表化）、Opik（优化 + 护栏）。
- 解释为什么到 2026 年 89% 的组织报告已有 Agent 可观测性。
- 实现一个基于标准库的追踪到仪表盘流水线，带 LLM 评判者评估。

## 问题

OTel GenAI（第 23 课）给你 schema。你仍然需要接收 span、运行评估、存储提示版本、并暴露回归的平台。三个竞争者各自强调生命周期的不同部分。

## 概念

### Langfuse（MIT）

- 每月 6M+ SDK 安装，19k+ GitHub stars。
- 功能：追踪、带版本控制 + 游乐场的提示管理、评估（LLM 作为评判者、用户反馈、自定义）、会话回放。
- 2025 年 6 月：先前商用的模块（LLM 评判者、标注队列、提示实验、Playground）在 MIT 下开源。
- 最强：端到端可观测性，带紧密的提示管理循环。

### Arize Phoenix（Elastic License 2.0）

- 更深层的 Agent 特定评估：追踪聚类、异常检测、RAG 检索相关性。
- 原生 OpenInference 自动仪表化。
- 与托管 Arize AX 配对用于生产。
- 无提示版本控制——定位为更广泛平台旁的漂移/行为回归工具。
- 最强：RAG 相关性、行为漂移、异常检测。

### Comet Opik（Apache 2.0）

- 通过 A/B 实验自动提示优化。
- 护栏（PII 编辑、主题约束）。
- LLM 评判者幻觉检测。
- Comet 自测基准：Opik 日志 + 评估 23.44s vs Langfuse 327.15s（约 14 倍差距）——将供应商基准视为方向性参考。
- 最强：优化循环、自动实验、护栏执行。

### 行业数据

据 Maxim（2026 年现场分析）：89% 的组织已有 Agent 可观测性；质量问题是最大的生产障碍（32% 的受访者提及）。

### 如何选择

| 需求 | 选择 |
|------|------|
| 带提示管理的一体化方案 | Langfuse |
| 深度 RAG 评估 + 漂移 | Phoenix |
| 自动优化 + 护栏 | Opik |
| 开放许可，非 ELv2 | Langfuse（MIT）或 Opik（Apache 2.0） |
| Datadog / New Relic 集成 | 任意——它们都导出 OTel |

### 此模式在何处会出错

- **无评估策略。** 没有评估的追踪只是昂贵的日志。
- **自建 LLM 评判者无真值锚定。** CRITIC 模式（第 05 课）适用——评判者需要外部工具做事实验证。
- **提示版本未绑定到追踪。** 当生产回归时，你无法二分查找是哪个提示导致的。

## 构建

`code/main.py` 实现一个基于标准库的追踪收集器 + LLM 评判者评估器：

- 接收 GenAI 形态的 span。
- 按会话来分组，标记失败的运行（护栏拦截、低置信度评估）。
- 一个脚本化 LLM 评判者，按评分量表对 Agent 响应评分。
- 一个类仪表盘摘要：失败率、top 失败原因、评估分数分布。

运行：

```bash
python3 code/main.py
```

输出：每个会话的评估分数和失败分类，匹配 Langfuse/Phoenix/Opik 所展示的内容。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 追踪（Tracing） | "Span 收集器" | 接收 OTel / SDK span；按会话索引 |
| 提示管理（Prompt management） | "提示 CMS" | 绑定到追踪的版本化提示 |
| LLM 作为评判者（LLM-as-judge） | "自动评估" | 独立 LLM 按评分量表对 Agent 输出评分 |
| 会话回放（Session replay） | "追踪回放" | 逐步回溯过往运行以进行调试 |
| RAG 相关性（RAG relevancy） | "检索质量" | 检索到的上下文是否匹配查询 |
| 追踪聚类（Trace clustering） | "行为分组" | 聚类相似运行以进行漂移检测 |
| 护栏执行（Guardrail enforcement） | "日志时策略" | 对已记录内容的 PII/毒性/范围检查 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 15 可靠性工程模块的工具选型课。前三节课讲了"什么会崩""怎么知道没崩""怎么防注入"——这节课回答"用哪个平台来看这些"。文档的价值在于把三个开源平台（Langfuse、Phoenix、Opik）从功能沼泽中提炼为三个清晰的最强项和一张决策表。不是"哪个更好"——是"你的需求匹配哪个最强项"。适合已理解 OTel 追踪、正在选型可观测性平台的架构师。

### 二、知识结构梳理

**平台对比层：** 三平台在三个维度上分化——Langfuse 最全（追踪+提示+评估）、Phoenix 最深（RAG+漂移+异常）、Opik 最自动（优化+护栏+实验）。许可证差异（MIT vs ELv2 vs Apache 2.0）对商业用户是硬约束。

**工程构建层：** `code/main.py` 展示了一个平台的底层逻辑——接收 span → 按会话分组 → 标记失败 → LLM 评分 → 聚合摘要。不管用哪个平台，这五个步骤是通用的。

**行业数据层：** 89% 的组织已有 Agent 可观测性——这个数字说明可观测性已从"前沿实践"变成"基线要求"。32% 把质量问题列为最大生产障碍——说明有可观测性不等于有质量，评估策略才是关键。

### 三、核心洞察（备课时的关键理解）

1. **"追踪没有评估 = 昂贵的日志"是本节课的核心公式。** 收集 span 只是第一步。如果你不对这些 span 做评估——不标记失败、不检测漂移、不给输出打分——你只是把日志从一个地方搬到了另一个地方。可观测性的价值在评估层，不在收集层。

2. **三平台的分化是"全栈 vs 深度 vs 自动"的经典竞争格局。** Langfuse = 瑞士军刀（什么都有）。Phoenix = 显微镜（RAG 和漂移看得最细）。Opik = 自动驾驶（优化和护栏自动跑）。没有"最好的"——有"和你的需求形态最匹配的"。

3. **Opik 的 14 倍速度差距声明需要质疑，但方向是对的。** 供应商基准测试不可信——但 Opik 的定位（轻量日志 + 快速评估）和 Langfuse 的定位（全栈平台）确实在延迟上有本质差异。选型时自己测量，不要信供应商数字。

4. **"提示版本未绑定到追踪"是生产回归排障的最大障碍。** Agent 变差了——是模型换了？是提示词改了？是工具变了？如果追踪里没有"这次运行用的是 prompt_v3"——你只能猜。Langfuse 是三个中唯一原生支持提示版本化的。

5. **RAG 相关性评估是 Phoenix 的差异化武器。** 大多数 Agent 失败的根本原因是检索到了不相关的内容。Phoenix 能告诉你"这个查询检索到的 5 篇文档中，只有 2 篇是相关的"——这个指标比任何通用评估都更有诊断价值。

6. **ELv2 许可证是 Phoenix 的商业限制。** Langfuse（MIT）和 Opik（Apache 2.0）对商业使用没有限制。Phoenix 的 ELv2 禁止托管服务竞争——如果你想把 Phoenix 作为 SaaS 的一部分提供，需要商业许可。

7. **"LLM 评判者需要外部工具锚定"是第 05 课 CRITIC 在可观测性领域的直接应用。** 用 LLM 评判 Agent 的输出——如果 LLM 评判者也幻觉，你就有了"两个 LLM 互相确认错误"的问题。CRITIC 的解决方案：让评判者以外部工具为锚定——不是"你觉得对吗"，是"测试通过了吗""事实能验证吗"。

### 四、教学建议

1. **用一个真实 Agent 的 OTel 导出开场。** 如果你有生产 Agent 的追踪数据——直接导入 Langfuse 免费层，展示按会话分组的失败分布。让学生看到"你的 Agent 上周的失败都集中在哪一种模式"——这是可观测性价值的直观冲击。

2. **三平台用"一句话总结"对比。** Langfuse = "如果你只能装一个，装这个。" Phoenix = "如果你的主要问题是 RAG 检索不准，装这个。" Opik = "如果你想让系统自动优化提示词，装这个。"让学生记住三个句子，比记住功能列表有用。

3. **"追踪 vs 评估"的区别值得一个对比 demo。** 展示两个仪表盘——一个只有 span 收集（原始追踪），一个有评估覆盖（标记失败+评分）。问学生"哪个能让你在凌晨三点找到问题？"

4. **提示版本化的重要性用"回归二分查找"来讲。** Agent 上周表现好，这周差了。四天里改了 8 次提示词。你只能一次一次回滚来测试——除非追踪绑定了提示版本，你直接看"prompt_v5 以后的通过率暴跌了 15%"。

5. **许可证差异对商业用户是硬约束，需要单讲。** 如果你公司在用 Phoenix 做内部可观测性——ELv2 没问题。如果你想在 SaaS 产品里嵌入 Phoenix——ELv2 禁止。让有商业背景的学生意识到"选框架时还要看许可证"。

6. **练习 5（自己测基准）是最诚实的练习。** 供应商说 Opik 比 Langfuse 快 14 倍。你在你的数据上跑——可能差距是 3 倍，可能是 20 倍。**不测就信供应商数字 = 工程上的盲信。**

7. **三个练习（导出追踪、写评分量表、对比版本控制 vs 聚类）应该在不同周做。** 导出追踪 → 第一周：建立基线。写评分量表 → 第二周：定义质量。对比 → 第三周：优化排障流程。可观测性的价值随时间累积。

### 五、值得补充的内容

1. **三平台的成本模型对比。** 自托管（你自己的服务器）vs 云（按 span 量计费）。Langfuse 云有免费层（50K span/月）。Phoenix 自托管免费。Opik 云有免费层。对于日均 10K span 的系统——每年成本多少？

2. **跨平台迁移成本。** 从 Langfuse 迁移到 Phoenix 需要重新仪表化吗？OTel 导出是标准化的——但评估、提示版本、会话回放是平台专有的。迁移成本不在追踪层，在评估和工作流层。

3. **评估的采样策略。** 你不能对 100% 的追踪做完整的 LLM 评判者评估——太贵了。采样 10%？基于风险分层？低置信度运行 100% 评估，高置信度 1%？

4. **与第 11 课（评估驱动开发）的集成。** 第 11 课教你建立 CI 中的评估套件。这节课的平台是这些评估的运行时——Langfuse/Phoenix/Opik 就是 CI 评估的数据源和可视化层。

5. **多 Agent 追踪的复杂度。** 单 Agent 追踪是一棵树。多 Agent 追踪（编排器 → 子 Agent A → 子 Agent B）是森林。三平台对多 Agent 追踪的支持差异——Phoenix 在聚类上更强，Langfuse 在会话回放上更强。

### 六、一句话总结

**可观测性平台不是"装上去就能看见问题"——是"你首先得定义了什么叫'有问题'（评估），平台才能告诉你问题在哪（追踪），然后你才能知道什么时候出了什么问题（提示版本 x 追踪 x 时间）。装平台是第一步，定义评估是第零步。"**

# 🎓 Agent 架构课：可观测性——三个平台，三个答案

同学们好。我是你们的 Agent 架构老师。

前面三节课——失败模式、评估驱动、提示注入——我教了你"什么会崩""怎么知道没崩""怎么防别人让你崩"。这些课有一个共同的前提：**你看得见你的 Agent。**

什么叫"看得见"？不是你看过它的输出——是你能回答这些问题：

- 上个星期三的下午两点，Agent 在第 7 步做了一个什么工具调用？
- 那个工具调用的参数是什么？
- 那次调用成功了还是失败了？
- 那之后的三步有什么连锁反应？
- 和上个月比，Agent 的失败率是升了还是降了？

如果你不能回答——你看不见你的 Agent。你只是在看它的输出，不是在观测它的行为。

今天这节课讲三个平台。一个给你全栈、一个给你深度、一个给你自动化。不讲"哪个更好"——讲"你的需求是什么，哪个匹配它。"

## 不是"装了就完了"

我先讲一个反直觉的事。**大多数团队装了可观测性平台后，唯一的改进是"日志更好看了"。**

为什么？因为他们跳过了第零步——**定义"什么叫有问题"。**

追踪告诉你"Agent 做了 A，然后做了 B"。但它不告诉你"A 对不对"。

评估告诉你"A 对不对"。但它需要你定义"对"是什么意思。

这三个平台的核心差异，不在于谁收集 span 更快——在于**他们在追踪之外还帮你做什么。**

## Langfuse：如果你只能装一个

Langfuse 是三个中最全的。追踪、提示管理、评估、会话回放——四个核心模块都在一个平台里。

**追踪：** 接收 OTel span 或其 SDK。按会话分组——你能看到"用户 X 的完整对话，从第一句话到 Agent 的最后回复"。

**提示管理：** 这是 Langfuse 和另外两个最大的区别。你的每一次提示词修改都有版本号。追踪里的每条 span 标记了"这次运行用的是 prompt_v3"。生产回归怎么办？直接看"prompt_v5 以后通过率跌了 18%"就找到了。

**评估：** LLM 评判者 + 用户反馈 + 自定义评估。你可以让 Langfuse 自己跑 LLM 评判者对每次会话的回复打分。

**会话回放：** 点进一条失败的追踪，逐步看 Agent 的每一步——就像看录像回放。

2025 年 6 月，Langfuse 做了一个关键决定——把之前商用的模块（LLM 评判者、标注队列、提示实验、Playground）在 MIT 下开源。这意味着**你现在不需要付费就能用它的全部功能。**

## Phoenix：如果你的 Agent 靠 RAG 活着

Phoenix 的核心不是"全"——是"深"。

大多数 Agent 失败的根本原因不是模型输出错了——是**检索回来的东西不对。** 用户问"怎么退款"，RAG 检索回来的是"退款政策已更新"——但这份文档是三年前的。Agent 基于过时内容回答了错误的退款流程。

Phoenix 的 RAG 相关性评估能告诉你：针对这个查询，检索回来的 N 篇文档中，有几篇是真正相关的。**当这个数字下降时——你要么换了 embedding 模型，要么文档库被污染了。**

Phoenix 还有两个独特能力：**追踪聚类**（自动把相似的失败运行聚在一起——"这 200 次失败其实全是同一类问题"）和**异常检测**（"本周的工具调用参数分布突然变了"）。

但它有一个限制：**ELv2 许可证。** 如果你想把 Phoenix 嵌入你的 SaaS 产品——ELv2 禁止。如果你只在内部用——没问题。

## Opik：如果你想让系统自己优化

Opik 的哲学是"自动化"。不是"我帮你看"——是"我帮你改"。

**自动提示优化：** Opik 跑 A/B 实验——提示词 A 和提示词 B 同时在线，看哪个评估分数更高，然后自动切换。

**护栏：** PII 编辑（自动检测并删除追踪中的个人信息）、主题约束（"Agent 不应该讨论这个话题"）——在日志层执行策略，不只是警告。

**LLM 评判者幻觉检测：** 这个很特别——Opik 用一个额外的检查来验证 LLM 评判者本身是否可靠。评判者说"这个答案 9/10"——Opik 的幻觉检测问"评判者给出的理由和评分一致吗？"——这是一层元评估。

Comet 说 Opik 比 Langfuse 快 14 倍。你不应该信这个数字——供应商基准测试不可信。但你应该自己测。在你的数据上跑，看真实的差距是多少。

## 89% 已经在用了——你呢？

2026 年的行业数据：89% 的组织报告已有 Agent 可观测性。这个数字的意思是：**如果你还没有，你的同行已经有了。** 你凭什么觉得你的 Agent 比他们的更不容易崩？

但同一个调查里有另一个数字：32% 的组织说质量问题是最大生产障碍。有可观测性 ≠ 有质量。装平台只是第一步。第二步是定义评估——什么算好、什么算坏。第三步是建立从"发现回归"到"定位原因"的流程（提示版本 x 追踪 x 时间）。

没有评估策略的追踪 = 昂贵的日志。没有提示版本绑定的追踪 = 回归时你只能猜。

## 结课清单

选可观测性平台：

1. 你需要提示版本管理吗？→ 只有 Langfuse 原生支持。
2. 你的主要问题是 RAG 检索不准吗？→ Phoenix 最深。
3. 你想让系统自动优化提示词吗？→ Opik 唯一。
4. 你在做 SaaS 产品？→ Langfuse（MIT）或 Opik（Apache 2.0），避免 ELv2。
5. 你已经有 Datadog/New Relic？→ 三个都导出 OTel，随便选。
6. 最重要的一条：**平台装上之后的第一件事不是庆祝——是定义你的评估。** 没有评估，平台只是昂贵的日志查看器。

**最后一句话：**

**可观测性平台给你的不是"看见问题的能力"——是"在问题发生之后，你能多快找到它、定位它、证明是哪个改动引起的"。快不快取决于三件事：追踪有没有评估、提示有没有版本、失败有没有聚类。做了这三件事，凌晨三点的告警从"Agent 崩了"变成"prompt_v5 的通过率从 94% 跌到了 87%，回滚到 v4，明天早上再说。"**
