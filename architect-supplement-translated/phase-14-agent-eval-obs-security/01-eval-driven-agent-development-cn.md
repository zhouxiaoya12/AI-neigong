# 评估驱动的Agent开发

> Anthropic 的指导方针："从简单提示开始，通过全面评估优化它们，仅在必要时添加多步骤的Agent系统。"评估不是最后一步。它是驱动阶段14中每一个其他选择的外部循环。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 阶段14全部内容。
**时间：** ~60分钟

## 学习目标

- 说出三层评估——静态基准测试、自定义离线评估、在线生产评估——以及每一层的用途。
- 解释评估器-优化器的紧密循环。
- 描述2026年的最佳实践：评估与代码共存、在CI中运行、对PR进行门控。
- 将阶段14的每一课与它所产生的评估案例联系起来。

## 问题

Agent在演示中表现良好，但在生产中却以演示无法预测的方式失败。基准测试回答的是"这个模型整体能力强吗？"，而不是"这个Agent是否为我的产品提交了正确的补丁？"。答案在于：三层评估，持续运行，每个护栏和学习规则都映射到一个评估案例。

## 核心概念

### 三层评估

1. **静态基准测试** — SWE-bench Verified用于代码（第19课），WebArena/OSWorld用于浏览/桌面（第20课），GAIA用于通用型（第19课），BFCL V4用于工具使用（第06课）。用于跨模型比较和回归门控。污染是真实存在的：SWE-bench+发现32.67%的解决方案泄露。始终报告Verified/+审计的分数。

2. **自定义离线评估** — 你的产品的形状：
   - LLM作为评判者（Langfuse、Phoenix、Opik——第24课）。
   - 基于执行（运行补丁，检查测试）。
   - 基于轨迹（将动作序列与黄金标准进行比较；OSWorld-Human显示顶级Agent是黄金标准的1.4-2.7倍）。

3. **在线评估** — 生产环境：
   - 会话回放（Langfuse）。
   - 护栏触发的告警（第16课、第21课）。
   - 每步成本/延迟跟踪（第23课OTel跨度）。

### 评估器-优化器（Anthropic）

紧密循环：

1. 提议者生成输出。
2. 评估者评判。
3. 优化直到评估者通过。

这是自优化（第05课）的泛化。你关心的任何Agent流程都可以用评估器-优化器包装以确保可靠性。

### 2026年最佳实践

- 评估与代码共存。
- 在每次PR的CI中运行。
- 基于评估分数进行合并门控（例如"与主分支相比回归不超过5%"）。
- 每个护栏映射到一个评估案例。
- 每个学习规则（Reflexion、pro-workflow learn-rule）映射到一个失败案例。

### 评估驱动开发失败的地方

- **没有基线。** 没有最后已知良好状态的评估是不可读的。存储基线。
- **LLM作为评判者没有基于事实。** 评判者也会产生幻觉。CRITIC模式（第05课）——评判者基于外部工具进行验证。
- **对评估过度拟合。** 为评估优化会偏离生产实用性。轮换案例。
- **不稳定的评估。** 非确定性案例导致误报。固定种子、快照状态。

1. 取你的一个生产失败。编写一个能够复现它的评估案例。你的Agent现在能通过吗？

## 📝 教师备课总结与读后感

**文档评价：** 这是一份极其精炼的"评估驱动Agent开发"教案，短短篇幅内将Anthropic的核心方法论（简单开始→评估优化→逐步增加复杂性）浓缩为可操作的三层评估框架。文档结构清晰——问题→概念→构建→使用→练习，符合"学以致用"的教学循环。最精彩的部分是"连接阶段14"的对照表，将评估从理论拉到实践，让学生看到每一堂课都在为某个具体的评估案例服务。弱点是缺少真实的失败案例故事（比如某公司跳过评估导致的生产事故），这在教学中可以用来打破"评估不重要"的错觉。

**知识结构：** 三层评估（静态基准→自定义离线→在线生产）构成了从模型选型到生产监控的完整评估链路。评估器-优化器是将评估嵌入Agent运行时的核心模式。CI门控将评估从"偶尔跑一下"升级为"每次提交都必须过"的工程纪律。最后，评估驱动开发的反模式（无基线、评判者无事实依据、过度拟合、不稳定性）是实际落地的关键陷阱。

**核心洞察（5-7条）：**
1. 评估不是Agent开发后的验证步骤，而是决定Agent设计的外层循环——这是范式翻转：评估先于设计。
2. SWE-bench+发现的32.67%解决方案泄露揭示了一个残酷事实：公开基准测试已经被模型训练数据污染，盲目引用分数可能严重高估能力。
3. LLM作为评判者本身也需要"事实依据"——CRITIC模式表明评判者必须像Agent一样接入外部工具进行验证，否则评判者本身成为幻觉源。
4. "不能区分失败来自模型能力还是系统设计"的问题，通过三层评估解决了：每层隔离不同的故障源。
5. 评估案例必须是"活的"——每个生产失败都应该立即转化为一个新的评估案例，这是评估套件的"自愈"机制。
6. 轨迹效率指标（Agent步数/人类专家步数）是一个被低估的评估维度——它不仅衡量是否成功，还衡量Agent的工作方式是否经济。
7. 评估的不稳定性（非确定性导致的误报）是CI门控的最大敌人——没有人会认真对待一个动不动就"假阳性"的评估系统。

**教学建议（5-7条）：**
1. 开门见山讲"你们见过多少次Agent在demo里完美、上线后崩溃？"作为引入，引发共鸣。
2. 先展示"没有评估"的Agent上线后果（使用真实案例或编一个有冲击力的故事），让学生明白为什么评估是"第一公民"。
3. 三层评估的讲解顺序应该倒过来：先讲在线评估（学生最熟悉的生产监控场景），再回溯到离线评估和基准测试——从应用场景推导理论需求。
4. "连接阶段14"的对照表应作为互动练习：遮住右列，让学生自己为每课设计评估案例，然后揭晓答案进行对比。
5. 评估器-优化器的概念应与学生已知的自优化模式（第05课）建立连接，强调"这是同一个模式在不同层次的应用"。
6. CI门控部分需要给出具体的阈值建议（比如"先设10%回归阈值，稳定后收紧到5%"），避免学生设得太严导致构建一直失败而放弃。
7. 布置学生"回到自己的Agent项目，找出最近一次生产失败，写出能复现它的评估案例"——这是从学到用的关键桥梁。

**值得补充（3-5条）：**
1. 补充一个完整的"评估驱动开发"完整案例——从发现生产Bug→编写评估案例→修复→CI集成→成功防止回归的全流程，让学生看到闭环。
2. 增加"评估经济学"的内容——什么时候投入评估资源是划算的？每个评估案例的成本是多少？ROI如何计算？
3. 补充评估案例的"腐烂"问题——随着产品演进，旧的评估案例可能变得过时或不再相关，需要定期审查和清理。
4. 增加多模型比较的场景——当你的Agent可以切换底层模型时，如何设计跨模型的评估策略？
5. 补充"评估反作弊"——当工程师开始"为评估优化"而非"为产品优化"时，如何检测和纠正这种偏离？

**一句话总结：** 评估不是Agent的影子，而是Agent的骨架——没有评估的Agent系统就像没有类型检查的代码库，一切看似运行正常，直到某天在不可预知的地方塌陷。

# 🎓 Agent 架构课：评估——你不是在验证Agent，你是在决定它长什么样

开门见山，我问你们一个问题：**你的Agent上一次在生产环境中出问题的时候，你是先改了Agent代码，还是先补了一个评估案例？如果答案是前者，你的评估系统没有在正常工作——或者说，根本没有评估系统在工作。**

我见过太多团队把评估当成了"QA的事"，放在上线前最后一关跑一遍。这种做法的问题在哪？想象你装修房子，施工队干完了三个月，最后一天请监理来看。监理说"柱子歪了5厘米"。你现在能怎么办？拆了重装？你只能在柱子外面包一层装饰板假装看不见。评估放在最后就是装修结束才找监理——你什么都改不了。

## 评估的真正位置：它是Agent设计的"外层循环"

我在生产系统中设计的评估体系，从来不是Agent的一个功能特性。它是包裹在Agent外面的一个循环。Agent生产输出 → 评估者打分 → 不是"好/坏"的判断，而是"这个Agent应该是什么样子"的设计反馈。

Anthropic把话说得很直白："start with simple prompts, optimize them with comprehensive evaluation, and add multi-step agentic systems only when needed." 注意这个顺序：评估在Agent系统之前。你不需要一个复杂的多Agent编排系统，如果你的简单提示已经能通过评估。这就是评估的威力——它不是让你知道"现在的Agent好不好"，而是让你知道"你该怎么设计Agent"。

## 两条路径：被动评估 vs 设计驱动评估

我把评估分成两条路径来理解，而不是三层。

**路径一：被动评估。** 基准测试跑一跑、离线评一评、上线后看看监控。这是"检验已经发生的事情"。99%的团队停留在这里。他们看SWE-bench分数选模型，看Langfuse的评分看质量——这本质上是用后视镜开车。

**路径二：设计驱动的评估。** 在写第一行Agent代码之前，你先定义"什么是成功"。不是"Agent完成了任务"，而是"Agent以什么方式完成了任务，花了多少步，做了什么决策，哪些决策绝对不能出现"。然后你的Agent是被这些评估案例反向塑形的。

我做一个具体对比。一个做代码审查Agent的团队，如果用被动评估，他们会说"模型改对了多少bug"——这是静态基准测试。如果用设计驱动评估，他们的评估案例包括："Agent是否在没有充分阅读上下文的情况下就提交了修改？""Agent是否跳过了必要的测试运行？""Agent是否改动了不在任务范围内的文件？"——这些评估案例决定了Agent的行为约束，而不仅仅是结果质量。

## 为什么CI门控是"评估"的最高形态

我说一句可能得罪人的话：**不在CI里跑的评估，等于没写。**

评估只在你自己机器上跑过，和你从来没写过有什么区别？它必须活在你的合并流程里。每次PR，评估套件自动触发，回归超过阈值就阻断合并。这不是"严格"，这是让评估获得"执行权"。

我在一个项目中设定的阈值是这样的：第一版设15%回归阈值——先让团队适应"合并被评估卡住"这件事。两个月后收紧到5%。再过一个月到2%。现在那个项目的评估套件有超过200个案例，每次合并都跑，每次失败都有日志，每次超阈值都有人需要写解释。

关键不是什么数字，关键是"可执行"。评估必须有权说不。

## 真正的反模式：LLM评判者没有事实依据

这里有一个隐藏的大坑。你们很多人都在用"LLM-as-judge"——用另一个LLM来评判Agent的输出。但你们有没有想过：**如果评判者LLM自己产生幻觉呢？**

举个例子：Agent回复说"根据2025年的法规，你需要做X"。评判者LLM也可能"认为"这个法规是对的，于是给高分。但事实上法规是2026年才生效的，Agent和评判者都在同一片幻觉的海洋里漂着。

这就是为什么CRITIC模式不只适用于Agent生成，同样适用于评估。评判者必须有外部工具来验证——搜索法规原文、查询数据库、运行实际代码——而不是凭空判断。一个没有事实依据的LLM评判者，就像一个从不看源码的代码审查者——他可能会点头说"看起来不错"，而实际上完全不对。

## 评估套件的"腐烂"问题

另一个少有人谈的问题。你的评估案例是活的还是死的？随时间推移，产品在变，模型的底层能力在变，曾经重要的评估案例可能变得无关紧要，曾经无关紧要的边界情况可能成为主要故障源。

我在生产系统中做的一件事是：每个评估案例都有一个"最后触发日期"和"触发频率"。如果一个案例超过3个月没有被触发过（即Agent从未在这条案例上失败），我会标记它：要么这个案例已经过时了，要么Agent在这个维度上已经完美——无论哪种，都需要重新审视。此外，每出现一个生产Bug，必须转化为一个新的评估案例——这就像是代码测试中的"回归测试"原则。

## 结语清单

在你离开这堂课之前，检查你的Agent项目：
1. 你是否有三层评估（基准测试、离线、在线）？如果只有基准测试，你只有1/3的覆盖。
2. 你的评估是否跑在CI里，并能阻断合并？
3. 每个评估案例是否与一个具体的生产失败或护栏规则对应？
4. 如果你在用LLM评判者，它是否有外部工具的事实依据？
5. 你的评估套件是否在"腐烂"？最近一次审查评估案例是什么时候？

**金句：评估不是你的Agent的质量检查点，它是你的Agent的设计语言。你在评估里写什么，你的Agent最终就会变成什么。所以，不要在评估里写你不想让Agent变成的样子。**
