# 评估驱动的 Agent 开发

> Anthropic 的指导原则："从简单的提示词起步，用全面的评估优化它们，仅在需要时才添加多步 Agent 系统。"评估不是最后一步。它是驱动 Phase 14 中每一个其他选择的外部循环。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 全部内容。
**用时：** 约 60 分钟

## 学习目标

- 说出三个评估层——静态基准、自定义离线、在线生产——以及每层的用途。
- 解释评估器-优化器紧循环。
- 描述 2026 年最佳实践：评估与代码并存、在 CI 中运行、门控 PR。
- 将 Phase 14 的每一课连接到它生成的评估用例。

## 问题

Agent 在 demo 中通过。它们以 demo 无法预测的方式在生产中失败。基准回答的是"这个模型普遍有能力吗？"而非"这个 Agent 在为我的产品提交正确的补丁吗？"答案：三层评估，持续运行，每个护栏和学到的规则都映射到一个评估用例。

## 概念

### 三个评估层

1. **静态基准**——SWE-bench Verified 用于代码（第 19 课）、WebArena/OSWorld 用于浏览/桌面（第 20 课）、GAIA 用于通才（第 19 课）、BFCL V4 用于工具使用（第 06 课）。用于跨模型比较和回归门控。污染是真实的：SWE-bench+ 发现 32.67% 的解决方案泄露。始终报告 Verified / + 审计分数。

2. **自定义离线评估**——你的产品形态：
 - LLM 作为评判者（Langfuse、Phoenix、Opik——第 24 课）。
 - 基于执行的（运行补丁，检查测试）。
 - 基于轨迹的（将动作序列与金标准比较；OSWorld-Human 显示顶级 Agent 是金标准的 1.4-2.7 倍）。

3. **在线评估**——生产：
 - 会话回放（Langfuse）。
 - 护栏触发的告警（第 16、21 课）。
 - 每步成本/延迟追踪（第 23 课 OTel span）。

### 评估器-优化器（Anthropic）

紧循环：

1. 提案者生成输出。
2. 评估者判断。
3. 精炼直到评估者通过。

这是 Self-Refine（第 05 课）的泛化。任何你关心的 Agent 流程都可以包裹在评估器-优化器中以获得可靠性。

### 2026 年最佳实践

- 评估与代码并存。
- 在 CI 中对每个 PR 运行。
- 基于评估分数门控合并（例如"相对于 main 无超过 5% 的回归"）。
- 每个护栏映射到一个评估用例。
- 每个学到的规则（Reflexion、pro-workflow learn-rule）映射到一个失败用例。

### 评估驱动开发在何处失败

- **无基线。** 没有 last-known-good 的评估是不可读的。存储基线。
- **LLM 评判者无真值锚定。** 评判者也会幻觉。CRITIC 模式（第 05 课）——评判者以外部工具为锚定。
- **对评估过拟合。** 为评估优化会偏离生产有效性。轮换用例。
- **不稳定评估。** 非确定性用例导致误报。锁定种子，快照状态。

## 构建

- 通过 CI 对每个 PR 运行。
- 在回归上使构建失败。
- 随时间追踪通过率。
- 将每个生产失败绑定到新用例。

## 练习

1. 拿一个你的生产失败。写一个复现它的评估用例。你的 Agent 现在能通过吗？
2. 为你的领域构建一个三维 LLM 评判者评分量表（事实性、语气、范围）。对 50 个会话评分。
3. 将评估套件接入 CI。在 ≥5% 回归时使构建失败。
4. 添加轨迹效率指标：Agent 采用了多少步相对于金标准轨迹？
5. 将 Phase 14 的每课映射到你的套件中的一个评估用例。有缺失的吗？那就是需要填补的缺口。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是整个 Phase 14/15 课程的**收官课**——不是教新概念，是把前面所有课串成一幅"怎么持续知道你的 Agent 有没有变差"的全景图。文档做了两件极少有课程敢做的事：① 给 Phase 14 的每一课对应一个具体的评估用例（那张表是本节课的核心交付物），② 把评估从"做完再测"重新定位为"驱动所有其他选择的外部循环"。适合已完成 Phase 14 全部课程、正在建立 Agent 质量保障体系的架构师。

### 二、知识结构梳理

**评估架构层：** 三层评估——静态基准（跨模型比较）、自定义离线（产品形态特定）、在线生产（真实用户反馈）——覆盖了 Agent 评估的完整生命周期。每层有不同的角色：基准做回归门控、离线做质量验证、在线做持续监控。

**工程实践层：** 四个 2026 最佳实践——评估与代码并存、CI 中运行、PR 门控、护栏映射用例——将评估从"测试团队的事"变成"每个工程师每次提交的事"。CI 门控："≥5% 回归 → 构建失败"——这不是测试，是工程纪律。

**课程串联层：** 那张 Phase 14 全课→评估用例的映射表是本节课的独特价值。它不是"这些课有这些概念"——是"每节课教你的东西，都必须有一个评估用例来验证它是否还 work。"

### 三、核心洞察（备课时的关键理解）

1. **"评估不是最后一步——是外部循环"是整个 Phase 14 的设计哲学。** 传统开发：实现 → 测试 → 上线。评估驱动开发：每个设计决策对应一个评估用例 → 实现 → 跑评估 → 不通过就改。评估在实现之前。这就是"evaluator-optimizer 紧循环"在课程设计层面的映射。

2. **三层评估对应三种受众。** 静态基准 → 研究者和模型选型者。自定义离线 → Agent 开发者。在线生产 → 运维和产品。如果只有一层——你缺了两个视角。

3. **"与代码并存"不是建议——是工程要求。** 评估用例不在另一个仓库、不在测试团队的文档里、不在"以后再说"的 TODO 里——在 Agent 代码的同一个仓库里，随每次 PR 一起跑。这是可维护性的最低要求。

4. **Phase 14 全课→评估用例的映射表是"覆盖范围检查清单"。** 如果你的评估套件里缺少某课的对应用例——那一课教的东西在你的系统里就是"没被测试的"。Agent 循环没测试 → 无限循环会漏。护栏没测试 → 恶意输入会漏。编排没测试 → 路由错误会漏。这张表的用途是**找出你的盲区。**

5. **SWE-bench+ 发现的 32.67% 解决方案泄露是一个足够震动的数字。** 基准污染不是理论问题——在 SWE-bench 上，三分之一的"正确"答案是模型在训练数据里见过的。这就是为什么文档坚持"报告 Verified / + 审计分数"——原始的 SWE-bench 分数已经不可信。

6. **"LLM 评判者也会幻觉"是评估的最大自反性问题。** 你用 LLM 评判 LLM 的输出——如果评判者也幻觉呢？CRITIC 模式的答案：评判者以外部工具为锚定。不是"你觉得这段代码对吗？"——是"跑一下测试，测试通过了吗？"外部工具是真值——评判者只是解释结果。

7. **"轨迹效率"是 Agent 评估中被最低估的指标。** 大多数评估只看"任务完成了吗？"但一个 Agent 用 40 步完成的"成功"和一个 Agent 用 10 步完成的"成功"是不同的质量水平。轨迹效率 = Agent 步数 / 人类专家最小步数——这是 Agent 的"推理效率"的量化。

### 四、教学建议

1. **这节课必须放在最后。** 学生需要理解所有课的内容，才能理解"每课对应一个评估用例"的映射。把它放在 Phase 14 第一课——学生不知道什么是 Reflexion、什么是 LangGraph 恢复——映射表是空的占位符。

2. **用"你上次 Agent 出 bug 是什么时候？"开场。** 让学生回忆最近一次 Agent 生产事故——然后问"你有评估用例能复现它吗？"大多数答案是"没有"。在感受了"没评估的代价"之后再讲评估架构。

3. **三层评估用"买车类比"来讲。** 静态基准 = 汽车安全碰撞测试评级（跨车型比较）。自定义离线 = 你试驾这条路（你的日常通勤路线，你的需求）。在线生产 = 你实际开了一年的油耗和维修记录。三层缺一不可。

4. **那张 Phase 14 映射表是课堂上应该让学生逐行填空的。** 发一张空白表，让学生为每课写一个评估用例。然后对比文档的答案。差异就是他们对"这课教的东西怎么验证"的理解差异。

5. **"与代码并存"的实操——让学生当场建一个 `evals/` 目录。** 在 Agent 代码仓库里 `mkdir evals`，写两个用例，跑一下。这个动作从"概念"变成"习惯"就是在课堂上发生的。

6. **LLM 评判者的幻觉问题值得一个现场 demo。** 让 LLM 评判同一段代码两次——一次给"这段代码好吗？"的自由文本提示，一次给"运行测试并基于测试结果评判"的 CRITIC 提示。对比结果——自由文本评判者可能说"很好"，CRITIC 评判者可能说"3 个测试不过"。

7. **练习 1（复现生产失败）是这节课最"疼"的练习。** 让学生拿一个真实的生产失败，写一个评估用例来复现它——然后跑，看 Agent 现在能不能通过。能通过 → 这个用例是你的回归测试。不能通过 → 这个 bug 还在你的系统里。

### 五、值得补充的内容

1. **评估用例的维护成本。** 100 个用例，每个 PR 跑一次，每次跑 30 秒——总计 50 分钟的 CI 时间。随着用例增长，需要分层运行（快速冒烟 + 完整套件）。

2. **在线评估的采样策略。** 你不能对 100% 的生产流量做完整的轨迹评估——太贵了。采样率是多少？1%？基于风险分层采样（高风险用户 100%，低风险 1%）？

3. **评估用例的"保鲜期"。** Agent 的行为随模型更新、提示词调整而改变。三个月前的评估用例可能仍在测"已经修复的 bug"——需要定期审查用例的相关性。

4. **跨模型可比性。** 在 GPT-5 上通过率 95% 的用例，在 Claude Sonnet 上可能只有 80%。评估套件需要标明"在哪个模型上建立的基线"——换了模型，基线要重新做。

5. **评估与 A/B 测试的关系。** 在线评估是观察性的（"Agent 在生产中表现如何？"），A/B 测试是实验性的（"新提示词比旧提示词好吗？"）。两者互补但目的不同。

### 六、一句话总结

**评估不是 Agent 开发的最后一步——是每一步的证明。不是"我做完了，测一下"——是"我怎么知道我做对了？"每个设计决策对应一个评估用例，每个用例在 CI 中每次 PR 都跑——这是 Agent 工程的实证基础。**

# 🎓 Agent 架构课：评估驱动开发——你怎么知道你的 Agent 不是在变差？

同学们好。我是你们的 Agent 架构老师。

前面九节课，我教了你们怎么搭 Agent、怎么选框架、怎么监控失败。这些课有一个共同的隐藏假设——**你知道你的 Agent 做对了。**

但你怎么知道？

"我看了一眼输出，看起来还行。"——这不是工程，这是信仰。

今天这节课是 Phase 14 的收官课。它不是教新东西——是教你**怎么持续地、可量化地、不会自欺欺人地知道你前面九节课搭的东西还在正确地工作。**

## 基准能告诉你什么，不能告诉你什么

SWE-bench。GAIA。WebArena。BFCL。你听过这些名字。

它们告诉你一件事——**这个模型在你的任务类别上大概行不行。** 在 SWE-bench Verified 上得分 49% 的模型能解决大约一半的真实 GitHub issue。对选模型有用。

它们不告诉你的事：
- 你的 Agent 具体在哪些类型的任务上崩？——基准是聚合分数，没有细粒度分类。
- 你的 Agent 在生产中多快？（基准只测端到端成功率，不测延迟/成本/轨迹效率。）
- 你的 Agent 有没有在训练数据里见过这道题？（SWE-bench+ 的研究发现 **32.67% 的解决方案在训练数据里泄露了**——三分之一的"对"是"背出来的"。）

基准是第一层——必要但不充分。不能只靠它。

## 三层评估：你需要的不是一层，是三层

### 第一层：静态基准——跨模型比较

SWE-bench Verified、GAIA、BFCL V4。这些是你在选模型时看的。你换了一个模型——跑一遍基准，看分数有没有掉。掉了 ≥10% → 不能换。

作用：**回归门控。**

### 第二层：自定义离线——你的产品形态

这是你真正需要的东西。它不是"SWE-bench 上表现怎么样"——是"你的 Agent 在处理你的用户实际会问的问题时表现怎么样"。

三个子类：

- **LLM 作为评判者。** 用另一个 LLM 读 Agent 的输出，在三个维度上评分——事实性（说的对不对？）、语气（礼貌吗？专业吗？）、范围（有没有做超了？）。但要小心——评判者 LLM 也会幻觉。CRITIC 模式（第 05 课）给答案：让评判者以外部工具为锚定。不是"你觉得这段代码对吗？"——是"跑一下测试，测试通过了吗？"

- **基于执行的。** 拿 Agent 的输出真的跑一下。Agent 说它创建了一个文件 → 检查文件是否存在，内容是否正确。Agent 说它修好了一个 bug → 跑测试套件。执行是最终的真值。

- **基于轨迹的。** Agent 的步骤序列和人类专家的金标准步骤对比。Agent 用了 40 步，人类只用 10 步——Agent 的效率是 25%。你的 Agent 可能"完成了任务"，但效率很差——轨迹评估暴露这一点。

### 第三层：在线生产——真实用户反馈

Agent 上线了。你怎么知道它现在表现如何？

- **会话回放。** Langfuse 录制用户会话，你可以回放看 Agent 的完整对话。不是采样——是完整的轨迹。
- **护栏触发的告警。** 输入护栏被触发了——意味着有人尝试了恶意输入。输出护栏被触发了——意味着 Agent 差点说出了 PII。这些是实时的失败信号。
- **成本/延迟追踪。** OTel span（第 23 课）让你看到每个步骤的成本和延迟。Agent 突然变慢了 50%——可能是某个工具 API 变慢了，可能是模型换了。

三层没得商量。基准告诉你选哪个模型。离线告诉你 Agent 具体哪里不行。在线告诉你用户实际在经历什么。

## 评估驱动开发：不是在代码之后——在代码之前

Anthropic 说了一句在整个行业被反复引用的话："从简单的提示词起步，用全面的评估优化它们，仅在需要时才添加多步 Agent 系统。"

注意顺序：**先有评估，再有优化。**

传统 Agent 开发：写代码 → 上线 → 出 bug → 修。

评估驱动开发：先写评估用例（"这个 Agent 应该对退款的用户说..."）→ 写 Agent → 跑评估 → 评估不通过 → 修 → 再跑。

好处：**你不需要上线才能知道 Agent 不行。** 评估在 CI 里跑——每次 PR，每个提交。你的 Agent 输出的质量变化在你合并代码之前就暴露了。

## 那张表：Phase 14 每一课对应一个评估用例

这是本节课最核心的交付物。我把它翻译成一句能记的话：

**你从 Phase 14 学的每一课——都必须在你的评估套件里有一个"证明它还在工作"的用例。**

Agent 循环 → 用例："Agent 不会在空输入上陷入无限循环。"
ReWOO → 用例："工具返回错误时，规划器重新规划而不是放弃。"
Reflexion → 用例："Agent 在上次犯错的任务上，这次用上了学到的反思。"
LangGraph → 用例："崩溃恢复后，状态精确复现，Agent 从断点继续。"
OpenAI Agents SDK → 用例："输入护栏在包含 PII 的消息上触发。"
失败模式 → 用例："级联错误在第一步就被检测到并截断。"

缺了任何一行的用例——那一课教的东西在你的系统里就是 **"没被测过的。"** 它是薛定谔的可靠性——你可能觉得它还在 work，但你没有证据。

你的评估套件如果有 20 个用例覆盖了这 20 行——你就覆盖了 Phase 14。你现在不是"我觉得我的 Agent 还可以"——你是"我有证据"。

## 什么时候评估驱动开发会不可靠

**没有基线就不知道有没有变差。** 这个月通过率是 85%。是好了还是坏了？你不知道——你没有上个月的通过率。基线 = 一个快照。每次改完 Agent，跑一遍评估，存下分数。下次跑的时候对比——你是上升了还是下降了。

**LLM 评判者没有真值锚定等于两个幻觉在对话。** 你的 Agent 输出了错误的事实。你的评判者 LLM 也说"看起来合理"。你花了两次 LLM 调用的钱获得了一个没被发现的错误。解决：评判者必须基于可验证的真值——测试结果、数据库查询、API 返回值。

**过拟合评估是训练数据泄露的 Agent 版本。** 你的评估套件有 50 个用例。你为这些用例优化了 Agent——在这些用例上它 100% 通过。然后上线——真实用户的问题不在你那 50 个用例里——Agent 崩了。解决：定期轮换用例，加入新的生产失败作为新用例。

**不稳定的评估会让你的 CI 变红一天。** 用例 7 有时候通过有时候不通过——因为它的输出有随机性。CI 红了——你排查——发现不是你的代码的问题，是评估本身的问题。这种误报消耗的是团队对评估的信任。解决：锁定随机种子、固定状态快照、标记已知的不稳定用例。

## 结课清单

建立你的 Agent 评估体系：

1. 三层：静态基准（跨模型比较）+ 自定义离线（你的产品形态）+ 在线生产（真实用户）。
2. 评估与 Agent 代码在同一个仓库——`evals/` 目录。
3. CI 中每次 PR 跑评估——≥5% 回归 → 构建失败。
4. 每个护栏、每个学到的规则、每个失败模式 → 至少一个评估用例。
5. Phase 14 全课映射表——逐行检查，盲区就是你没在测的东西。
6. 基线存储——每次跑完存分数，下次才能知道有没有变差。
7. LLM 评判者以外部工具为锚定——不是"你觉得对吗"，是"测试通过了吗"。
8. 每个生产失败 → 一个新的评估用例。你的评估套件是你的 bug 收藏馆。

**最后一句话——这是我整个 Phase 14/15 模块最想留给你的：**

**Agent 工程的实证基础不是"我觉得输出还不错"——是"这个设计决策对应了哪个评估用例，这个评估用例上次跑是什么分数，这个分数和上个月相比是升了还是降了。"把"我以为"从你的 Agent 架构词汇表里删掉，用"我有证据"替换它。**
