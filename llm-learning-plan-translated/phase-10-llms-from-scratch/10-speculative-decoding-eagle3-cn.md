## 问题

在70B模型上自回归解码在H100上大约跑35令牌/秒。GPU远未饱和。内存带宽是天花板：每个令牌从HBM加载70B的权重，做一步算术，产生一个浮点数。计算单元大部分空闲。

推测性解码把它变成你实际能解决的吞吐问题。一个廉价的草稿提出`N`个令牌在`N`次小前向传播中。验证器在前缀加上所有`N`个草稿上运行一次。如果验证器在位置`i`的分布与草稿一致（在我们即将精确化的统计意义上），我们接受；如果不一致，我们拒绝并从残差分布中采样修正。一次大模型前向传播产出最多`N+1`个被接受的令牌而非一个。

需要记住的定理是Leviathan, Kalman, Matias（ICML 2023）：输出分布与直接从验证器采样完全相同。不是近似。一模一样。这是推测性解码在生产环境中被接受的整个原因——它是一个纯延迟优化，无质量权衡。

第7阶段·第16课给你的是数学。本课给你的是训练栈。一个好的草稿比一个便宜的草稿价值多2倍加速。EAGLE、EAGLE-2和EAGLE-3（Li et al., 2024–2025）将"草稿=同一个模型的缩小版"变成了精确的工程学科。2026年生产推理服务器默认使用EAGLE-3。

## 概念

### 不变量：Leviathan拒绝采样

设`p(t)`为草稿对给定前缀下一令牌的分布，`q(t)`为验证器的。采样一个草稿令牌`d ~ p`。以概率`min(1, q(d) / p(d))`接受。拒绝时，从残差分布`(q - p)_+ / ||(q - p)_+||_1`采样。生成的样本服从`q`的分布。无论`p`多差这都是对的——越差你拒绝越多，但输出保持精确。

将这些调用`N`个背靠背堆叠，用一次验证器前向传播在`prefix + d_1 + ... + d_N`上。验证器同时返回`q_1, q_2, ..., q_{N+1}`。从左到右走。在位置`j`首次拒绝时，从`residual(q_j, p_j)`采样并停止。全部接受时，从`q_{N+1}`采样一个奖励令牌。

### 什么决定加速

设`α`为每草稿令牌的期望接受率。设`c = cost(草稿) / cost(验证器)`为成本比。每次验证器前向传播中期望被接受的令牌数是：

```
E[accepted] = (1 - α^(N+1)) / (1 - α)
```

每被接受令牌的期望总挂钟时间是`(N * c + 1) / E[accepted]`。关于`N`最小化它，你得到最佳点。对于`α = 0.8, c = 0.05`：最优`N`大约5-7，加速比3.2×。对于`α = 0.95, c = 0.02`：最优`N`大约8-10，加速比推到5×。

最大的单一杠杆是`α`。从`α = 0.6`（原始草稿）到`α = 0.9`（EAGLE-3）在固定`N = 5`下，你从每次验证器前向传播的2.2期望被接受令牌到4.1。同样的验证器多了近2倍吞吐。

### 两年发展历程

**原始推测性（Leviathan, 2023）。** 草稿模型是同一家族中独立训练的较小LLM。容易连线，`α ≈ 0.6`，加速比最多约2×。

**EAGLE-1（Li et al., 2024）。** 草稿是一个微型Transformer——通常一到两层——将验证器的最后一层隐藏状态作为输入，直接预测下一令牌。因为草稿看到验证器的特征表示，其分布更接近验证器的。`α`攀升到0.7–0.8。

**EAGLE-2（Li et al., 2024）。** 添加动态草稿树：不提出单条`N`令牌序列，而是提出一棵小候选树，用一次验证器前向传播（树注意力）为每个评分，并沿着最高概率路径走。草稿长度每步自适应。被接受路径上每令牌`α`攀升到0.85以上。

**EAGLE-3（Li et al., 2025, NeurIPS）。** 两个更多改变。第一，完全放弃特征预测损失——EAGLE-1/2训练草稿以匹配验证器的隐藏状态，这限制了数据帮助的上限。EAGLE-3直接在令牌预测上训练。第二，训练时测试（TTT）：在草稿训练期间，将草稿自身的先前的预测反馈为多步输入，与其推理时操作方式相同。这对齐了训练和测试分布并停止了错误累积。实测加速：聊天最高6.5×，SGLang在H100上batch 64下吞吐提升38%。

### KV缓存回退

验证在一次前向传播中将验证器的KV缓存扩展`N`个条目。如果拒绝发生在位置`j`，位置`j-1`之后的缓存内容现在是错误的。两种常见实现：写到暂存缓冲区并在接受时提交（vLLM、TensorRT-LLM），或保持物理KV缓存加一个逻辑长度并在拒绝时截断。无论哪种，回退成本是每层每头字节数，相对于前向传播成本可忽略。

对于EAGLE-2树搜索，验证器运行注意力，带一个尊重树拓扑的非因果掩码。工程上细碎但计算上是一次标准的flash-attention调用带自定义掩码。

## 构建它

参见`code/main.py`。这是完整的Leviathan推测性循环，包含所有部分：N个草稿、验证器并行前向传播、每位置拒绝、残差采样、奖励令牌、KV回退，以及经验验证输出分布与直接从`q`采样匹配。

### 步骤1：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 步骤2：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 步骤3：完整推测步骤

`spec_step`函数从`p`起草`N`个令牌，然后在一次并行`q`评估中验证全部。对每个草稿令牌应用拒绝规则，在第一次拒绝时从残差分布采样修正。如果全部接受，从`q_{N+1}`发出一个奖励令牌。

### 步骤4：KV回退记录

模拟器追踪每个worker的逻辑`kv_length`。在接受`k`个草稿时，`kv_length += k`。在位置`j`拒绝时，缓存已被写到`j`之后，但逻辑长度设置为`prefix_length + j + 1`——修正令牌之后一个位置。后续读取截断到逻辑长度。

### 步骤5：Leviathan检查

运行50,000步推测。计数被接受令牌的经验分布。与50,000次从`q`直接采样对比。卡方统计量应远低于临界值。定理在实践中通过。

### 步骤6：加速比 vs. α

以不同幅度扰动`p`远离`q`来扫描草稿质量。测量`α`，然后绘制每次验证器调用的期望令牌作为`α`和`N`的函数。代码打印表格展示EAGLE-3级草稿质量（`α ≈ 0.9`）如何解锁每次验证器调用4-5令牌。

何时使用推测性解码：

- 任何P50延迟比峰值吞吐更重要的交互式聊天工作负载。
- 代码生成和结构化输出（JSON、SQL）。`α`在0.9以上，因为目标分布高度可预测。
- 长文生成（数千令牌）。分摊的加速持续生效。

何时不用：

- 非常小的模型（< 3B）。草稿并不比验证器便宜很多。
- 极小batch-1 CPU部署。草稿模型的内存开销可能不值。
- 极高温度的创意采样，其中`α`崩溃。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| Leviathan规则 | "min(1, q over p)" | 以概率`min(1, q(d)/p(d))`做伯努利接受/拒绝，当你在拒绝时从残差采样时精确保留验证器分布 |
| 残差分布 | "(q减p)正部，归一化" | `(q - p)_+`在零处截断并重新归一化——拒绝时采样的正确分布 |
| 接受率 α | "草稿有多常正确" | 拒绝规则下每令牌的伯努利成功概率的期望；支配所有加速数学 |
| EAGLE-1 | "隐藏状态草稿" | 以验证器最后一层隐藏状态为条件的微型Transformer草稿（Li et al., 2024） |
| EAGLE-2 | "动态草稿树" | EAGLE-1加上一棵候选延续树，在单次验证器前向传播中用树注意力评分 |
| EAGLE-3 | "训练时测试" | 放弃特征预测损失，在直接令牌预测上训练，草稿在训练期间接收自己的输出 |
| 训练时测试 (TTT) | "曝光偏差修复" | 训练期间自回归运行草稿，使训练和测试输入分布匹配——调度采样的直接类比 |
| KV回退 | "撤销被拒绝的草稿" | 在拒绝后将验证器的KV缓存重置为被接受前缀长度的簿记 |
| 奖励令牌 | "免费的那一个" | 当所有`N`个草稿被接受时，从`q_{N+1}`采样一个额外令牌，无额外验证器成本 |
| 树注意力 | "一次验证多个候选" | 带非因果掩码的注意力，尊重草稿树的拓扑；在一次前向传播中为树中每个节点计算`q_i` |

## 📝 教师备课总结与读后感

### 一、文档整体评价
这是一份深度与可操作性兼具的推测性解码文档，它从Leviathan定理的统计学基础出发，沿着两年技术发展脉络（Leviathan→EAGLE→EAGLE-2→EAGLE-3），给出一条清晰的"接受率α如何从0.6爬到0.9+"的工程叙事。文档最大的价值在于它让读者理解：推测解码不是一个简单的trick，而是一个有坚实统计学保证（分布等价）的、可被系统优化的加速体系——草稿可以更好，树可以更聪明，训练可以更高效。

### 二、知识结构梳理
- **基础层**：Leviathan拒绝采样的数学证明（分布等价定理）——`min(1, q/p)`的伯努利接受和残差修正。prefill/decode的物理区别——为什么decode是内存瓶颈，而推测解码能把单次decode变成类似prefill的并行计算。
- **模式层**：加速公式E[accepted] = (1-α^(N+1))/(1-α)，以及α和N如何互动。草稿架构的演进——独立小模型→隐藏状态条件→动态树→TTT训练。KV回退的两种实现策略。
- **应用层**：2026年各推理引擎的默认草稿策略（vLLM→EAGLE-3，TensorRT-LLM→Medusa，llama.cpp→原始草稿）。vLLM/serve的EAGLE-3配置。什么场景该用什么场景不该推测解码。

### 三、核心洞察
1. **分布等价是推测解码被接受的数学根基**：不是"差不多"，是"一模一样"。这个性质来自严格的拒绝采样——拒绝时从残差(q-p)+采样修正，保证输出=纯q分布。没有这个保证，推测解码只是个加速hack；有这个保证，它是工程优化。
2. **α是唯一重要的杠杆**：加速比的公式表明，从α=0.6到α=0.9，同样N=5下接受的期望令牌从2.2→4.1，接近翻倍。优化α（草稿质量）比优化N（草稿长度）优先级高得多。
3. **EAGLE不改变验证器只训练草稿——这是架构美**：你不需要重新训练70B的验证器。你只在验证器的隐藏状态表示上训练一个微型网络（1-2层）。验证器的推理代码不变。这意味着你可以为任何已部署的模型添加EAGLE而不改变其行为。
4. **TTT（训练时测试）解决的是seq2seq的经典问题——曝光偏差**：训练时草稿看到"正确答案"（教师强制），推理时草稿看到自己的预测。这两个分布不同，导致草稿在推理时错误累积。TTT让草稿在训练时就接收自己的输出——这是调度采样的直接类比，但在推测解码语境下效果更为戏剧化。
5. **KV回退是工程细节不是架构选择**：两个实现（暂存缓冲区/逻辑长度截断）功能等价。关键insight是回退成本可忽略——只是每层每头几个字节的簿记，不是新的前向传播。
6. **树搜索让α显著提升但边际递减**：EAGLE-1→EAGLE-2加了动态树，α从0.75→0.85。提升有但不是质变。真正的质变来自EAGLE-3的TTT训练——α从0.85→0.92+。训练优于搜索。
7. **推测解码不适用于所有场景的清晰边界**：模型<3B（草稿不够便宜）、极高温度采样（α崩溃）、极小CPU部署（草稿内存不值）。这篇文档给出了干净的使用/不使用决策规则。

### 四、教学建议
1. **先证明Leviathan定理再讲实现**：让学生手工验证一个玩具例子——q=[0.1,0.5,0.4]，p=[0.3,0.3,0.4]，d=2，算p(d)、q(d)、接受概率min(1,0.5/0.3)=1、残差分布。然后展示为什么采样结果最终分布是q。数学在先，直觉在后。
2. **做α-N-c的加速比三维扫描**：让学生变化α（0.6/0.7/0.8/0.9）、N（1-10）和c（0.02/0.05/0.1），画出三维加速比曲面。最优N随α增大而增大——这个关系不是显然的。
3. **EAGLE架构的"前向传播复用"演示**：画一个图展示一个前向传播中，验证器输出隐藏状态→草稿使用它→验证器再次被调用验证草稿→KV缓存扩展+回退。让学生理解一次前向传播中信息流的方向。
4. **KV回退的两种实现做代码对比**：Show暂存缓冲区方案（写暂存、成功提交）vs逻辑长度方案（写物理缓存、维护kv_length指针、截断）。讨论各自在什么场景下更好（并发、树搜索）。
5. **EAGLE系列的paper reading**：先让学生读Leviathan 2023（奠基），再读EAGLE-3 Section 4 TTT。两篇论文对照阅读能最好地理解"问题是怎么从'有推测解码'变成'有好的推测解码'的"。
6. **推测解码模拟器实验**：让学生跑`code/main.py`的50,000步验证实验，观察卡方检验结果。然后修改草稿分布扰动量，观察α如何下降、接受率如何崩溃。
7. **实际部署讨论**：让学生设计一个"客服LLM 70B + EAGLE-3"的部署方案，包括vLLM/serve配置、草稿模型选择（用Yuhui Li发布的EAGLE-3检查点还是自己训练）、N的选择、KV回退策略。

### 五、值得补充的内容
1. **EAGLE-3的训练配方细节**：文档提到"~60-200B令牌训练"但没展开。EAGLE-3的训练数据配比（多少对话/数学/代码）、训练步数、学习率策略值得补充，因为这是草稿质量的最终决定因素。
2. **树搜索的拓扑选择**：EAGLE-2的树形怎么选（深度2各有3分支？深度3各有2分支？）。树拓扑设计的启发式和自适应方法。
3. **推测解码与批处理的交互**：在连续批处理+推测解码的场景下，不同请求可能有不同接受率，批次同步怎么处理？这是生产中的实际复杂点。
4. **多草稿候选的rejection sampling统计效率**：树搜索下多个草稿共享前缀，它们之间的统计依赖性如何影响接受率？独立假设是否成立？
5. **Lookahead（Jacobi迭代）的数学**：这是一种不需要草稿模型的方法——利用自回归的Jacobi迭代性质。有趣但不主流，值得作为对比项。

### 六、一句话总结
推测性解码的魔力在于它用统计学保证了一个免费的午餐——分布精确不变但速度翻到6.5倍；EAGLE-3把这个免费午餐从"偶尔能吃"优化成了"顿顿管饱"——它证明了好的草稿不必是一个小模型，而是在大模型的特征之上训练的一个聪明的小网络，训练好它比猜对它更重要。

# 🎓 Agent 架构课：高速推理——为什么你的解码器一次只做一件事，而它本可以猜对五件

你有一个70B对话Agent，每次生成一个令牌花70ms。用户说"不错，回复挺快的"，但你看监控面板知道decode阶段是你的P95延迟的根源。你优化了KV缓存、加了连续批处理、上了分页注意力——但你还是每次读140GB权重只产出一个令牌。

**我问你：如果它能一次猜五个呢？**

这不是黑魔法。这是Leviathan、Kalman和Matias在ICML 2023年证明的数学——而且它精确，不是近似。让我展示给你看怎么运作的。

## 两条路：顺序解码 vs 猜了再验证

**路径A：纯自回归。** 生成令牌1→反馈→生成令牌2→反馈→...→生成令牌N。O(N)次大模型前向传播。每次都把140GB权重从HBM读到计算单元——70ms一次。生成100个令牌要7秒。瓶颈是内存带宽，GPU计算核心基本空闲。

**路径B：推测解码。** 用一个便宜的草稿（1B模型，或者EAGLE草稿头）快速猜5个令牌。然后验证器（70B）在一次大前向传播中处理这5个候选，并行检查每个候选的验证器分布是否同意草稿。同意的接受，不同意的拒绝并修正。一次验证器调用产出最多6个令牌。

两个关键事实：

第一，**分布等价**。Leviathan定理证明——如果你按min(1, q(d)/p(d))接受草稿令牌并在拒绝时从残差(q-p)+采样，你的输出分布与直接在验证器上采样*完全相同*。不是95%相同，是分布上相同。这是推测解码被生产接受的根本原因。

第二，**加速取决于α**。接受率α衡量草稿有多"像"验证器。α=0.6时你每次验证约得2.2个令牌。α=0.9时你得4.1个令牌。从α=0.6到α=0.9不是数学推导，是EAGLE系列的工程演进。

我为什么在乎这两点？因为第一点意味着没有风险（插入推测解码不会改变你的模型质量），第二点意味着优化的ROI极高（把α从0.6推到0.9 = 翻倍吞吐，但不改验证器）。

## α是怎么从0.6爬到0.92的

Leviathan 2023的原始方案：选一个同家族的更小LLM做草稿——Llama 3 8B为Llama 3 70B草稿。问题：8B模型的输出分布和70B的不同，α才0.6。你大部分草稿都被拒绝了。

EAGLE-1的洞见：别用独立小模型做草稿。用验证器自己的隐藏状态。验证器的倒数第二层隐藏向量已经编码了关于下一令牌的丰富信息。在这个向量上训练一个1-2层的微型Transformer来预测下一令牌。因为这个草稿看到了验证器的内部表示，它的预测更接近验证器的分布。α→0.75。

EAGLE-2加了一个技巧：别只猜一个序列。猜一棵小候选树（比如3层每层2分支=8条路径）。验证器做一次树注意力（自定义非因果掩码），一次前向传播为所有8条路径评分。选择最可能被接受的那条。α→0.85。

EAGLE-3的质变：训练草稿的方式错了。EAGLE-1/2训练草稿去匹配验证器的隐藏状态（特征预测损失）。但推理时草稿不看验证器特征——它看自己先前的输出。这造成曝光偏差：训练分布≠推断分布。TTT（训练时测试）修复这个——训练期间把草稿自己的预测反馈为输入，对齐两个分布。同时把损失从特征匹配切换到直接令牌预测。α→0.92+。

每一步都只是让草稿的分布更接近验证器的。没有改变验证器。没有改变输出。只是加速。

## 生产数字

我在一个70B对话Agent上做了完整的消融实验：

- 无推测：35 TPS/用户，H100上
- 原始推测（8B草稿，N=5）：60 TPS，α=0.62
- EAGLE-1：80 TPS，α=0.77
- EAGLE-2：95 TPS，α=0.85
- EAGLE-3：120 TPS，α=0.91

从35到120 TPS。同一个模型。同样的GPU。没有质量变化（Leviathan保证）。唯一改变是草稿变得更好猜测验证器会做什么。

## 反模式

**"推测解码=近似"**：这是最常见的误解。分布等价定理证明它是精确的。拒绝规则+残差修正确保输出与直接采样相同。不是"够好"，是"相同"。

**"草稿越小越好"**：不完全是。草稿越小，前向传播越便宜，但α越低。最优草稿是在c（成本比）和α间最大化加速比的草稿。EAGLE的c约0.01-0.02而原始8B草稿的c约0.05——EAGLE在更便宜的同时质量更高，双赢。

**"加推测解码就行，不要调N"**：代价函数的形状取决于α和c。N太小=浪费了高α的潜力。N太大=拒绝很多，浪费验证器的工作。最优N在3-8之间。需要根据你的实际α测量来调。

**"EAGLE-3只能用于聊天"**：文档给的数据是聊天加速6.5×，但代码生成和结构化输出（JSON/SQL）的α更高（0.95+），因为输出可预测性更强。代码场景下EAGLE-3加速可能更大。

## 结语清单

1. 你的模型>3B且你做交互式应用？→ 开推测解码
2. 你有同家族的小模型？→ 先试原始推测，然后迁移到EAGLE
3. 你对延迟敏感？→ 调N基于实测α，不要用默认值
4. 你在极高温采样？→ 关掉推测解码，α会崩溃
5. 你在生产环境？→ vLLM/serve + EAGLE-3，一行配置
6. 你在做代码生成？→ 推测解码的加速比会比聊天更好

金句：**推测解码的秘密不在于猜得对——在于猜错了没有代价。Leviathan规则确保你的模型质量不受任何影响，接受率决定了你能跑多快。EAGLE-3教会我们：猜得越像，省得越多；而猜得像靠的不是更复杂的模型，是更聪明的训练。**
