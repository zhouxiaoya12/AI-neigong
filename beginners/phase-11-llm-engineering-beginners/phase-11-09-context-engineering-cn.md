# 上下文工程：200K 窗口不等于全塞满

> **原文：** Context Engineering: Windows, Budgets, Memory & Retrieval  
> **预计时间：** ~30 分钟  

---

## 🚗 开场翻车：10 万 token 塞满的提示，不如精心挑选的 1 万 token

Claude Opus 有 200K token 窗口。GPT-5 有 400K。Gemini 3 Pro 有 2M。Llama 4 号称 10M。

**于是一个团队把什么都往提示里塞。** 50 个工具定义、20 轮对话历史、30 个检索文档、5 个少样本示例。填到 150K token。觉得"窗口大能装，全装进去模型自然能搞清楚"。

结果：模型对中间的文档完全忽略了。第 3 轮的关键信息被第 18 轮的废话淹没了。

**窗口大 ≠ 注意力均匀分布。模型对开头和结尾的东西关注最多。中间的被"遗忘"。这叫做"迷失在中间"（Lost in the Middle）效应。**

---

## 🧠 上下文窗口 = RAM，不是硬盘

```
┌─────────────────────────────────────────┐
│          上下文窗口 (200K)               │
│  系统提示 ~500                           │
│  工具定义 ~5K                            │
│  检索文档 ~4K                            │
│  对话历史 ~6K                            │
│  少样本 ~2K                              │
│  用户问题 ~200                           │
│  留给回答 ~4K                            │
│  ─────────────────────                   │
│  还剩 ~175K 空着 ← 不是浪费，是克制      │
└─────────────────────────────────────────┘
```

**每多塞一个 token，就挤掉一个可以携带更相关信息的 token。** 无关的工具定义、过时的对话轮次、不能回答问题的文档——每一样都在偷偷降低模型表现。

---

## ⚠️ "迷失在中间"

模型注意力分布：
```
████████░░░░░░░░░░░░░░░░████████
  开头高      中间低        结尾高
```

**放在中间的信息（40%-70% 位置）检索准确率下降 10-20%。**

**怎么办：**
1. 最重要的信息放开头或结尾
2. 不要塞满窗口。精心挑的 10K token > 硬塞的 100K token
3. 长对话历史：只保留最近的 N 轮 + 摘要早期轮次

## 🏆 焊死在脑子里的东西

1. **窗口不是垃圾桶。** 精心策划的 10K > 硬塞的 100K。
2. **重要信息放开头或结尾。** 中间的信息模型可能忽略。
3. **对话历史用滑动窗口。** 只保留最近 N 轮 + 早期内容摘要。

---

## 📝 练习答案

1. 工具定义：只保留跟用户当前问题相关的 10 个（1.5K）
2. 对话历史：最近 5 轮（2.5K）+ 早期摘要（1K）
3. 检索文档：最相关的 3 个（4K）
4. 剩下 11K = 留给模型回答的空间

---

> **上下文工程的精髓不是"能塞多少塞多少"，而是"敢扔掉多少"。扔掉的噪音越多，剩下的信号越好。**
