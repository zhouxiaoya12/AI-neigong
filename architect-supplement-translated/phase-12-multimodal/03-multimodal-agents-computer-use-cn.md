# 多模态Agent与计算机使用（总复习）

> 2026年的前沿产品是一个多模态Agent：读取截图、点击按钮、导航网页UI、填写表单、端到端完成工作流。SeeClick和CogAgent（2024年）证明了GUI定位基元。Ferret-UI增加了移动端。ChartAgent为图表引入了视觉工具使用。VisualWebArena和AgentVista（2026年）是前沿追逐的基准——即便Gemini 3 Pro和Claude Opus 4.7在AgentVista的困难任务上也只有约30%的得分。这个总复习汇集了第12阶段的每一条线索：感知（高分辨率VLM）、推理（带工具使用的LLM）、定位（坐标输出）、长周期记忆和评估。

**类型：** 总复习
**语言：** Python（标准库，动作模式 + Agent循环骨架）
**前置条件：** 第12阶段 · 05（LLaVA），第12阶段 · 09（Qwen-VL JSON），第14阶段（Agent工程）
**时间：** ~240分钟

## 学习目标

- 设计一个多模态Agent循环：感知 → 推理 → 执行 → 观察 → 重复。
- 构建一个VLM能以JSON发出的GUI定位输出模式（点击坐标、输入文本、滚动、拖拽）。
- 对比纯截图Agent vs 无障碍树Agent vs 混合Agent。
- 在小型VisualWebArena切片上搭建多模态Agent基准评估。

## 问题

一个订票网站的工作流："帮我找4月15日飞东京的航班，要求靠过道座位、800美元以下，直接订。"

一个多模态Agent需要：

1. 截取浏览器截图。
2. 将截图 + URL + 目标解析成计划。
3. 发出结构化动作：点击（坐标x,y处）、输入"东京"（元素E处）、向下滚动、选择（单选按钮）。
4. 将动作应用到浏览器。
5. 观察新状态（下一张截图）。
6. 重复直到任务完成。

每一步都是一次多模态VLM调用。VLM输出必须是可解析的JSON。错误会跨步骤累积，所以恢复能力至关重要。

## 概念

### GUI定位——基元

GUI定位是指：给定一张截图和一条自然语言指令，输出要点击的 (x, y) 坐标（或其他动作）。

SeeClick（arXiv:2401.10935）是第一个规模化开源结果：在合成 + 真实GUI数据上微调VLM，以纯文本令牌输出坐标。有效。

CogAgent（arXiv:2312.08914）加入了1120x1120高分辨率编码用于密集UI。得分：网页导航约84%。

Ferret-UI（arXiv:2404.05719）专注于移动UI，集成iOS无障碍数据。

输出格式通常是JSON：

```json
{"action": "click", "x": 384, "y": 220, "element_desc": "搜索按钮"}
```

`element_desc` 帮助恢复：如果坐标在截图间漂移，语义提示让系统可以重新定位。

### 动作模式

典型的动作模式有6-10个动作类型：

- `click`：(x, y)
- `type`：(text, x?, y?)
- `scroll`：(direction, amount)
- `drag`：(x0, y0, x1, y1)
- `select`：(option_index)
- `hover`：(x, y)
- `navigate`：(url)
- `wait`：(ms)
- `done`：(success, explanation)

Agent每步发出一个动作。浏览器包装器执行并返回新状态。

### 纯截图 vs 无障碍树

两种输入模式：

- 纯截图：完整图像，无结构信息。最通用；适用任何应用。
- 无障碍树：结构化DOM / iOS无障碍信息。定位可靠得多；在树可用处有效。
- 混合：两者兼有，以树为原子动作的可靠定位器，以截图为语义上下文。

生产Agent在可能时使用混合。浏览器自动化（Selenium + 无障碍）总是有树；桌面应用有时有。

### 长周期记忆

一个20步的工作流产生20张截图。VLM的上下文快速填满。三种压缩策略：

- 摘要链：每5步后总结发生了什么，丢弃旧截图。
- 跳帧：保留第一张、最后一张和每第3张截图。
- 工具记录日志：执行动作，保留所做之事的文本日志；不再回看旧截图。

Claude的computer-use API使用日志模式。更简单，更可靠。

### 视觉工具使用

ChartAgent（arXiv:2510.04514）为图表理解引入了视觉工具使用：裁剪、缩放、OCR、调用外部检测。Agent可以发出"裁剪区域(100, 200, 300, 400)然后调用OCR"作为工具调用。工具返回文本；VLM继续推理。

这个模式可泛化：set-of-mark提示、区域标注和外部检测工具都适合同一个"发出工具调用，接收结构化响应"的模式。

### 2026年基准

- ScreenSpot-Pro。约1k网页截图的GUI定位。开源SOTA Qwen2.5-VL-72B ~85%。前沿 ~90%。
- VisualWebArena。端到端网页任务（购物、论坛、分类广告）。开源SOTA ~20%。Gemini 3 Pro ~27%。
- AgentVista（arXiv:2602.23166）。2026年最难的基准。跨12个领域的真实工作流。前沿模型得分27-40%；开源模型10-20%。
- WebArena / WebShop。更老的基准；已被前沿饱和。

### 为什么仍然难

Agent性能瓶颈：

1. 细粒度视觉定位。在小分辨率下"点击小X"经常失败。
2. 长周期规划。10步后Agent偏离目标。
3. 错误恢复。点击失败（按错按钮）时，检测 + 恢复的训练数据极少。
4. 跨页上下文。跳标签页或长表单丢失状态。

研究方向：记忆架构、显式重新规划、多模态验证（截图对比确认动作成功）。

### 总复习构建任务

总复习任务：构建一个计算机使用Agent，它：

1. 读取订票网站模拟页面的HTML + 截图。

3. 发出匹配动作模式的JSON动作。
4. 在固定的10任务切片上评估。

课程提供脚手架代码，易于扩展到真实浏览器。

## 📝 教师备课总结与读后感

### 一、文档整体评价
这是第12阶段的多模态Agent总复习，也是整个课程中"离产品最近"的一课。文档以订票网站为叙事锚点，从感知→推理→执行→观察的Agent循环到GUI定位基元，再到2026年基准对比，层层递进。定位基元（SeeClick、CogAgent、Ferret-UI）和评估基准（AgentVista 30%这个数字）的结合让学习者在架构设计和现实差距两个维度同时产生认知冲击。

### 二、知识结构梳理
- **基础层**：GUI定位的(x,y)坐标输出范式、VLM作为动作生成器的调用循环、JSON动作模式的序列化/反序列化
- **模式层**：Agent循环的四个阶段（感知→推理→执行→观察）、混合输入模式（截图+无障碍树）的互补性、长周期记忆的三种压缩策略
- **应用层**：ScreenSpot-Pro→VisualWebArena→AgentVista的基准演进路径、计算机使用Agent在订票/购物/导航等真实场景中的端到端工作流设计

### 三、核心洞察
1. **30%的AgentVista得分是整个领域的现实检查**——即便是Gemini 3 Pro也在2026年最难的基准上只得~30%。这不是悲观，是让学生理解AutoGPT式的"AI代理一切"叙事与工程现实之间的巨大鸿沟。
2. **GUI定位不是"检测加分类"的问题，是VLM的细粒度视觉理解问题**——传统的对象检测有BBox标注，但"点击搜索栏的放大镜"这类的GUI定位需要语义理解和像素级精度。这就是为什么专门微调的VLM（SeeClick 84%）强于通用检测。
3. **混合模式（截图+树）是生产级的正确选择**——树提供可靠的坐标定位，截图提供通用的视觉上下文。单独任一种都有盲区：纯树无法处理验证码，纯截图难以精确定位输入框。
4. **记忆压缩是长周期Agent的隐藏瓶颈**——20步20张截图填满32k上下文，摘要链/跳帧/日志三种策略分别适合不同场景。这不是"锦上添花"的优化，是10步以上任务的生存必需。
5. **错误恢复是训练数据的最大缺口**——几乎所有VLM都在"正确场景"上训练，极少数训练数据覆盖了"按钮没点中对不对该怎么办"。这是2026年Agent从演示到生产的最大障碍。
6. **`element_desc` 字段是工程实践者对学术输出的补充**——论文中很少讨论这个，但它是生产系统中坐标漂移时的救命稻草。

### 四、教学建议
1. **用"让你妈妈订机票"做故事开场**——"假设你妈打开一个网页，她能认出搜索框、填入目的地、选航班、填信用卡。但AI Agent必须做完全一样的事，而它不能'看'网页——它看到的是一串像素坐标。"让学生从第一分钟就理解问题的难度。
2. **现场演示Agent循环的每一步**——准备一个订票网站模拟页面，用LLaVA或Qwen-VL做实时演示：截图→JSON输出→解析→模拟点击。学生看过一次循环，胜过读十页文档。
3. **动作模式做互动设计工作坊**——给一个具体任务（比如"在淘宝找一条低于100块的瑜伽裤加购物车"），让学生自己设计需要的动作类型，然后对照标准动作模式讨论差异。
4. **AgentVista 30%这个数字作为核心教学内容**——不要只提一下。把这个数字及其含义讲透：它意味着即便最好的模型，每3个复杂任务就失败2个。让学生估算一个需要10步的任务在30%成功率下的完成概率（0.3^10 = 极低），引发对可靠性的深层思考。
5. **三种记忆策略做A/B/C实验**——同样的10步任务，分别用摘要链、跳帧、日志三种策略，让学生观察哪种策略下Agent更快偏离目标。实验结果比理论论证有说服力。
6. **坐标漂移模拟**——做一个简单HTML页面，每点击一次按钮，页面布局轻微变化（比如广告载入）。让学生体验`element_desc`字段的语义回退机制。
7. **"前沿模型30%——你能做什么"的批判性讨论**——引导学生思考：生产环境中，你可以通过哪些工程手段把30%提到可以接受的可靠性？（重试+更小步骤+人工审核+混合模式oracle建议等）

### 五、值得补充的内容
1. **set-of-mark提示技术的详细解释**——"在截图上画编号标记，让模型输出数字而非坐标"是Apple和Google在各自产品中使用的技术，更可靠。这个应该深度补充。
2. **ACT-1到Adept到Claude Computer Use的工业演进线**——从2022年的ACT-1创业公司到Adept的Fuyu模型再到Anthropic收购后的Claude Computer Use，讲一个工业界从研究到产品的完整故事线。
3. **无障碍树的实际获取代码**——一句代码 `document.querySelectorAll('*')` 和iOS的AXUIElementCopyAttributeValue 的实际调用，让学生知道"树"从哪里来。
4. **失败模式的可视化分类**——将AgentVista上的失败案例分成"定位失败"、"规划偏离"、"不支持的动作"、"环境变化"等类别，让学生有针对性地补习弱项。

### 六、一句话总结
多模态Agent的终极挑战不是"看懂页面"（感知），也不是"想出计划"（推理），而是"在连续20步中每一步都做对，并且知道自己做错了"——而这就是AgentVista 30%告诉我们还远远没到的那条线。

# 🎓 Agent 架构课：计算机使用Agent——为什么你的Agent会在第7步崩溃

要我猜猜你的计算机使用Agent是怎么死的吗？

第一步，它截图，认出搜索栏，在正确的位置输入"东京"。第二步，它点了搜索按钮。第三步，它看结果。一切正常。第四步它点了一个航班。第五步填入姓名。第六步——等等，这个复选框怎么移动了？第七步，它崩溃了。不是崩溃，是开始对着一个不相关的按钮疯狂点击。

**我很熟悉这个死法。我见过至少200次。**

让我告诉你Agent在第7步死掉的真正原因，以及"把无障碍树扔进上下文"这件事为什么比你想象的重要十倍。

---

## 问题本质：Agent不会"看到"——它在猜测

我问你一个看似简单的问题：当你说"点击搜索按钮"时，你的Agent实际上在做什么？

它在看一张1310x900像素的截图，然后输出 `(x: 680, y: 45)`。

这不是"看到"。这是根据一张压缩过的图像里的颜色分布，推断出"搜索按钮"可能在哪里。你的Agent在做的事和一个人蒙着眼睛、通过别人用文字描述的图像来猜位置——没有太大区别。唯一差別是Agent做这件事只需要0.3秒。

**我为什么在乎？** 因为坐标偏移1个像素在小屏幕上可能就是从"搜索"点到"清除"，偏移50个像素就是从"确认付款"点到"取消订单"。这不是一个"模型最好再提升2%准确率"的问题——这是"我的生产系统在帮用户订机票还是在打游戏"的问题。

---

## 两条路径：你给Agent什么信息

**路径A：纯截图**
- 给VLM一整张截图
- 模型必须从像素中推断UI结构
- 适用任何应用（浏览器、桌面App、手机App、游戏）
- 定位准确率：前沿模型~90%在ScreenSpot-Pro上（意味着10个按钮里点错1个！）

**路径B：截图 + 无障碍树（混合模式）**
- 将DOM树序列化为文本（`<button id="search-btn" bounds=(670,40,720,52)>搜索</button>`）
- VLM同时看截图和树的结构信息
- 需要浏览器环境，不适用纯桌面应用
- 定位准确率：在树可用时接近100%的原子动作

你看到这个差距了吗？一个10步工作流：
- 纯截图方案：0.9^10 = 35%的完成概率
- 混合方案：接近100%的原子动作完成率

35% vs 接近100%。不是"稍微好一点"。是"能用"和"不能用"的区别。

---

## 深入原理：为什么树比截图可靠这么多

这不是模型质量问题。这是信息论问题。

一张截图有1310x900x3 = 350万个数字。每对(x,y)坐标是2个数字。从350万个数字里精确选出2个，这是把世界塞进针眼。

一个序列化的无障碍树可能是："Window > div.header > div.search > input[type=text](780,200,900,220) > button[type=submit](920,200,960,220)"。60个词，把所有可交互元素的类型、层级和坐标告诉了你。

**模型最擅长的是什么？处理文本序列。模型最不擅长的是什么？精确空间推理。**

你在让模型用最不擅长的事情做最关键的决定。

但等一下——截图仍然需要。为什么？因为无障碍树有盲区：验证码、Canvas渲染的图表、自定义绘制的UI。混合模式的想法是"用树做精确的事情（坐标），用截图做语义的事情（这是什么玩意？）"。

## 生产数字：一个20步工作流的真实预算

在这里产品经理和架构师的对话通常是这样的：

PM："所以我们这个Agent能自动化客服后台的完整操作吧？大概30步？"

我："让我们算算。"

假设使用Claude Computer Use API（每步一次调用，包含截图）：
- 每张截图约150KB编码后送入API
- 20步 = 3MB图像数据 + 文本上下文
- 当前Claude上下文已经足够大，但token消耗暴增
- 每次调用约$0.15-0.30（带截图的完整上下文）
- 20步 = $3-6 per task

现在乘以每天1000个客服任务 = $3,000-6,000/天。

"哦，"PM说。"那能不能只用无障碍树不用截图？"

"不能。无障碍树看不到验证码。"

"那少截几张图？"

"可以，记忆压缩。只保留首帧、尾帧和每第三帧的截图。成本降60%。定位准确率可能降5%。"

这就是生产中的交易——每降低一个维度的成本，就在另一个维度增加风险。而谁来做这个交易？我。

## 反模式：把Agent循环当成线性管道

新手架构师通常会这样设计Agent循环：

```
state → LLM → action → execute → state → LLM → ...
```

看起来没问题。前进、执行、再前进。

**问题是什么？** 这个循环假设每一步都成功。它不包含"上一步失败了"的意识。

生产级Agent循环必须是这样的：

```
state → LLM → action → execute → 
    if success: next state
    if failure: recovery LLM → recovery action → re-execute → 
        if still failure: escalate to human
```

这个额外的"恢复分支"不是锦上添花。它是Agent能否跑完10步以上的唯一保障。而恢复LLM的提示模板和动作空间必须和主LLM不同——"按钮似乎不可用，你能看到附近有其他看起来相似的按钮吗？截图这个区域给我。"

我见过太多Agent Demo在演示中完美运行5步，然后上线第一天在第8步静默失败——因为没有人教Agent如何检测和恢复。

## 结语：架构师的决策清单

构建一个计算机使用Agent，第一步不是选模型——是选信息源：

1. **你的目标环境有结构化UI吗？** 如果有（网页、iOS），用混合模式。如果没有（游戏、自定义桌面App），纯截图是你唯一的选择。接受更低完成率，加更多人工审核。
2. **你的工作流有多长？** <5步：纯截图勉强可行。5-15步：必须有记忆压缩。>15步：需要显式的进度追踪和人工后备。
3. **每个动作的成本是多少？** 如果你的Agent API调用费用 > 人工操作成本，Agent的存在毫无意义。计算每一分钱。
4. **错误恢复你设计了吗？** 如果你的Agent循环中没有失败分支，你还没有Agent——你有一个演示。
5. **你的用户能接受多少失败率？** 订票Agent不能接受 >1% 的支付错误。备忘录Agent能接受 20% 的格式偏差。设计不同的容错策略。

**金句：Agent不是在"做对了多少步"这件事上竞争——是在"做错了之后怎么办"这件事上竞争。**

AgentVista 30%不是在告诉你模型多聪明。它在告诉你：在真实世界中，每一步都可能出问题，而出问题后会发生什么，我们几乎完全没有训练数据。
