# 人类在环：提案后提交

> 2026 年关于人类在环的共识是具体的。它不是"Agent 问，用户点批准。"它是提案后提交：拟议的操作以幂等键持久化到持久化存储；附带意图、数据来源、触及的权限、爆炸半径和回滚计划呈现给审查者；仅在积极确认后提交；执行后验证以确认副作用确实发生。LangGraph 的 `interrupt()` 加 PostgreSQL 检查点、Microsoft Agent Framework 的 `RequestInfoEvent`、Cloudflare 的 `waitForApproval()`——都实现了相同的形态。经典的失败模式是橡皮图章批准：未审查就点击"批准"。文档记录的缓解措施是带显式检查清单的挑战-应答。

**类型：** 学习
**语言：** Python（标准库，带幂等的提案后提交状态机）
**前置课程：** Phase 15 · 12（持久化执行）、Phase 15 · 14（触发线）
**用时：** 约 60 分钟

## 问题

Agent 采取了操作。用户必须决定：批准或不批。如果决策是瞬间的，很可能不是审查。如果决策是结构化的，它是慢的但是可信的。工程问题是**如何让结构化审查成为阻力最小的路径。**

2023 年时代的人类在环模式是同步提示："Agent 想发送邮件给 X，内容为 Y——批准？"用户点击批准。每个人都觉得系统是安全的。实践中这个界面被严重橡皮图章化：用户快速批准，批准几乎不预测什么，当 Agent 出错时，审计轨迹显示用户无法回忆起的一长串批准历史。

2026 年的模式——提案后提交——将人类在环移到持久化基座上，附带结构化元数据，并需要积极提交。每个受管理的 Agent SDK 都有一个版本：LangGraph `interrupt()`、Microsoft Agent Framework `RequestInfoEvent`、Cloudflare `waitForApproval()`。API 名称不同；形态相同。

## 概念

### 提案后提交状态机

1. **提案。** Agent 产出一个拟议操作。持久化到持久化存储（PostgreSQL、Redis、Durable Object）。包含：
 - 意图（Agent 为什么做这个）
 - 数据来源（什么来源导致了这个提案）
 - 触及的权限（哪些作用域/文件/端点）
 - 爆炸半径（最坏情况是什么）
 - 回滚计划（如果提交了，怎么撤销）
 - 幂等键（每次提案唯一；重复提交返回同一记录）
2. **呈现。** 审查者看到提案及其所有元数据。审查者是人（不是 Agent 审查自己）。
3. **提交。** 积极确认。操作执行。
4. **验证。** 执行后，副作用被读回并确认。如果验证步骤失败，系统处于已知坏状态并触发告警。

### 幂等键

没有幂等键，短暂失败后的重试可能双重执行一个已批准的操作。具体例子：用户批准"从 A 转账 $100 到 B。"网络闪断。工作流重试。用户批准了一次，但转账执行了两次。幂等键将批准绑定到单一、唯一的副作用；第二次执行是空操作。

这是 Stripe 和 AWS API 使用的同一个幂等模式。将其复用于 Agent 批准在 Microsoft Agent Framework 文档中是明确的。

### 持久化：为什么批准比进程活得更久

批准等待室是 Agent 不拥有的一段状态。工作流被暂停（第 12 课）。当批准到达时，工作流从该确切点恢复。这就是为什么 LangGraph 将 `interrupt()` 与 PostgreSQL 检查点配对而非仅内存状态——两天后的批准仍然能找到完整的工作流。

### 橡皮图章批准与挑战-应答缓解

人类在环的默认 UI（"批准"/"拒绝"按钮）产生快速批准且没有真正的审查。文档记录的缓解措施：一个挑战-应答检查清单，要求在批准按钮启用之前对特定问题给出积极回答。具体形态：

- "你是否理解此操作触及了什么资源？[ ]"
- "你是否验证了爆炸半径是可接受的？[ ]"
- "如果此操作失败，你是否有回滚计划？[ ]"

不是为了官僚主义而官僚主义——是一个强制函数。无法勾选这些框的审查者要么请求澄清（升级），要么拒绝（安全默认）。Anthropic 的 Agent 安全研究明确引用检查清单驱动的人类在环作为橡皮图章批准模式的缓解措施。

### 什么算作"后果性操作"

不是每个操作都需要提案后提交。2026 年指导：

- **后果性操作**（始终人类在环）：不可逆写入、金融交易、外发通信、生产数据库变更、破坏性文件系统操作。
- **可逆操作**（有时人类在环）：本地文件编辑、staging 环境变更、有明确回滚的可逆写入。
- **读取和检查**（永不人类在环）：读取文件、列出资源、调用只读 API。

### 执行后验证

"提交执行了"不等同于"副作用发生了。"网络分区和竞态条件可能产生一个工作流认为自己成功而后端没有持久化的情况。验证步骤在提交后重新读取目标资源以确认。这与带 `RETURNING` 子句的数据库事务或 `PutObject` 后的 AWS `GetObject` 是同一模式。

### 欧盟 AI 法案第 14 条

第 14 条要求对欧盟高风险 AI 系统进行有效的人类监督。"有效"不是装饰性的。监管语言特别排除了橡皮图章模式。提案后提交加挑战-应答是微软 Agent Governance Toolkit 合规文档中经受第 14 条审查的形态。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 提案后提交（Propose-then-commit） | "两阶段批准" | 持久化提案 + 积极提交 + 验证 |
| 幂等键（Idempotency key） | "重试安全令牌" | 每次提案唯一；第二次执行是空操作 |
| 数据来源（Data lineage） | "它从哪来的" | 导致提案的具体来源内容 |
| 爆炸半径（Blast radius） | "最坏情况" | 操作出错时的效应范围 |
| 橡皮图章（Rubber-stamp） | "快速批准" | 未经真正审查点击"批准" |
| 挑战-应答（Challenge-and-response） | "强制检查清单" | 审查者必须积极确认特定问题 |
| RequestInfoEvent | "MS Agent Framework 原语" | 带结构化元数据的持久化 HITL 请求 |
| `interrupt()` / `waitForApproval()` | "框架原语" | LangGraph / Cloudflare 的同一形态对等物 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 16 高级 Agent 工程中最"人类"的一课。前面四节课讲的全是系统层面——长程趋势、监管者架构、持久化执行、成本调控。这节课讲的是系统边界上那个最不可靠的组件——人类。文档的核心洞察是反直觉的：**加一个"批准"按钮不会让 Agent 更安全——它让 Agent 看起来更安全但实际风险不变。** 提案后提交模式通过持久化、结构化元数据和挑战-应答检查清单，将人类在环从"无审查的点击"变成"有结构的决策"。三个主流框架（LangGraph/MS/Cloudflare）独立收敛到同一 API 形态是一个强烈的信号。

### 二、知识结构梳理

**状态机层：** 提案后提交的四阶段——提案（持久化+元数据）、呈现（结构化审查）、提交（积极确认）、验证（读回确认）——将人类决策嵌入 Agent 工作流的确定位置。

**安全工程层：** 三个保障——幂等键（防双重执行）、持久化等待室（批准比进程活得久）、挑战-应答检查清单（防橡皮图章）——每个都是对一种特定失败模式的工程回答。

**合规层：** 欧盟 AI 法案第 14 条将结构化人类在环从"最好有"升级为"必须有"。橡皮图章模式在监管审查中不成立。提案后提交加挑战-应答是已知的合规形态。

### 三、核心洞察（备课时的关键理解）

1. **"批准按钮 = 安全幻觉"是这节课的出发点。** 加一个"批准？"提示不增加安全——它增加了对安全的错觉。用户在 200 毫秒内点批准。审计轨迹显示一连串用户无法回忆的批准。这个模式在安全工程中有一个名字——**橡皮图章。**

2. **挑战-应答检查清单是强制函数，不是官僚主义。** 三大问题——"你理解触及了什么资源？爆炸半径可接受吗？有回滚计划吗？"——这三问将"点一下"变成"被迫思考"。不能诚实勾选→要么升级要么拒绝。这不是增加摩擦——是让摩擦对齐到风险。

3. **幂等键在 Agent 批准中的重要性被普遍低估。** 用户批准了一次转账。网络闪断。工作流重试。没有幂等键→转账执行两次。有幂等键→第二次是空操作。这和 Stripe API 的幂等是同一个问题——但 Agent 开发者往往不认为"用户批准"需要幂等保护。

4. **"批准比进程活得久"是持久化执行在人类在环场景的直接应用。** 用户需要两小时后才看审批请求。如果批准等待状态在 Agent 进程内存中——Agent 崩溃，批准请求丢失。持久化存储中——Agent 崩溃，审批请求还在，用户回复后工作流从暂停点恢复。

5. **"执行后验证"补上了"提交成功了 ≠ 副作用发生了"的缺口。** 网络分区可能导致工作流认为提交成功但后端实际没有持久化。读回确认——AWS S3 的 `PutObject` 后 `GetObject` 模式——是分布式系统的基本功，但在 Agent 工作流中常被省略。

6. **后果性操作的分类（总是/有时/永不）简化了人类在环的工程实现。** 不是每个操作都需要四阶段审批。不可逆写入→始终。可逆编辑→有时。只读操作→永不。这个分类让人类在环从"每个操作都问"（审批疲劳）变成"只在后果不可逆时问"（有意义的人类决策）。

7. **LangGraph/MS/Cloudflare 三者的 API 收敛——`interrupt()`、`RequestInfoEvent`、`waitForApproval()`——说明人类在环的基础设施形态已经标准化。** 名称不同，四阶段状态机相同。选哪个框架的差异不在"HITL 怎么实现"——在每个框架的检查点后端和持久化策略。

### 四、教学建议

1. **用"批准按钮"的橡皮图章 demo 开场。** 让学生做一个实验：Agent 说"发邮件给客户说项目延期了"，审批提示弹出。计时——从弹出到点击"批准"的平均时间是多久？< 1 秒？→ 这不是审批，是路过。

2. **挑战-应答检查清单的"三问"让学生自己设计。** 场景："Agent 想给你的生产数据库加一个索引。"问学生——你会让审查者回答哪三个问题才允许点"批准"？对比他们的答案和文档的三问——意图/爆炸半径/回滚的覆盖范围。

3. **幂等键的双重执行 demo 最直观。** 跑一遍：正常流程→转账一次。去掉幂等键→网络闪断→转账两次。加上幂等键→闪断→转账一次。"一行哈希值保护了你的用户不被双重扣款。"

4. **"什么算后果性操作"值得一个分类练习。** 给 10 个操作（删除文件、读取配置、发送邮件、修改 staging、写入生产数据库、列出目录），让学生分三类——始终 HITL、有时、永不。讨论边界案例——"发送邮件"如果发送给内部团队呢？"修改 staging"如果 staging 和生产共享数据库呢？

5. **执行后验证用"事务的 RETURNING 子句"类比。** `INSERT ... RETURNING id`——你不假设插入成功，你读回确认。Agent 也如此——`write_file(path, content)`→然后 `read_file(path)` 确认内容正确。

6. **练习 4（为发 Twitter 设计检查清单）是让学生理解"后果不同→审查不同"的练习。** 发 Twitter 的问题不是"发邮件给客户"的那三个——你需要"内容审查？品牌调性？时间敏感度？"让学生发现检查清单不是通用的。

7. **练习 5（什么时候同步批准就够了）是理解 risk acceptance 的练习。** 让学生选一个他们愿意接受无持久化的场景——然后明确说出他们接受了什么风险。这是架构师的思维——不是避免所有风险，是有意识地接受某些风险。

### 五、值得补充的内容

1. **审批疲劳及其缓解。** 100 个"批准？"提示/天→第 50 个开始不看就点。缓解：后果性操作才 HITL；批量审批（"批准這 5 个安全的变更"）；信任等级（过去 100 次审批都对的 Agent→放宽审查频率）。

2. **人类在环的延迟预算。** 提案后提交的最快路径——提案→呈现→提交→验证——需要多久？5 秒？5 分钟？对延迟敏感的操作（实时聊天）不适合。

3. **回滚计划的自动化验证。** "回滚计划：执行 `DROP INDEX`"——这个回滚计划本身是否有效？自动化验证回滚计划的可行性——可以在 staging 上跑一次看看能不能回滚吗？

4. **与 Phase 15·05（验证门控）的关系。** 提案后提交是"人类确认操作"。验证门控是"代码确认制品"。两者互补——一个是人的判断，一个是确定性的检查——在 Agent 工作流的不同节点。

5. **多级批准——不是 Level 1 批准就够的。** 金融交易→金额 > $1000→需要两人批准。发布到生产→需要值班+发布经理。多级批准是提案后提交的扩展——多个审查者，每个需要积极提交。

### 六、一句话总结

**人类在环的真正价值不是"让人类点一下批准"——是让人类在被给予正确信息（意图、来源、爆炸半径、回滚）和被迫思考（挑战-应答检查清单）后做出决定。批准按钮是安全幻觉。提案后提交是安全工程。**

# 🎓 Agent 架构课：人类在环——为什么"批准"按钮是你的安全幻觉

同学们好。我是你们的 Agent 架构老师。

前面四节课我讲了长程趋势、监管者模式、持久化执行、成本调控。这四节课的共同画面是 Agent 在自主运行。今天这节课讲的是那个最不靠谱但最关键的组件——**人类。**

Agent 要做一个操作。它说："我想发这封邮件。"你点了一个按钮——"批准"。Agent 发了。你觉得系统是安全的——有人类把关。

让我问你：你刚才点了"批准"的一瞬间，你真的看邮件内容了吗？你看了收件人是谁吗？你确认了这封邮件说的是对的吗？

如果你在 200 毫秒内点了"批准"——你不是在审查。**你是在路过。**

这就是橡皮图章。2023 年的所有 Agent 系统都这样做。2026 年的共识是一个不同的模式——**提案后提交。**

## 不是"问一下"——是"提案-呈现-提交-验证"

提案后提交把人类决策从一个按钮变成了四个阶段的状态机。

**第一阶段：提案。** Agent 产出一个操作提案——不只说"发邮件给 X"，而是附带五样东西：意图（为什么发这封邮件）、数据来源（从哪得出需要发这封邮件）、权限（触及了什么文件/API）、爆炸半径（最坏情况——这个邮件发错了会怎样）、回滚计划（如果错了怎么挽回）。

这些元数据不是可选的装饰。它们是人类能做出真正审批决策所需的**信息。**

**第二阶段：呈现。** 人类看到提案和它的全部元数据。不只是"批准？"—是一个结构化的审查界面。

**第三阶段：提交。** 人类积极确认。不只是点一个按钮——是被迫回答三个问题后才能点。我们在挑战-应答部分展开这个。

**第四阶段：验证。** 操作执行后——Agent 读回确认副作用确实发生了。邮件在发件箱里吗？数据库记录存在吗？"提交执行了"≠"副作用发生了。"网络分区、竞态条件——分布式系统的经典问题在 Agent 工作流中同样存在。

## 为什么需要一个幂等键

用户批准了一次转账。网络闪断。工作流重试。

没有幂等保护——这次转账会被执行两次。用户批准了一次，但 Agent 转账了两次。银行账户扣了双倍。

幂等键——`hash(thread_id, action_signature)`——将每次审批绑定到一个唯一的副作用。第二次执行是空操作——持久化存储说"这个键的审批已经执行过了。"

这和 Stripe API 的幂等键是同一个模式。在 Agent 审批中的重要性完全一样——但被普遍忽略。

## 为什么批准需要比进程活得更久

Agent 在下午 3 点发出一个审批请求。人类在下午 5 点才看到——会议结束后。

如果审批在 Agent 进程的内存中——Agent 在这两小时内崩溃了。审批请求丢失。人类 5 点回来——什么都没有。

持久化执行（第 18 课）在这里是基础设施。审批等待室是一段持久化状态——存储在 PostgreSQL、Redis 或 Durable Object 中。Agent 崩溃了——审批还在。人类回复——工作流从暂停点恢复。

这就是为什么 LangGraph 将 `interrupt()` 与 PostgreSQL 检查点配对——不只是内存状态。两天后的审批仍然能找到完整的工作流。

## 挑战-应答：不能让人类路过

橡皮图章的问题是：人类不思考就点。

解决方案不是"让人类更认真"——是**让人类不能不思考。** 挑战-应答检查清单：

- "你是否理解此操作触及了什么资源？☐"
- "你是否验证了爆炸半径是可接受的？☐"
- "如果此操作失败，你是否有回滚计划？☐"

三个框没勾→"批准"按钮是灰色的。这不是官僚主义。这是强制函数。不能诚实勾选→要么请求更多信息（升级），要么拒绝（安全默认）。

这三个问题覆盖了人类审批决策的三个维度——**知道在做什么、知道最坏的后果、知道怎么挽回。** 少任何一个，审批是不完整的。

## 不是每个操作都值得人类的时间

如果你让人类审批每一个文件读取——人类会在第 50 次审批后完全忽视提示。审批疲劳 = 橡皮图章。

2026 年的分类：

- **后果性操作——始终人类在环。** 不可逆写入、金融交易、外发邮件、生产数据库变更、`rm -rf`。
- **可逆操作——有时人类在环。** 本地编辑、staging 变更、有明确回滚的写入。
- **读取和检查——永不人类在环。** 读文件、列资源、只读 API。这些不值得人类的时间。

边界线在哪？"发送邮件给客户"——后果性。发送错了你就失去了客户。"发送邮件给内部团队"——有时。内部邮件错了可以补救。

## 欧盟 AI 法案：这不仅是好实践，是法律

第 14 条要求对欧盟高风险 AI 系统进行有效人类监督。"有效"在监管语言中不是装饰。它具体排除了橡皮图章模式。提案后提交加挑战-应答检查清单是已知的合规形态。

如果你的 Agent 在欧盟有用户——人类在环不是你选的——是法律要求的。

## 结课清单

为 Agent 添加人类在环：

1. 提案后提交——不只是一个"批准"按钮。提案→呈现→提交→验证。四个阶段。
2. 提案附带元数据——意图、数据来源、权限、爆炸半径、回滚计划。人类需要这些才能做决定。
3. 幂等键——`hash(thread_id, action_signature)`——防止双重执行。
4. 持久化等待室——批准比 Agent 进程活得久。
5. 挑战-应答检查清单——三问必答才能点批准。
6. 执行后验证——"提交执行了"≠"副作用发生了"。读回确认。
7. 后果性操作才人类在环——不可逆=始终、可逆=有时、只读=永不。

**最后一句话：**

**人类在环的安全价值不是"有个人点了批准"——是"这个人在被给予了正确信息和被迫思考后做出了决定。"没有元数据的批准是路过。没有检查清单的批准是橡皮图章。没有幂等的批准是双重扣款。没有验证的批准是"但愿它成功了。"人类在环是你的最后一道防线——别把它设计成一个能被 200 毫秒绕过的弹出窗口。**
