# RAG：让 AI 读你的文档再回答

> **原文：** RAG (Retrieval-Augmented Generation)  
> **预计时间：** ~45 分钟  

---

## 🚗 开场翻车：AI 不知道你的退款政策

客户问你的聊天机器人："企业版退款政策是什么？"

LLM 回了一段通用 SaaS 退款政策的套话。而你们公司实际的退款政策——藏在 200 页内部 Wiki 里——写着"企业客户 60 天退款窗口，按比例退款"。

**LLM 没见过你的内部文档。它不可能知道。**

微调？花几千美元、文档一更新就过时、还看不出它依据了哪个文档。

RAG？把文档搜出来，塞进提示里，让模型基于它回答。文档几分钟更新，来源可追溯，模型不用动。

---

## 🧠 RAG 四步走

```
用户提问 → 检索相关文档 → 把文档塞进提示 → 生成答案
```

### 你需要什么

1. **文档** — 你的知识库
2. **Embedding 模型** — 把文字变成向量（语义相近的文字向量也相近）
3. **向量数据库** — 存向量 + 快速搜索
4. **LLM** — 读文档 + 生成答案

| | 微调 | RAG |
|---|---|---|
| 成本 | 每次 $1K-100K | 每次 $0.01-0.10 |
| 更新 | 重新训练 | 重新索引，几分钟 |
| 来源 | 查不到 | 能展示引用了哪个文档 |
| 幻觉 | 可能自由发挥 | 基于文档，有依据 |

## 🔧 最小 RAG 管线

```python
# 第一步：切文档（chunk）
# 文档太长模型读不完，切成小块
def chunk_document(text, chunk_size=500):
    """把长文档切成 500 字一块"""
    words = text.split()
    chunks = []
    for i in range(0, len(words), chunk_size):
        chunks.append(" ".join(words[i:i+chunk_size]))
    return chunks

# 第二步：生成 Embedding（把文字变成向量）
# 用 OpenAI 的 text-embedding-3-small
# "怎么重置密码" 和 "我需要改密码" → 几乎相同的向量
# "猫坐在垫子上" → 完全不同的向量

# 第三步：存到向量数据库
# ChromaDB、FAISS（本地）、Pinecone（云）
# 搜索时找向量最接近的 top_k 个文档块

# 第四步：把检索结果拼进提示
def build_rag_prompt(query, retrieved_docs):
    """把搜到的文档塞进提示，让模型基于它回答"""
    context = "\n\n".join(retrieved_docs)
    return f"""基于以下文档回答问题。如果文档中没有答案，说"文档中未找到"。

文档：
{context}

问题：{query}"""
```

## 🏆 焊死在脑子里的东西

1. **RAG = 搜文档 + 塞提示 + 生成答案。** 四步，不神秘。
2. **Embedding = 把文字变成向量。** 语义相近 → 向量相近。
3. **RAG 完胜微调于事实型知识。** 更便宜、更新鲜、可审计。微调只适合改风格/语气。
4. **切的块太大模型读不下，太小缺上下文。** 500-1000 token 是甜区。

---

## 📝 练习答案

两个都应该搜到。"退订"和"取消订阅"语义相同 → Embedding 向量接近 → 都会在搜索结果前面。这就是语义搜索比关键词搜索强的地方——词不同但意思一样。
