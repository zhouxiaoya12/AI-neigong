# 工具使用与函数调用

> Toolformer（Schick 等人，2023）首创了自监督工具标注。Berkeley 函数调用排行榜 V4（Patil 等人，2025）设定了 2026 年的标尺：40% Agent 场景、30% 多轮、10% 真实提示词、10% 非真实提示词、10% 幻觉检测。单轮调用已经基本解决。记忆、动态决策和长链条工具调用还没有。

**类型**：构建  
**语言**：Python（标准库）  
**前置要求**：阶段 14 · 01（Agent 循环）、阶段 13 · 01（函数调用深入）  
**预计时间**：约 60 分钟

## 学习目标

- 解释 Toolformer 的自监督训练信号：仅当执行结果降低了下一个 token 的损失时才保留工具标注
- 说出 BFCL V4 的五个评估类别及每项衡量什么
- 实现一个标准库工具注册表，包含 Schema 验证、参数强制转换和执行沙盒
- 诊断 2026 年的三个开放问题：长链条工具调用、动态决策和记忆

## 问题陈述

早期的工具使用问的是：模型能否预测一个正确的函数调用？现代的工具使用问的是：模型能否在 40 步的链条中调用工具——带记忆、部分可观测性、从工具失败中恢复、不幻觉出不存在的工具？

Toolformer 建立了基线：模型可以通过自监督学会何时调用工具。BFCL V4 定义了 2026 年的评估目标。两者之间的差距就是生产 Agent 所生活的空间。

## 核心概念

### Toolformer（Schick 等人，NeurIPS 2023）

思路：让模型在自己的预训练语料库中标注候选 API 调用。对每个候选调用，执行它。仅当包含工具结果后降低了对下一个 token 的损失时，才保留该标注。在过滤后的语料库上微调。

覆盖的工具：计算器、问答系统、搜索引擎、翻译器、日历。自监督信号纯粹围绕"工具是否帮助预测文本"——无需人工标注。

规模效应：工具使用能力随规模涌现。较小的模型因工具标注而受损；较大的模型获益。这就是为什么 2026 年的前沿模型内建了强大工具使用能力，而大多数 7B 模型需要显式的工具使用微调才能可靠。

### Berkeley 函数调用排行榜 V4（Patil 等人，ICML 2025）

BFCL 是 2026 年的事实评估标准。V4 组成：

- **Agent 场景（40%）**——完整 Agent 轨迹：记忆、多轮、动态决策。
- **多轮（30%）**——带工具调用链条的交互式对话。
- **真实提示词（Live，10%）**——用户提交的真实提示词（更难的分布）。
- **非真实提示词（Non-Live，10%）**——合成测试用例。
- **幻觉检测（10%）**——检测何时不该调用工具。

V3 引入了基于状态的评估：在工具序列之后，检查 API 的实际状态（例如"文件创建了吗？"），而不是匹配工具调用的 AST。V4 增加了网页搜索、记忆和格式敏感性类别。

2026 年的关键发现：单轮函数调用已接近解决。失败集中在记忆（跨轮次携带上下文）、动态决策（基于先前结果选择工具）、长链条（20 步以上发生漂移）和幻觉检测（当没有工具适合时拒绝调用）。

### 工具 Schema

每个厂商都有自己的 Schema。它们在细节上不同但共享相同的形态：

```
name: 字符串
description: 字符串（做什么、何时使用）
input_schema: JSON Schema（properties、required、types、enums）
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述字段是承重的——模型读取它来选择正确的工具。糟糕的工具描述是"选错工具"失败的第一根源。

### 参数验证

不信任任何工具调用。验证：

1. **类型强制转换。** 模型可能返回字符串"5"，但 Schema 说是整数。如果明确无误则强制转换；否则拒绝。
2. **枚举验证。** 如果 Schema 说 `status in {"open", "closed"}` 而模型输出 `"in_progress"`，以描述性错误拒绝。
3. **必需字段。** 缺少必需字段→立即以错误观察返回给模型，而非崩溃。
4. **格式验证。** 日期、邮件、URL——用具体的解析器验证，不用正则。

每次验证失败都应返回结构化的观察，以便模型可以用正确的格式重试。

### 并行工具调用

现代厂商支持在单轮助手回复中并行调用多个工具。循环：

1. 模型发出 3 个工具调用，带不同的 `tool_use_id`。
2. 运行时执行它们（如果独立则并行执行）。
3. 每个结果作为 `tool_result` 块返回，通过 `tool_use_id` 关联。

工程规则：把关联 ID 视为承重的。搞错它们就会造成错误工具-错误结果的路由。

### 沙盒

工具执行就是沙盒边界。详见第 09 课。简版：每个工具应指定读写范围、网络访问、超时、内存上限。通用的 `run_shell(cmd)` 是红旗；具体的 `git_status()` 更安全。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇"从历史到前沿"的压缩叙事——从 Toolformer 2023 的自监督基线到 BFCL V4 2026 的五类现实评估，60 分钟内把工具使用的进化史和开放问题全部覆盖。最大的亮点是定位精准：开门见山说"单轮已基本解决，记忆/动态决策/长链条才是硬骨头"——学生不需要浪费时间去优化一个已解决的问题。Toolformer 的"仅保留降低损失的工具调用"的自监督信号是本课最有教学价值的概念——它解释了为什么大规模模型可以"自学"工具使用而小模型不能。

### 二、知识结构梳理

**认知基础层**：Toolformer 的自监督标注机制（执行→降低损失→保留 vs 丢弃）、BFCL V4 五类评估的意图（Agent 长轨迹/多轮交互/真实分布/幻觉检测）

**工程模式层**：工具 Schema 的统一形态（name/description/input_schema）、四步参数验证管道（类型转换→枚举→必需字段→格式）、tool_use_id 的并行调用关联机制、沙盒的边界定义

**实践应用层**：2026 年三个开放问题的生产表现（长链条漂移/动态决策困难/记忆丢失）、BFCL 作为发版前基准测试的工程价值、多厂商 Schema 的转换层策略

### 三、核心洞察（备课时的关键理解）

1. **Toolformer 的自监督信号是本课最优雅的想法。** 不是人工标注"这里该调用工具"，而是让模型自己试——调了工具后下一个 token 预测更准就保留，否则丢掉。这纯粹是一个语言建模目标，却自动学会了工具使用的时机判断。

2. **"规模效应"是工具使用的隐藏条件。** Toolformer 的发现直接回应了"为什么 2026 年前沿模型能可靠调用工具而 7B 模型需要额外微调"——工具使用不是技术问题，是规模问题。让学生先读懂这个，后面的内容才有地基。

3. **BFCL V4 的五类不是随便分的——它们在模拟"一个 Agent 在现实世界中失败的所有方式"。** Agent 场景测长时间决策不漂移，多轮测上下文不丢失，真实提示词测分布外鲁棒性，幻觉检测测"知道什么时候闭嘴"。这是一个被精心设计的压力测试矩阵。

4. **基于状态的评估（state-based）是 BFCL V3 的关键创新。** 不检查工具调用长得对不对，而是检查调用之后世界的状态对不对——"文件创建了吗？""数据插入了吗？"这个转变从"语法正确"升级到了"行为正确"。

5. **参数验证不是安全功能——是恢复功能。** 验证失败不应该崩溃，应该以结构化错误形式返回给模型作为观察。这样模型才能自我修正。这呼应了上一节 Agent 循环中的"观察格式化器"概念。

6. **沙盒不是安全选项，是 Agent 工程的基本抽象。** 每个工具都必须声明它的边界——读写什么文件系统范围、有没有网络访问、超时多长、最多用多少内存。"通用 shell 执行"是一票否决的架构缺陷。

7. **BFCL 是"发布之前跑一遍"的基准，不是"跑完就赢了"的排行榜。** 它的价值在于告诉你你的 Agent 在哪些维度上会失败，而不是你排第几名。把 BFCL 当成调试工具，而不是评分工具。

### 四、教学建议

1. **Toolformer 的实验设计是最好的"科学研究方法"教学。** 在黑板上画：预训练语料→候选 API 调用→执行→算 loss→保留或丢弃→微调。每一步让学生自己预测结果——"如果模型很小会怎样？""如果没有过滤步骤会怎样？"引导学生自己发现规模效应和自监督的价值。

2. **BFCL V4 五类直接用饼图展示。** 画一个饼：40% Agent、30% 多轮、10% 真实、10% 非真实、10% 幻觉。然后在旁边标注每类测什么——"这是你 Agent 被现实锤的五个方向"。

3. **参数验证代码当场跑——故意喂脏数据。** 给 validator 传一个 string "5" 到 int 字段，一个不在 enum 里的值，一个缺 required 字段的调用。让学生看到验证器怎么处理——哪些它接住了，哪些它扔回去了。视觉反馈比文字有力。

4. **并行调用的 tool_use_id 用"快递单号"类比。** 你同时下了三个快递单，三个包裹可能以任意顺序到达。你怎么知道哪个包裹对应哪个订单？快递单号。tool_use_id 就是工具调用的快递单号。

5. **沙盒边界用"酒店房间"类比。** 每个工具的沙盒就是它在酒店里分配到的房间——能进去、能用的东西有限、有退房时间（超时）、不能去别的房间（不能访问其他工具的范围）。

6. **三个开放问题是让学生"看到终点线后面还有路"的关键。** 单轮已解决→多轮半解决→长链条未解决。画一条从"已解决"到"部分解决"到"开放问题"的进度条。让学生知道什么该优化、什么该研究。

7. **练习 1（no-op 工具）是理解"沉默是一种能力"的最佳练习。** 很多 Agent 不会说"我不该用任何工具"——它们会强制选一个最像的工具然后用错。no-op 工具教会 Agent：有时候正确答案是"什么都不调"。

### 五、值得补充的内容

1. **Toolformer 的算力成本。** 自监督标注需要对每个候选 API 调用执行并计算 loss——在预训练规模上这极昂贵。应该提一嘴为什么这不是一个每个人都跑一遍的步骤，而是"前沿实验室做一次，其他人享受成果"。

2. **BFCL 的局限性。** BFCL 只测试了固定工具集。真实 Agent 可能动态注册新工具、工具的 Schema 可能随 API 版本变化。这些场景在 BFCL 中未覆盖。

3. **中文工具调用的特殊挑战。** 中文函数名和参数的命名惯例不同——拼音、英文混合、中英夹杂。这在多语言工具 Schema 设计中增加了额外复杂度，值得作为实践注意点提出。

### 六、一句话总结

**2023 年的 Toolformer 教会模型"该不该用工具"，2026 年的 BFCL 告诉我们"用得好不好"。单轮调用已经闭环，但记忆、动态决策和长链条仍是 Agent 工程的开放前线。**

# 🎓 Agent 架构课：工具使用——单轮调用已经赢了的战场，以及还没赢的前线

同学们好。

上节课我们讲了 Agent 循环——观察→思考→行动→...那个不变的基石。今天我们把循环里最关键的一个节点放大：**工具调用。**

为什么工具调用值得单独一节课？因为它是 Agent 和外部世界之间唯一的接口。LLM 能做的所有事情中，只有工具调用能让它实际改变什么——改文件、发邮件、查数据库、调 API。其他一切都是内部生成。

而且——这是今天的核心洞察——**单轮工具调用已经基本解决了。** 2026 年的前沿模型在你给它一个明确的工具列表和明确的查询时，几乎不会选错工具。

但现实世界不是单轮的。现实世界的 Agent 要面对 40 步的工具链条、跨轮次的记忆、基于中间结果的动态决策、以及——什么时候该承认"我不该用任何工具"。

这就是今天的内容。从一个 2023 年的优雅实验开始。

## Toolformer——让模型自己学会"要不要用工具"

2023 年，Meta AI 的 Schick 等人问了一个问题：我们能不能不告诉模型"这里该用工具"，而是让模型自己发现？

他们的方案极优雅。做法是这样的：

1. 拿预训练语料库（海量文本）。
2. 在文本中随机插入候选 API 调用——"要不要在这里查一下天气？要不要在这里调一下计算器？"
3. 对每个候选，执行它，拿到结果。
4. 把结果也插到文本里，继续预测下一个 token。
5. 如果加了工具结果之后，模型对下一个 token 的预测更准了（loss 降低了）——保留这个工具调用标注。如果加了之后反而更差了或者没变——丢掉。
6. 在过滤后保留的语料上微调模型。

**全过程零人工标注。** 信号全部来自"工具是否帮助预测文本"。

结果呢？大规模模型从工具标注中获益。小模型反而被工具标注伤害——它们在工具调用的开销中丢失了对语言本身的学习能力。工具使用能力是**涌现的**——它随规模增长而自然出现。

这就是为什么 2026 年 GPT-5、Claude 4.7、Gemini 3 这些前沿模型内建了强大的工具使用能力——它们的规模已经跨过了涌现阈值。而 7B 的开源模型需要显式的工具使用微调，因为它们还没到那个规模。

**Toolformer 不是让你去复现的实验。是帮你理解"为什么前沿模型天生会用工具"的解释框架。**

## BFCL V4——2026 年的工具使用评分卡

有了基础能力，怎么衡量它？这就是 Berkeley 函数调用排行榜的用武之地。

BFCL V4 不是随机出题。它有五类测试，每一类模拟 Agent 在现实中被锤的一种方式：

**40% Agent 场景。** 不是"调一个工具回答一个问题"。是完整的 Agent 轨迹——十几步、几十步的决策链条。测什么？测你 Agent 会不会在第五步之后开始漂移——基于一个略有错误的结果做了一个略微偏离的决定，然后下一步再偏一点，再偏一点...最终答案完全跑偏。

**30% 多轮交互。** 用户说"帮我安排一个会议"，Agent 说"什么时间？"，用户说"周二下午"——对话在继续，Agent 必须记住上下文并持续调用正确的工具。测记忆。

**10% 真实用户提示词。** 不是研究人员精心构造的测试用例。是真实用户在日常使用中打出来的、可能包含拼写错误、模糊需求、不完整信息的提示词。测分布外鲁棒性。

**10% 合成测试用例。** 用于覆盖边界情况的系统化构造。补充真实数据的盲区。

**10% 幻觉检测。** 这个类别最特别——不是测"该用工具时用对了吗"，而是测"不该用工具时闭嘴了吗"。问模型"人生的意义是什么"——正确答案是不调用任何工具，直接回答。如果模型硬要调一个最像的工具然后用错，那就扣分。

**BFCL 最关键的创新不是题目分类，是评估方式。** V3 开始用"基于状态的评估"——不检查工具调用长得对不对，而是检查调用之后**世界的状态**。你调了 create_file("test.txt")——文件真的创建了吗？你调了 insert_row(data)——数据真的插进去了吗？检查结果，不检查语法。

## 工具 Schema——写给模型看的说明书

有了评估标准，回到工程。每个工具调用的基础是 Schema。

所有厂商的 Schema 本质是一样的：一个名字、一段描述、一个参数的 JSON Schema。

**描述是最被低估的字段。** 模型读它来决定什么时候用这个工具。一个模糊的描述——"获取天气"——会让模型在"该用 get_weather 还是 search_web"之间乱选。一个精准的描述——"获取指定城市的当前天气，返回温度和天气状况。不要用于查询历史数据或预测"——直接排除了歧义。

工程上，写工具描述就是做提示工程。只不过你提示的不是用户，是模型。

## 参数验证——信任零，验证全部

模型给了一个工具调用，带了一组参数。你的本能是直接执行。

**不要。**

模型可能传字符串 "5" 给整数参数。可能传不在枚举列表里的值。可能漏了必需字段。可能把 email 格式写错了。

验证分四步：

1. **类型强制转换。** "5" → 5。明确可转就转。模糊（"five"）就拒绝。
2. **枚举验证。** Schema 说 status 只能是 "open" 或 "closed"，模型传了 "in_progress"→拒绝，并告诉它合法选项是什么。
3. **必需字段。** 缺了就拒绝，告诉模型缺了什么。
4. **格式验证。** 日期、邮件、URL——用真实解析器，不用正则。

关键点：**每一次验证失败都应该以结构化错误字符串的形式返回给模型，包含三个要素——哪里错了、为什么错、正确的格式是什么。** 这样模型才能自我修正。这和上节课 Agent 循环中的"观察格式化器"是同一个原则。

## tool_use_id——并行调用的隐形支柱

现代模型支持在一次响应中同时调用多个工具。"帮我查北京、上海、广州的天气"→三个 get_weather 调用同时发出。

但三个调用的返回顺序是不确定的。可能上海的结果先回来，北京最后回来。你怎么知道哪个结果对应哪个调用？

**tool_use_id。** 每个工具调用生成时带着一个唯一 ID。工具执行完后，结果带着同一个 ID 回来。运行时按 ID 匹配。

这个设计不是锦上添花——是必需的。Anthropic 要求它，OpenAI 要求它，AWS Bedrock 要求它。三家都要求，说明它是工具调用协议的内在需求，不是某家的 API 癖好。

## 沙盒——工具的边界就是安全的边界

每个工具都应该有明确的边界声明：

- 它能读写哪些文件路径？
- 它能访问网络吗？
- 每次调用最多花多少时间？
- 最多占用多少内存？

通用的 `run_shell(anything)` 工具是一个巨大的架构缺陷。它把沙盒的墙拆了——模型可以执行任何东西。具体的 `git_status()` 工具是安全的——它只能做一件事，边界明确。

**工具的安全模型，就是 Agent 的安全模型。** 因为 Agent 能做的所有事情，都是通过工具做的。

## 2026 年的三个开放问题

最后，说点实话。不是所有工具调用的问题都解决了。

**问题一：长链条漂移。** 单轮调用准确率很高。但 20 步的链条呢？40 步呢？每一步的微小偏差积累起来，第 20 步的决策和第 1 步的质量已经不在一个量级上。这不是模型的问题——是任何长链条决策系统的固有问题。

**问题二：动态决策。** 当前大多数工具调用是"给你一个查询和一组工具，选一个"。但真实场景是：先调一个工具，看到结果，**基于结果决定下一个调什么**。这个"看到→判断→选择"的动态环，模型还做不好。

**问题三：记忆丢失。** 在多轮对话中，Agent 需要在十几条消息后仍然记得"用户之前说要查三月份的订单"。上下文窗口很大，但注意力不均衡——这是第 09 课上下文工程的"迷失在中间"效应的直接后果。

这三个问题加在一起，就是 2026 年 Agent 工具使用的前线。BFCL 的五类测试正是为这些前线设计的。

## 总结：Agent 架构师的工具使用检查清单

1. ✅ **Schema 质量**：每个工具的 description 是否清楚告诉模型"什么时候用、什么时候不用"？
2. ✅ **参数验证**：类型转换、枚举、必需字段、格式验证——四步都覆盖了吗？
3. ✅ **错误即观察**：验证失败返回结构化错误，模型可以自我修正——而不是崩溃？
4. ✅ **沙盒边界**：每个工具明确声明了读写范围、网络、超时、内存上限吗？
5. ✅ **tool_use_id**：并行调用有唯一关联 ID 吗？
6. ✅ **幻觉检测**：Agent 有没有能力说"我不需要任何工具，直接回答"？
7. ✅ **评估**：发版前用 BFCL 跑过了吗？知道自己的 Agent 在哪些维度上会失败吗？

记住一件事：**2023 年 Toolformer 教会了模型"该不该用工具"。2026 年 BFCL 告诉我们"用得好不好"。单轮调用已是闭环。记忆、动态决策、长链条——这才是 2026 年 Agent 工程师真正的前线。**
