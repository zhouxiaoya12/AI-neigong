# 精炼循环（Self-Refine / CRITIC）

> 给定任务、迭代预算和可用的验证器类型（工具真值验证或仅自我评估），为评估器-优化器循环输出提示词和停止策略。

**技能名称：** refine-loop
**版本：** 1.0.0
**阶段：** Phase 14 · 第 05 课
**标签：** self-refine, critic, evaluator-optimizer, guardrails, iteration

---

## 功能说明

给定一个任务、迭代预算和可用的验证器（工具真值验证或仅自我评估），为评估器-优化器循环输出提示词和停止策略。

输出以下内容：

1. **生成器提示词（Generator prompt）。** 用于首次输出的确定性生成器。明确陈述任务、输出格式和约束条件。
2. **评估器/验证器提示词（Evaluator/verifier prompt）。** 如果有工具可用（搜索、代码运行、测试、计算器、类型检查），指定如何调用它们以及如何生成结构化评审（JSON 格式，包含：`pass/fail`、`violations[]`、`suggested_fixes[]`）。如果只有自我评估可用，必须显式标注 Self-Refine 的"橡皮图章"风险，并使用结构上不同的提示风格（例如，对抗式——"至少找出一处缺陷"）。
3. **精炼器提示词（Refiner prompt）。** 必须引用先前的输出和评审（历史记录）。声明"不得重复先前迭代中已标记的失败模式"是强制要求。
4. **停止策略（Stop policy）。** 取以下条件的逻辑或：验证器通过 OR（自我评估通过 AND 迭代次数 ≥ 2）OR 迭代次数 ≥ 最大迭代次数。绝不允许单条件停止。
5. **可观测性钩子（Observability hooks）。** 按第 23 课的要求，将每次迭代记录为 OpenTelemetry GenAI span（`evaluate`、`optimize`），使完整的精炼轨迹可审计。

## 硬拒绝规则

- **生成器和评审器使用相同提示词。** 橡皮图章风险——模型会同意自己的输出。
- **无迭代上限。** 无限精炼循环会烧掉令牌；默认上限始终为 4。
- **要求自由文本散文反馈的验证器提示词。** 仅允许结构化 JSON——`pass/fail` 加上逐条列出的违规项。
- **在精炼器提示词中丢弃历史记录。** 论文表明，没有历史记录，质量会崩溃。

## 拒绝规则

- 如果任务没有验证器且无法构建验证器，**拒绝 CRITIC**，并说明 Self-Refine 是可用的较弱选项——警告用户橡皮图章风险。
- 如果 `max_iterations ≥ 10`，**拒绝**，并建议重新设计任务。超过 3-4 轮还未收敛的精炼通常意味着生成器提示词本身有误。
- 如果验证器调用破坏性工具（shell、git write），**拒绝**，并要求沙箱边界（第 09 课）。

## 输出格式

单个配置块，包含所有提示词、停止策略和工具列表，外加根据部署目标指向的"下一步阅读"说明：

- 第 16 课（OpenAI Agents SDK 护栏）
- 第 12 课（Anthropic 评估器-优化器）
- 第 30 课（评估驱动的 Agent 开发）

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份极其精准的操作规范——它不是教学文档，而是**技能规范**（skill specification）。它把 Self-Refine（Madaan et al., 2023）和 CRITIC 的论文级概念压缩为一套可执行的工程约束：五个输出项、四条硬拒绝规则、三条拒绝规则。文档的价值不在于"讲了什么新东西"，而在于**把容易犯的错固化为不可绕过的检查点**。目标受众是正在搭建 Agent 评估流水线的工程师，不适合初学者——它假设读者已经理解 Self-Refine 循环的基本概念。

### 二、知识结构梳理

**认知基础层：** Self-Refine（第 05 课核心内容）——LLM 自己生成、自己评估、自己修正的循环。CRITIC 是该模式的扩展，将"自我评估"替换为"外部工具验证"。学生需先理解"为什么 LLM 会同意自己"才能理解硬拒绝规则的设计意图。

**工程规范层：** 五个输出项构成一个完整的评估器-优化器流水线——① 生成（首次输出）、② 验证（工具或自我评估）、③ 精炼（带历史约束的修正）、④ 停止（三条件逻辑或）、⑤ 可观测（OpenTelemetry span）。加上四条不可绕过的硬拒绝规则，这是一个可以直接翻译为代码的接口规范。

**安全边界层：** 三条拒绝规则是该文档最被低估的部分。它们不是"建议"——是"拒绝服务"的触发条件。无验证器 → 降级警告、迭代过多 → 拒绝并建议重设计、破坏性工具 → 拒绝并要求沙箱。这是 Agent 安全工程的一个微观实践。

### 三、核心洞察（备课时的关键理解）

1. **"生成器和评审器用不同提示词"是防止橡皮图章的最小化设计。** LLM 在任何评估任务中的默认倾向是和给定的内容保持一致。如果你给同一个提示模板让 LLM 既生成又评审，它几乎一定会说"这很好"。对抗式要求（"至少找一处缺陷"）是低成本打破这个默认倾向的方法。

2. **停止策略必须三条件逻辑或，单条件是事故的温床。** 如果只靠"验证器通过"来停止，一个永远报错的验证器会让循环无限跑。如果只靠"迭代次数到上限"，你会在已经正确的输出上继续浪费令牌。三条件逻辑或同时保护了安全性和经济性。

3. **结构化 JSON 反馈不是可选的——它是让精炼器可工作的前提。** 自由文本反馈让 LLM 必须重新解析"到底哪里错了"——这引入了二次歧义。`{pass: false, violations: [...], suggested_fixes: [...]}` 这种格式让精炼器的输入是机器可读的，不给歧义留空间。

4. **在精炼器提示词中丢弃历史记录 = 质量崩溃。** Madaan et al. 的消融实验已经证明了这一点：不带历史记录的 refine 不比单次生成好多少。因为 LLM 不知道上次犯了什么错，它会犯同样的错或者引入新错，形成随机游走。

5. **迭代上限 4 是经验法则，不是教条。** 文档说"默认 cap 4"，但真正的洞察在拒绝规则里：如果 ≥ 10 次迭代还没收敛，问题不在精炼循环，在生成器提示词本身——它可能没说清楚任务目标。

6. **拒绝规则是 Agent 安全工程的微观范例。** 三条拒绝规则覆盖了三类典型风险：验证缺失（质量风险）、设计缺陷（架构风险）、破坏性工具（安全风险）。每条规则都是"检测 → 拒绝 → 给出替代方案"的完整模式，而非粗暴拒绝。

7. **OpenTelemetry span 不是可选的附加项——是审计能力的基础。** 每个 `evaluate` 和 `optimize` span 让你能回溯"第 3 轮为什么还失败了"。没有这条，精炼循环是个黑盒。

### 四、教学建议

1. **先让学生写一个没有硬拒绝规则的评估器-优化器，然后让它跑坏。** 演示"生成器 = 评审器提示词"导致的橡皮图章、演示无上限循环烧掉的令牌数量。先感受痛苦，再引入规则，记得最牢。

2. **用 Voyager（第 09 课）作为 Self-Refine 的示例。** Voyager 的迭代精炼正是这个规范的一个真实案例：生成器 = 写技能、验证器 = 环境执行 + 栈追踪、精炼器 = 带历史重写。跨课串联。

3. **停止策略的三条件逻辑或，画一个真值表。** 让学生看到四种停止路径的对比，理解"为什么一个条件不够"。这个设计决策如果不讲清楚，他们会把停止策略简化为 `while not pass: retry`。

4. **结构化反馈 vs. 自由文本反馈——用一个对比实验。** 同一个任务，分别用自由文本和 JSON 格式的验证器输出，看精炼器在哪种条件下的收敛速度更快。实验比说教有用。

5. **三条拒绝规则适合用"场景演练"教学。** 给出三个场景（无验证器的创意任务、已迭代 12 次的 bug 修复、需要 `rm -rf` 的部署脚本），让学生判断应该 CRITIC 还是 Self-Refine，接受还是拒绝，以及给出什么替代方案。

6. **OpenTelemetry span 的部分，现场展示一个带 eval/optimize span 的 trace 截图。** 可视化胜过文字。

7. **最后做一个"规范翻译"练习。** 把这份 SKILL.md 的约束翻译为 Python 代码（一个 `RefineLoop` 类），包括硬拒绝检查和停止策略。让学生把规范变成可运行的工程制品。

### 五、值得补充的内容

1. **验证器返回的"置信度"字段。** 当前规范只有 `pass/fail`，但实际验证器（尤其是自我评估）有置信度问题。建议在 JSON schema 中增加 `confidence: 0.0-1.0`。

2. **精炼器对历史记录的长度管理。** 当迭代到第 4 轮时，历史记录可能超出上下文窗口。需要截断策略——是只保留最近 N 轮，还是用摘要压缩早期轮次？

3. **多验证器融合。** 当一个任务同时有代码运行验证和 LLM 语义验证时，两个验证器的结论可能冲突——谁说了算？这需要一个"仲裁器"层。

4. **与 Voyager 三元反馈的对应关系。** 第 09 课的 Voyager 用 `success / error / self-verification failure` 三元信号。这套规范用 `pass/fail + violations[]` 二元——可以讨论哪种更适合不同的 Agent 架构。

### 六、一句话总结

**好的精炼循环不是"写→跑→改"的无限重试——它的质量边界由三样东西定义：不同化生成与评估、不丢弃历史、不省略停止条件。**

# 🎓 Agent 架构课：精炼循环——为什么"让它再多试几次"通常没用

同学们好。我是你们的 Agent 架构老师。

今天这节课讲一个你们一定做过的事：**让 Agent 改一版、再改一版、再改一版——然后结果还是不对。**

你以为多试几次就行。但有的时候，多试几次只是让 Agent 多犯几次同样的错。

我们今天把这个"改"的过程拆开看——它什么时候工作，什么时候不工作，以及怎么让它持续工作。

## 先看一个你天天在犯的错

你写了一个提示词，让 Agent 生成一段代码。

```python
# 生成器提示词：
"用 Python 写一个函数，输入三个数，返回中位数。"
```

Agent 输出了：

```python
def median(a, b, c):
    return sorted([a, b, c])[1]
```

现在你想验证一下。你又发了一个提示词：

```python
# 验证器提示词（坏的）：
"检查上面这个函数的正确性，给出你的评价。"
```

Agent 回复："这个函数看起来逻辑正确，使用了排序取中间值的方法，可以正确处理三个数的中位数计算。"

你满意了，继续下一个任务。

但你知道刚才发生了什么吗？**你让同一个 LLM 既写代码又检查自己的代码。** 这叫**橡皮图章**（rubber-stamp）。LLM 在任何评估任务中的默认倾向是和给出的内容保持一致。它不是在"检查"——它在"确认"。

这就是 Self-Refine 和 CRITIC 要解决的核心问题。今天我们不谈定义，我们谈**怎么设计一个不会自己骗自己的精炼循环。**

## 两条路，两种验证

你有一个任务——生成一段代码或一段文本。你觉得第一版可能不够好，想让它改进。这个"改"的过程怎么跑？

业界有两条路。

**第一条路：Self-Refine。** LLM 自己生成，自己检查，自己修正。你只给它文本，不给它任何外部工具。优点是零依赖——不需要测试框架、不需要编译器、不需要搜索引擎。缺点是它可能自己骗自己。让它"找至少三个问题"，它可能找三个不存在的"问题"来交差。

**第二条路：CRITIC。** 你把验证器从 LLM 的手里拿过来，换成**真实工具**——跑一下代码看有没有异常、搜一下资料看事实对不对、用计算器算一下数学结果。LLM 只负责生成和精炼。验证由外部工具完成。

这两条路的选择不是小事。它直接决定了你的精炼循环的可靠性上限。

## 设计一个不坑人的精炼循环：五个组件

好，我们来搭一个正经的评估器-优化器循环。不是"让 LLM 多试几次"——是一个有结构的、可验证的、可停止的工程系统。

### 组件一：生成器（Generator）——说清楚你要什么

生成器的提示词必须做到三件事：
- **明确任务。** 不要说"写一个好函数"，要说"写一个函数，输入三个数，返回中位数，要求处理浮点数和整数，不允许使用内置的 statistics 模块"。
- **指定输出格式。** "返回一个 Python 函数定义"——不是一个解释，不是一段带说明的代码块。
- **列出硬约束。** "边界值处理：三个数相等、两个数相等、包含负数——都必须输出正确结果。"

生成器的质量决定精炼的上限。如果生成器的提示词是模糊的，精炼 20 轮也救不回来。

### 组件二：验证器（Evaluator/Verifier）——验证不能用同一张嘴

这是整个循环里最容易出错的地方。

如果你有工具——**用工具。** 代码任务跑测试，数学任务用计算器，事实任务上网搜。工具验证 = 真值信号。不需争论。

如果你没有工具——**改变提示词的结构。** 不要用"检查上面的输出是否正确"这种提示词。用对抗式的："假设上面这段代码是别人写的，你必须作为一个严苛的代码审查者，至少找出一个逻辑错误或边界情况处理遗漏。如果确实找不到，解释为什么没有错误，并证明你的推理。"

关键：**验证器的提示词不能和生成器是同一套模板。** 如果是——橡皮图章。这是第一条硬拒绝规则。

验证器的输出必须是结构化的 JSON：

```json
{
  "pass": false,
  "violations": [
    {
      "severity": "high",
      "description": "边界情况未处理：三个输入相等时可能返回错误类型",
      "location": "line 2, sorted() 返回类型不确定"
    }
  ],
  "suggested_fixes": [
    "在返回前添加类型转换，确保输出是数字而非列表元素"
  ]
}
```

不是"我觉得这个函数还不错"——**结构化数据。逐条列出问题。逐条给出修复建议。**

### 组件三：精炼器（Refiner）——你必须有记性

这是最容易被偷工减料的地方。很多人把第二轮的提示词写成"再改一下"——不加历史，不加约束。

精炼器提示词必须包含三样东西：

1. **上一轮的输出。** "你上一轮生成的代码是：{previous_output}"
2. **验证器的反馈。** "验证器发现以下问题：{critique_json}"
3. **一个强制约束。** "修改代码以修复上述所有问题。**不得引入已在上一次迭代被指出为错误的新模式。**"

没有第三条，Agent 可能在修复 A 问题时引入 B 问题。没有前两条，你就在让盲人改画。

论文数据很清楚：**不带历史记录的精炼，质量不比单次生成好多少。**

### 组件四：停止策略（Stop Policy）——知道什么时候停

这是你以前可能没想过的事情：**精炼循环怎么停下来？**

坏的停止策略：`while not_verified: retry()`——验证器永远不通过怎么办？死循环。

好的停止策略：**三个条件，逻辑或，任意一个成立就停。**

```
停止条件 = 
  (验证器通过) 
  OR 
  (自我评估通过 AND 迭代次数 ≥ 2)
  OR 
  (迭代次数 ≥ max_iterations)
```

第一个条件：最容易理解——外部工具说通过了，就通过了。

第二个条件：防橡皮图章。如果你只有自我评估，不能第一轮自评通过就停。至少跑两轮，让精炼器有机会在第一轮自评的基础上再改进一轮。

第三个条件：防无限烧钱。默认 `max_iterations = 4`。四轮之后还没过——停，问题不在循环，在生成器提示词。

### 组件五：可观测性（Observability）——别在黑盒里精炼

每次迭代记两条 OpenTelemetry span：
- `evaluate` span：谁验证的、用了什么工具、结果是什么
- `optimize` span：精炼器输出了什么、和上一轮相比改了什么

这不是"以后再说"的东西。**没有 span，你的精炼循环是个黑盒。** 一个月后你看到一条失败的输出，你完全不知道它在第 3 轮到底发生了什么。审计能力是生产 Agent 的基本功。

## 三条防线：这个系统在什么时候该拒绝运行

好设计不仅有"做什么"——还有"什么时候不做"。

### 防线一：无验证器 → 拒绝 CRITIC，降级为 Self-Refine + 警告

你有一个创意写作任务——没有测试能跑，没有事实能查。你问我："能用 CRITIC 吗？"

不能。CRITIC 的前提是有外部工具做真值验证。没有验证器的时候，你只能用 Self-Refine。但 Self-Refine 有橡皮图章风险——你必须知道并接受这一点。系统不会默默从 CRITIC 降级——它会**明确告诉你降级了，并警告风险。**

### 防线二：迭代过多 → 拒绝，建议重新设计任务

你说："这个任务特别难，给我 15 轮精炼预算。"

拒绝。不是因为系统跑不动——是**生成器提示词有问题。** 在这个领域，3 到 4 轮精炼应该覆盖绝大多数改正。如果 4 轮还改不对，说明第一轮生成的质量太差——重新设计生成器提示词，比增加迭代次数省钱得多。

### 防线三：破坏性工具 → 拒绝，要求沙箱

你的验证器要跑 `rm -rf /` 或者 `git push --force`？

拒绝。验证器可以执行代码，但不能触碰文件系统、不能改 git 历史、不能调用外部 API——除非在沙箱里。第 09 课的沙箱边界是这里的硬前提。

这三条防线不是"建议"——是**拒绝服务**。触发条件满足，系统必须停下来告诉你为什么，而不是默默跑下去。

## 把这个和 Voyager 串起来

你还记得第 09 课的 Voyager 吗？它的迭代精炼就是这套循环的一个生产实例：

- **生成器 =** Agent 写出一条技能函数
- **验证器 =** 技能在 Minecraft 环境中执行——真值反馈。失败时还带栈追踪
- **精炼器 =** Agent 拿到错误信息，加上历史记录，重写技能
- **停止策略 =** 技能执行成功 OR 达到最大重试次数
- **三元反馈（success/error/self-verification failure）=** 比二元的 `pass/fail` 多了一层安全检查

你看，这不是两套东西——Voyager 就是这个规范在 Minecraft 里的实现。这个规范是 Voyager 精炼循环的形式化表达。

## 什么时候这个模式没用

1. **没有验证器的任务。** 如果你的验证来源只有 LLM 的嘴，精炼循环的上限很低。承认它，别假装它是 CRITIC。
2. **一次性的查询。** "今天天气怎么样？"——不需要精炼循环。不要为不需要迭代的东西建迭代。
3. **有硬截止时间的任务。** 精炼循环的时间是不可预测的。如果你的响应必须在 500ms 内，精炼循环不适合。

## 结课清单

如果你要给你的 Agent 加一个精炼循环，这是你的检查清单：

1. 生成器提示词和验证器提示词用不同的模板——防止橡皮图章。
2. 验证器输出结构化 JSON——`pass/fail` + `violations[]` + `suggested_fixes[]`。
3. 精炼器必须收到上一轮的输出、验证结果、和"不要重复已标记的错误"的硬约束。
4. 停止策略三条件：验证通过 OR（自评通过 AND ≥ 2 轮）OR 达到上限。
5. 默认最大迭代 = 4。更多意味着生成器提示词有问题。
6. 每次迭代记录 OpenTelemetry span（evaluate + optimize）。
7. 无验证器 → 降级 Self-Refine + 警告。≥ 10 轮 → 拒绝。破坏性工具 → 拒绝要沙箱。

最后一句话：

**精炼循环的价值不在"多试几次"——在每一次精炼都必须比上一次更有信息。生成不是评估，评估不是精炼，精炼不是遗忘。把这三件事混在一起，你就只有一个会重复犯错的黑盒。**
