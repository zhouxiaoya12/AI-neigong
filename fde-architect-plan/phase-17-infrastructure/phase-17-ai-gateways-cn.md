# AI 网关 — LiteLLM、Portkey、Kong AI Gateway、Bifrost

> 网关位于你的应用和模型提供商之间。核心功能包括：提供商路由、故障转移、重试、速率限制、密钥引用、可观测性、防护栏。2026 年市场格局：**LiteLLM** 是 MIT 开源的，支持 100+ 提供商，兼容 OpenAI 接口，但在约 2000 RPS 时会崩溃（8 GB 内存，公开基准测试中出现级联故障）；最适合 Python、<500 RPS、开发/原型场景。**Portkey** 定位为控制平面（防护栏、PII 脱敏、越狱检测、审计追踪），2026 年 3 月以 Apache 2.0 协议开源，延迟开销 20-40 毫秒，生产级 $49/月。**Kong AI Gateway** 基于 Kong Gateway 构建——Kong 在相同 12 CPU 上的基准测试：比 Portkey 快 228%，比 LiteLLM 快 859%；定价 $100/模型/月（Plus 套餐最多 5 个模型）；如果你已经在用 Kong，这是企业级选择。**Bifrost**（Maxim AI）——自动重试并可配置退避策略，OpenAI 429 时故障转移到 Anthropic。**Cloudflare / Vercel AI Gateway**——托管式、零运维、基础重试。数据驻留需求驱动自托管决策；Portkey 和 Kong 以 OSS + 可选托管的方式处于中间地带。

**前置课程：** 第 17 阶段 · 01（托管 LLM 平台）、第 17 阶段 · 16（模型路由）

## 学习目标

- 列举六大核心网关功能（路由、故障转移、重试、速率限制、密钥、可观测性、防护栏）。
- 将四个 2026 年网关（LiteLLM、Portkey、Kong AI、Bifrost）映射到其规模上限和使用场景。
- 引用 Kong 基准测试（比 Portkey 快 228%，比 LiteLLM 快 859%）并解释为什么这对 >500 RPS 至关重要。
- 在数据驻留和运维预算的约束下，选择自托管或托管方案。

## 问题

你的产品调用 OpenAI、Anthropic 和一个自托管的 Llama。每个提供商有不同的 SDK、错误模型、速率限制和认证方案。你希望实现故障转移（如果 OpenAI 返回 429，尝试 Anthropic）、统一的凭证存储、统一的可观测性，以及按租户的速率限制。

在应用层重新发明这些会将每个服务与每个提供商耦合。网关层将其整合到一个进程中，提供一个 API（通常兼容 OpenAI 接口），然后分发到各提供商。

## 概念

### 六大核心功能

1. **提供商路由** —— OpenAI、Anthropic、Gemini、自托管等，统一在一个 API 后面。
2. **故障转移** —— 遇到 429、5xx 或质量失败时，在其他地方重试。
3. **重试** —— 指数退避，限制尝试次数。
4. **速率限制** —— 按租户、按密钥、按模型。
5. **密钥引用** —— 在运行时从密钥库中拉取凭证（绝不出现在应用中）。
6. **可观测性** —— OTel + GenAI 属性（第 17 阶段 · 13）+ 成本归属。
7. **防护栏** —— PII 脱敏、越狱检测、允许主题过滤器。

### LiteLLM —— MIT 开源，Python

- 100+ 提供商，兼容 OpenAI 接口，路由器配置，故障转移，基础可观测性。
- 在 Kong 的基准测试中约 2000 RPS 崩溃；8 GB 内存占用，持续负载下级联故障。
- 最适合：Python 应用，<500 RPS，开发/预发布网关，实验性路由。
- 成本：OSS 免费；云免费层存在。

### Portkey —— 控制平面定位

- 2026 年 3 月起 Apache 2.0 开源。防护栏、PII 脱敏、越狱检测、审计追踪。
- 每次请求延迟开销 20-40 毫秒。
- 生产级 $49/月，含保留期 + SLA。
- 最适合：需要防护栏 + 可观测性捆绑的受监管行业。

### Kong AI Gateway —— 规模化之选

- 基于 Kong Gateway 构建（成熟的 API 网关产品，Lua + OpenResty）。
- Kong 在 12-CPU 等效环境下的基准测试：比 Portkey 快 228%，比 LiteLLM 快 859%。
- 定价：$100/模型/月，Plus 套餐最多 5 个模型。
- 最适合：已在使用 Kong；>1000 RPS；愿意购买许可证。

### Bifrost（Maxim AI）

- 自动重试并可配置退避策略。
- 在 OpenAI 429 时故障转移到 Anthropic 是一个经典配方。
- 较新的进入者；商业产品。

### Cloudflare AI Gateway / Vercel AI Gateway

- 托管式，零运维。基础重试和可观测性。
- 最适合：Cloudflare/Vercel 上的边缘服务 JavaScript 应用。
- 在防护栏和速率限制方面较 Kong/Portkey 有限。

### 自托管 vs 托管

数据驻留是决定性因素。医疗和金融默认自托管（LiteLLM 或 Portkey OSS 或 Kong）。消费者产品默认托管（Cloudflare AI Gateway）或中间层（Portkey 托管）。混合方案：受监管租户用自托管，其他用托管。

### 延迟预算

- LiteLLM：典型开销 5-15 毫秒。
- Portkey：开销 20-40 毫秒。
- Kong：开销 3-8 毫秒。
- Cloudflare/Vercel：开销 1-3 毫秒（边缘优势）。

网关延迟直接叠加到首 token 时间（TTFT）。对于 TTFT P99 < 100 毫秒的 SLA，用 Kong 或 Cloudflare。对于 P99 < 500 毫秒，任意均可。

### 速率限制语义很重要

简单的令牌桶在中等规模下可行。多租户需要滑动窗口 + 突发容忍 + 按租户分层。LiteLLM 提供令牌桶；Kong 提供滑动窗口；Portkey 提供分层。

### 网关 + 可观测性 + 路由的组合

第 17 阶段 · 13（可观测性）+ 16（模型路由）+ 19（网关）在生产中是同一层。选择一个同时覆盖这三者的工具，或小心地组合它们：大多数 2026 部署将 Helicone（可观测性）或 Portkey（防护栏）与 Kong（规模）结合，分工协作。

### 你应该记住的数字

- LiteLLM：约 2000 RPS 崩溃，8 GB 内存。
- Portkey：20-40 毫秒开销；2026 年 3 月起 Apache 2.0。
- Kong：比 Portkey 快 228%，比 LiteLLM 快 859%。
- Kong 定价：$100/模型/月，Plus 套餐最多 5 个。
- Cloudflare/Vercel：边缘 1-3 毫秒开销。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份高质量的 2026 年 AI 网关技术选型指南。文档以"问题驱动"的方式组织内容——先抛出一个真实痛点（多提供商 SDK 碎片化、故障转移需求、凭证管理混乱），再系统性地介绍六种网关方案及其适用边界。最令人印象深刻的是，它用具体的基准测试数字（Kong 比 Portkey 快 228%，比 LiteLLM 快 859%）代替了模糊的"性能更好"的描述，使得选型决策有据可依。文档还巧妙地将网关与可观测性（第 13 课）、模型路由（第 16 课）横向关联，体现了生产环境中这些组件协同工作的现实。

### 二、知识结构梳理

- **认知基础层**：理解为什么需要网关——多提供商碎片化、故障转移需求、统一的凭证管理和可观测性是网关存在的根本原因。六大核心功能（路由、故障转移、重试、速率限制、密钥引用、可观测性、防护栏）构成了网关的认知框架。
- **工程模式层**：四种网关方案（LiteLLM、Portkey、Kong、Bifrost）各有其规模上限和场景定位。延迟预算（3-40ms 不等）和自托管 vs 托管的决策矩阵是选型的关键约束。速率限制的语义差异（令牌桶 vs 滑动窗口 vs 分层）体现了工程细节的重要性。
- **落地实践层**：代码模拟器让学生体验路由故障转移的实际行为；`skill-gateway-picker.md` 产出物将理论转化为可操作的选型工具。练习设计从具体（计算命中率）到抽象（设计多租户速率限制策略），梯度合理。

### 三、核心洞察

1. **网关延迟直接叠加到 TTFT**——网关不是"免费"的抽象层；每增加一个中间层都会增加延迟。Kong 的 3-8ms 和 LiteLLM 的 5-15ms 开销在低延迟场景下不是可忽略的。为什么重要：TTFT 是用户体验的核心指标，选错网关可能让优化其他环节的努力付诸东流。
2. **数据驻留是自托管决策的"强制函数"**——医疗和金融行业不是因为运维能力强才自托管，而是因为合规要求。这个洞察反转了常见的"技术能力决定架构"的思路。为什么重要：它意味着即使托管方案在技术上更优，对某些客户也必须提供自托管选项。
3. **LiteLLM 的 2000 RPS 崩溃点是一个"隐性墙"**——团队可能在 1500 RPS 时觉得一切正常，1900 RPS 时只是偶尔抖动，到 2100 RPS 时系统完全崩溃。为什么重要：这种非线性故障模式在基准测试中清晰可见，但在实际监控中容易被误读为"偶发问题"。
4. **组合优于单一方案**——2026 年的最佳实践不是"选一个最好的网关"，而是"Helicone（可观测性）+ Kong（规模）+ Portkey（防护栏）"的分工协作。为什么重要：它提醒我们网关市场正在从"一体化"向"专业化组合"演进。
5. **Portkey 的 Apache 2.0 开源是一个信号**——2026 年 3 月的开源决策意味着防护栏功能正在从"差异化竞争"变为"基础设施标配"。为什么重要：它暗示商业网关的价值将从防护栏转向更高层次的治理和合规能力。
6. **Kong 的定价模型（$100/模型/月，最多 5 个）是为企业设计的**——这不是一个小团队会用到的模型；它是为已经有 API 网关采购流程的企业设计的追加销售。为什么重要：定价模型本身就是市场定位的反映，理解定价就是理解目标客户。

### 四、教学建议

1. **用"多提供商 SDK 地狱"作为开场痛点**——让学生先手动写一个调用三个不同提供商的代码，感受 SDK 碎片化、错误处理不一致的痛苦，然后再引入网关概念。亲身体验比任何讲解都有效。
2. **制作一个"网关选型决策树"**——将规模（<500 / 500-1000 / 1000+ RPS）、运维姿态（零运维 / 可运维 / 专业运维）、合规需求（无 / 数据驻留 / HIPAA）作为分支条件，可视化输出推荐方案。
3. **强调基准测试的"基准"是相对基准**——提醒学生 Kong 的基准测试是 Kong 自己做的，不能当作独立第三方评估。教会学生批判性地阅读厂商基准测试。
4. **做一次"网关故障演练"**——在模拟环境中注入错误（429、5xx、超时），让学生观察不同网关配置下的故障转移行为。故障不是理论，是生产常态。
5. **讨论"为什么不是 Nginx/Envoy"**——引导学生思考：通用 API 网关和 AI 专用网关的本质区别是什么？答案在于 LLM 特有的流式响应、token 级计费和模型特定的错误语义。
6. **用"延迟预算"的视角重新审视架构**——如果 SLA 是 TTFT P99 < 100ms，每个组件（网关、推理引擎、网络）的延迟预算分别是多少？这种"倒推"思维训练非常有工程价值。
7. **布置"网关迁移方案"作为综合练习**——给定一个使用 LiteLLM 的团队，RPS 从 300 增长到 1500，写一份迁移到 Kong 的方案，包括风险、回滚策略和监控指标。

### 五、值得补充的内容

1. **网关的流式处理能力差异**——不同网关对 SSE（Server-Sent Events）和流式响应的支持程度不同，这对 TTFT 和用户体验有直接影响，原文未深入讨论。
2. **缓存层与网关的关系**——像 GPTCache 这样的语义缓存在网关层之上或之下的部署位置，以及缓存命中如何影响故障转移逻辑。
3. **多云策略的网关挑战**——当一个团队同时使用 AWS Bedrock、Azure OpenAI 和 GCP Vertex AI 时，网关需要处理的不仅仅是 API 格式转换，还有云特定的认证和网络配置。
4. **开源网关的维护风险和社区健康度**——LiteLLM 和 Portkey OSS 的 commit 频率、issue 响应时间和核心维护者数量，这些"软指标"对长期依赖至关重要。
5. **网关的计费准确性**——网关层的 token 计数是否与提供商的实际计费一致？微小的差异在大规模下可能产生显著的财务影响。

### 六、一句话总结

AI 网关不是"锦上添花"的可选组件，而是生产环境中将"多提供商碎片化"转化为"统一抽象"的关键基础设施——选型时规模数字（2000 RPS 崩溃点）比功能列表更重要，而延迟预算（3-40ms 的差异）可能在低延迟 SLA 下成为决定性因素。

# 🎓 Agent 架构课：AI 网关——凌晨 2 点还在手写 SDK 适配层的工程师没有性生活

同学们好。我是你们的FDE工程老师，今天讲的是 AI 网关。

**你的代码库里有几个文件名叫 `provider_adapter_v3_final_real_fixed_FINAL.py`？** 如果答案不是零，你就需要一个 AI 网关。

AI 网关 = LLM 提供商前面的统一入口。上游是 OpenAI/Anthropic/Gemini/自托管模型，下游是你的 Agent 代码。网关做五件事：① **格式转换**——OpenAI 的 `tool_calls` ↔ Anthropic 的 `tool_use`。② **负载均衡**——三个相同的模型实例，轮询或最少连接。③ **速率限制**——每 API Key 每分钟最多 N 次调用。④ **降级**——主模型超时？自动切到备选模型。⑤ **可观测性**——所有请求的 token 数、延迟、错误率，一个地方看。

2026 年的选择：开源的有 LiteLLM、Portkey，云的有 OpenAI SDK Router、Anthropic 的网关方案。Phase 13 的函数调用转换器就是网关的最小骨架。

## 结课清单

1. **AI 网关 = 提供商差异不是你核心代码的事。** 翻译层独立。
2. **负载均衡 + 降级 = 可用性。** 一个模型挂了不影响整体。
3. **转换器模式推广 = 网关。** Phase 13 函数调用转换器的工程化版本。

最后一句话：**不要在你的核心 Agent 循环里写 if-anthropic-elif-openai。把它推到边缘——那叫网关。把你解放出来写真正重要的代码。**

# 💼 从业者故事：AI 网关——凌晨 2 点还在手写 SDK 适配层的工程师没有性生活

去年接手一个旧项目的时候，我在代码库里发现了一个文件叫 `provider_adapter_v3_final_real.py`。旁边还有 `provider_adapter_v3_final_real_fixed.py` 和 `provider_adapter_v3_final_real_fixed_FINAL.py`。

这三个文件加起来 1400 行，做的事情就是把 OpenAI、Anthropic 和一个自托管的 Llama 的 SDK 调用统一成一个接口。1400 行的 if-else，每一行都在说"我们当初没有用网关"。

每个提供商的错误码不一样——OpenAI 返回 429 是 rate limit，Anthropic 返回 529 是 overloaded，自托管 Llama 偶尔直接给你一个空响应。每个提供商的认证方式不一样——OpenAI 用 Bearer token，Anthropic 用 x-api-key，自托管看心情。你如果想做故障转移——"OpenAI 挂了自动切 Anthropic"——你就要在 1400 行 if-else 里面再加 200 行 try-catch。

这就是网关存在的原因。不是 v2 要加的功能，是 v0.1 就该有的基础设施。你让每个服务自己去对接三个提供商的 SDK，等于让每个服务都重写一遍 1400 行的适配层。这叫"人肉的 API 网关"，唯一的区别是没有人帮你排 bug。

## LiteLLM：美好的 MIT 协议，直到 RPS 超过 2000

LiteLLM 是一个我真心喜欢的产品。MIT 开源，兼容 100+ 提供商，Python 原生，`pip install litellm` 就能跑。我们在开发环境用了半年，一切都很美好。

然后流量涨了。从 300 RPS 涨到 800，再到 1500。1500 的时候偶尔有小抖动，我们没在意。到 2000 的时候，整个网关进程像被一拳打晕了——级联故障，一个请求卡住拖死后面所有请求，CPU 飙到 100%，内存从 2GB 涨到 7.8GB，然后 OOM Kill。重启。再 OOM。再重启。

Kong 那个基准测试说 LiteLLM 在约 2000 RPS 时崩溃——我可以作证，这是真的。不是一个"可能会慢一点"的问题，是"突然死亡"问题。如果你在选 LiteLLM，记住这个数字：2000。低于 500 你是安全的，500-1500 你要开始想退路，超过 1500 你应该已经在迁移了。

## 四个网关，四个完全不同的世界

Portkey 的定位是"控制平面"——防护栏、PII 脱敏、越狱检测、审计追踪。2026 年 3 月开源成 Apache 2.0，这是个大新闻，说明防护栏功能正在从商业溢价变成基础设施标配。20-40ms 的延迟开销在大多数场景下不是问题——但如果你的 TTFT P99 SLA 是 100ms，这 20-40ms 就是不可接受的税。

Kong AI Gateway 基于 Kong Gateway——Kong 本身就是一个带 Lua 和 OpenResty 的成熟 API 网关产品。基准测试说比 Portkey 快 228%，比 LiteLLM 快 859%，这个数据是 Kong 自己测的，不要太信，但方向是对的——Kong 确实是在规模化场景下性能最好的。定价 $100/模型/月，Plus 最多 5 个模型——这个定价模型不是给小团队的，是给已经有 API 网关采购预算的企业。

Cloudflare AI Gateway 是托管式的，零运维。延迟开销 1-3ms，因为跑在边缘网络上离用户近。但防护栏和速率限制功能比 Kong/Portkey 弱很多。适合已经在 Cloudflare/Vercel 生态里的 JavaScript 前端应用。

选网关不是选"最好的那个"，是选"最适合你现在规模和未来 12 个月规模的这个"。如果你的 RPS 还不到 500，LiteLLM 就够了，不需要为"未来可能需要"的规模提前购买复杂性。如果你已经在用 Kong 做 API 网关，加 AI Gateway 插件是零迁移成本。如果你在医疗或金融行业，数据驻留需求会直接把你推向自托管方案。

## 组合优于单体：2026 年的真实生产架构

我在实际部署中看到的最合理的架构是分层组合的：Kong 在边缘做速率限制和路由（性能最好），Portkey 在中间做防护栏和 PII 脱敏（合规能力最强），Helicone 做可观测性和成本归因（仪表板最好看）。三个工具各干各的，用 OpenTelemetry 上下文传播串起来。

这不是过度工程化——是每个领域有最好的工具，你要的不是一个"万能网关"（不存在），而是一个能串联的工具链。你不需要一次选对所有的，可以从一个开始，按需叠加。我的起步套餐：LiteLLM（路由）+ Helicone（可观测性）。上了规模之后叠加：Kong（网关层）+ Portkey OSS（防护栏）。

## 延迟预算：网关不是你唯一加延迟的地方

网关层增加的延迟直接叠加到 TTFT 上。如果你的 SLA 是 TTFT P99 ≤ 200ms，而网关加了 40ms，你就只剩下 160ms 给推理引擎和网络了。这就是为什么"延迟预算"这个概念很重要——你要把 200ms 像分切蛋糕一样分给每一个组件：网络传输 30ms，网关处理 10ms，排队 50ms，预填充 110ms。任何一个组件超支，总 SLA 就破。

Kong 的 3-8ms 和 Portkey 的 20-40ms 在宽松场景下无所谓，在严格 SLA 下就是生与死的差别。如果你的产品是代码实时补全（E2E < 1s 是底线），选 Kong 或 Cloudflare。如果是客服聊天机器人（用户可以接受 3-5s E2E），选什么网关都行。

**AI 网关是你 LLM 基础设施里的"统一入口层"——没有它，你的架构就是每个服务跟每个提供商单独领证结婚，每次离婚（提供商挂了）都要重新谈一次。有它，就是你跟网关结一次婚，网关帮你去搞外交。省下的不是代码行数，是你凌晨 2 点的睡眠。**
