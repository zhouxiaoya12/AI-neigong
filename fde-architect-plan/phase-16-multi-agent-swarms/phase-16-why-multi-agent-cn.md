# 为什么需要多智能体？

> 一个智能体撞墙了。聪明之举不是更大的智能体——而是更多的智能体。

**类型：** 学习
**语言：** TypeScript
**前置课程：** 第14阶段（智能体工程）
**时间：** 约60分钟

## 学习目标

- 识别单智能体上限（上下文溢出、混合技能、顺序瓶颈），并解释何时将任务拆分为多个智能体是正确的举措
- 比较编排模式（流水线、并行扇出、监管者、分层式）并为给定任务结构选择正确的模式
- 设计具有清晰角色边界、共享状态和通信合约的多智能体系统
- 分析多智能体复杂性的权衡（延迟、成本、调试难度）与单智能体简单性的对比

## 问题

你在第14阶段构建了一个单智能体。它能工作。它可以读取文件、运行命令、调用 API 并对结果进行推理。然后你把它指向一个真实的代码库：200 个文件、三种语言、依赖基础设施的测试，以及在编写代码前研究外部 API 的要求。

智能体卡住了。不是因为大语言模型笨，而是因为任务超出了单智能体循环能处理的范围。上下文窗口被文件内容填满。智能体忘记了 40 次工具调用前读取的内容。它试图同时成为研究员、编码员和审查员，但三件事都做得不好。

这就是单智能体上限。每当任务需要以下内容时，你就会碰到它：

- **超出单窗口容量的上下文**——读取 50 个文件会超过 200k 令牌
- **不同阶段需要不同的专业技能**——研究需要的提示与代码生成不同
- **可以并行处理的工作**——可以同时读取三个文件，为什么要顺序读？

## 概念

### 单智能体上限

一个单智能体是一个循环、一个上下文窗口、一个系统提示。想象一下：

```
┌─────────────────────────────────────────┐
│            单个智能体                    │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │         上下文窗口                 │  │
│  │                                   │  │
│  │  研究笔记                         │  │
│  │  + 代码文件                       │  │
│  │  + 测试输出                       │  │
│  │  + 审查反馈                       │  │
│  │  + API 文档                       │  │
│  │  + ...                            │  │
│  │                                   │  │
│  │  ██████████████████████ 已满 ███  │  │
│  └───────────────────────────────────┘  │
│                                         │
│  一个系统提示试图覆盖                   │
│  研究 + 编码 + 审查 + 测试              │
│                                         │
│  结果：每件事都做得很平庸               │
└─────────────────────────────────────────┘
```

三件事会出问题：

1. **上下文饱和**——工具结果不断堆积。到第 30 轮，智能体已经消费了 150k 令牌的文件内容、命令输出和先前的推理。第 5 轮的关键细节丢失了。

2. **角色混淆**——一个写着"你是研究员、编码员、审查员和测试员"的系统提示，产生的是一个半研究、半编码、从没完成审查的智能体。

3. **顺序瓶颈**——智能体读取文件 A，然后文件 B，然后文件 C。三次串行的大语言模型调用。三次串行的工具执行。没有并行性。

### 多智能体解决方案

拆分工作。给每个智能体一个工作、一个上下文窗口和一个为该工作调优的系统提示：

```
┌──────────────────────────────────────────────────────────┐
│                    编排器                                │
│                                                          │
│  "构建一个用户管理的 REST API"                           │
│                                                          │
│         ┌──────────┬──────────┬──────────┐               │
│         │          │          │          │               │
│         ▼          ▼          ▼          ▼               │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│   │ 研究员   │ │ 编码员   │ │ 审查员   │ │ 测试员   │  │
│   │          │ │          │ │          │ │          │  │
│   │ 阅读     │ │ 基于     │ │ 检查     │ │ 运行     │  │
│   │ 文档，   │ │ 研究     │ │ 代码     │ │ 测试，   │  │
│   │ 发现     │ │ + 规格   │ │ 质量，   │ │ 报告     │  │
│   │ 模式     │ │ 编写代码 │ │ 发现     │ │ 结果     │  │
│   │          │ │          │ │ 缺陷     │ │          │  │
│   └─────┬────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘  │
│         │           │            │             │         │
│         └───────────┴────────────┴─────────────┘         │
│                          │                               │
│                     合并结果                             │
└──────────────────────────────────────────────────────────┘
```

每个智能体都有：
- 一个聚焦的系统提示（"你是一个代码审查员。你唯一的工作是发现 bug。"）
- 自己独立的上下文窗口（不被其他智能体的工作污染）
- 清晰的输入/输出合约（接收研究笔记，输出代码）

### 做到这一点的真实系统

**Claude Code 子智能体**——当 Claude Code 用 `Task` 生成子智能体时，它创建了一个具有限定任务的子智能体。父智能体保持上下文干净。子智能体做聚焦的工作并返回摘要。

**Devin**——运行一个规划器智能体、一个编码员智能体和一个浏览器智能体。规划器将工作分解为步骤。编码员编写代码。浏览器研究文档。每个都有独立的上下文。

**多智能体编码团队（SWE-bench）**——在 SWE-bench 上表现最好的系统使用一个研究员读取代码库、一个规划器设计修复方案、一个编码员实现它。单智能体系统得分更低。

**ChatGPT Deep Research**——生成多个并行的搜索智能体，每个探索不同的角度，然后综合结果。

### 频谱

多智能体不是二元的。它是一个频谱：

```
简单 ──────────────────────────────────────────── 复杂

 单个        子智能体      流水线       团队        集群
 智能体

 ┌───┐       ┌───┐        ┌───┐───┐    ┌───┐───┐    ┌─┐┌─┐┌─┐
 │ A │       │ A │        │ A │ B │    │ A │ B │    │ ││ ││ │
 └───┘       └─┬─┘        └───┘─┬─┘    └─┬─┘─┬─┘    └┬┘└┬┘└┬┘
               │                │        │   │       ┌┴──┴──┴┐
             ┌─┴─┐          ┌───┘───┐    │   │       │共享   │
             │ a │          │ C │ D │  ┌─┴───┴─┐    │状态   │
             └───┘          └───┘───┘  │ 消息   │    └───────┘
                                       │ 总线   │
 1个循环    父+子任务      逐阶段   │       │    N个对等体，
 1个上下文                               └───────┘    涌现
                                       显式角色        行为
```

**单智能体**——一个循环，一个提示。适合简单任务。

**子智能体**——父智能体为聚焦的子任务生成子智能体。父智能体维护计划。子智能体汇报回来。这是 Claude Code 的做法。

**流水线**——智能体顺序运行。智能体 A 的输出成为智能体 B 的输入。适合分阶段工作流：研究 → 编码 → 审查 → 测试。

**团队**——智能体通过共享消息总线并行运行。每个有角色。编排器协调。适合同时需要不同技能的场景。

**集群**——许多相同或接近相同的智能体共享状态。没有固定的编排器。智能体从队列中拾取工作。适合高吞吐量并行任务。

### 四种多智能体模式

#### 模式 1：流水线

```
输入 ──▶ 智能体A ──▶ 智能体B ──▶ 智能体C ──▶ 输出
          (研究)      (编码)      (审查)
```

每个智能体转换数据并向前传递。简单易懂。一个阶段失败阻塞其余阶段。

#### 模式 2：扇出 / 扇入

```
                ┌──▶ 智能体A ──┐
                │              │
输入 ──▶ 拆分 ├──▶ 智能体B ──├──▶ 合并 ──▶ 输出
                │              │
                └──▶ 智能体C ──┘
```

将工作拆分到并行智能体，然后合并结果。适合可分解为独立子任务的任务。

#### 模式 3：编排器-工作者

```
                    ┌──────────┐
                    │  编排器  │
                    └──┬───┬───┘
                 任务 │   │ 任务
                 ┌─────┘   └─────┐
                 ▼               ▼
           ┌──────────┐   ┌──────────┐
           │ 工作者A  │   │ 工作者B  │
           └──────────┘   └──────────┘
```

一个智能编排器决定做什么，委派给工作者，并综合结果。编排器本身是一个智能体，拥有生成工作者的工具。

#### 模式 4：对等集群

```
         ┌───┐ ◄──── 消息 ────▶ ┌───┐
         │ A │                  │ B │
         └─┬─┘                  └─┬─┘
           │                      │
     消息  │    ┌───────────┐     │ 消息
           └───▶│  共享     │◄────┘
                │  状态     │
           ┌───▶│  / 队列   │◄────┐
           │    └───────────┘     │
     消息  │                      │ 消息
         ┌─┴─┐                  ┌─┴─┐
         │ C │ ◄──── 消息 ────▶ │ D │
         └───┘                  └───┘
```

没有中央编排器。智能体点对点通信。决策从交互中涌现。更难调试，但能扩展到许多智能体。

### 何时不用多智能体

多智能体增加了复杂性。智能体之间的每条消息都是潜在的故障点。调试从"读一个对话"变成"追踪五个智能体之间的消息"。

**保持单智能体当：**
- 任务适配在一个上下文窗口内（约 100k 令牌以内的工作数据）
- 你不需要为不同阶段使用不同的系统提示
- 顺序执行足够快
- 任务简单到拆分增加的开销大于价值

**复杂性成本：**
- 每个智能体边界都是有损压缩步骤：智能体 A 的完整上下文被压缩成给智能体 B 的消息
- 协调逻辑（谁做什么、何时、以什么顺序）本身是 bug 的来源
- 延迟增加：N 个智能体意味着最少 N 次串行大语言模型调用，如果需要反复通信则更多
- 成本倍增：每个智能体独立消耗令牌

经验法则：如果一个任务所需工具调用少于 20 次且适配 100k 令牌以内，保持单智能体。

## 构建它

### 第 1 步：超载的单智能体

这是一个试图做所有事情的单个智能体。它有一个庞大的系统提示和一个包含研究、代码和审查的上下文窗口：

```typescript
type AgentResult = {
  content: string;
  tokensUsed: number;
  toolCalls: number;
};

async function singleAgentApproach(task: string): Promise<AgentResult> {
  const systemPrompt = `You are a full-stack developer. You must:
1. Research the requirements
2. Write the code
3. Review the code for bugs
4. Write tests
Do ALL of these in a single conversation.`;

  const contextWindow: string[] = [];
  let totalTokens = 0;
  let totalToolCalls = 0;

  const research = await fakeLLMCall(systemPrompt, `Research: ${task}`);
  contextWindow.push(research.output);
  totalTokens += research.tokens;
  totalToolCalls += research.calls;

  const code = await fakeLLMCall(
    systemPrompt,
    `Given this research:\n${contextWindow.join("\n")}\n\nNow write code for: ${task}`
  );
  contextWindow.push(code.output);
  totalTokens += code.tokens;
  totalToolCalls += code.calls;

  const review = await fakeLLMCall(
    systemPrompt,
    `Given all previous context:\n${contextWindow.join("\n")}\n\nReview the code.`
  );
  contextWindow.push(review.output);
  totalTokens += review.tokens;
  totalToolCalls += review.calls;

  return {
    content: contextWindow.join("\n---\n"),
    tokensUsed: totalTokens,
    toolCalls: totalToolCalls,
  };
}
```

这种方法的问题：
- 上下文窗口在每个阶段增长。到审查步骤时，它既包含研究笔记又包含代码和先前的推理。
- 系统提示是通用的。不能为每个阶段调优。
- 没有并行运行的内容。

### 第 2 步：专业化智能体

现在拆分。每个智能体获得一个工作：

```typescript
type SpecialistAgent = {
  name: string;
  systemPrompt: string;
  run: (input: string) => Promise<AgentResult>;
};

function createSpecialist(name: string, systemPrompt: string): SpecialistAgent {
  return {
    name,
    systemPrompt,
    run: async (input: string) => {
      const result = await fakeLLMCall(systemPrompt, input);
      return {
        content: result.output,
        tokensUsed: result.tokens,
        toolCalls: result.calls,
      };
    },
  };
}

const researcher = createSpecialist(
  "researcher",
  "You are a technical researcher. Read documentation, find patterns, and summarize findings. Output only the facts needed for implementation."
);

const coder = createSpecialist(
  "coder",
  "You are a senior TypeScript developer. Given requirements and research notes, write clean, tested code. Nothing else."
);

const reviewer = createSpecialist(
  "reviewer",
  "You are a code reviewer. Find bugs, security issues, and logic errors. Be specific. Cite line numbers."
);
```

每个专家有一个聚焦的提示。每个获得一个干净的上下文窗口，仅包含它需要的输入。

### 第 3 步：通过消息协调

用显式的消息传递将专家连接起来：

```typescript
type AgentMessage = {
  from: string;
  to: string;
  content: string;
  timestamp: number;
};

async function multiAgentApproach(task: string): Promise<AgentResult> {
  const messages: AgentMessage[] = [];
  let totalTokens = 0;
  let totalToolCalls = 0;

  const researchResult = await researcher.run(task);
  messages.push({
    from: "researcher",
    to: "coder",
    content: researchResult.content,
    timestamp: Date.now(),
  });
  totalTokens += researchResult.tokensUsed;
  totalToolCalls += researchResult.toolCalls;

  const coderInput = messages
    .filter((m) => m.to === "coder")
    .map((m) => `[From ${m.from}]: ${m.content}`)
    .join("\n");

  const codeResult = await coder.run(coderInput);
  messages.push({
    from: "coder",
    to: "reviewer",
    content: codeResult.content,
    timestamp: Date.now(),
  });
  totalTokens += codeResult.tokensUsed;
  totalToolCalls += codeResult.toolCalls;

  const reviewerInput = messages
    .filter((m) => m.to === "reviewer")
    .map((m) => `[From ${m.from}]: ${m.content}`)
    .join("\n");

  const reviewResult = await reviewer.run(reviewerInput);
  messages.push({
    from: "reviewer",
    to: "orchestrator",
    content: reviewResult.content,
    timestamp: Date.now(),
  });
  totalTokens += reviewResult.tokensUsed;
  totalToolCalls += reviewResult.toolCalls;

  return {
    content: messages.map((m) => `[${m.from} -> ${m.to}]: ${m.content}`).join("\n\n"),
    tokensUsed: totalTokens,
    toolCalls: totalToolCalls,
  };
}
```

每个智能体只接收发给它的消息。没有上下文污染。研究员的 50k 令牌文档阅读从不会进入审查员的上下文。

### 第 4 步：比较

```typescript
async function compare() {
  const task = "Build a rate limiter middleware for an Express.js API";

  console.log("=== Single Agent ===");
  const single = await singleAgentApproach(task);
  console.log(`Tokens: ${single.tokensUsed}`);
  console.log(`Tool calls: ${single.toolCalls}`);

  console.log("\n=== Multi-Agent ===");
  const multi = await multiAgentApproach(task);
  console.log(`Tokens: ${multi.tokensUsed}`);
  console.log(`Tool calls: ${multi.toolCalls}`);
}
```

多智能体版本使用更多总令牌（三个智能体，三次独立的大语言模型调用），但每个智能体的上下文保持干净。每个阶段的质量因系统提示专业化而提高。

1. 添加第四个专家：一个"测试员"智能体，接收来自编码员的代码和审查员的审查反馈，然后编写测试

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|----------------------|
| 集群 | "一群 AI 智能体的蜂巢思维" | 一组对等智能体共享状态，没有固定领导者。行为从局部交互中涌现。 |
| 编排器 | "老板智能体" | 一个智能体，其工具包括生成和管理其他智能体。它计划和委派，但可能不做实际工作。 |
| 协调器 | "交通警察" | 一个非智能体组件（通常只是代码，不是大语言模型），基于规则在智能体之间路由消息。 |
| 共识 | "智能体们达成一致" | 多个智能体必须达成一致的协议。当冲突输出需要解决时使用。 |
| 涌现行为 | "智能体自己想出来的" | 从智能体交互中产生但未被显式编程的系统级模式。可以是有益的也可以是有害的。 |
| 扇出/扇入 | "智能体的 Map-Reduce" | 将任务拆分为并行智能体（扇出），然后合并它们的结果（扇入）。 |
| 消息传递 | "智能体互相交谈" | 智能体之间的通信机制：从一个智能体发送到另一个智能体的结构化数据，取代共享上下文窗口。 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份精心设计的多智能体入门教材，以"单智能体撞墙"的生动比喻开场，将抽象的多智能体架构必要性转化为直觉可感知的问题（上下文饱和、角色混淆、顺序瓶颈）。文档的架构设计值得赞赏：从问题出发→频谱定位→四种模式详解→TypeScript 代码对比→"何时不用"的反面清单——形成了一个完整的论证闭环。代码示例从超载单智能体到专业化多智能体的渐进式重构，让学生通过"看到问题→看到解决方案"的对比来学习，效果远胜于被告知"应该这样做"。

### 二、知识结构梳理

**认知基础：** 单智能体上限由三个机制决定——上下文饱和（窗口容量限制）、角色混淆（单一系统提示无法覆盖多种角色）、顺序瓶颈（串行执行限制吞吐量）。这三者分别对应空间（上下文）、质量（专精度）和时间（并行性）三个维度。

**工程模式：** 四种多智能体模式按复杂度递增排列——流水线（串行阶段）→ 扇出/扇入（Map-Reduce 模式）→ 编排器-工作者（中央委派）→ 对等集群（无中央协调）。频谱概念（单智能体→子智能体→流水线→团队→集群）帮助学生理解这不是二值选择，而是架构自由度。

**落地实践：** 20 次工具调用 / 100k 令牌的经验法则作为"转向多智能体"的阈值判断；每种智能体边界都是有损压缩的警告；Claude Code、Devin、SWE-bench 作为真实系统的参考锚点。

### 三、核心洞察

1. **上下文窗口是单智能体的真正瓶颈，不是模型能力。** 当 Claude Code 生成子智能体时，关键收益不是"更聪明的模型"而是"更干净的上下文"。父智能体的 50k 令牌规划垃圾不会污染子智能体的窗口。为什么重要？它将多智能体的价值主张从"用更多模型"重新定义为"给每个思考者一个干净的工作台"。

2. **每个智能体边界都是有损压缩步骤。** 当研究员智能体输出"Foo 库的认证流程摘要"时，它已经丢弃了原始文档中的细节。编码员智能体永远看不到那些细节。为什么重要？这意味着多智能体系统的质量上限是由"压缩质量"决定的——研究员的摘要越好，整个链的质量越高。

3. **"集群"和"团队"的区别在于组织哲学，而非智能体数量。** 团队有显式角色和编排器；集群没有固定领导者。两者都可以有 10 个智能体。区别在于决策模式：集中式 vs 涌现式。为什么重要？选择集群不是选择"更多智能体"，而是选择"放弃中央控制"——这是架构决策，不是规模决策。

4. **协调逻辑本身是 bug 的来源。** 当一个智能体需要等待另一个的结果、消息路由错误、或者合约不匹配时，bug 出现在协调层而非智能体内部。为什么重要？调试多智能体系统时，首先检查协调逻辑（谁发消息、消息去了哪、格式对不对），再检查智能体内部逻辑。

5. **"何时不用多智能体"这一节是整个文档最有价值的诊断工具。** 文档提供了清晰的阈值：20 次工具调用、100k 令牌。为什么重要？大多数工程过早优化的问题是"过早复杂化"——多智能体架构应该在需要时才引入，而非因为"看起来很酷"。

6. **TypeScript 代码示例的渐进式重构是教学设计的典范。** 同一个任务先用单智能体傻做，再用专业化智能体拆分，最后用消息总线协调。学生看到结构的变化、Token 消耗的变化、代码行数的变化——对比学习比单例学习效果提高数倍。

### 四、教学建议

1. **用真实的单智能体失败案例开场。** 让学生实际运行一个单智能体处理包含 50 个文件的代码库任务，亲眼看到上下文溢出和中段遗忘。亲历的挫败感是最好的动机。

2. **把频谱画在白板上作为"地图"。** 让学生对每个新出现的任务标记：这个任务应该在频谱的哪个位置？是单智能体就够了，还是需要流水线，还是需要集群？让学生习惯用频谱思维评估任务。

3. **"角色混淆"实验。** 让学生先用一个通用系统提示完成研究+编码+审查，再用三个专门提示分别做。比较输出质量。这种 A/B 对比比任何讲解都有说服力。

4. **强制让学生考虑"何时不用"。** 给一个简单任务（例如"写一个排序函数"），问学生：用单智能体还是多智能体？如果坚持用多智能体，列出增加的成本和风险。这个逆向练习训练"不做什么"的判断力。

5. **用消息日志可视化多智能体通信。** 实现一个简单的事件跟踪器，记录每条消息的发送方、接收方、时间戳和内容摘要。让学生"看到"信息流。

6. **让各组选择不同模式解决同一个问题。** 给一个通用任务（如"为一个遗留系统设计迁移计划"），不同组分别实现流水线、编排器-工作者和集群模式，然后比较结果质量和实现难度。

### 五、值得补充的内容

1. **多智能体通信的序列化格式讨论。** 消息应该是自然语言、结构化 JSON 还是混合格式？不同格式的压缩效率、可解析性和容错性差异。

2. **智能体的版本控制问题。** 当一个系统中的不同智能体有不同版本的模型或系统提示时，如何管理兼容性和行为一致性？Rainbow 部署的概念值得在入门阶段提及。

3. **子智能体的异常处理。** 当子智能体超时、返回空结果或无限循环时，父智能体应该如何响应？错误传播模式是整个文档体系中可以加强的部分。

4. **成本估算的定量模型。** 令牌成本的倍增不仅仅是"更多调用"——每个智能体的上下文窗口大小、工具调用频率和输出长度都会影响成本。提供一个简单的成本估算公式会让学生在做架构决策时更有依据。

5. **与 Function Calling 机制的关系。** 多智能体消息传递本质上可以看作智能体将其他智能体作为"远程函数"来调用——这个类比可以帮助有编程背景的学生更快理解。

### 六、一句话总结

多智能体的本质不是"用更多模型"而是"给每个思考者一个干净的工作台和一个明确的工作"——上下文隔离是主要机制，角色专业化是质量保证，而代价是协调复杂性和有损的信息压缩。

同学们好。我是你们的FDE工程老师，今天讲的是 多智能体架构。

你的单智能体在第 14 阶段跑得很好。能读文件、能跑命令、能调 API。然后你把它指向真实代码库——200 个文件，三种语言。它读到第 32 个文件时忘了第 3 个文件的内容。它同时当研究员、编码员、审查员——三件事都做得不好。这叫**单智能体上限**。

三个信号告诉你撞墙了：① **上下文溢出**——50 个文件超过 200k token 窗口。② **技能混合**——研究需要的 prompt 和代码生成完全不同。③ **串行瓶颈**——能同时读的三个文件为什么要顺序读？

多智能体的核心不是"让更多模型一起工作"——是**给每个任务分配合适的上下文窗口和角色范围**。Anthropic 的研究系统用 Opus 4 当监管者、Sonnet 4 当工作者，BrowseComp 上 80% 的方差仅由 token 用量解释——每个子智能体拿到新鲜上下文窗口是主因。

## 结课清单

1. **单智能体上限是真实瓶颈。** 上下文窗口是硬限制，不是"加点 prompt engineering 就能绕过的"。
2. **三个信号：上下文溢出、技能混合、串行瓶颈。** 看到任一个 = 该拆了。
3. **多智能体的价值 = 每个智能体拿到新鲜上下文 + 专精角色。** 不只是更多模型——是更合适的切分。

最后一句话：**你的单智能体不是神。它只是撞了墙。多智能体不是"让它更聪明"——是"不再让它一个人扛所有事"。**

# 💼 从业者故事：多智能体的必要性——你的单智能体不是神，它只是撞了墙

去年十月份，凌晨两点十五分，我被 PagerDuty 叫醒。我们那个"万能"智能体在处理一个 migration PR 的时候炸了——它要读 47 个文件理解数据模型，然后写迁移脚本，再自己 review。结果呢？它读到第 32 个文件的时候，把第 3 个文件的表结构忘了。生成的 SQL 把 `user_id` 定义成了 `VARCHAR`，但原始表里是 `UUID`。更骚的是它自己 review 自己写的代码——你会让写 bug 的人查自己的 bug 吗？它给了个大大的 ✅ "LGTM"。上线后数据库字段类型不匹配，凌晨告警就是这么来的。

那一天我学会了一个道理：**单智能体的上限不是模型不够聪明，是它只有一个脑子、一张桌子、一双眼睛。**

## 问题不是你想象的那样

很多人觉得多智能体的理由是"我需要更强的 AI"。错。你需要的是更多的 AI，不是更强的 AI。这两者的区别就像：你需要十个外卖骑手同时送餐，而不是一个世界短跑冠军骑手跑十趟。

单智能体的三个死穴，我用大白话重新说一遍：

**1. 上下文饱和——不是忘了，是根本看不见了。** 大语言模型的上下文窗口就像你电脑的内存。开到第 30 轮对话，前面的东西早就被挤出去了。它没忘——是压根没有"前面"这个概念了。你以为它记住了第 5 轮的关键信息，实际上它连第 5 轮是什么时候都不知道。这就像你跟一个人聊天聊了三个小时，然后问他"我第一句话说的是什么"——正常人也记不住，但你会怪他吗？你不会。那你为什么要怪一个 200k 窗口被塞到 198k 的模型？

**2. 角色混淆——一个人戴三顶帽子，帽子叠帽子。** 你试试给自己写一个系统提示："你是研究员、架构师、程序员、测试工程师、代码审查员、DevOps 工程师"。你猜模型会怎么做？它会在写代码的时候想着架构，在 review 的时候惦记着还没写完的测试。最后出来的东西就是：什么都会一点，什么都不精。这就像你让一个厨师同时负责炒菜、算账、洗碗、接待客人——最后菜糊了，账算错了，客人跑了。

**3. 顺序瓶颈——单线程跑多线程的活。** 你的智能体在读文件 A → 文件 B → 文件 C，每一次都是一次 API 调用 + 一次模型推理。真 实世界里，你的眼睛可以同时扫三行代码（好吧其实也不能，但至少你可以开三个 tab）。智能体不行。它就是一根筋：读完一个再读一个。50 个文件？50 次串行调用，每次 2 秒，100 秒起步。

## 多智能体的思维模型——不是架构图，是分活

别一上来就看那个四象限架构图。我告诉你我是怎么跟新人讲的：

**多智能体就是把一个全能傻子的工作拆给一群专业傻子。**

每个"傻子"只做一件事，但他做那件事的时候非常清醒。研究员傻子只读文档，他那个 200k 的窗口里全是文档内容，没有一行代码。编码员傻子只写代码，他看到的输入是研究员傻子给的结构化摘要——干净、聚焦、不超过 2k token。审查员傻子只看代码质量，他不知道需求是什么，也不管技术选型为什么这么选——他只找你写的 bug。

关键不是"更多模型调用"。关键是一个词：**上下文隔离**。每个智能体的窗口只装它自己那摊活儿。研究员读的 50 个文件的垃圾永远不会跑到编 码员的窗口里。这就是多智能体真正的价值——不是算力，是注意力。

## 四种模式的本质——不是模式，是控制权的分配

流水线、扇出、编排器、对等集群——这四个东西我换个角度给你讲：它们不是"架构模式"，是**控制权怎么分配**。

- **流水线**：控制权在"顺序"手里。前一个人干完，后一个人才能干。适合"后一步必须依赖前一步结果"的场景。代价：一个环节卡住全链卡死。
- **扇出/扇入**：控制权在"拆分-合并"手里。你把同一个问题拆成三块，同时丢给三个智能体，然后人工（或程序）合并答案。适合"子任务完全独立"的场景。代价：合并逻辑可能比子任务本身还复杂。
- **编排器-工作者**：控制权在一个"老板"手里。老板想清楚怎么分活，然后分给手下。老板不干活，老板只分活和验收。适合"任务结构不固定、需要动态决策"的场景。代价：老板本身就是一个单点故障。
- **对等集群**：没有老板。每个人自己从工作队列里捡活，干完了自己标记。适合"大量同质化并行任务"的场景。代价：调试像在蜂窝里找一只特定的蜜蜂。

你看，选什么模式本质上是在选**你能承受多大的协调成本和单点风险**。不是"哪种模式最好"，是"哪种死法你能接受"。

## 生产现实——那些文档不会告诉你的

多智能体系统的真实痛点不是架构设计，是三个魔鬼：

**魔鬼一：有损压缩。** 研究员读 50k 文档，产出 2k 摘要。那 48k 被丢掉了。编码员永远不知道研究员丢了什么。如果研究员把"API 有个 edge case 只会在闰年触发"给丢了——而你的代码刚好在 2028 年 2 月 29 日出错——你猜怎么着？你查 bug 查了三天，最后发现是研究员智能体的摘要策略有问题。你敢信？你要去 debug 一个 NLP 摘要。

**魔鬼二：消息漂移。** 智能体 A 发消息给智能体 B。消息是自然语言。自然语言有歧义。"这个字段需要是唯一的"——是指数据库 UNIQUE 约束？还是业务逻辑唯一性检查？还是只是建议？三个智能体可能理解出三种含义。你写的 TypeScript 类型定义不会撒谎，但你写的自然语言消息会。

**魔鬼三：协调层的 bug 比智能体内部的 bug 更难查。** 你的审查员报告说代码有问题，但这条消息被错误路由到了测试员而不是编码员。测试员收到代码审查意见一脸懵，然后忽略它直接跑测试。测试通过。你上线。炸了。你翻日志翻了半小时才发现——消息路由写错了 `to: "tester"` 应该写 `to: "coder"`。这 bug 不是 AI 的错，是你自己写的胶水代码的错。

## 什么时候绝对不要用多智能体

给一个硬标准，不是"视情况而定"那种废话：

**如果你的任务在整个生命周期内的总上下文 < 100k token，且工具调用 < 20 次——单智能体。别多想。直接单智能体。**

为什么？因为多智能体系统的最小编排成本大概在 500-1000 token 左右（消息封装、路由、状态管理）。如果你的任务本身才 30k token，加上编排开销就 31k——为了那一丁点的"架构美感"多花了 3% 的成本和 50% 的复杂性？不值得。

我见过太多工程师一上来就说"我们用多智能体吧！"——问他们为什么，答案是"这样更 scalable"。兄弟，你现在的 traffic 是每天 3 个请求，你跟我说 scalable？先把单智能体跑稳了，等它真的撞墙了再拆。墙什么时候来？你会知道的。凌晨两点的 PagerDuty 会告诉你。

## 最后一句

多智能体不是"让 AI 更聪明"，是**承认一个 AI 不够聪明，然后给一群不够聪明的 AI 每人一块干净的工作台**——上下文隔离才是多智能体的灵魂，其他都是实现细节。
