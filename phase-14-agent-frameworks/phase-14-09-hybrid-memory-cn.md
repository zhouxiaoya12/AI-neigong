# 混合记忆：向量 + 图 + KV（Mem0）

> Mem0（Chhikara 等人，2025）将记忆视为三个并行存储——向量用于语义相似度，KV 用于快速事实查找，图用于实体关系推理。检索时由评分层融合三者。这是 2026 年外部记忆的生产标准。

**类型**：构建  
**语言**：Python（标准库）  
**前置要求**：阶段 14 · 07（MemGPT）、阶段 14 · 08（Letta Blocks）  
**预计时间**：约 75 分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅 KV）对 Agent 记忆是不够的
- 说出 Mem0 的三个并行存储及每个针对什么优化
- 描述 Mem0 的融合评分——相关性、重要性、新鲜度——以及为什么它是加权和而非层级结构
- 用标准库实现一个玩具级三存储记忆：一个 `add()` 写入所有三个存储，一个 `search()` 融合结果

## 问题陈述

对于三类查询，单一存储总会出问题：

- **语义相似度**——"我们上周关于 Agent 漂移讨论了什么？"向量赢了；KV 和图都找不到。
- **事实查找**——"用户的电话号码是什么？"KV 赢了；向量太浪费，图是杀鸡用牛刀。
- **关系推理**——"哪些客户共享同一个开票实体？"图赢了；向量和 KV 都回答不了。

生产 Agent 在单次会话中会发出所有三种类型的查询。单一存储记忆对其中两种永远是错的。Mem0 的贡献是将三种存储连接在同一个 `add`/`search` 接口后面，用一个评分函数融合它们。

## 核心概念

### 三个并行存储

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text, user_id, metadata)` 时：

1. 从文本中提取候选事实（LLM 驱动的步骤）。
2. 将每条事实写入向量存储（Embedding）用于语义搜索。
3. 将每条事实按 (user_id, fact_type, entity) 键写入 KV 存储，实现 O(1) 查找。
4. 将每条事实作为类型化边写入图存储（Mem0g），用于关系查询。

在 `search(query, user_id)` 时：

1. 向量存储通过 Embedding 余弦相似度返回 top-k。
2. KV 存储根据查询推导出的 (user_id, type, entity) 键返回精确命中。
3. 图存储返回从查询实体可达的子图。
4. 评分层融合三者。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性**——向量余弦、KV 精确匹配、图路径权重。
- **重要性**——写入时标记或学习得出（某些事实更重要：姓名、ID、政策）。
- **新鲜度**——自上次写入或读取以来的指数衰减。

权重按产品调优。聊天 Agent 提高 `w_recency`；合规 Agent 提高 `w_importance`；检索 Agent 提高 `w_relevance`。

### Mem0g 与时间推理

Mem0g 添加了冲突检测器。当新事实与已有边发生冲突时，已有边被标记为无效但不删除。时间查询（"用户三月份在哪个城市？"）遍历该时间点有效的子图。

这是 Letta 的失效模式所泛化的合规级行为。

### 基准数据

Mem0 论文报告（2025 年）：

- **LoCoMo**（长对话记忆）：91.6
- **LongMemEval**（长时间跨度情景记忆）：93.4
- **BEAM 1M**（1M token 记忆基准）：64.1

对比基线（完整上下文 128k LLM、平面向量存储、平面 KV）全部落后 10 分以上。基准测试本身不能证明选择——操作形态才是——但这些数字表明融合设计不是舍入误差。

### 范围分类

Mem0 按范围划分记忆：

- **用户记忆**——跨会话持久，按 `user_id` 键存储。
- **会话记忆**——在单个线程内持久。
- **Agent 记忆**——每个 Agent 实例的状态。

每次写入选择一个范围。检索可以跨范围查询，带每个范围的权重。不经思考地混合范围，就是"助手把 Alice 的事告诉了 Bob"这类事件的原因。

### 这个模式在哪里出错

- **Embedding 漂移。** 前一百次查询看起来正常的向量结果，随语料库增长而退化。定期重新对 top-N 使用记录做 Embedding。
- **KV Schema 蠕变。** `(user_id, type, entity)` 看起来简单，直到每个团队添加自己的 `type`。每季度审计一次 type 集合。
- **图爆炸。** 一个嘈杂的提取器每条消息添加 50 条边。对每次 `add` 调用限制图写入数；丢弃低置信度边。

## 动手构建

`code/main.py` 用标准库实现三存储模式：

- `VectorStore`——用 token 重叠相似度替代 Embedding 的朴素实现。
- `KVStore`——按 `(user_id, fact_type, entity)` 键的字典。
- `GraphStore`——类型化边 (subject, relation, object, valid)。
- `Mem0`——顶层门面，包含 `add()`、`search()`、融合评分和范围感知检索。
- 一个多用户、多会话对话的完整轨迹。

运行：

```bash
python3 code/main.py
```

输出展示三个独立的召回路径和融合后的 top-k。翻转 `main()` 顶部的评分权重，观察排序如何变化。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是记忆系列的"工程升级课"——从 MemGPT 的 OS 类比（两层架构）跃迁到 Mem0 的三存储融合架构。最大的架构洞察是"单一存储对三种查询类型中的两种永远是错的"——这个分类（语义相似度/事实查找/关系推理）让三存储设计从"做加法"变成了"做数学"。融合评分公式（relevance + importance + recency）是本课的核心工程资产——它把"怎么组合检索结果"从一个启发式变成了一个可调参、可衡量、可按产品迭代的函数。

### 二、知识结构梳理

**认知基础层**：三种查询类型与三种存储的映射（语义→向量、事实→KV、关系→图）、单一存储的必然失败（两个查询类得不到正确答案）

**工程模式层**：三存储的 add 路径（事实提取→并行写入向量/KV/图）、三存储的 search 路径（向量 top-k + KV 精确命中 + 图子图→融合评分）、融合评分三因子（w_relevance/w_importance/w_recency）的产品调优

**实践应用层**：范围分类（user/session/agent）的安全含义、时间失效（软删除而非硬删除）的合规价值、三个生产陷阱（Embedding 漂移/KV Schema 蠕变/图爆炸）的预防

### 三、核心洞察（备课时的关键理解）

1. **三存储不是"加功能"——是"解方程组"。** 三种查询类型构成了三个独立的检索需求，没有一种存储能同时满足。三存储是必要条件，不是锦上添花。

2. **融合评分是加权和，不是层级——这是一个被低估的设计决策。** 层级意味着"先看向量，不够再看 KV，还不够再看图"。加权和意味着三种存储的结果始终被同时考虑，权重按产品场景调节。前者在失败时无声降级，后者在所有时刻都透明。

3. **Mem0g 的"标记无效但不删除"是合规的基石。** 金融审计需要知道"用户二月份时地址是 A"——如果删除旧数据就丢了。软删除 + 时间遍历子图是时间维度上的查询能力，这在 MemGPT 两层架构中不存在。

4. **范围分类不是命名空间——是安全边界。** 用户记忆、会话记忆、Agent 记忆混在一起，就是跨用户数据泄漏的系统性根源。Mem0 把范围作为第一等概念是吸取了 OpenAI Assistants 早期多租户问题的教训。

5. **三个陷阱都是"看起来不紧急，生产环境压死人"的类型。** Embedding 漂移在 1000 次写入后开始显现。KV Schema 蠕变在第三个团队接入时爆炸。图爆炸在一个嘈杂事实提取器上线后瞬间发生。这些不是设计缺陷，是生长痛——但架构设计时就要考虑。

6. **融合权重的按产品调优是把"魔法参数"变成"产品决策"。** 一个合规 Agent 的 w_importance 高于 w_recency——政策的重要性超过它是不是昨天发布的。一个聊天 Agent 的 w_recency 最高——用户更关心最近说了什么。权重不是超参数搜索的参数，是产品经理的决定。

### 四、教学建议

1. **三种查询类型用三个具体问题开场。** 不要先讲架构。先给三个问题：A)"上周我们讨论了什么？"B)"用户的电话号码是多少？"C)"哪些客户属于同一个开票实体？"让学生自己尝试用一种存储回答全部三个——学生自己就会撞墙，然后"发明"三存储。

2. **Mem0 的 add 路径画成三叉戟。** 输入文本→事实提取器→三条并行箭头分别指向向量/KV/图。这个图形让学生一眼看到"add 是扇出，search 是扇入"——写入时分散，检索时聚合。

3. **融合评分不要讲公式——要在黑板上调权。** 在黑板左边写三条检索结果，右边写三个滑动条（relevance/importance/recency）。让学生自己拉滑动条，看哪个结果升到第一位。调 w_recency 到最大——"用户昨天的偏好"排到了第一位。调 w_importance 到最大——"用户的付款方式"排到了第一位。这不是学公式，是学产品直觉。

4. **时间失效必须用"地址变更"的例子。** "用户一月份住在北京，二月份搬到了上海。"在图上画两条边：(用户→住在→北京, valid_until=2025-02-01) 和 (用户→住在→上海, valid_from=2025-02-01)。问学生"一月份用户在哪个城市"——遍历 valid_at("2025-01-15") 的子图。这个例子比任何技术解释都直观。

5. **三个陷阱要在讲完 happy path 之后立刻揭示。** 学生刚理解了架构有多优雅，立刻问："如果在同一个用户上跑 10 万次 add，会发生什么？"学生自然会发现 Embedding 漂移、KV 膨胀、图爆炸——然后自己提出问题，你再给出修复方案。

6. **范围分类用"医院病历"类比。** 用户记忆 = 病人的永久健康档案。会话记忆 = 这次就诊的临时笔记。Agent 记忆 = 辅助诊断系统的工作状态。不能把永久档案和临时笔记混在一起——这是医疗事故。安全和隐私的直觉从类比来。

7. **练习 4（用户反馈维度）是理解"评分可以被操纵"的最佳入口。** 学生加了点赞权重后，Agent 可能陷入过滤气泡——只返回用户已经喜欢过的内容。让学生自己发现这个问题，再讨论如何设计反操纵机制。

### 五、值得补充的内容

1. **事实提取器的质量对三存储的影响。** LLM 驱动的提取步骤是本课最大的隐藏依赖。如果事实提取器把"用户喜欢 Python"和"用户讨厌缩进规则"提取为两条不相关的事实而不是一条关系，图存储的关联推理就失效了。提取质量的上限决定了融合质量的上限。

2. **存储之间的同步一致性。** 三个存储是并行写入的，但写入之间有时间差。如果向量写入了但 KV 写入失败怎么办？分布式事务在这种架构中被刻意回避（性能换简单性），但应该提到这个 tradeoff。

3. **多用户共享记忆时的访问控制。** 企业场景下，Alice 和 Bob 共享一个项目空间。项目相关的事实是否对两人都可见？谁写的不能谁删？这超出了 Mem0 的范围分类，但值得作为扩展点提及。

### 六、一句话总结

**MemGPT 给了 Agent 两本书——一本常在手边，一本在书架上。Mem0 给了它一个图书馆系统——按主题分类、按关系交叉索引、按时间标记变更。一种存储解决一种查询，三种加起来才能回答 Agent 在真实世界中遇到的所有"还记得吗"。**

# 🎓 Agent 架构课：混合记忆——当一种存储不够，用三种

同学们好。

上一节课我们讲了 MemGPT——两层记忆架构，OS 虚拟内存类比。但有一个问题我没有展开讲：当你 Agent 需要查询的记忆类型不止一种时，**两层不够。**

想象你的 Agent 同时面对三个问题：

A) "我们上周关于那个客户退款讨论了什么？"
B) "我妈妈的生日是哪天？"
C) "哪些客户是同一个母公司的子公司？"

这三个问题需要三种完全不同的检索方式。A 是语义搜索——近似匹配，理解意思。B 是精确查找——一个 key，一个 value。C 是关系推理——实体之间怎么连接的。

**没有一种存储能同时做好这三件事。** 用向量存储去查电话号码——杀鸡用牛刀，还不一定精确。用 KV 存储去查"讨论了什么"——关键词对不上就全军覆没。用图存储去查两者——前两个问题图根本不会。

Mem0 的答案是：**三种存储都要，并行写入，检索时融合结果。**

## 三种查询，三种存储

先说清楚为什么一种不够。

**语义相似度查询**："我们上周关于 Agent 漂移讨论了什么？"你需要找到跟这句话意思相近的记忆——不是精确匹配关键词。向量 Embedding 天然擅长这个：把记忆变成向量，把查询变成向量，找余弦距离最近的。

**事实查找**："用户电话是多少？"这不是语义问题。这是精确查找——一个 key（电话号码），一个 value（138xxxx）。KV 存储——字典、Redis、键值对——就是为此设计的。O(1) 查找，不浪费一分算力。

**关系推理**："哪些客户共享同一个开票实体？"这需要图。实体是节点，关系是边。A 公司→属于→X 集团。B 公司→属于→X 集团。遍历"共享同一个父节点"的子图。向量和 KV 都做不到。

你的 Agent 在同一次对话中会交替发出所有三种查询。一个只支持一种存储的记忆系统，对另外两种查询是无能为力的。

## Mem0：三叉戟架构

Mem0 的核心设计就是：一条 add 路径，分叉到三个存储。一条 search 路径，三个存储的结果融合回来。

**写入时（add）：**

1. 接收一段文本——"用户张三，电话 138xxxx，住在北京，是 Acme 集团的员工。"
2. 调用 LLM 提取候选事实——(张三, 电话, 138xxxx)、(张三, 住在, 北京)、(张三, 属于, Acme 集团)。
3. 并行写入：
   - 向量存储：每条事实 Embedding 后存入，带文本和元数据。
   - KV 存储：按 (user_id=张三, fact_type=电话, entity=phone) 键写入 value。"张三的电话→138xxxx"。
   - 图存储：添加类型化边——(张三)→[has_phone]→(138xxxx)、(张三)→[lives_in]→(北京)、(张三)→[works_at]→(Acme集团)。

**检索时（search）：**

1. 向量存储：Embedding 查询，返回语义上最相似的 top-k。
2. KV 存储：从查询中推导 (user_id, fact_type, entity)，精确命中。
3. 图存储：从查询实体出发，遍历可达子图，返回相关节点和边。
4. 融合评分：三个来源的结果按公式打分排序。

## 融合评分——不是在排序，是在做产品决策

融合公式：

```
score = w_relevance × 相关性 + w_importance × 重要性 + w_recency × 新鲜度
```

三个因子，三种考量。

**相关性**：结果和查询有多匹配。向量用余弦相似度，KV 用精确匹配加分，图用路径权重。

**重要性**：有些事实天生比别的更重要。用户的付款方式 > 用户喜欢的颜色。合规政策 > 最近的闲聊。重要性可以在写入时标注（"标记为重要"），也可以从事实类型自动推断。

**新鲜度**：昨天的偏好比去年的偏好更相关。指数衰减——一周前的记忆权重降到一半，一个月前降到四分之一。聊天 Agent 把这个权重调得最高（用户上一句话比上个月那句话重要得多）。

**权重不是魔法数字，是产品决策。** 做聊天 Agent？w_recency = 0.6。做合规 Agent？w_importance = 0.7。做研究助手？w_relevance = 0.8。同一个架构，不同的产品，调不同的权重。

## 图存储与"时间旅行"查询

Mem0 的图存储（Mem0g）有一个独特的能力：时间推理。

当新事实与已有事实冲突时——"用户一月份住在柏林"→"用户二月份搬到了里斯本"——旧的边不被删除。它被标记为 `valid_until: 2025-02-01`。新的边标记为 `valid_from: 2025-02-01`。

这样你就可以做时间查询："用户在 2025 年 1 月 15 日住在哪里？"——遍历那个时间点有效的子图，得到柏林。"现在住在哪里？"——遍历当前有效的子图，得到里斯本。

**这不是数据库优化。这是合规需求。** 银行需要知道客户在某个时间点的地址。审计需要知道政策在某个时间点的版本。医疗需要知道患者的过敏史在什么时候被更新。硬删除是数据丢失；软删除是时间维度上的可查询性。

## 三个陷阱——Architecture is what happens when you scale

**陷阱一：Embedding 漂移。** 你的向量存储在初始 100 条记忆上检索很准。加到 10,000 条时，语义空间越来越拥挤，"相关的"和"不相关的"越来越难区分。到 100,000 条时，top-k 检索可能返回一堆看起来有点像但其实无关的东西。修复：定期对高频访问的记录重新 Embedding——不是重写所有数据，是保持"热数据"的向量质量。

**陷阱二：KV Schema 蠕变。** `(user_id, fact_type, entity)` 的键结构很清晰。第一个月。然后产品团队加了 `fact_type="subscription_status"`。然后增长团队加了 `fact_type="last_login_device"`。然后合规团队加了 `fact_type="gdpr_consent_v4"`。一年后，没有人知道有哪些 fact_type，没有人知道哪个 type 对应什么 value 格式。每季度审计一次 type 集合，保持命名规范。

**陷阱三：图爆炸。** 事实提取器把"用户说他喜欢 Python，但觉得缩进有点烦，不过整体上还是不错的语言"提取成了 8 条边而不是 1 条。每条消息 8 条边 × 1000 条消息 = 8000 条边。图变得无法遍历。对每次 add 限制写入边数，丢弃低置信度边。

## MemGPT + Mem0 = 一个完整的 Agent 记忆架构

这两节课合在一起，构成了 2026 年 Agent 记忆的完整方案：

- **MemGPT 提供架构模式**：主上下文（RAM）+ 外部存储（硬盘）+ 换页中断。这是控制流。
- **Mem0 提供存储实现**：向量 + KV + 图 + 融合评分。这是数据模型。

两者不是竞争——是互补。一个定义 Agent 怎么跟记忆交互。一个定义记忆本身怎么被组织和检索。

## 总结：Agent 架构师的混合记忆检查清单

1. ✅ **查询类型**：你的 Agent 发出的查询是语义的、精确的还是关系的？需要覆盖几种？
2. ✅ **存储选择**：向量用于模糊匹配，KV 用于精确查找，图用于关系推理——三种都覆盖了吗？
3. ✅ **融合权重**：w_relevance、w_importance、w_recency 按你的产品场景调好了吗？
4. ✅ **时间查询**：冲突事实是硬删除还是软删除？审计和合规需要时间遍历吗？
5. ✅ **范围隔离**：用户记忆、会话记忆、Agent 记忆——是分开的，还是混在一起的？
6. ✅ **漂移预防**：有定期重新 Embedding 热数据的机制吗？有 type 集合审计吗？有边写入上限吗？

记住一件事：**MemGPT 教 Agent 怎么去书架上拿书。Mem0 把一个书架变成了图书馆——按主题分类、按关系交叉索引、按时间标记变更。单一存储是一种答案的引擎。混合存储是所有答案的引擎。**
