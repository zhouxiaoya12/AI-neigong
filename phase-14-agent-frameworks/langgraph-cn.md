# LangGraph：有状态图与持久化执行

> LangGraph 是 2026 年级别低层有状态编排的参考实现。Agent 是状态机；节点是函数；边是转换；状态不可变，每一步后做检查点存储。从任何失败处精确恢复。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 01（Agent 循环）、Phase 14 · 12（工作流模式）
**用时：** 约 75 分钟

## 学习目标

- 描述 LangGraph 的核心模型：带不可变状态的状态机、函数节点、条件边、以及步后检查点。
- 说出文档强调的四种能力：持久化执行、流式传输、人类在环、综合记忆。
- 解释 LangGraph 支持的三种编排拓扑：监管者、点对点（群体）、层级（嵌套子图）。
- 实现一个基于标准库的状态图，包含不可变状态、条件边和检查点/恢复循环。

## 问题

Agent 和工作流共享一个难题：当一个 40 步的运行在第 38 步失败时，你希望从第 38 步恢复，而不是从头开始。二等状态模型让运维人员在假定新运行的库周围拼凑重试逻辑。

LangGraph 的设计答案：状态是一等类型化对象，变更显式声明，检查点在每个节点后持久化。恢复就是一次 `load_state(session_id)` 调用。

## 概念

### 图

一个图由以下定义：

- **状态类型（State type）。** 一个类型化字典（或 Pydantic 模型），每个节点读取并变更它。
- **节点（Nodes）。** 纯函数 `(state) -> state_update`。更新在返回后合并到状态中。
- **边（Edges）。** 节点之间的条件或直接转换。
- **入口与出口（Entry and exit）。** `START` 和 `END` 哨兵节点标记边界。

示例：一个包含 `classify`、`refund`、`bug`、`sales`、`done` 节点的 Agent——一个路由工作流作为图。

### 持久化执行

每个节点返回后，运行时序列化状态并写入检查点存储（SQLite、Postgres、Redis、自定义）。在第 N 步失败时，运行时可以 `resume(session_id)` 并从第 N+1 步以精确的状态继续。

LangGraph 文档显式强调了这很重要的生产用户：Klarna、Uber、J.P. Morgan。宣称的并非图的形状——而是图的形状加上检查点存储使得恢复变得廉价。

### 流式传输

每个节点可以产出一部分输出。图向调用者流式传输每个节点的增量事件，使 UI 随图的运行而更新。

### 人类在环

在节点之间检查和修改状态。实现方式：在关键节点前暂停、将状态呈现给人类、接受修改、恢复。检查点存储使这变得容易，因为状态已经序列化。

### 记忆

短期（一次运行内——对话历史在状态中）和长期（跨运行——通过检查点存储加上独立的长期存储持久化）。LangGraph 通过工具与外部记忆系统（Mem0、自定义）集成。

### 三种拓扑

1. **监管者。** 中央路由器 LLM 分派给专家子 Agent。`langgraph-supervisor` 中的 `create_supervisor()`（尽管 LangChain 团队在 2026 年建议通过直接工具调用来做这件事，以获得更多的上下文控制）。
2. **群体 / 点对点。** Agent 通过共享工具面直接转交。无中央路由器。
3. **层级。** 监管者管理子监管者，作为嵌套子图实现。

### 此模式在何处会出错

- **检查点太小。** 仅对对话轮次做检查点会让工具状态和记忆写入不可恢复。完整状态必须序列化。
- **非确定性节点。** 恢复假设节点输入产生相同的状态更新。随机种子、时钟、外部 API 必须被捕获。
- **条件边过度使用。** 每条边都是条件边的图是一个无法推理的状态机。优先使用线性链加偶尔的分支。

## 构建

`code/main.py` 实现一个基于标准库的有状态图：

- `State`——一个类型化字典，包含 `messages`、`step`、`route`、`output`、`human_approval`。
- `Node`——一个可调用对象，接收状态并返回更新字典。
- `StateGraph`——节点 + 边 + 条件边 + 运行 + 恢复。
- `SQLiteCheckpointer`（内存假实现）——在每个节点后序列化状态；`load(session_id)` 恢复。
- 一个演示图：分类 -> 分支（退款 / bug / 销售）-> 人类闸门 -> 发送。

运行：

```bash
python3 code/main.py
```

输出显示首次运行在人类闸门处失败、持久化、然后恢复产生最终输出。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 状态图（State graph） | "Agent 即状态机" | 类型化状态 + 节点 + 边 + reducer |
| 检查点存储（Checkpointer） | "持久化后端" | 每个节点后序列化状态；使恢复成为可能 |
| Reducer | "状态合并器" | 将当前状态与节点更新合并的函数 |
| 条件边（Conditional edge） | "分支" | 由状态函数选择的边 |
| 子图（Subgraph） | "嵌套图" | 作为节点用在另一个图中的图 |
| 持久化执行（Durable execution） | "从失败恢复" | 在最后成功的节点处以精确状态重启 |
| 监管者（Supervisor） | "路由器 LLM" | 为专家子 Agent 做中央分派 |
| 群体（Swarm） | "P2P Agent" | Agent 通过共享工具转交；无中央路由器 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 14 课程中从"模式"转向"基础设施"的关键一课。前三节课讲了技能库、工作流、编排拓扑——这些是"你要搭什么"。这节课讲的是"搭在哪上面"——一个带检查点的有状态图执行引擎。文档的价值在于三个设计决策的阐发：① 状态不可变（更新合并而非原地修改）、② 每步序列化（持久化执行）、③ resumable by session_id（恢复是查询而非重放）。适合已理解工作流/编排概念、正在选型执行引擎的架构师。

### 二、知识结构梳理

**认知基础层：** Agent 循环（第 01 课）是"运行时"概念——while 循环 + 工具调用。LangGraph 将这个运行时形式化为一个可检查点的状态机。工作流模式（第 12 课）提供了"五种模式是什么"的知识——LangGraph 提供了"这五种模式如何在有状态引擎上运行"的知识。

**引擎模型层：** 四个核心概念——状态（typed dict）、节点（纯函数）、边（条件/直接）、检查点（步后序列化）——构成了 LangGraph 的运行时模型。这不是 Agent 框架——这是状态机引擎，Agent 只是它的一种用例。

**拓扑层：** 三种编排拓扑（监管者、群体、层级）通过嵌套子图实现。这是上节课的编排模式在 LangGraph 运行时上的具体实现。

### 三、核心洞察（备课时的关键理解）

1. **"Agent 是状态机"不是比喻——是 LangGraph 的架构定义。** 状态是类型化字典、节点是纯函数、边是转换规则、每步后做检查点。这不是"Agent 框架"——这是"带 LLM 节点的状态机引擎"。Agent 只是这个引擎的一种应用。

2. **不可变状态 + reducer 合并是该模型最被低估的设计决策。** 如果状态是可变的，节点可以直接修改状态——恢复时必须重放所有节点才能知道"当时的状态是什么"。不可变状态意味着：检查点 = 当前状态的完整快照。恢复不需要重放——只需要加载快照。

3. **检查点不是"崩溃恢复"——是"人类在环"的基础设施。** 文档说的"inspect and modify state between nodes"只有在状态已经被序列化并可通过 session_id 访问时才可能。人类在环不是独立功能——它依赖检查点。

4. **"每步序列化"的成本在选择检查点存储后端时体现。** SQLite 是起步选择——单次写入几毫秒。Postgres 适合多副本读取。Redis 适合高频检查点。自定义存储适合合规场景（加密、审计日志）。选择取决于"你的图每秒有多少个节点完成"。

5. **非确定性节点是持久化执行的暗礁。** 如果你的节点调用了外部 API、用了 `random()`、读了系统时钟——恢复时第二次运行可能得到不同结果。LangGraph 要求你捕获这些外部状态作为检查点的一部分。不是框架帮你做——是你必须自己设计。

6. **条件边的过度使用是状态机的通病，LangGraph 不免疫。** 当你的图有 15 个节点，每个节点有多条条件边时——这个图已经无法被人类推理。文档的建议是"优先线性链加偶尔分支"。

7. **三种拓扑（监管者/群体/层级）在 LangGraph 中是嵌套子图的三种用法。** 监管者 = 一个子图作为路由器。群体 = 等价的子图互相调用。层级 = 子图嵌套子图。同一个运行时，三种不同的嵌套模式。

### 四、教学建议

1. **用"第 38 步崩溃"的场景开场。** 讲述一个 40 步的 Agent 任务（比如数据 ETL pipeline），在第 38 步因为 API 超时崩溃。让学生先感受"必须从头重来"的痛苦，再引入检查点=恢复。

2. **状态不可变 vs 可变的对比值得一个现场 demo。** 写一个可变状态的 Agent 循环，崩溃后尝试恢复——状态已损坏。再写一个不可变+检查点的——精确恢复。对比胜于理论。

3. **画一个 5 个节点的图（classify → refund/bug/sales → done），标注每条边、每个节点的输入输出状态。** 可视化状态在节点间的流动，比读代码直观得多。

4. **Reducers 是教学难点。** 用"两个节点同时更新 messages 列表"的例子——默认合并会覆盖，自定义 reducer（append）才会累积。让学生理解"不可变+reducer=可组合的并行更新"。

5. **检查点的三种后端（SQLite/Postgres/Redis）用成本-延迟表对比。** 让学生为三个场景选后端：本地开发、生产单区域、全球多区域部署。

6. **非确定性节点的问题适合作为练习。** 给出一个有 `random()` 调用的节点和一个有外部 API 调用的节点——让学生设计"如何在检查点中捕获这些非确定性"。这是从"用户"到"设计者"的思维转变。

7. **将前四节课的模式映射到 LangGraph 实现。** Voyager 的技能库循环怎么在 LangGraph 状态机里实现？五种工作流模式怎么映射为节点和边？四个编排拓扑怎么映射为子图？这是跨课串联的复习。

### 五、值得补充的内容

1. **检查点存储的垃圾回收。** 1000 次运行 × 40 个节点 = 40000 个检查点。存储会增长。LangGraph 的检查点存储设计假设无限保留吗？需要补充清理策略。

2. **检查点的大小估算。** 消息历史的序列化大小、工具调用参数的序列化大小——这些决定了每次写入的存储和延迟成本。需要一个经验公式。

3. **与 Temporal/Cadence 的对比。** LangGraph 的持久化执行模式非常类似工作流引擎（Temporal、Cadence、AWS Step Functions）。它们之间的设计哲学差异在哪里？

4. **子图的隔离性。** 嵌套子图时——子图的状态是父图状态的子集还是独立命名空间？出错了是只恢复子图还是整个父图？

5. **LangGraph 与 LangChain 的关系演变。** LangGraph 开始是 LangChain 的 Agent 执行引擎，但 2026 年它独立发展。这对已有 LangChain 代码库的团队意味着什么？

### 六、一句话总结

**LangGraph 的价值不是"又一个 Agent 框架"——是它把 Agent 执行从一个 while 循环升级为一个可检查点、可恢复、可流式传输的状态机。状态不可变、每步序列化、恢复不重放——这三个决定构成了它的工程基础。**

# 🎓 Agent 架构课：LangGraph——为什么你的 Agent 需要状态机

同学们好。我是你们的 Agent 架构老师。

前三节课我讲了技能库、工作流、编排拓扑。那些是"你要搭什么"。今天这节课讲的是**"搭在哪上面"**——执行引擎。

我说一个场景，看你们有没有碰到过。

## 第 38 步的崩溃

你在跑一个数据 ETL pipeline。40 步——拉取数据、清洗、转换、验证、写入。Agent 在第 38 步挂掉了。不是代码错了——是外部 API 超时了 5 秒。

你会怎么做？

如果你的 Agent 是一个简单的 while 循环——**从头再跑一遍。** 37 步已经完成了的工作，全部重做。如果其中有一步调用了一个每分钟限流 10 次的 API——你已经在第 38 步耗尽了额度，从头跑意味着第一轮调用就被限流拒绝。

这个问题的根因是什么？**你的 Agent 的执行模型不知道"运行到哪一步了"。** 它的状态和它的执行是耦合在一起的。状态没了，执行没了，一切归零。

LangGraph 解决的就是这个问题。它不是"又一个 Agent 框架"——它是一个带检查点的状态机引擎。Agent 只是它的一个用例。

## 核心模型：状态机，不是 while 循环

我们来看 LangGraph 是怎么重新定义"执行一步 Agent 任务"的。

**状态。** 一个类型化字典。`messages`（对话历史）、`step`（当前步数）、`route`（路由决策）、`output`（最终输出）——所有信息都在这个字典里。每个节点都能读完整的当前状态。

**节点。** 纯函数。输入是当前状态，输出是一个"更新字典"（只包含这个节点要改的字段）。节点不修改状态——它提议修改。

**合并。** 运行时拿到节点的更新字典，用 reducer 把它合并进状态。Reducer 决定"这个字段应该覆盖还是追加"。

**检查点。** 合并完成后，运行时把整个状态序列化，写入一个持久化后端——SQLite、Postgres、Redis。

**恢复。** 任何时候你调用 `resume(session_id)`，运行时从检查点存储读回状态，从最后完成的节点的下一步继续执行。

一个 5 节点的图长这样：

```
START → classify → [refund | bug | sales] → human_gate → send → END
```

`classify` 读消息，决定 `route = "refund"`。条件边根据 route 值跳到对应节点。`human_gate` 在状态上标记 `human_approval = "pending"` 然后暂停。人类审批后，`resume` 恢复。

## 不可变状态：这个设计的精妙之处

LangGraph 做了一个你可能觉得反直觉的决定：**状态是不可变的。**

为什么不能用可变状态？考虑一下：如果节点直接修改状态字典——`state["messages"].append(msg)`——那么在第 5 步崩溃后恢复时，你怎么知道"第 3 步的状态是什么"？

答案是：你不知道。可变状态意味着"当前状态是累积的，但累积的历史被丢失了。" 你只能从第 1 步重放。不可变状态意味着：每一步的检查点存储保存了那一瞬间的完整状态快照。恢复是 `load_checkpoint(step_4)`——不需要重放步骤 1-3。

不可变状态的另一个好处是**并行节点。** 如果两个节点同时运行，它们各自读取同一个不可变状态，各自产出一个更新字典。Reducer 把两个更新合并——如果 reducer 是"追加到列表"，两个节点的输出都会被保留。

但这里有一个工程细节你必须知道：**非确定性是持久化执行的天敌。** 如果你的节点调了 `random()`、读了系统时钟、调了外部 API——第二次运行同一个节点时，相同输入可能产生不同输出。回复假设确定性，所以你必须把这些非确定性源作为状态的一部分捕获。随机种子要存入检查点、API 响应要存入检查点。框架不会帮你做——这是你的设计决策。

## 四种能力：不止是不崩

文档列出 LangGraph 的四种核心能力。我用自己的话解释。

### 能力一：持久化执行——不止是崩溃恢复

持久化执行让你在任何步骤失败后精确恢复。但更重要的是——**它消除了"从头重跑"的恐惧。** 你可以在第 38 步的 API 超时后只重跑第 38-40 步。你可以在第 5 步的 LLM 返回了异常响应时，回到第 4 步的状态，重试第 5 步。

LangGraph 的文档点名了三个生产用户：Klarna、Uber、J.P. Morgan。这些不是"我们在用 LangGraph 做 demo"——是"我们的恢复成本被这个检查点存储降到近乎为零"。

### 能力二：流式传输——UI 跟随图更新

每个节点可以产出部分输出。图每跑一步，调用者收到一个事件——"这个节点的这部分结果"。UI 不需要等整个图跑完才能显示东西——它可以逐节点展示进度。

### 能力三：人类在环——在节点之间插入人

检查点存储让人类在环变得自然。在关键节点前（比如"给用户退款"）暂停。状态已序列化——呈现给人类审核者。人类说"批准"，修改 `human_approval = "approved"`，恢复。人类说"拒绝"，修改状态，走不同的边。

这不是一个独立功能。它依赖检查点存储。没有状态序列化，你只能把状态 dump 到文件里等人来看——这就是"二等状态模型"的现实。

### 能力四：记忆——短期内 + 长期外

LangGraph 区分两种记忆：
- **短期记忆：** 一次运行内的对话历史，在状态字典的 `messages` 字段里。运行结束即销毁（或存入长期）。
- **长期记忆：** 跨运行的持久化。检查点存储（同一 session 内的运行间恢复）加上独立的长期存储（跨 session 知识积累）。LangGraph 通过工具与 Mem0 等外部记忆系统集成。

这个区分很重要——很多 Agent 框架把两种记忆混在一起，导致检查点膨胀（每次运行都存储整个对话历史）或者上下文污染（上一个 session 的对话混进了这个 session）。

## 三种拓扑：同一个引擎，三种嵌套方式

上节课我们讲了四种编排拓扑。这节课告诉你在 LangGraph 里怎么实现。实质就是嵌套子图。

- **监管者。** 一个图，其中"路由"节点本身是一个子图——监管者 LLM 加上工具调用。负责人：`create_supervisor()`（但 2026 年建议用直接工具调用而非这个函数）。

- **群体。** 等价的 Agent 子图，每个都可以调用其他子图。用共享工具面实现转交。没有中心路由节点——转交逻辑分布在每个子图内部。

- **层级。** 子图嵌套子图。监管者子图调用子监管者子图，子监管者调用工作者节点。

同一个运行时引擎，状态、检查点、恢复机制完全一致。区别只在于子图之间的边怎么画。

## 反模式：什么时候 LangGraph 是错误答案

### 反模式一：用 LangGraph 做线性链路

如果你的任务就是 `A → B → C`——三个 LLM 调用顺序执行——不需要 LangGraph。第 12 课的提示链足够。LangGraph 的检查点、条件边、状态类型——对一个线性链来说都是过载。

### 反模式二：检查点粒度太小

你只在"对话轮次"的粒度做了检查点。工具调用的中间状态没有持久化。崩溃恢复后——对话历史恢复了，但那次工具调用的结果丢了。全部状态必须序列化——不只是 `messages`。

### 反模式三：把图当编程语言

你的图有 20 个节点、35 条条件边。每个节点的行为都依赖之前 5 个节点的输出。这个图你三个月后读不懂，你的同事现在就读不懂。条件边是分支——不是 goto。

## 结课清单

如果你在考虑用 LangGraph 作为你的 Agent 执行引擎：

1. 你的任务能从检查点恢复中获益吗？— 如果最长运行不超过 5 步，不需要。
2. 状态被完整序列化了吗？— 不只是 `messages`，工具调用结果、API 响应、随机种子都要在状态里。
3. 节点的非确定性源被捕获了吗？— `random()` 的种子、外部 API 的响应。
4. 图的复杂度可控吗？— 如果超过 15 个节点、20 条边，考虑拆成子图。
5. 检查点存储后端选对了吗？— SQLite 起步、Postgres 生产、Redis 高频。
6. 短期记忆和长期记忆分离了吗？— 当前 session 在状态里、跨 session 在独立存储里。
7. 能用工具调用实现监管就不要用 `create_supervisor`？

**最后一句话：**

**LangGraph 的核心不是"又一个 Agent 框架"——是它把 Agent 的执行从"重试 while 循环"升级为"可恢复的状态机"。状态不可变、每步做快照、恢复是读取而非重放——这三个决定让崩溃成本从"重跑 40 步"降为"接上第 38 步"。**
