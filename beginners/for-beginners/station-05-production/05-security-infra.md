# 基础设施安全：密钥、脱敏和网络隔离

> 预计阅读：20分钟 | 难度：进阶 | 前置条件：理解了生产部署
>
> 你将学到：生产 Agent 的密钥管理、PII 脱敏、网络隔离——三条防线不让你的凭据出现在别人的日志里。

---

## 🚗 开场翻车：.env 文件提交到 GitHub，8 小时后才修完

一个实习生把 `.env` 文件提交到了 GitHub。里面是生产环境的 OpenAI API Key 和 AWS 密钥。

17 分钟后 GitGuardian 扫描发现。但"修"这个问题的流程是：
1. 在 Slack 通知团队
2. 在密钥管理后台轮换密钥
3. 更新 40 个配置文件
4. 重新部署所有服务
5. 确认所有服务拿到新密钥

**8 小时后，一半服务已更新，另一半还在等部署窗口。** 8 小时内攻击者可以用泄露的密钥干什么？

---

## 🔑 第一道防线：密钥管理

### 绝不在这些地方放密钥

| ❌ 不能放的地方 | 应该用什么替代 |
|----------------|---------------|
| `.env` 文件（可能被提交） | 密钥库（Vault/AWS Secrets Manager） |
| 硬编码在代码里 | 运行时从密钥库拉取 |
| 电子表格共享 | 密钥库 + 按需授权访问 |
| CI/CD 变量明文 | OIDC（无密钥的 CI/CD 认证） |

### AI 网关模式——2026 年标准

```
应用 → 网关 → 从密钥库拉取 API Key → 转发给模型提供商
```

密钥不离开密钥库。网关在每次请求时动态拉取。在密钥库轮换密钥 → 网关自动获取新密钥 → 无需重新部署。

### 轮换 + 扫描

- 所有密钥 **≤ 90 天轮换一次**
- 每次提交用 TruffleHog / Gitleaks 扫描
- 扫描到密钥 = 阻止 PR + 立刻轮换

## 🛡️ 第二道防线：PII 脱敏

你的用户可能会在提示里写："我的身份证号是 110101199001011234"。

这段文字会被发送到 OpenAI 的 API。即使你签了数据处理协议，最佳实践是**在脱离你基础设施之前脱敏**。

### 怎么做

```python
# 一致性令牌化：同一个值每次替换成同一个占位符
# "110101199001011234" → "<ID_NUM_1>"
# 下次同样的值 → 还是 "<ID_NUM_1>"
# Agent 能理解"ID_NUM_1"是一个占位符，不影响语义
```

在网关层统一脱敏——所有请求先过脱敏器再过 API 边界。

## 🌐 第三道防线：网络隔离

Agent 能访问的 API 要严格白名单：

```
✅ api.openai.com
✅ api.anthropic.com  
❌ 所有其他出站流量
```

**如果 Agent 的 Pod 能访问任意互联网主机——攻击者可以通过 DNS 查询把数据带出去。**

## 🏆 焊死在脑子里的东西

1. **密钥不进代码、不进配置文件、不进 git。** 只进密钥库。网关在运行时拉取。

2. **PII 在你基础设施的出口处脱敏。** 不要等它到了 OpenAI 的服务器再后悔。

3. **LLM Pod 只能访问白名单域名。** 所有其他出站流量 = 阻断。

4. **每次提交自动扫密钥。** TruffleHog/Gitleaks 跑在 CI 里，扫到就拦。

---

## 📝 练习答案

**优先级：**

1. **先修密钥管理**（最紧急）——`.env` 已经在 git 历史里了。立刻轮换密钥 + 迁移到密钥库。

2. **再修网络隔离**——LLM Pod 能访问外网意味着数据泄露通道敞开着。

3. **最后修 PII 脱敏**——重要但优先级在"密钥泄露"和"网络通道敞开"之后。密钥泄露 = 直接损失。PII 泄露 = 合规风险。

---

> **生产 Agent 安全的底线：密钥不进代码、PII 不出边界、网络不出白名单。三条线都守住，你才能在凌晨三点安稳睡觉。**
