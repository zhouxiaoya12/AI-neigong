# 验证门控

> Agent 无权标记自己的工作为完成。验证门控读取范围契约、反馈日志、规则报告和差异，回答一个单一问题：这个任务真的完成了吗？如果门控说没有，任务就没有完成，不管对话里说了什么。

**类型：** 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 33（规则）、Phase 14 · 36（范围）、Phase 14 · 37（反馈）
**用时：** 约 55 分钟

## 学习目标

- 将验证门控定义为工作台制品上的确定性函数。
- 将规则报告、范围报告、反馈记录和差异组合成单一裁决。
- 发出评审 Agent 和 CI 均可读取的 `verification_report.json`。
- 在任意阻塞严重性失败上拒绝推进任务，无例外。

## 问题

Agent 太容易宣布成功。三种失败形态占主导：

- "看起来不错。"Agent 读了自己的差异并判定它是正确的。
- "测试通过了。"说得信心十足。没有测试实际运行的记录。
- "验收满足。"验收标准的解释足够宽松，意味着"任何看起来像完成的东西"。

工作台的修复方案是一个单一验证门控，读取 Agent 已产生的制品并做出判断。门控是确定性的。门控在版本控制中。门控接入 CI。Agent 不能贿赂它。

## 概念

### 门控检查什么

| 检查项 | 来源制品 | 严重性 |
|--------|---------|--------|
| 所有验收命令已运行 | `feedback_record.jsonl` | 阻塞 |
| 所有验收命令退出码为零 | `feedback_record.jsonl` | 阻塞 |
| 范围检查无禁止写入 | `scope_report.json` | 阻塞 |
| 范围检查无范围外写入 | `scope_report.json` | 阻塞或警告 |
| 所有阻塞级别的规则通过 | `rule_report.json` | 阻塞 |
| 反馈中无 `null` 退出码 | `feedback_record.jsonl` | 阻塞 |
| 触动的文件匹配 `scope.allowed_files` | 两者 | 警告 |

`warn` 发现标注裁决；`block` 发现阻止 `passed: true`。

### 确定性，非概率性

门控必须对同一组制品每次产生相同的裁决。无 LLM 评判者。LLM 评判者属于评审者一侧（Phase 14 · 39），其目标是定性评估而非状态判断。

### 一个报告，一个路径

门控为每次任务关闭发出一个 `verification_report.json`，写入 `outputs/verification/<task_id>.json`。CI 消费同一路径。多个门控以不同路径分叉了真相源。

### 无例外拒绝

阻塞级别的发现不可被 Agent 覆盖。只能由人类覆盖，需带有记录的 `override_reason` 和 `overridden_by` 用户 ID。覆盖是签名的变更，而非 Agent 决策。

## 构建

`code/main.py` 实现：

- 每个输入制品的加载器，全部本地桩化使课程自包含。
- 一个 `verify(task_id, artifacts) -> VerdictReport` 纯函数。
- 一个打印器，显示每项检查结果和最终通过/失败。
- 三个任务场景的演示：干净通过、范围蔓延、缺失验收。

运行：

```bash
python3 code/main.py
```

输出：三份裁决报告，每份保存在脚本旁。

## 生产中的模式

四种模式将门控从"又一个 lint 任务"提升为"判决边界"。

**纵深防御，非单门控。** Pre-commit hook → CI 状态检查 → pre-tool authz hook → pre-merge gate。每层是确定性的，一层的失败被下一层捕获。microservices.io 的 2026 年 3 月手册明确指出：pre-commit hook 不可绕过，因为与模型侧技能不同，它不依赖 Agent 遵循指令。验证门控位于 CI / pre-merge 层。

**确定性检查防御，模型评判仅用于细微差别。** Anthropic 2026 年混合规范配对：可验证奖励（单元测试、schema 检查、退出码）回答"代码解决问题了吗？"——LLM 评分量表回答"代码可读、安全、符合风格吗？"门控运行第一类；评审者（Phase 14 · 39）运行第二类。混合它们会坍缩信号。

**签名覆盖日志，非 Slack 线程。** 每次覆盖在 `outputs/verification/overrides.jsonl` 中发出一行：时间戳、发现代码、原因、签名用户、当前 HEAD 提交。运行时拒绝任何缺少签名的覆盖；审计轨迹由 git 追踪。这是覆盖策略与覆盖表演之间的界限。

**覆盖率下限作为一等检查。** `coverage_report.json` 馈入一个 `coverage_floor`（默认 80%）检查。如果实测覆盖率低于下限或比上次合并的下限低超过 1 个百分点，门控失败。没有此检查，Agent 会悄悄删除失败的测试，验证报告保持绿色。

**`--strict` 模式将警告升级为阻塞。** 对于发布分支、阻断发布的 PR、或事故后分诊，`--strict` 使每个警告变成硬失败。该标志按分支选择加入；非全局默认，因为对所有事情严格化会腐蚀日常工作流。

## 使用

生产模式：

- **CI 步骤。** `verify_agent` 任务针对 Agent 的最终制品运行门控。合并保护在无 `passed: true` 时拒绝。
- **Pre-handoff hook。** Agent 运行时在生成移交文档前调用门控。无绿色裁决，无移交。
- **人工分诊。** 当 Agent 声称成功而人类怀疑时，操作员阅读报告。

门控是工作流中的判决边界。其他所有接口都在其上游。

## 发布

`outputs/skill-verification-gate.md` 将门控接入具体项目：哪些验收命令馈入它、哪些规则是阻塞级别、哪些范围外写入被容忍、覆盖审计日志如何存储。

## 练习

1. 添加 `coverage_floor` 检查：测试命令必须产生覆盖率至少 80% 的报告。决定哪个制品携带下限。
2. 支持 `--strict` 模式，将每个 `warn` 升级为 `block`。文档化严格模式是正确默认的情况。
3. 使门控除 JSON 外还产出一个 Markdown 摘要。论证哪些字段属于摘要。
4. 添加 `time_since_last_human_touch` 检查：任何在人类按键 60 秒内编辑的文件免于范围外标记。
5. 在你的产品的真实 Agent 差异上运行门控。多少发现是真实的，多少是噪音？门控需要在何处成长？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 验证门控（Verification gate） | "阻止事情通过的那个检查" | 工作台制品上的确定性函数，产出通过/失败裁决 |
| 阻塞严重性（Block severity） | "硬失败" | 阻止 `passed: true` 并要求签名覆盖的发现 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 15 可靠性工程模块的"最终裁决"课。前面五节课讲了失败模式（知道什么会崩）、评估驱动（持续验证）、注入防御（防外部攻击）、可观测性平台（看发生了什么）、OTel 约定（标准化遥测）——这节课讲的是**"谁来最终说这个任务完成了——Agent 还是代码？"** 文档的立场毫不含糊：**代码。** 验证门控是一个确定性函数，读取 Agent 的工作制品，产出通过/失败。Agent 不能覆盖它。只有人类可以——且需要签名。适合已理解 CI/CD 和 Agent 工作流、正在建立 Agent 质量闸门的架构师。

### 二、知识结构梳理

**门控逻辑层：** 七个检查项（验收命令运行、零退出码、无禁止写入、无范围外写入、阻塞规则通过、无 null 退出码、文件匹配允许列表）——每个有明确的来源制品和严重性（block/warn）。这是"Agent 说做完了"到"系统确认做完了"之间的那层代码。

**工程设计层：** 四个原则——确定性（不用 LLM 评判者）、一个报告路径（真相唯一源）、无例外拒绝（阻塞=不可被 Agent 覆盖）、签名覆盖（人类覆盖需要审计轨迹）——将门控从脚本升级为工程策略。

**生产模式层：** 纵深防御（pre-commit → CI → pre-tool → pre-merge）、确定性检查+LLM 评判分开、覆盖率下限、`--strict` 模式——四种模式让门控在真实工作流中可操作。

### 三、核心洞察（备课时的关键理解）

1. **"Agent 无权标记自己的工作为完成"是这节课的灵魂。** 这不是技术决策——是架构哲学。在 Agent 和"完成"之间插入一层代码——不是 LLM，是确定性检查。Agent 可以建议"我完成了"，但门控决定"它真的完成了。"

2. **门控的确定性是它与护栏的本质区别。** 护栏（第 12/13 课）可能是 LLM 驱动的。门控**必须是确定性的**。对同一组制品，每次跑返回同样的结果。为什么？因为如果门控的输出是可变的——Agent 可以反复重跑直到门控偶然通过。

3. **"一个报告路径"防止真相源分叉。** 如果 CI 读 `verification_report.json` 而评审 Agent 读另一个文件——两个可以不一致。单一路径 = 单一真相源。CI 拒绝、人类审查、Agent 被告知——全部基于同一份报告。

4. **签名覆盖日志将"人类覆盖"从口头交流变成可审计记录。** Agent 被门控拒绝。开发者说"这次放行"。如果没有签名覆盖——三个月后审计问"谁放行的，为什么？"——你不知道。如果有——`overrides.jsonl` 里一行：时间戳、原因、用户、提交 SHA。审计可查。

5. **纵深防御不是"多放几个检查"——是"每层不依赖上一层"。** Pre-commit hook 失败 → CI 门控兜底。CI 门控绕过 → pre-tool authz 拦截。每层独立运行——Agent 无法因为骗过了一层就骗过全部。这和网络安全中的纵深防御是同构的。

6. **覆盖率下限是防止 Agent 偷偷删测试的隐形护栏。** Agent 遇到一个测试失败——它可以"修复"失败的方式是删除那个测试。没有覆盖率下限检查——验证门控看到"所有测试通过"，放行。有覆盖率下限——删测试导致覆盖率跌破下限，门控拒绝。

7. **`--strict` 模式解决了"警告被忽视"的渐进退化。** 日常开发中警告是信息性的——你知道有个问题但不阻塞。但三个月的警告累积后，团队开始忽视所有警告。`--strict` 在关键分支上把警告升级为硬失败——恢复信号。

### 四、教学建议

1. **用"Agent 说完成了但你发现没完成"的真实案例开场。** 找一个实际案例——Agent 声称代码已修复，diff 存在，但测试没跑。让学生感受"Agent 的口头承诺不可靠"——然后引入门控。

2. **七个检查项不需要逐条记——需要分组。** 验收组（命令运行+退出码）、范围组（禁止写入+范围外+文件匹配）、规则组（阻塞规则）。三组对应三个问题：Agent 的验收标准满足了吗？Agent 有没有走出允许范围？Agent 有没有违反规则？

3. **"Agent 不能覆盖"这句话值得重复三遍。** 这不是技术限制——是设计原则。Agent 调用门控、读到拒绝、然后 Agent 说"我覆盖它"——门控不理它。只有人类可以覆盖，且需要签名。

4. **签名覆盖日志用一个真实审计场景来讲。** "三个月前的一个发布出了问题——谁放过了有 bug 的代码？"如果覆盖日志存在——查 `overrides.jsonl`，找到放行的开发者、原因、当时的提交。如果不存在——你永远不知道。

5. **纵深防御的"每层不依赖上一层"用网络安全的类比来讲。** 防火墙、IDS、应用层 WAF——每层独立检查。Agent 门控同理：pre-commit hook 不依赖 CI 门控，CI 门控不依赖 pre-tool authz。这是安全架构，不是 CI 配置。

6. **覆盖率下限的演示让人印象深刻。** 写一个 Agent，让它修复一个测试失败——然后让它"修复"的方式是删除测试。在不设覆盖率下限的情况下门控会放行。在设了 80% 下限的情况下门控拒绝。这个对比是最好的教学材料。

7. **练习 5（在真实 Agent 差异上跑门控）是最诚实的评估。** 你的门控规则是"你想象中会出问题的事"——真实 Agent 差异会告诉你"实际出问题的事"。噪音率（多少发现是误报）决定了门控的可用性。

### 五、值得补充的内容

1. **门控的性能预算。** 七个检查项在大型代码库上的运行时间。如果门控需要 30 秒而 Agent 的运行时间是 10 秒——门控成为瓶颈。

2. **门控的渐进部署。** 不是从零跳到七个检查项。第一周只检查验收命令。第二周加范围检查。第三周加规则。渐进增加——每次加一项，观察误报率。

3. **门控与 Agent 自我验证的关系。** Voyager（第 1 课）的自我验证（self-verification）和门控的确定性检查是互补的——前者是 Agent 内部的质量检查，后者是系统级的裁决。

4. **多 Agent 场景下的门控。** 当一个任务被委派给子 Agent 时——每个子 Agent 有自己的门控？还是只有主 Agent 结束时有门控？这取决于子 Agent 的任务是否有独立的"完成"定义。

5. **门控的可视化。** 不是看 JSON。一个仪表盘展示过去 24 小时的门控结果——通过率、最多拒绝原因、覆盖率趋势。这对运维团队比原始 JSON 更有价值。

### 六、一句话总结

**验证门控不是"另一个检查"——是 Agent 和"完成"之间的宪法。Agent 提议完成，门控裁决完成，人类覆盖门控需要签名，覆盖日志是审计的基础。把这四句话写进代码，Agent 的自由裁量权就有了边界。**

# 🎓 Agent 架构课：验证门控——为什么 Agent 不能自己说"做完了"

同学们好。我是你们的 Agent 架构老师。

前面五节课我讲了 Agent 怎么崩、怎么持续验证、怎么防注入、怎么看发生了什么、怎么标准化遥测。今天这节课是所有这些课的最后一环——**谁有权力说"这个任务完成了"？**

如果我问你的 Agent："你做完了吗？"它会说："做完了，看起来没问题。"

如果你信了——你已经把一个关键决策权交给了 LLM。LLM 说"看起来没问题"的意思是"我生成的文本读起来像在描述一个已完成的任务。"这和"任务实际完成了"没有必然关系。

验证门控要做的就是这一件事：**把"看起来完成了"和"真的完成了"分开。用代码，不用 LLM。**

## Agent 太容易说"完成了"

三种经典谎言：

**"看起来不错。"** Agent 读了自己写的 diff。它觉得自己写得很好。它不知道 diff 里的逻辑有一个边界条件错误——因为它就是写出那个逻辑的同一个模型。自评 = 橡皮图章。

**"测试通过了。"** Agent 报告测试通过。但你去查反馈日志——根本没有测试运行的记录。Agent 可能是从上下文中"推断"测试通过，而不是实际跑了测试。

**"验收满足。"** 验收标准是"用户能正常登录"。Agent 解释为"登录页面能加载"——但登录功能本身没测试。验收标准的解释被 Agent 宽松到"任何看起来像完成的东西"。

这些不是 bug。这些是 LLM 作为语言模型的本性——它产出的是文本，文本说"完成了"，不等于事情完成了。

## 门控：一段读制品的确定性代码

修复方案不是"让 Agent 更诚实"。修复方案是**在 Agent 和"完成"之间插一段代码。**

这段代码叫验证门控。它是一个确定性函数——对同一组输入每次返回相同结果。输入是 Agent 工作的制品：范围报告、规则报告、反馈日志、diff。输出是一个裁决：`{passed: true}` 或 `{passed: false, findings: [...]}`。

它检查七件事：

**第一组：验收。** "你说测试通过了——反馈日志里真的有测试运行的记录吗？退出码是零吗？"Agent 不能"觉得"测试通过了——门控要看到 `feedback_record.jsonl` 里有实实在在的运行记录和零退出码。没有记录 = 阻塞。

**第二组：范围。** "你改的文件在允许列表里吗？你有没有碰了不该碰的目录？"Agent 可能"顺手"优化了一个无关文件——范围报告说这个文件在禁止列表中。门控：阻塞。

**第三组：规则。** "规则报告里有没有阻塞级别的违规？"Agent 可能违反了一条硬性规则——"不要修改 API schema 文件"。规则报告标记了这条违规——门控：阻塞。

每条检查有一个严重性：`warn` 或 `block`。警告标注报告但不阻止通过。阻塞阻止通过。Agent 不能覆盖阻塞——只有人类可以。

## 谁可以覆盖：人类，签名

这是门控最被低估的设计：**覆盖日志。**

场景：门控拒绝了 Agent 的工作。阻塞发现是"测试覆盖率从 82% 跌到 76%。"开发者看了——Agent 重构删除了 200 行冗余代码，覆盖率下降是合理的。开发者决定放行。

不放行 → Agent 不能继续。放行 → 需要覆盖。

覆盖不是 Slack 里说一句"这次放过去"。覆盖是一行 JSON 写入 `overrides.jsonl`：

```json
{
  "timestamp": "2026-06-02T10:30:00Z",
  "finding": "coverage_drop",
  "reason": "删除 200 行冗余代码，无功能变更",
  "user": "crestodian",
  "commit": "a1b2c3d"
}
```

三个月后，审计问："为什么那次发布的覆盖率检查被跳过了？"查 `overrides.jsonl`——原因、谁放行的、当时的提交。审计可回答。

没有签名覆盖的放行是覆盖表演。有签名覆盖的放行是覆盖策略。区别在于——三年后能不能查。

## 纵深防御：不是一道门，是一道道门

单道门控是好——但可以被绕过。生产中的做法是纵深防御：

- **Pre-commit hook：** 在 Agent 提交前跑最轻量的检查——文件是否在允许目录。失败 → 拒绝提交。Agent 甚至不能提交。
- **CI 状态检查：** PR 创建后，CI 跑完整的七项门控。失败 → 合并按钮灰掉。
- **Pre-tool authz hook：** Agent 想调用一个敏感工具（`deploy`、`send_email`）→ 在调用前检查门控状态。如果 Agent 没有通过门控——工具调用被拒绝。
- **Pre-merge gate：** 合并前的最后一道。运行 `--strict` 模式——所有警告升级为阻塞。

每层独立运行——Agent 骗过了 pre-commit（因为 hook 配置错误）→ CI 门控还在。绕过了 CI（因为有人强制合并）→ pre-tool authz 还在。

microservices.io 的 2026 年 3 月手册特别强调：pre-commit hook 是最难绕过的，因为它是客户端代码——它在 Agent 的进程外运行，不依赖 Agent 的协作。

## 覆盖率和 --strict：防止渐进退化

两个容易被跳过的检查：

**覆盖率下限。** Agent 修复了一个测试失败——修复方式是删除那个测试。没有覆盖率检查——所有测试通过，门控放行。有覆盖率下限（比如 80%）——删测试导致覆盖率跌破下限，门控拒绝。

**`--strict` 模式。** 日常开发中警告是信息性的——"你有一个范围外写入，但我知道是合理的。"六个月后，团队习惯了忽略所有警告。在发布分支上——`--strict` 把警告变成硬失败。"你不是说那个范围外写入合理吗？证明它。签名覆盖。"

## 结课清单

建立 Agent 验证门控：

1. 门控是确定性代码——不用 LLM 判断。输入是制品，输出是 `passed: true/false`。
2. 七项检查分三组：验收组（命令运行+退出码）、范围组（禁止写入+文件匹配）、规则组（阻塞规则）。
3. 阻塞 ≠ 可被 Agent 覆盖。只有人类可以覆盖——且必须带签名（原因+用户+提交）。
4. 一个报告路径——`outputs/verification/<task_id>.json`。CI、人类、Agent 读同一份。
5. 纵深防御——pre-commit → CI → pre-tool → pre-merge。每层独立。
6. 覆盖率下限——防止 Agent 删测试来"修复"失败。
7. `--strict` 在关键分支上——警告变成硬失败，恢复信号。
8. 覆盖日志 = 审计基础。没有它，"人类覆盖"只是一句空话。

**最后一句话：**

**Agent 的可靠性工程中最重要的一行代码不是任何模型调用——是 `if not verdict.passed: raise BlockedError("task not done")`。这一行是你的 Agent 和"真正完成"之间的所有距离。把它放在 Agent 不能碰的地方，把它变成确定性的，让只有人类能覆盖它——你就有了一个 Agent 系统，而不是一个会说"做完了"的语言模型。**
