# 长运行后台 Agent：持久化执行

> 生产级长程 Agent 不在 `while True` 中运行。每次 LLM 调用变成带有检查点、重试和重放的活动。Temporal 的 OpenAI Agents SDK 集成于 2026 年 3 月正式发布。Claude Code Routines（Anthropic）在没有持久本地进程的情况下运行定时 Claude Code 调用。会话在需要人类输入时暂停，在部署后存活，并从以 `thread_id` 为键的最新检查点恢复。在新的易用性背后是一个旧模式——工作流编排——加上一个全新的输入：LLM 调用作为非确定性活动，必须在恢复时确定性重放。

**类型：** 学习
**语言：** Python（标准库，最小持久化执行状态机）
**前置课程：** Phase 15 · 10（权限模式）、Phase 15 · 01（长程 Agent）
**用时：** 约 60 分钟

## 问题

设想一个运行四小时的 Agent。它调用了三个工具、提示用户两次、做了四十次 LLM 调用。中途，运行它的主机重启了。发生了什么？

- 在朴素的 `while True` 循环中：一切丢失。运行从头开始。三个工具调用（有真实副作用）再次执行。用户被再次提示他们已经批准的事情。四十次 LLM 调用被重新计费。
- 在持久化执行中：运行从最近的检查点恢复。已完成的活动不重新执行；其结果从持久化日志中重放。用户不重新批准他们已经批准的事。已完成的 LLM 调用不重新计费。

工作流引擎已经交付这个模式十年了（Temporal、Cadence、Uber 的 Cherami）。新的是 LLM 调用现在是一种活动——非确定性、昂贵、有副作用——并且它们干净地适配这个模式。

本课贯穿始终的主题：长程可靠性会衰减（METR 观察到"35 分钟退化"——成功率随跨度大致呈二次方下降）。持久化执行使得运行可以超过可靠性剖面所支持的长度，这在设计正确时是一种新的安全失败方式，在设计错误时是一种新的不安全失败方式。

## 概念

### 活动、工作流与重放

- **工作流（Workflow）：** 确定性编排代码。定义活动的顺序、分支、等待。必须是确定性的，以便从事件日志重放时不产生意外分歧。
- **活动（Activity）：** 一个非确定性的、可能失败的工作单元。LLM 调用、工具调用、文件写入、HTTP 请求。每个活动以输入和（一旦完成）输出记录。
- **事件日志（Event log）：** 持久化后备存储。每次活动启动、完成、失败、重试以及每个工作流决策都被记录。
- **重放（Replay）：** 恢复时，工作流代码从头重新运行；每个已完成的活动返回其已记录的结果而不重新执行。只有尚未完成的活动才被实际运行。

这与 React 对虚拟 DOM 重新渲染、或 Git 从提交重建工作树的形态相同。编排器中的确定性使持久化变得廉价。

### 为什么 LLM 调用适配这个模式

LLM 调用是：
- 非确定性的（temperature > 0；即使 temperature 0 跨模型版本也会漂移）。
- 昂贵的（金钱和延迟）。
- 可能失败的（速率限制、超时）。
- 有副作用的（如果它们调用工具）。

这正是活动的特征。将每次 LLM 调用包装为活动，你就获得了指数退避重试、跨重启检查点、以及用于调试的可重放轨迹。

### 以 `thread_id` 为键的检查点

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects 和 Claude Code Routines 都收敛到同一 API 形态：一个 `thread_id`（或等价物）标识会话；每次状态转换持久化到后端（默认 PostgreSQL、开发用 SQLite、缓存用 Redis）；恢复读取最新检查点。

后端选择很重要：

- **PostgreSQL：** 持久化、可查询、跨部署存活。LangGraph 的默认选择。
- **SQLite：** 仅本地开发；跨主机丢失数据。
- **Redis：** 快但短暂，除非配置 AOF/快照。
- **Cloudflare Durable Objects：** 透明分布式；由唯一键划定范围；存活数小时到数周。

### 人类输入作为一等状态

提案后提交（第 15 课）需要一个持久化的"等待人类"状态。工作流暂停，外部队列持有待处理请求，一个审批从该确切点恢复。没有持久化，这是尽力而为；有它，一个隔夜审批到达，工作流在早晨继续。

### 35 分钟退化

METR 观察到，每个被测量的 Agent 类别在约 35 分钟的连续操作后都表现出可靠性衰减。任务时长翻倍大致使失败率翻四倍。持久化执行不修复这一点；它让你运行得超过可靠性剖面支持的长度。安全的模式是将持久化与需要重新进入时新鲜人类在环校验的检查点结合，以及与限制总计算量的预算断路器（第 13 课）结合，无论墙钟时间多长。

### 持久化执行是错误答案的时候

- 短于几分钟且没有人类输入的运行。开销 > 收益。
- 严格只读的信息检索。
- 正确性要求在一个上下文窗口内端到端完成的任务（某些推理任务；某些一次性生成）。

## 使用

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 工作流（Workflow） | "Agent 的脚本" | 确定性编排代码；可从事件日志重放 |
| 活动（Activity） | "一个步骤" | 非确定性单元（LLM 调用、工具调用）；执行前后记录 |
| 事件日志（Event log） | "后备存储" | 每次状态转换的持久化记录 |
| 重放（Replay） | "恢复" | 重新运行工作流；已完成活动返回已记录结果而不重新执行 |
| 检查点（Checkpoint） | "保存点" | 以 thread_id 为键的持久化状态；恢复时取最新 |
| thread_id | "会话键" | 划定持久化状态范围的标识符 |
| 35 分钟退化 | "可靠性衰减" | METR：成功率随跨度大致呈二次方下降 |
| 非确定性（Non-determinism） | "重放时漂移" | 墙钟、随机、LLM 输出；必须注册为副作用 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 16 高级 Agent 工程中从"模式"转向"运行时"的关键一课。长程 Agent（第 16 课）揭示了跨度翻倍的趋势，监管者模式（第 17 课）给了解决上下文过载的架构——这节课回答的是这些长程架构在物理上怎么存活：跨重启、跨部署、跨故障。将十年前的工作流引擎模式（Temporal/Cadence）重新解读为 Agent 运行时基础设施，是本节课最大的洞察。核心观点：LLM 调用是一种活动——非确定、昂贵、可失败、有副作用——这正是工作流引擎原生处理的对象类型。

### 二、知识结构梳理

**工作流引擎层：** 三个核心概念——工作流（确定性编排）、活动（非确定性执行单元）、事件日志（持久化存储）——和重放机制（恢复时不重执行已完成活动）构成了持久化执行的完整模型。

**Agent 适配层：** LLM 调用作为活动的论证——非确定性、昂贵、可失败、有副作用——四个属性完美匹配工作流引擎的 activity 定义。

**基础设施收敛层：** `thread_id` 作为跨平台通用 API（LangGraph/MS/Cloudflare/Claude Code）和检查点后端选择矩阵（PostgreSQL/SQLite/Redis/Cloudflare DO）——不同平台独立收敛到同一模式。

### 三、核心洞察（备课时的关键理解）

1. **"LLM 调用是一种 activity"是将 Agent 运行时纳入工作流引擎的直接入口。** Temporal 做了十年的工作流引擎——活动定义、事件日志、重放语义——和 Agent 系统的 LLM 调用共享完全相同的执行剖面。不需要为 Agent 发明新的执行模型——把 LLM 调用注册为一种 activity 类型就够了。

2. **编排器的确定性和 LLM 的非确定性之间的张力是设计关键。** 工作流的编排代码必须是确定性的（分支逻辑、循环条件）——这样恢复时重放不会分歧。LLM 的调用是非确定性的（temperature > 0）——但它在执行时产生一次结果后就变成"确定性的日志条目"。重放时读取日志而非重新调用 LLM——这解决了非确定性。

3. **"35 分钟退化"意味着持久化执行不是让你跑得更久——是让你在退化发生后安全恢复。** 你的 Agent 跑了 30 分钟开始出错。没有持久化→从头再跑→在新的 35 分钟内又出错→无限循环。有持久化→从 25 分钟的检查点恢复，只有最后 10 分钟需要重新推理→更短的连续运行→更低的退化暴露。

4. **`thread_id` 是跨平台 API 收敛的最强信号。** LangGraph、MS Agent Framework、Cloudflare Durable Objects、Claude Code Routines——四个独立的设计团队，在 2026 年各自独立地选择了以会话 ID 为键的检查点 API。这不是碰巧——是"你需要一个键来标识谁的运行正在恢复"这个问题的自然答案。

5. **检查点后端选择不是技术偏好——是故障域决策。** PostgreSQL = 持久且跨部署存活。SQLite = 本地开发，跨主机丢数据。Redis = 快但短暂。Cloudflare DO = 无运维但供应商锁定。后端选择决定了"什么类型的故障会带走你的 Agent 的状态"。

6. **人类在环在持久化执行中是自然实现的——在工作流引擎中它是原生概念。** 工作流暂停在"等待审批"→事件日志记录这个状态→人类审批到达→工作流从暂停点恢复。没有持久化，这是"最好能恢复"。有了它，这是"必然恢复"。

7. **朴素 `while True` vs 持久化执行的重试成本差异可以用练习 1 的 demo 量化。** 三活动工作流，第二次活动崩溃。朴素重试 = 3 次活动执行（第一次白跑了两次）。重放 = 2 次活动执行（第一次重放已记录结果，第三次是新执行）。差异随活动数增加呈线性放大。

### 四、教学建议

1. **用"四小时 Agent 崩溃"的场景开场。** 讲一个故事：Agent 跑了四小时，做了三个工具调用（真实副作用）、提示用户两次（需要审批）、四十次 LLM 调用。主机重启。发生了什么？让学员先回答，然后展示朴素重试和持久化执行的结果差异。

2. **活动/工作流/重放的三个概念用"React 虚拟 DOM"或"Git 重建工作树"来类比。** 工作流 = React 组件树（声明式、确定性）。活动 = DOM 操作（有副作用）。事件日志 = VDOM diff 记录。重放 = 重新渲染——已挂载的 DOM 节点不重新创建。

3. **"LLM 调用是 activity"用四个属性来讲。** 非确定（每次结果可能不同）→ 和 API 调用一样。昂贵（花令牌和钱）→ 和数据库写入一样。可失败（限流、超时）→ 和网络请求一样。有副作用（可能调工具）→ 和支付操作一样。四个属性加在一起 = 这必须被记录、重试、检查点。

4. **重放 demo 是最有冲击力的。** 运行 `code/main.py`，展示朴素重试的 trace（活动 1 执行了两次）vs 重放的 trace（活动 1 只执行了一次，结果从日志中取）。把重试成本差异可视化。

5. **非确定性在重放中的问题用练习 3 来讲。** 在工作流决策中插入 `time.time()`——第一次运行"时间>12:00 所以走 A 分支"，重放时"现在是 14:00，时间>12:00 仍然走 A 分支"——但如果分支条件是"时间在 12:00-13:00"就分歧了。展示墙钟时间如何破坏重放确定性，以及 Temporal 的 `Workflow.now()` API 如何解决（将时间戳注册为副作用，重放时取记录值而非实时值）。

6. **检查点后端选择矩阵应该让学生为三个场景做选择。** 场景 A：本地开发。场景 B：单区域生产部署。场景 C：全球多区域、10K 并发 Agent。每场景对应一个后端。

7. **练习 5（设计 6 小时编码任务的检查点策略）是最综合的。** 学生必须回答：在哪些步骤之间做检查点？（每次工具调用后？每 5 分钟？）恢复时什么需要重新批准？（"删除文件"操作需要人类在环吗？）结合 35 分钟退化——连续运行 >35 分钟风险加速——检查点间距是否应该 ≤ 35 分钟？

### 五、值得补充的内容

1. **事件日志的增长和清理策略。** 一个 6 小时运行的 Agent 可能有 500 个活动。事件日志持续增长——需要归档策略。Temporal 有命名空间级保留策略。

2. **确定性工作流中的"非确定性输入"处理。** 外部 API 调用（非 LLM）也是非确定的——返回可能随重放变化。Temporal 的解决方案：将外部调用的结果作为 activity 的一部分记录，重放时读取记录值。

3. **跨版本 workflow 的兼容性。** 用 v1 工作流定义启动的 Agent——在 v2 部署后恢复。重放 v1 定义还是 v2？Temporal 用 `GetVersion` API 解决。LangGraph 的检查点包含图版本吗？

4. **与 Phase 16·01（长程 Agent）的精确串联。** METR 的 35 分钟退化是本课的核心数据点——持久化执行的意义不在于"跑得更久"，在于"每段连续运行不超过 35 分钟，检查点之间恢复重启"。

5. **Claude Code Routines 和 Temporal 的哲学差异。** Claude Code Routines = 无状态的定时调用（每次调用是新的 Claude Code 进程）。Temporal = 有状态的持久化工作流（进程内跨调用保持状态）。两种哲学适用于不同的 Agent 类型。

### 六、一句话总结

**持久化执行不是 Agent 的新发明——是工作流引擎在 Agent 时代的回归。LLM 调用是 activity、线程 ID 是 session 键、检查点是恢复点。当你的 Agent 运行四小时中途崩溃——问题不是"能不能恢复"，是"你的事件日志够不够让重放只执行未完成的步骤。"**

# 🎓 Agent 架构课：持久化执行——当你的 Agent 跑到一半，主机重启了

同学们好。我是你们的 Agent 架构老师。

前两节课我讲了跨度翻倍和监管者模式——长程 Agent 怎么运行，怎么分解任务。今天这节课讲一个更根本的问题：**Agent 跑着跑着，主机重启了。然后呢？**

你写了一个 Agent，它要跑四个小时。第三小时的时候，它做了三件事：调了一个 API（有真实副作用——比如创建了一条数据库记录）、提示用户审批一个操作（用户点了"同意"）、做了四十次 LLM 调用。

然后主机重启了。你的 Agent 怎么办？

## 两种答案

**第一种答案：从头再来。** `while True` 循环里没有状态保存。重启 → 从零开始。那三条数据库记录——再创建一次（现在有四条了，其中两条是重复的）。那个用户审批——再弹出来一次（用户："我不是已经同意过了？"）。那四十次 LLM 调用——重新付钱。

**第二种答案：从检查点恢复。** 每次 LLM 调用都是一个"活动（activity）"。活动执行前记录输入，执行后记录输出。主机重启 → 读取事件日志 → 已完成的活动不重新执行，直接取记录的结果 → 只运行还没做的活动。数据库记录只有三条。用户只被问一次。LLM 调用只付一次。

第二种答案叫**持久化执行。** 这不是 Agent 领域的发明。Temporal 做这个做了十年了。Cadence 做这个做了十年了。只是现在，LLM 调用变成了一种新的 activity 类型。

## LLM 调用为什么是一种 activity

Temporal 定义 activity 有四个属性：非确定性、可能失败、昂贵、有副作用。

LLM 调用呢？**全部符合。**

- **非确定性：** temperature > 0，同样的 prompt 每次返回不同的结果。即使 temperature = 0，跨模型版本的输出也可能漂移。
- **可能失败：** 速率限制。超时。内容过滤被触发。
- **昂贵：** 每次调用花令牌，花的是真钱。
- **有副作用：** LLM 的输出可能触发工具调用——写入文件、发送邮件、调用外部 API。

这四个属性放在一起的意思就是：**LLM 调用必须被记录。** 输入记录（prompt 和参数）、输出记录（响应内容）、执行状态记录（完成/失败/重试）。事件日志就是这些记录的整体。

恢复时重放：工作流代码从头跑到尾——但遇到"已完成"的活动时，不重新调 LLM——直接从事件日志里取上次的响应。只有"未完成"的活动才真正执行。

## 这不是 Agent 的发明——这是工作流引擎的回归

如果你做过微服务，你可能用过 Temporal 或 Cadence。如果你用过 React，你接触过虚拟 DOM 的重渲染。如果你用过 Git，你知道工作树是从提交历史重放的。

持久化执行是同一个模式：**编排逻辑是确定性的，可以从事件日志重放。执行副作用的活动只运行一次——第一次。重放时用记录值。**

在 Agent 的语境下：
- **工作流 =** Agent 的主循环代码。分支逻辑、循环条件、等待条件。必须是确定性的——不能用 `time.time()` 或 `random()` 在工作流里做分支决策。
- **活动 =** LLM 调用、工具调用、HTTP 请求。非确定性、可失败、有副作用。
- **恢复 =** 重放工作流、跳过已完成活动、只运行未完成的。

## `thread_id`——全行业在同一个答案上收敛了

四个独立的设计团队，在 2026 年各自独立地选择了同一个 API 形态：

- **LangGraph：** `checkpointer.get(thread_id)` → 最新状态。
- **Microsoft Agent Framework：** session ID → 检查点。
- **Cloudflare Durable Objects：** 由唯一键划定范围 → 透明持久化。
- **Claude Code Routines：** thread ID → 会话恢复。

这不是碰巧。这是"你需要一个键来标识谁的运行正在恢复"这个问题的自然答案。`thread_id` 就是那个键。

检查点存哪？四个选择：

- **PostgreSQL：** 持久、可查询、跨部署存活。生产默认。
- **SQLite：** 本地开发，一套就够。跨主机丢数据。
- **Redis：** 亚毫秒读写，但不配置 AOF/快照就会丢数据。
- **Cloudflare Durable Objects：** 无运维，但供应商锁定。

选什么取决于一个简单的问题：**当主机重启时，你要多快恢复，以及你愿意为持久化付多少钱？**

## METR 的 35 分钟——为什么持久化不是为了"跑更久"

METR 的数据说：Agent 在约 35 分钟连续运行后，可靠性开始以二次方速度衰减。任务时长翻倍，失败率大致翻四倍。

持久化执行不是这个问题的答案——**你不能靠持久化让 Agent 连续跑 2 小时不出错。** 可靠性衰减是模型行为的问题，不是存储的问题。

持久化做的是另一件事：**它让你把 2 小时的运行拆成 4 段 30 分钟的运行。** 每段之间——检查点，恢复，重新人类在环。Agent 实际上跑的总时间是 2 小时——但每段连续运行不超过 30 分钟，保持在可靠性衰减的阈值之下。

这就是持久化执行和 35 分钟退化的安全结合方案。

## 结课清单

为长程 Agent 添加持久化执行：

1. LLM 调用包装为 activity——记录输入/输出到事件日志。
2. 编排代码保持确定性——不在工作流决策中使用墙钟/随机。
3. `thread_id` 作为会话键——跨重启、跨部署恢复状态。
4. 检查点后端选 PostgreSQL（生产）、SQLite（开发）、Redis（缓存）。
5. 人类在环暂停是持久化状态——审批到达后从暂停点恢复。
6. 检查点间距 ≤ 35 分钟——在可靠性退化阈值内保持每段连续运行。
7. 恢复时对敏感操作重新要求人类审批——"从检查点恢复"不等于"不需要重新确认。"

**最后一句话：**

**持久化执行不是一个 Agent 功能——是 Agent 的基础设施。它不是在告诉 Agent"怎么跑"——是在回答"你跑着跑着崩了以后，事件日志够不够让你只重做没做完的事。"十年工作流引擎的经验加上一个全新类型的活动——LLM 调用——这就是 2026 年长程 Agent 的运行时。**
