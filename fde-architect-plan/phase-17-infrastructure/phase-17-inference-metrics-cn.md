# 推理指标 — TTFT、TPOT、ITL、有效吞吐、P99

> 四个指标决定推理部署是否正常工作。TTFT 等于预填充加排队加网络。TPOT（等价于 ITL）是每个 Token 的内存受限解码成本。端到端延迟是 TTFT 加 TPOT 乘以输出长度。吞吐量是跨集群聚合的每秒 Token 数。但真正对产品有意义的是有效吞吐（goodput）— 同时满足所有 SLO（服务水平目标）的请求占比。高吞吐低有效吞吐意味着你处理了大量 Token，但从未按时送到用户面前。Llama-3.1-8B-Instruct 在 TRT-LLM 上 2026 年的参考数据：平均 TTFT 162 毫秒，平均 TPOT 7.33 毫秒，平均 E2E 1,093 毫秒。始终报告 P50、P90、P99 — 永远不要只报告均值。还要注意测量陷阱：GenAI-Perf 在 ITL 计算中排除 TTFT，而 LLMPerf 包含它；两个工具对同一运行报告的 TPOT 不同。

**类型：** 学习
**语言：** Python（标准库，简单的百分位计算器和有效吞吐报告器）
**前置课程：** Phase 17 · 04（vLLM 服务内部原理）
**时间：** 约 60 分钟

## 学习目标

- 准确定义 TTFT、TPOT、ITL、E2E、吞吐量和有效吞吐，并指明每个指标测量哪个组件。
- 解释为什么均值对 LLM 服务是错误的统计量，以及如何解读 P50/P90/P99。
- 构建一个 SLO 多约束条件（如 TTFT<500ms AND TPOT<15ms AND E2E<2s）并对照它计算有效吞吐。
- 指出两个对同一运行报告不同 TPOT 的基准测试工具并解释原因。

## 问题

"我们的吞吐量是每秒 15,000 个 Token。"那又怎样？如果 40% 的请求端到端延迟超过 2 秒，用户已经放弃了会话。吞吐量单独不能告诉你产品是否正常工作。

推理有多个维度的延迟，每个维度的失败原因不同。预填充受计算限制，随提示长度增加。解码受内存限制，随批次大小增加。排队延迟是运维问题。网络是物理距离问题。你需要每个维度的独立指标，需要百分位数，还需要一个单一综合指标来表示"用户是否得到了他们期望的结果"——这就是有效吞吐。

## 概念

### TTFT — 首 Token 时间

`TTFT = 排队时间 + 网络请求 + 预填充时间`

当提示较长时，预填充占主导。在 H100 上的 Llama-3.3-70B FP8，32k 提示的纯预填充约需 800 毫秒。排队时间是负载下的调度器行为。网络请求包括 TLS 的传输时间。TTFT 是用户在收到任何流式返回之前感受到的延迟。

### TPOT / ITL — Token 间延迟

同一个量有多个名称。`TPOT`（每个输出 Token 的时间）、`ITL`（Token 间延迟）、`每 Token 解码延迟` — 都是一回事。它是首 Token 之后连续流式 Token 之间的时间。

`TPOT = (解码前向时间 + 调度器开销) / 产生的 Token 数`

在同一套 Llama-3.3-70B H100 上，启用分块预填充时 TPOT 均值约 7 毫秒。不使用分块预填充时，如果相邻序列正在进行长预填充，TPOT 会飙升至 50 毫秒。要关注 P99，不是均值。

### E2E 延迟

`E2E = TTFT + TPOT * 输出 Token 数 + 网络响应`

对长输出（>500 Token），E2E 受 TPOT 主导。对长提示的短输出，E2E 受 TTFT 主导。报告按输出长度分档的 E2E。

### 吞吐量

`吞吐量 = 总输出 Token 数 / 经过时间`

聚合指标。告诉你集群效率。不告诉你单个请求的健康状态。

### 有效吞吐 — 你真正关心的指标

`有效吞吐 = 满足 (TTFT <= a) AND (TPOT <= b) AND (E2E <= c) 的请求占比`

SLO 是多约束条件。一个请求只有每个约束都满足才算"好"。有效吞吐就是这个比例。高吞吐配 60% 有效吞吐是失败的。低吞吐配 99% 有效吞吐才是目标。

2026 年，有效吞吐已被用于 MLPerf Inference v6.0 提交和 AI 平台提供商的内部 SLA 跟踪。

### 为什么均值是错误的统计量

LLM 延迟分布是右偏的。一个解码批次有一半序列 TPOT 约 7 毫秒，20 个序列 TPOT 约 60 毫秒。均值 TPOT 是 9 毫秒。P99 TPOT 是 65 毫秒。用户经常命中 P99 — 这就是他们离开的原因。

始终报告三元组 (P50, P90, P99)。对用户体验而言，P99 是你优化的目标。

### 参考数据 — Llama-3.1-8B-Instruct 在 TRT-LLM 上，2026 年

- 平均 TTFT：162 毫秒
- 平均 TPOT：7.33 毫秒
- 平均 E2E：1,093 毫秒
- P99 TPOT：根据分块预填充配置在 10-25 毫秒之间变动。

这是 NVIDIA 发布的参考值。这些值会随模型大小（70B 会显示 3-5 倍）、硬件（H100 vs B200 约 3 倍）和负载而变化。

### 测量陷阱

2026 年最常用的两个基准测试工具对同一运行报告不同的 TPOT：

- **NVIDIA GenAI-Perf**：在 ITL 计算中排除 TTFT。ITL 从第 2 个 Token 开始计算。
- **LLMPerf**：包含 TTFT。ITL 从第 1 个 Token 开始计算。

对于一个 TTFT 500 毫秒、100 个输出 Token 在 700 毫秒总解码时间内完成的请求，GenAI-Perf 报告 `ITL = 700/99 = 7.07 ms`，而 LLMPerf 报告 `ITL = 1200/100 = 12.00 ms`。工具选择改变了数字。

始终说明使用的是哪个工具。始终公布定义。

### 构建 SLO

2026 年 70B 聊天模型的一个合理的面向用户 SLO：

- TTFT P99 <= 800 毫秒。
- TPOT P99 <= 25 毫秒。
- E2E P99 <= 3 秒（<300 Token 输出）。
- 有效吞吐目标 >= 99%。

企业级 SLO 收紧 TTFT（200-400 毫秒）并放松 E2E。关键是要写出来，测量三个指标，并将有效吞吐作为单一综合指标跟踪。

### 如何测量

- 运行真实流量或逼真的合成流量（LLMPerf 带 `--mean-input-tokens 800 --stddev-input-tokens 300 --mean-output-tokens 150`）。
- 针对 2 倍峰值并发运行基准测试。
- 运行 30-50 次迭代，对组合样本取百分位。
- 公布时注明工具名称、工具版本、模型、硬件、并发数、提示分布。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| TTFT | "首 Token 时间" | 排队 + 网络 + 预填充；长提示时受预填充主导 |
| TPOT | "每输出 Token 时间" | 首 Token 之后每个 Token 的内存受限解码成本 |
| ITL | "Token 间延迟" | 大多数工具中与 TPOT 相同（但并非全部 — 参见 GenAI-Perf） |
| E2E | "端到端" | TTFT + TPOT * 输出长度；再加上响应侧网络 |
| 吞吐量 | "tok/s" | 集群效率；没有延迟百分位数则毫无用处 |
| 有效吞吐 | "SLO 达标率" | 同时满足每个 SLO 约束的请求占比 |
| P99 | "尾部" | 1/100 的最坏延迟；用户体验指标 |
| SLO 多约束 | "联合条件" | 三个延迟边界的 AND；任一个被违反则请求失败 |
| GenAI-Perf vs LLMPerf | "工具陷阱" | 两个工具对 ITL 是否包含 TTFT 意见不一致 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇「工程精确度」极高的指标教学文档 — 不只是列出定义，而是深入解释了每个指标测量哪个物理组件、为什么均值是错误统计量、以及两个主流基准工具对同一运行的不同测量结果。TTFT=排队+网络+预填充、TPOT=解码/Token 的分解公式使抽象的延迟概念变得可操作化。最大亮点是"有效吞吐"概念的引入 — 将三个 SLO 约束的 AND 作为单一的部署门禁指标，这对 CI/CD 实践者来说极其有用。测量陷阱部分是独家知识 — GenAI-Perf vs LLMPerf 的差异在中文资料中少见。不足在于没有提供真实 Prometheus/Grafana 仪表板方案，停留在理论的 SLO 构建。

### 二、知识结构梳理

- **指标定义层**：TTFT（首 Token）、TPOT/ITL（Token 间延迟）、E2E（端到端）、吞吐量（聚合效率）、有效吞吐（SLO 达标率）。每个指标精确测量哪个组件。
- **统计方法层**：均值 vs 百分位数（P50/P90/P99），右偏分布下均值的误导性，P99 是用户体验指标。测量陷阱 — 不同工具的 ITL 定义差异。
- **工程实践层**：SLO 的构建方法（多约束 AND），参考数据（Llama-3.1-8B），测量方法（流量、并发、迭代次数），有效吞吐作为部署门禁。

### 三、核心洞察

1. **有效吞吐是唯一你应该在告警中监控的指标。** 吞吐量单独毫无意义 — 40% 的请求可能已超时。有效吞吐将多个 SLO 约束合成为一个单一数字，适合 CI/CD 门禁。
2. **LLM 延迟分布是右偏的，均值掩盖了用户实际体验到的痛苦。** P99 是用户离开的原因。如果只报告均值，你根本看不到那 1% 的极慢请求。
3. **TTFT 和 TPOT 的物理瓶颈完全不同。** TTFT 的计算瓶颈在预填充（GPU 算力），TPOT 的瓶颈在 KV 缓存带宽（内存）。优化策略应分开设计 — 提示压缩降低 TTFT，批处理策略影响 P99 TPOT。
4. **GenAI-Perf vs LLMPerf 的 ITL 差异是真实的工程陷阱。** 同一个运行，两个工具报告差 70%。在内部报告和对外沟通中必须说明使用的是哪个工具以及其定义。
5. **分块预填充保护 P99 TPOT 而不是均值 TPOT。** 因为它把长预填充切成小块，减少了对并发序列的暂停时间，但不改变平均解码时间。
6. **SLO 多约束条件的 AND 逻辑是严格的。** 一个请求只需要在一个维度上失败就被计入"无效"。这意味着优化必须关注最弱的那个维度，而不是平均数。
7. **"合成流量比峰值并发高 2 倍"的测量方法是务实的。** 不是压到系统崩溃，而是在有意义的负载下测量。30-50 次迭代的样本量对百分位数统计来说足够了。

### 四、教学建议

1. **从"15,000 tok/s, 那又怎样？"开场。** 展示一个高吞吐但 40% 超时的实际截图，让学员理解"吞吐量迷思"的陷阱。
2. **画延迟分布的可视化图。** 右偏分布（长尾在右侧） vs 正态分布 对比，让学员直观看到为什么均值 9 毫秒但 P99 是 65 毫秒。
3. **有效吞吐的 AND 逻辑用一个表格展示。** 列出 10 个请求，每列的 TTFT/TPOT/E2E 是否满足 SLO，最后一列 AND 结果。视觉化"一个维度失败就全失败"。
4. **测量陷阱用数字对比教学。** 同一个请求的 GenAI-Perf 和 LLMPerf 计算过程并排展示，差异一目了然。
5. **让学员自己写 SLO。** 给一个场景（客服聊天机器人、代码补全、语音助手），让他们定义 TTFT/TPOT/E2E 的 P99 目标并解释理由。
6. **强调"分块预填充保护 P99 但不保护均值"需要画时间线。** 展示两个序列交替执行时，不分块的长预填充把另一个序列的 TPOT 从 7ms 推到 50ms 的时序图。
7. **练习 5（工具差异）可以扩展为小组讨论。** 除了 ITL，还有哪些指标两个工具可能定义不同？Throutput 的计算方式、排队时间的归属 — 鼓励学员去读两个工具的源码。

### 五、值得补充的内容

1. **H100 vs B200 vs 消费级 GPU 的参考数据对比。** 文档只给了 H100 的数据，加上 B200 和 RTX 4090 的对比能帮助不同预算的团队校准预期。
2. **实际 Grafana/Prometheus 仪表板配置。** 概念讲清楚了，但学员可能不知道如何在生产环境中实际收集和展示这些指标。
3. **有效吞吐 vs 延迟的帕累托边界。** 在给定的 GPU 预算下，吞吐量和有效吞吐之间的最优权衡曲线 — 这是一个决策工具。
4. **不同模型架构（MoE vs Dense）的延迟特征差异。** MoE 模型的路由开销对 TTFT 和 TPOT 的影响与 Dense 模型不同，值得区分。

### 六、一句话总结

**推理指标不是越多越好 — TTFT 告诉你用户等了多久看到第一个字，TPOT 告诉你阅读体验是否流畅，有效吞吐告诉你多少人真正满意；扔掉均值，盯住 P99，写明你用的是哪个工具。**

# 🎓 Agent 架构课：推理指标——你的 VP 在问吞吐量，但你的用户在骂延迟

同学们好。我是你们的FDE工程老师，今天讲的是 推理指标。

**VP 问"吞吐量怎么样"——你骄傲地说 15,000 tok/s。但你的用户不关心吞吐量——他们在乎"为什么每次回复要等 8 秒"。**

四个核心指标：① **TTFT（首 token 时间）**——用户体感延迟。1 秒以下是"即时"，3 秒以上是"卡了"。② **TPOT（每输出 token 时间）**——生成速度。流式输出第一个 token 之后，每个后续 token 的间隔。③ **吞吐量**——系统全局 tok/s，这是你关心的不是用户关心的。④ **TTLT（总延迟）**——从发请求到收完最后一个 token，端到端。

关键洞察：**TTFT 和 TPOT 是用户体验指标。吞吐量是成本指标。别在 VP 面前用后者解释前者。**

## 结课清单

1. **TTFT = 用户体感延迟。** <1秒好，>3秒差。
2. **吞吐量 ≠ 用户体验。** 15,000 tok/s 的系统可以每个用户等 10 秒。
3. **四个核心指标缺一不可。** 各管一类问题诊断。

最后一句话：**你监控的指标决定了你优化什么。只监控吞吐量 = 只优化成本，用户会骂你慢。监控 TTFT = 用户感到快，然后你再去控制成本。**

# 💼 从业者故事：推理指标——你的 VP 在问吞吐量，但你的用户在骂延迟

VP 在周会上问："咱们系统的吞吐量怎么样？"我把监控屏投上去：15,000 tok/s，还挺得意的。VP 点点头，满意地继续下一个议题。

然后产品经理私下跟我说："用户满意度掉了 12 个点，你知道为什么吗？"我不知道。我去翻了真实用户的延迟数据——40% 的请求端到端延迟超过 2 秒。用户输入一句话，等了 2 秒才开始出字。在 2026 年，2 秒的首字延迟等于永恒。但我之前只用均值回答 VP 的问题——均值没问题！

这就是推理指标最经典的坑：**你用吞吐量向上汇报，用户用 P99 体感投票。** 在 LLM 推理延迟的右偏分布下，均值和 P99 之间可能差 5-7 倍。你以为系统很健康，其实 1% 的用户每次都在骂。

## 拆开看：TTFT 和 TPOT 根本不是一回事

我第一次看 LLM 推理延迟的时候，脑子里只有一个数字："从发送请求到收完回复，多久？"这个叫 E2E，它其实是一块三明治：TTFT（首 Token 时间）是上面那片面包，TPOT（每个 Token 的间隔）是中间的肉，输出长度决定了这片三明治有多厚。

TTFT = 排队 + 网络 + 预填充。排队长的时候 TTFT 爆炸，因为你的请求在 GPU 外面等着。预填充长的时候 TTFT 也爆炸，因为 GPU 要一口吃掉你整个 prompt 才能开始吐第一个字。

TPOT = 解码 / Token 数。这个数字小（H100 上约 7ms），但用户在盯着屏幕看每一个字蹦出来，7ms 还是 25ms 的区别就是"打字流畅"和"卡顿感"的区别。

这两个指标指向完全不同的瓶颈。TTFT 高 → 优化排队策略或分块预填充。TPOT 高 → 优化批处理大小或换更快的 GPU。别混为一谈。

## 有效吞吐：我唯一相信的指标

吞吐量是 15,000 tok/s，有效吞吐只有 60%。翻译成人话：你的系统在高速公路上开得很快，但每 3 辆车就有 1 辆开错了出口。

有效吞吐的定义特别简单：同时满足 TTFT ≤ 800ms AND TPOT ≤ 25ms AND E2E ≤ 3s 的请求占比。三个条件，缺一个就判失败。因为用户不管你 TTFT 多快——如果打字打一半突然卡 50ms，他一样觉得"这系统不行"。

我在团队里立了规矩：任何优化 PR 的 commit message 里必须写"有效吞吐从 X% 提升到 Y%"，而不是"吞吐量提升了"。后者是给投资人看的，前者是给用户看的。你服务的对象是用户，不是投资人的 Excel 表格。

2026 年 MLPerf Inference v6.0 已经开始用有效吞吐作为提交标准了。这个趋势意味着：只会报吞吐量的团队正在失去 benchmark 的话语权。

## GenAI-Perf vs LLMPerf：两个工具，同一个请求，两个不同的数字

这是一个真实的段子：我们组用 GenAI-Perf 测了一个 TPOT，对外宣传"我们的每 Token 延迟只有 7ms"。一个用户跑来质疑，他用自己的 LLMPerf 测到的是 12ms。两边都觉得自己没错。

根因特别狗血：GenAI-Perf 计算 ITL 的时候，把首 Token 排除在外了——它从第 2 个 Token 开始计时。LLMPerf 从第 1 个 Token 开始。同样一批数据，因为"第一个 Token 到底算不算 ITL"这个定义上的分歧，结果差了 40%。

这就是为什么文档里说"始终说明使用的是哪个工具"——不是客气话，是你迟早会遇到有人跟你吵这个数字。我的做法：所有内外部报告中标注"Measured with GenAI-Perf vX.Y（ITL excludes first token）"。这样别人想挑战你，至少知道从哪挑战。

## P99 是用户的眼睛，均值是工程师的幻觉

LLM 推理延迟的分布不是正态分布，是右偏的，尾巴拖得老长。一个批次里 19 个用户的 TPOT 是 7ms，1 个用户是 65ms。均值是 9ms。你会说"平均 9ms，挺好的"。但那个 65ms 的用户每次用你的产品都会觉得"怎么打字还有卡顿"，然后默默换到竞品去。

P99 就是盯住最惨的那 1%。如果你的 P99 TTFT 超过 800ms，并不是 1% 的用户偶尔慢——在大规模下，每天可能有几千次请求命中这个尾巴。几千次"你的系统很慢"的体感，足够让用户评分掉 0.5 分。

我现在的监控三件套：P50（中位数 — 大部分用户体感）、P90（开始不满的边界）、P99（用户离开的原因）。三个数字必须同时看，缺少任何一个就像开车只看速度表不看油量表——你迟早会停在路边。

## 构建 SLO：别猜，写下来

很多团队的 SLO 是"系统要快"。这不是 SLO，这是许愿。真正的 SLO 长这样：

- TTFT P99 ≤ 800ms
- TPOT P99 ≤ 25ms
- E2E P99 ≤ 3s（输出 ≤ 300 Token）
- 有效吞吐 ≥ 99%

有数字，有百分位，有边界条件（输出长度分段）。这才是一个能被 CI/CD 检索引用的门禁，而不是一句心灵鸡汤。

企业级 SLO 收得更紧：TTFT P99 ≤ 200ms，因为企业用户比个人用户更没有耐心。消费者聊天机器人可以放宽到 E2E 5s，但在代码补全场景下 E2E 超过 1s 就是产品报废——没人愿意等 AI 补全代码。SLO 不是抄模板，是你的产品体验的数学表达式。

**吞吐量取悦你的老板，有效吞吐取悦你的用户，P99 保护你不在凌晨被叫醒。把这三个数字写在仪表板最顶上，让团队的每一个人在推代码之前都看它一眼。**
