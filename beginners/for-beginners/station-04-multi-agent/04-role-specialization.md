# 给 Agent 分工：规划者、执行者、批评者、验证者

> 预计阅读：20分钟 | 难度：进阶 | 前置条件：理解了多 Agent 编排
>
> 你将学到：四个经典角色怎么组合、为什么验证者是最重要的、怎么避免"三个 Agent 犯同一个错"。

---

## 🚗 开场翻车：三个编码员写出同一段烂代码

一个团队搭了一个多 Agent 代码生成系统。三个 Agent，都叫"编码员"——都基于同一个模型，不同的提示。用户说"写一个排序函数"。

Agent A 写了一个冒泡排序。Agent B 说"我同意"。Agent C 说"看起来不错"。**三个 Agent 全都通过了，但写出来的代码在生产环境 O(n²) 的性能，数据量一大就崩了。**

问题不是 Agent 不够多——是**角色不够不同**。三个编码员共享同一个盲区。没有人在问："这段代码的性能够吗？"

---

## 🤔 四个经典角色

2026 年最成熟的多 Agent 分解方式是这四个角色：

```
规划者（Planner）→ 执行者（Executor）→ 批评者（Critic）→ 验证者（Verifier）
```

| 角色 | 做什么 | 输出 | 判断方式 |
|------|--------|------|---------|
| **规划者** | 读需求，拆步骤，出规格 | 结构化计划 | -- |
| **执行者** | 按计划做实际工作 | 产物（代码/文档/数据） | -- |
| **批评者** | 对照计划审查产物质量 | 通过/拒绝 + 理由 | 主观（LLM 判断） |
| **验证者** | 跑确定性检查 | 通过/失败 + 证据 | 客观（代码/测试） |

---

## ⭐ 为什么验证者是承重角色

MAST 研究（Cemri 等，NeurIPS 2025）追踪了 1642 次多 Agent 失败。**21.3% 的直接原因是"没人验证"。** 剩下 79% 往往追溯到"有个检查悄悄失败或根本没跑"。

**一个真实的对比：**

PwC 在 CrewAI 上部署了一套文档审核系统。初始准确率：10%。

他们加了一个结构化验证循环——每个 Agent 的输出必须通过独立验证。准确率：**70%。** 7 倍提升。

这一个角色值 7 倍。

---

## 🧠 批评者 vs 验证者——别搞混

| | 批评者 | 验证者 |
|---|---|---|
| 本质 | 另一个 LLM 在"看" | 代码在"测" |
| 擅长 | 品味、可读性、设计合理性 | 功能正确性、性能、安全漏洞 |
| 盲区 | 看起来对但实际跑不通 | 跑通了但设计稀烂 |
| 成本 | 便宜（一次 LLM 调用） | 贵（需要沙箱、测试套件） |

**两者都要用。** 先跑批评者（便宜，抓设计问题），再跑验证者（慢但抓硬 bug）。

## 🏆 焊死在脑子里的东西

1. **不要三个同质的 Agent——要四个不同角色的 Agent。** 规划、执行、批评、验证。各司其职。

2. **验证者必须是确定性的。** 不能是"我觉得可以"——必须是"测试通过了"。

3. **批评者 + 验证者互补。** 先跑便宜的（LLM 审查），再跑贵的（代码测试）。

4. **加一个验证者 = 可能提升 7 倍准确率。** PwC 的生产数据，不是实验室数据。

---

## 📝 练习答案

**自动写单元测试系统：**

1. **规划者：** 读源代码 → 输出"需要覆盖的测试用例列表"
2. **执行者：** 读列表 → 写出测试代码
3. **批评者：** 看测试代码 → 判断"覆盖全了吗？边界条件测了吗？"
4. **验证者：** 实际跑测试 → 通过/失败 + 覆盖率报告

**验证标准：** 覆盖率 > 80% + 全部测试通过 = 成功。

---

> **多 Agent 系统的质量不取决于 Agent 的数量——取决于角色的多样性。三个编码员不如一个规划者+一个执行者+一个批评者+一个验证者。差异产生质量，同质产生盲区。**
