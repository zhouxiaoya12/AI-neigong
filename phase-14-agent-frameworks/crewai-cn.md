

# CrewAI：基于角色的工作组与工作流

> CrewAI 是 2026 年级别基于角色的多 Agent 框架。四个原语：Agent、Task、Crew、Process。两种顶层形态：Crew（自主的、基于角色的协作）和 Flow（事件驱动的、确定性的）。文档直言不讳："对于任何生产就绪的应用，从 Flow 起步。"

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** Phase 14 · 12（工作流模式）、Phase 14 · 14（Actor 模型）
**用时：** 约 75 分钟

## 学习目标

- 说出 CrewAI 的四个原语（Agent、Task、Crew、Process）及其各自拥有的属性。
- 区分顺序、层级和计划中的共识流程；为不同工作负载选择。
- 区分 Crew（自主的基于角色）和 Flow（事件驱动的确定性），并解释文档的生产建议。
- 用 `@tool` 装饰器和 `BaseTool` 子类挂接工具；对结构化输出 vs 自由文本做推理。
- 说出四种 CrewAI 记忆类型及其各自的回报点。
- 实现一个基于标准库的三 Agent 工作组（研究员、作者、编辑）来生成一份简报。
- 识别三种 CrewAI 失败模式：提示膨胀、管理器 LLM 税收、脆弱的转交。

## 问题

采用多 Agent 框架的团队都会撞到同一堵墙。"自主协作"在 demo 里听起来很棒。然后客户提交了一个 bug，你需要确定性回放。或者财务问一个 LLM 路由的工作组每次运行成本是多少。或者值班需要知道哪个 Agent 在凌晨三点卡住了。

自由形式的 LLM 路由工作组无法干净地回答这些问题。纯 DAG 能全部回答，但丢失了头脑风暴 Agent 需要的探索形态。

CrewAI 的拆分诚实地面对了这个 trade-off。Crew 用于协作的、基于角色的、探索性的工作。Flow 用于事件驱动的、代码拥有的、可审计的生产。同一个框架，两种形态，按场景选。

## 概念

### 四个原语

CrewAI 的界面很小。记住这些，其余都是配置。

- **Agent。** `role + goal + backstory + tools + (可选) llm`。背景故事（backstory）是关键负载。它塑造语气、判断、以及 Agent 何时停止。工具是 Agent 可以调用的函数（详见下文）。
- **Task。** `description + expected_output + agent + (可选) context + (可选) output_pydantic`。一个可复用的工作单元。`expected_output` 是契约。`context` 列出其输出被传入的上游任务。`output_pydantic` 强制结构化输出。
- **Crew。** 容器。拥有 Agent 列表、Task 列表、Process，以及可选的 `memory` + `verbose` + `manager_llm` 设置。
- **Process。** 执行策略。顺序（Sequential）、层级（Hierarchical）、共识（Consensus，计划中）。选择运行的形态。

Agent 不直接看到彼此。Task 引用 Agent。Crew 编排 Task 的顺序。Process 决定谁选择下一个 Task。这就是完整的思维模型。

> **已验证：** CrewAI 0.86（2026-05）。较新版本可能重命名或合并流程类型；在依赖特定形态前，请检查 [CrewAI Processes 文档](https://docs.crewai.com/concepts/processes)。

### 顺序 vs 层级 vs 共识

- **顺序（Sequential）。** 任务按声明顺序运行。任务 N 的输出作为 `context` 对任务 N+1 可用。成本最低。最可预测。当顺序固定时使用。
- **层级（Hierarchical）。** 一个管理器 Agent（独立的 LLM 调用）在专家组之间路由。CrewAI 从你的 `manager_llm` 配置或默认值生成管理器。管理器每轮选择下一个任务，并可以拒绝或重新路由。当你有四个或更多专家且顺序确实依赖先前输出时使用。
- **共识（Consensus）。** 计划中，尚未在公共 API 中实现。文档为该名称预留了未来的投票型流程。目前不要依赖它。

层级流程在每轮之上增加一次 LLM 调用（管理器）。令牌成本在五步运行上可能翻三倍。只在需要路由时才为此付费。

### Crew vs Flow

这是文档在 2026 年引领的框架。

- **Crew。** LLM 驱动的自主。框架在运行时选择形态。适合：研究、头脑风暴、初稿、任何路径本身就是答案一部分的场景。难以回放。难以测试。原型化成本低。
- **Flow。** 事件驱动图，由你拥有。`@start` 标记入口。`@listen(topic)` 标记一个步骤，当另一个步骤发出该主题时触发。每个步骤是普通 Python（可以在内部调用 Crew）。适合：生产。可观察。可测试。确定性。

文档 2026 年的生产建议：从 Flow 起步。在自主性值得其成本时，将 Crew 作为 `Crew.kickoff()` 调用折叠进 Flow 步骤中。Flow 给你审计边界，Crew 给你探索。组合，不要二选一。

### 工具集成

三种方式给 Agent 配备工具。选最简单的那个。

1. **`@tool` 装饰器。** 纯函数变成工具。签名即 schema；docstring 是 LLM 看到的描述。最适合一次性辅助。

 ```python
 from crewai.tools import tool

 @tool("Search the web")
 def search(query: str) -> str:
     """返回查询的 top 结果。"""
     return run_search(query)
 ```

2. **`BaseTool` 子类。** 基于类的工具，带有显式参数 schema、异步支持、重试。当工具有状态（客户端、缓存）或需要结构化参数时使用。

 ```python
 from crewai.tools import BaseTool
 from pydantic import BaseModel

 class SearchArgs(BaseModel):
     query: str
     limit: int = 10

 class SearchTool(BaseTool):
     name = "web_search"
     description = "搜索网络并返回 top 结果。"
     args_schema = SearchArgs

     def _run(self, query: str, limit: int = 10) -> str:
         return self.client.search(query, limit=limit)
 ```

3. **内置工具包。** CrewAI 提供第一方适配器：`SerperDevTool`、`FileReadTool`、`DirectoryReadTool`、`CodeInterpreterTool`、`RagTool`、`WebsiteSearchTool`。一次导入即可接入。

结构化输出使用 Pydantic。在 Task 上传递 `output_pydantic=MyModel`。CrewAI 根据模型验证 LLM 响应，要么强制转换，要么重试。将其与严格的 `expected_output` 字符串配对。自由文本输出适合草稿；结构化输出是下游 Flow 可以消费的。

### 记忆钩子

CrewAI 开箱提供四种记忆类型。它们是可组合的：一个 Crew 可以同时启用全部四种。

> **已验证：** CrewAI 0.86（2026-05）。近期版本通过统一的 `Memory` 系统路由，包裹了这四种存储。以下概念模型仍然成立，但公共类的 surface 可能在较新版本中收缩为单一 `Memory` 入口点；请检查 [CrewAI memory 文档](https://docs.crewai.com/concepts/memory) 了解当前 API。

- **短期（Short-term）。** 单次运行内的对话缓冲。结束时擦除。
- **长期（Long-term）。** 跨运行持久化。存储在向量数据库（默认 Chroma，可替换）。按与当前任务的相似度检索。
- **实体（Entity）。** 按实体存储事实。"客户 X 是企业方案用户。"按实体键控，而非按相似度。跨运行保存。
- **上下文（Contextual）。** 装配时检索。在 Agent 需要记忆的时刻拉取相关记忆，而非预加载。

在 Crew 上通过 `memory=True` 或按类型配置启用。由你配置的嵌入提供者支持（默认 OpenAI，可替换为本地）。记忆是 CrewAI 相较更轻薄框架的一个重要价值点；纯 LangGraph 需要你自己为每一项做挂接。

### CrewAI 适合的场景

- 三到六个 Agent，有命名的角色和协作工作流。起草、审查、规划、头脑风暴。
- LLM 关于下一步的判断是价值一部分的路由（层级）。
- 团队更愿意阅读 `role + goal + backstory` 而非阅读图定义的任何场景。

### CrewAI 不适合的场景

- 严格顺序的确定性 DAG。使用 LangGraph（第 13 课）。图形态是正确的抽象；CrewAI 的角色框架是摩擦。
- 亚秒级延迟预算。层级增加往返。即使是顺序模式也会序列化包含背景故事和先前输出的提示词。
- 单 Agent 循环。跳过框架；Agent 循环（第 01 课）加工具注册表更短。

第 17 课（Agent 框架权衡）将此展开为一个矩阵。简版：CrewAI 位于"协作式基于角色"的角落。

### 依赖形态

独立于 LangChain。Python 3.10 至 3.13。使用 `uv`。Star 数：参见 [crewAIInc/crewAI](https://github.com/crewAIInc/crewAI)（2026-05 快照）。AWS Bedrock 集成已有文档；供应商基准测试报告在 QA 工作负载上相比 LangGraph 有显著加速，但方法论（数据集、硬件、评估指标）未公布，因此仅将框架供应商的数据视为方向性参考。

### 此模式在何处会出错

- **背景故事造成的提示膨胀。** 每个 Agent 2000 词的背景故事，一个五 Agent 工作组在第一次工具调用前就烧光了上下文预算。将背景故事保持在 200 词以下。跨 Agent 复用短语；不要把公司风格重复五遍。
- **管理器 LLM 的令牌税。** 层级流程在每次专家调用前增加一次管理器 LLM 调用。在一个五任务工作组上，这是六次 LLM 调用而非五次，且管理器调用携带完整任务列表加上先前输出。切换到顺序模式，除非路由依赖输出。
- **脆弱的转交。** 任务 N 的 `expected_output` 是"一个大纲"。任务 N+1 将其作为 `context` 读取，试图解析三个部分。LLM 生成了四个。下游 Agent 即兴发挥。通过任务 N 上的 `output_pydantic` 修复，以便任务 N+1 读取类型化对象而非自由文本。
- **Crew 直入生产。** 自由形式的 Crew 在没有 Flow 包装的情况下部署到生产。输出变异性高；回放不可能；值班不能将坏运行与好运行做 diff。用 Flow 包装。

## 构建

`code/main.py` 实现两种形态的标准库版本加上一个三 Agent 工作组。

形态：

- `Agent`、`Task` 数据类，匹配 CrewAI 的 surface。
- `SequentialCrew.kickoff(inputs)` 按声明顺序运行任务，将输出作为 `context` 传递。
- `HierarchicalCrew.kickoff(topic)` 添加一个管理器 Agent，每轮选择下一个专家，在"done"时停止。
- `Flow`，带有 `@start` 和 `@listen(topic)` 装饰器、一个小型事件循环和追踪。
- `tool(name)` 装饰器，镜像 CrewAI 的 `@tool` 形态。
- `Memory`，包含 `short_term`、`long_term`、`entity` 存储；模拟相似度使用 numpy。
- 模拟 LLM 响应为按角色和输入前缀键控的硬编码字符串。无网络。确定性。

具体演示：研究员、作者、编辑工作组生成一份关于"agent engineering 2026"的简报。研究员拉取（模拟的）来源。作者起草。编辑收紧。同一工作组通过 Flow 运行以展示确定性形态。

运行：

```bash
python3 code/main.py
```

追踪覆盖：顺序工作组通过 `context` 传递输出、层级工作组的管理器选择（研究员、作者、编辑、然后"done"）、按显式主题（`researched`、`drafted`、`edited`）运行相同三步的 flow、通过 `@tool` 路由的工具调用，以及跨两次 kickoff 存活的长期记忆。

Crew 的追踪是流动的；管理器原则上可以重新排序。Flow 的追踪是固定的。这个选择就是这节课的核心。

## 使用

- **CrewAI Flow** 用于生产。即使 Flow 只有一个步骤调用 `Crew.kickoff()`。Flow 提供审计边界。
- **CrewAI Crew（顺序）** 用于明确排序的协作工作，特别是初稿和审查循环。
- **CrewAI Crew（层级）** 当路由依赖输出且你有四个或更多专家时。
- **LangGraph**（第 13 课）用于显式状态机、持久化恢复、严格排序。
- **AutoGen v0.4**（第 14 课）用于 Actor 模型并发和故障隔离。
- **OpenAI Agents SDK**（第 16 课）用于以 OpenAI 为主的产品，带转交和护栏。
- **Claude Agent SDK**（第 17 课）用于以 Claude 为主的产品，带子 Agent 和会话存储。

## 发布

`outputs/skill-crew-or-flow.md` 为任务选择 Crew 或 Flow，并脚手架化最小实现。硬拒绝：Crew 无背景故事、Flow 无显式主题、层级流程不足三个专家。

## 陷阱

- **背景故事作为调味料。** 它塑造输出。每个 Agent 测试三个变体；方差是真实的。选一个，冻结它。
- **跳过 `expected_output`。** 没有每任务的契约，下游任务接收 LLM 产生的任意内容。Crew 运行了；审计失败了。
- **记忆始终开启。** 长期记忆每次运行都写入。向量数据库增长。检索变得嘈杂。将写入范围限定到事实是持久性的任务。
- **管理器提示漂移。** 层级的管理器提示是隐式的。如果路由变得奇怪，在 verbose 模式下 dump 它并阅读。
- **Crew 中的工具副作用。** Crew 调用工具的次数可能超出预期。POST、DELETE、支付属于 Flow 步骤，永远不属于 Crew 工具。

## 练习

1. 将顺序 Crew 转换为 Flow。统计变异性下降的接触点。注意可读性下降的位置。
2. 为工作组添加实体记忆：关于客户的跨 kickoff 持久化事实。验证检索拉取了正确的实体。
3. 实现一个层级流程，其中管理器在作者输出至少有 3 段之前拒绝路由到编辑。追踪重试。
4. 为（模拟的）网页搜索挂接一个 `BaseTool` 子类。对比追踪形态与 `@tool` 装饰器版本。
5. 为编辑任务添加 `output_pydantic=Brief`，其中 `Brief` 有 `title`、`summary`、`sections`。让作者任务输出一次格式错误的 JSON；在追踪中验证 CrewAI 的重试行为。
6. 阅读 CrewAI 文档介绍。将玩具移植到真实的 `crewai` API。标准库版本跳过了哪些保证？
7. 将 AgentOps 或 Langfuse（第 24 课）接入真实运行。标准库版本漏了哪些追踪？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| Agent | "人设" | Role + goal + backstory + tools |
| Task | "工作单元" | Description + expected output + 负责人 + 可选结构化输出 |
| Crew | "Agent 团队" | Agent + Task + Process 的容器 |
| Process | "执行策略" | 顺序 / 层级 / 共识（计划中） |
| Flow | "确定性工作流" | 事件驱动的、代码拥有的、可测试的 |
| Backstory（背景故事） | "人设提示" | 塑造 Agent 语气和判断的提示 |
| `@tool` | "函数工具" | 将函数变成 Agent 可调用的工具的装饰器 |
| `BaseTool` | "类工具" | 基于类的工具，带参数 schema、重试、异步支持 |
| 实体记忆（Entity memory） | "按实体事实" | 按客户/账户/问题范围限定的记忆 |
| 长期记忆（Long-term memory） | "跨运行记忆" | 向量支持的在 kickoff 之间存活的记忆 |
| 上下文记忆（Contextual memory） | "即时检索" | 在 Agent 需要记忆的时刻拉取 |
| 管理器 LLM（Manager LLM） | "路由 Agent" | 层级流程中选择下一个任务的额外 LLM |
| `expected_output` | "任务契约" | 告诉 Agent（和审计）返回什么形态的字符串 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 14 课程中"多 Agent 框架"模块的收官之作。前面依次讲了 LangGraph（执行引擎）、OpenAI Agents SDK（轻量原语）、Claude Agent SDK（产线 harness）。这节课讲的是 CrewAI——一个以"角色驱动"为设计中心的框架。文档的最大亮点是 **Crew vs Flow 的诚实二分**："自主协作很酷，但生产先从确定性 Flow 开始。"这种"框架官方告诉你别先把框架最酷的功能用于生产"的坦率，在工具文档中极少见。适合已理解编排拓扑和三种执行引擎、正在评估"基于角色的多 Agent"方案的架构师。

### 二、知识结构梳理

**认知基础层：** 工作流模式（第 12 课）——Flow 是五种工作流模式的事件驱动实现。编排拓扑（第 13/28 课）——层级流程 = 监管者模式，顺序流程 = 提示链。Actor 模型（第 14 课）——CrewAI 的 Agent 是被动角色，不是 Actor，这个对比很重要。

**框架原语层：** 四个原语（Agent/Task/Crew/Process）是该框架的最小概念集合。Agent = 角色+目标+背景故事，Task = 描述+期望输出+负责人，Crew = 容器，Process = 执行策略。背景故事（backstory）被文档强调为"关键负载"——它不是可选装饰，是 Agent 行为的核心塑造者。

**操作模式层：** Crew vs Flow 是文档最核心的区分。Crew = LLM 自主路由（适合探索），Flow = 事件驱动确定性图（适合生产）。文档的推荐是"从 Flow 起步，需要时嵌入 Crew"——不是"选一个"，是"组合使用"。

### 三、核心洞察（备课时的关键理解）

1. **"背景故事是关键负载"是 CrewAI 与其他框架最本质的区别。** LangGraph 的节点是纯函数——没有角色。OpenAI Agents SDK 的 Agent = LLM + 指令 + 工具——没有背景故事。Claude Agent SDK 的子 Agent 是独立上下文——没有角色。只有 CrewAI 把一个 Agent 的**身份**作为一等概念：你是谁、你的目标是什么、你的背景是什么。背景故事不是 flavor ——它直接决定了 Agent 的输出语气、判断、以及"我什么时候该说做完了"。

2. **Crew vs Flow 的二分是诚实的架构 trade-off，不是营销。** 大多数框架说"我们又灵活又可靠"。CrewAI 的文档说："Crew 灵活但不可靠。Flow 可靠但不灵活。用 Flow 做生产，需要探索时嵌入 Crew。"这不是卖点——这是工程建议。

3. **层级流程的管理器 LLM 税收是一个可量化的成本决策。** 文档直言：五任务工作组，顺序流程 = 5 次 LLM 调用，层级流程 = 6 次（加上管理器调用，且管理器调用携带完整的任务列表和先前输出）。令牌成本翻倍到三倍。这个成本是否值得取决于"任务顺序是否确实依赖先前输出"——如果可以预先固定顺序，层级流程就是过度支付。

4. **`expected_output` 是 CrewAI 对"脆弱的转交"问题的回答。** 任务 N 产出自由文本，任务 N+1 基于它工作——这是跨 Agent 通信的经典断裂点。CrewAI 的解决方案：① `expected_output` 描述期望格式（松散契约），② `output_pydantic` 强制类型化（严格契约）。两者配合，下游 Agent 收到的不是"一段文本"，是"一个类型化对象"。

5. **四种记忆类型覆盖了 Agent 记忆的全部频谱。** 短期 = 运行内缓冲，长期 = 跨运行向量检索，实体 = 按实体键控的事实，上下文 = 即时拉取。LangGraph 只做前两种（状态中的 messages + 检查点存储）。CrewAI 将记忆作为一个完整的子系统，而不是状态的附属。

6. **Flow 的 `@listen(topic)` 是事件驱动架构在 Agent 框架中的直接应用。** 一个步骤说"我完成了科研"并发出 `researched` 主题。另一个步骤说"我会在有人完成科研后开始写作"并监听 `researched`。这是发布-订阅模式——和 AWS Step Functions 或 Temporal 中的事件驱动流转是同构的。Flow 不是"Crew 的简化版"——它是完全不同的编程模型。

7. **文档的"陷阱"部分比正文更有教学价值。** 五种陷阱——背景故事膨胀、跳过 expected_output、记忆始终开启、管理器提示漂移、Crew 中的工具副作用——不是理论，是生产中的伤疤。特别是最后一条（"POST/DELETE/支付属于 Flow 步骤，永远不属于 Crew 工具"）——是安全架构的底线。

### 四、教学建议

1. **用"角色扮演 vs 状态机"的对比开场。** 左边：LangGraph 的图——节点、边、检查点。右边：CrewAI 的 Crew——研究员、作者、编辑，各有性格。问学生"哪个更容易向产品经理解释？哪个更容易在凌晨三点调试？"——答案通常不同。

2. **背景故事不是装饰——用一个实验来证明。** 同一个 Agent，同一个任务——三次不同背景故事（"你是激进的创新者""你是保守的检查者""你是中立的分析者"）。展示输出差异。让学生感受"背景故事是工程控制，不是文学创作"。

3. **Crew vs Flow 适合用"同一个任务两种方式"来教。** 拿三 Agent 简报生成任务——一次用 Crew（顺序），一次用 Flow（事件驱动）。对比追踪：Crew 的 trace 可能有变异，Flow 的 trace 每次都一样。从差异中引出"Crew 适合探索，Flow 适合生产"的结论。

4. **四种记忆类型用"购物类比"来讲。** 短期记忆 = 你脑子里正在想的（本次购物）。长期记忆 = 你记得上周买过什么（向量检索）。实体记忆 = 你记得某家店的价格（按店名检索）。上下文记忆 = 走到货架前才想起"上次这个牌子不好用"（即时拉取）。

5. **`expected_output` + `output_pydantic` 的组合适合用一个对比演示。** 任务 A 产出自由文本 → 任务 B 接到后开始"即兴发挥"。同一个任务加上 `output_pydantic=Brief` → 任务 B 接到类型化对象 → 输出一致。结构化输出的价值在 Agent 间通信中会被放大。

6. **工具副作用在 Crew 中的风险是一个关键的"安全架构"讨论点。** 问学生"你的 Crew 里有一个退款工具——Crew 用了三次退款工具（一次正常，两次因为 LLM 幻觉）——你怎么审计？"答案：退款工具不属于 Crew 工具——属于 Flow 中的显式步骤。

7. **练习 1（Crew → Flow 转换）是最有教学价值的实操。** 让学生亲手体验"把自主性去掉后哪里变好了、哪里变差了"。这是"知道自己为什么选择 Flow"的唯一途径。

### 五、值得补充的内容

1. **CrewAI 与 LangGraph 在"确定性"谱系上的位置。** 画一个谱系：左边完全确定性（LangGraph DAG），右边完全自主（CrewAI 层级 Crew），中间是 Flow + 嵌入 Crew。这是选型决策的地图。

2. **共识流程（Consensus）的设计预测。** 虽然尚未实现，但文档预留了名称。它与第 25 课的辩论模式、以及 Voyager 的自我验证有什么异同？

3. **Flow 中的错误传播和重试语义。** `@listen` 步骤失败了——事件是否丢失？Flow 如何恢复？这与 LangGraph 的检查点恢复有什么区别？

4. **四记忆同时启用的上下文预算影响。** 四种记忆启用意味着每次 LLM 调用的提示词中注入检索到的记忆。这对上下文预算的影响需要量化。

5. **CrewAI 的"角色"概念在跨语言和跨文化场景下的表现。** 背景故事（backstory）是用英文写的——如果最终用户是中文或日文，角色设定怎么翻译和本地化？这是多语言 Agent 的隐形成本。

### 六、一句话总结

**CrewAI 的诚实是它的最大资产：它告诉你"自主协作很酷但不可靠，确定性工作流可靠但不灵活——从 Flow 起步，需要时嵌入 Crew。"这不是框架的妥协，是工程判断的 API 化。**

# 🎓 Agent 架构课：CrewAI——为什么"角色扮演"有时候就是你需要的全部

同学们好。我是你们的 Agent 架构老师。

前面六节课我讲了 LangGraph 的状态机、OpenAI Agents SDK 的五个原语、Claude Agent SDK 的产线 harness。这些都是很"硬"的框架——它们提供的是执行引擎、工具抽象、会话存储。

今天这节课讲一个很"软"的框架。CrewAI 的核心不是技术原语——是**角色。** 它的回答是："如果你的 Agent 有一个名字、一份简历、一个性格——它会做得更好吗？"

答案是：在很多场景下，是的。

## 不是每个 Agent 都需要是状态机

我先讲一个反直觉的事实：**你给产品经理解释 LangGraph 的时间，可能比 LangGraph 运行的时间还长。**

"你看，这是一个带不可变状态和检查点存储的有状态图，节点是纯函数，状态通过类型化字典定义，恢复通过 session_id 做检查点加载……"

产品经理看着你："所以那个写博客的 Agent，它写得好吗？"

CrewAI 给的是另一种回答。

```python
researcher = Agent(
    role="技术研究员",
    goal="找到 Agent 工程领域的最新趋势",
    backstory="你是一位资深研究员，从业十年，擅长从海量信息中提取关键洞察。"
)

writer = Agent(
    role="技术作者",
    goal="将研究结果写成一篇清晰的简报",
    backstory="你是一位体验丰富的技术写作者，擅长将复杂概念说清楚。你讨厌 jargon。"
)

editor = Agent(
    role="编辑",
    goal="确保简报准确、简洁、没有废话",
    backstory="你是一位严格的编辑，日报编辑背景。你不会放过任何模糊的表述。"
)
```

三个角色。三个背景故事。一个工作组。

你知道产品经理会说什么吗？"我懂了。研究员找资料，作者写，编辑改。"

这不是技术极简主义——这是一个不同的设计哲学：**当角色逻辑足够清晰时，Agent 的行为不需要通过图来定义——通过"你是谁"和"你的目标是什么"来定义就够了。**

## 四个原语：比你以为的更小

CrewAI 的概念表面非常小。Agent、Task、Crew、Process。就四个。

**Agent** = 一个角色 + 一个目标 + 一个背景故事 + 一组工具。注意：背景故事不是可选的。文档明确说："背景故事是关键负载。它塑造语气、判断、以及 Agent 何时停止。"如果你写的背景故事是"你是一个有用的助手"——你没在用它。如果你写的背景故事是"你是一位资深编辑，从业十年，日报编辑背景，从未漏过一个语法错误，但你也知道什么时候该放行一个风格选择"——你开始真正用它了。

**Task** = 一个描述 + 一个期望输出 + 一个指定的 Agent。`expected_output` 是关键。它不是提示词——是契约。任务 N 说"期望输出是三段结构的大纲"，任务 N+1 基于这个契约工作。如果没有 `expected_output`，任务 N+1 接到的是"一段 LLM 大脑里的东西"——它不知道应该 parse 几个段落。

**Crew** = 容纳 Agent 和 Task 并选择一个进程的容器。**Process** = 顺序、层级，或计划中的共识。顺序 = 任务按你声明的顺序跑。层级 = 任务间插入一个管理器 LLM，它每轮决定"下一个该谁做"。共识 = 未来会有，现在别指望。

这就是全部。四个概念。Agent 不直接调用 Agent。Task 引用 Agent。Crew 编排 Task。Process 决定谁选下一个。整个思维模型可以在一张便利贴上画完。

## Crew vs Flow：CrewAI 最诚实的决定

CrewAI 做了一个很少有框架敢做的事：**它告诉你不要把最酷的功能用于生产。**

**Crew**：LLM 自动路由。框架在运行时决定谁做什么。自由、灵活、探索性——但输出不可预测，出问题时无法回放，凌晨三点值班工程师没法 diff 一个坏运行和一个好运行。

**Flow**：事件驱动图，你拥有控制权。`@start` 标记入口。`@listen(topic)` 标记"当某某做完时我开始做"。每个步骤是普通 Python 函数——你可以在里面调用一个 Crew，但外层是确定性的。

文档的原话是："对于任何生产就绪的应用，从 Flow 起步。"

为什么？因为在生产系统中，你需要的东西 Crew 给不了你：你需要审计追踪（"谁在什么时间做了什么"），Flow 的事件图就是审计追踪。你需要确定性回放（"同样的输入每次得到同样输出"），Flow 的函数调用就是确定性的。你需要当 bug 出现时能定位"是第几步出错了"——Flow 的 `@listen` 每一步都是显式的，Crew 的内部路由是 LLM 自由选择的。

**正确的组合方式：Flow 做骨架，Crew 做肌肉。** 你的生产流水线是一个 Flow——事件驱动、可审计、确定性。但在某个步骤里——比如"研究分析"这个步骤——你嵌入一个 Crew。研究员、作者、编辑在里面自主协作。Flow 给这个自主盒子划定了边界和审计点。

## 层级流程的成本：管理器 LLM 税收

层级流程（Hierarchical Process）听起来很酷：一个管理器 Agent 自动决定"研究员先跑，然后作者，然后编辑"。不需要手动排任务顺序。

但文档告诉你一个其他框架未必会告诉你的数字：**令牌成本可能翻三倍。**

为什么？因为层级流程在每次专家调用之前，插入一次管理器 LLM 调用。五个任务 = 五次专家调用 + 五次管理器调用 = 十次 LLM 调用。而且管理器调用携带整个任务列表和所有先前输出——它的提示词比专家的长得多。

对比顺序流程：五个任务 = 五次专家调用。没有管理器税。

什么时候值得付这个税？当你不确定任务顺序的时候。当你确实需要 LLM 的动态判断来决定"下一步该做什么"的时候。如果你可以事先固定顺序（"研究员 → 作者 → 编辑"是固定的）——顺序流程。不要为不需要的动态路由支付税款。

## 四种记忆：从"这次对话"到"我认识你"

CrewAI 的记忆系统分四层，每层解决不同的问题。

**短期记忆：** 就是这次 run 内的对话缓冲。研究员说了什么，作者引用了什么——都在。run 结束就清空。这是最轻的——不需要向量数据库。

**长期记忆：** 跨 run 持久化。上周你研究过"Agent 安全性"这个话题，这周又研究"Agent 安全性"——上次的研究结论通过向量相似度被检索回来。存哪？默认 Chroma 向量数据库，可换。这是 CrewAI 比纯 LangGraph 堆内存成本更低的地方——你不必自己从头挂接向量存储。

**实体记忆：** 不是按话题检索——是按实体检索。"客户 X 是企业方案。"——不管当前任务是什么，只要涉及客户 X，这条事实就被拉取。实体记忆的键是客户 ID，不是语义向量。

**上下文记忆：** 前面三种是"加载时注入"。上下文记忆是"需要时拉取"。Agent 在推理中碰到一个概念，在那一刻去检索相关记忆。不是预加载——是即时。

四层全部启用 = 每次 LLM 调用的提示词里注入更多上下文。文档警告："不要把记忆始终开启。只对事实是持久性的任务做写入。"记忆不是免费的——它在吃你的上下文预算。

## 三条命脉：输出契约

让我讲一个具体场景，说明为什么 CrewAI 的三个输出相关概念是它的命脉。

研究员做完研究，产出是一段自由文本："Agent 工程在 2026 年有三个趋势：多 Agent 编排、技能库、上下文压缩。"作者接到这个，开始写。它写了几段，其中有一段标题是"多 Agent 编排"，但内容混进了技能库的东西。编辑接到，改来改去，但总有些信息混搭。

问题在哪？**任务之间的接口是脆弱的。** 研究员给作者的是"一段文本"——作者不知道应该怎么结构化地消费它。

CrewAI 给了三个层次的修复：

1. **`expected_output`**（松散契约）："期望输出是一个 JSON，包含三个字段：trends（趋势列表）、sources（来源列表）、confidence（置信度 0-1）。"作者知道会收到什么格式。

2. **`output_pydantic`**（严格契约）：研究员的任务挂了一个 Pydantic 模型 `ResearchOutput(trends: list[str], sources: list[str], confidence: float)`。输出不是 JSON 字符串——是被验证过的类型化对象。如果 LLM 输出格式不对，CrewAI 自动重试或强制转换。

3. **`context`**（上下游链接）：作者任务的 `context = [research_task]`——明确告诉框架"作者的上下文包括研究员任务的输出"。

三者配合：研究员产出通过了 Pydantic 验证的 `ResearchOutput` → 作者在 `context` 里收到它 → 作者的 `expected_output` 描述它要产出什么。自由文本的脆弱性被这三层逐步收紧。

## 五种陷阱：生产中才会发现的事

**陷阱一：背景故事膨胀。** 五个 Agent，每个 2000 词的背景故事。第一个工具还没调用，上下文预算已经烧光。规则：背景故事 ≤ 200 词。短，具体，可测试。

**陷阱二：跳过 `expected_output`。** "我期望输出是一份简报。"— 什么格式？几个部分？什么语气？没有这些，下游任务在工作时是盲的。

**陷阱三：记忆始终开启。** 长期记忆每次 run 都写入。100 次 run 后向量 DB 里有 100 条记录，检索开始噪声化。只在记忆有价值时才写。

**陷阱四：管理器提示漂移。** 层级流程的管理器提示是隐式的——你看不到它。突然有一天它开始把"编辑"路由到"作者"前面。原因：管理器提示被不断增长的任务列表和背景故事挤压了。在 verbose 模式下 dump 管理器提示。

**陷阱五：Crew 中的工具副作用 ⚠️。** 这是最危险的。你的 Crew 里有一个"退款"工具。LLM 在推理中调用了三次——第一次是正常的，第二次是因为误解用户说要"退款"（其实用户只是问怎么退款），第三次是幻觉。三次退款操作。**任何有副作用的工具（POST、DELETE、支付、退款、发邮件）——不放 Crew 里。放 Flow 的显式步骤里。** Crew 工具只能是读操作、搜索、计算——纯函数。

## 什么时候选 CrewAI，什么时候选别的

- **CrewAI：** 你的任务有明确的角色分工（研究员、作者、编辑）。你的团队更喜欢"角色描述"而非"图定义"。你需要四种记忆类型中的至少两种。你的工作负载以协作为主，而非严格的确定性。

- **LangGraph：** 你需要步级状态管理和检查点恢复。你的任务是严格确定性 DAG。

- **OpenAI Agents SDK：** 你需要转交、护栏、零配置追踪。你的产品以 OpenAI 模型为主。

- **Claude Agent SDK：** 你需要子 Agent 的上下文隔离。你的产品以 Claude 模型为主。

## 结课清单

如果你在用 CrewAI：

1. Agent 的背景故事是工程控制——不是文学创作。短（≤200 词）、具体、可测试。
2. 每个 Task 必须有 `expected_output`——它是任务间的契约。
3. 结构化输出（`output_pydantic`）是防止跨 Agent 通信断裂的最小化设计。
4. 从 Flow 起步做生产——即使 Flow 只有一步调用 `Crew.kickoff()`。
5. 层级流程的令牌成本是顺序的 2-3 倍——只在路由依赖输出时才付这个税。
6. 四种记忆不全开——只在记忆有价值时才写。
7. 有副作用的工具不属于 Crew 工具——属于 Flow 步骤。

**最后一句话：**

**CrewAI 教给 Agent 架构师的最重要的事不是"怎么用角色"——是"什么时候角色够用了，什么时候你需要把自主性关进确定性的盒子里"。Flow 是盒子，Crew 是盒子里的大脑。先造盒子，再放大脑。**
