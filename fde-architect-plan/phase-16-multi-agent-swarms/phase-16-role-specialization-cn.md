# 角色专业化——规划者、批评者、执行者、验证者

> 2026 年最常见的多智能体分解：一个智能体规划，一个执行，一个批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示中的 SOP——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过"聊天链"串联设计师、程序员、审查员、测试员，并带有"沟通式去幻觉"（智能体显式请求缺失的细节）。验证者是承重角色：Cemri 等人（MAST，arXiv:2503.13657）表明每个多智能体故障都可以追溯到缺失或损坏的验证。PwC 报告在 CrewAI 中通过结构化验证循环实现了 7 倍准确率提升（10% → 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置课程：** 第16阶段 · 04（原语模型），第16阶段 · 05（监管者）
**时间：** 约60分钟

## 问题

泛化的多智能体系统产生泛化的输出。群聊中的三个编码员写出同一平庸代码的三种变体。你可以添加更多智能体，添加更多轮次，但仍然无法跨越质量门槛。

解决方案不是更多智能体——而是*不同*的智能体。分配不同的角色。给批评者规划者没有的工具。给验证者一个客观的测试套件。现在系统有了基于证据的内部分歧和纠正，而不仅仅是并行猜测。

## 概念

### 四个经典角色

**规划者。** 读取目标，生成步骤列表或规格说明。工具：知识检索、文档。输出：结构化计划。

**执行者。** 一次读取一个计划步骤，生成产物。工具：实际工作工具（代码编译器、Shell、API 客户端）。输出：产物。

**批评者。** 对照规划者的意图读取执行者的输出。工具：对产物的只读访问、静态分析。输出：接受/拒绝及理由。

**验证者。** 读取产物并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败及证据。

批评者是主观的、有意见的，通常基于大语言模型。验证者是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示：

- **产品经理**编写 PRD。
- **架构师**生成系统设计。
- **项目经理**拆分任务。
- **工程师**实现。
- **QA 工程师**运行测试。

每个角色有严格的输入/输出模式。角色提示说明角色是*什么*以及必须*产出什么*。`Code = SOP(Team)` 表述——确定性的 SOP 将一个大语言模型团队变成可预测的流水线。

### ChatDev 的沟通式去幻觉

ChatDev 添加了一个关键动作：当执行者需要一个计划中没有的特定细节时，它在继续之前显式询问设计师。这防止了经典的大语言模型失败——切实际地发明细节。

实现：角色提示包含"当你需要你没有收到的特定信息时，在生成输出之前按名称询问相关角色。"

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 个多智能体执行故障。21.3% 是验证缺口——系统发出了没有人检查过的答案。其余 79% 往往追溯到"有一个检查静默失败或从未运行"。验证是承重角色。

PwC 报告（CrewAI 部署，2025 年）添加结构化验证循环将准确率从 10% 提升到 70%。一个角色带来 7 倍提升。

### 批评者 vs 验证者

- 批评者是一个大语言模型审查产物的质量。主观的。可能被看似合理的文本欺骗。
- 验证者是在产物上运行的确定性程序。客观的。给出通过/失败及证据。

两者都使用。批评者捕捉验证者无法表达的品味问题。验证者捕捉批评者看不到的 bug，因为它们只在运行时出现。

### 反模式

系统中的每个角色都是大语言模型，每个角色的输出是"看起来不错"。这是经典的 MAST 故障模式。添加至少一个验证者，其通过/失败由代码决定，而非大语言模型。

### 框架映射

- **CrewAI**——`Agent(role, goal, backstory)` 是教科书式的专业化表面。
- **LangGraph**——节点可以有专业化的提示；边强制执行流水线。
- **AutoGen**——带有单字名称的角色专用 ConversableAgent 在 GroupChat 中。
- **OpenAI Agents SDK**——角色专业化智能体之间的转交工具。

## 构建它

`code/main.py` 实现了一个构建简单 Python 函数的 4 角色流水线：

- **规划者**生成规格说明。
- **执行者**生成代码字符串。
- **批评者**（大语言模型模拟）标记明显问题。
- **验证者**在沙箱中运行生成的代码（`exec`）并对照测试用例。

演示运行两次：一次执行者生成正确代码（批评者 + 验证者都通过），一次执行者产生偏离规格的代码（批评者因为代码看起来合理而漏掉 bug，验证者因为测试失败而捕获它）。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-role-designer.md` 接受任务并生成角色名单（3-5 个角色）、每个角色的输入/输出模式以及验证者检查。在将智能体接入框架之前使用。

## Ship It

检查清单：

- **至少一个确定性验证者。** 绝不全部是大语言模型。
- **每个角色有显式的输入/输出模式。** 规划者返回规格说明，不是散文；执行者读取该模式。
- **沟通式去幻觉。** 执行者在缺少信息时必须询问规划者；绝不发明信息。
- **批评者/验证者排序。** 先运行批评者（便宜，捕获设计问题），再运行验证者（慢，捕获 bug）。
- **循环预算。** 在升级给人类之前最多 2 轮批评者-执行者修订循环。

## 练习

1. 运行 `code/main.py` 并观察验证者如何捕获批评者漏掉的 bug。添加一个静态分析检查（计算 `return` 出现次数）作为额外的验证者。它捕获到了运行时测试漏掉的什么？

2. 添加第 5 个角色："需求分析员"，将用户愿望转化为规划者可用的规格说明。哪些沟通式去幻觉请求应该向上流向它？

4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。识别沟通式去幻觉在哪里切断了一个否则会无限的循环。

5. PwC 的 7 倍准确率提升来自验证循环。假设三个添加验证者不会有帮助的任务——其中确定性的正确性检查不可能或成本高得惊人。

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一份将学术研究（MetaGPT、ChatDev、MAST）与工业实践（PwC 7 倍提升）紧密结合的角色专业化教材。文档最突出的贡献是清晰区分了"批评者"和"验证者"——前者是 LLM 的主观审查，后者是代码的客观检查——这一区分在大多数多智能体教材中是模糊甚至缺失的。"全大语言模型反模式"（所有角色输出'看起来不错'）是一个精准的诊断工具，将 MAST 的抽象故障分类法转化为工程师可直觉感知的警告。

### 二、知识结构梳理

**认知基础：** 泛化多智能体（三个编码员）输出平庸，因为缺乏结构化的内部分歧。解决方案不是更多智能体，而是不同角色——规划者设定方向，执行者实现，批评者主观检查，验证者客观通过/失败。

**工程模式：** MetaGPT 的 SOP 编码（严格输入/输出模式→确定性流水线）、ChatDev 的沟通式去幻觉（缺细节时询问而非编造）、批评者-验证者双重检查的经济性排序（便宜的主观审查先于昂贵的客观测试）。

**落地实践：** PwC 7 倍提升的定量证据；MAST 21.3% 验证缺口的故障根因；2 轮修订循环的上限预算；CrewAI/LangGraph/AutoGen/OpenAI Agents SDK 的框架映射。

### 三、核心洞察

1. **"批评者 vs 验证者"的二分法是整个文档体系中最有价值的单一概念。** 批评者是 LLM 的"品味审查"（主观、可被欺骗），验证者是代码的"事实检查"（客观、不可欺骗）。两者需要共存。为什么重要？它直接回应了"AI 审查 AI 是否可靠"的递归信任问题——答案是不可靠，所以必须有非 AI 的检查。

2. **"沟通式去幻觉"是一个优雅的故障预防机制。** 当执行者缺少细节时，它的选择不是"编造一个"而是"询问规划者"。这在工程上是成本很小的修改（一行角色提示），但在质量上是质的飞跃。为什么重要？它证明了多智能体系统的质量有时不完全取决于模型能力，而取决于信息流的设计——让信息在正确的时刻流向正确的角色。

3. **PwC 的 7 倍提升（10%→70%）是一个教科书级的 ROI 案例。** 添加一个验证者角色的成本是增量令牌消耗，收益是准确率翻 7 倍。为什么重要？它为"是增加更多智能体还是增加验证"这个问题提供了确凿的定量证据——增加验证者。

4. **21.3% 的验证缺口意味着五分之一的多智能体故障是可预防的。** MAST 分类法说的不是"验证很难"，而是"验证被跳过了"。不是能力问题，是流程问题。为什么重要？它将问题从"我们需要更好的验证技术"重新定义为"我们需要确保验证不被跳过"——这是一个文化/流程问题，不是技术问题。

5. **SOP 编码的哲学是"确定性从约束中涌现"。** MetaGPT 的每个角色有严格的输入/输出模式。当角色提示说"你必须以特定 JSON 格式输出"时，大语言模型的行为可预测性大大提高。为什么重要？它揭示了一个反直觉的真相：对 LLM 施加更多结构约束（减少自由度）反而提高系统的稳定性。

6. **"全大语言模型反模式"是一个可操作的代码审查检查。** 审查多智能体实现时，问一个简单问题："哪个角色的输出是由代码而非大语言模型决定的？"如果答案是"没有"，系统有 MAST 漏洞。为什么重要？它提供了一个一行诊断，无论系统多复杂都适用。

### 四、教学建议

1. **先让学生构建一个"全大语言模型"的 3 角色系统（规划者+执行者+审查员），再添加验证者。** 展示审查员如何对明显有 bug 的代码说"看起来不错"。MAST 反模式必须被亲历才能被真正理解。

2. **把批评者和验证者做成两张"角色卡片"。** 批评者卡片写"主观的、有意见的、可以被欺骗的"；验证者卡片写"客观的、确定性的、不可欺骗的"。让学生根据任务类型决定什么时候该用哪个，什么时候两个都要。

3. **让学生扮演"沟通式去幻觉"中的询问流程。** 一个学生是规划者（给出模糊计划），另一个是执行者（必须询问缺失细节才能继续）。然后反过来——执行者编造细节，审查员发现产物偏离计划。对比两种结果。

4. **用 PwC 的 7 倍提升作为"投资回报"练习。** 让学生估算：添加验证者增加的令牌成本 vs 不添加验证者的错误成本（在不同业务场景中）。让"添加验证者"从一个技术建议变成一个经济决策。

5. **让学生阅读 MetaGPT 或 ChatDev 论文的角色提示设计。** 标注出输入格式、输出格式和角色描述。对比这些学术设计与自己写的角色提示的质量差异。

6. **将 4 个角色映射到真实软件开发团队。** 让学生识别：在一个真人团队中，谁在做规划、执行、批评和验证？这些角色在真人团队中如何沟通？AI 智能体版的突破和局限在哪里？

### 五、值得补充的内容

1. **验证者设计的图谱。** 不同类型任务的最优验证策略不同——代码任务用单元测试，数据任务用模式验证，文档任务用事实核查。一个系统化的"按任务类型选择验证者"指南会很有价值。

2. **批评者提示的工程技巧。** "你是代码审查员" vs "你是代码审查员，必须找到至少一个问题"——后者产生的输出质量完全不同。强制性约束对 LLM 输出质量的影响值得单独讨论。

3. **验证报告的可读性问题。** 验证者产生"测试 X 失败：预期 Y，得到 Z"的输出——这对人类是有用的，但对上游智能体（规划者）的重新规划有多有用？需要什么格式的验证输出才能成为有效的反馈信号？

4. **角色间的信任模型。** 如果批评者总是通过执行者的输出（校准错误的批评者），系统如何检测？"反向验证"——使用另一个批评者审查批评者的判断——是否可行？成本是否合理？

5. **SOP 模式在非软件工程领域的泛化。** MetaGPT 和 ChatDev 都聚焦于软件开发。相同的 SOP 编码方法如何应用于法律合规审查、医疗诊断或金融分析？角色集合需要如何调整？

### 六、一句话总结

多智能体系统的质量不取决于你用了多少智能体，而取决于你是否设置了一个不依赖 LLM 的客观验证者——批评者发现品味问题，验证者捕获无法狡辩的 bug，两者共存才能阻止"看起来不错"这个最常见的故障模式。

# 🎓 Agent 架构课：角色专业化——没有验证者的多智能体系统等于没有刹车的车

同学们好。我是你们的FDE工程老师，今天讲的是 多 Agent 角色设计。

**三个泛化的 Agent 写出同一段平庸代码的三种变体。解决方案不是加更多 Agent——是加不同角色的 Agent。**

四个经典角色：① **规划者**——读目标，出步骤。② **执行者**——按步骤产出产物。③ **批评者**——主观审查，给意见。④ **验证者**——客观检查，跑测试。批评者是"我觉得这里不对"，验证者是"类型检查失败，第 47 行返回类型不匹配"。

MAST 论文的结论：每个多 Agent 故障都可以追溯到缺失或损坏的验证。PwC 在 CrewAI 中通过结构化验证循环实现了 7 倍准确率提升（10%→70%）。MetaGPT 将其形式化为 SOP——产品经理、架构师、工程师、QA——遵循 Code = SOP(Team)。

## 结课清单

1. **验证者是承重角色。** 没有客观验证 = 没有质量保证。
2. **批评者 ≠ 验证者。** 前者主观有意见，后者跑客观测试。
3. **泛化 Agent 输出泛化结果。** 专业化角色是质量杠杆的最便宜支点。

最后一句话：**三个泛化的智能体不如一个规划者+执行者+验证者。角色专业化不是加人——是给每个人配不同的刀。**

# 💼 从业者故事：角色专业化——没有验证者的多智能体系统等于没有刹车的车

去年我 review 了一个同事的多智能体 PR。他做了一个"代码审查系统"：规划者写 spec，执行者写代码，审查者检查。听起来很完美对吧？三个角色，各司其职，像一个小型软件团队。

我在 PR 下面写了两个字："验证者呢？"

他回："审查者就是验证者啊。"

我说："你的审查者是一个 LLM。你把一段有 bug 的代码给它，它说'看起来不错'的概率是多少？"

他沉默了。然后做了一个实验：故意在代码里埋了一个 `else if` 条件写反的 bug，让审查者 review。审查者给了满分，评论是"代码清晰，逻辑合理"。那个 `if (role === 'admin' || role === 'superadmin')` 被写成了 `if (role === 'admin' && role === 'superadmin')`——一个永远为 false 的条件。但审查者读的是源代码，不是执行路径，它看到 `admin` 和 `superadmin` 都在条件里就觉得没问题。

**这就是为什么"全大语言模型"的多智能体系统跟没有刹车一样。所有角色靠嘴说，没有一个角色动手跑。**

## 四个角色的本质——不是分工，是信任机制

很多人把角色专业化理解成"把活分给不同的人"。那只是表面。角色专业化的本质是**在系统内部建立不信任机制。**

一个单智能体的信任模型是："我相信你写的东西是对的。"

一个多智能体系统的信任模型应该是："我不相信你。让批评者质疑你，让验证者跑测试打你的脸。"

四个经典角色翻译成大白话：

- **规划者**：说"我们要做什么"。它不写代码，它写 todo list。
- **执行者**：说"我做完了"。它不检查自己，它只负责产出产物。
- **批评者**：说"我觉得有问题"。它是 LLM，主观的。它可以被一段写得很漂亮的 bug 欺骗。但它快，它便宜，它先跑。
- **验证者**：说"测试失败了"。它不跟你辩论。它跑代码，看结果。第 37 行预期 `true`，实际 `false`。无法狡辩。

**批评者和验证者的区别，是整个多智能体架构最重要的一对概念。** 批评者是品酒师——他可以告诉你酒好不好喝，但他可能被漂亮的包装骗了。验证者是呼气测醉器——它不关心酒好不好喝，它只告诉你血液酒精含量。

MAST 论文（Cemri 等人）追踪了 1642 个多智能体故障，21.3% 的直接原因是验证缺口——答案发出去了，但没有人跑过检查。剩下的 79% 呢？追根溯源，也往往是"有一个检查静默失败了，或者从未运行"。所以现实是：**几乎 100% 的多智能体故障都跟验证有关。** 这不是夸张，这是数据。

## 三个让你的系统从"能跑"到"能上线"的设计模式

**模式一：MetaGPT 的 SOP 编码——你不给格式，它就给你散文。**

MetaGPT（arXiv:2308.00352）把软件开发流程编码成了角色提示。产品经理必须输出特定格式的 PRD。架构师必须输出特定格式的系统设计。每个角色的 prompt 里，一半在说"你是谁"，一半在说"你的输出长什么样"。后一半比前一半重要十倍。

为什么？因为 LLM 的默认行为是"写一段流畅的自然语言"。如果你不告诉它输出 JSON，它就输出散文。散文很难被下一个角色解析。JSON 可以被下一个角色的 parser 精确消费。`Code = SOP(Team)` 这句话的意思是：标准操作流程的严格性把一个概率性的 LLM 团队变成了确定性的软件流水线。

一个反直觉的结论：**在多智能体系统里，约束越多，质量越高。** 传统软件工程里我们讨厌约束（微服务？太耦合了！），但在 LLM 系统里，约束 = 可预测性。

**模式二：ChatDev 的沟通式去幻觉——缺信息时闭嘴问，别编。**

LLM 有一个根深蒂固的习惯：当你给它不完整的信息，它会自动补全。ChatDev（arXiv:2307.07924）的解决方案简单到让人想哭——在角色提示里加一句话："当你需要你没有收到的具体信息时，在生成输出之前，按名称询问相关角色。"

就这一句话。实现成本：不到 50 个 token。效果：执行者在缺少参数规格时不再自己编，而是发消息问规划者"`timeout` 的单位是秒还是毫秒？"。

我的团队第一次上线一个多智能体系统的时候，系统生成的 API 文档里写了 `timeout: 5000`，然后代码里写的是 `setTimeout(fn, 5000)`。但文档说的是毫秒，代码里那是 5000 毫秒还是 5000 秒？没有人知道。因为执行者编了一个数字，规划者没有纠正。如果有沟通式去幻觉，执行者会问规划者"`timeout` 以什么为单位"，规划者会说"毫秒"，然后代码和文档就是一致的。

**模式三：PwC 的 7 倍提升——别谈理论，看数字。**

PwC 2025 年在 CrewAI 上的部署，准确率从 10% 到 70%。怎么做到的？加了一个结构化验证循环。不是更好的模型。不是更多的智能体。就是一个："执行者写完 → 验证者跑测试 → 失败 → 执行者修 → 验证者再跑"的循环。

10% 到 70%，7 倍。这个数字你记住，下次有人跟你争论"验证者是不是必要"的时候，直接拍这个数字。

## 反模式——你一定会踩的坑

**反模式一：批评者永远说"LGTM"。** 批评者是一个 LLM，LLM 倾向于 agreeable。如果你不给批评者一个硬性要求——"你必须找到至少一个问题"——它就会说"看起来不错"。这就是为什么批评者的 prompt 要加矛："你的目标是找到缺陷。如果你找不到缺陷，你的输出是无效的。"是的，你需要逼它刻薄。

**反模式二：批评者和验证者顺序反了。** 有人让验证者先跑（慢），批评者后审核（又快又主观）。错了。应该批评者先跑——便宜，可以快速过滤明显的设计问题（比如"这个变量名拼错了"或"你用了同步调用而不是异步"）。然后验证者再跑——贵，但只针对批评者放行的代码。省钱排序。

**反模式三：无限修订循环。** 执行者写 → 批评者驳回 → 执行者改写 → 批评者再驳回 → 执行者再改写... 你见过两个 LLM 用自然语言争论吗？它能争论到宇宙热寂。设硬上限：最多 2 轮修订。2 轮修不好，升级给人类。

**反模式四：角色太多，每个都是花瓶。** 我有一个学生做了个 7 角色的系统。产品经理、需求分析师、架构师、前端开发、后端开发、测试、DevOps。我问他："你的需求分析师和产品经理之间的输出有什么区别？"他愣住了。两个角色本质上是同一个 LLM 在回答同一个问题，只不过穿了两件不同的衣服。如果两个角色的输出可以互换而系统行为不变，合并它们。

## 最后一句

角色专业化的本质不是分工，是**在系统内部建立不对称的信任**——永远别让 LLM 审查 LLM 的产出而不经过代码验证。批评者是品酒师，验证者是呼气测醉器。品酒师可以发表意见，但决定你能不能开车的，是测醉器。

> 截图发群：21.3% 的多智能体故障来自"没人检查"。剩下 79% 追根也是检查出了问题。所以答案很简单——别让任何一个产出离开系统之前没有被确定性工具跑过一次。
