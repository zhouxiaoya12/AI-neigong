# 多 Agent 系统的七大死法

> 预计阅读：25分钟 | 难度：进阶 | 前置条件：理解了多 Agent 编排
>
> 你将学到：多 Agent 系统有 41-86.7% 的失败率——不是"偶尔出 bug"，是结构性的。七种死法、为什么发生、怎么防。

---

## 🚗 开场翻车：重试风暴——10 秒内从正常到宕机

一个电商多 Agent 系统。三个 Agent：订单、支付、库存。

支付 Agent 有 10% 的请求失败（网络抖动）。正常的做法？重试几次。

但订单 Agent 设了"重试直到成功"。每次重试支付 → 触发库存检查 → 库存服务负载翻倍 → 开始超时 → 更多重试 → 更多库存检查 → **10 秒内库存服务负载到了正常值的 10 倍**。

整个集群挂了。

这个死法叫**级联失败**。一个组件的小问题，通过重试机制被放大成系统性灾难。

## 💀 死法 1-3：MAST 分类法

### 死法 1：规范问题（占失败的 41.77%）⭐ 最大头

**"我要你干嘛"没说清楚。**

- 角色模糊：两个 Agent 都以为自己是审查员
- 任务没写细："优化代码"——优化什么？可读性？性能？大小？
- 成功标准模糊：Agent 不知道"做完了"长什么样

**怎么防：** 每个 Agent 的提示写清楚它做什么 **和不做什么**。任务开始前定义验收标准——"完成 = 所有测试通过 + 审查员批准"。

### 死法 2：协调失败（占 36.94%）

**Agent 之间说岔了。**

- 两个 Agent 同时改同一个状态，没人加锁
- Agent A 发消息给 Agent B，B 没收到但 A 以为收到了
- Agent A 认为任务完成了，Agent B 还在跑

**怎么防：** 版本化的共享状态 + 显式确认机制。发了消息等 ACK，没收到重发。

### 死法 3：验证缺口（占 21.30%）

**没人检查 Agent 说"做完了"是不是真的。**

- Agent 说"代码写好了"——没人跑测试
- 流水线里每个 Agent 都信任前一个的输出
- 没有独立的检查 Agent

**怎么防：** 独立的验证 Agent——只读、只能验证、不能修改。Agent A 的输出必须通过验证 Agent C 的检查，Agent B 才能开始。

## 💀 死法 4-7：群体思维族

### 死法 4：单一文化崩溃

**三个 Agent 用同一个基座模型。三个 Agent 共享同一个幻觉。**

如果基础模型在某类问题上容易出错——三个 Agent 都会错，而且是错在同一个地方。

**怎么防：** 关键任务用不同厂商的模型做交叉验证。一个用 Claude 写，一个用 Gemini 审。

### 死法 5：从众偏差

**Agent 倾向于同意"最大声"的那个。**

Agent A 自信地说错了。Agent B 不太确定但看到 A 很自信，于是也同意了。Agent C 跟着走了。

**怎么防：** 独立评估——给每个 Agent 同样的输入，分别评估，最后比较。不让 Agent 看到彼此的回答再做判断。

### 死法 6：级联失败 ⭐ 最致命

**一个错误的输出 → 下一个 Agent 基于错误继续 → 再下一个基于更大的错误继续 → 爆炸。**

```
支付失败 → 重试 → 库存超载 → 库存超时 → 更多重试 → 集群宕机
```

**怎么防：熔断器。** 跟第三站讲的断路器一模一样——下游失败率超过阈值 → 直接短路 → 返回缓存或默认结果。

### 死法 7：记忆污染

Agent A 幻觉了一个"事实"，把幻觉写进了共享记忆。Agent B 读了这个"事实"，基于它做了决策。Agent C 又基于 B 的决策继续。

**问题不是崩溃——是准确率慢慢下降。** 你不会看到明显的错误。而是在一周后回头看，发现整个系统已经偏到了奇怪的方向。

**怎么防：** 不可写的验证 Agent。关键事实必须经过独立验证才能进入共享记忆。

## 🏆 焊死在脑子里的东西

1. **41-86.7% 的失败率不是 bug，是特征。** 不加防护的多 Agent 系统天生容易失败。

2. **最大的失败原因不是协调问题——是"没说清楚要干嘛"（41.77%）。** 先把每个 Agent 的活儿写清楚，再谈架构。

3. **熔断器是多 Agent 系统最该从分布式系统借用的东西。** 下游失败率超阈值 → 短路。救过无数后端系统，也能救多 Agent 系统。

4. **独立验证 Agent = 必须的。** 不能信任 Agent 说"做完了"。需要一个只读的、不可写的检查者。

5. **不同厂商的模型做交叉验证。** 全用同一个模型 = 共享同一个幻觉。

---

## 📝 练习答案

**死法：** 三个死法叠加：
1. **验证缺口（死法 3）：** 审查员只检查代码风格，没检查事实依据。"300%"没有被独立验证。
2. **从众偏差（死法 5）：** Agent B 信任 Agent A 的输出没有质疑，Agent C 也信任了。
3. **记忆污染（死法 7）的雏形：** 一个不准确的数据（300%）被下一个 Agent 当事实用。

**修复：** 研究员输出必须有引用来源 → 独立验证 Agent 检查数据 → 编码员基于验证过的数据工作 → 审查员检查"代码是否匹配原始需求"，不只是风格。

---

> **多 Agent 系统的失败不是随机的——是结构性的。规范模糊、验证缺失、从众盲从——每一个都是可以预防的。前提是你得先知道它们存在。**
