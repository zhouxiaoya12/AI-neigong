# Agent 框架权衡——LangGraph vs CrewAI vs AutoGen vs Agno

> 每个框架都卖同样的 demo（研究 Agent 写一份报告），隐藏同一个 bug（状态 schema 与编排层打架）。选那个其抽象匹配你问题形态的框架；其余的都是你写两遍的胶水代码。

**类型：** 学习
**语言：** Python
**前置课程：** Phase 11 · 09（函数调用）、Phase 11 · 16（LangGraph）
**用时：** 约 45 分钟

## 问题

你有一个需要多次 LLM 调用的任务。可能是一个研究工作流（计划、搜索、总结、引用）。可能是一个代码审查流水线（解析 diff、评审、打补丁、验证）。可能是一个多轮助手，它能订机票、写邮件、报销费用。你选了一个框架。

三天后，你发现框架的抽象层在渗漏。CrewAI 给了你角色，但当"研究员"需要把一个结构化计划交给"作者"时它和你打架。AutoGen 给了你 Agent 之间的对话，但没有一等状态，所以你的检查点是一个对话日志的 pickle。LangGraph 给了你状态图，但强迫你在知道 Agent 会做什么之前就命名每一个转换。Agno 给了你单 Agent 抽象，但当你想扇出到三个并发工作者时它叫唤了。

解决办法不是"选最好的框架"。是**把框架的核心抽象匹配到你的问题形态上。** 这节课画这张地图。

## 概念

### 核心抽象对比

四种框架主导 2026 年的格局。它们的核心抽象并不相同。

| 框架 | 核心抽象 | 最适合 | 最不适合 |
|------|---------|--------|---------|
| **LangGraph** | `StateGraph`——类型化状态、节点、条件边、检查点存储。 | 带显式状态和人类在环中断的工作流；需要时间旅行调试的生产 Agent。 | 松散的、角色驱动的头脑风暴，其中拓扑未知。 |
| **CrewAI** | `Crew`——角色（goal、backstory）、任务、流程（顺序或层级）。 | 角色扮演或人物驱动的、带短线性/层级计划的工作流。 | 超出工作组轮次历史的任何有状态需求；复杂分支。 |
| **AutoGen** | `ConversableAgent` 对——两个或更多 Agent 轮流发言直到退出条件。 | 多 Agent *对话*（师生、提案-评审、演员-审查者），其中思考从对话中涌现。 | 带已知 DAG 的确定性工作流；任何需要跨重启持久化状态的需求。 |
| **Agno** | `Agent`——单个 LLM + 工具 + 记忆，可组合为团队。 | 快速构建的单 Agent 和轻量团队；强多模态和内置存储驱动。 | 深度的、显式分支图，带自定义 reducer。 |

### "抽象"到底意味着什么

一个框架的核心抽象是你在推销架构时在白板上画的那个东西。

- **LangGraph** → 你画一张图。节点是步骤，边是转换，每一点上的状态对象是类型化的。思维模型是状态机。
- **CrewAI** → 你画一张组织架构图。每个角色有岗位描述，一个管理者路由任务。思维模型是一个小团队。
- **AutoGen** → 你画一个 Slack 私信。两个 Agent 互相发消息；如果你需要主持人，第三个加入。思维模型是聊天。
- **Agno** → 你画一个盒子，上面挂着工具。把盒子并排放就是团队。思维模型是"自带电池的 Agent"。

### 状态问题

状态是大多数框架选择在生产中崩溃的地方。

- **LangGraph。** 类型化状态（`TypedDict` 或 Pydantic 模型），按字段 reducer，一等检查点存储（SQLite/Postgres/Redis）。恢复、中断和时间旅行是免费的。*（见 Phase 11 · 16。）*
- **CrewAI。** 状态通过 `context` 字段以字符串形式在任务间流动，或通过 `output_pydantic` 结构化。开箱无持久化工作组存储；如果工作组必须跨重启存活，你自己挂接。
- **AutoGen。** 状态是对话历史和你定义的任何 `context`。对话记录持久化；任意工作流状态不会，除非你写适配器。
- **Agno。** 通过 `storage=` 挂接到 Agent 的内置存储驱动（SQLite、Postgres、Mongo、Redis、DynamoDB）——对话会话和用户记忆自动持久化。不是完整的图检查点存储；是一个会话存储。

### 分支问题

每个非平凡的 Agent 都有分支。谁决定分支很重要。

- **LangGraph** — 你通过条件边决定。路由是带命名分支的 Python 函数。分支在编译图中是一等的；检查点存储记录哪个分支被走。
- **CrewAI** — 管理器在层级模式下决定；顺序模式下你在构建时决定。路由隐含在任务列表中；在管理器提示之外没有一等"if"。
- **AutoGen** — Agent 通过对话决定。分支从谁下一个发言中涌现。`GroupChatManager` 选择下一个发言者；你可以手写 `speaker_selection_method`，但默认是 LLM 驱动的。
- **Agno** — Agent 通过调用哪个工具来决定。团队有协调/路由/协作模式；超出此范围的分支是开发者的责任。

### 可观测性问题

- **LangGraph** — OpenTelemetry，通过 LangSmith 或任何 OTel 导出器。每个节点转换是一个追踪 span；检查点兼作可回放追踪。LangSmith 是官方选项；Langfuse/Phoenix 也有适配器。
- **CrewAI** — 自 2025 年末起一等 OpenTelemetry；与 Langfuse、Phoenix、Opik、AgentOps 集成。
- **AutoGen** — 通过 `autogen-core` 集成 OpenTelemetry；AgentOps 和 Opik 有连接器。追踪粒度是每 Agent 消息，而非每节点。
- **Agno** — 内置 `monitoring=True` 标志加上 OpenTelemetry 导出器；与 Langfuse 紧密集成以做会话追踪。

### 成本与延迟

四个框架都增加每次调用开销（框架逻辑、验证、序列化）。递增开销的大致顺序：Agno ≈ LangGraph < CrewAI ≈ AutoGen。差异由框架做了多少额外 LLM 路由决定。CrewAI 的层级管理器花费令牌决定谁下一步；AutoGen 的 `GroupChatManager` 同理。LangGraph 只在你写 `llm.invoke` 的地方花费令牌。Agno 的单 Agent 路径很薄。

当每次运行的成本重要时，优先选显式路由（LangGraph 边、AutoGen `speaker_selection_method`）而非 LLM 选择的路由。

### 互操作性

- **LangGraph** ↔ **LangChain** 工具、检索器、LLM。一等 MCP 适配器（工具作为 MCP 服务器导入）。
- **CrewAI** ↔ 工具继承 `BaseTool`；LangChain 工具、LlamaIndex 工具和 MCP 工具均可适配。通过 `allow_delegation=True` 实现工作组到工作组的委派。
- **AutoGen** → `FunctionTool` 包装任何 Python 可调用对象；MCP 适配器可用。紧密耦合于 AG2 生态的 Agent 到 Agent 模式。
- **Agno** → `@tool` 装饰器或 BaseTool 子类；MCP 适配器；工具可跨 Agent 和团队共享。

## 技能

> 你能用一句话解释，为什么某个框架对某个 Agent 问题是正确的。

预构建检查清单：

1. **画出形态。** 这是一个图（类型化状态、命名转换）？一个角色扮演（专家交接工作）？一个对话（Agent 交谈直至完成）？一个带工具的单 Agent？
2. **决定谁做分支。** 开发者决定的分支 → LangGraph。管理器 Agent 决定的 → CrewAI 层级。对话涌现的 → AutoGen。工具调用决定的 → Agno。
3. **检查状态预算。** 你需要从检查点恢复吗？时间旅行？运行中途的人类中断？如果需要，LangGraph 是默认选项；Agno 会话覆盖对话范围状态。
4. **检查成本预算。** LLM 选择的路由每轮花费额外令牌。如果 Agent 每天运行数千次，优先选显式路由。
5. **预算框架开销。** 每个框架都是另一个依赖。如果任务是两次 LLM 调用和一个工具，写 30 行纯 Python；没有框架比没有框架更便宜。

**拒绝**在你能够画出图、组织架构图、聊天记录或 Agent 盒子之前就伸手拿框架。
**拒绝**选一个强迫你对抗其状态模型才能得到你实际需要的东西的框架。

## 决策矩阵

| 问题形态 | 首选框架 | 原因 |
|----------|---------|------|
| 带类型化状态、人类审批、长运行的工作流 DAG | LangGraph | 一等状态、检查点存储、中断、时间旅行。 |
| 带明确角色分工的研究/写作流水线 | CrewAI（顺序）或 LangGraph 子图 | 角色-任务映射在 CrewAI 中表达成本低；分支变复杂时升级到 LangGraph。 |
| 提案-评审或师生对话 | AutoGen | 双 Agent 对话是其原生形态。 |
| 带工具、会话、记忆的单 Agent | Agno | 最薄的设置，内置存储和记忆。 |
| 数千个带 reducer 的并行扇出 | LangGraph + `Send` | 唯一有一等并行分发 API 的。 |
| 快速原型，无框架承诺 | 纯 Python + 供应商 SDK | 没有框架是最快的框架。 |

## 练习

1. **简单。** 拿同一个任务——"调研 Anthropic 的总部，写一份 200 词的简报，引用来源"——用 LangGraph（四个节点：plan、search、write、cite）和 CrewAI（三个角色：研究员、作者、编辑）实现。报告每次运行的令牌成本和代码行数。
2. **中等。** 用 AutoGen（研究员 ↔ 作者对话，编辑通过 `GroupChat` 加入）和 Agno（带 `search_tools` 和 `write_tools` 的单 Agent，加会话存储）构建同一任务。在 (a) 每次运行成本、(b) 崩溃后恢复能力、(c) 在写步骤前注入人类审批的能力上进行排序。
3. **困难。** 构建一个决策树脚本 `pick_framework.py`，接受简短的问题描述（JSON：`{has_typed_state, has_roles, has_dialogue, has_parallel_fanout, needs_resume}`），返回带一句话理由的推荐。在你自设计的六个案例上验证。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 编排（Orchestration） | "Agent 如何协调" | 决定哪个节点/角色/Agent 下一步运行的层 |
| 持久化状态（Durable state） | "重启后恢复" | 跨越进程死亡而存活的状态，挂接到检查点或会话存储 |
| LLM 选择的路由（LLM-selected routing） | "让模型决定" | 一个规划 LLM 每轮选择下一步；灵活但每次决策花费令牌 |
| 显式路由（Explicit routing） | "开发者决定" | 一个 Python 函数或静态边选择下一步；便宜且可审计 |
| Crew | "一个 CrewAI 团队" | 角色 + 任务 + 流程（顺序或层级）绑定为单一可运行体 |
| GroupChat | "AutoGen 的多 Agent 对话" | N 个 Agent 之间的管理对话，带发言者选择器 |
| Team（Agno） | "多 Agent Agno" | 在一组 Agent 上的路由/协调/协作模式 |
| StateGraph | "LangGraph 的图" | 类型化状态、节点、条件边、检查点存储抽象 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是整个 Phase 14 多 Agent 框架模块的收官课——不是讲"怎么用"，是讲"怎么选"。文档做了一个行业里极少有人做的事：把四个主流框架的核心抽象、状态模型、分支机制、可观测性、成本和延迟，放在一张表格里横向对比。最大的价值在于"决策矩阵"——六种问题形态直接映射到首选框架加一句话理由。这是架构师的cheatsheet，不是开发者的教程。适合已通读前面七节课、正在做一个真实的"我该用哪个框架"决策的架构师。

### 二、知识结构梳理

**对比维度层：** 七个维度——核心抽象、状态模型、分支机制、可观测性、成本延迟、互操作性、决策矩阵——构成一个完整的框架选型 checklist。每个维度不是独立的——状态模型决定了恢复能力，分支机制决定了成本和可审计性，核心抽象决定了"白板上画什么"。

**预构建检查清单层：** 五步法——画形态 → 定分支 → 查状态 → 查成本 → 预算开销——是一个可执行的决策流程。最后两条拒绝规则（"画不出来就别选框架""别选要你对抗其状态模型的框架"）是工程经验的结晶。

**实操验证层：** 三个难度递进的练习，要求学生用同一个任务跑四个框架，然后在成本、恢复、人工审批三个维度上排序。这不是"学框架"——是"学选框架"。

### 三、核心洞察（备课时的关键理解）

1. **"核心抽象 = 你在白板上画的那个东西"是整个文档的关键洞察。** 选框架不是在比较 feature list——是在比较"这个框架要求你用什么样的思维模型来构建你的系统"。图？组织架构图？聊天记录？Agent 盒子？如果你的问题形态是图但你选了一个"聊天"作为核心抽象的框架——你会一直在对抗框架。

2. **状态是框架选择的沉默杀手。** 框架文档很少在首页提状态模型——它们在 demo 里用"一次运行"展示了所有功能。真实系统需要恢复、中断、时间旅行——这时候状态模型的差异爆炸性显现。LangGraph 有步级检查点，其他三个没有。

3. **"谁决定分支"是成本和可审计性之间的 trade-off。** LLM 选择路由（CrewAI 层级管理器、AutoGen GroupChatManager）= 灵活但贵且不可审计。开发者选择路由（LangGraph 条件边、Agno 工具调用）= 便宜且可审计但不够灵活。这不是"哪个更好"——是"你的任务需要哪种"。

4. **"没有框架是最快的框架"是文档最被低估的声明。** 在课程序列的这个节点——八节课学完了七个框架——这一句是必要的解毒剂。"如果任务就是两次 LLM 调用和一个工具，写 30 行 Python 就够了。"

5. **互操作性不是 feature——是迁移成本。** 如果你的现有系统用 LangChain 工具和检索器——LangGraph 和 CrewAI 都可以直接吃它们。如果你的团队用 MCP 服务器——四个框架都有适配器。互操作性的真正价值是"你不需要为换框架重写所有集成"。

6. **可观测性在四个框架中趋同。** OpenTelemetry 正在成为共同标准。四个框架都支持 OTel 导出，Langfuse 和 AgentOps 作为通用集成层。这意味着架构师选框架时可以把可观测性从"差异化因素"中排除——四个都够用。

7. **三个练习的设计本身就是教学方法。** 同一个任务、四个框架、在三个维度上排序——这不仅验证了"哪个框架更好"，而且验证了"每个框架在哪个维度上更好"。学生从"LangGraph 好还是 CrewAI 好？"变成"LangGraph 好在状态管理，CrewAI 好在角色表达，AutoGen 好在对话涌现，Agno 好在轻量上手。"

### 四、教学建议

1. **这节课应该放在整个 Phase 14 模块的最后。** 学生需要先理解每个框架独立的样子，才能理解对比。前面七节课是"学框架"，这节课是"学选框架"。

2. **用"同一个任务四种实现"的对比 demo 开场。** 拿练习 1 的任务——研究、写作、引用——用四个框架各写一个实现。LangGraph 版（图）、CrewAI 版（角色）、AutoGen 版（对话）、Agno 版（单 Agent+工具）。让学生看到同一个任务在不同抽象下的表达方式。

3. **白板上画"核心抽象"的对比是最有效的视觉化。** 画四个白板：一张图、一张组织架构图、一个聊天框、一个带工具的盒子。然后问——"你的问题最适合画成哪个？"

4. **"状态问题"用三个场景来教。** 场景 A：Agent 在第 7 步崩溃→需要从第 7 步恢复。场景 B：需要在第 3 步注入人工审批然后继续。场景 C：需要回退到第 2 步检查当时的状态。每个场景让学生在四个框架中排"能做什么、做不到什么"。

5. **"LLM 路由 vs 显式路由"的成本对比值得量化。** 同一个 5 步任务，CrewAI 层级（5 次 LLM 调用 + 5 次管理器调用）vs LangGraph 条件边（5 次 LLM 调用）。令牌成本差异的百分比——这就是"灵活性的价格标签"。

6. **"没有框架是最快的框架"应该在课程结束时着重强调。** 学生学完八节课后容易产生"每个项目都需要一个框架"的错觉。用一个反例收尾——"你最常用的 3 个 Agent 功能，统计一下没有框架你能用多少行代码解决"。

7. **练习 3（`pick_framework.py`）是这节课的最终交付物。** 让学生写出一个决策脚本——不是"我觉得"，是"给定这 5 个参数，输出框架推荐+理由"。这是把知识转化为工具。

### 五、值得补充的内容

1. **框架的版本兼容性和迁移成本。** LangGraph 从 0.1 到 0.3.0 有 breaking changes。CrewAI 从 0.5 到 0.8 重写了大部分 API。AutoGen 从 0.2 到 0.4 是完全的架构重写（事件驱动）。框架选择的"保鲜期"是多少？

2. **四框架的社区规模和商业支持对比。** LangGraph = LangChain 公司。CrewAI = CrewAI 公司。AutoGen = 微软研究院。Agno = 初创公司。社区规模决定你debug时能在GitHub issue里找到多少答案。

3. **框架在非 Python 生态的可用性。** LangGraph 有 JS/TS 版本。CrewAI 只有 Python。AutoGen 有 .NET 版本。Agno 只有 Python。如果你的团队是 Node.js 为主的——选型立即缩小。

4. **框架对多供应商 LLM 的支持。** 文档说 Agno 支持 20+ LLM 供应商。LangGraph 通过 LangChain 生态支持多供应商。CrewAI 主要围绕 OpenAI 和 Anthropic。AutoGen 强在微软生态（Azure OpenAI）。多供应商需求能立即缩小选择范围。

5. **"混合架构"——两个框架同时用。** 实际生产中常见 LangGraph（状态机 shell）+ CrewAI Crew（探索性子任务）或 Agno Agent（简单工具调用）+ LangGraph 检查点。文档提到的互操作性为这些混合架构铺路，但未展开。

### 六、一句话总结

**选 Agent 框架不是在比 feature——是在比"你需要在白板上画什么才能解释你的系统"。画图 → LangGraph。画组织架构图 → CrewAI。画聊天框 → AutoGen。画带工具的盒子 → Agno。画不出来 → 先把问题搞清楚再选框架。**

# 🎓 Agent 架构课：怎么选框架——一堂关于"匹配"的课

同学们好。我是你们的 Agent 架构老师。

前面七节课，我带你们看了四个框架——LangGraph、CrewAI、AutoGen、Agno——还有 OpenAI 和 Claude 的 SDK。每节课我都说了它擅长什么、不擅长什么。

今天这节课不一样。今天不做"怎么用"。今天做一件事：**给你一个具体的 Agent 任务，你怎么知道你该用哪个框架？**

在我给你们对比表之前，我先讲一件事。这件事我在生产里见过太多次了。

## 同一个 bug，四个框架

有一个任务在 Agent 演示中极其常见：研究型 Agent。计划 → 搜索 → 总结 → 引用。一个团队用四个框架实现了它。

**LangGraph 版：** 四个节点，四条边。状态是类型化字典。检查点存在 SQLite 里。Agent 在第 3 步崩溃——恢复，从第 3 步继续。

**CrewAI 版：** 研究员、作者、编辑。每个有性格、背景故事。角色清晰，输出像人写的。但当研究员需要把一个结构化 plan 对象传给作者时——`context` 字段只传字符串。plan 对象被压扁成一段文本，作者要重新 parse。

**AutoGen 版：** 研究员和作者互发消息。"我找到了这些资料""好的，我在写""这一段需要更多引用"。对话流畅，自然。但当 Agent 在第 7 轮对话崩溃时——恢复是加载整个聊天记录并重放。状态在对话里，不在检查点里。

**Agno 版：** 一个 Agent，`search_tool` + `write_tool`。30 行代码跑起来。但当你想让三个 Agent 并行搜索不同资料时——Agno 的 Team 模式能做，但 reducer 逻辑你得自己写。

同一个任务。同样的输入。四个框架用了四种完全不同的"形状"来表达它。

没有人错。是**问题本身在不同框架的抽象下呈现了不同的形状。** 这节课给你画一张地图——哪个问题用哪个框架。

## 核心抽象：你白板上画的是什么

框架不是在比较功能。框架在比较的是**你被要求用什么方式思考你的系统。**

- **LangGraph** → 状态机。你在白板上画一张图——节点是步骤，边是转换，每一步的状态是可检查点的。你的问题看起来像流程图——用 LangGraph。

- **CrewAI** → 组织架构图。你在白板上画一个团队——研究员汇报给谁，谁产出最终稿。你的问题看起来像角色分工——用 CrewAI。

- **AutoGen** → 聊天记录。你在白板上画两个方块互相发消息。你的问题看起来像对话——用 AutoGen。

- **Agno** → 带工具的盒子。你在白板上画一个 Agent 盒子，挂上搜索工具、写工具、数据库工具。你的问题看起来像一个全能助手——用 Agno。

如果选错了——比如你的问题本质上是一个状态机但你用了"角色扮演"框架——你会一直在对抗框架的底层假设。研究员传给作者的不是"一段文本"——是一个 `Plan` 对象。CrewAI 不支持。你开始写胶水代码。胶水代码就是"你用框架不适合的部分"。

## 三个隐藏的杀手

框架 demo 展示的东西都一样——一个 Agent 跑成功了。demo 不展示的是：

### 杀手一：状态——"Agent 在第 7 步崩溃了，怎么恢复？"

- **LangGraph**：`resume(session_id)`。检查点精确存储了第 6 步结束时的完整状态。恢复 = 加载 + 从第 7 步的节点开始跑。
- **CrewAI**：没有内置的步级恢复。Crew 是一次性 `kickoff()`。崩溃了？从头再来。
- **AutoGen**：对话历史持久化了，但任意工作流状态不会。恢复 = 加载聊天记录 + 重放——重放可能有副作用（工具被重复调用）。
- **Agno**：`storage=` 持久化了会话。恢复 = 加载会话 + 继续。但不会精确地从"那个工具调用的中间"恢复——从轮次粒度恢复。

如果你的 Agent 会跑 10 分钟或更长——**状态模型是你选框架的第一考量。** 崩溃后以什么粒度恢复？节点级？轮次级？没有？

### 杀手二：分支——"谁决定下一步？"

- **LangGraph**：你决定。`route(state) -> "plan" | "search" | "write"`——一个 Python 函数。便宜、可审计、可测试。
- **CrewAI（层级）**：管理器 LLM 决定。每轮一次额外 LLM 调用，花费令牌来选"下一个该谁"。灵活——但贵。
- **AutoGen**：Agent 通过对话决定。"我觉得现在该编辑了"——涌现式。自然——但不可预测。
- **Agno**：Agent 通过工具调用决定。调 `search` 与否——Agent 的判断。简洁——但分支逻辑分散在工具定义中。

"谁决定分支"直接关联两件事：① 每次运行的成本（LLM 路由花的令牌），② 出问题时你能不能复现决策路径（显式路由 → 能，LLM 涌现路由 → 难）。

成本敏感？选显式路由。灵活性至关重要？选 LLM 路由。两个都需要？LangGraph 条件边 + 在某个节点内用 CrewAI 层级路由——混合架构。

### 杀手三：开销——"我的 Agent 每天跑一万次"

框架不是免费的：
- **LangGraph**：状态序列化、检查点写入、条件边评估——开销低且可预测。
- **CrewAI**：层级模式的 manager LLM 每轮额外翻倍 LLM 调用。
- **AutoGen**：GroupChatManager 每轮选下一个发言者——每次发言都要选。
- **Agno**：单 Agent 模式开销最低。Team 模式增加到路由开销。

大概的顺序：Agno（单 Agent）≈ LangGraph < CrewAI（顺序）< AutoGen < CrewAI（层级）。

如果你的 Agent 每天跑几千次——这个排序就是你的成本决策表。

## 决策矩阵

我不念这张表了——你看文档。但我要强调几个"为什么会这样排"的理由。

**研究/写作流水线 → CrewAI 或 LangGraph 子图。** 角色清晰（研究员、作者、编辑）→ CrewAI 表达成本最低。但当分支变复杂（"如果搜索不到资料，走人工审批路线"）→ 升级到 LangGraph。CrewAI 的角色表达有优势，LangGraph 的分支表达有优势。不是"选一个"——是"从哪个起步，什么时候切"。

**提案-评审 → AutoGen。** 两个 Agent 对话——"我提案""我评审""我修改"——这是 AutoGen 的原生形态。用 LangGraph 也能做（两个节点交替），但会丢失"对话的涌现感"。用 CrewAI 也能做（两个角色），但角色之间不直接对话——它们通过任务 context 传递。

**单 Agent + 工具 → Agno。** 最薄的设置。`agent = Agent(tools=[search, write], storage=SqliteStorage())`。三行。不需要图、不需要角色、不需要聊天管理。

**最后一行：纯 Python。** 如果这个 Agent 就是两次 LLM 调用和一个工具——你不需要框架。框架是依赖项、学习曲线、版本兼容性问题的来源。纯 Python 的 30 行 = 零框架开销 + 完全控制 + 零版本风险。

## 拒绝规则

最后，两条规则，如果你只记住这节课的两件事，记住这两条：

**规则一：在你画不出白板之前，别选框架。** 你能画出这个系统在 LangGraph 里像什么吗？在 CrewAI 里像什么？在 AutoGen 里像什么？在 Agno 里像什么？四张白板画不出来——你的问题还没想清楚。先搞清楚问题，再选框架。

**规则二：别选需要你对抗其状态模型的框架。** 你需要步级恢复→CrewAI 就是错误的答案。你需要角色驱动的团队→LangGraph 单用会很痛苦。你需要涌现式对话→Agno 不是为这个建的。框架的状态模型是你的约束——你的需求应该落入框架的舒适区，而不是被迫适配。

## 结课清单

做一个框架选型决策时：

1. 画出问题在四个框架下的形态。
2. 列出你不可妥协的需求——步级恢复？角色驱动？对话涌现？
3. 匹配：需求的形态和框架的核心抽象是否一致？
4. 计算：这个任务每天跑多少次 × 框架的每次开销 = 每日成本。
5. 检查：你能接受"LLM 决定分支"带来的不可审计性吗？
6. 总问：这个任务能用 30 行纯 Python 吗？

**最后一句话——这是我整个 Phase 14 模块最想说的一句话：**

**每一个 Agent 框架都在回答同一个问题——"你的系统里，谁来画箭头？"如果你画 → LangGraph。如果你的角色画 → CrewAI。如果你的 Agent 通过聊天画 → AutoGen。如果你的 Agent 通过工具调用画 → Agno。如果你发现这个系统不需要箭头 → 你不需要框架，30 行 Python 就够了。**
