# 监管者 / 编排器-工作者模式

> 一个主导 Agent 做计划和委派；专业工作者在并行上下文中执行并回报。这是 Anthropic 研究系统背后的模式（Claude Opus 4 作为主导，Sonnet 4 作为子 Agent），在内部研究评估中比单 Agent Opus 4 高出 +90.2%。Anthropic 的工程文章报告，BrowseComp 上 80% 的方差仅由令牌使用量解释——多 Agent 的优势主要来自每个子 Agent 获得一个新的上下文窗口。本课从原语构建监管者模式，并涵盖从生产部署中获得的 2026 年工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置课程：** Phase 16 · 04（原语模型）
**用时：** 约 75 分钟

## 问题

研究是单 Agent 系统失灵的典型任务。你问"在多 Agent 系统方面，2023 年到 2026 年之间有什么变化？"单个 Agent 按顺序读了五篇论文，把一半的上下文塞满了论文正文，然后试图对所有论文一起推理。到第五篇时它已经忘了第一篇。它无法并行化。

监管者模式修复了这一点：一个主导 Agent 规划搜索，将每个子问题委派给一个工作者，然后综合。每个工作者针对一个狭窄问题获得自己的 200k 令牌窗口。主导从未看到原始论文——只看到工作者摘要。

Anthropic 的生产研究系统报告在内部研究评估中比单 Opus 4 高出 +90.2%。同一篇文章指出，BrowseComp 上 80% 的方差仅由*令牌使用量*解释。每个子 Agent 的新上下文是主要机制。

## 概念

### 模式

```
 ┌──────────────┐
 │ 主导 │ 规划、分解、
 │ (Opus 4) │ 综合
 └──┬────┬───┬──┘
 │ │ │
 ┌───────┘ │ └───────┐
 ▼ ▼ ▼
 ┌─────────┐ ┌─────────┐ ┌─────────┐
 │ 工作者1 │ │ 工作者2 │ │ 工作者3 │
 │(Sonnet) │ │(Sonnet) │ │(Sonnet) │
 └─────────┘ └─────────┘ └─────────┘
 新 新 新
 上下文 上下文 上下文
```

主导从未读取原始材料。在主导综合之前，工作者彼此看不到对方的工作。每条箭头是一个带有狭窄制品的转交。

### 为什么它能赢

三种机制：

1. **每个子 Agent 的新上下文。** 一个正在探索"FIPA-ACL 遗产"的工作者不携带主导用于规划的 40K 令牌。它为一个问题获得一个 200K 窗口。
2. **通过提示词实现专业化。** 主导的提示词是"分解和综合"，不是"研究"。每个工作者的提示词是狭窄的："找到 X 中发生了什么变化"。专注的提示词产出专注的输出。
3. **并行化。** 工作者并发运行。墙钟时间约为 `max(工作者时间) + 规划 + 综合`，而不是 `sum(工作者时间)`。

### 工程经验（Anthropic 2025）

Anthropic 文章列出了几条在 2026 年仍然相关的生产经验：

- **将投入规模匹配到查询复杂度。** 简单查询：一个 Agent，3-10 次工具调用。复杂查询：10 个以上 Agent。主导必须自己估计这一点，而非由调用者决定。
- **先广后深。** 先分解为广泛的子问题，然后如果答案值得深入，再为每个子问题生成更多工作者。
- **彩虹部署。** Agent 是长运行且有状态的。传统的蓝绿部署不适用。Anthropic 使用彩虹部署：新版本逐步上线，旧版本逐步排空。
- **令牌使用占主导。** 多 Agent 大约是单 Agent 令牌量的 15 倍。仅在任务价值证明成本合理时才运行它。

### LangGraph 转折

LangGraph 最初以 `langgraph-supervisor` 库发布，带有一个高级的 `create_supervisor` 辅助函数。2025 年 LangChain 将建议改为通过工具调用直接实现监管者模式，因为工具调用对*监管者看到什么*提供了更多控制（上下文工程）。该库仍然有效；文档现推荐工具调用形式。

### 失败模式

- **主导幻觉出计划。** 如果主导生成的子问题没有分解真正的问题，工作者在错误的目标上做精确的研究。
- **工作者过度探索。** 没有明确的范围边界，工作者会漂移到超出被分配的子问题，污染综合步骤。
- **综合冲突。** 两个工作者返回矛盾的事实。主导必须要么重新询问（增加一轮），要么明确标注分歧。静默选择一边是最糟糕的失败：用户永远不知道存在分歧。

### 什么时候监管者是错误的

- **顺序任务。** 如果第 2 步确实需要第 1 步的输出，并行化没有收益。使用流水线（CrewAI 顺序、LangGraph 线性图）。
- **简单查询。** 单 Agent 处理得更快更便宜。在生成工作者之前使用主导的"规模投入"检查。
- **严格确定性。** 监管者使用 LLM 选择的委派。当审计/重放比适应性更重要时，静态图更好。

## 构建

`code/main.py` 使用 `threading` 实现一个三个并行工作者的监管者。主导将查询分解为子问题，工作者在每个子问题上并发运行，主导综合。没有真实 LLM——工作者被脚本化以模拟获取和摘要。

关键结构：

- `Lead.plan(query)` 将一个查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回一个假摘要（在生产中可以是任何使用工具的 Agent）。
- `Lead.run(query)` 在线程中启动工作者、等待完成、然后综合。

运行：

```bash
python3 code/main.py
```

输出显示计划、带开始/结束时间戳的并行工作者轨迹、以及最终综合。你可以看到墙钟时间的优势：三个 0.3 秒的工作者在大约 0.35 秒内运行，而不是 0.9 秒。

## 使用

`outputs/skill-supervisor-designer.md` 接收用户查询并产出一个监管者模式设计：主导系统提示、工作者角色、子问题分解规则和综合模板。在构建新的研究型 Agent 系统之前使用此工具。

## 发布

部署监管者模式前的检查清单：

- **模型配对。** 主导使用推理层模型（Opus 级、`o3` 级）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 任何超过 2 倍中位运行时间的工作者被终止；主导要么以更窄的范围重新生成，要么在没有它的情况下继续。
- **每工作者令牌上限。** 硬限制（例如 10 倍预期综合输入）防止失控工作者耗尽预算。
- **可观测性。** 追踪主导的计划、每个工作者的工具调用和综合。这是任何事后调试的基础。
- **彩虹部署。** 有状态的长运行 Agent 需要渐进版本转换，而非热替换。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 监管者（Supervisor） | "主导 Agent" | 一个做规划、委派和综合的编排器 Agent。它不自己做具体工作。 |
| 工作者（Worker） | "子 Agent" | 一个由监管者调用、范围狭窄、拥有自己上下文窗口的聚焦 Agent。 |
| 编排器-工作者（Orchestrator-worker） | "监管者模式" | 同一个概念，不同的名字。2026 年文献两者都用。 |
| 新上下文（Fresh context） | "干净窗口" | 工作者的上下文从系统提示和被分配的问题开始，不携带主导的历史。 |
| 彩虹部署（Rainbow deployment） | "渐进上线" | 长运行有状态 Agent 需要版本化的排空替换，而不是蓝绿部署。 |
| 令牌占主导（Token dominance） | "上下文是变量" | 80% 的研究评估方差来自总令牌使用量，而非模型选择，据 Anthropic。 |
| 规模投入（Scale effort） | "将 Agent 数量匹配到复杂度" | 主导估计查询难度，相应生成 1 个 vs 10+ 个工作者。 |
| 综合冲突（Synthesis conflict） | "工作者意见不一" | 两个工作者返回矛盾事实；主导必须暴露分歧，而非静默选择一边。 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是 Phase 16 高级 Agent 工程中的"旗舰模式"课。长程 Agent（第 16 课）讲了跨度翻倍的趋势——这节课讲的是解决这个问题的具体架构模式：监管者-工作者。文档以 Anthropic 的真实生产研究系统为蓝本，给出了一个罕见的数字——+90.2% 超单 Agent，80% 方差由令牌使用量解释。这两个数字加在一起揭示了一个反直觉的真相：多 Agent 的主要优势不是"多个大脑"，是"多个上下文窗口"。适合已理解编排模式（Phase 14）和长程 Agent 挑战（Phase 16·01），正在设计研究型 Agent 系统的架构师。

### 二、知识结构梳理

**模式定义层：** 监管者的三角架构——主导规划/综合，工作者并行执行。三个获胜机制——新上下文、专业化提示、并行墙钟时间——每个是独立的工程杠杆。

**生产经验层：** Anthropic 的四条经验——规模投入、先广后深、彩虹部署、令牌占主导——将学术模式翻译为运维可操作的实践。彩虹部署是其中最被低估的：有状态长运行 Agent 不能用蓝绿部署。

**工程构建层：** `code/main.py` 用 threading 展示了并行化的墙钟优势——三个 0.3s 工作者在 0.35s 内完成而非 0.9s。

### 三、核心洞察（备课时的关键理解）

1. **"+90.2% 和 80% 令牌方差"放在一起读，揭示了一个反直觉的真相。** 多 Agent 的优势主要不是"多个 Agent 一起推理"——是"每个子 Agent 有自己的上下文窗口，不被其他 Agent 的历史污染。"主导的上下文窗口装 200K 工作者的原始研究材料 vs 每个工作者用自己的 200K 窗口只装一个子问题——上下文管理才是多 Agent 的真正机制。

2. **"主导不做具体工作"是监管者模式区别于编排器-工作者（Phase 14）的关键。** Phase 14 的编排器-工作者中编排器可以做一部分工作。在生产研究中，Anthropic 的主导 Opus 4 只做计划和综合——不碰原始数据。这是专业化在 Agent 层面的应用：推理模型做推理，快模型做检索。

3. **"彩虹部署"暴露了 Agent 运维和传统微服务运维的根本差异。** 蓝绿部署假设服务是无状态的——切流量就行。Agent 是有状态的——一个正在运行 6 小时的研究 Agent 不能"切流量"。彩虹部署：新版本 Agent 启动处理新任务，旧版本 Agent 运行完当前任务后退役。这是"状态感知"的部署策略。

4. **"规模投入"意味着主导必须自己判断复杂度——而不是收到指令"生成 N 个工作者"。** 简单问题 1 个 Agent 就够了。复杂问题 10+ 个。主导的系统提示需要包含这个判断逻辑——这是一个元认知任务（"这个问题需要多少资源？"）。

5. **综合冲突是监管者模式最难以自动化的环节。** 两个工作者返回矛盾事实——主导怎么裁决？Anthropic 的选择是"要么重新问，要么明确标注分歧"。静默选一边是最差的——用户被欺骗。但"重新问"有成本，"标注分歧"让用户自己判断——两者都放弃了全自动化的幻想。

6. **工具调用监管者 vs `create_supervisor`——这是上下文工程控制权的斗争。** `create_supervisor` 帮你管理上下文——但你不看到主导看到了什么。工具调用形式让你精确控制——只传子答案不传原始上下文。Anthropic 选择后者——因为 80% 的方差在令牌使用量，上下文控制是性能变量。

7. **工作者超时和令牌上限不仅是运维配置——是防止"长程 Agent 失控"的最小化护栏。** 一个工作者无限探索一个子问题 → 超时终止 → 主导收到部分结果或没有 → 决定是否重试。没有这两个限制——长程 Agent 的成本问题（Phase 16·01）在这里具体化为一个工作者的预算爆炸。

### 四、教学建议

1. **用 Anthropic 的 +90.2% 开场。** "单一 Opus 4 的研究得分如果是 50，加上 Sonnet 4 子 Agent 后变成 95——为什么不是多个 Opus？答案是模型配对——推理做规划，快模型做检索。"

2. **"新上下文"是这节课最核心的概念，用一个可视化 demo 来讲。** 显示主导的上下文窗口被 5 篇论文的全文占满 vs 每个工作者窗口只有 1 篇论文的摘要任务。窗口占用率从 95% 降到 20%。这个对比比任何文字都有说服力。

3. **彩虹部署用一个"不能重启的 Agent"的故事来讲。** "你的 Agent 跑了 5 小时 59 分，马上要完成一个复杂分析了。运维说'要升级 Agent 版本了，重启一下。'——你会说什么？"这就是为什么彩虹部署存在。

4. **令牌占主导（80% 方差）这个数字值得在黑板上写下来。** 然后问学生："如果你能选——把模型从 Sonnet 升到 Opus，或把上下文窗口从 200K 升到 400K——哪个对研究评估分数的影响更大？"根据 Anthropic 的数据——后者。模型选择不是主要变量，上下文是。

5. **综合冲突检测（练习 3）是最有价值的动手练习。** 让学生设计一个"不用 LLM 检测矛盾"的算法——关键词重叠、日期冲突、数字矛盾。这是信息检索 101 在 Agent 场景的应用。

6. **LangGraph 的 `create_supervisor` vs 工具调用监管者——让学生看两个代码版本。** `create_supervisor` 版本：5 行配置。工具调用版本：30 行手写。但后者让你看到"主导传给工作者什么"。控制粒度 vs 便利性的 trade-off。

7. **练习 4（对比 Anthropic 生产实践）是连接"玩具"和"生产"的桥梁。** 让学生列出他们的 demo 和 Anthropic 生产系统之间的差距——超时、令牌上限、彩虹部署、冲突检测——每一个差距就是"为什么你的 toy 不能直接上线"的理由。

### 五、值得补充的内容

1. **主导的"规模投入"判断逻辑。** 怎么让主导估计任务复杂度？基于查询长度？关键词密度？历史数据？这是一个值得深入讨论的元认知设计问题。

2. **工作者之间的信息共享——什么时候打破"彼此不见"的规则。** 如果工作者 2 的研究依赖于工作者 1 发现的一个关键事实——主导应该在生成工作者 2 时传递这个事实吗？这是上下文隔离和协同效率之间的权衡。

3. **监管者模式的成本模型。** 15 倍令牌 ≠ 15 倍成本。因为工作者用的是更便宜的模型（Sonnet），主导用更贵的模型（Opus）。加权成本 = 1 × Opus 成本 + N × Sonnet 成本。

4. **与 Swarm 模式（Phase 14）的对比。** 监管者 = 有中心路由，工作者彼此不见。Swarm = 无中心，Agent 直接转交。监管者模式在研究中占优（因为主导的综合步骤是必需的），Swarm 在低延迟场景中占优。

5. **非研究场景的监管者变体。** 代码审查（主导分配审查者）、客户支持（主导分类并路由）、数据分析（主导分解查询并综合图表）——监管者模式在不同领域的具体形态。

### 六、一句话总结

**监管者模式的价值不在"多个 Agent"——在"多个上下文窗口"。一个 Opus 4 做规划和综合，N 个 Sonnet 4 各有一个干净的 200K 窗口做狭窄检索。+90.2% 的提升主要来自上下文隔离，而非并行推理。选择监管者不是因为你需要多个大脑——是因为单个窗口装不下所有证据。**

# 🎓 Agent 架构课：监管者模式——多 Agent 的真正力量不在"多"

同学们好。我是你们的 Agent 架构老师。

上节课我讲了长程 Agent——跨度每 7 个月翻一番，14 小时的运行已经是现实。今天这节课回答一个自然的问题：**既然单个 Agent 能跑 14 小时了，为什么还要多 Agent？**

我先给你两个数字。

Anthropic 在内部研究评估上测了一个系统：主导 Agent 是 Claude Opus 4，子 Agent 是 Sonnet 4。对比单 Opus 4，多 Agent 版本得分高出 +90.2%。

第二个数字更耐人寻味：**BrowseComp 基准上，80% 的得分方差仅由令牌使用量解释。** 不是模型架构，不是提示工程，不是工具选择——就是用了多少 token。

把这两个数字放在一起读：多 Agent 系统赢了 +90.2%，而 80% 的赢面来自"用了更多 token。"**多 Agent 的真正力量不是"多个大脑一起推理"——是"每个大脑有一个不被污染的上下文窗口。"**

## 监管者不做事

监管者模式的架构很简单——主导做三件事：计划、委派、综合。工作者做一件事：专注执行。

主导是 Opus 4——推理层模型。它不读论文全文。它说："这个问题有三个子方向——FIPA-ACL 的遗产、2024 年的 LangGraph 转向、OpenAI Swarm 的兴衰。"然后它把这三个子问题委派给三个 Sonnet 4 工作者。

每个工作者拿到的是什么？**一个干净的 200K 上下文窗口。** 里面只有系统提示和被分配的狭窄问题。没有主导的 40K 规划 token。没有其他工作者的中间推理。只有一个问题和搜索结果的完整空间。

工作者 A 读 3 篇关于 FIPA-ACL 的论文。它的上下文窗口用掉了 60K tokens——装满了相关证据。它产出 500 词的摘要。

工作者 B 读 4 篇关于 LangGraph 的文章。同样——自己的窗口，自己的证据，自己的摘要。

主导收到了三份 500 词的摘要。它的上下文窗口现在有 1.5K 的精华——而不是 200K 的全文。它的推理精度没有被稀释。

这就是 +90.2% 的来源。不是"多 Agent 更聪明"——是"每个 Agent 有更好的信息密度。"

## 为什么不是多个 Opus 4

你可能想问：为什么工作者不用 Opus 4？用更强的模型不是更好吗？

因为工作者不需要推理——工作者需要**检索和摘要。** 这不是一个需要 Opus 级推理能力的任务。Sonnet 4 做得足够好，而且便宜。一对多：主导的强度 × 工作者的效率。

这是模型配对——用最匹配任务需求的模型，而不是最强的模型。

## 四个工程现实

Anthropic 在生产中跑这套系统，总结了四条经验。每一条都能救你的预算。

**第一条：让主导自己判断任务有多难。** 不是调用者说"给我生成 10 个工作者"。主导看到查询——"这个只需要 1 个 Agent 就够了"或"这个需要 10 个。"元认知——"这个任务需要多少资源？"——是主导系统提示的一部分。

**第二条：先广后深。** 不要一上来就深度研究每个子方向。先广泛检索——哪个方向真的有料？发现"FIPA-ACL 方向有几篇 2025 年的新论文"→ 再生产 2 个工作者深入。这是研究策略——不是架构配置。

**第三条：彩虹部署，不是蓝绿。** 你的 Agent 正在跑一个需要 6 小时的研究任务。运维要升级 Agent 版本。蓝绿部署——切流量——意味着终止一个跑了 5 小时 59 分的 Agent。彩虹部署：新版本 Agent 接新任务，旧版本跑完当前任务再退役。有状态的 Agent 需要状态感知的部署。

**第四条：令牌是主要成本变量。** 多 Agent 是单 Agent 令牌量的 15 倍。不是所有任务都值得付这个成本。如果你问的问题能用单 Agent 3-10 次工具调用解决——别用监管者。只在任务价值证明 15 倍令牌时升级。

## 什么时候监管者会崩

三种失败模式，每种都来自主导的"判断"。

**主导幻觉出计划。** 你问"多 Agent 系统在 2023-2026 年有什么变化。"主导分解成三个子问题——"FIPA-ACL 的历史"、"LangGraph 的性能优化"、"Agent 的内存管理"。三个子问题都和原问题相关——但没有一个直接回答"发生了什么变化"。工作者在错误的方向上做精确的研究。综合产出了一份精确但跑题的报告。

**工作者过度探索。** 主导说"找到 FIPA-ACL 遗产的相关论文。"工作者找到了 10 篇。然后它开始探索 FIPA 的历史、FIPA 的替代协议、FIPA 在电信行业的应用——主导没问这些。工作者带回来 3000 词的输出而不是 500 词。主导的综合被稀释了。

**综合冲突。** 工作者 A 说"2024 年的关键变化是 LangGraph 引入了检查点。"工作者 B 说"2024 年的关键变化是 AutoGen 发布了 v0.4 的事件驱动架构。"两者都对——但在"什么是关键变化"上冲突。主导怎么做？选 A？选 B？还是说"存在分歧"？Anthropic 的选择是明确的：要么重新问（一轮额外的调查），要么明确标注"关于这个问题存在分歧，两边的证据如下。"静默选一边是最差的——你欺骗了用户。

## 结课清单

部署监管者模式：

1. 模型配对：主导用推理模型（Opus 级），工作者用快模型（Sonnet 级）。
2. 主导不读原始材料——只读工作者摘要。上下文窗口留给综合推理。
3. 主导自己判断任务复杂度——1 个工作者还是 10 个。元认知是系统提示的一部分。
4. 工作者超时 + 令牌上限——防止一个失控工作者炸掉预算。
5. 彩虹部署——有状态 Agent 需要渐进版本转换。
6. 冲突检测——两个工作者矛盾 → 要么重问，要么明确标注。静默选一边 = 欺骗用户。
7. 只在任务价值证明 15 倍令牌成本时升级。

**最后一句话：**

**监管者模式的秘密不在"多"——在"隔离"。每个工作者拿到一个不被污染的上下文窗口，只处理一个狭窄问题。主导的综合只基于摘要而非全文。+90.2% 的提升不是来自并行推理——是来自信息密度的提升。如果你的问题塞不进一个窗口，不是因为模型不够强——是因为你需要多个窗口。**
