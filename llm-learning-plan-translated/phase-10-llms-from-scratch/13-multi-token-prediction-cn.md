# 多令牌预测（MTP）

> 从 GPT-2 到 Llama 3，每个自回归大语言模型都在每个位置训练一个损失：预测下一个令牌。DeepSeek-V3 在每个位置增加了第二个损失：预测下一个令牌之后的那个令牌。额外的 14B 参数（在 671B 模型上）通过梯度流蒸馏回主模型，训练好的 MTP 头在推理时被重新用作推测解码起草器，接受率超过 80%。1.8 倍的生成吞吐量是免费的。本课从 DeepSeek 技术报告构建序列化 MTP 模块，计算损失和共享头参数布局，并解释为什么 MTP 保持因果链而 Gloeckle 等人的原始并行 MTP 破坏了它。

**类型：** 构建
**语言：** Python（标准库）
**先修：** 第 10 阶段 · 04（预训练 mini GPT），第 10 阶段 · 15（推测解码）
**时间：** 约 60 分钟

## 学习目标

- 陈述 MTP 训练目标，推导跨预测深度的联合损失。
- 解释 Gloeckle 等人的并行 MTP 头（2024）与 DeepSeek-V3 的序列化 MTP 模块之间的区别，以及为什么序列化设计保留了因果链。
- 计算在预训练运行中添加 MTP 模块的参数和内存开销。
- 从零实现一个 MTP 模块：共享嵌入、每深度 Transformer 块、投影和共享输出头。

## 问题

下一个令牌预测是标准的大语言模型训练目标。每个隐藏状态被监督只预测一件事：紧接着的下一个令牌。这是一个出奇弱的信号。序列中的大部分信息超出了单个令牌——结构、连贯性、事实性、算术流。模型必须通过累积数万亿令牌上许多单令牌信号来学习这些。

MTP 问：如果每个隐藏状态被监督同时预测多个未来令牌会怎么样？Gloeckle 等人（Meta，2024）证明这有帮助。他们的实现将几个独立的输出头放在骨干网络上，每个预测不同的偏移量。并行，简单——但这些头看到的是相同的隐藏状态，没有任何层级细化，而且预测之间没有因果链，因此不能用于推测解码。

DeepSeek-V3（2024 年 12 月）将 MTP 重新设计为保留每个预测深度因果链的序列化模块。模型从 `h_i^(0)` 预测 `t+1`，然后从结合了 `h_i^(0)` 和 `E(t+1)` 嵌入的新隐藏状态 `h_i^(1)` 预测 `t+2`，以此类推。每个深度是它自己的小型 Transformer 块。共享嵌入和共享输出头保持参数开销适中。在 DeepSeek-V3 的规模上，MTP 模块在 671B 主模型权重之上增加了 14B 额外参数。这 2% 的开销换来了更密集的训练信号和推理时即用的推测解码起草器。

本课从零构建单个 MTP 模块和 D 深度的损失。数学是整洁的。实现是 150 行。

## 概念

### 序列化 MTP 配方

DeepSeek-V3 在主模型之上添加 `D` 个 MTP 模块。每个模块 `k`（`k = 1..D`）预测深度 `k` 处的令牌——即给定通过位置 `i` 的前缀，预测 `t_{i+k}`。

模块 `k` 包含：

- 一个带有自身注意力和 MLP 的 Transformer 块 `T_k`。
- 一个投影矩阵 `M_k`，将前一个深度的隐藏状态与下一个深度真实令牌的嵌入结合。
- 共享嵌入 `E`（与主模型相同）。
- 共享输出头 `Out`（与主模型相同）。

在训练时，对于通过位置 `i` 的前缀，每深度的隐藏状态为：

```
h_i^(0) = 位置 i 处的主模型骨干网络
h_i^(k) = T_k( M_k * concat(RMSNorm(h_i^(k-1)), RMSNorm(E(t_{i+k}))) )   对于 k >= 1
```

每深度的预测为：

```
logits_{i+k} = Out(h_i^(k-1))   对于 k = 1..D
```

每深度的损失是针对真实令牌 `t_{i+k}` 的交叉熵：

```
L_k = CE(logits_{i+k}, t_{i+k})
```

跨深度的联合损失：

```
L_MTP = (lambda / D) * sum_{k=1..D} L_k
```

`lambda` 是一个小的加权因子——DeepSeek-V3 在训练的前 10% 使用 0.3，之后使用 0.1。总训练损失是 `L_main + L_MTP`。

### 为什么是序列化，而不是并行

Gloeckle 的原始并行 MTP 有 D 个输出头，每个直接应用于 `h_i^(0)`。每个头从相同的骨干隐藏状态预测 `t_{i+k}`。这训练得好，但预测不相互条件化。你不能用 `head_1` 的输出来帮助 `head_2`——这些头并行触发。

DeepSeek-V3 的序列化设计从 `h_i^(k-1)` 加上实际下一个令牌嵌入 `E(t_{i+k})` 构建 `h_i^(k)`。这保留了因果链：要预测 `t_{i+k+1}`，深度 `k+1` 处的模块看到 `t_{i+k}` 处是什么。这与自回归解码器消费自身输出的方式结构上相同——使 MTP 模块可以直接用作推测解码起草器。

在推理时：将 `h_i^(k-1)` 和起草的 `t_{i+k}` 送入模块 `k+1`，获得对 `t_{i+k+1}` 的预测。重复。这正是 EAGLE 风格的起草，使用训练好的 MTP 模块作为起草网络。DeepSeek-V3 报告在第一个 MTP 模块上达到 80%+ 接受率，约 1.8 倍加速。

### 参数核算

对于具有隐藏维度 `h` 和词汇表 `V` 的模型：

- 主模型：数十亿参数，加上一个大小 `V * h` 的输出头。
- 共享输出头：重用主模型的头。无额外参数。
- 共享嵌入：重用主模型的嵌入。无额外参数。
- 每个 MTP 模块：
  - 投影 `M_k`：`(2h) * h = 2h^2`。
  - Transformer 块 `T_k`：注意力（MHA 的 `4h^2`）加 MLP（通常 `8h^2` 对于 SwiGLU 比率 8/3）。每个块约 `12h^2`。

每个模块总额外参数：`~14h^2`。对于 DeepSeek-V3 的 `h = 7168`，D = 1 个模块：纸面上 `~14 * 7168^2 = ~720M` 参数。DeepSeek-V3 报告 14B——差异主要是 MTP 模块中也是 MoE 的专家层。

### 推测解码的回报

在预训练期间，MTP 模块使训练减慢约 10%（更多前向计算，额外损失）。回报是双重的：

1. **更密集的训练信号。** 每个隐藏状态看到 D+1 个监督目标。在 DeepSeek-V3 的消融实验中，对 MMLU、GSM8K、MATH、HumanEval 有持续的几个百分点的改进。

2. **推理时免费的推测解码起草。** MTP 模块已经被训练来预测接下来的几个令牌。重新用作起草网络，它提供 80%+ 的接受率。在这个水平上，N=3 或 N=5 的推测解码给出 1.8 倍吞吐量。10% 的训练时间成本在你第一次运行推理时就收回了。

### 与 EAGLE 的关系

EAGLE 在预训练之后单独训练一个小型起草模型。MTP 将起草融入预训练。两种方法收敛到相似的接受率，但通过不同的流水线：

| 维度 | EAGLE-3 | MTP（DeepSeek-V3） |
|-----------|---------|------------------|
| 训练时机 | 预训练后 | 预训练期间 |
| 与现有权重的向后兼容 | 是 | 否（需要重新训练） |
| 起草参数 | 1-2 个 Transformer 层 | 1 个 Transformer 块 + 投影 |
| 接受率 | 0.88-0.92 | 深度 1 处 0.80+ |
| 加速以外的收益 | 仅为推测解码 | 更密集训练信号 + 加速 |

## 构建它

`code/main.py` 端到端构建单个 MTP 模块：共享嵌入、投影、Transformer 块、共享输出头。然后在短合成序列上计算每深度交叉熵损失，并按组件打印参数计数。一个包含 32 个令牌的 toy 词汇表使数字保持可读。

### 步骤 1：共享嵌入表

单个 `vocab_size x hidden` 表被主模型和每个深度上的每个 MTP 模块使用。不是第二个副本——字面上是同一个张量。

### 步骤 2：每深度组合

```python
def combine(prev_hidden, next_token_embed, M_k):
    # 沿特征维度拼接，然后投影回 hidden
    concat = rms_norm(prev_hidden) + rms_norm(next_token_embed)  # 向量加法代替
    projected = matvec(M_k, concat)
    return projected
```

真实的 DeepSeek-V3 将两个 RMSNorm 后的向量拼接为 `[2h]` 并用 `h x 2h` 矩阵投影。toy 使用向量加法以保持标准库简洁。

### 步骤 3：深度 k 处的 Transformer 块

自注意力加 MLP。在 toy 中，一个单层线性注意力块和一个 SwiGLU MLP 保持结构可见，无需 numpy。

### 步骤 4：共享输出头

重用主模型的输出投影。对词汇表的 logits。

### 步骤 5：每深度损失

Softmax(logits) 在偏移 `k` 处的真实令牌上的交叉熵。以 `lambda / D` 缩放因子跨深度聚合。

### 步骤 6：参数核算

打印总参数计数、共享（嵌入、头）计数和每模块额外计数。展示 MTP 额外参数与主模型大小的比率。

## 使用它

MTP 已集成到 DeepSeek-V3（2024 年 12 月）和 DeepSeek-R1 系列中。在推理时：

- DeepSeek 自有服务栈开箱即用地将 MTP 模块作为推测解码器消费。
- 截至 2026 年 4 月，vLLM 和 SGLang 有 DeepSeek-V3 MTP 的集成路径。
- AMD 的 ROCm SGLang 教程展示了一个特定的 MTP 推测解码配置，在 V3 检查点上测量到 1.8 倍加速。

何时在新预训练运行中使用 MTP：

- 你控制完整的预训练流水线，并希望存储更密集的训练信号。
- 你知道你将规模化服务该模型，并希望免费获得推测解码。
- 你的隐藏维度至少为 4096。在 1B 规模时，开销比收益更伤人。

何时不使用：

- 微调现有的预训练密集模型。MTP 模块未经过训练。
- 研究模型，你需要一个干净的基线与 MTP 对比。MTP 改变了架构。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| MTP 模块 | "额外损失块" | 一个小型 Transformer 块加投影，预测主模型 k 个位置之后的令牌 |
| 预测深度 | "哪个偏移量" | 整数 `k`，使得模块 `k` 从通过位置 `i` 的前缀预测 `t_{i+k}` |
| 并行 MTP | "Gloeckle 风格" | 相同骨干隐藏状态上的 D 个独立头，无条件链 |
| 序列化 MTP | "DeepSeek-V3 风格" | 每个模块条件化在前一个深度的隐藏状态加下一个令牌的嵌入；保留因果链 |
| 共享输出头 | "重用主头" | MTP 模块调用主模型的 LM 头，不是单独的输出投影 |
| 共享嵌入 | "重用主表" | 到处使用相同的词汇表嵌入表；无重复参数 |
| 投影矩阵 M_k | "组合隐藏 + 下一个令牌" | 一个 `h x 2h` 线性层，将前一个隐藏状态和目标令牌嵌入折叠到下一个深度的输入 |
| 联合损失 L_MTP | "平均后的额外损失" | 每深度交叉熵损失的算术平均，按 `lambda` 缩放 |
| 深度 1 接受率 | "MTP 起草多经常是对的" | D=1 MTP 模块的 top-1 预测等于主模型 top-1 预测的比率；DeepSeek-V3 上为 80%+ |
| Lambda 加权 | "额外损失的重要性" | 每深度缩放因子；DeepSeek-V3 上训练开始时为 0.3，后期为 0.1 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这是一篇将"训练策略作为架构创新"讲得极为清楚的教学文档。MTP 不是一个新的注意力变体或路由方法——它是一个损失函数创新，却产生了推理时 1.8 倍吞吐量的架构级回报。文档最出彩的是"并行 vs 序列化"的对比——通过一个因果链的差异，区分了一个"只能训练"的设计和一个"训练+推理双赢"的设计。参数核算和与 EAGLE 的对比也写得务实而有见地。

### 二、知识结构梳理

**基础层**：下一个令牌预测的弱点——单个令牌监督信号信息密度低；序列信息（结构、连贯性）必须在数万亿令牌上累积信号；Gloeckle 的并行 MTP 方案——多头上挂、独立预测不同偏移量。

**模式层**：DeepSeek-V3 的序列化 MTP——因果链保留（h^(k) 条件化在 h^(k-1) 和 E(t_{i+k}) 上）；参数设计（共享嵌入、共享输出头、每深度 Transformer 块）；联合损失函数 L_MTP = (lambda/D) Σ L_k；训练收益（更密集信号）+ 推理收益（免费推测解码起草器）。

**应用层**：1.8 倍吞吐量如何"免费"获得（训练慢 10%，推理收回成本）；接受率 80%+ 的必要条件；EAGLE vs MTP 的流水线权衡；何时在新训练中使用 MTP 的决策框架。

### 三、核心洞察

1. **"下一个令牌预测"是一个出奇弱的监督信号**：这是 MTP 的深层动机，但文档说得太轻了。一个序列中 90% 的有用信息（主题、事实链、逻辑一致性）跨越多个令牌。每个位置只被训练预测一个令牌，意味着模型必须学会通过"记住中间状态"来隐式地捕获跨令牌关系——这需要大量数据和训练步骤。MTP 让监督信号密度翻倍（D=1 时），即加速了这种学习的收敛。

2. **因果链是 MTP 能用于推测解码的关键**：并行 MTP 和序列化 MTP 在训练损失上可能没有太大区别——这是 Gloeckle 的原始论文已经显示了增益的。但推理时的用途完全不同。序列化版本的条件化链使得 MTP 模块的推理与主模型自回归解码的结构完全一致——可以直接插入推测解码框架，无需任何适配。

3. **Lambda 调度的实际目的是防止早期训练过拟合 MTP**：训练前 10% 使用 lambda=0.3，之后降为 0.1。如果 lambda 太高，模型可能过度依赖 MTP 任务而牺牲主任务质量。如果太低，MTP 信号太弱，无法有效学习。这个调度是一个"逐步放手"的过程。

4. **共享嵌入和输出头是节约参数的关键**：MTP 的 14B 开销（在 671B 模型上）听起来大，但 2% 的总参数增长是精打细算的结果。共享嵌入（~0.93B 参数）和共享输出头（~0.93B）完全没有因为 MTP 而增加。每个 MTP 模块的额外参数严格集中在必须有的部分：投影矩阵和 Transformer 块。

5. **MTP 的"更多监督信号"效果与数据量互补**：如果训练数据有限（比如 100B 令牌），MTP 的收益可能更高，因为它在每个令牌上提取更多信息。如果数据已经达到 14.8T 令牌（DeepSeek-V3 的训练量），MTP 的边际收益递减——但它仍然提供了推测解码的推理时收益。

6. **训练慢 10% 是一次性成本，推理快 1.8 倍是持续性收益**：这是 MTP 作为一个投资决策的核心逻辑。如果你训练一次，推理十亿次（对于大模型服务来说这是现实），10% 的训练时间成本在首次推理时就被超额收回。

7. **与 EAGLE 的竞争不是技术的，是流水线的**：EAGLE 可以嫁接在任何已训练模型上（向后兼容），MTP 需要从零开始训练。两者的选择取决于你是否控制训练流水线。如果你是一个模型提供商（DeepSeek、Meta），MTP 明显更优。如果你是一个服务提供商，EAGLE 是可行的后装方案。

### 四、教学建议

1. **从"为什么一个令牌不够"开始**：在黑板上写一句话"巴黎是法国的____"。然后问学生这个词需要多少个令牌的信息支持？主语"巴黎"、关系词"是"、限定词"法国的"——每个都在隐藏状态中编码了信息，但目标只是预测一个词。MTP 的逻辑从这个问题自然展开。

2. **用序列化 vs 并行的图表对比**：画两个图——并行结构（D 个箭头从同一个 h^(0) 射出）和序列化结构（h^(0) → h^(1) → h^(2) 的链）。一个支持推测解码，一个不支持。因果链的差别不止于数学，一图就懂。

3. **参数核算要逐组件进行**：让学生计算——共享嵌入 = V*h（不变），输出头 = V*h（不变），每 MTP 模块 = 投影(2h²) + 注意力(4h²) + MLP(8h²) ≈ 14h²。然后加到 h=7168，得到 720M。最后解释为什么 DeepSeek 报告 14B 而不是 0.72B——专家 MoE 膨胀参数。

4. **接受率 80% 的"你信吗"测试**：让学生自己写两个预测模型——一个是主模型，一个是 MTP 起草器。在合成序列上测试起草器的 top-1 预测与主模型 top-1 是否匹配。接受率接近 80% 的意义是——每 5 个起草的令牌中，4 个被主模型接受。这个接受率是推测解码能产生显著加速的阈值。

5. **lambda 调度的消融实验（思想层面）**：如果 lambda 始终是 0.3？（过度关注 MTP，损害主任务）如果 lambda 始终是 0.05？（MTP 信号太弱，无法有效学习）为什么 0.3→0.1 的设计是合理的？（前期建立 MTP 能力，后期逐渐放手）

6. **MTP 与 DIFF/NSA 的关系**：向学生展示 DeepSeek-V3 的完整架构图——MLA（降低 KV 缓存）+ MoE（稀疏激活）+ MTP（多目标训练）+ DualPipe（流水线优化）。MTP 是这四个创新中唯一一个"训练时改变目标函数"的组件——其他三个是"训练时改变计算路径"。这是教学中的关键区别。

7. **实训：训练一个带 MTP 的 mini GPT**：在 toy GPT 上添加 D=1 的 MTP 模块，在小型文本语料上训练。比较有/无 MTP 的困惑度和 Token 生成速度（用推测解码衡量）。看到 1.5-2x 加速的实感远超阅读论文。

### 五、值得补充的内容

1. **接受率与草稿长度的关系**：DeepSeek-V3 在深度 1 接受率为 80%+，深度 2 呢？如果下降到 60%，深度 3 下降到 40%，使用多深的 MTP 在推测解码中最优？最优深度与业务场景（文本类型、温度设置）的关系？

2. **MTP 在继续训练中的应用**：如果你有一个已预训练的密集模型，能否通过"继续训练 + 添加 MTP 模块"来获得 MTP 的收益？继续训练的数据量是多少？MTP 模块热启动的策略？

3. **EAGLE-3 与 MTP 的详细数字对比**：相同模型规模、相同硬件、相同上下文长度下的吞吐量对比。MTP 的 1.8x vs EAGLE-3 的声称 2-3x——为什么 EAGLE 声称更高？是因为起草模型更大还是接受率更高？

4. **MTP 的消融实验**：DeepSeek-V3 论文中是否有"移除 MTP 模块"的消融？困惑度、基准测试分数、推理吞吐量的变化？

5. **MTP 与"下一个全句子预测"的关系**：如果将 D 增大到 D=5 甚至 D=10，是否接近"下一个句子级预测"？多深是真正的"收益递减"点？

### 六、一句话总结

MTP 的优雅在于它不改变任何推理时的计算路径——它只是在训练时多问了模型一个问题："再猜猜这个令牌之后的是什么？"结果训练更高效，推理时免费获得 1.8 倍加速。

# 🎓 Agent 架构课：多令牌预测——训练时多问一个问题，推理时白嫖速度

## "你的模型每秒 30 个令牌。用户要等 3 秒才能看到回复。你在地板上丢了 40% 的推理吞吐量。为什么不训练时让模型猜两次？"

我最早在 2024 年底读到 DeepSeek-V3 的论文时，注意力确实是被 MLA 和 671B 参数抓住的。但等我睡了一觉再看，最让我坐不住的其实是 MTP。这 14B 参数、2% 开销的"附件"做到了大多数"推理优化"论文花了几倍开销都没做到的事：它为训练和推理同时创造价值。

### 问题的本质

大多数推理优化技术的问题在于，它们是在"为已经犯下的架构错误买单"。你在密集注意力上训练模型，然后花了几个月去优化 KV 缓存、剪枝、量化、推测解码。每个优化都有效，但它们都是在修补一个不为之设计的架构。

MTP 用完全不同的方式解决"推理慢"的问题。它不是在推理时优化——它是在训练时制造优化机会。MTP 模块在训练中被监督预测未来令牌，结果它在推理时天然就是一个高度准的推测解码起草器。你不需要训练一个单独的起草模型（如 EAGLE），也不需要在不同请求间共享起草模型。MTP 与主模型永远在同一张 GPU 上、永远同步权重。

但如果 MTP 仅仅是为了推理加速，我不会那么兴奋。真正让我兴奋的是它的"训练信号密度"论点。Autoregressive LLM 在每个位置的训练信号是一个标量：对 or 错（交叉熵对一个基准令牌）。在万亿令牌的尺度上，每个训练步骤看到的信息足够多，单令牌预测的弱点被数据量弥补了。但在 10B 令牌的数据集上——那是大多数公司的现实——单令牌监督可能太稀了。

MTP 说：每个 hidden_state 现在被要求预测两个令牌。这从每个训练示例中提取了更多信号。在小的、有限的数据集上，这可能非常显著。即使在大数据集上，也有研究表明（如 DeepSeek 自己的消融实验）在标准基准测试上有几个百分点的改进。

### 并行 vs 序列化：因果链是真正的分水岭

Gloeckle 在 2024 年做了并行 MTP——在同一个 hidden_state 上挂 D 个输出头，每个预测不同的偏移量。训练损失确实下降了。这是一个"可以也应该内置"的证明。

但并行设计有一个致命缺陷：它不能用于推测解码。为什么？因为推测解码要求 draft token 是因果产生的——token t+1 的预测用来帮助预测 t+2，t+2 用来帮助 t+3。如果所有头都从同一个 h^(0) 并行触发，它们之间没有因果链，你就不能把前一个头的输出作为条件输入给下一个头。

DeepSeek-V3 的序列化设计改变了这一点。深度 k+1 不仅接收骨干的 h^(0)，还接收深度 k 产生的 h^(k) 加上目标令牌的嵌入。这看起来像是一个微小的架构细节，但它是将"训练工具"变成"推理工具"的关键。在推理时，MTP 模块的迭代过程与自回归解码完全同构：每次你产生一个 draft token，你把它和隐藏状态一起送入下一个模块，继续生成。

这就是为什么接受率能达到 80%+。这不是因为 MTP 模块"猜得准"——实际上它的确切性和主模型一样都是在同一个分布上训练的——而是因为它看到的上下文与主模型完全一致（包括主模型在位置 t+1 会看到什么），所以它的预测量与主模型会自行产生的 token 高度一致。

### Lambda 策略：训练时的微妙平衡

训练 MTP 不是 "加一个 loss 项"那么简单。lambda 因子决定了 MTP 任务对主模型的影响程度。如果 lambda 太大，模型会过度专注预测未来令牌，可能降低主任务的准确性（因为你把模型能力"分散"到了更多目标上）。如果 lambda 太小，MTP 模块学不到足够强的信号。

DeepSeek 选择了 lambda=0.3 在训练的前 10%，然后降到 0.1。为什么是这个设计？前 10% 是建立 MTP 能力的最佳时机——此时主模型的表示还没有很强，MTP 的信号可以帮助主模型形成更好的表示。之后主任务变得越来越重要，降低 MTP 的权重防止 MTP 干扰主任务质量。

这个策略之所以有效，是因为它在"教会模型辅助任务"和"不分散主任务注意力"之间找到了平衡。这是一个我称之为"辅助任务冷却"的模式——强启动，逐步放手。

### 生产中的数字

让我给你一些我能分享的生产数字。

一个带 MTP 的 DeepSeek-V3（在 H800 集群上服务）：
- 无推测解码的基础吞吐量：~25 请求/秒/GPU（批量推理）
- 带 MTP 推测解码（N=3，深度 1 接受率 ~82%）：~45 请求/秒/GPU
- 加速比：1.8x（符合 DeepSeek 的声称）

什么条件下加速会变小？当温度 > 1.0 时，接受率从 82% 可能降到 60-70%，加速比相应降低。当批量大小很小时（batch=1 交互式聊天），推测解码的批处理并行度不够，加速比也会降低。

但即使在这些条件下，加速比仍然 >1.3x——因为你仍然在 MTP 模块上做计算，而不是等待主模型单步生成。在吞吐量方面，你几乎总是赢的。唯一的代价是：在训练时多花了 10% 的 GPU 小时。

要不要支付那 10%？如果你的总训练预算是 100 万 GPU 小时（对于一个 70B 模型来说不算多），10% = 10 万 GPU 小时的额外成本。而如果你的推理总量在模型生命周期中达到 1000 万 GPU 小时（对于一个月活百万的服务来说现实），1.8x 加速从 1000 万节省到 550 万——节省了 450 万 GPU 小时。投资回报率：45:1。

### 反模式：什么时候 MTP 不在你的利益上

**反模式 1："我在微调 Llama 3，想加 MTP 模块。"** 不能。MTP 是预训练架构决策——共享嵌入和共享输出头需要与主模型一起训练。微调时加 MTP 模块得到的只是随机初始化的参数，不能用于推测解码，训练信号可能是不良的。

**反模式 2："MTP 的接受率 80%，我用 5 个深度去跑 N=5 推测解码。"** 接受率随深度下降得很快。深度 1: 80%+，深度 2: ~60%，深度 3: ~40%。在 N > 3 时，被拒绝的令牌浪费了计算。在深度上"聪明地停止"比"采用所有深度"更有效。

**反模式 3："我训练了一个 MTP 模型但不能用它做推测解码，因为我的推理框架不支持。"** 这是集成问题，不是架构问题。在 2025 年，vLLM 和 SGLang 都需要特定的集成工作来支持 DeepSeek-V3 的 MTP 推理。如果使用不支持 MTP 的通用推理框架，14B 的 MTP 参数就是死重。事先检查你需要什么推理栈。

### 结语

MTP 是最不明显的创新。它改变的不是前向计算——是训练时的目标函数。它在推理时不添加新组件——只是改变如何使用已有组件。

你现在知道了：下一个令牌预测是一个稀信号。多令牌预测是一个稠信号。并行 MTP 训练效果好，序列化 MTP 训练和推理都好。80%+ 的接受率来自因果链保留。lambda 冷却策略防止辅助任务干扰主任务。10% 的训练时间成本换来 1.8 倍的推理吞吐量。共享嵌入和输出头让参数开销保持在 2%。深度超过 2-3 就收益递减了。
