# 多头注意力（Multi-Head Attention）

> 一个注意力头一次学一种关系。八个头学八种。头是免费的。多拿几个。

**类型：** 构建
**语言：** Python
**前置要求：** Phase 7 · 02（从零实现自注意力）
**时间：** 约75分钟

## 问题所在

单个自注意力头计算一个注意力矩阵。这个矩阵捕获一种关系——通常是那个在训练信号上最小化损失的关系。如果你的数据同时包含主谓一致、共指消解、长程语篇和句法分块，单头会把这些全部涂抹进一个 softmax 分布，丢失一半的信号。

2017年 Vaswani 论文的修复：并行运行多个注意力函数，每个有自己的 Q、K、V 投影，然后拼接输出。每个头在维度 `d_model / n_heads` 的更小子空间中操作。总参数量不变。表达能力上升。

多头注意力是2026年每个 Transformer 默认标配。唯一的争论在于*多少个头*，以及键和值是否共享投影（分组查询注意力GQA、多查询注意力MQA、多头潜在注意力MLA）。

## 核心概念

![多头注意力：分拆、注意力、拼接](../assets/multi-head-attention.svg)

**分拆。** 取形状为 `(N, d_model)` 的 `X`。投影到 Q、K、V，各形状为 `(N, d_model)`。重塑为 `(N, n_heads, d_head)`，其中 `d_head = d_model / n_heads`。转置为 `(n_heads, N, d_head)`。

**并行注意力。** 在每个头内部运行缩放点积注意力。每个头产生 `(N, d_head)`。这些头在嵌入的不同子空间上操作，在注意力计算本身期间从不交流。

**拼接并投影。** 将头堆回 `(N, d_model)`，乘以可学习的输出矩阵 `W_o`，形状为 `(d_model, d_model)`。`W_o` 是头之间混合的地方。

**为什么有效。** 每个头可以专业化，而不与其他头竞争表示预算。2019-2024年的探针研究揭示了不同的头角色：位置头、关注前一个令牌的头、复制头、命名实体头、归纳头（支撑上下文学习的关键电路）。

**2026年变体系列：**

| 变体 | Q 头数 | K/V 头数 | 使用者 |
|------|--------|---------|--------|
| 多头注意力（MHA） | N | N | GPT-2, BERT, T5 |
| 多查询注意力（MQA） | N | 1 | PaLM, Falcon |
| 分组查询注意力（GQA） | N | G（例如 N/8） | Llama 2 70B, Llama 3+, Qwen 2+, Mistral |
| 多头潜在注意力（MLA） | N | 压缩到低秩 | DeepSeek-V2, V3 |

GQA 是现代默认配置，因为它以 N/G 的因子削减 KV 缓存内存，同时保持近乎全部质量。MLA 进一步将 K/V 压缩到潜在空间，然后在计算时回投影——消耗计算量，大幅节省内存。

## 动手构建

### 步骤 1：从单头注意力分拆出头

从第02课取 `SelfAttention`，用分拆/拼接对包装。参见 `code/main.py` 的 numpy 实现；逻辑是：

```python
def split_heads(X, n_heads):
    n, d = X.shape
    d_head = d // n_heads
    return X.reshape(n, n_heads, d_head).transpose(1, 0, 2)  # (heads, n, d_head)

def combine_heads(H):
    h, n, d_head = H.shape
    return H.transpose(1, 0, 2).reshape(n, h * d_head)
```

一次重塑和一次转置。没有循环。这正是 PyTorch 在 `nn.MultiheadAttention` 底层做的事。

### 步骤 2：按头运行缩放点积注意力

每个头获得自己的 Q、K、V 切片。注意力变成批量矩阵乘法：

```python
def mha_forward(X, W_q, W_k, W_v, W_o, n_heads):
    Q = X @ W_q
    K = X @ W_k
    V = X @ W_v
    Qh = split_heads(Q, n_heads)         # (heads, n, d_head)
    Kh = split_heads(K, n_heads)
    Vh = split_heads(V, n_heads)
    scores = Qh @ Kh.transpose(0, 2, 1) / np.sqrt(Qh.shape[-1])
    weights = softmax(scores, axis=-1)
    out = weights @ Vh                    # (heads, n, d_head)
    concat = combine_heads(out)
    return concat @ W_o, weights
```

在真实硬件上，`Qh @ Kh.transpose(...)` 是一次 `bmm`。GPU 将其视为一个批量矩阵乘法，形状为 `(heads, N, d_head) × (heads, d_head, N) → (heads, N, N)`。增加头数是免费的。

### 步骤 3：分组查询注意力（GQA）变体

只有键和值投影改变。Q 获得 `n_heads` 组；K 和 V 获得 `n_kv_heads < n_heads` 组，然后重复以匹配：

```python
def gqa_project(X, W, n_kv_heads, n_heads):
    kv = split_heads(X @ W, n_kv_heads)       # (kv_heads, n, d_head)
    repeat = n_heads // n_kv_heads
    return np.repeat(kv, repeat, axis=0)      # (n_heads, n, d_head)
```

推理时这节省内存，因为只有 `n_kv_heads` 份副本存在于 KV 缓存中，而不是 `n_heads` 份。Llama 3 70B 使用64个查询头和8个 KV 头——缓存缩减8倍。

### 步骤 4：探测每个头学到了什么

用4个头对短句子运行 MHA。对每个头，打印 `(N, N)` 注意力矩阵。即使随机初始化，你也会看到不同头捕捉不同结构——部分是信号，部分是子空间中的旋转对称性。

## 实战使用

PyTorch 的一行版本：

```python
import torch.nn as nn

mha = nn.MultiheadAttention(embed_dim=512, num_heads=8, batch_first=True)
```

PyTorch 2.5+ 的 GQA：

```python
from torch.nn.functional import scaled_dot_product_attention

# scaled_dot_product_attention 在 CUDA 上自动分派 Flash Attention。
# 对于 GQA，传入形状为 (B, n_heads, N, d_head) 的 Q
# 和形状为 (B, n_kv_heads, N, d_head) 的 K, V。PyTorch 处理重复。
out = scaled_dot_product_attention(q, k, v, is_causal=True, enable_gqa=True)
```

**多少个头？** 2026年生产模型的参考经验值：

| 模型规模 | d_model | n_heads | d_head |
|---------|---------|---------|--------|
| 小型（~125M） | 768 | 12 | 64 |
| 基础（~350M） | 1024 | 16 | 64 |
| 大型（~1B） | 2048 | 16 | 128 |
| 前沿（~70B） | 8192 | 64 | 128 |

`d_head` 几乎总是落在64或128。这是一个头能"看见"多少信息的单位。降到32以下，头开始与缩放因子 `sqrt(d_head)` 对抗；超过256，你会丧失"许多小专家"的收益。

## 练习

1. **简单。** 取 `code/main.py` 中的 MHA，在 `d_model=64` 固定的情况下将 `n_heads` 从1变到16。在合成复制任务上绘制一个微小的单层模型的损失。更多头是帮助、达到平台还是有害？
2. **中等。** 实现 MQA（一个 KV 头在所有查询头之间共享）。测量参数计数比完整 MHA 下降了多少。计算在 N=2048 推理时 KV 缓存大小缩减了多少。
3. **困难。** 实现微小的多头潜在注意力版本：将 K、V 压缩到秩为 `r` 的潜在表示，在 KV 缓存中存储该潜在表示，注意力时解压。在什么 `r` 下缓存内存低于完整 MHA 的 1/8 而质量保持在验证困惑度 1 bit 内？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 头（Head） | "单个注意力电路" | 维度 `d_head = d_model / n_heads` 的一个 Q/K/V 投影，有自己的注意力矩阵。 |
| d_head | "头维度" | 每头的隐藏宽度；生产中几乎总是64或128。 |
| 分拆/合并 | "重塑技巧" | `(N, d_model) ↔ (n_heads, N, d_head)` 在注意力周围的重塑+转置。 |
| W_o | "输出投影" | 拼接头后应用的 `(d_model, d_model)` 矩阵；头混合的地方。 |
| MQA | "一个 KV 头" | 多查询注意力：单个共享 K/V 投影。最小的 KV 缓存，有质量损失。 |
| GQA | "Llama 2 以来的默认" | 分组查询注意力，`n_kv_heads < n_heads`；重复以匹配 Q。 |
| MLA | "DeepSeek 的绝招" | 多头潜在注意力：K、V 压缩到低秩潜在表示，注意力时解压。 |
| 归纳头 | "上下文学习背后的电路" | 一对头检测先前的出现并复制其后跟随的内容。 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这篇多头注意力教材在单头自注意力基础上完成了自然的知识跃迁。从"分拆→并行→拼接→投影"的四步走讲清楚了 MHA 不是新东西，只是单头的复用加组合。尤其难得的是直接覆盖了 2024-2026 年的四代变体（MHA/MQA/GQA/MLA），这在大多数教学材料中都是缺失的。

### 二、知识结构梳理

- **基础层**：reshape+transpose 的"分拆技巧"、批量矩阵乘法的 GPU 语义、W_o 输出投影的维度匹配
- **模式层**：四个变体的架构差异（重点在 KV 共享程度与缓存内存的权衡）、d_head 的最优选择经验法则
- **应用层**：不同模型规模的注意力配置表、PyTorch 2.5+ 的 `enable_gqa` 标志、归纳头与上下文学习的关系

### 三、核心洞察

1. **多头不是"加法"是"乘法"**：8个头不是把表达能力乘8，而是让不同头占据不同的表示子空间——每个头少了几分之一的维度，但多了平等的发言权。
2. **reshape+transpose 就是你需要的全部并行化**：不需要线程池、不需要进程通信。两个张量操作就完成了"切成 heads → 同时计算 → 拼回来"的整个并行流程。
3. **KV 缓存的真正成本在 GQA/MLA 里才显露**：64个头的 MHA 在推理时需要64组 K/V 向量。8个 KV 头的 GQA 只需要8组。节省的是显存，换来的是吞吐量。
4. **MLA 是"以计算换内存"的极端案例**：DeepSeek 把 K/V 投影到低秩空间存储，注意力时解压——解压这个 FLOPs 成本远小于 HBM 访问成本。这反映了 GPU 架构的根本不对称：计算便宜，内存访问贵。
5. **归纳头是上下文学习的"硬件"**：某些注意力头专门负责"之前出现的模式 X 后面跟了什么？"——这就是为什么模型能从 prompt 里的几个例子中学会新任务。
6. **d_head=64 这个数字不是偶然**：低于32，softmax scaling 开始出问题；高于128，单个头"看到"太多维度和全局注意力重叠，多头收益递减。
7. **W_o 是头混合器，不是一个可有可无的线性层**：去掉 W_o 等于剥夺了不同头"对话"的能力——每个头的输出只是简单拼接，没有互动。

### 四、教学建议

1. **从单头直接改 MHA——让学生看代码的差异**：代码改动只有 split_heads 和 combine_heads 两个函数，其余完全不变。学生会惊呼"就这？"
2. **用不同颜色的粉笔画不同头的注意力矩阵**：4个黑色/红色/蓝色/绿色的方阵叠在一起——让学生直观感受"头在处理不同信息"。
3. **做一个"头剪枝"实验**：训练好的模型，逐个移除注意力头，测量损失变化——某些头移除后损失几乎不变，某些头是"关键的"。这能教会学生"头的重要性不同"。
4. **对比 MHA、MQA、GQA 在同一批次中的 KV 缓存大小**：具体到 MB 的数字，比如"Llama 70B 在128K上下文：MHA=32GB, GQA=4GB"——学生永远不会忘记这个差一个数量级的结果。
5. **讲归纳头时引用 Olsson 原论文的图**：那个"检测模式→复制后继"的机制图是上下文学习最优雅的解释。
6. **把 W_o 单独抽出来讲解**：大部分教材把 W_o 当作一个"收尾的线性层"。实际上它是头混合层——没有 W_o，attention heads 的丰富性无法转化为更好的 token 表示。
7. **做一次"调 d_head 看性能"的实验**：固定 d_model=1024，n_heads 分别取 4（d_head=256）、8（128）、16（64）、32（32），在一个小任务上看 PPL——学生会看到 U 形曲线。

### 五、值得补充的内容

1. **注意力头的结构多样性分析**：BERTology 研究（Clark et al. 2019）发现有些头关注相邻令牌、有些关注特定词性、有些关注分隔符——可以选几张经典注意力可视化图展示。
2. **Head Pruning 的生产实践**：训练时砍掉"无用"的头可以加速推理——这种做法在大规模部署中的 ROI 分析。
3. **Cross-Attention vs Self-Attention 的头行为对比**：在编码器-解码器架构中，cross-attn 的头行为和 self-attn 的头行为完全不同。
4. **Attention Sink 现象**：第一个令牌在注意力中吸收了不成比例的大量权重，导致其他令牌信息被淹没——这个现象在流式解码中是重大工程挑战。
5. **Multi-Latent Attention 的数学细节**：低秩压缩的具体矩阵分解形式、解压时的计算开销公式、为什么解压开销远小于 HBM 访问成本。

### 六、一句话总结

多头注意力的本质不是"多加几个头"，而是"在同一个表示空间中开辟多个并行信令通道"——每个通道以更低的维度和更高的特异性捕获不同的语言关系，W_o 是这些通道的协调器。

# 🎓 Agent 架构课：多头注意力——为什么你要拿8个而不是1个？

**副标题：从"一个头看全局"到"八个专家各自为政"的架构权衡**

---

先问一个问题——

如果你有一个1280维的表示空间，你要让注意力在这个空间里去"查东西"。你会用一个"1280维的大窗口"去看，还是用8个"160维的小窗口"分别去看？

……

你的直觉可能告诉你：一个大窗口"看到的多"，信息量更大。但**在注意力机制里，这个直觉是错的。错得很惨。**

让我解释。

单头注意力的问题不是"它不够大"。问题是它只有一个注意力分布。一个概率分布。一个"注意力预算"。所有关系——语法的一致关系、实体的指代关系、长短距离的话篇关系——全部在这个分布里竞争。最后训练信号只选一条路径梯度最强的方向推进。其他关系？一路被埋进噪声里。

多头注意力的核心洞察是：**给每个关系类型一个独立的注意力预算。** 你切8个头，你就有8个概率分布。头1可以专门看前一个令牌（预测下一个词），头2可以专门看句子开头（主题一致性），头3可以专门看最近的名词短语（句法一致性）。这些头不会互相争夺——它们各自有各自的 softmax，各自有各自的 Q/K/V 投影。

这就是为什么 d_head=64 是行业标准。不是"够用"，而是**恰好够让一个头有足够的维度去看清一种关系，又不至于维度太高让它和全局注意力重叠太多。**

现在让我给你两条路径：

**路径一：MHA——每个头有自己独立的 K/V。**
64个查询头，64个 K/V 头。训练时没问题——显存里的 K/V 矩阵是临时的，计算完就扔了。推理时？灾难。每个令牌每层你都要存储64组 K/V 向量。在 128K 上下文的 Llama 70B 上，KV 缓存吃掉32GB显存。**你的 A100-40GB 几乎全给缓存了，留给模型参数和计算的不到8GB。**

**路径二：GQA——8个查询头共享1个 KV 头。**
同样的64个查询头，但只有8个 K/V 头。每个 K/V 向量被8个查询头复用。KV 缓存从32GB降到4GB。损失的质量？在大多数基准上，困惑度的差异不到0.05。**你用0.05的困惑度换来了8倍的吞吐量。**

我在给生产环境选型的时候，从来没有在"纯 MHA"和"GQA"之间犹豫过。除非你只做训练不做推理部署，否则 GQA 是唯一选项。2026年的标准是 n_kv_heads = n_heads / 8 或 /4。

但 GQA 的细节有一个坑，我必须警告你。**K/V 头是重复的，不是插值的。** 如果你有8个 KV 头、64个 Q 头，就是把每组 K/V 复制8份。这意味着被分到同一组的那8个查询头"看到的东西是严格相同的键值信息"。它们的差异只能来自不同的查询投影。如果你的模型把语义头全部塞到一组里——实践中确实会发生——那一组的表达能力就被限制了。

MLA（DeepSeek 的方案）更进一步。**它不重复 K/V，它把它们压缩到一个低秩潜在空间，然后在注意力时解压。** 缓存里只存压缩后的向量——大概是原始大小的 1/16 或更小。注意力时解压的 FLOPs 成本远低于 HBM 访问成本，因为 GPU 的 SRAM 比 HBM 快10倍。**这是在用计算买内存——2026年最好的交易。**

再给你看一个具体的反模式。我见过不止一个团队在培训 Transformer 时把 n_heads 设成不能被 d_model 整除的数。这种事不应该发生，但它确实发生了——因为他们在写配置时不手动做整除此检查。如果你的 d_model=768、n_heads=7，d_head=109.7……这不是一个整数。你的 split_heads 要么报错（pytorch 会），要么静默地对不齐维度。调试这个发疯。

还有一个更隐蔽的反模式：**把 W_o 当成"可选的线性层"而省略。** 没有 W_o，头之间唯一的交互是拼接——不同的头输出不同的通道，但通道之间没有混合。加了 W_o，头之间开始"对话"：头1可以说"我看到一个位置关系"，头2可以说"我看到一个语义关系"，W_o 可以学会"当它们都激活时才输出强特征"。这是 Transformer 内部最重要的信息整合步骤之一。

**关于教案里提到的"归纳头"。** 这不是一个学术概念，它解释了你每天在用的东西。当你给 GPT 一个示例——"中文：你好 → 英文：hello，中文：谢谢 → 英文："——能回答问题，是因为某些注意力头专精于"之前的 pattern X 后面跟着什么"的检测和复制。这解释了为什么模型在少数示例中就能学会新行为——注意力头已经被训练成了 pattern-copy 机。

最后的清单，如果你在设计一个 Transformer 架构：

- ✅ 你的 d_head 落在 64-128 范围内吗？——不在的话，你大概率在浪费容量或制造不稳定。
- ✅ 你的推理场景用了 GQA 或 MLA 而不是纯 MHA 吗？——纯 MHA 在长上下文推理中是显存自杀。
- ✅ 你的 W_o 是可训练的吗？——不可省略。
- ✅ 你有监控"哪些头实际被用了"的机制吗？——大部分的生产模型里，20%的头贡献了80%的信号。知道哪些头是冗余的可以指导量化或剪枝。
- ✅ 如果你用 GQA，你确认了 KV 分组结构在初始化时与 Q 头对齐吗？——不对齐的静默错误是最难抓的。

**金句：多头注意力不是"多双眼睛看同一个画面"——是"把一个大房间隔成八个小办公室，每个办公室有一个专家，W_o 是让他们开会用的话筒。"**

回到开头的问题：为什么拿8个而不是1个？不是一个"越多越好"的答案。而是因为**一个注意力分布只能表达一个关系结构的分词方式**——而语言里有几十种关系同时发生。多几个头，你不需要额外的参数，不需要额外的计算量，你只需要 reshape 一次、transpose 一次——**然后单次 attention 变成并行 attention，信息密度提升的不是倍数而是质变。**
