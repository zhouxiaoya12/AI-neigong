# 提示缓存与上下文缓存

> 你的系统提示是4000个令牌。你的RAG上下文是20000个令牌。你每次请求都发送两者。你也每次都付钱——每次。提示缓存让提供商在他们那边保持这个前缀处于热状态，并在重用时只按正常价格的10%计费。正确使用时，它将推理成本削减50-90%，首令牌延迟削减40-85%。

**类型：** 构建 | **语言：** Python | **前置要求：** Phase 11 · 01, 05, 11 | **时间：** 约60分钟

## 问题

一个编程Agent在对话的每一轮向Claude发送相同的15000令牌系统提示。20轮×$3/百万输入令牌=$0.90仅输入成本——在任何用户实际消息之前。乘以每天10000次对话，账单达到$9000/天——为从未改变的文本。

**你不能缩小提示而不损害质量。你不能避免发送它。** 唯一的办法是停止为提供商已经看过的前缀付全价。

那就是提示缓存。Anthropic在2024年8月发布（2025年推出1小时扩展TTL变体），OpenAI在当年晚些时候自动化了它，Google随Gemini一同发布了显式上下文缓存。三者现在都在其前沿模型上将其作为一等功能提供。

## 概念

### 机制

当请求的前缀与近期请求的前缀匹配时，提供商从之前的运行中提供KV缓存，而不是重新编码令牌。首次你付一个小的写入溢价，之后每次都付大的读取折扣。

### 2026年三家提供商的风格

| 提供商 | API风格 | 命中折扣 | 写入溢价 | 默认TTL | 最小缓存量 |
|--------|---------|---------|---------|---------|-----------|
| Anthropic | 显式`cache_control`标记 | 输入9折 | 25%附加费 | 5分钟（可扩展至1小时） | 1024令牌 |
| OpenAI | 自动前缀检测 | 输入5折 | 无 | 最多1小时（尽力而为） | 1024令牌 |
| Google (Gemini) | 显式`CachedContent` API | 存储计费；读取约正常价格的25% | 存储费/token·小时 | 用户设置（默认1小时） | 4096令牌 |

### 缓存友好的布局

```
[系统提示]          ← 缓存这个
[工具定义]           ← 缓存这个
[少样本示例]         ← 缓存这个
[检索文档]          ← 如果重复使用则缓存，否则不缓存
[对话历史]          ← 缓存到最后一轮为止
[当前用户消息]       ← 永不缓存（每次都不同）
```

### 盈亏平衡计算

Anthropic的25%写入溢价意味着缓存块至少需要读取2次才能净省钱。1次写入+1次读取=每请求平均0.675x成本（节省32%）；1次写入+10次读取=平均0.205x（节省80%）。

**经验法则：缓存任何你预计在TTL内至少重用3次的内容。**

## 构建

### Anthropic显式缓存

```python
import anthropic

SYSTEM = [{
    "type": "text",
    "text": "You are a senior Python reviewer..." + RUBRIC,
    "cache_control": {"type": "ephemeral"},
}]

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=1024,
    system=SYSTEM,
    messages=[{"role": "user", "content": code}],
)

# 检查缓存命中
print(response.usage.cache_creation_input_tokens)  # 首次：15023
print(response.usage.cache_read_input_tokens)       # 后续：15023（0.1x价格）
```

### OpenAI自动缓存

```python
# 零代码变更——自动前缀匹配
# 检查是否工作：
print(response.usage.prompt_tokens_details.cached_tokens)
```

### Google显式缓存

```python
# 创建缓存内容资源
cache = genai.caching.CachedContent.create(
    model="gemini-1.5-pro",
    system_instruction="You are a helpful...",
    ttl=datetime.timedelta(hours=1),
)
# 在后续请求中引用缓存
model.generate_content("query", cached_content=cache.name)
```

## 缓存命中率

| 提示类型 | 典型缓存命中率 | 节省 |
|---------|-------------|------|
| 系统提示（静态） | 95-100% | 30-50% 总成本 |
| 工具定义（静态） | 95-100% | 10-20% 总成本 |
| RAG上下文（可变） | 10-40% | 5-15% 总成本 |
| 对话历史（逐轮增长） | 60-90% | 15-25% 总成本 |
| 用户消息 | 0-5% | 可忽略 |

最佳情况（系统提示+工具+历史全部命中）：总成本削减65-90%。

## 关键术语

| 术语 | 含义 |
|------|------|
| 提示缓存 | KV缓存在请求间保持热状态，避免重新编码匹配的前缀 |
| KV缓存 | 每个令牌在注意力层中的键和值向量——提示缓存重用的就是这个 |
| 写入溢价 | 缓存块的首次请求收取更高的输入费率 |
| TTL | 缓存在最近一次访问后保持存活的时间 |
| 盈亏平衡点 | 块必须被重用的次数才能产生净节省 |

## 📝 教师备课总结与读后感

### 一、文档整体评价

这篇文档将"提示缓存"从一个省钱技巧提升为架构必须。它不是"要不要缓存"，而是"如果每一轮都发相同的15000令牌系统提示，那你就是在烧钱"。精华在于三家提供商的对比——Anthropic的显式控制（90%折扣）、OpenAI的零代码（50%折扣）、Google的按小时存储——以及"缓存友好布局"的原则：稳定部分放前面，可变部分放后面。

### 二、核心洞察

1. **前缀顺序是缓存命中率的第一决定因素**：稳定内容必须在前，否则全部失效
2. **Anthropic的90%折扣是最好的成本杠杆**：但需要显式标记和25%写入溢价
3. **OpenAI的50%折扣自动且免费**：代码零改动但不能控制什么被缓存
4. **盈亏平衡点是至少3次重用**：1次写入+1次读取=净节省32%，写入+10次读取=80%
5. **系统提示缓存命中率95-100%**：这是一个"免费的钱"的机会

### 三、一句话总结

**提示缓存不是优化——它是纠正。为每次都相同的文本每次付全价不是"使用API"，是"浪费预算"。**

# 🎓 Agent 架构课：提示缓存——为什么你在为相同的文本反复付钱

15,000个令牌。这就是你的系统提示。每次对话的每一轮都被发送。20轮×$3/百万令牌=$0.90。每天10,000次对话=$9,000。仅系统提示。

这不是"使用AI的成本"。这是"重复的税"。

注意力的KV缓存（键值缓存）——Transformer在计算自注意力时产生的中间表示——在相同前缀的连续请求中可以被复用。Anthropic在2024年把它变成了一个产品功能：标记你希望缓存的部分，首次付25%的写入溢价，90%的折扣在后续命中时读取。

这是"不再为相同的工作付两次钱"的LLM等价物。

## 结语清单

1. ☐ 系统提示是否标记为`cache_control: ephemeral`（Anthropic）或前缀足够长以触发自动缓存（OpenAI）？
2. ☐ 静态内容（系统、工具、少样本示例）是否放在提示的最前面？
3. ☐ 动态内容（RAG上下文、历史）是否放在变量部分，当重用时可选缓存？
4. ☐ 用户消息是否从不放在可能匹配缓存前缀的位置？
5. ☐ 是否监控了`cache_read_input_tokens`以确保缓存命中？
6. ☐ 对于Anthropic，是否知道盈亏平衡点是至少3次重用？

**一句金句：你不是在"发送提示"，你是在"每次为相同的工作付费"——提示缓存让你停止为重复买单。**
